Javascript 101

Cohort 2.0 · 5 lessons · SDE-2 (3+ yrs) interview prep

Concept summary — all 5 steps

Coding consistency check
A habit/motivation prompt, not a technical concept: the point is that fluency in JS comes from daily reps (small programs, dev-tools console experiments) rather than passive reading. Interview angle is indirect — consistent practice is what makes you fast at whiteboard/live-coding rounds where you can't lean on autocomplete.
What JavaScript is
JS is a high-level, dynamically-typed, single-threaded language executed by an engine (V8, SpiderMonkey) that JIT-compiles source to machine code rather than purely interpreting it. It runs in browsers and on servers (Node) via the same core language (ECMAScript) plus a host-provided runtime (DOM APIs vs Node APIs). Interviewers probe whether you understand the language is separate from its runtime, and that 'single-threaded' refers to the call stack while async I/O is offloaded.
Core properties of JS
Key properties: dynamic and weak typing (types resolved at runtime, implicit coercion), prototype-based inheritance (objects delegate to a prototype chain, not classes under the hood), first-class functions (functions are values — passed, returned, stored), and a single-threaded event-loop concurrency model. These properties explain most 'gotcha' questions: coercion bugs, closures, `this` binding, and why blocking the main thread freezes everything.
JavaScript syntax
Covers declarations (`var` function-scoped/hoisted-as-undefined vs `let`/`const` block-scoped with temporal dead zone), operators (`===` strict vs `==` coercing, `??` vs `||`), control flow, and function forms (declarations, expressions, arrow functions). The load-bearing interview points are scoping rules, hoisting, and that arrow functions lexically bind `this` and lack their own `arguments`.
Complex (reference) types
Objects, arrays, and functions are reference types: variables hold a pointer, so assignment and function arguments copy the reference, not the value — mutations are shared and `===` compares identity, not structure. Contrast with primitives (string, number, boolean, null, undefined, symbol, bigint) which are copied by value and immutable. This is the root of shallow-vs-deep-copy questions, accidental shared-state mutation bugs, and why `{} === {}` is false.

Real-world example

Debugging a shared-state mutation in a shopping-cart module

You build a cart service where a default line-item template `const DEFAULT_ITEM = { qty: 1, options: {} }` is spread into each new item: `const item = { ...DEFAULT_ITEM, sku }`. QA reports that setting a gift-wrap option on one product silently applies it to every product added afterward. The bug ties the whole topic together: the spread does a shallow copy, so every item's `options` still points to the same nested object (reference type), and mutating `item.options.giftWrap = true` mutates the shared original. The fix requires understanding value-vs-reference semantics (create a fresh nested object per item, e.g. `structuredClone` or an explicit `{ ...DEFAULT_ITEM, options: {}, sku }`), plus knowing that `===` on the leaked objects would show they're the same identity — exactly the primitives-vs-objects distinction taught in this track.

Interview Q&A (SDE-2)

Q1. What does it mean that JavaScript is single-threaded, and how does it handle asynchronous work?
There is one call stack, so only one piece of JS runs at a time — the language itself has no parallelism on the main thread. Async work (timers, network, I/O) is delegated to the host runtime (browser Web APIs or libuv in Node); when it completes, a callback is queued and the event loop pushes it onto the stack once the stack is empty. This is why a long synchronous loop freezes the UI, and why microtasks (Promises) drain fully before the next macrotask (setTimeout) runs.
Q2. Explain the difference between == and === and why you'd prefer one.
`===` is strict equality: it returns true only if type and value match, with no coercion. `==` performs type coercion before comparing, following non-obvious rules (e.g. `0 == ''` and `null == undefined` are true, but `null == 0` is false). Prefer `===` to avoid coercion surprises; the one common idiomatic use of `==` is `x == null` to check for both null and undefined in a single expression.
Q3. Contrast var, let, and const in terms of scope and hoisting.
`var` is function-scoped and hoisted such that the binding exists (initialized to `undefined`) from the top of the function, which enables buggy access before the assignment line. `let` and `const` are block-scoped and also hoisted, but sit in the temporal dead zone until their declaration executes, so accessing them early throws a ReferenceError. `const` additionally prevents reassignment of the binding — but it does not make the referenced object immutable, so `const obj = {}` still allows `obj.x = 1`.
Q4. What are reference types vs primitives, and what breaks when engineers forget the difference?
Primitives (number, string, boolean, null, undefined, symbol, bigint) are copied by value and are immutable. Objects, arrays, and functions are reference types: the variable holds a pointer, so assignment or passing to a function copies the reference, and two variables can mutate the same underlying object. Forgetting this causes shared-state bugs (mutating a 'copy' that's actually the original), broken equality checks (`[1] !== [1]` because identity differs), and shallow-copy leaks where nested objects are still shared after a spread.
Q5. How does prototypal inheritance actually work in JS?
Every object has an internal link (`[[Prototype]]`, exposed via `Object.getPrototypeOf` / `__proto__`) to another object. Property lookups that miss on the object walk up this prototype chain until found or the chain ends at `null`. `class` syntax is sugar over this — methods live on `Constructor.prototype`, and instances delegate to it — so there's no classical class copying; it's live delegation, which is why adding a method to a prototype affects all existing instances.
Q6. Why do arrow functions behave differently with `this`, and when is that a problem?
Arrow functions don't bind their own `this`, `arguments`, or `super` — they capture `this` lexically from the enclosing scope at definition time. This is ideal for callbacks (e.g. inside `array.map` or `setTimeout`) where you want the surrounding `this`. It's wrong for object methods that rely on the call-site receiver, or for event handlers where you expect `this` to be the element, and arrow functions also can't be used as constructors.
Q7. What is type coercion and give an example where implicit coercion causes a real bug.
Coercion is JS automatically converting a value from one type to another in an operation. The classic bug is `+` doing string concatenation when one operand is a string: `1 + '1'` yields `'11'`, so summing values pulled from an input field or JSON-as-string produces `'0' + userInput` garbage instead of arithmetic. Another is truthiness — `if (arr.length)` is fine but `if (value)` treats `0`, `''`, and `NaN` as false, which is why `??` (nullish coalescing) exists to fall back only on null/undefined rather than any falsy value.
Q8. Describe the difference between a shallow copy and a deep copy in JS and how to produce each.
A shallow copy (`{...obj}`, `Object.assign`, `array.slice`) duplicates the top level but nested objects remain shared references, so mutating a nested field affects both copies. A deep copy recursively clones nested structures so they're fully independent — `structuredClone(obj)` is the modern built-in, while `JSON.parse(JSON.stringify(obj))` works only for JSON-safe data (it drops functions, undefined, symbols, and mangles Dates). Choosing wrong is a frequent source of state-mutation bugs in reducers and caches.
