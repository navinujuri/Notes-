Docker Swarm

Cohort 2.0 · 5 lessons · SDE-2 (3+ yrs) interview prep

Concept summary — all 5 steps

Intro
Docker Swarm is Docker's native container orchestrator: it turns a pool of Docker Engines into a single logical cluster with built-in service discovery, load balancing, rolling updates, and desired-state reconciliation. Interview angle: know why it exists (simpler than Kubernetes, ships with Docker, uses the same CLI/Compose files) and its trade-off — lower operational overhead but a smaller ecosystem and less feature depth, which is why most production shops standardize on Kubernetes.
Architecture
A swarm has manager nodes (maintain cluster state and schedule work) and worker nodes (run tasks); managers use the Raft consensus algorithm to replicate state, so you run an odd number (3 or 5) to tolerate failures while keeping a quorum. Interview angle: quorum math — N managers tolerate (N-1)/2 failures, so 3 managers survive 1 loss, 5 survive 2; losing quorum freezes scheduling (existing containers keep running) until it's restored. Also mention the built-in overlay network, gossip protocol for node membership, and mutual TLS auto-rotation between nodes.
Services, tasks, containers
This is the core object model: a service is the declarative desired state (image, replica count, ports, update policy); a task is a single scheduled slot the manager assigns to a node; and a container is the running instance that fulfills that task. Interview angle: tasks are immutable and atomic — a failed task isn't restarted in place, the reconciler kills it and schedules a fresh one to converge back to desired replica count. Know the two service modes: replicated (N copies) vs global (one per node, e.g. for log/monitoring agents).
Create a 2 node swarm
`docker swarm init --advertise-addr <IP>` bootstraps the first manager and prints a `docker swarm join --token ...` command; run the worker token on the second node to join it. Interview angle: understand join tokens (separate manager vs worker tokens, rotatable via `docker swarm join-token --rotate`), the advertise/listen address distinction for multi-NIC hosts, and required ports — 2377/tcp (cluster management), 7946/tcp+udp (node gossip), 4789/udp (VXLAN overlay data plane).
Deploying a service
`docker service create --name web --replicas 3 -p 8080:80 nginx` schedules tasks across nodes; the routing mesh then makes the published port reachable on every node's IP regardless of where the replicas actually run. Interview angle: explain the ingress routing mesh vs host-mode publishing, how `docker service scale`/`docker service update --image` trigger rolling updates (with `--update-parallelism`, `--update-delay`, and rollback on failure), and prefer `docker stack deploy -c docker-compose.yml` for multi-service declarative deploys.

Real-world example

Rolling out a zero-downtime API update across a small production cluster

You run a payments API on a 5-node swarm (3 managers spread across availability zones, 2 dedicated workers) defined in a Compose stack deployed via `docker stack deploy`. The `api` service runs 6 replicas published on port 443 through the ingress routing mesh, sitting behind an external load balancer that health-checks every node IP. To ship v2 you run `docker service update --image registry/api:v2 --update-parallelism 2 --update-delay 15s --update-failure-action rollback api`: the reconciler drains and replaces two tasks at a time, waiting for each batch to pass its healthcheck before proceeding, and auto-reverts if a batch fails. Mid-rollout one manager's host dies — because you kept an odd number of managers you still have quorum (2 of 3), so scheduling continues uninterrupted and the lost node's tasks are rescheduled onto healthy nodes. When you later debug an intermittent 502, the usual suspects are a failing container-level HEALTHCHECK marking tasks unhealthy, or a blocked overlay port (4789/udp) breaking the routing mesh between nodes.

Interview Q&A (SDE-2)

Q1. How does Swarm decide how many manager nodes you should run, and what happens when you lose quorum?
Managers replicate cluster state via Raft, which requires a majority (quorum) to commit changes, so you run an odd number to maximize fault tolerance per node — 3 managers tolerate 1 failure, 5 tolerate 2. If you lose quorum (e.g. 2 of 3 managers down), the swarm can no longer schedule, update, or reconcile anything and management API calls fail, but critically the already-running containers on worker nodes keep serving traffic. Recovery means bringing managers back or force-creating a new cluster with `docker swarm init --force-new-cluster` from a surviving manager.
Q2. Explain the difference between a service, a task, and a container in Swarm.
A service is the declarative spec of desired state — which image to run, how many replicas, ports, and update/restart policy. The manager decomposes that into tasks, each of which is a single scheduling unit bound to one node and backed by exactly one container. Tasks are immutable and have a lifecycle that ends terminally; if a container dies, Swarm doesn't restart that task in place — it removes the failed task and creates a new one to reconcile back to the desired replica count.
Q3. What is the routing mesh and how does ingress publishing differ from host-mode?
The routing mesh means a service's published port is exposed on every node in the swarm; a request hitting any node's IP on that port is transparently routed (via IPVS load balancing over the ingress overlay network) to a healthy task, even if that node runs no replica of the service. This is convenient for putting an external LB in front of all nodes, but adds a hop. Host-mode publishing (`--publish mode=host`) instead binds the port only on nodes actually running a task, skipping the mesh — useful for high-throughput or when you want the client IP preserved, at the cost of losing the every-node reachability guarantee.
Q4. Walk through how a rolling update works and how you'd make it safe.
`docker service update` replaces tasks in batches controlled by `--update-parallelism` (how many at once) and `--update-delay` (pause between batches). Combined with a container `HEALTHCHECK`, Swarm waits for new tasks to become healthy before continuing, and `--update-failure-action rollback` (plus `--update-max-failure-ratio`) auto-reverts if too many fail. To make it safe you keep parallelism low, set a realistic delay and healthcheck, run enough replicas that draining a batch doesn't overload the rest, and optionally use `--update-order start-first` so a new task is up before the old one is torn down.
Q5. Which ports must be open between swarm nodes and what does each carry?
Three: 2377/tcp for cluster management (the manager control plane / API), 7946 on both tcp and udp for node-to-node membership and gossip, and 4789/udp for the VXLAN data plane that overlay networks (including the ingress mesh) tunnel over. A very common production bug is that 2377 is open but 4789/udp or 7946 is blocked by a firewall or security group — nodes join fine but cross-node overlay traffic and the routing mesh silently break, producing intermittent connection failures.
Q6. Replicated vs global services — when do you use each?
Replicated (the default) runs a fixed number of tasks that the scheduler spreads across eligible nodes; you use it for stateless app tiers where you scale by count. Global runs exactly one task on every node (present and future) that matches the constraints, which is the right model for per-host agents like log shippers, monitoring/metrics collectors, or a node-level proxy. A key property of global mode is that adding a new node automatically gets a task without you re-scaling.
Q7. How does Swarm compare to Kubernetes and when would you still choose it?
Swarm is built into Docker, uses the same CLI and Compose files, has a gentle learning curve, and gives you TLS, service discovery, load balancing, and rolling updates out of the box with almost no setup. Kubernetes is far more powerful and extensible (CRDs, operators, autoscaling, a huge ecosystem) but much heavier to operate. You'd reasonably pick Swarm for small teams, edge/on-prem deployments, or internal tooling where operational simplicity beats feature depth — though be aware the industry and hiring has consolidated heavily around Kubernetes, and Swarm's future investment is uncertain.
Q8. How does Swarm handle secrets and configuration, and why is that better than env vars or bind mounts?
Swarm has first-class `docker secret` and `docker config` objects: secrets are stored encrypted in the Raft log, sent to only the nodes running tasks that need them over mutual TLS, and mounted into the container as an in-memory tmpfs file at `/run/secrets/<name>` rather than exposed as an environment variable. That's safer than env vars (which leak via `docker inspect`, child processes, and crash dumps) and safer than bind-mounting files (which requires the secret to sit in plaintext on every host's disk). Secrets and configs are also immutable, so rotating one means creating a new object and updating the service to reference it.
