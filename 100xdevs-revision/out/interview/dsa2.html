Concept summary — all 13 steps

Class roadmap
Overview of the four array-preprocessing patterns covered: prefix sums, frequency arrays, difference arrays (prefix + freq for ranges), and sliding window. Matters because the majority of 'count subarrays/substrings' interview problems collapse into one of these four, so recognizing which one applies is the real skill.
Prefix sums
Precompute P[i] = sum of the first i elements so any range sum [l,r] = P[r+1] - P[l] in O(1) after an O(n) build. This is the foundation for O(1) range queries and, combined with a hashmap, for counting subarrays that satisfy a sum condition.
Running sum (LC 1732 Highest Altitude)
Given per-leg altitude gains, the highest point is just the max over the running prefix sum starting at 0. Trivial application, but the interview signal is instantly reframing 'cumulative/highest-so-far' as a prefix scan rather than a nested loop.
Frequency / count arrays
When the value domain is small and bounded (26 letters, 128 ASCII, 0..K-1), index an array by value to count occurrences in O(1) with no hashing overhead and better cache behavior than a hashmap. It is effectively a perfect hash for dense small-integer keys.
Most frequent character
Build a size-26 (or 128) count array in one pass, then argmax. The nuance interviewers probe is the tie-break rule (first-seen vs lexicographically smallest) and whether input is guaranteed lowercase ASCII vs arbitrary Unicode, which would break the fixed-size array assumption.
Difference array (range updates)
To add v across [l,r], do diff[l]+=v and diff[r+1]-=v, then a single prefix sum materializes the final array — turning m range updates from O(m·n) into O(m+n). This is the write-heavy dual of prefix sums and the basis of line-sweep counting.
Maximum platforms needed
Minimum platforms = maximum number of trains present at any instant. Either +1 at each arrival / -1 at each departure+1 on a time-indexed difference array then prefix-sum for the running max, or sort arrivals and departures and two-pointer merge (O(n log n)), which wins when timestamps are large/sparse. Same shape as Meeting Rooms II.
Prefix + hashmap counting pattern
The meta-technique: the number of subarrays meeting a sum condition equals the number of earlier prefix-states that pair with the current one. Maintain a map of prefix-state -> count and, at each index, add the count of the matching state before recording the current — the engine behind divisible-by-K, sum=k, and equal-0/1 problems.
Subarray sums divisible by K (LC 974)
A subarray sum is divisible by K iff two prefix sums share the same remainder mod K, so count prefixes by remainder and add C(count,2) pairs. Initialize remainder 0 with count 1 for the empty prefix, and normalize negatives with ((sum % K) + K) % K to avoid language-specific negative modulo. O(n) time, O(K) space.
Sliding window (two pointers)
Maintain a window [left,right] with an invariant, expanding right and shrinking left, for overall O(n). Applies only when contribution is monotone (typically non-negative), so growing the window can only push a metric one way — that monotonicity is exactly what lets you move pointers greedily instead of re-scanning.
Fixed-size window (LC 2269 K-Beauty)
Slide a length-k window across the digit string of the number and count windows whose value is non-zero and divides the original number. Classic fixed-window: maintain the current window value incrementally rather than reparsing k digits each step.
Variable window + contribution counting (LC 1358, all 3 chars)
For each right endpoint, track the last-seen index of a, b, and c; every substring ending at right that starts at or before min(last_a,last_b,last_c) contains all three, so add min(...)+1. This counts valid substrings ending at each position in O(1), giving O(n) instead of O(n^2) enumeration.
Exactly-K window (LC 1248 Nice Subarrays)
Map odd->1, even->0, reducing 'exactly k odds' to 'subarrays with sum exactly k'. Solve as atMost(k) - atMost(k-1) with a monotone sliding window, or with the prefix-count hashmap. The atMost trick generalizes to any 'exactly k' counting problem where 'at most k' is a clean window.

Real-world example

Capacity planning and abuse detection for an API gateway

You own an API gateway and build an analytics-plus-guardrails pipeline over per-second traffic. Prefix sums answer 'how many requests hit /checkout between 09:00 and 09:15' in O(1) after one pass over the second-buckets. A difference array over each connection's [start,end] interval yields the peak concurrent long-lived connections per second — the exact 'maximum platforms' computation that sizes your worker pool. A frequency array over HTTP status codes (bounded 100-599) tallies the error mix cheaply for a dashboard. Sliding windows enforce per-client rate limits ('at most k requests in any rolling 60s') and flag bursts. When you need to count how many 5-minute buckets had a volume divisible by an autoscaling step, or had exactly k spike-seconds, you reach for the prefix-sum-plus-hashmap counting pattern. The two bugs that actually bite in review: taking a modulo of a negative delta without normalizing, and an off-by-one at the difference array's r+1 decrement boundary.

Interview Q&A (SDE-2)

Q1. How does prefix-sum plus hashmap count subarrays with sum divisible by K, and how do you handle negative numbers?
A subarray (l,r] has sum divisible by K iff prefix[r] and prefix[l] have the same remainder mod K, so group prefixes by remainder and every pair within a group is a valid subarray. Iterate once, and at each index add count[rem] to the answer before incrementing count[rem]; seed remainder 0 with count 1 for the empty prefix. For negatives, normalize with ((sum % K) + K) % K because in most languages % of a negative is negative, which would split one true remainder class into two buckets.
Q2. When can you NOT use a sliding window for 'subarray sum equals k', and what do you use instead?
When the array contains negative numbers. Sliding window relies on monotonicity — extending right only increases the sum and shrinking left only decreases it — so you can move pointers greedily. Negatives destroy that invariant, so you fall back to prefix sum plus a hashmap of prefix values, which is O(n) and sign-agnostic. Sliding window is valid only when the window metric changes monotonically with window size.
Q3. Explain the difference array technique and how it solves the maximum-platforms problem.
To apply +v over [l,r], set diff[l]+=v and diff[r+1]-=v; a single prefix sum over diff reconstructs the final array, turning m range updates from O(m·n) into O(m+n). For platforms, each train occupies [arrival,departure]: do +1 at arrival and -1 at departure+1 on a time-indexed diff array, prefix-sum to get the concurrent-train count at each instant, and take the max. If timestamps are large or sparse, instead sort arrivals and departures and two-pointer merge, counting overlaps in O(n log n).
Q4. How do you count subarrays with 'exactly k' of something using sliding windows?
Direct 'exactly k' is awkward, but 'at most k' is a clean monotone window, so exactly(k) = atMost(k) - atMost(k-1). For Nice Subarrays, map odd->1 and even->0; atMost(k) counts subarrays with at most k ones by expanding right and shrinking left when the count exceeds k, adding (right-left+1) each step. Subtracting atMost(k-1) leaves exactly k, all in O(n) with O(1) extra space, and the trick generalizes to any countable window property.
Q5. When do you choose a fixed-size frequency array over a hashmap?
Use a fixed array when keys are small, dense, and bounded — 26 lowercase letters, 128 ASCII, digits, or remainders 0..K-1 — because you get O(1) access with no hashing constant factor, contiguous cache-friendly memory, and naturally ordered iteration. Use a hashmap when the domain is large, sparse, or unbounded (arbitrary 32/64-bit ints, strings) to avoid allocating a huge mostly-empty array. The array is essentially a perfect hash whenever the key is a dense small integer.
Q6. Walk through the contribution-counting trick in 'number of substrings containing all three characters' (LC 1358).
Fix the right endpoint and maintain the most recent index at which each of a, b, c appeared. Any substring ending at right whose start is at or before min(last_a,last_b,last_c) must contain all three characters, so exactly min(last_a,last_b,last_c)+1 substrings ending here are valid, and you add that to the answer. This counts valid substrings ending at each position in O(1), giving O(n) overall instead of the O(n^2) of enumerating every substring.
Q7. When do prefix sums stop being the right tool, and what replaces them?
Static prefix sums give O(1) range queries but O(n) point updates, since changing one element forces rebuilding the whole suffix. If the array is mutated between queries, switch to a Fenwick/BIT or segment tree for O(log n) update and query. Prefix sums are the read-heavy, immutable-data choice; the difference array is the write-heavy dual — batch many range updates cheaply, then materialize with one final prefix pass.
