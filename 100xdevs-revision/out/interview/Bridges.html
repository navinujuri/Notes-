Bridges

Cohort 3.0 · 9 lessons · SDE-2 (3+ yrs) interview prep

Concept summary — all 9 steps

Web3 Cohort / Context
Sets the EVM + Solidity foundation: accounts, gas, transactions, and tooling (Hardhat or Foundry, ethers/viem, testnets). Interview angle: be able to explain that a 'bridge' is not magic — chains are isolated state machines that cannot read each other, so any cross-chain movement is really a lock/burn on one chain plus an off-chain actor triggering a mint/unlock on the other.
Lock (vault) contract for ERC-20
A source-chain vault that pulls the user's ERC-20 into custody via approve + transferFrom and emits a Deposit/Locked event carrying (sender, amount, destination address, nonce). It matters because the locked balance is the collateral backing whatever gets minted on the other side — the vault is the single source of truth and the prime hack target.
Deploying it
Deploy the vault to a testnet (e.g. Sepolia) using a deploy script, a funded key in .env, and Etherscan verification so the ABI/source is public. Interview angle: never commit private keys, understand nonce/gas at deploy time, and know that a deployed address is immutable — upgradeability requires a proxy pattern chosen up front.
What a bridge is
A bridge moves value across chains using lock-and-mint (lock native token on source, mint a wrapped/pegged version on destination) or burn-and-mint. The core insight: total supply is conserved because minted destination tokens are always 1:1 backed by locked source tokens — if that invariant ever breaks (bad mint, double-spend), the peg collapses.
Contract + system structure
Architecture is three parts: a source lock contract, a destination mint contract (an ERC-20 with a restricted mint), and an off-chain relayer/validator that watches source events and submits mint transactions. The relayer is the trust boundary — this is a trusted/centralized bridge, and the design question is who is authorized to call mint.
One-way bridge
A deliberate simplification: value flows only source→destination (lock→mint), with no burn-and-unlock path back. It matters because it halves the attack surface and state you must track, and it forces you to reason clearly about the single direction before adding the harder reverse leg (which needs its own events, nonces, and unlock authorization).
Code / implementation
Solidity for both sides: the vault does transferFrom + emits an event with a monotonically increasing nonce; the destination token exposes mint(to, amount, nonce) guarded by onlyRelayer/onlyOwner access control. Interview angle: use SafeERC20 for non-standard tokens, follow checks-effects-interactions to avoid reentrancy, and emit events for every state change since the relayer relies on them.
Nonces / replay protection
Each deposit gets a unique nonce, and the destination stores mapping(uint256 => bool) processed to reject any nonce it has already minted for. Without this, a relayer (or attacker replaying a signed message) could submit the same mint twice and print unbacked tokens — nonces enforce exactly-once processing across an inherently at-least-once messaging layer.
Testing the one-way bridge
With Hardhat/Foundry, deploy a mock ERC-20 + both contracts, then simulate the full flow in one test env: approve, lock, assert the Deposit event and vault balance, then call mint as the relayer and assert the destination balance. Critically test the failure paths: replaying a used nonce reverts, and a non-relayer calling mint reverts.

Real-world example

Bridging a game token from Ethereum to an L2

You run a game whose reward token GEM lives as an ERC-20 on Ethereum, but gameplay happens on a cheaper L2. You build a one-way lock-and-mint bridge: a Vault contract on L1 where a player calls approve(vault, amount) then deposit(amount, l2Address), which runs transferFrom to custody the GEM and emits Locked(player, amount, l2Address, nonce). An off-chain relayer subscribes to that event, waits for enough confirmations to survive a reorg, and calls mint(l2Address, amount, nonce) on the wGEM contract on the L2. wGEM's mint is onlyRelayer and checks a processed[nonce] mapping so a crashed-and-restarted relayer that re-submits the same event cannot double-mint. The invariant you monitor in prod: total wGEM minted on L2 must always equal GEM locked in the L1 vault — a divergence means a bug or a compromised relayer key, and you'd pause minting immediately.

Interview Q&A (SDE-2)

Q1. Why can't a smart contract on Ethereum just read state from another chain to bridge tokens?
Each chain is an isolated, deterministic state machine; a node validating Ethereum has no access to another chain's state, and allowing external network calls would break determinism and consensus. So cross-chain communication is always mediated by an off-chain actor (a relayer/oracle/validator set) that observes an event or proof on the source chain and submits a corresponding transaction on the destination. That off-chain component is why bridges introduce a trust assumption that pure on-chain logic doesn't have.
Q2. Walk me through lock-and-mint and why supply stays conserved.
The user locks N tokens in a source-chain vault, and a relayer mints N wrapped tokens on the destination chain to the user's address. Total economic supply is unchanged because every minted wrapped token is 1:1 collateralized by a token frozen in the vault; the wrapped token is a claim on that locked collateral. The whole system's safety rests on this invariant — if you ever mint without a matching lock, the wrapped token becomes under-collateralized and its peg breaks.
Q3. Why approve + transferFrom instead of having users just transfer tokens to the vault?
A plain transfer to the contract address moves tokens but the contract gets no execution hook and no reliable record tying the deposit to a user intent, so it can't emit the structured Deposit event the relayer needs (and funds can be stranded). The approve/transferFrom pattern lets the vault pull a specific amount inside its own deposit() call, so it can atomically record the sender, destination address, and nonce and emit the event in the same transaction. The trade-off is the two-transaction UX (approve then deposit), which permit/EIP-2612 can collapse into one.
Q4. What exactly do the nonces protect against, and how?
They enforce exactly-once minting over a messaging layer that is only at-least-once — relayers can crash and re-send, and signed mint messages can be replayed by anyone who sees them. Each deposit carries a unique, monotonically increasing nonce, and the destination keeps a mapping(uint256 => bool) processed that it sets on first mint and checks on every mint, reverting on a repeat. Without it, a single legitimate deposit could be minted multiple times, creating unbacked supply — a classic double-mint bridge exploit.
Q5. This is a trusted bridge. Who can mint, and what are the main attack vectors?
Minting is gated by access control (onlyRelayer/onlyOwner), so whoever holds that key can mint arbitrarily — that key is the crown jewel. The dominant vectors are: compromise of the relayer/owner key (Ronin, ~$625M, was stolen validator keys), a bug in mint validation or signature verification (Wormhole, ~$325M), and missing replay/nonce checks. This is why bridges are the single largest category of DeFi losses, and why production designs move toward multisig/MSC validator sets, rate limits, and a pausable circuit breaker.
Q6. The relayer sees a Deposit event and mints immediately — what could go wrong?
Block reorganizations: if the relayer mints on the strength of an event that later gets reorged out, tokens are minted on the destination with no backing lock remaining on the source. The fix is to wait for sufficient finality/confirmations (or hard finality on chains that offer it) before minting, trading latency for safety. You also handle the relayer being at-least-once via nonces so a re-observed-and-re-sent event can't double-mint.
Q7. How do you test a cross-chain bridge when your test harness is a single EVM?
You deploy all three contracts (mock ERC-20, vault, wrapped token) into one in-memory chain and simulate the message passing manually: run the lock, read the emitted event's args, then call mint with those exact args from an account impersonating the relayer. Assert on both sides — vault balance up and Locked event correct after lock, destination balance up after mint. The high-value tests are the negatives: a non-relayer mint reverts, a replayed nonce reverts, and depositing without approval reverts.
Q8. What would it take to make this a two-way bridge, and what new risks appear?
You add a reverse leg: burn the wrapped token on the destination (emitting a Burn event with its own nonce space), have a relayer observe it, and call unlock() on the source vault to release the original tokens. New risks include a second replay surface (burn-side nonces), the unlock authorization becoming a mint-equivalent privilege over real collateral, and the need to keep the accounting invariant (locked == minted) consistent under concurrent flows in both directions. Most real exploits target the unlock/withdraw path precisely because it releases genuine assets.
