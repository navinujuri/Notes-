Turborepo and Monorepos

Cohort 2.0 · 18 lessons · SDE-2 (3+ yrs) interview prep

Concept summary — all 18 steps

What is a monorepo
A single version-controlled repo that holds multiple projects (apps + libraries) rather than one repo per project (polyrepo). Interview angle: a monorepo is a repository topology, not a deployment topology — it has nothing to do with being a monolith; you can ship many independently deployed services from one monorepo.
Why monorepos
They give you atomic cross-package commits, a single source of truth for dependency versions, trivial code sharing, and unified tooling/CI, which makes wide refactors (rename an API across 20 consumers in one PR) safe. The trade-offs an SDE-2 must name: slower/heavier CI, VCS scaling pain at large size, weaker access control granularity, and the need for a task orchestrator to keep builds fast.
Node.js monorepo tooling
Two distinct layers: package-manager workspaces (npm/yarn/pnpm workspaces, or pnpm-workspace.yaml) that link internal packages and hoist deps, and task orchestrators/build tools (Turborepo, Nx, Lerna, Rush, Bazel) that schedule and cache tasks across the graph. Interview angle: workspaces solve linking; they do NOT solve build ordering or caching — that's why you add Turborepo on top.
History of Turborepo
Created by Jared Palmer, acquired by Vercel in late 2021; originally written in Go and later rewritten in Rust. Its design goal was Bazel-class incremental caching with near-zero configuration, positioned as far simpler to adopt than Bazel or (at the time) Nx.
Build system vs orchestrator
A build system actually compiles/bundles code (tsc, esbuild, webpack, vite, next build). An orchestrator does not compile anything itself — it reads the task dependency graph, decides run order and parallelism, and caches results. Getting this distinction right is the crux of most Turborepo interview questions.
Turborepo as an orchestrator
Turbo wraps your existing package.json scripts; it builds a task graph from turbo.json, runs independent tasks in parallel, enforces topological order for dependent ones, and caches each task's outputs. It is a thin coordination + caching layer, which is why adopting it rarely requires changing your actual build commands.
Initializing a Turborepo
`npx create-turbo@latest` scaffolds a workspace with apps/ and packages/, a root package.json declaring workspaces, a turbo.json, and shared config packages. Convention-over-configuration means you get a working incremental pipeline immediately.
Folder structure
By convention apps/ holds deployable units (web, docs) and packages/ holds shared internal libraries (ui, eslint-config, typescript-config). This split communicates intent — apps consume packages, packages don't consume apps — and keeps the dependency graph acyclic.
Running the project
`turbo run dev` (or `turbo dev`) fans out to each workspace's matching script in parallel; `turbo run build` respects graph order. Know `--filter=<pkg>` to scope to one package and its deps, and that dev servers run as long-lived persistent tasks that stream interleaved output.
Root package.json
Declares `workspaces` globs (or defers to pnpm-workspace.yaml), a `packageManager` field that pins the tool via Corepack, shared dev tooling, and root scripts that delegate to turbo. Pinning packageManager is what makes builds reproducible across dev machines and CI.
packages/ui (shared library)
An internal library consumed via the workspace protocol ("@repo/ui": "workspace:*"), never published to npm. Interview nuance: it can be a 'just-in-time' package (raw TS transpiled by the consuming app) or a compiled package (pre-built with its own build step and outputs) — the choice affects caching and build ordering.
apps/web (consumer app)
A Next.js app that imports from @repo/ui, wired through workspace linking and (for JIT packages) Next's transpilePackages. It demonstrates how an app resolves an internal dependency to local source instead of a registry download.
Adding a new page
A hands-on exercise: add a route that renders a component from the shared UI package, proving the end-to-end sharing loop. The takeaway is that editing packages/ui is instantly reflected in the app with no publish step.
turbo.json pipeline
The heart of the setup: the `tasks` map (formerly `pipeline`) defines each task's `dependsOn` (`^build` means run upstream dependencies' build first), `outputs` (what to cache), `inputs`, `env`/`globalEnv`, `cache`, and `persistent` for servers. Misconfiguring outputs or env here is the most common source of stale/incorrect cache bugs.
Adding React projects
Add a second app or package and wire it to the shared eslint/tsconfig and ui packages so tooling and components are reused rather than copied. This shows the marginal cost of a new project in a well-structured monorepo is small.
Caching
Turbo hashes each task's inputs — source files, dependency graph, relevant env vars, and config — into a cache key, then stores outputs plus captured logs. On a hit it restores outputs and replays logs instead of re-running ("never do the same work twice"). Local cache lives under node_modules/.cache/turbo; remote cache (Vercel or self-hosted) shares hits across teammates and CI, which is the biggest real-world CI speedup.
Adding a Node.js app
Add a backend (e.g., Express) app so frontend and backend live in one graph and can share code. This is where monorepos pay off: shared types and validation between client and server in a single atomic PR.
Adding a common module
Create a shared package (types, utils, validation schemas) consumed by both the web app and the Node app, demonstrating cross-stack reuse. The payoff is a single definition — change the type once and every consumer is type-checked against it in the same commit.

Real-world example

Full-stack product monorepo with shared types and fast CI

You own a product with a Next.js web app (apps/web), an Express API (apps/api), and shared packages: @repo/types (Zod schemas + inferred TS types), @repo/ui (React components), and @repo/config (eslint/tsconfig). All are linked via pnpm workspaces with "workspace:*". You define API request/response shapes once in @repo/types; both the API handlers and the web fetch layer import them, so a breaking field change surfaces as a type error in the same PR across client and server. turbo.json declares build.dependsOn = ["^build"] so @repo/types builds before apps that consume it, marks dev as persistent + uncached, and lists .next/ and dist/ as outputs. Locally you run `turbo dev --filter=web...` to boot only the web app and its deps. The real win is CI: with Vercel Remote Caching enabled, a PR that only touches apps/web restores cached build/lint/test outputs for @repo/types, @repo/ui, and apps/api instead of rebuilding them, cutting a ~9-minute pipeline to under 2. The classic bug you debug: the web build reads NEXT_PUBLIC_API_URL but you forgot to list it under the build task's env, so Turbo served a stale cached bundle pointing at the old URL across environments — fixed by adding it to the task's env/globalEnv so the cache key busts when it changes.

Interview Q&A (SDE-2)

Q1. How does Turborepo's caching work, and what determines a cache hit?
For each task Turbo computes a hash over its inputs: the package's source files, its resolved dependency graph, declared env vars, and relevant config. That hash is the cache key; the first run stores the task's outputs (the paths you list under `outputs`) plus its captured stdout/stderr. On a later run with the same key, Turbo restores the outputs and replays the logs instead of executing the task. Remote caching (Vercel or self-hosted) uploads these artifacts so a teammate or CI job can get a hit for work someone else already did.
Q2. What does `dependsOn: ["^build"]` mean, and how is it different from `"build"`?
The `^` prefix means 'run this task in all of the package's topological dependencies first' — so before building app X, Turbo builds every internal package X depends on. Without the caret, a bare `"build"` would refer to another task within the same package (e.g., build depending on the same package's codegen). This is how Turbo enforces correct build order across the workspace graph while still parallelizing independent branches.
Q3. Do you still need workspaces if you use Turborepo, or does Turbo replace them?
You need both; they solve different problems. Package-manager workspaces (pnpm/npm/yarn) handle installing and symlinking internal packages via the workspace protocol so imports resolve to local source. Turborepo sits on top and handles task scheduling, graph ordering, parallelism, and caching. Turbo reads the dependency graph that workspaces establish — it does not link packages or install dependencies itself.
Q4. What's the most common way to get a stale or wrong cache in Turborepo, and how do you prevent it?
Env-var cache poisoning: a task's behavior depends on an environment variable (like an API URL or feature flag) that isn't part of the cache key, so Turbo serves a cached artifact built with a different value. You prevent it by declaring those variables in the task's `env` (or `globalEnv` for repo-wide ones) so changing them busts the hash. The same class of bug happens when `outputs` is incomplete — Turbo caches a 'success' but restores nothing, leaving you with missing build artifacts.
Q5. What is a `persistent` task and why can't other tasks depend on it?
A persistent task is a long-running process that never exits on its own — typically a dev server or watcher. Turbo marks it persistent so it won't try to cache it or wait for it to 'finish,' and it forbids other tasks from listing it in `dependsOn`, because a task that never completes would deadlock the graph. Practically this is why `dev` is persistent and uncached while `build` is cached.
Q6. Turborepo vs Nx — when would you pick each?
Turborepo is minimal-config, wraps your existing npm scripts, and its core value is task orchestration plus local/remote caching, so it's fast to adopt on an existing JS/TS monorepo. Nx is a heavier platform: it offers computation caching too but adds code generators, project graph visualization, module-boundary lint rules, and richer plugin/executor abstractions, at the cost of more buy-in. Choose Turbo when you want caching and ordering with low ceremony; choose Nx when you want an opinionated framework with enforced architecture and generators across a large org.
Q7. When would you advise against a monorepo?
When teams need hard access-control boundaries (a monorepo makes fine-grained per-project permissions awkward), when the repo would grow large enough to strain Git/CI without investment in tooling like sparse checkout and remote caching, or when projects are genuinely unrelated with no shared code or release coordination. Monorepos also concentrate blast radius — a bad root-level change or dependency bump can break everyone at once — so without disciplined CI and ownership (CODEOWNERS), the coordination cost can outweigh the sharing benefits.
Q8. How do you run a task for just one package and its dependencies?
Use `--filter`. `turbo run build --filter=web` scopes to the web package; `--filter=web...` includes its dependencies, and `--filter=...web` includes its dependents. Combined with Turbo's caching and change detection (e.g., `--filter=[origin/main]` to target packages changed since a git ref), this is how you keep local dev and CI from rebuilding the entire graph on every change.
