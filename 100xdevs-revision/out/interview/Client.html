Client side ETH

Cohort 3.0 · 7 lessons · SDE-2 (3+ yrs) interview prep

Concept summary — all 7 steps

Web3 cohort intro / framing
Sets the mental model that a dApp is a normal frontend/backend plus a connection to a blockchain node; the 'client side' half is just your app translating user intent into RPC calls. Interview angle: be able to draw the boundary between off-chain code (your JS) and on-chain code (the deployed contract) — most confusion comes from blurring the two.
Recap of prior concepts
Consolidates the primitives everything downstream depends on: accounts (EOA vs contract), private keys/addresses, transactions vs calls, gas, and that contracts are just bytecode at an address with a defined ABI. Interview angle: an SDE-2 should crisply distinguish an EOA (controlled by a private key) from a contract account (controlled by code) and know that only EOAs can originate transactions.
Ethereum JSON-RPC
All node communication is JSON-RPC 2.0 over HTTP or WebSocket — methods like eth_call, eth_sendRawTransaction, eth_getBalance, eth_estimateGas, eth_getLogs, eth_chainId. Libraries (ethers/viem/web3) are thin ergonomic wrappers over these methods; the node is usually a hosted provider (Alchemy/Infura) or your own geth/reth. Interview angle: know which operations are pure node queries vs which require a signed payload, and that eth_getLogs is how you read past events for indexing.
Connecting to an existing contract from the browser
To talk to a deployed contract you need three things: its address, its ABI (the interface describing function selectors and argument/return types), and a provider or signer. In the browser the provider comes from an injected EIP-1193 object (window.ethereum from MetaMask), which you wrap with ethers/viem to build a typed Contract instance. Interview angle: the ABI is off-chain metadata — the chain stores only bytecode — so a wrong or partial ABI silently breaks encoding/decoding.
Calling read-only (view/pure) functions
view/pure functions are executed via eth_call, which runs the function against a node's local copy of state and returns the ABI-decoded result without creating a transaction — so it is free, needs no wallet signature, and is synchronous from the user's view. You can pin the query to a block tag (latest, a number, or 'pending') to read historical state. Interview angle: eth_call costs no gas because nothing is broadcast or mined; it never mutates state and only needs a provider, not a signer.
Writing to a contract from the browser
State-changing functions require a signed transaction: the app builds the call, the wallet (MetaMask) prompts the user to sign, and it is broadcast via eth_sendRawTransaction, costing gas and returning a tx hash you must then wait on for confirmation. You deal with nonces, EIP-1559 fees (maxFeePerGas/maxPriorityFeePerGas), gas estimation, and the fact that success is asynchronous and can revert. Interview angle: the returned hash means 'accepted into the mempool', not 'succeeded' — you must await the receipt and check status to know it mined and didn't revert.
Interacting from a Node.js server
Server-side there is no injected wallet, so you construct a signer directly from a private key (loaded from a secret manager/env) and connect it to an RPC provider — used for backends, relayers, bots, and indexers. Read paths are identical to the browser; write paths mean you now own key security and nonce management for concurrent transactions. Interview angle: the big new risks are leaking the private key and nonce collisions when firing multiple txs in parallel from one account.

Real-world example

ERC-20 token dashboard with a server-side airdrop distributor

You build a dApp for an ERC-20 token. The React frontend connects to the user's MetaMask via window.ethereum, wraps it in an ethers Signer, and instantiates the token Contract from its address + ABI. Balances and allowances render through read-only eth_call queries (free, no signature). When a user clicks 'Transfer', you call the state-changing transfer(), MetaMask prompts them to sign, you broadcast it, then await the receipt to flip the UI to 'confirmed' or surface a revert. Separately, a Node.js backend runs a scheduled airdrop: it loads a hot-wallet private key from a secret manager, builds a signer against an Alchemy RPC provider, and sends batched transfer transactions — here you must manage the nonce yourself (incrementing per tx rather than trusting 'pending') so parallel sends don't collide, set EIP-1559 fees sanely, and retry stuck transactions. An indexer process on the same backend uses eth_getLogs to read Transfer events and reconcile off-chain balances. Debugging this system means knowing exactly which failures are RPC-layer (rate limits, wrong chainId), encoding-layer (stale ABI), or on-chain (reverts, insufficient gas, nonce gaps).

Interview Q&A (SDE-2)

Q1. What is the difference between eth_call and eth_sendTransaction, and when do you use each?
eth_call executes a function against a node's local state view and returns the decoded result without broadcasting anything — it is free, needs only a provider, and is used for view/pure reads. eth_sendTransaction (really eth_sendRawTransaction with a pre-signed payload) broadcasts a signed transaction into the mempool to change state, costs gas, requires a signer, and only returns a tx hash. You use eth_call for reads and simulations (including dry-running a write to catch reverts cheaply) and eth_sendTransaction when you actually need to mutate on-chain state.
Q2. Why don't read-only calls cost gas, given that gas exists to pay for computation?
Gas pays for computation that every node in the network must redundantly execute and for the state changes they must persist. An eth_call runs only on the single node you queried, against its local state, and its result is discarded — it is never mined into a block or replicated. Since nothing is broadcast or committed, there is no consensus cost to compensate, so the caller pays nothing (the node operator absorbs the trivial CPU cost, which is why hosted providers rate-limit).
Q3. How does a browser dApp get a provider, and what is EIP-1193?
A wallet extension like MetaMask injects an EIP-1193 provider object at window.ethereum — a standardized interface exposing a request({method, params}) function plus events like accountsChanged and chainChanged. Your library (ethers BrowserProvider / viem custom transport) wraps this object so RPC methods and account access route through the wallet, which mediates permissions and signing. Standardizing on EIP-1193 is what lets one frontend work across MetaMask, Rabby, Coinbase Wallet, etc., instead of coding to each wallet's proprietary API.
Q4. What is an ABI and why is it required off-chain when the contract is already deployed?
The ABI (Application Binary Interface) is a JSON description of a contract's functions and events — names, argument types, return types, and mutability. The chain stores only compiled bytecode, which has no self-describing type information, so your client needs the ABI to encode calls (compute the 4-byte function selector and pack arguments) and to decode return values and event logs. A stale or incomplete ABI causes silent encoding/decoding failures or missing functions even though the contract itself is fine.
Q5. A user's write transaction returns a hash immediately, but the token balance hasn't changed. What's happening and how do you handle it?
The hash only means the signed transaction was accepted into the mempool, not that it mined or succeeded. You must await the transaction receipt (e.g., tx.wait()) and check receipt.status — a successful mine is status 1, while a mined-but-reverted tx is status 0 and still consumed gas. Properly you wait for a chosen number of confirmations to guard against reorgs, and you handle the tx being dropped or stuck (underpriced) by re-broadcasting or replacing it with the same nonce and higher fee.
Q6. How does signing differ between the browser and a Node.js server, and what new risks does the server introduce?
In the browser the private key never touches your code — the wallet holds it and the user manually approves each signature via EIP-1193. On a server you instantiate a signer directly from a raw private key you control, so there is no human in the loop and signing is automatic. The new risks are key custody (a leaked key drains the account, so keys belong in a KMS/secret manager or HSM, never in code or logs) and nonce management, since your process is now solely responsible for issuing correct sequential nonces.
Q7. Why do concurrent server-side transactions from one account collide, and how do you fix it?
Every transaction from an EOA carries a nonce that must increment by exactly one; nodes reject duplicates and queue gaps. If you fire several transactions in parallel and each fetches the 'pending' nonce independently, they all read the same value and collide, so only one lands. The fix is to serialize nonce assignment in your own process — maintain a local counter (seeded from getTransactionCount) and hand out incrementing nonces under a lock/queue — plus handle replacement and gap recovery when a tx gets dropped.
Q8. In ethers.js, what is the practical difference between a Provider and a Signer?
A Provider is a read-only connection to the chain: it can query state and call view functions but cannot authorize anything. A Signer wraps a Provider with the ability to produce signatures, so it can send transactions and sign messages on behalf of a specific address. In the browser the Signer is backed by the wallet (user approves); on the server it is backed by a private key you hold. A Contract built with a Provider can only read; one built with a Signer can also write.
