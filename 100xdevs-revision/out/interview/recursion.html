Recursion

Cohort 2.0 · 6 lessons · SDE-2 (3+ yrs) interview prep

Concept summary — all 6 steps

Introduction to Recursion
A function that solves a problem by calling itself on a smaller input, defined by a base case (stops the recursion) and a recursive case (reduces toward the base case). Each call pushes a frame onto the call stack, so understanding the stack is essential — it's where stack-overflow bugs, space cost (O(depth)), and execution order all come from. Interview angle: interviewers check that you always state the base case first and prove the input strictly shrinks.
Nearby Squares (warm-up problem)
An early hands-on problem that forces you to express a concrete computation (e.g. a value derived from a number/series) purely in terms of a smaller subproblem plus a base case. The point is mechanical fluency: mapping an iterative loop into f(n) = combine(f(n-1)), and tracing the unwinding of the stack. This builds the habit of trusting the recursive call rather than mentally simulating every frame.
Divide and Conquer
Split the input into independent subproblems, solve each recursively, then combine the results — merge sort, quicksort, binary search, and closest-pair are the canonical examples. Runtime follows a recurrence you analyze with the Master Theorem (e.g. T(n)=2T(n/2)+O(n) → O(n log n)). Interview angle: be ready to derive the recurrence and to explain why the 'combine' step's cost dominates or doesn't.
How to think recursively
The mental framework: define the problem in terms of a strictly smaller version of itself, assume the recursive call already returns the correct answer for that smaller input (the 'recursive leap of faith'), then only write the combine step and the base case. Correctness reduces to three checks: base case is right, each call makes progress toward it, and the combine logic is right assuming the subcall is. This is what lets you write and reason about recursion without tracing the full call tree.
Backtracking
A refinement of recursion for exploring a decision tree: at each step make a choice, recurse, then undo the choice (restore state) before trying the next option — used for permutations, subsets, N-Queens, Sudoku, and maze/path problems. The key lever is pruning: abandon a branch the moment it can't lead to a valid solution, which is what separates it from naive brute force. Interview angle: articulate the choose/explore/unchoose template and how constraints prune the search space.
Problem: Staircase (climbing stairs)
Count the ways to climb n stairs taking 1 or 2 steps at a time: ways(n) = ways(n-1) + ways(n-2), i.e. Fibonacci. Naive recursion is O(2^n) because subproblems overlap; memoization (top-down) or a bottom-up DP collapses it to O(n) time, and rolling two variables brings space to O(1). This is the standard bridge from recursion to dynamic programming — recognizing overlapping subproblems is the whole lesson.

Real-world example

Recursively computing sizes over a nested file/asset tree

You're building a storage dashboard that shows, for each folder, the total size of everything beneath it. The natural implementation is recursive: size(node) = own size if it's a file (base case), otherwise the sum of size(child) over all children (the divide-and-conquer combine step). Two production issues surface immediately. First, some deployments have very deep directory trees, and a naive recursive crawl blows the call stack — so you rewrite it with an explicit stack (iterative DFS), which is the same recursion with your own heap-allocated stack instead of the language's. Second, the storage graph is actually a DAG because symlinks and shared assets are referenced by multiple folders, so the same subtree gets recomputed repeatedly — the same overlapping-subproblem trap as the Staircase problem, fixed by memoizing size(node) in a map keyed by node id. When a user then asks 'which combination of top-level folders fits under a 10 GB export budget,' that's a backtracking search over subsets with pruning (abandon a branch once the running total exceeds the budget). One feature exercises base-case/recursive-case discipline, divide-and-conquer aggregation, memoization, stack-depth limits, and backtracking.

Interview Q&A (SDE-2)

Q1. When would you choose iteration over recursion, and vice versa?
Prefer recursion when the problem is naturally self-similar — trees, divide-and-conquer, backtracking — because it maps cleanly to the structure and is far more readable. Prefer iteration when depth can be large and unbounded (risking stack overflow) or when the language lacks tail-call optimization, since each recursive frame costs O(depth) stack space. Any recursion can be converted to iteration with an explicit stack; you'd do that specifically to control depth or memory, trading readability for safety.
Q2. What is tail recursion, and can you rely on the runtime optimizing it away?
A call is tail-recursive when the recursive call is the very last operation, so nothing needs to be done with its result — the current frame can be reused instead of pushing a new one. Languages/compilers with tail-call optimization (Scheme, Scala, some C/C++ setups) turn it into a loop with O(1) stack. But you cannot rely on it in Java, Python, or JavaScript engines generally — the JVM and CPython deliberately do not perform TCO — so deep tail recursion there still overflows and you must rewrite it iteratively.
Q3. How do you analyze the time complexity of a recursive function?
Write the recurrence relation from the code: how many recursive calls, on what fraction/size of input, plus the non-recursive work per call. For divide-and-conquer of the form T(n)=aT(n/b)+f(n), apply the Master Theorem by comparing f(n) against n^(log_b a). For tree-shaped recursion, count nodes in the recursion tree times work per node — e.g. naive Fibonacci is O(2^n) because the tree roughly doubles each level, while memoization prunes it to O(n) distinct nodes.
Q4. Explain the difference between backtracking and plain brute force.
Both explore the space of candidate solutions, but backtracking builds solutions incrementally and abandons a partial candidate the instant it violates a constraint (pruning), whereas brute force generates complete candidates and only then checks validity. Pruning can cut the effective search space by orders of magnitude — N-Queens is intractable by brute force but fast with backtracking because most placements are rejected early. The mechanical signature of backtracking is the 'undo' step: after recursing on a choice you restore state so the next choice starts clean.
Q5. Why is naive recursion for the staircase/Fibonacci problem slow, and how do you fix it?
Because subproblems overlap: ways(n-1) and ways(n-2) both recompute ways(n-3) and below, producing an exponential O(2^n) call tree with massive redundancy. Memoization (top-down) caches each result the first time it's computed, so every distinct subproblem is solved once — O(n) time, O(n) space. Bottom-up DP does the same iteratively, and since each state only depends on the previous two, you can roll two variables and drop space to O(1).
Q6. How do you argue that a recursive function is correct?
Three checks. First, the base case returns the right answer directly. Second, every recursive call operates on a strictly smaller input that provably moves toward the base case (guaranteeing termination). Third, assuming the recursive call returns the correct answer for the smaller input — the 'recursive leap of faith' — the combine step produces the correct answer for the current input. This is essentially proof by induction, and it lets you reason locally without tracing the entire call tree.
Q7. What are the space costs of recursion that engineers often overlook?
Even when a recursive algorithm's 'obvious' space looks O(1), the call stack itself consumes O(depth) memory because every unreturned frame is live simultaneously — for unbalanced recursion (like a degenerate quicksort partition or a linked-list-shaped tree) depth can be O(n), risking overflow. This is why balanced divide-and-conquer recurses on the smaller partition first or caps depth, and why deep production traversals often use an explicit heap-allocated stack, which moves the frames off the limited call stack onto the much larger heap.
