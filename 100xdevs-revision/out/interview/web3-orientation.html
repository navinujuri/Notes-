Web3 Cohort by 100xDevs (orientation)
Course kickoff and orientation framing: what Web3/blockchain engineering covers and how the cohort is structured. Non-technical, but sets the mental model that the rest of the track builds a blockchain bottom-up from ledgers, hashing, and consensus rather than starting from a framework.
Cohort 3.0 Exclusive Hackathon
Logistics for a cohort hackathon — a chance to ship an end-to-end dApp/protocol under time pressure. Interview-relevant only in that portfolio projects (a working smart contract, a mini-chain, a wallet integration) are strong signals for Web3 SDE roles.
Why blockchains?
Motivates the core value proposition: a shared, append-only, tamper-evident state that multiple mutually-distrusting parties can agree on without a trusted intermediary. The key interview point is knowing the trade-off — blockchains buy decentralization, censorship resistance, and immutability at a large cost in throughput/latency/complexity, so they're justified only when no trusted central party exists.
How to create a new currency? (building a ledger)
Currency is derived first-principles as a ledger of balances/transactions rather than physical coins, exposing the two hard problems: authorization (proving you own funds) and the double-spend problem (preventing the same funds being spent twice). This frames why you need digital signatures for authorization and a decentralized ordering mechanism (consensus) to prevent double-spends — the setup for everything after.
Intro to hashing
Cryptographic hash functions (e.g. SHA-256): deterministic, fixed-size output, preimage/second-preimage/collision resistant, with an avalanche effect where one bit flips ~half the output. They're the primitive behind block linking (each header embeds the previous block's hash), Merkle trees for efficient integrity proofs, and Proof of Work — expect to be asked how these properties make tampering detectable.
Intro to Proof of Work
PoW is a Sybil-resistant consensus mechanism: miners brute-force a nonce until the block-header hash falls below a difficulty target, which is expensive to produce but trivial to verify. Difficulty auto-retargets to keep block time roughly constant, and the cost of hashing is what makes rewriting history economically infeasible — the classic interview angle is how this replaces identity-based voting with work-based voting.
Intro to Bitcoin
Bitcoin ties the pieces together: a UTXO-based ledger, ECDSA signatures for authorization, hash-linked blocks, PoW mining with the heaviest/longest-chain rule (Nakamoto consensus), and a fixed 21M supply with halvings. Know the UTXO model vs account model, why deeper confirmations mean stronger finality, and what a 51% attack can and cannot do.

Real-world example

Building a tamper-evident audit log with a mini-blockchain

An SDE-2 on a fintech team is asked to build an append-only audit log for a compliance system that regulators must trust even if an internal admin turns malicious. They model each batch of events as a block whose header stores the SHA-256 hash of the previous block plus a Merkle root of that batch, so editing any past record changes its hash and breaks every downstream prev-hash pointer — making tampering instantly detectable. To make silent rewrites expensive rather than instant, they bolt on a lightweight proof-of-work (find a nonce so the block hash has N leading zeros) and sign each block with the service's private key so authorship is provable. This exercise mirrors Bitcoin's core design and forces the real architectural decision: does this need full decentralization and public consensus, or is a signed, hash-chained log periodically anchored to a public chain sufficient — which is exactly the 'why blockchains?' trade-off in production form.

Interview Q&A (SDE-2)

Q1. What properties must a cryptographic hash function have, and why do they matter for a blockchain?
It must be deterministic, produce fixed-size output, be fast to compute, and be preimage-, second-preimage-, and collision-resistant, with an avalanche effect where flipping one input bit changes ~half the output. In a blockchain these guarantee that block linking is tamper-evident (you can't find a different block with the same hash) and that Proof of Work is hard to satisfy but cheap to verify. If collision resistance broke, an attacker could substitute a fraudulent block that hashes to the same value and slip it into the chain undetected.
Q2. What is the double-spend problem and how does Bitcoin solve it?
Digital data is trivially copyable, so a naive coin could be spent twice by broadcasting two conflicting transactions. Bitcoin solves it with a globally-ordered ledger: each transaction consumes specific prior UTXOs, and nodes reject any transaction spending an already-spent output. Ordering is agreed via PoW plus the heaviest-chain rule, so only one of two conflicting transactions can ever be confirmed, and waiting for more confirmations (block depth) makes a reversal exponentially less likely.
Q3. How does Proof of Work provide Sybil resistance and consensus?
PoW ties influence to computational work rather than to identities or IP addresses, so an attacker cannot cheaply create many nodes to sway the network — they would need a majority of global hashrate. Miners vary a nonce until the header hash is below the difficulty target; the network retargets difficulty to hold block time roughly constant as hashrate changes. Because valid blocks cost real electricity and hardware, rewriting confirmed history means out-mining the honest majority, which is economically prohibitive.
Q4. When would you deliberately NOT use a blockchain, and what would you use instead?
If a trusted central party exists, if you don't need censorship resistance, or if you need high throughput and low latency, a replicated database (Postgres, Kafka, a signed audit log) is far cheaper, faster, and easier to evolve. Blockchains trade several orders of magnitude of performance for decentralized trust and immutability. They're justified only when multiple mutually-distrusting parties must share tamper-evident state with no acceptable intermediary; otherwise it's over-engineering.
Q5. How are blocks linked, and why does that make the chain tamper-evident?
Each block header contains the hash of the previous block's header, forming a hash chain (often with a Merkle root summarizing that block's transactions). Altering any historical transaction changes that block's hash, which invalidates the prev-hash pointer of every subsequent block and their PoW. To rewrite one old block, an attacker must redo PoW for it and all blocks after it, faster than the honest network keeps extending the tip — which is why depth equals security.
Q6. Explain the difficulty target and nonce in Proof of Work.
A block is valid only if its header hash, interpreted as a number, is below the current target; a lower target means fewer valid hashes and thus more work. Miners search by incrementing the nonce (and other malleable fields) and rehashing until they find a qualifying value. Difficulty is proportional to max_target/current_target, and Bitcoin retargets every 2016 blocks (~two weeks) to keep the average block time near 10 minutes as total hashrate rises or falls.
Q7. How does Bitcoin authorize spending, and what is the UTXO model?
Funds are locked to a public key (address); to spend, the owner signs the transaction with the corresponding private key using ECDSA on secp256k1, and every node verifies the signature — proving authorization without exposing the key. Bitcoin tracks state as UTXOs (unspent transaction outputs) rather than account balances: a transaction fully consumes input UTXOs and creates new ones, with the difference (minus change) as the fee. This differs from Ethereum's account model and enables straightforward parallel validation and clear ownership provenance.
Q8. What is the longest-chain rule, and what can a 51% attacker actually do?
Nodes treat the chain with the most cumulative Proof of Work (the 'heaviest', loosely 'longest') as canonical, which deterministically resolves forks. An entity with majority hashrate can out-mine the honest network to reorganize recent blocks and double-spend their own coins, and can censor transactions. Crucially they cannot forge signatures to steal others' funds, mint coins out of protocol rules, or cheaply rewrite deeply-buried history, since re-mining all subsequent blocks against the honest chain is infeasible for old blocks.
