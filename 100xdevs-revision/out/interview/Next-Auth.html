Next Auth

Cohort 2.0 · 9 lessons · SDE-2 (3+ yrs) interview prep

Concept summary — all 9 steps

What is NextAuth
NextAuth (now Auth.js) is an authentication library for Next.js that manages the full auth lifecycle: OAuth/OIDC flows, sessions, CSRF protection, and provider callbacks. It matters because it removes the need to hand-roll cookie handling, token rotation, and provider redirect dances — the parts of auth most likely to be built insecurely. Interview angle: know that it's provider-agnostic and can persist sessions either as a stateless JWT or as DB-backed sessions via an adapter.
Why not JWT + localStorage
Storing a JWT in localStorage exposes it to any injected script, so a single XSS turns into full account takeover with no way to revoke the leaked token before expiry. The correct pattern is an httpOnly, Secure, SameSite cookie that JS cannot read, which is exactly what NextAuth sets by default. Interview angle: articulate the XSS-vs-CSRF trade-off — httpOnly cookies neutralize token theft via XSS but need CSRF defenses (SameSite + CSRF token), which NextAuth handles.
NextAuth core setup
The library is configured through an options object (providers, session strategy, callbacks, custom pages) that you export from the auth handler. This central config is where session strategy (`jwt` vs `database`), the `jwt`/`session` callbacks for enriching tokens, and secret signing are wired. Interview angle: the `session` and `jwt` callbacks are the standard place to attach role/userId claims onto the session the client sees.
Catch-all routes
Next.js dynamic routes with `[...slug]` (catch-all) or `[[...slug]]` (optional catch-all) match an arbitrary number of path segments and expose them as an array. This matters because auth needs many sub-endpoints (`/signin`, `/callback/:provider`, `/session`, `/signout`, `/csrf`) served from one file. Interview angle: know the difference between `[param]` (one segment), `[...param]` (one or more), and `[[...param]]` (zero or more).
Mount NextAuth on the catch-all
You register the handler at `pages/api/auth/[...nextauth].js` (or `app/api/auth/[...nextauth]/route.ts` in the App Router) so every `/api/auth/*` request is routed into NextAuth. This single mount point is what lets the library own signin, OAuth callbacks, session reads, and signout without you defining each route. Interview angle: the callback/redirect URLs you register with OAuth providers must point at `/api/auth/callback/<provider>` under this path.
Credentials provider
CredentialsProvider lets you authenticate against your own store (email/password, LDAP, etc.) via an `authorize(credentials)` function that returns a user object or null. It matters when you can't or won't outsource identity to an OAuth provider. Interview angle: using Credentials forces the `jwt` session strategy (DB sessions aren't supported for it), and you are responsible for password hashing (bcrypt/argon2) and rate limiting — NextAuth does not do that for you.
Google provider
GoogleProvider wires up OAuth2/OIDC with Google using a `clientId`/`clientSecret` from the Google Cloud console and an authorized redirect URI of `/api/auth/callback/google`. It matters for frictionless social login and because Google returns verified email/profile claims. Interview angle: be ready to explain the OAuth authorization-code flow, why the client secret must stay server-side, and how `allowDangerousEmailAccountLinking` / account-linking behaves when the same email arrives from two providers.
GitHub provider
GitHubProvider is the same OAuth pattern registered via a GitHub OAuth App, with callback `/api/auth/callback/github`. The reuse here is the point: adding providers is largely declarative config, and NextAuth normalizes each provider's profile into a common user shape. Interview angle: note that GitHub may not return a public email unless you request the `user:email` scope, a common real-world gotcha.
Custom signup page
NextAuth ships default auth pages but you override them via the `pages` option (`signIn`, `error`, etc.) to render your own branded UI, then call `signIn('credentials', {...})` from it. Signup specifically is not a NextAuth concept — you build a separate registration endpoint that creates+hashes the user, then sign them in. Interview angle: distinguish sign-IN (NextAuth's job) from sign-UP (your job); the custom page just replaces the default UI and hands credentials to `signIn`.

Real-world example

Adding social + password login to a B2B SaaS dashboard

You're an SDE-2 shipping auth for an internal analytics dashboard. Marketing users want 'Sign in with Google', engineers prefer GitHub, and a few service accounts need email/password. You mount NextAuth at `app/api/auth/[...nextauth]/route.ts`, register GoogleProvider and GitHubProvider with credentials pulled from env vars (redirect URIs `/api/auth/callback/google` and `/api/auth/callback/github`), and add a CredentialsProvider whose `authorize` looks up the user and verifies an argon2 hash. You switch the session strategy to `jwt`, and in the `jwt`/`session` callbacks you attach `userId` and `role` so route guards can authorize without a DB round-trip. You override the `pages.signIn` option to render your branded login screen, and build a separate `/signup` API route that creates the user (hashing the password) before calling `signIn('credentials')`. During testing you hit the classic bug: GitHub logins arrive with `email: null` until you add the `user:email` scope, and you confirm the session cookie is httpOnly+Secure so a reflected-XSS attempt can't exfiltrate the token.

Interview Q&A (SDE-2)

Q1. Why are httpOnly cookies preferred over localStorage for session tokens, and what new risk do they introduce?
localStorage is readable by any JavaScript on the page, so an XSS injection can steal the token and impersonate the user with no revocation possible. An httpOnly cookie is invisible to JS, closing that exfiltration path, and adding Secure ensures it's only sent over HTTPS. The trade-off is CSRF: because cookies are sent automatically on cross-site requests, you need SameSite=Lax/Strict plus a CSRF token — both of which NextAuth applies by default.
Q2. What is the difference between NextAuth's `jwt` and `database` session strategies, and when do you pick each?
With the `jwt` strategy the session lives entirely in a signed, encrypted cookie — stateless, fast, and scales horizontally with no session store, but you can't easily revoke a session before it expires. The `database` strategy stores sessions server-side via an adapter, giving instant revocation and server-side introspection at the cost of a DB read per request. Choose `jwt` for stateless/edge-friendly apps or when using CredentialsProvider (which requires it); choose `database` when you need immediate logout/revocation or centralized session control.
Q3. Why does NextAuth use a catch-all route, and what would break without it?
Authentication requires many sibling endpoints — signin, per-provider OAuth callbacks, session, signout, csrf — and the catch-all `[...nextauth]` lets a single handler own all of `/api/auth/*`. Without it you'd have to define each endpoint manually and keep them in sync with the library's expectations. The provider redirect URIs (`/api/auth/callback/<provider>`) specifically resolve into this catch-all, so OAuth callbacks would 404 without it.
Q4. What are your responsibilities when using CredentialsProvider that NextAuth does not handle?
NextAuth only calls your `authorize` function and manages the resulting session; it does not store users, hash passwords, or throttle attempts. You must hash with bcrypt/argon2 and compare in constant time, enforce rate limiting and lockout to blunt brute force, and validate input yourself. You're also locked into the `jwt` session strategy, since database sessions aren't supported for credentials.
Q5. How do you get custom data like a user role or ID into the session the client reads?
Use the `jwt` and `session` callbacks. In the `jwt` callback you enrich the token (e.g. `token.role = user.role`) on initial sign-in, and in the `session` callback you copy it onto `session.user` so it's exposed to the client. This avoids extra DB lookups on every request when using the JWT strategy, and keeps authorization claims flowing through to middleware and route guards.
Q6. Walk through the OAuth authorization-code flow that fires when a user clicks 'Sign in with Google'.
NextAuth redirects the browser to Google's authorization endpoint with the client ID, requested scopes, redirect URI, and a state parameter for CSRF protection. The user consents, and Google redirects back to `/api/auth/callback/google` with a short-lived authorization code. NextAuth exchanges that code (plus the client secret, server-side) for access/ID tokens, verifies them, normalizes the profile into a user, and establishes the session. The client secret never touches the browser, which is why the exchange happens on the server.
Q7. A user signs in with Google and later with GitHub using the same email. What happens, and what's the concern?
By default NextAuth treats them as distinct accounts (keyed by provider + provider account id) to avoid silently linking identities. You can enable automatic linking, but only when the provider verifies the email — `allowDangerousEmailAccountLinking` is named that way because linking on an unverified email lets an attacker who controls an OAuth app claim someone else's account. The safe pattern is to link only verified emails, or require the user to confirm the link while already authenticated.
Q8. How is 'sign up' different from 'sign in' in NextAuth, and how do you implement a custom signup page?
NextAuth handles sign-IN — verifying an identity and creating a session — but has no built-in registration concept. You override `pages.signIn` to render your own UI, but for signup you build a separate endpoint that creates the user record and hashes the password, then call `signIn('credentials', ...)` to log them in immediately after. For OAuth providers there's effectively no separate signup: first successful login provisions the account (often via an adapter's `createUser`).
