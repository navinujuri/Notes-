Middlwares in next

Cohort 2.0 · 5 lessons · SDE-2 (3+ yrs) interview prep

Concept summary — all 5 steps

What are middlewares?
A middleware is a function that sits in the request/response pipeline and runs before the request reaches its final handler, so it can inspect, mutate, short-circuit, or pass the request along. It is the canonical place for cross-cutting concerns (auth, logging, rate limiting, header injection) so that logic isn't duplicated across every route. Interview angle: describe it as a composable chain where each layer decides whether to call `next()` or terminate the request early.
Middleware in Next.js
Next.js middleware is a single `middleware.ts` at the project (or `src/`) root that runs on the Edge runtime before a request hits pages, route handlers, or even cached content. Because it executes at the CDN edge close to the user, it is ideal for fast redirects, rewrites, A/B routing, geolocation, and optimistic auth gates. Interview angle: it runs on the Edge runtime, so no native Node APIs (no `fs`, no raw DB drivers) and no heavy CPU/DB work — keep it lightweight.
Basic middleware code
You export a function named `middleware(request: NextRequest)` that returns a `NextResponse` — typically `NextResponse.next()` to continue, `NextResponse.redirect(url)` to send the client elsewhere, or `NextResponse.rewrite(url)` to serve a different path transparently. Interview angle: know the difference — redirect changes the URL the browser sees (3xx), rewrite keeps the URL but serves different content server-side.
Reading state + passing data downstream
Middleware reads cookies and headers via `request.cookies` / `request.headers` (e.g. verify a JWT) and forwards enriched data to the route using `NextResponse.next({ request: { headers } })`, since you cannot directly mutate the downstream handler's arguments. You can also respond immediately with `NextResponse.json(...)` to block a request entirely. Interview angle: the standard pattern for propagating derived values (like a decoded user id) is setting request headers, not global/context state, because each request is isolated at the edge.
Selectively running middleware (matcher)
By default middleware runs on every request, so you scope it with the `config.matcher` array or with early-return conditionals inside the function. Matchers support path globs, negative lookaheads to exclude static assets (`/((?!_next/static|favicon.ico).*)`), and `has`/`missing` conditions on headers/cookies/query. Interview angle: the matcher must be a statically-analyzable constant (evaluated at build time) and scoping it well matters for performance since middleware fires on matching static files too.

Real-world example

Edge auth gate for a multi-tenant dashboard

You own a Next.js app where `/dashboard/*` is private and `/login`, `/`, and marketing pages are public. You add a root `middleware.ts` with `config.matcher = ['/dashboard/:path*']` so it only fires on protected routes. On each matched request it reads the `session` cookie and verifies the JWT signature using a Web Crypto (edge-compatible) library — an optimistic check, not a DB lookup. If the token is missing or invalid it returns `NextResponse.redirect(new URL('/login', request.url))`; if valid it decodes the tenant/user id and forwards it downstream via `NextResponse.next({ request: { headers } })` so server components read it without re-parsing the cookie. Because this runs at the edge, unauthenticated users are bounced before any page render or data fetch, cutting latency and load. The debugging gotcha you hit: full authorization (is this user allowed in THIS tenant?) still needs a DB check in the data layer, because doing it in middleware would mean a slow database round-trip on every request from the edge — middleware only does the cheap, stateless gate.

Interview Q&A (SDE-2)

Q1. Why does Next.js run middleware on the Edge runtime, and what does that constrain?
The Edge runtime executes middleware in a lightweight V8 isolate deployed close to the user (CDN edge), which minimizes latency for redirects, rewrites, and auth gates that must run on every request. The trade-off is a restricted API surface: no native Node.js modules (`fs`, `net`), no raw TCP database drivers, and tight CPU/time budgets. As a result you use Web-standard APIs (fetch, Web Crypto) and keep logic stateless and fast; anything requiring the full Node runtime or heavy I/O belongs in a route handler or server component.
Q2. What is the difference between NextResponse.redirect, rewrite, and next()?
`next()` passes the request through to the normal handler (optionally with modified request headers). `redirect()` returns a 3xx and changes the URL the browser navigates to — the client sees the new URL. `rewrite()` keeps the original URL in the address bar but internally serves content from a different path, which is how you implement transparent proxying, A/B tests, or localized routing without exposing internal paths.
Q3. How do you pass data computed in middleware (like a decoded user id) to a route handler or server component?
You cannot mutate the handler's arguments or share in-memory context, so you set request headers via `NextResponse.next({ request: { headers: newHeaders } })` and read them downstream with `headers()`. Each edge request is isolated, so header propagation is the reliable channel. Avoid trying to stash state in module-level variables — isolates are reused unpredictably and it leaks across requests.
Q4. How does the matcher work and why not just check the path inside the function?
`config.matcher` declares which paths invoke middleware at all, and it must be a static, build-time-analyzable constant (you can't compute it at runtime). Using it means the middleware bundle never even runs for excluded paths, which is more efficient than always invoking the function and returning early — important because without a matcher, middleware fires on every request including `_next/static` assets and images. In practice you combine both: a matcher for coarse scoping plus conditionals for fine-grained logic.
Q5. Can you have multiple middleware files, and how do you compose logic?
No — Next.js supports exactly one middleware entry point (`middleware.ts` at root or `src/`). To run multiple concerns you compose them manually inside that single function, typically as a chain of helper functions each taking the request and returning either a response (short-circuit) or null (continue). Libraries like `next-connect` or hand-rolled composition patterns are used to keep this readable.
Q6. Why is doing full authentication/authorization purely in middleware considered an anti-pattern?
Middleware runs at the edge with no cheap access to your database, so doing full session or permission verification there forces a slow round-trip on every request and can't be trusted as the only gate. The recommended pattern is optimistic checks in middleware — verify a signed JWT's signature/expiry to bounce obviously-unauthenticated users fast — while the authoritative authorization (does this user own this resource?) happens in the data access layer close to the DB. This keeps the edge fast and keeps security enforced where the data actually lives.
Q7. In what order does middleware run relative to caching, routing, and static files?
Middleware runs before routing resolution and before cached responses are served, on every request that matches the matcher — including matched static assets unless you exclude them. This is why a poorly-scoped matcher hurts performance and why middleware can invalidate CDN caching benefits if it forces dynamic behavior. It executes first, decides the fate of the request, and only then does Next.js resolve the route or serve cached content.
Q8. How would you rate-limit or geo-restrict traffic using Next.js middleware, and what are the limits?
You read `request.geo` (country/region, populated on supporting hosts like Vercel) or the client IP and apply logic — e.g. redirect EU users to a localized path or reject requests from a blocked region. For rate limiting you need shared state, which the edge lacks, so you call an external edge-compatible store (Upstash Redis, Vercel KV) over fetch and return `NextResponse.json({error}, {status:429})` when exceeded. The limitation is that any per-request external call adds latency and cost, so you cache aggressively and keep the store geographically close.
