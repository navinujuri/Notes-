React

Cohort 3.0 · 24 lessons · SDE-2 (3+ yrs) interview prep

Concept summary — all 24 steps

Starting a React project locally
Scaffold with Vite (`npm create vite@latest`) rather than CRA — Vite uses esbuild for dev and Rollup for prod, giving near-instant cold start and true HMR. Interview angle: know that JSX is transpiled by the bundler (esbuild/Babel/SWC) into `React.createElement`/`jsx()` calls, and that the dev server is not what ships to prod.
Components
A component is a pure function of props (and state) that returns JSX describing UI; composition replaces inheritance. Interview angle: distinguish a component (the function) from an element (the object it returns) and know rendering should be side-effect-free so React can call it multiple times safely.
useState
`useState` gives a component local state; calling the setter schedules a re-render and state is preserved across renders by call order. Key points: updates are asynchronous and batched, you must treat state as immutable, and use the functional updater `setX(x => x+1)` when the new value depends on the previous to avoid stale reads.
Tracking re-renders
A component re-renders when its own state changes, its parent re-renders, or its context value changes — not simply because a prop 'looks' different. Interview angle: use the React DevTools Profiler / 'highlight updates' to see wasted renders, and understand a re-render recomputes JSX but only commits actual DOM diffs.
useEffect
Runs side effects (data fetching, subscriptions, manual DOM) after the render is committed and painted; the dependency array controls when it re-runs and the returned function is the cleanup. Interview angle: missing deps cause stale closures, empty deps `[]` means run-once, and StrictMode double-invokes effects in dev to surface missing cleanup.
props
Props are the read-only inputs passed parent-to-child, enforcing one-way (top-down) data flow. Interview angle: props are immutable from the child's view; to communicate upward you pass callbacks down, and mutating props directly is a bug.
Conditional Rendering
Render branches with ternaries, `&&`, or early returns; returning `null` renders nothing. Interview angle: the classic footgun is `count && <X/>` rendering a literal `0` when count is 0 — guard with an explicit boolean (`count > 0 && ...`).
children
`children` is the implicit prop holding whatever JSX is nested inside a component's tags, enabling generic wrapper/slot components (cards, modals, layouts). Interview angle: it powers composition and inversion of control, letting you build reusable containers without them knowing their contents.
Lists and Keys
Render collections with `.map()` and give each element a stable, unique `key`. Interview angle: keys let React match elements across renders during reconciliation — using the array index as key corrupts component state and causes subtle bugs when items are inserted, removed, or reordered.
Inline styling
The `style` prop takes a JS object with camelCased properties (`backgroundColor`) rather than a CSS string. Interview angle: inline styles are convenient for dynamic values but can't use pseudo-classes/media queries and create a new object every render; prefer `className`/CSS modules for static styling.
Class vs functional components
Class components use `this.state`/lifecycle methods; functional components use hooks and are now the standard. Interview angle: hooks won because they let you reuse stateful logic without HOC/render-prop wrapper hell, avoid `this`-binding bugs, and colocate related logic instead of scattering it across lifecycle methods.
Lifecycle events
The three phases are mount, update, and unmount; in classes these are `componentDidMount`/`componentDidUpdate`/`componentWillUnmount`. Interview angle: be able to map them to hooks — mount+update is `useEffect(fn)`, mount-only is `useEffect(fn, [])`, and unmount cleanup is the effect's returned function.
Error boundary
A class component implementing `getDerivedStateFromError`/`componentDidCatch` that catches render-phase errors in its subtree and shows fallback UI instead of crashing the whole app. Interview angle: they do NOT catch errors in event handlers, async code, or SSR — those need try/catch; there is still no hook equivalent, so boundaries remain class-based.
Fragment
`<React.Fragment>` / the shorthand `<>...</>` groups siblings without adding an extra DOM wrapper node. Interview angle: it satisfies the single-root-element rule while keeping the DOM (and flex/grid layouts) clean; use the explicit `<Fragment key=...>` form when a fragment is a list item.
Single page applications, routing
An SPA loads one HTML shell and swaps views on the client; React Router intercepts navigation via the History API so there's no full page reload. Interview angle: understand client-side vs server-side routing trade-offs (fast transitions and shared state vs SEO/initial-load cost) and that deep links require server fallback to index.html.
Layouts
Shared chrome (nav, sidebar, footer) is expressed as a layout route whose child renders into an `<Outlet/>`, using nested routes. Interview angle: layout routes avoid re-mounting/re-fetching the shell on every navigation and keep persistent UI mounted across route changes.
Building a website
Integration step: assemble components, routing, layouts, state, and data fetching into a working multi-page app. Interview angle: demonstrates folder structure, component boundaries, and where state should live (colocated vs lifted) in a realistic app.
useRef
Returns a mutable `{ current }` container that persists across renders WITHOUT triggering a re-render; used for DOM node access and for holding mutable values (timers, previous values). Interview angle: contrast with state — writing `ref.current` is a side effect that doesn't re-render, so never store rendered UI data in a ref.
Custom hooks
A function prefixed with `use` that composes built-in hooks to extract and share stateful logic (e.g. `useFetch`, `useDebounce`). Interview angle: they share logic, not state — each call gets its own isolated state — and they must obey the Rules of Hooks (top-level, from React functions only).
Rolling up (lifting) state, unoptimal re-renders
When two components need the same state you lift it to their nearest common ancestor and pass it down — but that ancestor's re-render now cascades to its whole subtree. Interview angle: this is the core tension — centralizing state creates broad re-renders, mitigated by state colocation, `React.memo`, and splitting components.
Prop drilling
Passing a prop through many intermediate components that don't use it, just to reach a deep consumer. Interview angle: it couples the tree and is noisy, which motivates Context (or a state library) — but note drilling one or two levels is often clearer than reaching for context prematurely.
Context API
`createContext` + `Provider` broadcasts a value to any descendant via `useContext`, removing prop drilling for cross-cutting data (theme, auth, locale). Interview angle: the caveat is that EVERY consumer re-renders whenever the context value's identity changes, so memoize the provider value and split contexts by update frequency.
Testing the Context API
Verifying context behavior means asserting that consumers receive the right value and re-render only when expected. Interview angle: demonstrate that an unmemoized provider `value={{...}}` object re-renders all consumers every parent render, and that Context is a dependency-injection/broadcast tool, not an optimized state manager.
Introducing Recoil
Recoil models state as atoms (units of state) and selectors (derived state); components subscribe to specific atoms via `useRecoilState`, so only components reading a changed atom re-render. Interview angle: this granular subscription model solves Context's all-consumers-re-render problem and adds async/derived state — the same problem Redux, Zustand, and Jotai target with different trade-offs.

Real-world example

A multi-tenant analytics dashboard SPA

You're building the front end for a SaaS analytics dashboard. You scaffold with Vite and set up React Router with a persistent layout route (sidebar + top nav) whose pages render through an <Outlet/>. Auth state and the light/dark theme live in Context so any page can read them without prop drilling, and data-fetching per page is done in useEffect with cleanup to cancel stale requests on route change. Report tables render lists keyed by stable record IDs (never array index) and use conditional rendering for loading/empty/error states, wrapped in an Error Boundary so one failing widget doesn't blank the whole dashboard. You extract repeated logic into custom hooks like useFetch and useDebouncedSearch, and use useRef to focus the filter input and hold interval IDs for the live-refresh timer. Trouble starts when you put the frequently-updating 'live metrics' object into the same Context as auth: every ticking update re-renders the entire tree. You first mitigate with React.memo and by splitting the fast-changing context from the slow-changing one, then migrate the metrics to Recoil atoms so only the specific tiles that read a given metric re-render — turning a re-render storm into surgical updates.

Interview Q&A (SDE-2)

Q1. Why does a React component re-render, and how do you eliminate unnecessary re-renders?
A component re-renders when its own state changes, its parent re-renders, or a context it consumes changes value — not because a prop's value 'differs'. To cut waste: colocate state as low as possible, wrap pure children in React.memo, stabilize callback/object props with useCallback/useMemo, and split large contexts. Crucially, memoization only helps if the props' referential identity is actually stable, so fixing the source of new object/function identities matters more than sprinkling memo everywhere.
Q2. Walk through useEffect's dependency array and cleanup, and the common bugs.
The effect runs after commit/paint; the dependency array decides re-runs (`[]` = mount only, omitted = every render) and the returned function runs cleanup before the next run and on unmount. The classic bugs are stale closures from omitting dependencies (the effect captures old state/props) and leaks from missing cleanup on subscriptions/timers. In dev StrictMode, effects mount-unmount-mount to surface missing cleanup, so effects must be idempotent.
Q3. What is the re-render problem with Context, and how does Recoil address it?
Every component calling useContext re-renders whenever the provider's value identity changes, even if it only cares about one field — so a fast-changing value in a broad context re-renders the whole subtree. Mitigations are memoizing the value object and splitting contexts by update cadence. Recoil instead exposes atoms that components subscribe to individually, so changing one atom re-renders only the components reading that atom; Zustand and Jotai solve the same granularity problem differently.
Q4. Why is using the array index as a list key dangerous?
Keys tell React's reconciler which element is which across renders so it can preserve component state and DOM. With index keys, inserting/removing/reordering shifts every index, so React mismatches old state to new items — you get wrong input values, lost focus, or animation glitches. A stable domain ID (e.g. record.id) fixes identity to the data, not the position; index keys are only safe for static, never-reordered lists.
Q5. What do Error Boundaries catch and not catch, and how are they implemented?
They catch errors thrown during rendering, in lifecycle methods, and in constructors of the components below them, showing fallback UI via getDerivedStateFromError and logging via componentDidCatch. They do NOT catch errors in event handlers, async code (setTimeout/promises), server-side rendering, or errors thrown in the boundary itself — those need ordinary try/catch. There's still no hook version, so boundaries remain class components (or you use react-error-boundary).
Q6. When would you use useRef instead of useState?
Use a ref for a mutable value that should persist across renders but must NOT trigger a re-render when it changes — DOM element references, interval/timeout IDs, previous-value tracking, or a mutable flag. Use state when the value affects the rendered output. The rule of thumb: if changing it should update the UI, it's state; if it's out-of-band bookkeeping, it's a ref, and you should never read/write refs during render.
Q7. Lifting state up, prop drilling, and Context all move state around — how do you choose?
Lift state to the nearest common ancestor only when multiple components genuinely need to share it; over-lifting broadens re-renders and couples the tree. Prop drilling is fine for one or two levels and keeps data flow explicit. Reach for Context when a value is truly cross-cutting (theme, auth, locale) and read by many distant components — but if that value updates frequently or you need derived/granular subscriptions, a dedicated state library (Recoil/Zustand/Redux) is the better tool.
