Recoil

Cohort 3.0 · 7 lessons · SDE-2 (3+ yrs) interview prep

Concept summary — all 7 steps

React Context API
Context lets you pass shared state down the tree without prop-drilling via a Provider/Consumer (or useContext). The interview angle: Context is a dependency-injection mechanism, not a state manager — any change to the Provider's value re-renders every consuming component, which becomes the core motivation for Recoil.
Introducing Recoil
Recoil is a state-management library from Meta that models state as a graph of atoms (units of state) and selectors (derived state), consumed via hooks like useRecoilState. It matters because it gives you fine-grained, subscription-based reactivity with a React-idiomatic API and minimal boilerplate compared to Redux.
Recoil vs Context API
With Context, one value object drives all consumers, so an update forces every consumer to re-render regardless of which field changed; Recoil subscribes each component only to the specific atoms/selectors it reads, so unrelated components don't re-render. The trade-off: Context is built-in and zero-dependency (great for low-frequency, truly shared config like theme/auth), while Recoil adds a dependency but scales far better for frequently-changing, granular state.
Atom
An atom is the smallest unit of shared, writable state, defined with a unique key and default, and read/written via useRecoilState (or useRecoilValue / useSetRecoilState). Any component subscribing to an atom re-renders only when that atom changes — the key mechanism behind Recoil's performance story; the unique key also enables persistence and debugging.
React.memo
React.memo memoizes a component so it skips re-rendering when its props are shallow-equal to the previous render. In the Recoil discussion it matters because atom-level subscriptions plus memo let you contain re-renders to exactly the components whose data changed — and it illustrates the difference between preventing re-renders via props (memo) versus via granular subscriptions (atoms).
Alternate approach (setter-only / read-write split)
Instead of useRecoilState, use useSetRecoilState in components that only mutate state and useRecoilValue where you only read it. This avoids subscribing a write-only component to value changes, so a button that sets state never re-renders when the value updates — a common Recoil optimization pattern.
Selector
A selector is derived state: a pure function (get) that computes a value from atoms/other selectors, with optional set to make it writable, and it memoizes and recomputes only when its dependencies change. It matters because it moves computed/aggregated logic out of components into the data graph, supports async data (returning Promises that integrate with Suspense), and keeps derived values consistent and cached.

Real-world example

A dashboard navbar counter that re-renders the whole page

You build an admin dashboard where a top navbar shows a live 'pending notifications' count while a large table, sidebar, and charts render below. Initially you keep the count in a Context Provider wrapping the app. Every time the count polls and updates, the entire subtree under the Provider re-renders — the heavy table and charts flicker and the page janks, because Context has no per-field subscriptions. You migrate the count to a Recoil atom: the navbar badge reads it with useRecoilValue and re-renders on every tick, while the table and charts (which never read that atom) stay untouched. You then add a notificationSummarySelector that derives 'unread vs total' from the atom so the badge label logic lives in the data graph, and the polling component uses useSetRecoilState so it can write the count without itself subscribing to it. React.memo on the table row component guarantees rows don't re-render from unrelated parent updates. The result: updates that once re-rendered the whole dashboard now touch only the small badge.

Interview Q&A (SDE-2)

Q1. Why does the Context API cause unnecessary re-renders, and how does Recoil avoid them?
When a Context Provider's value changes, React re-renders every component that calls useContext on it, because the subscription is to the whole value object with no field-level granularity — memoization on the consumer doesn't help since the context value itself changed. Recoil instead has each component subscribe to specific atoms/selectors it reads, so only components whose exact piece of state changed re-render. That fine-grained subscription model is the primary reason to reach for Recoil over Context for frequently-updating state.
Q2. When would you still use Context over Recoil?
Use Context for low-frequency, broadly-shared values that rarely change — theme, locale, current user/auth, or a DI container for services — where the re-render cost is negligible and you want zero extra dependencies. Context is also fine when the shared value is essentially static after load. Recoil earns its keep when state changes often and is read by many components that should update independently, or when you need derived/async state.
Q3. What is an atom and how do useRecoilState, useRecoilValue, and useSetRecoilState differ?
An atom is a unit of shared writable state identified by a unique key with a default value. useRecoilState returns [value, setter] and subscribes the component to changes; useRecoilValue returns only the value and subscribes; useSetRecoilState returns only the setter and does NOT subscribe. Choosing the read-only or write-only hook prevents unnecessary subscriptions — a write-only button won't re-render when the value changes, which is a standard optimization.
Q4. What is a selector and how is it different from an atom?
A selector is derived state defined by a pure get function that computes a value from atoms or other selectors; an atom is a source of truth you write to directly. Selectors are memoized and only recompute when their tracked dependencies change, and they can be made writable by adding a set function. They keep computed/aggregated logic out of components and guarantee derived values stay consistent with their inputs.
Q5. How does Recoil handle asynchronous data?
A selector's get can return a Promise, and Recoil integrates that with React Suspense — components reading the selector suspend until it resolves and show a fallback, with errors surfaced to error boundaries. Recoil caches the result by dependency values, so it acts as a lightweight async data cache. This lets you model server-fetched, derived data declaratively in the same graph as synchronous state, though for heavy server-cache needs many teams pair it with or prefer React Query.
Q6. Why is every atom and selector required to have a unique key?
The key is a stable string identifier Recoil uses to track the node in its internal state graph, deduplicate definitions, and support features like persistence and time-travel debugging. Duplicate keys throw (or warn), and hot-reload can surface key collisions. It's essentially the node's identity across the app and across serialization boundaries.
Q7. How do atoms plus React.memo work together, and what problem does each solve?
They address re-renders at different layers: React.memo skips a component's render when its incoming props are shallow-equal, protecting it from parent-driven re-renders; atom subscriptions ensure a component only re-renders when the specific state it reads changes. Atoms cut re-renders caused by shared state, memo cuts re-renders caused by prop propagation from parents. Used together you can constrain updates to precisely the components whose data actually changed.
Q8. What are the downsides or risks of adopting Recoil?
Recoil was experimental for a long time and its maintenance/activity has been uncertain, so betting a large codebase on it carries ecosystem risk versus more established options like Redux Toolkit, Zustand, or Jotai (which shares the atomic model). It requires a RecoilRoot wrapper, global unique keys can collide, and its testing/devtools story is thinner than Redux's. For simple apps the granular API can also be overkill compared to Context or a small store.
