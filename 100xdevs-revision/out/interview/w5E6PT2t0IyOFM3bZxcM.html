Frontend Deployments on AWS

Cohort 2.0 · 9 lessons · SDE-2 (3+ yrs) interview prep

Concept summary — all 9 steps

Static hosting model for frontends
A modern SPA (React/Vue) compiles to static HTML/CSS/JS, so it needs no application server — the canonical AWS pattern is S3 (storage) + CloudFront (CDN/TLS) + Route 53 (DNS). Interview angle: articulate why you don't put a React app on EC2/nginx — no runtime, no autoscaling, near-zero cost, global low-latency delivery, and the whole thing is cache-friendly and immutable.
Build the production bundle
`npm run build` (Vite/CRA) runs bundling, minification, tree-shaking and code-splitting to emit a `dist/`/`build/` folder with content-hashed filenames (e.g. `app.4f8a2.js`). The content hash is the key detail: it enables aggressive far-future caching because any code change produces a new filename, so cache invalidation is implicit for assets — only `index.html` needs to change.
CDNs and edge caching
A CDN caches your assets at globally distributed edge PoPs so users are served from a nearby location instead of a single origin region, cutting latency and offloading the origin. Interview angle: understand cache key, TTL, cache hit ratio, and that the CDN also gives you TLS termination, HTTP/2/3, and DDoS absorption at the edge — not just speed.
Create the S3 bucket (object store)
S3 is a flat key-value object store (not a filesystem) — each file is an object with a key, metadata, and content-type. You create a bucket to hold the build output; keep it private and treat it purely as the CloudFront origin rather than exposing it publicly.
Upload the bundle to S3
Use `aws s3 sync ./dist s3://bucket --delete` to push the build; the `--delete` flag prunes stale objects. Set `Content-Type` correctly and differentiated `Cache-Control` headers — long `max-age=31536000, immutable` for hashed assets, `no-cache`/short TTL for `index.html` — because S3 metadata drives how CloudFront and browsers cache each object.
Access via the S3 endpoint (and its limits)
S3 offers a website endpoint (`*.s3-website-<region>.amazonaws.com`) supporting index/error documents and redirects but HTTP-only and no custom TLS, versus the REST endpoint (`*.s3.amazonaws.com`) which supports HTTPS and private access. Interview angle: the website endpoint alone can't give you HTTPS on a custom domain — that's the motivation for putting CloudFront in front.
Put CloudFront in front of S3
CloudFront becomes the public entry point: it fronts the private bucket, terminates TLS with an ACM cert, caches at the edge, and locks down the origin via Origin Access Control (OAC) so the bucket stays private and is only readable by that distribution. Interview angle: know OAC (current, SigV4) vs the deprecated OAI, and that you configure a default root object of `index.html`.
Attach a custom domain
Request an ACM certificate for your domain — it MUST be in us-east-1 for CloudFront regardless of bucket region because CloudFront is a global service — add the domain as an Alternate Domain Name (CNAME) on the distribution, then create a Route 53 alias A/AAAA record pointing at the distribution. Alias records (vs plain CNAME) let you point the zone apex at CloudFront for free.
Error pages and SPA routing
Client-side routers own paths like `/dashboard`, but S3 has no such object, so a hard refresh returns 403 (OAC/REST origin) or 404. Fix it with CloudFront custom error responses mapping 403/404 to `/index.html` with a 200 so the router can take over — but scope this carefully so genuine 404s for missing assets still surface, and use a real error page where you don't want SPA fallback.

Real-world example

Shipping an internal analytics dashboard to production on AWS

You own the React dashboard for an internal analytics tool. You run `vite build`, producing content-hashed assets plus `index.html`. You create a private S3 bucket `dash-prod-assets`, then `aws s3 sync ./dist s3://dash-prod-assets --delete`, tagging hashed JS/CSS with `Cache-Control: max-age=31536000, immutable` and `index.html` with `no-cache`. You put a CloudFront distribution in front with OAC so the bucket is never public, attach an ACM cert issued in us-east-1, and add `dash.company.com` as a Route 53 alias to the distribution. Everything works until a colleague reports that refreshing on `dash.company.com/reports/42` returns an access-denied error — because that key doesn't exist in S3 and OAC returns 403. You add a CloudFront custom error response mapping 403 -> `/index.html` (200), and the SPA router resolves the deep link. On the next deploy you notice users still see the old UI: the hashed assets updated but browsers cached the old `index.html`, so you fix the `Cache-Control` on `index.html` and issue a one-off `create-invalidation --paths '/index.html'` to flush the edge.

Interview Q&A (SDE-2)

Q1. Why deploy a React app to S3 + CloudFront instead of running it on EC2 or a container?
A built SPA is just static files with no server-side runtime, so there's nothing to execute on a box — S3 + CloudFront gives you global edge delivery, automatic scaling, and near-zero cost without patching servers or managing load balancers. You'd only need a server for SSR/SSG-at-request-time (Next.js server rendering), API endpoints, or logic that can't live in the browser. For a pure client-rendered SPA, a server is wasted operational overhead and a worse latency profile.
Q2. You have a private S3 bucket. How do you serve it over HTTPS on a custom domain while keeping the bucket private?
Put CloudFront in front using Origin Access Control (OAC), which signs origin requests with SigV4 so only that distribution can read the bucket, and keep 'Block Public Access' on. Attach an ACM certificate to the distribution for the domain, add the domain as an alternate name, and point a Route 53 alias record at the distribution. The bucket is never publicly reachable; all traffic flows through CloudFront which terminates TLS at the edge.
Q3. Why must the ACM certificate for CloudFront be created in us-east-1, and what breaks if it isn't?
CloudFront is a global service whose control plane lives in us-east-1, so it only reads ACM certs from that region regardless of where your S3 bucket or users are. If you issue the cert in another region, it simply won't appear in the distribution's certificate dropdown and you can't attach it. Certs for regional services (ALB, API Gateway) follow normal per-region rules — this us-east-1 requirement is specific to CloudFront.
Q4. A user refreshes the page on /orders/123 and gets an error, but navigating there in-app works. What's happening and how do you fix it?
Client-side routes exist only in the JS router; S3 has no object at key `orders/123`, so the origin returns 404 (website endpoint) or 403 (REST endpoint with OAC), which the browser shows on a hard load. The fix is a CloudFront custom error response mapping 403/404 to `/index.html` with a 200 status so the SPA boots and the router resolves the path. Scope it thoughtfully — blanket-mapping all 404s to index.html hides genuinely missing assets and can hurt SEO for public sites.
Q5. Describe your cache-control and invalidation strategy for deploys. Why not just invalidate everything each time?
Give content-hashed assets `max-age=31536000, immutable` since a code change yields a new filename — cache busting is automatic and you never need to invalidate them. Give `index.html` `no-cache` or a short TTL so it revalidates and picks up references to the new hashed files. Blanket `/*` invalidations are slow, cost money beyond the first 1,000/month, and momentarily tank your cache hit ratio; with hashing you at most invalidate just `/index.html`, or skip invalidation entirely by letting its short TTL expire.
Q6. What's the difference between the S3 website endpoint and the REST endpoint, and which should CloudFront use?
The website endpoint (`s3-website-<region>`) supports index/error documents and redirect rules but is HTTP-only, public, and doesn't support OAC. The REST endpoint (`s3.amazonaws.com`) supports HTTPS and private access via OAC/bucket policy but has no website-style routing. Best practice is CloudFront -> REST endpoint with OAC for a private, secure origin, and you replicate the index/error-document behavior with CloudFront's default root object and custom error responses. One gotcha: with the REST origin, missing keys come back as 403, so your SPA fallback should handle 403, not just 404.
Q7. How does CloudFront edge caching actually decide a cache hit, and what levers control it?
CloudFront builds a cache key from the request (path plus whatever headers, query strings, and cookies your cache policy includes) and serves from the edge if a fresh entry exists, otherwise it fetches from the origin and stores it per the TTL. TTL comes from origin `Cache-Control`/`Expires` bounded by the distribution's min/default/max TTL settings. Keep the cache key minimal (don't forward unnecessary query strings/cookies) to maximize hit ratio, and use a cache policy rather than the legacy forwarded-values config.
Q8. How would you set up automated frontend deployments to this stack in CI?
In CI: build the app, `aws s3 sync ./dist s3://bucket --delete` with per-file `Cache-Control` (sync hashed assets first with long TTL, then upload `index.html` last with no-cache to avoid a window where index references not-yet-uploaded assets), then optionally `cloudfront create-invalidation --paths '/index.html'`. Use an OIDC-federated IAM role scoped to that bucket and distribution rather than long-lived keys. For zero-downtime and rollback, keep versioning on the bucket or deploy to a prefix and flip the CloudFront origin path.
