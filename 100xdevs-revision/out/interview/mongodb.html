MongoDB, Zod, Error handling and Input validation

Cohort 3.0 · 13 lessons · SDE-2 (3+ yrs) interview prep

Concept summary — all 13 steps

Context: why persistence
In-memory arrays/globals reset on every process restart and can't be shared across horizontally-scaled server instances. A database gives durable, concurrent, queryable state — this framing is the interview-standard justification for introducing a DB tier instead of storing data in the app.
What databases are
A database is a system for durable storage, concurrent access, and efficient querying/indexing, with guarantees (ACID for relational, tunable consistency for many NoSQL). Interview angle: know the axes that differentiate them — data model (relational/document/key-value/graph), consistency model, and scaling story (vertical vs horizontal sharding).
MongoDB / NoSQL model
MongoDB is a document database storing BSON (binary JSON) documents in collections, with a flexible/implicit schema instead of rigid tables. It suits denormalized, read-heavy, evolving-shape data; the key trade-off vs SQL is no cross-document joins by default and app-enforced schema, in exchange for easier horizontal sharding and developer velocity.
Provision Atlas cluster
MongoDB Atlas gives a free-tier managed cluster; you create a DB user, allow-list IPs, and get a SRV connection string. Interview-relevant hygiene: the connection string holds credentials so it belongs in an env var/secret, never committed, and IP allow-listing plus least-privilege DB users are baseline security.
Seeding data
Seeding pre-populates the DB with baseline/reference or test data via a script (insertMany or a seed file) so dev and CI environments start in a known state. It matters for reproducible local setup and integration tests, and the script should be idempotent so re-running doesn't create duplicates.
CRUD operations
Create/Read/Update/Delete map to insertOne/find/updateOne/deleteOne (native driver) or Mongoose model methods. SDE-2 depth: understand query filters, projection, atomic update operators ($set/$inc/$push) that avoid read-modify-write races, and that update/delete take a filter — a missing filter or a typo can touch far more documents than intended.
Todo backend with Express + Mongoose
A REST backend wires Express routes to Mongoose models, connecting once at startup and mapping HTTP verbs to CRUD. Interview angle: define a schema/model, keep the DB connection as a singleton (don't reconnect per request), and structure routes/controllers so validation and error handling can be layered cleanly.
Testing the app
Exercise each endpoint (Postman/curl or automated tests) covering happy path and failure cases — bad payloads, missing auth, not-found IDs. The point is to verify status codes and response shapes, not just that the DB write happened; for real tests use a dedicated/ephemeral test DB (or mongodb-memory-server) so tests are isolated and repeatable.
Improvements / hardening
Refactor the naive app toward production shape: authentication (JWT), middleware for cross-cutting concerns, separation of routes/controllers/models, and config via env vars. This step is where you introduce the concerns that the next steps formalize — secure credentials, robust errors, and validated input.
Password hashing (bcrypt)
Never store plaintext passwords — hash with a slow, salted adaptive algorithm like bcrypt (salt is generated per-password and embedded in the hash; a cost/rounds factor makes brute-force expensive). Interview must-knows: hashing is one-way (vs reversible encryption), per-user salts defeat rainbow tables, and you compare via bcrypt.compare rather than re-hashing and string-matching.
Error handling
Wrap async route logic so failures surface as proper HTTP responses instead of unhandled rejections/500s, ideally via a centralized Express error-handling middleware (the 4-arg (err,req,res,next) signature). SDE-2 depth: distinguish operational errors (bad input, not found → 4xx) from programmer errors, avoid leaking stack traces/internals to clients, and remember async errors must be forwarded to next() or caught by a wrapper.
Input validation with Zod
Zod defines a schema and validates/parses untrusted request input at the boundary, giving both runtime safety and static TypeScript types via z.infer. Why it matters: never trust client input (injection, malformed data, type confusion); use safeParse to return structured 400 errors, and validation belongs before any DB write, typically as middleware.
Relationships in MongoDB
Model relationships either by embedding subdocuments (denormalized, single-read, good for contained one-to-few data) or by referencing ObjectIds and joining with populate/$lookup (normalized, good for shared or unbounded data). The core trade-off is read performance and atomicity (embedding) vs data duplication and update complexity (referencing) — choose based on access patterns and cardinality.

Real-world example

Multi-tenant Todo/Task API backed by MongoDB Atlas

You're building the backend for a task-management app: users sign up, log in, and manage their own todos. You provision a free Atlas cluster and keep the SRV connection string in an env var. Mongoose models define User and Todo, with each Todo referencing its owner via an ObjectId (userId) rather than embedding, since a user's todo list is unbounded and frequently updated. On signup you hash the password with bcrypt before saving; on login you bcrypt.compare and issue a JWT. Every mutating route runs Zod validation middleware first (rejecting malformed bodies with a 400 and a structured error list), then performs the CRUD operation scoped to the authenticated user's id so one user can never read or delete another's todos. All async handlers are wrapped so thrown errors — a duplicate email, an invalid ObjectId, a DB timeout — flow into a central error-handling middleware that maps them to clean 4xx/5xx responses without leaking stack traces. When you later add a 'fetch todos with owner name' endpoint, you use populate to resolve the referenced user. This single app touches every step: Atlas setup, schema/CRUD, hashing, validation, error handling, and reference-based relationships.

Interview Q&A (SDE-2)

Q1. When would you embed documents versus reference them with ObjectIds in MongoDB?
Embed when the related data is owned by and read together with the parent, is bounded in size, and doesn't need to be queried independently — e.g. a user's address or a post's small comment list — because it gives single-read access and atomic updates on the whole document. Reference when the data is shared across parents, grows unboundedly, or is updated independently — e.g. users↔orders — to avoid duplication and the 16MB document limit. The deciding factors are access patterns, cardinality, and how often the sub-data changes; reference reads require a populate/$lookup join, which is the cost you pay for normalization.
Q2. Why hash passwords with bcrypt instead of a fast hash like SHA-256, and what does the salt do?
Fast hashes are designed for speed, which is exactly wrong for passwords — an attacker with the hash dump can compute billions of guesses per second. bcrypt is deliberately slow with a configurable cost factor, so you can tune it to stay expensive as hardware improves. The salt is a unique random value per password (bcrypt generates and stores it inside the hash string) that ensures identical passwords produce different hashes, defeating precomputed rainbow tables and making each hash independently attackable. Verification uses bcrypt.compare, which re-derives the hash using the embedded salt.
Q3. How does error handling work in Express for async route handlers, and why is a centralized error middleware preferred?
Express's default error handling only catches synchronous throws; an error thrown or a rejected promise inside an async handler must be passed to next(err) (or caught by an async-wrapper utility) or it becomes an unhandled rejection and the request hangs or 500s. A centralized error-handling middleware — the special four-argument (err, req, res, next) signature registered last — is preferred because it puts all response formatting, logging, and status-code mapping in one place instead of duplicating try/catch response logic in every route. It also lets you distinguish operational errors (map to 4xx) from unexpected ones (500) and ensures you never leak stack traces to clients.
Q4. Why validate input with Zod at the API boundary when Mongoose already has schema validation?
They operate at different layers and serve different goals. Zod validates untrusted request input before it reaches business logic or the DB, gives you structured, client-friendly 400 errors, coerces/transforms shapes, and — crucially in TypeScript — produces a static type via z.infer so your validated data and your types can't drift. Mongoose validation is a last-line DB-schema guard but runs later, couples validation to the persistence layer, and produces less friendly errors. Validating early at the edge follows 'never trust the client' and prevents malformed data from ever touching your handlers.
Q5. What's the difference between Zod's parse and safeParse, and which belongs in a request handler?
parse throws a ZodError on invalid input, while safeParse returns a discriminated result object ({ success: true, data } or { success: false, error }) without throwing. In a request handler, safeParse is usually cleaner because you branch on success and return a 400 with the formatted error, avoiding try/catch for expected validation failures. parse is fine when you've centralized error handling and want a ZodError to bubble up to your error middleware, which then formats it — either works, but the choice should be consistent with how the rest of your errors flow.
Q6. How do you keep atomic, race-free updates in MongoDB?
Use atomic update operators like $set, $inc, and $push in updateOne/findOneAndUpdate rather than reading a document into the app, mutating it, and writing it back — the read-modify-write pattern has a lost-update race under concurrency. Operators like $inc apply server-side atomically per document. For invariants that span the read and the write (e.g. 'decrement stock only if > 0'), express the condition in the query filter itself, or use findOneAndUpdate with the condition so the check and update are one atomic operation. Multi-document atomicity requires transactions on a replica set.
Q7. Why should the MongoDB connection be established once at startup rather than per request?
Opening a connection is expensive (TCP + TLS handshake + auth), and the driver maintains an internal connection pool that's meant to be reused across requests. Connecting per request exhausts the pool, adds latency to every call, and can crash the DB with connection storms under load. The correct pattern is to connect once at boot (awaiting it before serving traffic) and let the driver's pool multiplex concurrent queries; in serverless environments you additionally cache the connection across warm invocations to avoid reconnecting on every cold-ish call.
Q8. How would you test the todo backend without polluting or depending on production data?
Point tests at an isolated database — either a dedicated test cluster, a per-run ephemeral DB, or an in-memory instance like mongodb-memory-server — and seed a known state before each test, cleaning up after so tests don't leak into each other. Cover both happy paths and failure modes: invalid payloads (expect 400 from validation), unauthenticated/unauthorized access (401/403 and correct user-scoping), and missing resources (404). Assert on status codes and response shape, not just that a write occurred, so the API contract itself is under test.
