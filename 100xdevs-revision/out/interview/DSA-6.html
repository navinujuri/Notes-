Concept summary — all 5 steps

Strings (C-strings vs std::string)
Two worlds coexist: C-style null-terminated char arrays (raw pointer, manual sizing, O(n) strlen, easy to overflow) and std::string (owns a heap buffer, tracks size + capacity, RAII cleanup, SSO for small strings). Interview angle: know that std::string uses Small String Optimization to store short strings inline without heap allocation, that .size() is O(1) while strlen is O(n), and when to pass std::string_view to avoid copies.
Pointers
A pointer holds a memory address; dereferencing (*p) reads/writes the pointee and p->m accesses members. They enable indirection, aliasing, polymorphism (base* to derived), and building linked structures. Interview angle: nullptr vs uninitialized/dangling pointers, pointer-vs-reference (references can't rebind or be null), pointer arithmetic scaling by sizeof(T), and why raw owning pointers are discouraged in favor of smart pointers.
Dynamic allocation (new/delete, heap)
new allocates on the heap and returns a pointer whose lifetime you manage manually; you must delete (or delete[] for arrays) exactly once or you leak/double-free. Interview angle: pair new with delete and new[] with delete[], understand that the heap outlives the current scope (unlike stack/automatic storage), and know that modern C++ replaces raw new/delete with std::make_unique / std::make_shared so RAII guarantees cleanup even on exceptions.
Function call arguments (pass semantics)
C++ passes by value by default (copies the argument), so use pass-by-reference (T&) to mutate the caller's object, const T& to avoid copying large objects without allowing mutation, and T* when the argument is optional/nullable. Interview angle: const-ref for read-only heavy types, the cost of unnecessary copies, and move semantics (T&&) / passing by value + std::move for sink parameters that take ownership.
Classes and objects
A class bundles data (members) with behavior (methods) plus access control (public/private/protected) and lifetime hooks: constructor, destructor, copy/move constructor and assignment. Interview angle: RAII (acquire in ctor, release in dtor), the Rule of 0/3/5, when the compiler synthesizes special members, and the difference between stack objects (deterministic destruction at scope end) and heap objects (destroyed on delete).

Real-world example

A parser that reads a large log file into custom record objects

Imagine building a log-ingestion utility that reads a multi-GB access log and turns each line into a LogRecord object. You store parsed fields in std::string members (letting SSO keep short tokens like HTTP verbs off the heap) and pass each raw line into your parser as `const std::string&` to avoid copying millions of lines. For a batch of records too large to size at compile time you originally used `new LogRecord[n]` and had to remember `delete[]` on every early-return path — a classic leak source, so you refactor to `std::vector<LogRecord>` / `std::make_unique`, letting RAII destructors free memory even when parsing throws. The LogRecord class owns its buffers, so you define (or =default/=delete) its copy and move members to control whether records are cheaply moved into the vector or accidentally deep-copied. Pointers appear when you hand out non-owning `LogRecord*` into an index map for O(1) lookup, and you must ensure those pointers don't dangle when the vector reallocates and moves its elements.

Interview Q&A (SDE-2)

Q1. What is the difference between a C-style string and std::string, and when would you still use a C-string?
A C-string is a raw `char*`/`char[]` terminated by '\0' with no stored length, manual memory management, and O(n) length lookup; std::string owns a resizable heap buffer, tracks size and capacity, and cleans itself up via RAII. std::string is the default for safety and convenience. You still reach for C-strings at C API boundaries (POSIX/Win32 calls), for compile-time string literals, or when interfacing with libraries expecting `const char*` — you get one via `.c_str()`.
Q2. Explain Small String Optimization (SSO).
SSO lets std::string store short strings (typically up to ~15 chars on 64-bit libstdc++/libc++) directly inside the string object's own storage instead of allocating on the heap. This avoids a malloc/free for the common case of short strings, improving performance and locality. The trade-off is a larger sizeof(std::string) (usually 24-32 bytes) and a branch on every access to check whether the data is inline or heap-allocated.
Q3. Pointer vs reference — what are the concrete differences?
A reference must be bound at initialization, can never be null, and cannot be rebound to another object — it's an alias. A pointer can be null, can be reseated, supports arithmetic, and requires explicit dereferencing. Use references for mandatory, non-null parameters (especially const& to pass large objects cheaply); use pointers when the value is optional/nullable, when you need to represent 'no object', or when you need to iterate/rebind.
Q4. What goes wrong with raw new/delete, and how do smart pointers fix it?
Raw new/delete require you to free exactly once on every code path; exceptions, early returns, and mismatched new[]/delete cause leaks, double-frees, or undefined behavior. std::unique_ptr and std::shared_ptr apply RAII: the destructor frees automatically when the owner goes out of scope, even during stack unwinding. unique_ptr is zero-overhead single ownership; shared_ptr adds a reference count (atomic, thus not free) for shared ownership. Prefer make_unique/make_shared for exception safety and a single allocation.
Q5. When should a function take a parameter by value, by const reference, or by rvalue reference?
Use const T& for read-only access to objects that are expensive to copy — no copy, no mutation. Use by-value for cheap/trivially-copyable types (int, pointers) or when the function needs its own copy anyway (a 'sink' where you then std::move it into storage). Use T& when you must mutate the caller's object, and T&& (or the by-value-then-move idiom) for perfect forwarding and taking ownership of temporaries efficiently.
Q6. What is the Rule of Three/Five, and when does it apply?
If a class manually manages a resource and needs a custom destructor, it almost certainly also needs a custom copy constructor and copy assignment (Rule of Three) to avoid shallow-copy double-frees; in C++11+ add the move constructor and move assignment (Rule of Five) for efficiency. The Rule of Zero is the modern preference: don't manage resources manually — wrap them in std::string/vector/unique_ptr so all five special members are correctly compiler-generated and you write none of them.
Q7. Explain RAII and why it's central to C++ resource management.
RAII (Resource Acquisition Is Initialization) ties a resource's lifetime to an object's lifetime: acquire in the constructor, release in the destructor. Because C++ guarantees destructors run deterministically at scope exit — including during exception stack unwinding — resources like memory, file handles, locks, and sockets are released automatically and exactly once. This is why std::lock_guard, std::unique_ptr, and std::fstream are leak-safe without try/finally, which C++ doesn't even have.
Q8. A pointer into a std::vector suddenly points to garbage after inserting elements. Why?
When a vector exceeds its capacity it allocates a larger buffer, moves/copies its elements there, and frees the old buffer — invalidating all pointers, references, and iterators into the old storage. Any raw LogRecord* you cached now dangles. Fixes: reserve() enough capacity up front so no reallocation occurs, store indices instead of pointers, or use a container with stable addresses (std::deque, std::list, or a node-based map). This is a frequent source of subtle heap-use-after-free bugs caught by AddressSanitizer.
