Websockets

Cohort 3.0 · 5 lessons · SDE-2 (3+ yrs) interview prep

Concept summary — all 5 steps

WebSockets: the protocol
A WebSocket is a persistent, full-duplex, bidirectional channel over a single TCP connection, established by 'upgrading' an ordinary HTTP request (client sends Upgrade: websocket + Sec-WebSocket-Key, server replies 101 Switching Protocols). After the handshake, both sides push framed messages at any time with no per-message HTTP header overhead. Interview angle: know when to reach for it vs HTTP polling, long-polling, or SSE — WebSockets win when you need low-latency server-to-client push AND client-to-server messages (chat, live cursors, trading, multiplayer).
WebSockets in Node.js (why it fits)
Node's single-threaded, non-blocking event loop is well suited to holding many idle-but-open socket connections cheaply, since each connection is mostly waiting on I/O rather than burning CPU. Native Node has no browser-style WebSocket server API, so you use the `ws` library, which piggybacks on an existing http.Server and handles the framing/handshake. Interview angle: the constraint isn't CPU per message, it's file descriptors and memory per open connection, plus the fact that CPU-heavy message handling blocks all connections on that event loop.
Server-side ws implementation
You instantiate `new WebSocketServer({ server })` (or `{ port }`), then handle the `connection` event to get a per-client socket, and on that socket handle `message`, `close`, and `error`; you send with `ws.send()` and broadcast by iterating `wss.clients`. Interview angle: be ready to discuss tracking connection state (rooms/user maps), heartbeat/ping-pong to detect dead peers (`ws.isAlive` + `terminate()`), and backpressure via `ws.bufferedAmount` for slow consumers.
Client-side (browser) code
The browser exposes the WebSocket API: `new WebSocket(url)` with `onopen`, `onmessage`, `onclose`, `onerror` handlers and a `send()` method; `wss://` is the TLS variant. Production clients must add reconnection with exponential backoff + jitter, app-level heartbeats, and re-subscription/state resync after reconnect. Interview angle: the network can drop a connection silently, so you can't trust `onclose` alone — heartbeats and idempotent resync are what separate a toy from a real client.
Scaling WebSocket servers
WebSockets are stateful — a client is pinned to one server instance for the life of the connection — so a naive multi-instance deploy breaks broadcasts because user A on server 1 can't reach user B on server 2. Solve fan-out with a shared pub/sub backplane (Redis Pub/Sub, Kafka, or NATS): each node subscribes to channels and relays published messages to its locally connected clients; use sticky sessions / connection-aware load balancing at the LB. Interview angle: discuss the connection ceiling per node (fds, memory), horizontal scaling limits, and why the pub/sub layer (not the ws nodes) usually becomes the bottleneck.

Real-world example

Multi-instance live chat / collaboration service

You're building the real-time layer for a collaboration app (think shared docs presence + chat). A browser client opens a `wss://` connection and joins room-123. On the server, `ws` accepts the upgrade, stores the socket in an in-memory map keyed by room, and runs a 30s ping/pong heartbeat to evict dead connections. When a user sends a message, the node persists it and needs to broadcast to everyone in room-123 — but you run 4 Node instances behind an ALB, and the room's members are spread across all of them. So instead of only iterating local `wss.clients`, each node publishes the message to a Redis channel `room:123`; every node is subscribed and relays it to its own locally-connected room members. The LB uses sticky sessions so a client's HTTP upgrade and subsequent frames land on the same node. On the client you wrap the socket in a reconnect-with-backoff helper that re-joins the room and refetches missed messages after any drop. Classic debugging story: messages work with one instance but silently fail to fan out after you scale to two — the fix is the pub/sub backplane.

Interview Q&A (SDE-2)

Q1. How does a WebSocket connection get established, and how does it relate to HTTP?
It starts as a normal HTTP/1.1 GET with `Upgrade: websocket`, `Connection: Upgrade`, and a base64 `Sec-WebSocket-Key`. The server validates it, echoes back a hashed `Sec-WebSocket-Accept`, and returns `101 Switching Protocols`; from that point the same TCP connection is used for full-duplex WebSocket frames instead of HTTP. This reuse of the HTTP handshake is why WebSocket servers can share a port/server with HTTP and why WebSockets traverse most proxies and firewalls.
Q2. When would you choose WebSockets over SSE or long-polling?
Use WebSockets when you need low-latency, bidirectional traffic — the client sends frequently and the server pushes frequently (chat, multiplayer, collaborative editing, live trading). SSE is simpler and fine when you only need server-to-client push over HTTP and are okay with text/one-way (dashboards, notifications), and it auto-reconnects. Long-polling is a fallback for environments that block upgrades, but it has higher latency and overhead; WebSockets are the right default when true duplex is the requirement.
Q3. Why are WebSockets harder to scale horizontally than stateless HTTP?
A WebSocket is a long-lived, stateful connection pinned to a single server instance, so the usual 'any request to any node' model breaks. Broadcasting to a logical group (a room) requires reaching clients that may be connected to different nodes, so you need a shared pub/sub backplane (Redis/Kafka/NATS) for fan-out and sticky routing at the load balancer. You also cap out on per-node resources — open file descriptors and memory per connection — long before CPU, so scaling is about connection count, not request throughput.
Q4. How do you detect and handle dead or half-open WebSocket connections?
TCP can silently drop a peer (network partition, laptop sleep) without a clean `close` frame, so you can't rely on the `close` event. Implement application-level heartbeats: the server sends periodic pings, marks each socket `isAlive=false` before sending, sets it true on `pong`, and `terminate()`s any socket that didn't respond by the next interval. On the client, run a matching heartbeat plus reconnect-with-backoff, because `onclose`/`onerror` may fire late or not at all.
Q5. What is backpressure in a WebSocket server and how do you handle a slow consumer?
If you push messages faster than a client can receive them, they queue in the server's send buffer (`ws.bufferedAmount` grows), consuming memory and eventually risking OOM across many slow clients. Handle it by checking `bufferedAmount` before sending and dropping/coalescing messages, applying per-connection rate limits, or disconnecting clients that fall too far behind. This matters most for fan-out heavy workloads like market data where one stalled client shouldn't degrade the whole node.
Q6. Walk through how a broadcast to a chat room works across multiple server instances.
Each instance keeps only its locally-connected sockets, grouped by room, in memory. When a message arrives, the receiving node publishes it to a shared channel (e.g. Redis `PUBLISH room:123 payload`); every instance is subscribed to that channel and, on receipt, relays the payload to its own local members of room-123. This decouples senders from receivers so any node can originate a broadcast, and the load balancer uses sticky sessions so a given client's frames consistently hit the node that holds its socket. The pub/sub layer, not the ws nodes, typically becomes the scaling bottleneck.
Q7. What are the main security concerns with WebSockets?
First, authentication/authorization: the browser WebSocket API can't set custom headers, so you typically auth via a token in the query string or a cookie during the upgrade, then validate it before accepting the connection — and re-check authz per message for sensitive actions. Second, CSWSH (Cross-Site WebSocket Hijacking): WebSockets aren't bound by the same-origin policy, so you must validate the `Origin` header server-side. Also enforce `wss://` (TLS), input validation on messages, and per-connection rate limiting to prevent abuse.
Q8. What happens to WebSocket connections during a deploy or restart, and how do you minimize disruption?
A restart drops every open connection on that instance simultaneously, causing a reconnect storm that can hammer the new instances and any auth/pub-sub dependencies. Mitigate with graceful shutdown (stop accepting new connections, send `close` frames, drain over a window), client reconnect with exponential backoff + jitter to spread the herd, and idempotent state resync so a reconnected client refetches anything missed. Rolling deploys plus a connection-draining-aware load balancer keep only a fraction of connections churning at once.
