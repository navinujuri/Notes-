Creating a web based wallet, RPCs

Cohort 2.0 · 9 lessons · SDE-2 (3+ yrs) interview prep

Concept summary — all 9 steps

Context: what a wallet actually is
A blockchain wallet stores keypairs, not coins; the network holds balances as on-chain state, and the private key is just the authority to sign state transitions from your address. This framing matters in interviews because it explains why 'losing a wallet' means losing keys, and why a wallet is fundamentally a key-management + transaction-signing app rather than a database of funds.
keccak-256 hashing
Ethereum's core hash function, used to derive addresses (last 20 bytes of keccak256 of the uncompressed public key) and 4-byte function selectors. The interview gotcha: Ethereum uses the original Keccak padding (0x01), NOT the finalized NIST SHA3-256 (0x06), so keccak256 and sha3-256 produce different digests despite the confusing naming.
Frontend vs backend and the trust boundary
For a web wallet, key generation and signing must run entirely client-side (browser/JS); the private key or seed must never be transmitted to a server. The 'backend' is essentially an RPC node you read from and broadcast to, so the security-critical logic lives in the frontend and the network boundary only ever sees already-signed transactions and public data.
RPC and JSON-RPC
Blockchains expose their state through Remote Procedure Calls using the JSON-RPC 2.0 protocol: a POST body of {jsonrpc, method, params, id} returning {jsonrpc, result|error, id}. It matters because every wallet interaction (read balance, send tx, estimate gas) is a JSON-RPC request; interviewers probe whether you know it's stateless request/response over HTTP, with WebSockets used for subscriptions and arrays used for batching.
Wei and Lamports (base units)
Chains denominate value in integer base units to avoid floating-point error: 1 ETH = 10^18 wei, 1 SOL = 10^9 lamports. The practical point is that all balance and fee math must use BigInt/BN, never JS Number (which loses precision beyond 2^53), and you only convert to human-readable decimals at the display layer.
RPC server / node
The RPC server is the node (or managed gateway like Alchemy, Infura, QuickNode, Helius) that accepts JSON-RPC and answers from chain state. Interview angle: full node vs archive node (historical state), self-hosted vs managed trade-offs, and awareness of rate limits, request quotas, and eventual consistency across load-balanced nodes.
Common Solana RPC calls
Core methods: getBalance, getAccountInfo, getLatestBlockhash, sendTransaction, getSignatureStatuses/confirmTransaction, and requestAirdrop (devnet only). Key concept: a Solana transaction must embed a recent blockhash that expires after ~150 slots, which provides replay protection and a natural expiry window instead of an ETH-style nonce.
Common Ethereum RPC calls
Core methods: eth_getBalance, eth_getTransactionCount (the account nonce), eth_estimateGas / eth_gasPrice (or EIP-1559 fee fields), eth_sendRawTransaction (broadcast a signed tx), eth_call (read-only contract execution), and eth_getTransactionReceipt (confirmation + logs). The recurring theme is per-account sequential nonces and locally signing before broadcasting the raw hex.
Assembling the web wallet
Tie it together: generate a BIP39 mnemonic to a seed, derive HD keypairs via BIP32/BIP44 paths (m/44'/60'/... for ETH, m/44'/501'/... for Solana), render balances from RPC reads, and build/sign/broadcast transactions client-side. This is the capstone that combines cryptography, unit handling, HD derivation, and the RPC read/write cycle into one product.

Real-world example

A minimal multi-chain web wallet (lightweight MetaMask/Phantom)

You're an SDE-2 building a browser wallet that supports both Ethereum and Solana from a single seed phrase. On first run you generate a BIP39 mnemonic in the browser, derive an ETH keypair at m/44'/60'/0'/0/0 and a Solana keypair at m/44'/501'/0'/0' from the same seed; the mnemonic is encrypted with a user password and kept in local storage, never sent anywhere. The dashboard reads balances via JSON-RPC against Alchemy endpoints: eth_getBalance for ETH (rendered by dividing wei by 10^18 using BigInt) and getBalance for SOL (lamports / 10^9). To send ETH, you fetch the nonce with eth_getTransactionCount(address,'pending'), estimate fees with eth_estimateGas plus EIP-1559 fields, sign the transaction locally, and broadcast with eth_sendRawTransaction, then poll eth_getTransactionReceipt. To send SOL, you fetch getLatestBlockhash, attach it to the transaction, sign, call sendTransaction, and confirm via getSignatureStatuses. A bug you'd realistically debug: two sends in quick succession both grab nonce N and one gets dropped as a replacement, or a Solana tx fails with 'blockhash not found' because it was signed too long after fetching the blockhash and expired past the ~150-slot window.

Interview Q&A (SDE-2)

Q1. Why must a web wallet generate keys and sign transactions client-side, and what is the trust boundary?
The private key (or seed) is the sole authority over funds, so if it ever reaches a server that server can drain the wallet and becomes a catastrophic single point of compromise. In a non-custodial web wallet the browser is the trust boundary: it holds and uses the key locally, and the only things that cross the network are public reads and already-signed transactions. The RPC node therefore never needs the key and can be fully untrusted for confidentiality (though it can still censor or lie about reads, which is why some clients verify against multiple providers).
Q2. What's the difference between keccak-256 and SHA3-256, and where does it bite in Ethereum?
They use the same Keccak sponge construction but different padding: Ethereum adopted the original Keccak submission (0x01 domain padding) before NIST finalized SHA-3 with different padding (0x06), so the two produce different digests for the same input. This matters because Ethereum addresses, function selectors, and event topics all depend on keccak-256; using a library's 'sha3' that actually implements NIST SHA3-256 will silently produce wrong addresses and selectors. Always confirm the library returns Ethereum-style keccak256.
Q3. Walk through how an Ethereum transaction gets from a wallet to the chain.
You read the sender's next nonce with eth_getTransactionCount, determine fees (EIP-1559 maxFeePerGas/maxPriorityFeePerGas or legacy gasPrice) and a gas limit via eth_estimateGas, then construct and RLP-encode the transaction and sign it locally with the private key (ECDSA over secp256k1). You broadcast the signed hex with eth_sendRawTransaction, which returns the tx hash immediately, and then poll eth_getTransactionReceipt until it's mined to get status and logs. The nonce is the tricky part: it must be strictly sequential per account, so concurrent sends or a stuck pending tx can block or replace later transactions.
Q4. Why do base units like wei and lamports exist, and how do you handle them safely in JavaScript?
Chains store value as unsigned integers to make arithmetic deterministic and avoid floating-point rounding, so 1 ETH is 10^18 wei and 1 SOL is 10^9 lamports. JavaScript's Number is an IEEE-754 double that only holds integers exactly up to 2^53-1, far below 10^18, so doing balance or fee math with Number will silently corrupt values. You must use BigInt (or a big-number library) for all on-chain quantities and only convert to a decimal string at the presentation layer.
Q5. What is a recent/latest blockhash on Solana and why is it required in a transaction?
Every Solana transaction must reference a recent blockhash (from getLatestBlockhash), which the runtime uses both to prevent replay of old transactions and to give each transaction a short expiry window of roughly 150 slots (about a minute). If you sign a transaction and submit it too late, validators reject it as expired ('blockhash not found'), which is a common source of failed sends in a wallet. This differs from Ethereum's model, where sequential per-account nonces provide replay protection instead of a time-boxed blockhash.
Q6. Describe the JSON-RPC 2.0 request/response shape and when you'd use batching or WebSockets.
A request is a JSON object with jsonrpc:'2.0', a method name, a params array, and an id; the response echoes the id and returns either result or an error object with code and message. It's normally sent as an HTTP POST and is stateless, so each call is independent. You can batch by POSTing an array of request objects to cut round-trips, and you switch to a WebSocket connection for push-style subscriptions (eth_subscribe / logsNotification or Solana's accountSubscribe) when you need live updates instead of polling.
Q7. How does HD wallet derivation (BIP39/32/44) let one phrase control many chains and accounts?
BIP39 turns entropy into a human-readable mnemonic and then into a 512-bit seed; BIP32 defines hierarchical deterministic derivation so a single seed produces a tree of child keys via a derivation path; BIP44 standardizes that path as m/44'/coin'/account'/change/index, where the coin type distinguishes chains (60 for ETH, 501 for Solana). This means one backed-up phrase deterministically regenerates every account across chains, and hardened derivation (the apostrophe) prevents a leaked child key plus parent public key from exposing siblings. The trade-off is that the mnemonic is the master secret, so its storage and encryption are the entire security model.
Q8. When would you run your own RPC node versus using a managed provider like Alchemy or Infura?
Managed providers give you instant, highly-available JSON-RPC with archive data and no ops burden, which is right for most wallets, but you pay per request, hit rate limits, and trust them not to censor or return stale/incorrect reads. Running your own node removes that trust and cost-per-call and can lower latency, but it's expensive to operate, needs an archive node for historical state, and you must handle sync, storage, and uptime yourself. A common middle ground is using multiple providers with failover and cross-checking critical reads to avoid a single point of trust.
