Docker End to End

Cohort 2.0 · 22 lessons · SDE-2 (3+ yrs) interview prep

Concept summary — all 22 steps

Why Docker?
Docker solves 'works on my machine' by packaging an app with its exact runtime, libs, and OS-level deps into a portable image that runs identically on any host with a container runtime. Interview angle: it standardizes dev/CI/prod environments and enables dense, fast-starting deployments compared to VMs.
Containerization
Containerization isolates a process using Linux kernel primitives — namespaces (PID, net, mount, UTS, IPC, user) for isolation and cgroups for resource limits — while sharing the host kernel. This is why containers start in milliseconds and are far lighter than VMs, which each boot a full guest kernel.
History of Docker
Docker (2013) didn't invent containers (LXC, Solaris Zones, cgroups predate it) but made them usable with an image format, a registry, and a simple CLI/Dockerfile workflow. It later split into containerd (runtime, now CNCF) and moby, and the OCI spec standardized image/runtime formats so the ecosystem isn't Docker-locked.
Installing Docker
Install Docker Engine (Linux) or Docker Desktop (Mac/Windows, which runs a lightweight Linux VM since containers need a Linux kernel). Know the client/daemon split: the `docker` CLI talks to the dockerd daemon over a socket, and on Linux adding your user to the `docker` group grants root-equivalent access.
Inside Docker (architecture)
The architecture is CLI -> dockerd (REST API) -> containerd -> runc, where runc actually creates the container via kernel syscalls. Understanding this layering explains why Kubernetes can drop dockerd and talk to containerd directly via CRI.
Images vs containers
An image is an immutable, layered, read-only template; a container is a running (or stopped) instance with a thin writable layer on top via a copy-on-write filesystem. Interview angle: analogous to class vs object — many containers share one image's read-only layers, and data written to the container layer is lost on removal unless persisted in a volume.
Port mapping
Containers get their own network namespace, so a process listening on port 3000 inside is unreachable from the host until you publish it with `-p hostPort:containerPort`. Docker sets up NAT/iptables rules to forward host traffic; EXPOSE in a Dockerfile is only documentation and does not publish anything.
Common Docker commands
Core lifecycle: `run` (create+start), `ps -a`, `stop`/`kill`, `rm`, `logs`, `exec -it` (shell into a running container), `images`, `rmi`, `pull`/`push`. Know `exec` vs `attach` (exec spawns a new process; attach hooks into PID 1's stdio) — a common practical distinction.
Dockerfile
A Dockerfile is a declarative recipe: FROM, RUN, COPY/ADD, WORKDIR, ENV, EXPOSE, CMD/ENTRYPOINT. Interview angle: ENTRYPOINT sets the fixed executable while CMD provides default args (overridable at `docker run`); prefer exec form (`CMD ["node","index.js"]`) so signals reach PID 1 for clean shutdown.
Building images
`docker build -t name:tag .` sends the build context to the daemon and executes each instruction, caching layers. Practical points: a .dockerignore trims the context (faster builds, smaller images, no secret leakage), and BuildKit (`DOCKER_BUILDKIT=1`) adds parallelism, better caching, and build secrets.
Running images
`docker run` turns an image into a container; key flags are `-d` (detached), `-it` (interactive TTY), `--name`, `--rm` (auto-cleanup), `--restart`, and resource limits like `--memory`/`--cpus`. Know that each `run` creates a fresh container — reusing state means `docker start` on an existing one, not repeated `run`.
Passing env variables
Inject config at runtime with `-e KEY=val` or `--env-file`, keeping the same image across environments (12-factor config). Interview angle: env vars set at runtime override Dockerfile ENV; never bake secrets into image layers since they persist in history — use env-file, secrets managers, or BuildKit `--mount=type=secret`.
More commands
Beyond the basics: `inspect` (JSON metadata/IP/mounts), `stats` (live resource usage), `system df`/`system prune` (reclaim disk), `cp`, `commit`, `tag`, `history` (see layers). These are the day-to-day debugging and housekeeping tools an SDE-2 reaches for.
Pushing to Docker Hub
Tag as `registry/user/repo:tag` (Docker Hub is the default registry), `docker login`, then `docker push`. Interview angle: use immutable, meaningful tags (git SHA/semver) rather than relying on `latest` in prod, and understand a registry stores content-addressable layers so unchanged layers aren't re-uploaded.
Layers in Docker
Each Dockerfile instruction creates a read-only layer identified by a content hash; images are stacks of layers unified by an overlay filesystem. This underpins caching and deduplication — multiple images sharing a base only store those layers once.
Layers practically
Practically, an unchanged layer plus all layers below it are pulled from cache, but changing one layer invalidates every layer after it. `docker history` reveals layer sizes, and each RUN/COPY adds a layer — chaining commands in one RUN reduces layer count and intermediate bloat.
Why layers?
Layers give three wins: build-cache speed (reuse unchanged steps), storage/transfer efficiency (shared and deduplicated across images), and reproducibility. The trade-off to understand: ordering instructions from least- to most-frequently-changed maximizes cache hits.
Optimising Dockerfile
Key techniques: multi-stage builds (compile in a fat builder stage, copy only artifacts into a slim runtime), order layers so dependency installs (e.g. `COPY package.json && npm ci`) cache before source copies, use small bases (alpine/distroless/slim), and combine RUN steps. Goal: smaller images, faster builds, smaller attack surface, and running as a non-root USER.
Networks and volumes
These are Docker's two persistence/connectivity primitives: networks let containers talk to each other and the outside world, while volumes persist data beyond a container's lifecycle. Both are first-class, managed objects you create and attach rather than implicit behavior.
Volumes
Volumes (Docker-managed) and bind mounts (host-path) persist data outside the container's writable layer, surviving `docker rm`. Interview angle: named volumes are the portable, backup-friendly choice for databases; bind mounts are ideal for live-reloading source in dev; tmpfs keeps sensitive data in memory only.
Network
The default `bridge` isolates containers (they reach each other only by IP), but a user-defined bridge network adds automatic DNS so containers resolve each other by name — essential for multi-service apps. Other drivers: `host` (share host netns, no isolation), `none`, and `overlay` for multi-host/Swarm.
docker-compose
Compose declares a multi-container app (services, networks, volumes, env, depends_on) in one YAML file, brought up with `docker compose up`. Interview angle: it creates a shared user-defined network so services reach each other by service name, and it's the standard for local multi-service dev — though orchestration at scale moves to Kubernetes.

Real-world example

Containerizing and shipping a Node API + Postgres for a team

You own a Node/Express service backed by Postgres. Locally you write a multi-stage Dockerfile: a builder stage runs `npm ci` and compiles TypeScript, and a slim runtime stage copies only the built `dist/` and production node_modules, runs as a non-root USER, and sets `CMD ["node","dist/index.js"]`. You order the Dockerfile so `COPY package*.json` + `npm ci` sit above `COPY . .` so dependency layers stay cached when only source changes. For local dev you write a docker-compose.yml with two services — `api` and `db` — on a shared user-defined network so the API connects using the hostname `db` (Compose DNS) instead of an IP; the DB uses a named volume so data survives `docker compose down`. Database credentials come in via `--env-file`/Compose `environment` rather than being baked into the image. You publish the API with `-p 8080:3000` for host access. In CI you build with BuildKit, tag the image with the git SHA (not `latest`), and `docker push` to your registry; only changed layers upload. When a container OOM-crashes in staging, you debug with `docker logs`, `docker stats`, `docker inspect`, and `docker exec -it` to get a shell — then set `--memory` limits and add a HEALTHCHECK.

Interview Q&A (SDE-2)

Q1. What is the difference between a container and a virtual machine, and why does it matter?
A VM virtualizes hardware and runs a full guest OS with its own kernel via a hypervisor, whereas a container shares the host's kernel and isolates processes using namespaces and cgroups. Containers are therefore far lighter (MBs vs GBs), start in milliseconds, and pack more densely, but they offer weaker isolation and are constrained to the host kernel's OS family. That trade-off is why Docker Desktop on Mac/Windows actually runs a Linux VM underneath to provide a Linux kernel.
Q2. Explain the difference between CMD and ENTRYPOINT.
ENTRYPOINT defines the executable that always runs, while CMD supplies default arguments that are easily overridden by args passed to `docker run`. If both are set, CMD's values become arguments to ENTRYPOINT. Best practice is the exec form (JSON array) so the process runs as PID 1 and receives OS signals directly, enabling graceful shutdown; the shell form wraps it in `/bin/sh -c`, which can swallow SIGTERM.
Q3. How does Docker's layer caching work, and how do you structure a Dockerfile to exploit it?
Each instruction produces a content-hashed layer; on rebuild Docker reuses a cached layer if that instruction and all preceding layers are unchanged. The moment one layer's inputs change, every subsequent layer is invalidated and rebuilt. So you order instructions from least- to most-frequently-changing — copy dependency manifests and install deps before copying application source — so a code edit doesn't bust the expensive dependency-install cache.
Q4. What are multi-stage builds and what problem do they solve?
A multi-stage build uses multiple FROM statements: an early 'builder' stage contains compilers, dev dependencies, and build tooling, and a final slim stage copies only the resulting artifacts via `COPY --from`. This keeps build-time bloat out of the shipped image, drastically shrinking size and attack surface while still allowing a full build toolchain. It's the standard way to ship compiled languages or bundled JS without leaking source, secrets, or dev deps.
Q5. How do containers communicate, and what does a user-defined bridge network give you over the default bridge?
Each container has its own network namespace; on the default bridge they can only reach each other by IP, which is brittle. A user-defined bridge network adds an embedded DNS server so containers resolve each other by container/service name, and it provides better isolation since only containers on that network can talk. This is exactly why docker-compose puts services on a shared user-defined network — the `api` service reaches Postgres via the hostname `db`.
Q6. Volumes vs bind mounts — when do you use each?
Named volumes are managed by Docker in its own storage area, are portable across hosts, easy to back up, and are the right choice for stateful services like databases in production. Bind mounts map a specific host directory into the container and are ideal for development (live-reloading source code) but couple you to the host's filesystem layout. Both persist data beyond the container's writable layer, which is otherwise destroyed on `docker rm`.
Q7. How should secrets be handled in Docker, and what's the pitfall with ENV/ARG?
Never bake secrets into image layers: values set via ENV or ARG, or via `COPY` of a secrets file, persist in the image history and layer contents and can be extracted by anyone with the image. Inject secrets at runtime with `--env-file`, orchestrator secret stores, or use BuildKit's `--mount=type=secret` which exposes a secret to a single build step without writing it to a layer. Combine with a .dockerignore so local secret files never enter the build context.
Q8. Why should you avoid the `latest` tag in production, and what's a better tagging strategy?
`latest` is a mutable, floating pointer — the image it references can change out from under you, breaking reproducibility and making rollbacks ambiguous. Better is immutable tags tied to a version or git SHA (e.g. `myapp:1.4.2` or `myapp:9f3a1c`), which pin exactly what's deployed and enable deterministic rollbacks. Since registries store content-addressable layers, using distinct tags costs almost nothing because unchanged layers aren't re-stored or re-pushed.
