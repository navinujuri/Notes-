Rate limitting, DDoS and Captcha

Cohort 2.0 · 10 lessons · SDE-2 (3+ yrs) interview prep

Concept summary — all 10 steps

Why rate limiting
Capping how many requests a client can make in a window protects you from brute-force attacks, credential stuffing, scraping, accidental client retry storms, and runaway third-party costs (SMS/email/LLM calls). The interview angle: rate limiting is about fairness and abuse prevention at the application layer, which is distinct from DDoS mitigation (volumetric, network layer) and from simple throttling for capacity.
Where to add rate limits
Limits can live at the CDN/edge (Cloudflare), the load balancer / API gateway (nginx, Kong), or in application middleware. Earlier layers are cheaper because they reject traffic before it consumes your compute, but only the app layer has enough context for per-user or per-endpoint keys; production systems usually layer both.
Build a password-reset endpoint
A typical forgot-password / send-OTP endpoint accepts an email or phone and triggers an out-of-band message (token link or numeric OTP). It matters because it is unauthenticated, side-effecting (sends real messages, costs money), and a common target, so it is the canonical example for why input endpoints need protection.
Exploiting the unprotected endpoint
With no limit, an attacker scripts thousands of requests: OTP/email bombing a victim, brute-forcing a short numeric OTP, enumerating which emails exist, and amplifying your SMS/email bill. This shows that an endpoint being functionally correct is not the same as being safe under adversarial load.
Exploiting one in production
Demonstrating the attack against a live but consenting/owned system (e.g. a simple loop of curl/fetch calls) makes the blast radius concrete: cost spikes, degraded service, and a real DoS on the messaging provider. The takeaway is that these are not theoretical bugs and must be caught in design review, not after the bill arrives.
Adding the rate limit
Fix it by keying limits on IP and on the target email/phone, using an algorithm like token bucket or sliding window (in-memory map for a demo, Redis in reality). Per-target keys matter because IP-only limits let an attacker bomb a single victim from many IPs while staying under any per-IP cap.
Problem: in-memory state doesn't scale
An in-process counter breaks the moment you run more than one instance behind a load balancer, since each replica sees only its own slice of traffic, so the effective limit is N times too loose. The fix is a shared store (Redis with atomic INCR/Lua or a sliding-window structure); also note IP keys are unreliable behind NAT/CGNAT and trivially rotated.
Captchas
A CAPTCHA forces a challenge (image, invisible risk score, or proof-of-work) to distinguish humans from bots, complementing rate limits when the abuse comes from distributed or low-volume-per-IP bots. It matters because rate limiting alone can't stop a botnet spreading load thinly across many IPs; the trade-off is added user friction, so it's usually gated to risky flows.
Adding captcha via Cloudflare Turnstile
Turnstile renders a widget that returns a token to the client, which your backend must verify server-side by POSTing the token + secret to the siteverify endpoint before honoring the request. The critical detail is that the client-side token is meaningless without server-side validation, and tokens are single-use and short-lived.
DDoS protection in prod
Volumetric attacks are absorbed at the edge via an anycast CDN/scrubbing network (Cloudflare, AWS Shield) that handles L3/L4 floods, plus a WAF and edge rate rules for L7 (HTTP) floods. Interview angle: know the L3/L4 (SYN floods, UDP amplification) vs L7 (HTTP request floods that look legitimate) distinction, because they need different mitigations and origin IPs must be hidden behind the edge.

Real-world example

The password-reset endpoint that ran up a $40k Twilio bill

You ship an unauthenticated POST /auth/forgot-password that takes an email or phone and sends a 6-digit OTP via Twilio. It works in the demo. A week later an attacker (or a buggy client) discovers it and loops it: they bomb one user with hundreds of SMS, brute-force the short OTP (10^6 space, guessable if you allow unlimited attempts), and drive the SMS bill through the roof while degrading delivery for real users. As the SDE-2 on call, you (1) add a Redis-backed sliding-window limiter keyed on BOTH source IP and the target email/phone (e.g. 3 requests / 15 min per target, plus a coarser per-IP cap), because your service runs 4 replicas so an in-memory counter would have been 4x too loose; (2) cap OTP verification attempts and lengthen/expire the code; (3) put Cloudflare Turnstile on the form and verify the token server-side via siteverify before sending anything; and (4) move the origin behind Cloudflare so edge WAF + rate rules absorb L7 floods and the origin IP is hidden from direct volumetric attacks. The layers matter: rate limits stop the cheap script, captcha stops the distributed bot that rotates IPs, and the edge stops the volumetric flood.

Interview Q&A (SDE-2)

Q1. Compare fixed-window, sliding-window, token-bucket, and leaky-bucket rate limiting. When would you pick each?
Fixed window (count per calendar minute) is simplest but allows a 2x burst at the window boundary. Sliding window (log or weighted counter) smooths that boundary at higher memory/CPU cost. Token bucket lets you allow controlled bursts (tokens accumulate up to a cap, refilled at a steady rate) and is the common choice for APIs that want burst tolerance. Leaky bucket enforces a strictly smooth output rate (good for shaping traffic to a downstream), but drops or queues bursts. For most web APIs, token bucket or sliding window in Redis is the practical pick.
Q2. How do you implement rate limiting correctly across a horizontally scaled service?
You cannot use in-process counters because each replica sees only its own traffic, making the effective limit multiply by the number of instances. Use a shared store like Redis with an atomic operation: INCR + EXPIRE for fixed window, a sorted set (ZADD/ZREMRANGEBYSCORE) for sliding window, or a Lua script for token bucket to keep the read-modify-write atomic and avoid race conditions. The trade-off is a network hop per request and Redis becoming a hot dependency, so you make failures fail-open or fail-closed deliberately and consider local pre-checks to shed obvious abuse.
Q3. What should you key your rate limits on, and why is IP alone a bad choice?
For an abuse-sensitive endpoint you want multiple keys: per-IP, per-account/target (email, phone, userId), and sometimes per-API-key. IP alone is weak because attackers rotate through cloud IPs or botnets to stay under a per-IP cap, and conversely CGNAT/corporate NAT puts thousands of legitimate users behind one IP, so a tight per-IP limit causes false positives. Keying on the target resource (e.g. the email being reset) stops victim-focused bombing that an IP limit would miss.
Q4. Rate limiting and CAPTCHA both fight abuse. Why do you often need both?
Rate limiting caps request volume per key, but a distributed botnet can spread load across thousands of IPs so each stays under the limit while the aggregate is still an attack. A CAPTCHA raises the per-request cost for automation regardless of source distribution, catching bots that rate limits let through. The trade-off is user friction, so the usual pattern is: allow normal traffic freely, apply rate limits as a baseline, and trigger CAPTCHA only on risk signals or sensitive flows.
Q5. How does server-side CAPTCHA verification (e.g. Cloudflare Turnstile / reCAPTCHA) actually work, and what's the common mistake?
The widget runs in the browser and returns an opaque token to your frontend, which sends it with the form submission. Your backend must POST that token plus your secret key to the provider's siteverify endpoint and only proceed if it returns success. The classic mistake is trusting the presence of a token client-side without server verification, which is trivially bypassed; also remember tokens are single-use and expire quickly, so verify once and don't cache them.
Q6. Distinguish L3/L4 from L7 DDoS attacks and their mitigations.
L3/L4 attacks are volumetric/protocol floods (SYN floods, UDP/DNS/NTP amplification) that try to saturate bandwidth or connection tables; they're mitigated by anycast scrubbing networks and providers like Cloudflare or AWS Shield that absorb and filter traffic before it reaches you. L7 attacks are HTTP request floods that look like legitimate requests (e.g. hammering an expensive search endpoint), so they need application-aware defenses: WAF rules, edge rate limiting, JS/CAPTCHA challenges, and caching. A key operational point is hiding your origin IP behind the edge so attackers can't bypass mitigation by hitting the origin directly.
Q7. What's the difference between failing open and failing closed in a rate limiter, and how do you choose?
Fail-open means if the limiter's backing store (Redis) is unavailable, you allow the request through; fail-closed means you reject it. Fail-open preserves availability but removes your protection exactly when things are already stressed, which is dangerous for abuse-critical endpoints like login or OTP. Fail-closed protects the endpoint but can turn a Redis blip into a full outage. The usual answer is to fail-open on best-effort throttling of normal endpoints and fail-closed (or degrade to a stricter local limit) on security-critical ones, and to make Redis itself highly available.
Q8. How would you prevent OTP brute-force even with rate limiting in place?
Rate limiting the send endpoint doesn't protect the verify endpoint, so you separately cap verification attempts per code/account and lock or invalidate the OTP after a small number of failures. Use a sufficiently large code space and short TTL so guessing is impractical within the validity window, and never reveal whether the email exists (return a generic response) to avoid enumeration. Adding exponential backoff and a CAPTCHA after a few failures further raises the cost of automated guessing.
