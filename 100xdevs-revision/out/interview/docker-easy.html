Actionable Docker

Cohort 2.0 · 6 lessons · SDE-2 (3+ yrs) interview prep

Concept summary — all 6 steps

What Docker actually is
Docker packages an app plus its full dependency tree into an immutable image that runs identically anywhere, using OS-level containerization rather than a full guest OS. The interview angle is the container-vs-VM distinction: containers share the host kernel and isolate via Linux namespaces (PID, net, mount) and cgroups (resource limits), so they start in milliseconds and cost far less overhead than a hypervisor VM.
Installing Docker
On Linux you install Docker Engine directly (the dockerd daemon + CLI run natively on the host kernel); on macOS/Windows, Docker Desktop transparently spins up a lightweight Linux VM because containers need a Linux kernel. Know the layered runtime underneath: CLI talks to dockerd, which delegates to containerd, which uses runc to actually create the container via kernel primitives.
What we use Docker for
The killer use cases are reproducible dev environments and dependency isolation, killing 'works on my machine' by shipping the runtime, libs, and config together. As an SDE-2 you mainly use it to run backing services (Postgres, Redis, Kafka) locally without polluting the host, to produce a single deployable artifact that CI builds once and ships to every environment, and to keep conflicting language/tool versions isolated per project.
Where packages/images come from
Images are pulled from registries: Docker Hub is the default public one, and teams use private registries like AWS ECR, GHCR, GCR, or ACR. An image is addressed as registry/repository:tag (e.g. postgres:16-alpine); tags are mutable pointers while a sha256 digest is immutable, so pinning by digest (or at least a specific tag) matters for reproducibility and supply-chain safety. Prefer official/verified images and slim variants like alpine or distroless to shrink attack surface.
Common commands to know
Core lifecycle: docker pull/build to get an image, docker run (with -d detached, -it interactive, -p host:container for ports, -v for volumes, -e for env, --name), then ps, logs, exec -it <c> sh to shell in, stop/rm, and image/volume prune to reclaim space. Interview nuance: run creates a new container from an image every time, whereas start restarts an existing stopped one and exec runs a process inside an already-running container.
Common packages/images
Day-to-day images an SDE-2 pulls are the backing services and base runtimes: postgres, mysql, redis, mongo, rabbitmq/kafka, nginx, and language bases like node, python, golang, plus alpine as a minimal base. Knowing these lets you stand up a full local stack in seconds and pick lean, well-maintained official images rather than random third-party ones.

Real-world example

Standing up a local dev stack for a payments API

You join a service that needs Postgres 16 and Redis 7. Instead of installing them on your laptop, you run `docker run -d --name pg -e POSTGRES_PASSWORD=dev -p 5432:5432 -v pgdata:/var/lib/postgresql/data postgres:16-alpine` and a similar one for redis:7-alpine. Docker pulls both images from Docker Hub, maps the container ports to localhost so your app connects at 127.0.0.1:5432, and the named volume persists your seed data across `docker rm`/`docker run` cycles. When migrations fail, you `docker logs pg` to see the error and `docker exec -it pg psql -U postgres` to inspect tables live. A teammate hits a 'port already allocated' error because they still have a native Postgres on 5432; the fix is remapping `-p 5433:5432` or stopping the host service. On CI, the same images back the integration tests, so the environment is byte-for-byte identical to what everyone runs locally, and the app itself is built once into an image and promoted unchanged from staging to prod.

Interview Q&A (SDE-2)

Q1. What is the concrete difference between a container and a VM?
A VM virtualizes hardware and runs a full guest OS with its own kernel on top of a hypervisor, so it is heavy (GBs, seconds to boot). A container shares the host's kernel and is just an isolated process, achieving isolation through Linux namespaces (what the process can see) and cgroups (what resources it can use). The trade-off: containers are lighter and faster but provide weaker isolation than a hardware-virtualized VM, which is why untrusted multi-tenant workloads sometimes still use VMs or micro-VMs like Firecracker.
Q2. What is the difference between an image and a container?
An image is an immutable, read-only template made of stacked filesystem layers plus metadata (entrypoint, env, ports). A container is a running (or stopped) instance of an image with a thin writable layer on top, so all changes made at runtime live in that layer and are lost when the container is removed unless you use a volume. One image can spawn many independent containers.
Q3. Walk through Docker's architecture from CLI to running container.
The docker CLI is a client that sends REST calls over a socket to the dockerd daemon. dockerd handles image management and networking, then hands container lifecycle to containerd, which in turn invokes runc to create the container using kernel namespaces and cgroups. Images are pushed to and pulled from a registry. Understanding this explains why a container is 'just a process' the daemon supervises, not a mini-machine.
Q4. How does data persistence work, and why use volumes instead of writing to the container filesystem?
The container's writable layer is ephemeral and disappears when the container is removed, and it also incurs copy-on-write overhead. Named volumes (`-v pgdata:/var/lib/postgresql/data`) are managed by Docker outside the container lifecycle, so data survives recreation and gets better I/O. Bind mounts map a host directory into the container and are ideal for live-reloading source code in dev, but tie you to the host path layout.
Q5. What does `-p 8080:80` do, and why isn't a container reachable without it?
It publishes the container's port 80 to the host's port 8080, so traffic to host:8080 is NATed into the container. By default a container's network namespace is isolated, so its ports are only reachable by other containers on the same Docker network, not from the host. The left side is always the host port and the right is the container port; forgetting the mapping is the most common 'my service is up but I can't hit it' bug.
Q6. Explain image layers and build caching, and how they affect Dockerfile ordering.
Each Dockerfile instruction creates a layer, and layers are cached by their content and the layers above them. If an early layer changes, every subsequent layer's cache is invalidated and rebuilt. So you order from least- to most-frequently changing: copy dependency manifests and install deps before copying application source, so a code edit doesn't reinstall all dependencies. Layers are also shared across images, saving disk and pull time.
Q7. Tags versus digests, and why does it matter for reproducibility?
A tag like `postgres:16` is a mutable human-friendly pointer that can be re-pushed to point at a different image over time, whereas a `sha256:...` digest is a content-addressed, immutable reference. Relying on `:latest` or a floating tag means your build can silently change or break between runs. For reproducible and secure builds, pin to a specific tag and ideally the digest, so you get the exact same bytes every pull.
Q8. What's the difference between `docker run`, `docker start`, and `docker exec`?
`docker run` creates and starts a brand-new container from an image every time it is invoked, so repeated runs leave you with multiple containers. `docker start` restarts an existing stopped container, preserving its writable layer and identity. `docker exec` runs an additional process inside an already-running container, which is how you shell in for debugging (`docker exec -it <c> sh`) without disturbing the main process.
