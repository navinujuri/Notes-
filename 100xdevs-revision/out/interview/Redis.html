Redis (Pub subs and Messaging queues)

Cohort 2.0 · 9 lessons · SDE-2 (3+ yrs) interview prep

Concept summary — all 9 steps

Scope & mental model
Sets up the two distinct Redis capabilities covered: durable-ish list-based queues (producer/worker fan-in) and fire-and-forget Pub/Sub (fan-out messaging). Interview angle: know which primitive fits which problem — queues for work distribution, Pub/Sub for real-time notifications — they are not interchangeable.
What Redis is
Redis is an in-memory, single-threaded key-value data store that also ships higher-level structures (lists, sets, sorted sets, hashes, streams) and messaging primitives. It matters because sub-millisecond latency and the single-threaded command loop mean every command is atomic, which is why it doubles as a cache, queue, and coordination layer.
Running Redis locally
Spin up Redis via Docker (`docker run -p 6379:6379 redis`) or a native install, and inspect it with `redis-cli`. The interview-relevant detail is the default port 6379 and that redis-cli lets you verify queue/pubsub behavior manually before wiring code.
Redis as a database / KV store
Basic `SET`/`GET`, exp_iry with `EXPIRE`/`SET ... EX`, and the caveat that data lives in RAM. It matters because you must reason about persistence (RDB snapshots vs AOF log) and eviction policies — Redis is durable only if configured for it, otherwise a restart loses data.
Redis as a queue
Use a list as a FIFO queue: producers `LPUSH` jobs, workers `RPOP` (or the blocking `BRPOP`) to pull them. This matters for offloading slow work (e.g. code execution, email sending) to background workers; the blocking pop avoids busy-polling and lets multiple workers compete for jobs with each job delivered to exactly one worker.
Talking to Redis from Node.js
Connect with a client (`redis` or `ioredis`), `await client.connect()`, then call the same commands (`lPush`, `brPop`, `set`, `get`) as promises. Key interview point: a blocking `BRPOP` needs its own dedicated connection, since a connection blocked on a pop cannot issue other commands concurrently.
Pub/Sub semantics
Publishers `PUBLISH channel message`; subscribers `SUBSCRIBE channel` and receive pushed messages. Critically it is fire-and-forget with no persistence or acknowledgement — a message is delivered only to clients connected at publish time, and offline subscribers miss it entirely. That trade-off (broadcast fan-out vs no durability) is the single most common interview question here.
Pub/Sub in Node.js
A subscriber connection must be duplicated/dedicated (`client.duplicate()`) because once subscribed it enters subscriber mode and cannot run normal commands. You then register a callback per channel to react to incoming messages — the pattern for pushing real-time updates back to a WebSocket/SSE layer.
Assignment / applied build
Consolidates the pieces into an end-to-end system: a primary server enqueues work, a worker pool consumes the queue, processes jobs, and publishes results back over Pub/Sub. The exercise cements when to reach for a queue (work distribution, one consumer per job) versus Pub/Sub (broadcast, many consumers).

Real-world example

A LeetCode-style code-submission pipeline

A user submits code on your judge platform. The primary API server shouldn't run untrusted code inline (slow, unsafe), so it `LPUSH`es a submission job {submissionId, language, code} onto a Redis list `problems` and immediately returns 202 Accepted. A pool of stateless worker processes each block on `BRPOP problems`, so exactly one worker grabs each job, runs it in a sandbox, and computes the verdict. When done, the worker `PUBLISH`es the result to a channel like `result:<submissionId>`. The WebSocket/SSE gateway that the browser is connected to has `SUBSCRIBE`d to that channel and pushes the verdict back to the user in real time. This cleanly separates the two primitives: the list is a durable-ish work queue giving load-balanced, exactly-one-consumer delivery, while Pub/Sub gives low-latency fan-out to whichever gateway currently holds the user's live connection. An SDE-2 debugging this must know that if the gateway subscribes after the worker publishes, the result is lost — which is why the verdict is also persisted (e.g. `SET submission:<id> <verdict>`) so a reconnecting client can fetch it, and for stronger delivery guarantees you'd graduate the queue to Redis Streams with consumer groups.

Interview Q&A (SDE-2)

Q1. When would you use Redis Pub/Sub versus a Redis list-based queue?
Use a queue (LPUSH/BRPOP) for work distribution where each job must be processed by exactly one consumer and shouldn't be lost if no worker is momentarily free — the job waits in the list. Use Pub/Sub for broadcast/fan-out where every currently-connected subscriber should get the message and staleness is acceptable, e.g. real-time notifications or cache-invalidation signals. The decisive difference is delivery semantics: queues buffer and deliver once to one consumer; Pub/Sub is fire-and-forget to all connected subscribers with no buffering.
Q2. What are the delivery guarantees of Redis Pub/Sub, and what breaks because of them?
Redis Pub/Sub is at-most-once with no persistence: a published message is delivered only to clients subscribed at that instant, and anyone offline or reconnecting misses it — there's no replay. It also has no acknowledgement, so a slow subscriber can have messages dropped once its output buffer overflows. This makes plain Pub/Sub unsuitable as a reliable message bus; for durability and replay you use Redis Streams with consumer groups (XADD/XREADGROUP/XACK) or a real broker like Kafka/RabbitMQ.
Q3. Why is BRPOP preferred over a polling loop of RPOP, and what's the connection caveat?
BRPOP blocks server-side until an item is available (or a timeout), so workers get jobs with near-zero latency and without hammering Redis with empty RPOP calls that waste CPU and network. The caveat is that a blocked connection can't do anything else — so in Node you must give the blocking consumer its own dedicated connection and keep your normal request-serving connection separate. Multiple workers can BRPOP the same key and Redis hands each pushed item to exactly one of them.
Q4. Redis is single-threaded — how does that affect using it as a queue and why isn't it a bottleneck?
The single command-execution thread makes every operation atomic, so LPUSH/BRPOP need no external locking and there are no partial-read race conditions on the list. It's rarely a bottleneck because commands are in-memory and O(1), so a single instance handles well over 100k ops/sec; the real limits are network round-trips and large payloads, mitigated with pipelining and keeping values small. If you do saturate one core, you scale out with sharding/Redis Cluster rather than threads.
Q5. How do you make Redis-based messaging survive a restart or crash?
Plain Pub/Sub can't — messages are never persisted. For queues, list contents are persisted according to your durability config: RDB gives point-in-time snapshots (fast, but you can lose the seconds since the last snapshot) while AOF logs every write and replays it on restart (more durable, larger/slower). For messaging that needs replay and consumer offsets you use Redis Streams, which retain entries and track per-consumer-group acknowledgements, so a crashed worker's un-ACKed messages can be reclaimed via XPENDING/XCLAIM.
Q6. A subscriber connection can't run normal commands — why, and how do you handle it in Node?
Once a connection issues SUBSCRIBE it enters 'subscriber mode' and the protocol only permits (un)subscribe commands plus receiving pushed messages, so you can't GET/SET on it. In Node with node-redis or ioredis you call `client.duplicate()` (or open a second client) to get a dedicated subscriber connection and keep the original for regular commands. You then attach a message callback per channel; forgetting the separate connection is a classic bug that surfaces as 'commands not allowed in subscriber mode' errors.
Q7. How would you scale Pub/Sub across multiple backend instances behind a load balancer?
The problem is that a user's WebSocket lives on only one app instance, but the worker that finishes their job may publish from anywhere — so every app instance subscribes to Redis, and Redis fans the published message out to all of them; the instance holding that user's socket forwards it, the rest ignore it. This is exactly the model Socket.IO's Redis adapter uses. Note that in Redis Cluster, standard PUBLISH only propagates within shard boundaries unless you use sharded Pub/Sub (SPUBLISH/SSUBSCRIBE) or route pub/sub through a single logical channel, which is a common scaling gotcha.
Q8. What are the risks of using a Redis list as your only queue, and when do you outgrow it?
A list gives you FIFO and one-consumer-per-job, but no built-in acknowledgement, retry, dead-letter, or visibility timeout — if a worker BRPOPs a job and then crashes mid-processing, that job is silently gone. You also can't inspect in-flight work or guarantee at-least-once without extra bookkeeping (e.g. RPOPLPUSH into a processing list, then removing on completion). You outgrow it when you need reliable redelivery, ordering guarantees per key, consumer groups, or high throughput with replay — at which point you move to Redis Streams or a dedicated broker like RabbitMQ/Kafka/SQS.
