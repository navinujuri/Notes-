Prometheus and Grafana

Cohort 2.0 · 16 lessons · SDE-2 (3+ yrs) interview prep

Concept summary — all 16 steps

Revise monitoring
The three observability pillars: metrics (cheap, aggregatable numeric time series ideal for alerting/dashboards), logs (high-cardinality event records), and traces (a request's path across services). Interview angle: know RED (Rate, Errors, Duration) for request-driven services and USE (Utilization, Saturation, Errors) for resources, and why metrics are the backbone of alerting while logs/traces are for drill-down.
Prometheus overview
An open-source, pull-based monitoring system with a multi-dimensional data model (metric name + key/value labels) and its own local TSDB. It scrapes HTTP /metrics endpoints on an interval instead of receiving pushes. Interview angle: pull vs push trade-offs — pull gives target liveness (up==0) and centralized service discovery for free but struggles with short-lived batch jobs (solved via Pushgateway).
Adding raw metrics
Hand-exposing a /metrics endpoint in the Prometheus text exposition format (e.g. http_requests_total 42) before pulling in a client library. It demonstrates the format is just plain text over HTTP — name, optional labels, value, plus # HELP and # TYPE comments. Matters for understanding what client libs generate under the hood and for debugging broken exporters.
Add Prometheus client
Wiring in the official client library (prom-client for Node) which owns a registry, ships default process/runtime metrics, and serializes everything to the exposition format on demand. Why it matters: hand-rolling counters is error-prone under concurrency, and the registry prevents duplicate-metric registration bugs.
Counters
A monotonically increasing cumulative value that only goes up or resets to 0 on restart — request totals, errors, bytes processed. You never read the raw number; you apply rate()/increase() in PromQL to derive a per-second rate. Interview angle: counter + rate() is resilient to restarts and scrape gaps because rate() detects and corrects for resets.
Better structure
Refactoring instrumentation into a dedicated module/middleware so metrics are defined and registered exactly once, not recreated per request (which throws on re-registration). Centralizing also enforces consistent label sets and makes cardinality easier to reason about — scattered ad-hoc metrics are a common source of label drift.
Gauge
A value that can move both up and down — in-flight requests, queue depth, current memory, active connections. Unlike a counter, you read the instantaneous value directly (often via inc()/dec()). Interview angle: pick a gauge for point-in-time state and a counter for accumulating events; using the wrong type yields meaningless graphs.
Histograms
Samples observations (e.g. latency) into predefined cumulative buckets, exposing _bucket, _sum, and _count series, enabling server-side quantiles via histogram_quantile(). Interview angle: histogram vs summary — histogram bucket counts are additive so you can aggregate p99 across replicas, whereas summaries compute quantiles client-side and cannot be aggregated. Bucket boundaries should straddle your SLO.
Final instrumented code
The complete service: a registry, a counter for request totals, a gauge for in-flight requests, and a latency histogram, all served from a single /metrics handler wrapped in middleware. The takeaway is a clean, low-cardinality instrumentation layer that covers the RED signals with minimal per-request overhead.
Starting Prometheus
Running the server against a prometheus.yml whose scrape_configs define job_name, targets, and scrape_interval; Prometheus then discovers and scrapes targets into its TSDB. This is also where relabeling and service discovery live. Matters because misconfigured targets or intervals silently produce no data — the number one 'why is my graph empty' cause.
PromQL queries
The query language over instant vectors and range vectors, with label matchers and functions like rate(), sum by (label), and histogram_quantile(). Interview angle: rate() requires a range-vector window (e.g. [5m]), aggregation operators group across labels, and you should be able to write an error-rate or p99-latency query on the spot.
Graphs in Prometheus
The built-in expression browser/graph tab for ad-hoc PromQL exploration and debugging queries. It is deliberately for investigation, not production dashboards — no persistence, sharing, variables, or rich panels — which is precisely the gap Grafana fills.
Grafana
A visualization/dashboarding platform that queries data sources (Prometheus, Loki, etc.) and renders shareable, templated dashboards with variables and its own alerting. It decouples visualization from storage. Interview angle: Prometheus stores and queries the data; Grafana only reads and visualizes it — distinct layers of the stack.
Grafana in docker-compose
Adding a grafana service next to prometheus with ports, a named volume for persistence, and provisioning files for data sources/dashboards. Matters for a reproducible dev stack and for understanding config-as-code provisioning versus fragile manual UI setup that is lost on container recreation.
Add Prometheus as a data source
Pointing Grafana's Prometheus data source at the server URL and then running PromQL from panels. Key gotcha: inside a container, use the compose service name (http://prometheus:9090), not localhost, which resolves to the Grafana container itself. Ideally provision the source as code rather than clicking through the UI.
Alerting
Defining alert rules on metric expressions that fire only after the condition holds for a duration (for:), routed via Alertmanager (Prometheus) or Grafana alerting to receivers like Slack/PagerDuty. Interview angle: alert on user-facing symptoms (error rate, latency/SLO burn) not causes (high CPU), use for: to prevent flapping, and know Alertmanager's grouping, silencing, and inhibition.

Real-world example

Instrumenting a checkout API to catch latency and error regressions

You own a Node checkout service. After a Black Friday incident where p99 latency quietly spiked and nobody noticed until customers complained, you add prom-client instrumentation: a counter http_requests_total{route,method,status}, a gauge http_in_flight_requests, and a histogram http_request_duration_seconds with buckets tuned around your 300ms SLO (crucially normalizing routes to /order/:id so cardinality stays bounded). You expose /metrics, add a Prometheus service in docker-compose scraping it every 15s, and stand up Grafana pointed at Prometheus by its compose hostname (http://prometheus:9090, not localhost). In Grafana you build a RED dashboard: request rate via sum(rate(http_requests_total[5m])), an error ratio, and fleet-wide p99 via histogram_quantile(0.99, sum(rate(http_request_duration_seconds_bucket[5m])) by (le)). Finally you add an alert that fires when the 5-minute error rate exceeds 1% for 10 minutes, routed to Slack via Alertmanager — so the next regression pages you before it pages your customers.

Interview Q&A (SDE-2)

Q1. Why is Prometheus pull-based, and when do you still need push?
Prometheus scrapes targets over HTTP on an interval. Pulling gives target liveness for free (a failed scrape sets up==0), centralizes scrape config and service discovery, and stops clients from overwhelming the server. The weakness is short-lived or batch jobs that exit before being scraped — for those you push to a Pushgateway, which Prometheus then scrapes. Fully push-based systems like StatsD suit ephemeral jobs but lose that built-in health signal.
Q2. How do you choose between a counter, gauge, and histogram?
Use a counter for monotonically increasing event counts (requests, errors) — always consumed via rate(). Use a gauge for values that move both directions (in-flight requests, queue depth, memory). Use a histogram for distributions where you want quantiles or SLO buckets (latency, payload size). Picking a counter where you need current state, or a gauge where you need a rate, produces graphs that are numerically meaningless.
Q3. Why do you apply rate() to a counter instead of reading its value?
A raw counter value is meaningless on its own and resets to zero whenever the process restarts. rate() and increase() compute the per-second change over a range window and automatically treat any decrease as a reset. That makes derived rates robust to restarts and missed scrapes, and lets you sum comparable rates across instances.
Q4. Histogram vs summary — what's the trade-off?
Histograms bucket observations client-side with predefined boundaries and expose cumulative bucket counters; quantiles are computed server-side with histogram_quantile(), and because bucket counts are additive you can aggregate across replicas. Summaries compute quantiles on the client — exact for that instance but impossible to aggregate and more expensive. In distributed systems you almost always want histograms; the price is that quantile accuracy depends on how well your buckets straddle the region of interest.
Q5. What is label cardinality and why is it dangerous?
Cardinality is the number of unique label-value combinations, and each one is a separate time series held in memory and on disk. Putting unbounded values — user IDs, request IDs, raw URLs with query params — into labels causes a cardinality explosion that can OOM Prometheus and slow every query. Keep labels low-cardinality and bounded, e.g. normalize /users/123 to /users/:id.
Q6. How does histogram_quantile compute a fleet-wide p99?
Histograms expose cumulative _bucket{le="..."} counters. To get a p99 across instances you first combine them with sum(rate(..._bucket[5m])) by (le), then wrap that in histogram_quantile(0.99, ...), which linearly interpolates within the bucket where the 99th-percentile rank falls. Accuracy is bounded by the bucket width around that quantile, so you tune bucket boundaries near your SLO target.
Q7. In docker-compose, why does pointing Grafana at http://localhost:9090 fail?
Inside a container, localhost refers to that container's own network namespace, not the host or the Prometheus container, so Grafana connects to itself and fails. Compose services share a network and are addressable by service name, so the data source URL should be http://prometheus:9090. Persist Grafana with a named volume and provision the data source as code so it survives container recreation.
Q8. What makes a good alerting strategy in Prometheus/Grafana?
Alert on user-facing symptoms — error rate, latency/SLO burn, availability — rather than causes like high CPU, which cuts noise and false pages. Use the for: clause so a condition must hold for a duration before firing, preventing flapping on transient spikes. Alertmanager then handles routing, grouping related alerts into one notification, silencing during maintenance, and inhibition (suppressing downstream alerts when a parent alert is already firing).
