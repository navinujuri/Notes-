Kubernetes Part 1

Cohort 2.0 · 20 lessons · SDE-2 (3+ yrs) interview prep

Concept summary — all 20 steps

What is Kubernetes
Kubernetes (K8s) is an open-source container orchestrator that manages deployment, scaling, networking, and self-healing of containerized workloads across a cluster of machines. It gives you a declarative control plane: you describe desired state, and controllers reconcile actual state toward it. Interview angle: it's an orchestrator, not a container runtime (it drives containerd/CRI-O), and its core value is the reconciliation loop.
Before Kubernetes
Pre-K8s ops meant manually provisioning VMs, SSHing to deploy, hand-rolled bash/Ansible for config, and bespoke scripts for scaling, health checks, and failover. This was error-prone, hard to reproduce, and gave no standard way to bin-pack containers or recover from node failure. Interview angle: understand the pain points (snowflake servers, no self-healing, poor resource utilization) that K8s exists to solve.
After Kubernetes
With K8s you declare workloads as manifests, and the cluster handles scheduling, rollouts, restarts of dead containers, horizontal scaling, and service discovery automatically. Infrastructure becomes declarative and version-controllable (GitOps). Interview angle: the shift is from imperative 'how' to declarative 'what' plus automatic drift correction.
Creating a K8s cluster
A cluster = a control plane (API server, etcd, scheduler, controller-manager) plus worker nodes running kubelet + a container runtime. For learning you spin one up with kind, minikube, or k3s; in production you use a managed offering (EKS/GKE/AKS) or kubeadm. Interview angle: know the control-plane vs node split and that etcd is the single source of truth for cluster state.
Kubernetes API
Everything in K8s is a REST resource exposed by the kube-apiserver; all reads/writes (kubectl, controllers, kubelets) go through it and persist to etcd. Resources are versioned and grouped (e.g. apps/v1, v1) and extensible via CRDs. Interview angle: the API server is the single front door and enforces authn/authz, admission control, and validation before anything is stored.
kubectl
kubectl is the CLI that translates commands into API-server calls using your kubeconfig context and credentials. Core verbs: apply, get, describe, logs, exec, delete, port-forward. Interview angle: prefer `kubectl apply -f` (declarative) over `kubectl create/run` (imperative) for reproducibility, and know `describe`/`logs` are your first debugging stops.
Creating a Pod
A Pod is the smallest deployable unit: one or more tightly-coupled containers sharing a network namespace (same IP/localhost) and volumes. You rarely create bare Pods directly because nothing reschedules them if the node dies. Interview angle: Pods are ephemeral and mortal; a bare Pod has no self-healing, which is why you use controllers instead.
Stop the Pod
Deleting a Pod triggers graceful termination: it's sent SIGTERM, given a terminationGracePeriodSeconds window to clean up, then SIGKILL. A standalone Pod stays gone; a Pod owned by a controller gets recreated. Interview angle: this difference (bare Pod vs controller-owned) demonstrates why self-healing needs a ReplicaSet/Deployment above the Pod.
Kubernetes manifest
A manifest is a YAML/JSON declaration of desired state with four key top-level fields: apiVersion, kind, metadata, and spec. Applying it lets the relevant controller reconcile reality to match. Interview angle: manifests are the declarative, Git-versionable contract; `spec` is your intent and `status` (managed by K8s) is observed reality.
Checkpoint
A consolidation point to confirm the mental model so far: cluster + API server + kubectl + Pods + manifests. Interview angle: be able to trace a single `kubectl apply` end-to-end from CLI through the API server to etcd and onto a kubelet scheduling a Pod.
Deployment
A Deployment is a higher-level controller that manages ReplicaSets to provide declarative rollouts, rollbacks, and versioned updates of stateless Pods. Changing the Pod template creates a new ReplicaSet and rolls traffic over gradually (rolling update). Interview angle: this is the object you deploy stateless apps with; it adds rollout strategy and revision history on top of a ReplicaSet.
ReplicaSet
A ReplicaSet ensures a specified number of identical Pod replicas are running, recreating any that die, using a label selector to track its Pods. Interview angle: you almost never manage ReplicaSets directly - Deployments own them - but understanding it explains where self-healing and replica-count reconciliation actually happen.
Series of events (ReplicaSet)
When a ReplicaSet is created or a Pod dies, the ReplicaSet controller compares desired vs actual replica count, creates missing Pods via the API server, and the scheduler binds them to nodes where kubelets start them. Interview angle: this is the reconciliation loop in action - watch, diff, act - the pattern behind every K8s controller.
Create a ReplicaSet
Written as kind: ReplicaSet with replicas, a selector, and a Pod template; applying it makes the controller drive the cluster to that replica count. Interview angle: know that the selector must match the template labels or the API rejects it, and that scaling is just editing `replicas` and re-applying.
Create a Deployment
kind: Deployment wraps the same replicas/selector/template plus a rollout strategy (RollingUpdate with maxSurge/maxUnavailable, or Recreate). Applying an updated image triggers a controlled rollout you can monitor with `kubectl rollout status` and reverse with `kubectl rollout undo`. Interview angle: be ready to explain maxSurge/maxUnavailable trade-offs for zero-downtime deploys.
Why do you need Deployment?
Because bare Pods and even ReplicaSets don't give you versioned, zero-downtime rollouts or one-command rollback. Deployments manage successive ReplicaSets so you can update the image, pause/resume, and revert to a known-good revision safely. Interview angle: the answer is rollout orchestration + revision history + rollback, not just replica management.
How to expose the app?
Pods get ephemeral IPs that change on every reschedule, so you can't address them directly; you need a stable endpoint in front of the replica set. Interview angle: this motivates Services - the problem is that Pod IPs are non-durable and you need a stable virtual IP + load balancing across healthy Pods.
Services
A Service is a stable virtual IP and DNS name that load-balances across a dynamic set of Pods matched by a label selector, backed by Endpoints/EndpointSlices. Types: ClusterIP (internal, default), NodePort (fixed port on every node), LoadBalancer (external). Interview angle: know how a Service decouples clients from ephemeral Pod IPs and that kube-proxy/iptables (or IPVS) implements the routing.
LoadBalancer service
type: LoadBalancer provisions an external cloud load balancer that routes internet traffic to the Service, which then fans out to Pods; it builds on NodePort/ClusterIP underneath. Interview angle: it's one LB (and often one cloud bill) per Service, which is why an Ingress controller is usually preferred to expose many HTTP services behind a single external LB.
Series of events (Service/LoadBalancer)
Creating a LoadBalancer Service makes the cloud-controller-manager provision an external LB, the Service controller populates EndpointSlices from selector-matched healthy Pods, and kube-proxy programs node routing rules so traffic reaches Pods. Interview angle: trace external request -> cloud LB -> NodePort -> kube-proxy rules -> Pod, and note Endpoints update automatically as Pods come and go.

Real-world example

Deploying and exposing a stateless REST API on EKS

You're an SDE-2 shipping a stateless Node.js order-service to a managed cluster (EKS). You write a Deployment manifest (apiVersion: apps/v1, kind: Deployment) with replicas: 3, a label selector app=order-service, a Pod template pointing at your container image, plus liveness/readiness probes and a RollingUpdate strategy (maxSurge: 1, maxUnavailable: 0) for zero-downtime deploys. `kubectl apply -f deployment.yaml` hits the API server, which persists to etcd; the Deployment controller creates a ReplicaSet, the scheduler places 3 Pods, and kubelets pull the image and start containers. To make it reachable you add a Service (initially ClusterIP for internal callers, then a LoadBalancer for external traffic), whose selector matches app=order-service so the cloud-controller-manager provisions an ELB and EndpointSlices track the 3 healthy Pods. When you push v2, you bump the image tag and re-apply; the Deployment spins up a new ReplicaSet and shifts Pods over one at a time while readiness probes gate traffic. A bad v2 is a one-liner to fix: `kubectl rollout undo`. When a node dies at 2am, the ReplicaSet notices the missing replica and reschedules it, and the Service's endpoints update automatically - no pager, no manual intervention.

Interview Q&A (SDE-2)

Q1. What's the difference between a Pod, a ReplicaSet, and a Deployment, and which should you deploy with?
A Pod is the smallest unit - one or more co-located containers sharing network/storage - but it has no self-healing on its own. A ReplicaSet sits above Pods and guarantees a desired replica count, recreating dead Pods via its label selector. A Deployment sits above ReplicaSets and adds versioned rollouts, rollback, and rollout strategy. You deploy stateless apps with a Deployment; it owns the ReplicaSet for you, so you rarely touch Pods or ReplicaSets directly.
Q2. Walk through what happens end-to-end when you run `kubectl apply -f deployment.yaml`.
kubectl authenticates via kubeconfig and sends the manifest to the kube-apiserver, which runs authn/authz, admission controllers, and validation, then persists the object to etcd. The Deployment controller sees the new/changed object, creates or updates a ReplicaSet; the ReplicaSet controller creates Pod objects to hit the desired count. The scheduler assigns each unbound Pod to a node based on resources/affinity, and the kubelet on that node pulls the image via the container runtime and starts the containers, reporting status back to the API server.
Q3. Why can't clients just talk to Pod IPs directly, and how does a Service solve it?
Pod IPs are ephemeral - they change whenever a Pod is rescheduled, scaled, or replaced during a rollout - so hardcoding them breaks constantly. A Service provides a stable virtual IP and DNS name plus load balancing across a dynamic set of Pods selected by labels, backed by EndpointSlices that update automatically as Pods come and go. kube-proxy (via iptables or IPVS) programs each node to route Service traffic to a healthy backing Pod, decoupling clients from Pod churn.
Q4. Explain a rolling update and how maxSurge/maxUnavailable control it.
On a Pod template change, the Deployment creates a new ReplicaSet and gradually scales it up while scaling the old one down, so both versions coexist briefly. maxSurge is how many extra Pods above desired count can be created during the rollout (throughput of the roll), and maxUnavailable is how many below desired can be down at once (availability floor). For zero-downtime you typically set maxUnavailable: 0 and maxSurge: 1+; readiness probes gate when new Pods receive traffic, and `kubectl rollout undo` reverts to the previous ReplicaSet.
Q5. Compare ClusterIP, NodePort, and LoadBalancer Service types and when you'd use each.
ClusterIP (default) exposes the Service only inside the cluster on a virtual IP - use it for internal service-to-service traffic. NodePort opens a fixed port on every node's IP and forwards to the Service - useful for on-prem or as a building block, but awkward for public traffic. LoadBalancer provisions an external cloud load balancer in front of NodePort/ClusterIP - use for exposing a service externally, but since it's typically one LB per Service, an Ingress controller is usually preferred to route many HTTP services behind a single external LB.
Q6. What is the reconciliation loop and why is it central to Kubernetes?
Controllers continuously watch the API server for the desired state (spec) and compare it against observed state (status), then take actions to close the gap - create, delete, or update resources - and repeat forever. This makes K8s declarative and self-healing: if a Pod dies or a node fails, the loop notices the drift and corrects it without human intervention. It's the shared pattern behind Deployments, ReplicaSets, Services, and every CRD-backed operator.
Q7. What role does etcd play and why does it matter operationally?
etcd is the cluster's single source of truth - a consistent, distributed key-value store holding all API objects and their state, and the API server is the only component that talks to it directly. Its consistency (Raft) is what lets controllers reliably reconcile. Operationally it's critical: losing etcd means losing cluster state, so it needs regular backups, an odd-numbered quorum of members, and low-latency disk; etcd health issues surface as a slow or unresponsive control plane.
Q8. How does a controller-owned Pod behave differently from a bare Pod when it's deleted or its node fails?
A bare Pod is mortal: delete it or lose its node and it's gone permanently, because nothing owns it. A Pod owned by a ReplicaSet/Deployment is tracked by that controller's replica count, so on deletion or node failure the controller recreates it (the scheduler places the replacement on a healthy node). This ownership - expressed via ownerReferences and the label selector - is exactly why you use controllers instead of bare Pods for anything you want to survive failures.
