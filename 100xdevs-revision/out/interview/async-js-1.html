Asynchronous Javascript, Callbacks and more

Cohort 2.0 · 9 lessons · SDE-2 (3+ yrs) interview prep

Concept summary — all 9 steps

Introduction
Frames why async matters in JS: it's single-threaded with one call stack, so any long-running work must be pushed off the main thread or it freezes everything. The interview angle is understanding that 'async' in JS is about non-blocking scheduling, not multi-threading or parallelism.
Normal (synchronous) functions
A plain function call pushes a frame onto the call stack, runs to completion, and returns before the next line executes. This is the mental baseline: JS executes statements top-to-bottom on a single stack, and nothing else runs until the current function returns.
Synchronous code
Synchronous code blocks the single thread — each operation must finish before the next starts, so a slow line stalls the entire program (including UI rendering in the browser). Interviewers probe whether you can identify blocking calls (e.g. fs.readFileSync, a tight loop) and explain their cost.
I/O heavy operations
I/O (disk reads, network calls, DB queries, timers) is slow relative to CPU and is handled outside the JS engine by the OS/libuv, not by the JS thread itself. Doing them synchronously wastes the thread waiting; doing them async lets the thread keep working while the OS completes the I/O.
I/O-bound vs CPU-bound tasks
I/O-bound tasks spend most time waiting on external resources (network/disk) and benefit from async concurrency on one thread; CPU-bound tasks (hashing, image resize, big loops) saturate the core and async does NOT help — they still block the event loop. The trade-off: offload CPU work to worker threads / child processes / a queue, keep the main thread for I/O orchestration.
Doing I/O in the real world
In practice you use async APIs — fs.promises, fetch/axios, DB drivers — so the request thread isn't held while waiting, which is what lets a single Node process handle thousands of concurrent connections. The key insight: a Node server's throughput comes from not blocking on I/O, not from many threads.
Functions as arguments (first-class functions)
Functions are values in JS: they can be passed as arguments, returned, and stored — this is the mechanism that makes callbacks possible. Know the distinction between passing a reference (setTimeout(fn, 100)) and invoking it (setTimeout(fn(), 100)), a classic bug source.
Asynchronous code & callbacks
A callback is a function you hand to an async API to be invoked later when the operation completes; the outer call returns immediately and the callback runs on a future tick. Interview depth: Node's error-first callback convention (err, data), and 'callback hell' / inversion of control, which motivated Promises and async/await.
JS architecture for async (event loop)
The runtime = call stack + Web/libuv APIs + callback (macrotask) queue + microtask queue, coordinated by the event loop, which only pushes a queued callback when the stack is empty. This is the single most-tested concept: microtasks (Promise .then, queueMicrotask) drain fully before the next macrotask (setTimeout, I/O), which explains ordering surprises.

Real-world example

An image-upload endpoint on a Node API server

You build a POST /upload endpoint on an Express/Node service. The handler must (1) read the uploaded file, (2) call an S3 API to store it, (3) write a row to Postgres, then (4) return 200. Steps 1-3 are I/O-bound, so you use async APIs (fs.promises, the AWS SDK, a pg driver) — while each awaits, the single Node thread is free to serve other users' requests, giving high concurrency from one process. Then a product requirement lands: generate a thumbnail. That resize is CPU-bound, and if you run it inline it pins the event loop and every other in-flight request stalls until it finishes. The fix is architectural: keep the I/O on the main thread but offload the CPU work — push a job to a queue (BullMQ) processed by a worker, or run it in a worker_thread — so the event loop stays responsive. Debugging this teaches the whole topic: sync vs async, I/O- vs CPU-bound, callbacks/Promises, and why the event loop is what makes or breaks throughput.

Interview Q&A (SDE-2)

Q1. JavaScript is single-threaded — so how does it do things concurrently?
The JS engine runs your code on one thread with one call stack, but the runtime (browser Web APIs or Node's libuv) handles I/O, timers, and network on separate OS-level threads/mechanisms. When that work completes, its callback is queued and the event loop pushes it onto the stack only when the stack is empty. So concurrency comes from offloading waiting to the environment, not from running JS in parallel.
Q2. Explain the difference between I/O-bound and CPU-bound work and why it matters in Node.
I/O-bound work spends most of its time waiting on external systems (disk, network, DB); the CPU is idle during the wait, so async lets one thread handle many such tasks concurrently. CPU-bound work (hashing, compression, image processing) keeps the core busy and blocks the event loop no matter how you write it — async syntax doesn't help. The practical rule: async is the answer for I/O, but CPU work must be moved off the main thread via worker_threads, child processes, or an external job queue.
Q3. Predict the output: console.log(1); setTimeout(()=>console.log(2),0); Promise.resolve().then(()=>console.log(3)); console.log(4);
Output is 1, 4, 3, 2. Synchronous logs 1 and 4 run first. When the stack clears, the event loop drains the microtask queue fully before any macrotask, so the Promise callback (3) runs before the setTimeout callback (2), even with a 0ms delay. This demonstrates that microtasks always have priority over macrotasks.
Q4. What is 'callback hell' and what problem beyond readability does it cause?
Callback hell is deeply nested callbacks from sequencing dependent async operations, which hurts readability and makes error handling and early returns awkward. The deeper problem is inversion of control: you hand your continuation to someone else's code and must trust it to call your callback exactly once, at the right time, with correct arguments — a callback called twice, never, or synchronously causes subtle bugs. Promises and async/await restore control flow and give a single, composable error channel.
Q5. What's the difference between setTimeout(fn, 0) and Promise.resolve().then(fn) in terms of when fn runs?
Both defer fn, but they land in different queues. Promise.resolve().then(fn) schedules a microtask, which runs as soon as the current synchronous code finishes and before the next macrotask. setTimeout(fn, 0) schedules a macrotask, which runs only after the entire microtask queue is drained. So the Promise callback effectively runs sooner, and a flood of microtasks can starve timers.
Q6. Why does fs.readFileSync hurt a Node web server but might be fine in a CLI script?
readFileSync blocks the single event-loop thread until the disk read completes. In a web server serving concurrent requests, that blocks every other request for the duration, tanking throughput and latency. In a one-shot CLI script or during startup there's nothing else to serve, so blocking is acceptable and the simpler code is fine. The distinction is whether the thread has other work it should be doing.
Q7. What's the difference between passing a callback as a reference versus invoking it, and why does it bite people?
setTimeout(fn, 100) passes the function value to be called later; setTimeout(fn(), 100) calls fn immediately and passes its return value (often undefined) as the 'callback'. The second runs fn synchronously right now and schedules nothing useful. This is a common bug with event handlers and timers, and it stems from functions being first-class values — you must pass the reference, or wrap it in an arrow function if you need to bind arguments.
Q8. When would you deliberately choose worker_threads or a job queue over async/await in Node?
When the work is CPU-bound. async/await only helps when there's I/O waiting to overlap; a heavy computation still monopolizes the single thread and blocks the event loop regardless of async syntax. worker_threads give you true parallelism within the process for CPU work, while an external queue (e.g. BullMQ/SQS + workers) decouples the work entirely, adds durability and retries, and lets you scale processing independently of the request-serving tier.
