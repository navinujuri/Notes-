Binary search, two sum and strings

Cohort 2.0 · 9 lessons · SDE-2 (3+ yrs) interview prep

Concept summary — all 9 steps

Binary Search (fundamentals)
Halving-search over a sorted or monotonic search space in O(log n) time, O(1) space. The whole game is the loop invariant and boundary handling: pick one template (closed [lo,hi] or half-open [lo,hi)) and stick to it, use mid = lo + (hi-lo)/2 to avoid integer overflow, and make sure the window strictly shrinks so it terminates. Preconditions matter: the array must be sorted, or more generally the predicate you search on must be monotonic.
Binary Search in Action
Applying the template to the canonical case: find a target's index in a sorted array (or report absence). Interviewers watch how you handle the off-by-one and empty-range cases; the reliable move is the half-open interval so hi is always exclusive and lo==hi means 'not found'. This is the muscle memory you reuse for every variant that follows.
Question: Two Sum
Find two indices whose values sum to a target. The optimal answer is a single-pass hash map (value -> index) in O(n) time / O(n) space, checking for target - x before inserting x. Classic because it tests whether you reach for hashing instead of the O(n^2) double loop, and it exposes edge cases like duplicate values and using the same element twice.
Two-Sum Variant on a sorted array (inferred)
The natural follow-up: given a sorted array, find a pair summing to target using the two-pointer technique (lo from left, hi from right, move inward based on sum vs target) in O(n) time / O(1) space. It contrasts with the hash approach and generalizes to 3Sum/4Sum via sort + fix-one + two-pointer. The insight — sortedness lets you prune a whole direction each step — is what bridges into lower_bound thinking.
lower_bound / upper_bound searches
Boundary searches instead of exact-match: lower_bound returns the first index with a[i] >= x, upper_bound the first with a[i] > x. These power 'search insert position', first/last occurrence, and counting (count = upper_bound - lower_bound). The interview point is that most 'find the smallest/largest index satisfying P' problems reduce to a lower_bound on a monotonic predicate.
lower_bound application question (inferred)
A direct use of boundary search — e.g. count occurrences of a value, or find the first and last position of an element in a sorted array — solved as two binary searches in O(log n) rather than a linear scan. It cements that once you can locate boundaries you compose them to answer range/count queries cheaply.
Taking it to the next level: binary search on the answer
When the answer is a number and 'is X feasible?' is a monotonic predicate (false...false,true...true), binary search the answer space and run an O(n) feasibility check per candidate. Canonical problems: Koko eating bananas, capacity to ship packages in D days, minimize the largest subarray sum. This is the single highest-leverage binary-search pattern in interviews because it turns optimization into a log-factor search — total O(n log(range)).
Q - Vasya and String (Codeforces 676C)
Given a binary string of a's and b's and at most k allowed flips, find the longest substring of one repeated character. Solved with a sliding window: expand right, track the count of the minority char, shrink left when flips-needed exceeds k; run it once assuming target 'a' and once assuming 'b', take the max — O(n). It's a great example of 'longest window satisfying a constraint', and note it can also be phrased as binary search on the window length (O(n log n)), which is a weaker but instructive alternative.
String sliding-window follow-up (inferred)
A second string problem extending the window theme — e.g. longest substring without repeating characters or longest substring with at most k distinct characters — using a two-pointer window with a frequency map in O(n). The takeaway is recognizing the 'variable-size window with a validity condition you can incrementally maintain' family, which covers a large share of string interview questions.

Real-world example

An SRE health-window and capacity service for a service-mesh dashboard

You are building the backend for a reliability dashboard. Requests arrive with monotonically increasing timestamps stored in a sorted array, so seeking the start of a query window ('errors in the last 15 minutes') is a lower_bound over timestamps, and the count in a range is upper_bound - lower_bound — the exact technique time-series stores use to slice range queries in O(log n) instead of scanning. For an anomaly rule you correlate metrics: given a sorted list of per-endpoint error counts and a target budget, a two-sum/two-pointer pass finds two endpoints whose combined errors exactly blow the budget, so you can flag the pair. Capacity planning is binary search on the answer: 'what is the minimum number of pods so every request finishes within the SLA?' is monotonic (more pods never makes it worse), so you binary-search the pod count and run an O(n) feasibility check that simulates the load per candidate. Finally, health-check streaks map straight onto Vasya and String: encode each probe as 'g' (green) or 'r' (red), and 'the longest stretch we can call healthy if we tolerate up to k transient red checks' is a sliding window over the sequence. One dashboard, and four steps of this topic each show up as the natural, optimal tool.

Interview Q&A (SDE-2)

Q1. Why write mid = lo + (hi - lo) / 2 instead of (lo + hi) / 2, and how do you guarantee your binary search terminates?
lo + hi can overflow a fixed-width integer when both are large (the classic bug in Java's Arrays.binarySearch, fixed in 2006); lo + (hi - lo)/2 computes the same midpoint without ever exceeding hi. Termination requires the search window to strictly shrink every iteration: with a half-open [lo, hi) loop you set lo = mid + 1 or hi = mid, and since mid is always in [lo, hi), one of the bounds always moves. The infinite-loop trap is a closed-interval template where mid rounds toward lo and you set lo = mid without the +1.
Q2. Explain lower_bound vs upper_bound and how you'd count occurrences of a value in a sorted array.
lower_bound returns the index of the first element >= x; upper_bound returns the first element > x. Both are binary searches on a monotonic predicate over a half-open interval: for lower_bound, if a[mid] < x move lo = mid+1 else hi = mid. The count of x is simply upper_bound(x) - lower_bound(x), giving an O(log n) range query, and lower_bound also directly answers 'search insert position'.
Q3. Compare the hash-map and two-pointer solutions to Two Sum. When do you pick each?
On an unsorted array, a single-pass hash map (value -> index) is O(n) time / O(n) space and preserves original indices — the default answer. On an already-sorted array (or when you can sort and indices don't matter), the two-pointer sweep is O(n) time / O(1) space, which is better on memory and extends to 3Sum/4Sum. If sorting is required just to enable two pointers, you pay O(n log n), so for a one-off query on unsorted data the hash map usually wins.
Q4. What signals that a problem is 'binary search on the answer', and what's the complexity?
The answer is a single number within a bounded range, and feasibility is monotonic: if value v works then every v on one side also works (false...false, true...true). You binary-search the value range and run a feasibility predicate at each step, e.g. Koko eating bananas or 'ship packages in D days'. Cost is O(f * log(range)) where f is the predicate cost (often O(n)); the key correctness detail is proving monotonicity and choosing the initial [lo, hi] so the true answer is inside it.
Q5. How would you solve Vasya and String, and what is the alternative framing?
Sliding window: keep a window and the count of the minority character inside it; expand the right edge, and whenever the number of flips needed (the minority count) exceeds k, advance the left edge. Run it once treating the target as 'a' and once as 'b', take the max window length — O(n) time, O(1) extra space. The alternative is binary search on the window length L: for a fixed L you can check in O(n) whether some length-L window needs <= k flips, and feasibility is monotonic in L, giving O(n log n) — correct but strictly slower than the two-pointer sweep.
Q6. How does binary search adapt to a rotated sorted array?
At each step compute mid; because the array is one rotation of a sorted array, at least one of [lo, mid] and [mid, hi] is fully sorted. Determine which half is sorted by comparing a[lo] with a[mid], check whether the target lies within that sorted half's value range, and recurse into the half that can contain it. It stays O(log n); the subtlety is picking consistent inclusive/exclusive comparisons to handle duplicates and the pivot correctly.
Q7. Binary search usually assumes distinct sorted data — how do you handle duplicates or a floating-point search space?
For duplicates, stop doing exact-match search and switch to lower_bound/upper_bound so you deterministically land on the first or last matching index rather than an arbitrary one. For a real-valued answer (e.g. binary-searching a ratio or a physical threshold), there's no integer step, so you either iterate a fixed number of times (~100 iterations is well past double precision) or loop while hi - lo > epsilon, avoiding the exact-equality comparisons that never terminate with floats.
