Web Dev + Devops Orientation

Cohort 2.0 · 8 lessons · SDE-2 (3+ yrs) interview prep

Concept summary — all 8 steps

Cohort scope: full-stack + DevOps
An orientation to a program that deliberately couples web development (frontend + backend) with DevOps (deploy, CI/CD, infra). The point for an SDE-2 is the 'own it end-to-end' mindset: writing code is table stakes; being able to ship it, observe it in prod, and roll it back is what separates SDE-1 from SDE-2/staff.
Avoid the spikes (consistency over cramming)
Learning strategy: steady, spaced practice beats binge-then-burnout spikes. Skill retention follows spaced repetition and deliberate practice; a sawtooth effort curve loses most of the gains between sessions. The same principle maps to careers — sustainable throughput matters more than heroic sprints.
'What would I do if I was you?' — build, don't just watch
Mentor advice to bias toward building projects and fundamentals over passively consuming tutorials or chasing every new framework. Interview angle: interviewers probe depth on core concepts (HTTP, the DOM, the event loop, data structures) that transfer across stacks, not framework trivia that expires.
Select your level (self-assessment)
Pick a beginner vs. intermediate track by honestly benchmarking your baseline. Meta-learning skill: entering material that's too easy wastes time and too hard causes churn; the productive zone is just past your current ability (the 'desirable difficulty').
Good things to hash out (expectations + prerequisites)
Align on ground rules up front: prerequisites, tooling, environment, time commitment, and how to ask for help. Analogous to a project kickoff or design-doc review — clarifying scope and constraints before writing code prevents expensive rework later.
VSCode installation & editor tooling
Set up the editor plus the workflow around it: extensions (linter, formatter, language servers), integrated terminal, and debugger. Why it matters: consistent formatting/linting via config committed to the repo (e.g., ESLint + Prettier, EditorConfig) enforced in pre-commit hooks is the first link in a DevOps quality chain, keeping diffs clean and reviews focused on logic.
HTML — semantic document structure
HTML defines the structure and semantics of a page via elements the browser parses into the DOM. Using semantic tags (header, nav, main, article, button) over div soup improves accessibility (screen readers, keyboard nav), SEO, and maintainability. Interview staple: the difference between semantic markup and presentational markup, and why native elements beat re-implementing behavior on divs.
CSS — styling, layout, and the cascade
CSS controls presentation via the cascade, specificity, and inheritance, plus the box model and layout engines (Flexbox for 1D, Grid for 2D). Core SDE-2 knowledge: how specificity resolves conflicts, why box-sizing: border-box simplifies sizing, and how layout choices affect rendering performance (reflow vs. repaint).

Real-world example

Ship a personal portfolio site with an automated deploy pipeline

You build a static portfolio page: semantic HTML for structure (header/nav/main/section/footer) and CSS with a Grid page layout plus Flexbox for the nav bar, using box-sizing: border-box and a small responsive breakpoint set. In VSCode you wire Prettier + a linter and commit the config to the repo so formatting is deterministic across machines. That's the 'web dev' half. The 'DevOps' half: you push to a Git branch, a CI job runs the linter and a build step, and on merge to main a CD job publishes the static assets to a host/CDN (e.g., GitHub Pages, Netlify, or an S3 + CloudFront setup) with cache headers. When a deploy makes the site look broken, you debug across the whole chain — is it a CSS specificity/cascade bug, a caching issue where the CDN served a stale stylesheet, or a build step that didn't emit the file? This single project exercises exactly what the orientation promises: writing the code, shipping it automatically, and reasoning about it in production.

Interview Q&A (SDE-2)

Q1. Why prefer semantic HTML over generic divs, concretely?
Semantic elements carry built-in accessibility semantics and behavior: a <button> is focusable, keyboard-activatable, and announced as a button by screen readers, while a <div onclick> requires you to re-add role, tabindex, and key handlers by hand — and people usually forget. Semantics also aid SEO and give other developers (and browser features like reader mode) a machine-readable document outline. The rule of thumb: reach for the native element that already does the job before styling a div to imitate it.
Q2. Explain the CSS box model and why box-sizing: border-box is the common default.
Every element is a box of content + padding + border + margin. With the default content-box, width applies only to the content, so padding and border add to the rendered width, making layout math error-prone. border-box makes width include padding and border, so an element set to width: 200px is actually 200px wide regardless of its padding. Teams typically set box-sizing: border-box globally (often via *, *::before, *::after) to make sizing predictable.
Q3. Flexbox vs. Grid — how do you choose?
Flexbox is one-dimensional: it lays out items along a single axis (a row or a column) and is ideal for nav bars, toolbars, and distributing space among a set of items. Grid is two-dimensional: it controls rows and columns simultaneously and is ideal for page-level layouts or any structure where alignment in both axes matters. A common pattern is Grid for the overall page skeleton and Flexbox for the components inside each cell; they compose rather than compete.
Q4. How does CSS resolve which rule wins when several target the same element?
The cascade resolves in order: importance and origin first (an !important author rule beats a normal one; inline the trickiest), then specificity, then source order as the tiebreaker. Specificity is counted as (inline, IDs, classes/attributes/pseudo-classes, elements) — an ID selector outranks any number of classes. Because high-specificity selectors and !important are hard to override later, maintainable CSS keeps specificity low and flat (favoring classes) rather than winning fights with ever-more-specific selectors.
Q5. What's the difference between reflow (layout) and repaint, and why care?
Repaint happens when a change affects only appearance (e.g., color, background) — the browser redraws pixels without recomputing geometry. Reflow (layout) happens when geometry changes (width, adding/removing nodes, font size), forcing the browser to recompute positions and sizes, which can cascade to descendants and is more expensive. Performance-wise you avoid layout thrash (interleaving DOM reads and writes in a loop) and prefer animating transform/opacity, which can be composited on the GPU without triggering layout or paint.
Q6. Compare CSS position values: static, relative, absolute, fixed, sticky.
static is the default, in normal flow, ignoring top/left. relative stays in flow but is offset from its normal spot and establishes a positioning context for descendants. absolute is removed from flow and positioned relative to the nearest positioned ancestor (else the initial containing block). fixed is positioned relative to the viewport and stays put on scroll. sticky is a hybrid — it behaves like relative until a scroll threshold, then sticks like fixed within its container.
Q7. How does editor tooling like VSCode + linters/formatters fit into a DevOps workflow?
The goal is that code quality is enforced by config, not by memory or reviewer nagging. You commit editor and tool config (EditorConfig, ESLint/Prettier settings) so every machine formats identically, run format-on-save locally, gate commits with a pre-commit hook (e.g., husky + lint-staged), and re-run the same lint/format/build checks in CI so nothing merges unformatted. This keeps diffs small and reviews focused on logic — the first, cheapest quality gate in the pipeline before build, test, and deploy stages.
Q8. At a high level, what happens between typing a URL and seeing a styled page?
DNS resolves the domain to an IP, a TCP (and for HTTPS, TLS) connection is established, and the browser sends an HTTP request. The server responds with HTML, which the browser parses into the DOM while fetching linked CSS (into the CSSOM) and JS. The DOM and CSSOM combine into a render tree, the browser computes layout (positions/sizes), paints pixels, and composites layers to the screen — and render-blocking CSS or synchronous JS in the <head> can delay first paint, which is why asset placement and loading strategy matter for perceived performance.
