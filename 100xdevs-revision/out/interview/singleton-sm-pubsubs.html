Singleton Pattern, Backend State management and Pub Subs

Cohort 2.0 · 6 lessons · SDE-2 (3+ yrs) interview prep

Concept summary — all 6 steps

Topic framing: where backend state lives
Sets up the core question — should application state live inside the server process or be externalized, and how do the Singleton pattern plus in-process pub/sub let you share and broadcast that state cleanly within one Node process. The interview thread is knowing when in-memory state is fine and exactly where it breaks under horizontal scale.
Stateful vs Stateless Backends
Stateless servers keep no per-client memory between requests (context is carried in the request or fetched from a shared store), so any instance serves any request — trivial load balancing, safe restarts, easy autoscaling. Stateful servers hold session/connection/game state in memory for lower latency and real-time use, but need sticky sessions and complicate failover. Interview angle: articulate the trade-off and know that the standard fix is to keep only the connection stateful and externalize shared/durable state to Redis or a DB.
State in the JS process
Node runs a single-threaded event loop over one shared process heap, so any module-scoped variable persists across requests for the process lifetime — fast, but lost on crash/restart, unbounded (memory-leak risk), and NOT shared across workers. Interview angle: under PM2 cluster or k8s replicas each process has its own heap, so process memory silently diverges — the classic 'works in dev, breaks in prod' bug.
Classes and the Singleton pattern
Singleton guarantees one instance with a single access point; in Node you get it almost free by exporting one instance (the require/import cache dedupes per process), or explicitly via a static getInstance() with lazy init. Used for shared resources: DB pools, config, in-memory stores, the pub/sub hub. Interview angle: it's a controlled global — downsides are hidden coupling, hard testing/mocking, and it's only 'single' per process, so DI is often cleaner.
Pub/Sub combined with Singleton
Pub/Sub decouples publishers from subscribers through named topics/channels so components communicate without direct references (fan-out, no tight coupling). The event manager is made a Singleton so every part of the app publishes and subscribes against the same registry. Interview angle: an in-memory manager only reaches subscribers in the same process — cross-instance fan-out requires an external broker like Redis pub/sub, Kafka, or NATS.
Pub/Sub + Singleton implementation
Concretely: a class holding a Map of topic -> Set of callbacks, with subscribe/publish/unsubscribe, exported as one instance; often it also bridges an external Redis subscription to in-process WebSocket connections. Interview angle: the load-bearing details are unsubscribe-on-disconnect to avoid leaking dead sockets/closures, try/catch around each callback so one bad subscriber doesn't break the fan-out, and choosing sync vs async dispatch.

Real-world example

Real-time stock price WebSocket fan-out service

You build a WebSocket service where clients subscribe to stock symbols (AAPL, TSLA...). A SubscriptionManager singleton holds a Map of symbol -> Set of client sockets, and on startup opens one Redis connection subscribed to the price channels. When a Redis message arrives, the singleton looks up subscribers for that symbol and pushes to each socket; on client disconnect the 'close' handler unsubscribes to avoid leaking dead sockets. The singleton is the right call because the connection handler, the Redis message handler, and the disconnect handler all need one authoritative registry and one Redis connection without threading references everywhere. The key SDE-2 insight surfaces when you scale out: each WS instance keeps its own in-memory socket registry (stateful, process-local), but the shared truth lives in Redis — Redis pub/sub fans every price tick out to all instances, and each instance delivers only to its locally connected clients. If you had relied on an in-process EventEmitter instead of Redis, a publish on instance A would never reach a client connected to instance B — the exact bug that passes in dev with one process and fails in production behind a load balancer.

Interview Q&A (SDE-2)

Q1. Why are stateless backends preferred for horizontal scaling, and when is statefulness unavoidable?
Stateless servers hold no per-client memory between requests, so any instance can serve any request — that gives trivial load balancing, safe restarts, and easy autoscaling. Statefulness becomes unavoidable for long-lived WebSocket connections, real-time games, or matchmaking where round-tripping to a store on every event is too slow. The standard pattern is to keep only the connection itself stateful and externalize any shared or durable state to Redis/DB so the servers stay effectively stateless for routing.
Q2. How does Node's module system give you a 'free' singleton, and what are the limits?
require/import caches a module after first evaluation, so every importer receives the same exported object — export an instance and you have a process-wide singleton with no extra code. The limits: it's per-process only (a cluster/PM2/k8s deployment gets one per worker), the cache can be defeated by different resolved module paths or by clearing require.cache, and it makes the dependency implicit and global, which hurts testability and hides coupling.
Q3. Compare a module-level singleton with a getInstance() singleton.
A module-level export creates the instance eagerly at import time — simplest and enough for most cases. getInstance() with a static field enables lazy initialization (created on first use), can take init parameters, and gives an explicit access point, at the cost of boilerplate. Both are single-per-process; pick the lazy form when construction is expensive or must wait until config/env is ready.
Q4. What are the real downsides of the Singleton pattern?
It is effectively a controlled global, so it introduces hidden shared state and coupling that make code paths depend on each other invisibly. It complicates unit testing because you must reset or mock the shared instance between tests, and it can cause initialization-order and circular-dependency problems. It also gives a false sense of 'exactly one' since it is only one per process — dependency injection is often a cleaner, more testable alternative.
Q5. Why is in-memory pub/sub insufficient at scale, and how do you fix it?
An in-process event manager or EventEmitter only reaches subscribers living in the same process, so with multiple instances behind a load balancer a publish on instance A never reaches subscribers connected to instance B. You fix it with an external broker — Redis pub/sub, Kafka, or NATS — where every instance subscribes to the channel and then re-fans-out to its own local in-memory subscribers. The in-memory singleton becomes the last-hop delivery mechanism, while the broker handles cross-instance distribution.
Q6. How do you implement a basic pub/sub manager without leaking memory?
Keep a Map of topic -> Set of callbacks: subscribe adds a callback, publish iterates the set and invokes each, unsubscribe removes it. The critical hygiene is removing subscriptions when a client disconnects (for WebSockets, on the 'close' event), deleting topic entries once their set is empty, and wrapping each callback in try/catch so one throwing subscriber doesn't abort the whole fan-out. Skipping unsubscribe-on-disconnect leaks closures and references to dead sockets, which grows the heap until the process falls over.
Q7. In a WebSocket fan-out service, why keep the socket registry in a singleton rather than passing it around?
The registry is shared mutable state that several independent code paths — the connection handler, the broker/Redis message handler, and the disconnect handler — all must reach. A singleton provides one authoritative registry and a single shared broker connection without threading references through every function or middleware. The trade-off is the usual singleton coupling, but for a single logical resource per process it's a pragmatic and common choice.
Q8. How does process-level state behave under PM2 cluster mode or Kubernetes with multiple replicas?
Each worker or replica is a separate OS process with its own heap, so module-level variables and singletons are duplicated and not shared — a counter incremented on one instance is invisible to the others. Sticky sessions can pin a given client to one instance, but any truth that must be consistent across clients has to live in an external store like Redis or a database. This is precisely why relying on in-process state passes in single-process dev and breaks once you scale to many instances in production.
