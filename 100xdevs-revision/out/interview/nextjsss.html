Recap of prior parts
Consolidation checkpoint anchoring Part 3 on top of App Router fundamentals: file-based routing, Server vs Client Components, layouts, loading/error boundaries, and data fetching. Interview angle: be able to state where the App Router draws the server/client boundary (`"use client"`) and why RSCs are the default.
Route groups ( )
A folder wrapped in parentheses like `(marketing)` organizes routes and lets you scope layouts without adding a segment to the URL. It matters for giving distinct sections their own root layout (e.g. `(auth)` vs `(dashboard)`) or splitting one layout tree into several; interviewers probe that it is purely organizational and does not change the path.
Dynamic routes [ ]
A `[id]` folder maps a single path segment to a param exposed via `params` (e.g. `/blog/[slug]`). Core to any detail page; the interview angle is that params arrive as strings, are available in the page/layout/route-handler, and that you pre-render them at build via `generateStaticParams`.
Catch-all [...slug]
`[...slug]` matches one OR MORE trailing segments and delivers them as an array (`/docs/a/b/c` -> `['a','b','c']`). Useful for docs trees, nested category paths, or proxy-style routes; the key distinction interviewers want is that it requires at least one segment, so the bare parent path does NOT match.
Optional catch-all [[...slug]]
Double brackets make the catch-all optional so the parent route matches too — `/shop` and `/shop/a/b` both hit `[[...slug]]`, with `params.slug` being `undefined` for the base case. It matters when one route must serve both an index and arbitrary nesting; the interview point is precisely this: `[...x]` needs a segment, `[[...x]]` does not.
Static Site Generation (SSG)
Pages are rendered to HTML at build time (or via ISR revalidation) rather than per request, giving CDN-cacheable, fast responses. In the App Router this is the default when a route has no dynamic APIs; dynamic segments are enumerated with `generateStaticParams`, and `revalidate`/`dynamicParams` control ISR and on-demand fallback. Interview angle: know SSG vs SSR vs ISR trade-offs and what silently opts a route into dynamic rendering (cookies, headers, `no-store`).
Hydration
Hydration is React attaching event listeners and reconciling client state onto the server-rendered HTML so a static page becomes interactive. It matters because mismatches between server and client render (dates, random values, `window` access) throw hydration errors, and heavy client trees delay interactivity — the interview angle is reducing hydration cost by keeping components as Server Components and minimizing `"use client"` boundaries.
Middleware
`middleware.ts` at the project root runs on the Edge runtime BEFORE a request is completed, ideal for auth gating, redirects/rewrites, A/B routing, i18n, and setting headers/cookies. Scoped via a `matcher` config; the interview angle is its constraints — Edge runtime only (no Node APIs), must be fast, cannot read a request body, and returns a `NextResponse` rather than rendering.
Assignment / Project
Applied build combining route groups, dynamic + catch-all segments, SSG with `generateStaticParams`, and middleware-based auth into one app. The takeaway is wiring these primitives together correctly — e.g. a statically generated content site with a protected dashboard — which is exactly the system-design shape an SDE-2 is expected to reason about.

Real-world example

A statically generated docs + blog site with a gated dashboard

You build a company site on the App Router. Marketing and legal pages live under a `(marketing)` route group so they share a clean root layout without polluting URLs, while the app itself lives under `(dashboard)` with its own layout. Docs use `app/docs/[[...slug]]/page.tsx` so `/docs` renders an index and `/docs/guides/routing` renders a deep page from the same route, with all known paths pre-rendered at build via `generateStaticParams` and ISR (`revalidate = 3600`) so CDN-cached HTML stays fresh. The blog uses `app/blog/[slug]` for individual posts. A `[...path]` catch-all under `/legacy` rewrites old CMS URLs. Root `middleware.ts` runs on the Edge, checks a session cookie, and redirects unauthenticated users away from `(dashboard)` routes via a `matcher`. When a 'text content does not match' hydration error appears on a post page, you trace it to a `new Date().toLocaleString()` rendered in a Client Component and fix it by formatting on the server or gating on `useEffect`, keeping the interactive island small so hydration stays cheap.

Interview Q&A (SDE-2)

Q1. What is the difference between `[...slug]` and `[[...slug]]`, and when would you choose each?
`[...slug]` is a catch-all that matches one or more segments and requires at least one, so the bare parent path (e.g. `/docs`) is a 404. `[[...slug]]` is an optional catch-all: the parent path also matches and `params.slug` is `undefined` in that case. Choose the optional form when a single route must serve both an index page and arbitrary nested paths — a docs section where `/docs` and `/docs/a/b` share one component.
Q2. Do route groups affect the URL, and why use them?
No — a folder like `(marketing)` is stripped from the URL entirely; it is purely for organization. The main reasons to use them are to give different sections of the app their own layout hierarchy (e.g. an auth layout vs a dashboard layout) without a shared parent segment, to split one large layout tree into multiple root layouts, and to co-locate related routes. A gotcha: you cannot have two routes resolve to the same final path across different groups, or you get a collision.
Q3. How does SSG work in the App Router, and what silently turns a route dynamic?
A route is statically rendered by default at build time when it uses no request-time (dynamic) APIs; dynamic segments are enumerated with `generateStaticParams`. Using `cookies()`, `headers()`, the `searchParams` prop, `fetch` with `cache: 'no-store'`, or setting `export const dynamic = 'force-dynamic'` opts the route into per-request server rendering. ISR sits in between — `export const revalidate = N` regenerates the static page in the background after N seconds, and `dynamicParams` controls whether params not returned by `generateStaticParams` are rendered on demand or 404.
Q4. What causes a hydration mismatch and how do you fix it?
It happens when the HTML React renders on the client during hydration does not match what the server produced — common causes are `Date.now()`/`new Date()`, `Math.random()`, locale-dependent formatting, reading `window`/`localStorage` during render, or invalid HTML nesting. Fixes: compute non-deterministic values on the server and pass them down, defer client-only logic to `useEffect` (so it runs after hydration), or use `suppressHydrationWarning` for intentionally divergent nodes like timestamps. The deeper point is that hydration replays the server render, so render output must be deterministic across environments.
Q5. What are the constraints of Next.js middleware and what is it good for?
Middleware runs on the Edge runtime before the request is completed, so it must be fast and cannot use Node-only APIs (fs, native modules) or read the request body; it returns a `NextResponse` (next, redirect, rewrite, or header/cookie mutation) rather than rendering UI. It is ideal for cross-cutting concerns: auth/session gating, redirects and rewrites, i18n locale routing, A/B bucketing, and bot/geo handling. Scope it with a `matcher` in the config export so it does not run on every asset request, and keep heavy logic (DB calls, verification) minimal or push it to the route itself.
Q6. How do you statically pre-render a dynamic route like `/blog/[slug]`?
Export an async `generateStaticParams` from the segment that returns an array of param objects, e.g. `[{ slug: 'hello' }, { slug: 'world' }]`; Next.js pre-renders one HTML file per entry at build time. Combine it with `revalidate` for ISR and with `dynamicParams` (default `true`) to decide whether slugs not in the list are rendered on demand and cached, or return a 404 (`false`). This is the App Router replacement for the old `getStaticPaths` + `getStaticProps` pairing.
Q7. Why are Server Components the default, and how does that relate to hydration cost?
Server Components render on the server and ship zero JS for their own logic, so only their serialized output crosses the wire — this shrinks the client bundle and reduces the amount of JavaScript that must be downloaded, parsed, and hydrated. Only components marked `"use client"` (and their subtree) hydrate. The practical rule is to keep `"use client"` boundaries as low and small as possible — pass server-rendered content into client components as children/props rather than pulling large trees into the client — which directly lowers time-to-interactive.
Q8. Middleware vs a layout vs a route handler for auth — which and why?
Middleware is best for coarse, fast gate-keeping (redirect unauthenticated users away from a section) because it runs before rendering and can short-circuit with a redirect at the Edge. A layout/page can enforce auth too via server-side session checks and is the right place when you need data-backed decisions or to render a partial UI, but it runs later and after routing. Route handlers protect API endpoints specifically. In practice you layer them: middleware for the cheap redirect, plus a server-side check in the protected route so security does not depend on middleware alone (defense in depth).
