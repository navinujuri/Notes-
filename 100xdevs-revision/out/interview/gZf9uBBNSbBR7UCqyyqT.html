Prisma

Cohort 2.0 · 12 lessons · SDE-2 (3+ yrs) interview prep

Concept summary — all 12 steps

What are ORMs
An ORM (Object-Relational Mapper) maps DB tables/rows to language objects so you query with typed code instead of raw SQL. Interview angle: know the mapping strategies — Active Record (model = row, e.g. Sequelize/Rails) vs Data Mapper (separate persistence layer, e.g. TypeORM/JPA). Prisma is neither classic pattern; it's a query builder + generated client.
Why ORMs
They give type safety, autocompletion, migrations, and portability across SQL dialects, and eliminate hand-written boilerplate for CRUD. The trade-off worth stating in interviews: ORMs abstract SQL but can produce inefficient queries (N+1, over-fetching) and hide the actual execution plan, so you still need to read generated SQL and drop to raw for hot paths.
What is Prisma
Prisma is a Node/TypeScript toolkit: a declarative schema (schema.prisma), a migration engine (prisma migrate), an auto-generated type-safe client (@prisma/client), and Studio. It is not a runtime ORM library you import models from — it generates a client from your schema, which is why types stay in sync with the DB.
Installing Prisma in a fresh app
`npm i -D prisma`, `npm i @prisma/client`, then `npx prisma init` scaffolds prisma/schema.prisma and a .env with DATABASE_URL. Note the split: prisma CLI is a devDependency (tooling), @prisma/client is a runtime dependency (your app imports it).
Selecting your database
The datasource block's provider (postgresql, mysql, sqlite, sqlserver, mongodb) plus DATABASE_URL picks the DB. This matters because provider changes what's supported — e.g. enums, arrays, and JSON differ; Mongo uses a different connector with no SQL migrations. Keep the connection string in .env, never in schema.
Defining your data model
You declare models, fields, scalar types, @id/@unique/@default/@relation attributes, and enums in schema.prisma — this is the single source of truth for both DB shape and generated types. Interview point: map-level attributes like @map/@@map decouple pretty TS names from actual column/table names, and @@index/@@unique define composite constraints.
Exploring your database
`prisma db pull` (introspection) reverse-engineers an existing DB into the schema, while Prisma Studio (`npx prisma studio`) gives a GUI to browse/edit rows. Know the two directions: db pull (DB → schema) for existing/legacy DBs vs migrate/db push (schema → DB) for greenfield.
Generating the Prisma Client
`npx prisma generate` reads the schema and emits a fully typed client into node_modules/@prisma/client. Key gotcha for interviews/CI: the client is generated code, so you must re-run generate after every schema change and in build pipelines (postinstall hook), or you get stale types/runtime mismatches.
Creating your first app
Instantiate `new PrismaClient()` once and reuse it (singleton) to run typed queries like prisma.user.findMany(). The common bug this teaches: instantiating PrismaClient per-request or per-hot-reload (Next.js dev) exhausts the DB connection pool — cache it on globalThis in dev.
Relationships
Prisma models 1-1, 1-many, and many-many via @relation with scalar foreign-key fields; implicit m-n creates a hidden join table, explicit m-n lets you model the join with extra columns. Query angle: use `include`/`select` to fetch relations, and nested writes (create/connect) for atomic multi-table inserts — this is where N+1 and over-fetching are decided.
Todo functions
The CRUD surface: create/createMany, findUnique/findMany with where/orderBy/skip/take, update/upsert, delete/deleteMany. Know upsert (insert-or-update in one call), findUnique vs findFirst (unique index vs first match), and that pagination is offset (skip/take) or cursor-based — cursor is preferred for large/stable feeds.
Expressify it
Wire the Prisma client into an Express REST layer: route handlers call client methods, await results, and shape JSON responses with proper status codes and try/catch error handling. The real lesson is separation of concerns — keep the singleton client in one module, handle Prisma error codes (e.g. P2002 unique violation, P2025 not found) and map them to HTTP 4xx.

Real-world example

A multi-tenant Todo/Task API backed by Postgres

You're building the backend for a team task app: an Express + TypeScript service on Postgres. In schema.prisma you define User, Project, and Todo models — a User has many Projects (1-many), a Project has many Todos (1-many), and Todos carry tags via an explicit many-to-many join so you can store when a tag was attached. You run `prisma migrate dev` to version the schema, `prisma generate` to get the typed client, and instantiate a single PrismaClient exported from db.ts. Express routes expose CRUD: POST /todos does a nested write (create the todo and connect existing tags atomically), GET /projects/:id/todos uses `include` with `take`/`cursor` for pagination, and PATCH /todos/:id uses upsert. When a duplicate title hits the @@unique constraint you catch Prisma error P2002 and return 409; a missing id (P2025) returns 404. Debugging later, you enable query logging, spot an N+1 from looping findMany per project, and collapse it into one query with `include`, dropping p95 latency.

Interview Q&A (SDE-2)

Q1. How does Prisma differ from a traditional ORM like TypeORM or Sequelize?
Prisma isn't a classic Active Record or Data Mapper library — you don't define model classes you import and instantiate. Instead you write a declarative schema.prisma, and Prisma generates a type-safe client from it, so the query API and TypeScript types are always derived from one source of truth. Practically this means fewer decorator/entity mismatches and stronger autocompletion, but it also means you must regenerate the client on every schema change, and Prisma historically had thinner support for very complex raw SQL and some advanced DB features.
Q2. What is the N+1 problem and how does Prisma help or hurt here?
N+1 is when you run one query to fetch a list, then one additional query per row to fetch a relation — e.g. fetch 100 projects then loop calling findMany for each project's todos, yielding 101 queries. Prisma can cause it if you naively loop queries, but it solves it two ways: `include`/`select` batches relations into an efficient set of queries, and Prisma's query engine also batches concurrent findUnique calls via a DataLoader-style mechanism. The interview point is that the ORM doesn't automatically save you — you have to eager-load intentionally.
Q3. Explain migrate dev vs db push vs db pull.
`prisma migrate dev` generates and applies versioned SQL migration files — this is the source-controlled, production-path workflow. `prisma db push` syncs the schema to the DB without creating migration history, good for prototyping/early dev where you don't care about migration files. `prisma db pull` goes the other direction, introspecting an existing database to generate/update the schema, which is how you onboard Prisma onto a legacy DB.
Q4. Why must you be careful instantiating PrismaClient, especially in Next.js or serverless?
Each PrismaClient opens its own connection pool. In dev with hot reload, or in serverless where every invocation is a fresh module, creating a new client per request/reload rapidly exhausts the database's max connections. The fix is a singleton — cache the client on globalThis in development and reuse one instance in the app; in serverless you additionally use a connection pooler like PgBouncer or Prisma's Accelerate/Data Proxy because functions scale beyond raw DB connection limits.
Q5. How do implicit vs explicit many-to-many relations differ in Prisma?
An implicit m-n relation is declared by putting a list field on both models; Prisma silently creates and manages a hidden join table with just the two foreign keys. An explicit m-n means you model the join table as its own Prisma model with two 1-many relations, which you must do when the join needs extra columns (e.g. createdAt, role, sort order) or its own primary key. Rule of thumb: use implicit for a pure link, explicit the moment the relationship carries data.
Q6. What's the difference between findUnique, findFirst, and upsert?
findUnique looks up by a unique field or primary key and can leverage the unique index; it only accepts fields that are @id/@unique. findFirst returns the first row matching arbitrary where/orderBy conditions, so it works on non-unique filters. upsert is a single atomic operation that updates a row if it exists (matched by a unique where) or creates it otherwise, avoiding a race-prone find-then-write in application code.
Q7. How should you handle Prisma errors in an API layer?
Prisma throws typed errors — PrismaClientKnownRequestError with codes like P2002 (unique constraint violation), P2025 (record not found for update/delete), and P2003 (foreign key constraint). In an Express handler you wrap calls in try/catch, inspect error.code, and map them to appropriate HTTP statuses (P2002 to 409, P2025 to 404, validation issues to 400) rather than leaking a 500. This keeps the DB layer's semantics translated into a clean API contract.
Q8. When would you drop to raw SQL, and how does Prisma support it?
You go raw for queries the client can't express efficiently — complex aggregations, window functions, recursive CTEs, or hand-tuned performance-critical paths. Prisma offers $queryRaw (returns rows, use the tagged-template form so parameters are safely bound) and $executeRaw (returns affected row count), plus $queryRawUnsafe when you must build dynamic SQL (with your own injection safeguards). The trade-off is you lose the generated type safety, so you type the result manually and confine raw usage to a small, well-tested layer.
