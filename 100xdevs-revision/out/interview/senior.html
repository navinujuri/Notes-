5 tools for senior engineers (Code without your mouse)

Cohort 2.0 · 8 lessons · SDE-2 (3+ yrs) interview prep

Concept summary — all 8 steps

Introduction — why go mouse-free
The thesis is that keeping your hands on the home row and driving everything (editing, navigation, window/session management, browsing) via keyboard cuts context-switch latency and RSI, and compounds over thousands of small actions per day. The interview angle isn't the tools themselves but demonstrating you optimize your own workflow and understand where your time actually goes.
vi / Vim — modal editing
vi (and modern Vim) is a modal editor: Normal mode for movement/operators, Insert for typing, Visual for selection, plus operator+motion composition (e.g. d2w, ci"). It matters because it's on virtually every Unix box (bastion hosts, containers, CI shells) where you have no GUI, so `vimtutor`-level fluency is a baseline ops skill, and its grammar (verb+noun) is the mental model every other tool on this list reuses.
Vimium — Vim keys in the browser
Vimium is a browser extension giving Vim-style keybindings: `f` to link-hint and follow, `j/k` scroll, `H/L` history, `/` find, `d/u` half-page. It matters as a low-cost way to keep the keyboard-first mental model consistent across your editor and browser during research/docs/PR review, eliminating the reach-for-mouse tax.
Neovim — the extensible fork
Neovim is a Vim fork with a first-class embedded Lua runtime, async job control, built-in LSP client, Treesitter parsing, and a stable RPC/msgpack API for external UIs. It matters because it turns the editor into a real IDE (completion, go-to-definition, diagnostics) via LSP without leaving the terminal, and Lua config (vs Vimscript) makes plugins faster and maintainable — this is what people mean today by 'daily-driving Vim'.
Vim mode in VS Code
The VSCodeVim / vscode-neovim extensions bring modal editing into VS Code, letting you keep muscle memory while retaining VS Code's debugger, extension ecosystem, and remote/dev-container tooling. The trade-off worth articulating: VSCodeVim reimplements Vim in JS (some fidelity gaps, occasional lag on large files) whereas vscode-neovim embeds a real nvim process for near-perfect semantics — pick based on how much of your config you need.
Window managers — tiling the desktop
A tiling window manager (i3/sway on Linux, yabai/Rectangle on macOS, PowerToys FancyZones/komorebi on Windows) auto-arranges windows into non-overlapping tiles addressed by keyboard, with workspaces/tags for grouping. It matters because it removes manual window dragging/alt-tab hunting and gives deterministic, reproducible layouts — the OS-level analog of the same keyboard-driven philosophy.
tmux — terminal multiplexer
tmux multiplexes one terminal into multiple sessions/windows/panes and, crucially, detaches: your processes keep running server-side after you disconnect and you reattach later. It matters for remote work — survive SSH drops, resume long builds/REPLs, and share a session for pairing — making it the terminal-side complement to a tiling WM and a common reason `screen`/`tmux` shows up in ops interviews.
Shell config / dotfiles
Shell rc files (.bashrc/.zshrc, .zprofile) plus tool configs (.vimrc/init.lua, .tmux.conf, git config) define aliases, functions, PATH, prompt, and completions; version-controlling them as 'dotfiles' makes any machine reproducible in minutes. Know the login vs interactive distinction (which files run when, e.g. .zprofile once at login vs .zshrc per interactive shell) — a frequent source of 'my alias/PATH isn't loading' bugs.

Real-world example

Debugging a prod incident on a bastion host, keyboard-only

You're paged for elevated 5xx on a service reachable only through a jump host with no GUI. On your laptop, a tiling WM snaps a browser (driven with Vimium to jump through the runbook and Grafana) beside your terminal. In the terminal you SSH to the bastion and attach a persistent tmux session so a flaky VPN won't kill your work; you split panes to tail logs in one, run the app in a REPL in another, and edit the config in the third. The remote box only has vi, so you fix a bad timeout value with `ci"` and `:wq` — no mouse, no GUI needed. Back on your workstation you reproduce locally in Neovim, where the LSP flags the type error your patch introduced before you even save, and you commit using the git aliases from your version-controlled dotfiles. The whole loop — browse, connect, edit, verify, commit — never left the keyboard, and the tmux session is still alive when you reattach the next morning to confirm the fix held.

Interview Q&A (SDE-2)

Q1. Explain Vim's modal editing model and why operator+motion composition is powerful.
Vim separates modes: Normal (navigate/command), Insert (type), Visual (select), plus command-line. The power is a composable grammar where operators (d delete, c change, y yank) combine with motions or text objects (w word, } paragraph, i" inside quotes), so `daw`, `ci(`, `y2j` are all learned from a small set of primitives rather than memorized shortcuts. This means your editing vocabulary scales multiplicatively instead of additively, and the same verbs work across counts, motions, and text objects.
Q2. What does Neovim add over classic Vim, and why does it matter for daily use?
Neovim adds an embedded Lua interpreter (faster, saner config than Vimscript), async job/event handling, a built-in LSP client, Treesitter for accurate syntax/highlighting/structural edits, and a stable RPC API so external GUIs and tools can drive it. Practically, LSP + Treesitter give you IDE features — completion, diagnostics, go-to-def, semantic highlighting — inside a terminal editor. It matters because you get modern tooling without abandoning the modal workflow or the terminal.
Q3. VSCodeVim vs vscode-neovim — how do they differ and when would you choose each?
VSCodeVim reimplements Vim semantics in TypeScript inside the extension, so it's easy to install and integrates with VS Code settings, but it can diverge from real Vim on complex commands, macros, and large-file performance. vscode-neovim embeds an actual Neovim process and delegates Normal-mode editing to it, giving near-perfect fidelity and letting you reuse your init.lua/plugins, at the cost of setup complexity and needing nvim installed. Choose VSCodeVim for light modal editing; choose vscode-neovim if you rely on real Vim macros, custom mappings, or an existing config.
Q4. Why is tmux valuable for remote work, and how is detach/attach implemented conceptually?
tmux runs a persistent server process that owns your sessions, windows, and panes; your client terminal merely attaches to it. When your SSH connection drops or you detach with the prefix + d, the server and all child processes keep running, and you reattach later with `tmux attach` to the exact same state. This is what lets long builds, REPLs, and monitoring survive disconnects, and enables session sharing for pair debugging.
Q5. How do login vs interactive shells affect where you put configuration, and what's a common bug?
A login shell (e.g. first SSH login) sources profile files like .zprofile/.bash_profile once; interactive non-login shells source .zshrc/.bashrc every time a new shell opens. PATH and one-time environment exports belong in the profile, while aliases, functions, and prompt setup belong in the rc file. The classic bug is putting an alias in .zprofile (so it doesn't show up in new terminal tabs) or setting PATH in .zshrc so it gets appended repeatedly or doesn't apply to non-interactive contexts like cron.
Q6. What problem does a tiling window manager solve, and what's the trade-off versus a stacking WM?
A tiling WM automatically arranges windows into non-overlapping regions addressed by keyboard, with workspaces/tags for grouping, eliminating manual dragging, resizing, and alt-tab hunting and giving reproducible layouts. The trade-off is a steeper learning curve and friction with apps that assume floating windows (dialogs, some GUIs), which is why most tiling WMs support float exceptions and per-app rules. It's the desktop-level expression of the same keyboard-first, deterministic philosophy as tmux and Vim.
Q7. You edit config on a remote container that only has vi — what's your minimal survival kit?
Enter insert mode with `i`/`a`, escape with Esc, save-and-quit with `:wq`, quit-without-saving `:q!`, delete a line `dd`, undo `u`. For targeted edits, `/pattern` to search, `ciw`/`ci"` to replace a word or quoted value, and `:%s/old/new/g` for a global substitution. Knowing these means you're never blocked on a bastion host or minimal image that lacks nano or a GUI editor.
Q8. Why version-control your dotfiles, and how would you make them portable across machines?
Version-controlling dotfiles (rc files, editor/tmux/git configs) makes any new machine reproducible in minutes, gives you history/rollback, and documents your environment. Portability techniques include a bare git repo or a symlink manager (GNU stow, chezmoi), guarding OS- or host-specific blocks behind conditionals (e.g. `case $(uname)`), keeping secrets out via templating or a separate untracked include, and bootstrapping tool installs from a single script. The goal is idempotent setup: clone, run one command, and the environment is identical.
