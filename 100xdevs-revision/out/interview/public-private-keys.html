Public Key Cryptography

Cohort 2.0 · 10 lessons · SDE-2 (3+ yrs) interview prep

Concept summary — all 10 steps

Banks vs Blockchains: trust models
A bank is a centralized trusted intermediary that owns the ledger and can reverse or block entries; a blockchain is a decentralized append-only ledger where no single party is trusted and correctness is enforced by cryptography plus consensus. The interview angle is recognizing that public-key crypto is what replaces the institution: ownership and authorization become 'can you produce a valid signature' rather than 'is your name in the bank's database'.
Bits and bytes
A bit is a binary digit and a byte is 8 bits; keys, hashes, and addresses are just fixed-length byte arrays (e.g., a 256-bit private key is 32 bytes). This matters because entropy, key strength, and address length are all measured in bits/bytes, and getting sizes wrong (leading-zero bytes, endianness) is a classic source of wallet bugs.
Encodings (hex, base64, base58)
Encoding maps raw bytes to a printable text representation and is fully reversible with no secret — hex, base64, Base58Check (Bitcoin), and 0x-hex (Ethereum) are common. The interview trap is conflating encoding with encryption or hashing: encoding provides zero security, and Base58/Base58Check exist mainly to drop visually ambiguous characters and add a checksum to catch typos.
Hashing vs encryption
Hashing (SHA-256, Keccak-256) is a one-way, deterministic, fixed-size digest used for integrity, commitments, and deriving addresses; encryption is reversible using a key and is used for confidentiality. Interviewers probe this because people say 'encrypt the password' when they mean 'hash it' — hashes can't be decrypted, and preimage/collision resistance is the property that matters.
Asymmetric encryption
Asymmetric crypto uses a mathematically linked keypair: one key does an operation the other reverses, so you can encrypt-to-public/decrypt-with-private, or sign-with-private/verify-with-public. It solves the key-distribution problem of symmetric crypto; note that blockchains use it almost entirely for digital signatures (authorization/authentication), not for encrypting transaction contents.
Creating a public/private keypair
The private key is a large random number; the public key is derived by elliptic-curve point multiplication on secp256k1 (public = privateKey · G). This is a one-way function due to the elliptic-curve discrete log problem, so you can safely publish the public key/address, and interviewers expect you to know the derivation direction is irreversible and that address = hash(public key).
How transactions work on the blockchain
A sender constructs a transaction, signs its hash with their private key (ECDSA on secp256k1), and broadcasts it; nodes recover/verify the signature against the sender's public key/address and check the nonce and balance before including it in a block. The key insight is that the private key never leaves the wallet — the signature proves authorization, and the nonce prevents replay/double-spend.
Hierarchical Deterministic (HD) wallets
An HD wallet (BIP32/39/44) generates an entire tree of keypairs deterministically from a single seed, itself represented as a human-readable mnemonic phrase, using derivation paths like m/44'/60'/0'/0/0. This means one backup restores every address, you can generate a fresh address per transaction for privacy, and hardened vs non-hardened derivation is a common deep-dive question.
How to create a wallet
In practice: gather secure entropy → encode as a BIP39 mnemonic → stretch to a seed (PBKDF2) → compute the BIP32 master key → derive child keys along a path → hash the public key into an address. Libraries like ethers.js or bitcoinjs do this, but SDE-2s should know the pipeline so they understand what a seed phrase actually protects and why entropy quality is critical.
Assignments / hands-on practice
Applied exercises to cement the pipeline — generate a keypair, derive an address, sign and verify a message, and reconstruct keys from a mnemonic. The point is to move from theory to being able to debug real key/signature/encoding mismatches.

Real-world example

Building 'Sign-in with Ethereum' (wallet-based auth) for a web app

Your team wants passwordless login: instead of storing password hashes, users authenticate by proving they control a wallet address. The flow ties the whole topic together. The backend issues a random nonce challenge; the frontend asks the user's HD wallet (a keypair derived from their BIP39 seed at path m/44'/60'/0'/0/0) to sign the message with its secp256k1 private key. The server receives the signature (hex-encoded), recovers the public key, hashes it to an address, and checks it matches the claimed account — no private key ever crosses the wire, and no shared secret is stored. Along the way you hit every concept: you debug an 'invalid signature' bug caused by encoding the message as UTF-8 vs hex bytes (encodings), you realize you must hash-then-sign not encrypt (hashing vs encryption), you add the nonce to prevent replay (transaction/signature semantics), and you explain to the team why compromising the DB leaks nothing because there are no secrets to steal — only public addresses. This is exactly the centralized-trust-to-cryptographic-trust shift from the Banks vs Blockchains framing, applied to plain auth.

Interview Q&A (SDE-2)

Q1. Why do blockchains use asymmetric cryptography for signatures rather than encrypting transactions?
The problem being solved is authorization and integrity, not confidentiality — the network must publicly verify that the rightful owner approved a transaction. A digital signature lets anyone verify authenticity using the public key while only the private-key holder could have produced it, and the transaction data stays public so every node can validate balances and consensus. Encrypting contents would defeat the whole point of a transparent, independently verifiable ledger.
Q2. Explain the relationship between a private key, public key, and address, and why you can't work backwards.
The private key is a random 256-bit number; the public key is derived via elliptic-curve point multiplication (public = priv · G on secp256k1); the address is typically a hash of the public key (e.g., Keccak-256 then truncate for Ethereum). Working backwards is infeasible: recovering the private key from the public key requires solving the elliptic-curve discrete logarithm problem, and recovering the public key from an address requires inverting a cryptographic hash. This one-way chain is why publishing your address or public key is safe.
Q3. What is the difference between hashing and encryption, and where does each appear in a wallet?
Hashing is a one-way deterministic function with fixed-size output used for integrity and derivation — it can't be reversed. Encryption is reversible using a key and provides confidentiality. In a wallet, hashing derives addresses from public keys, hashes transactions before signing, and stretches mnemonics into seeds (PBKDF2); encryption typically only appears when the keystore file itself is encrypted at rest with the user's password. Calling address derivation 'encryption' is a red flag in an interview.
Q4. What does an HD wallet give you over generating independent keypairs, and what's the risk model?
An HD wallet derives an unbounded tree of keys deterministically from one seed (BIP32/39/44), so a single mnemonic backs up every current and future address, and you can hand out a fresh receiving address per transaction to improve privacy. The trade-off is blast radius: the seed is a single point of failure — whoever has the mnemonic controls all derived accounts forever. A subtle risk is that leaking an extended public key (xpub) plus one non-hardened child private key can expose the parent, which is why the standard uses hardened derivation for account-level nodes.
Q5. Walk through how a transaction is authorized and why a nonce is needed.
The sender builds the transaction, computes its hash, and signs the hash with their private key using ECDSA; the signature and transaction are broadcast. Validating nodes recover the signer's public key from the signature, derive the address, and confirm it owns the funds and that the included nonce is the next expected value. The nonce (a per-account monotonically increasing counter) prevents replay and double-spending — without it, a valid signed transaction could be rebroadcast repeatedly.
Q6. A signature verification is failing in your app even though the key is correct. How do you debug it?
Almost always it's a byte/encoding or hashing mismatch, not the key. Check that both sides agree on the exact message bytes (raw bytes vs UTF-8 vs hex-decoded), that any prefix/domain-separation (like Ethereum's personal_sign prefix or EIP-712 typed-data hashing) is applied identically, and that the same hash function is used before signing. Also verify signature format (r,s,v vs DER), the recovery id/chain id, and that leading-zero bytes weren't stripped from the key or hash.
Q7. Why is encoding (hex/base58) not a security mechanism, and why does Bitcoin use Base58Check?
Encoding is a reversible, keyless representation of bytes as text — anyone can decode it, so it provides zero confidentiality or integrity by itself. Base58Check is chosen for addresses because it removes visually ambiguous characters (0/O, I/l) to reduce transcription errors and appends a checksum so a mistyped address fails validation instead of sending funds into a black hole. It's about human-safety and typo detection, not cryptographic protection.
Q8. How much entropy does a private key or mnemonic need, and why does it matter?
A secp256k1 private key has ~256 bits of entropy, and BIP39 mnemonics encode 128–256 bits (12–24 words) plus a checksum. High entropy is what makes brute-forcing or guessing keys computationally infeasible across the entire address space. The practical failure mode is weak randomness — using a non-CSPRNG, a predictable seed, or low entropy — which has led to real fund thefts; key generation must use a cryptographically secure random source.
