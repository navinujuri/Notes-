Concept summary — all 15 steps

Pick the stack (edge-first TS monorepo)
Hono as the backend framework running on Cloudflare Workers, PostgreSQL via Prisma with connection pooling (Prisma Accelerate), Zod for validation, a shared 'common' package for types, and a React (Vite) frontend. The interview angle is justifying an edge/serverless runtime: no persistent processes, Web-standard Request/Response, cold-start-free isolates, and the constraints that flow from that (DB connections, secret handling).
Scaffold the Hono backend
Bootstrap with `npm create hono@latest`, choose the cloudflare-workers template, and get `wrangler.toml` plus the fetch-handler entrypoint. Matters because the Worker export is a Web `fetch` handler, not a long-running Node server, so there is no `process.env`, no filesystem, and startup code runs per-isolate.
Structure route handlers
Define handlers on the Hono app (`app.get/post`), organize with sub-routers and `app.route()` for mounting, and type the app with generics for `Bindings`/`Variables`. The point is clean separation of concerns and correctly typing `c.env` (secrets/bindings) and `c.get/c.set` (per-request context) up front.
Set up Prisma + connection pooling
Model `User` and `Post` in `schema.prisma`, then connect through Prisma Accelerate (or a pgBouncer-style pooler) rather than a raw connection string. This is the crux interview point: Workers spin up many short-lived isolates that can't hold TCP connections, so a direct Postgres connection exhausts the DB — a pooling proxy over HTTP is required at the edge.
Auth routes (signup/signin + JWT)
Implement `/signup` and `/signin` that persist users and return a signed JWT (Hono's `sign`/`verify` from `hono/jwt`). Interview angle: stateless JWT fits serverless because there is no shared session store, and the signing secret comes from `c.env`, never hardcoded.
Auth middleware
A middleware reads the `Authorization` header, verifies the JWT, and stashes the userId via `c.set('userId', …)` so downstream handlers read it with `c.get`. Matters for showing you understand Hono's middleware chain (`await next()`), centralized auth, and returning 401 early instead of duplicating checks per route.
Blog CRUD routes
Protected routes for create, update, get-by-id, and a bulk/list endpoint, scoped to the authenticated author for writes. Depth here is authorization (author-only edits), and shaping list responses (select only needed fields, add pagination) rather than returning full rows.
Manual testing via Postman
Exercise each endpoint with Postman: set the Bearer token, craft JSON bodies, and confirm status codes and payloads before touching the frontend. It's the fast feedback loop that isolates backend bugs from frontend ones and validates the auth flow end to end.
Deploy to Cloudflare Workers
Ship with `wrangler deploy` and inject secrets via `wrangler secret put` (never committed), exposed at runtime through `c.env`. The interview point is the serverless deploy model — global edge distribution, no server to provision, and env/secret management that differs from a Node `.env` file.
Zod input validation
Validate request bodies with Zod schemas and derive TypeScript types via `z.infer` so the type and the runtime check share one definition. Matters because untrusted input must be validated server-side regardless of client checks, and `z.infer` kills type/validation drift.
Create the shared 'common' package
Extract the Zod schemas and inferred types into a standalone package (published to npm or wired as a workspace) so backend and frontend consume one source of truth. Trade-off to discuss: eliminates FE/BE contract drift, but adds versioning/publish friction — a monorepo workspace often beats a published package for a solo/small team.
Consume common in the backend
Install the common package and import its schemas to validate inputs, replacing locally-defined Zod objects. Reinforces the single-source-of-truth contract and that the same schema now guards both ends of the wire.
Initialize the frontend (Vite + React + TS)
Scaffold a Vite React-TS app, add Tailwind, and install the common package plus Axios. The relevant point is a typed client that reuses backend types, so API request/response shapes are checked at compile time.
Add react-router-dom
Wire client-side routing with `BrowserRouter`, `Routes`, and `Route` for pages like `/signup`, `/signin`, `/blogs`, and `/blog/:id`. Matters for SPA navigation without full reloads and for understanding route params, nested layouts, and protecting authenticated routes.
Build the components
Compose UI from reusable components (Auth form, Appbar, BlogCard, BlogList, full-post view), fetch through Axios with the JWT in headers, and type props/responses using the shared package. The angle is component decomposition, loading/skeleton states, and reusing the backend contract on the client.

Real-world example

Shipping a Medium-style blogging platform on the edge

You're an SDE-2 building a multi-author blogging site. The backend is a Hono app deployed to Cloudflare Workers, talking to managed Postgres through Prisma Accelerate so thousands of short-lived Worker isolates never exhaust the database's connection pool. Users sign up and sign in against `/signup` and `/signin`, which return a JWT signed with a secret stored via `wrangler secret`; an auth middleware verifies that token and injects `userId` into the request context for every protected blog route. All request bodies are validated with Zod schemas that live in a shared `common` package — the exact same schemas the React frontend imports, so the 'create post' shape can't silently diverge between client and server. The React app (Vite + react-router-dom) renders a blog list and per-post pages, calling the API with Axios and the Bearer token. When a bug appears — say, published posts show a stale author name — you reproduce it in Postman first to confirm whether it's a backend `select`/join issue or a frontend problem, then fix it in one place because the types are shared.

Interview Q&A (SDE-2)

Q1. Why do you need Prisma Accelerate (or a connection pooler) when running Prisma on Cloudflare Workers, instead of a plain DATABASE_URL?
Workers execute in many ephemeral, isolated V8 contexts that can't maintain persistent TCP connections and may spin up per request. Pointing each isolate at Postgres directly rapidly exhausts the database's max connections. Accelerate (or a pgBouncer-style proxy) sits in front, multiplexes/pools connections, and lets the Worker talk to it over HTTP — which is the only connection model the edge runtime reliably supports. It also adds optional query caching.
Q2. Why put the Zod schemas in a shared package rather than defining them separately in backend and frontend?
It creates a single source of truth for the API contract: the same schema validates input on the server and typechecks forms/requests on the client, and `z.infer` gives both sides identical TypeScript types. The trade-off is publishing/versioning overhead — every change means a version bump and reinstall. For a small team a monorepo workspace (pnpm/npm workspaces) gives the same benefit without the npm publish cycle, which is usually the better call.
Q3. Why JWT for auth here rather than server-side sessions?
Serverless/edge Workers have no shared, long-lived memory or session store, so stateless JWTs fit naturally — the token itself carries the userId and is verified with a secret from `c.env` on each request, no DB or cache lookup needed. The cost is that you can't easily invalidate a token before it expires; mitigations are short TTLs plus refresh tokens, or a denylist if hard logout is required.
Q4. How is Hono different from Express, and why does it matter for this project?
Hono is built on the Web-standard `Request`/`Response` and `fetch` handler model, so it runs natively on Cloudflare Workers, Deno, and Bun — Express assumes the Node HTTP server and `req`/`res` objects. Practically, that means no `process.env` (secrets come from `c.env`), context via `c.get/c.set` instead of mutating `req`, and a tiny footprint suited to cold-start-sensitive edge runtimes.
Q5. Where should validation live — client, server, or both?
Both, but the server is non-negotiable. Client-side Zod gives fast UX feedback, but the client is untrusted and can be bypassed, so the backend must re-validate every request body before it hits the database. Sharing one schema across both means you get the UX win and the security guarantee without maintaining two definitions.
Q6. How would you implement the 'list all blogs' endpoint so it scales?
Return only the fields the list view needs via Prisma `select` (id, title, snippet, author name, date) rather than full post bodies, and add pagination. Cursor-based pagination (using the last seen id/createdAt) is preferable to offset/limit for large or frequently-changing datasets because offsets get slower deep into the list and can skip/duplicate rows when data shifts. Add an index on the sort column.
Q7. How do you manage secrets like the JWT signing key and database URL in a Workers deployment?
They are set with `wrangler secret put` (or the dashboard) and accessed at runtime through the typed `c.env` bindings — never committed to the repo or read from a `.env` shipped in the bundle. In `wrangler.toml` you declare the binding names; the values are encrypted at rest by Cloudflare, which keeps them out of source control and the client bundle.
