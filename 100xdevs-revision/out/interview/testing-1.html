Testing in the MERN Stack

Cohort 2.0 · 13 lessons · SDE-2 (3+ yrs) interview prep

Concept summary — all 13 steps

Testing in Node.js + Express
Establishes the testing model for server-side JS: a runner (Jest/Vitest) provides describe/it/expect, lifecycle hooks (beforeEach/afterAll), and assertions, while Node code under test is just imported and invoked. Interview angle: know that Express handlers are plain functions of (req, res, next), so most logic can be tested without a network by injecting fake req/res or by driving the app with an HTTP client like supertest.
Testing a simple app (pure unit tests)
Start with pure functions (validators, formatters, business rules) that have no I/O — deterministic input/output, no mocks needed. Interview angle: pure functions are the cheapest, fastest, most valuable tests; a strong sign of good architecture is that core logic is extracted out of route handlers so it can be unit-tested in isolation.
Testing an Express app (supertest)
Use supertest to send real HTTP requests against the Express `app` object (not a bound port) and assert on status codes, headers, and JSON bodies. Interview angle: you export `app` separately from `server.listen()` precisely so tests can `request(app).get('/...')` without occupying a port, enabling parallel test files.
Slightly more complex endpoint
Cover routes with path params, query strings, request bodies, auth middleware, and multiple response branches — asserting both happy paths and error paths (400/401/404/500). Interview angle: interviewers probe whether you test the failure and edge cases (missing fields, invalid IDs, unauthorized) rather than only the 200 case, since that's where real bugs hide.
Moving from Jest to Vitest
Vitest offers a near drop-in Jest-compatible API (describe/it/expect, `vi` instead of `jest`) with first-class ESM/TypeScript support, faster startup via Vite's transform pipeline, and shared config with the app's Vite setup. Interview angle: be able to articulate the migration motivation — native ESM without babel/ts-jest gymnastics, watch-mode speed, and unified tooling — plus the main gotcha that `jest.*` calls and some globals config must be swapped to `vi.*`/vitest config.
Simple Express project with Vitest
Wire Vitest into a real project: `vitest.config.ts` (test.environment 'node', globals, setup files), npm scripts for run/watch/coverage, and organizing test files alongside or under a __tests__ folder. Interview angle: know `test.globals: true` vs explicit imports, and that Node vs jsdom environment must match what you're testing (backend = node).
Adding a database
Introduces integration tests that hit a real (test) database — seeding data, isolating state per test (transactions/truncation or a fresh in-memory/containerized DB), and cleaning up in hooks. Interview angle: the key trade-off is realism vs speed/flakiness; discuss test DB isolation strategies (mongodb-memory-server, testcontainers, per-test transaction rollback) and why sharing state across tests causes order-dependent flakiness.
Mocking dependencies
Replace external modules/collaborators (DB client, third-party SDK, mailer, fetch) with `vi.mock('module')` so a unit test exercises only the code under test without real I/O. Interview angle: mock at boundaries you don't own (network, filesystem, payment APIs); over-mocking your own code produces tests that assert implementation details and break on every refactor.
Deep mocking
When a dependency exposes nested objects/chained methods (e.g. a Prisma client or an ORM query builder), a shallow mock isn't enough — deep mocking auto-stubs the whole object tree (via `mockDeep` from vitest-mock-extended or recursive auto-mocks). Interview angle: recognize deep mocks are powerful but brittle and can hide contract drift, so they're best for chained fluent APIs where hand-stubbing every level is impractical.
Mocking return values
Control what a mock returns per scenario: `mockReturnValue`/`mockResolvedValue`/`mockRejectedValue` for defaults and `mockReturnValueOnce` for sequenced calls, so you can simulate success, empty results, and failures. Interview angle: use `mockResolvedValueOnce` chains to test retry/pagination logic, and always assert both the mocked outcome's effect and that the mock was called with the expected args.
Spies vs Mocks
A mock replaces an implementation entirely; a spy (`vi.spyOn(obj,'method')`) wraps a real method to record calls while optionally keeping or overriding behavior via `.mockImplementation`. Interview angle: use a spy when you want the real behavior to run but still assert it was called (e.g. verifying a logger fired); use a full mock when the real behavior is expensive or has side effects — and remember to restore spies (`mockRestore`/`restoreAllMocks`) to avoid leaking state across tests.
Adding a CI/CD pipeline
Run the test suite automatically on every push/PR (GitHub Actions: install, lint, test with coverage, block merge on failure) and gate deployment on green. Interview angle: discuss required status checks as merge gates, caching node_modules for speed, running tests against a service container (e.g. a Postgres/Mongo service) for integration tests, and separating fast unit tests from slower e2e stages.
Unit vs integration vs e2e tests
Unit tests isolate a single unit with mocked collaborators (fast, many); integration tests verify several real components together, e.g. route + DB (slower, fewer); e2e drive the full deployed stack through the UI/API like a user (slowest, fewest). Interview angle: articulate the test pyramid — push volume to the base for fast feedback, reserve e2e for critical user journeys — and the trade-off between confidence/realism and speed/flakiness/maintenance cost.

Real-world example

Testing a MERN order-checkout API

You own the `/api/orders` endpoint in a MERN backend: it validates a cart, calls a Stripe SDK to charge a card, writes an order document to MongoDB, and emails a receipt. You write pure unit tests for the cart-total/validation logic, then supertest-driven tests hitting the Express `app` for the POST route asserting 201 on success and 400/402 on bad input or declined payment. Stripe and the mailer are external boundaries you don't own, so you `vi.mock` them and use `mockResolvedValueOnce` to simulate a successful charge and `mockRejectedValueOnce` to simulate a decline; you use `vi.spyOn` on the logger to assert the failure was logged without silencing it. For the DB you run integration tests against mongodb-memory-server, seeding a product and truncating between tests so runs are order-independent. Finally a GitHub Actions workflow runs lint + unit + integration tests with coverage on every PR as a required merge check, and a single e2e test drives the real checkout flow end-to-end before deploy — giving you the classic pyramid: many fast unit tests, fewer integration tests, one high-value e2e.

Interview Q&A (SDE-2)

Q1. Why do you export the Express `app` separately from `server.listen()`, and how does that help testing?
Separating them lets tests import the `app` object and pass it directly to supertest (`request(app)`) without binding to a real port. This avoids port conflicts, lets test files run in parallel, and keeps tests fast since no real network socket is opened. The `listen()` call lives in a thin entrypoint that's only used in production/dev.
Q2. What's the difference between a spy and a mock in Vitest, and when do you use each?
A mock (`vi.fn()` or `vi.mock`) replaces an implementation entirely; a spy (`vi.spyOn(obj,'method')`) wraps an existing real method so you can observe calls while, by default, keeping the original behavior. Use a spy when you want the real code to run but still assert it was invoked (e.g. a logger or analytics call); use a full mock when the real behavior is slow, non-deterministic, or has side effects like network/DB access. Always restore spies with `mockRestore`/`restoreAllMocks` so behavior doesn't leak into other tests.
Q3. How do you decide what to mock versus what to exercise for real?
Mock at boundaries you don't own or that are expensive/non-deterministic: third-party APIs, payment gateways, email, network, clock, filesystem. Exercise your own business logic and, where feasible, your database via integration tests, because mocking things you own tends to assert implementation details and makes tests break on harmless refactors. The rule of thumb is to mock I/O and side effects, not your domain logic.
Q4. What are the trade-offs of the test pyramid (unit vs integration vs e2e)?
Unit tests are fast, isolated, and numerous, giving pinpoint feedback but low confidence that components work together. E2e tests exercise the whole stack like a real user, giving the highest confidence but are slow, flaky, and expensive to maintain. Integration tests sit in between (e.g. route + real DB). The pyramid says keep most tests at the fast base and reserve a small number of e2e tests for critical journeys, balancing confidence against speed and flakiness.
Q5. How do you keep database-backed tests fast and non-flaky?
Isolate state so tests don't depend on each other's order: use a dedicated test DB (mongodb-memory-server or a testcontainer), and reset between tests via truncation, dropping collections, or wrapping each test in a transaction that rolls back. Seed exactly the data each test needs in `beforeEach` and clean up in `afterEach`/`afterAll`. Avoid shared mutable global state, and in CI run the DB as a service container so integration tests have a real engine to hit.
Q6. Why migrate from Jest to Vitest, and what breaks in the migration?
Vitest gives native ESM and TypeScript support without babel or ts-jest configuration, faster startup and watch mode via Vite's transform pipeline, and shared config with a Vite-based frontend. The API is largely Jest-compatible, so the main changes are swapping the `jest` global for `vi` (`vi.fn`, `vi.mock`, `vi.spyOn`), moving config into `vitest.config.ts`, and setting `test.environment` (node for backend). Watch out for Jest-specific globals, timer APIs, and module-mock hoisting semantics that differ subtly.
Q7. What is deep mocking and when is it justified?
Deep mocking auto-stubs an entire nested object graph — every property and chained method returns a mock recursively — typically via `mockDeep` from vitest-mock-extended. It's justified for dependencies with fluent/chained APIs or large surface areas (ORM/query builders, complex SDK clients) where hand-stubbing each level is impractical. The downside is brittleness and that it can hide real contract changes, so prefer shallow, explicit mocks for simple collaborators and reserve deep mocks for genuinely deep interfaces.
Q8. What should a CI pipeline enforce for a tested MERN service?
On every push/PR it should install dependencies (with caching), run lint, type-check, and the test suite with coverage, and be configured as a required status check so failing tests block merge. Integration tests should run against a service container (Mongo/Postgres) rather than mocks, and the slower e2e stage can be separated so unit feedback stays fast. Deployment is gated on a green pipeline, ideally with coverage thresholds and separate fast/slow stages to keep developer feedback quick.
