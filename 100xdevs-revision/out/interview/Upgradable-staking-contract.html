Upgradable staking contract

Cohort 3.0 · 6 lessons · SDE-2 (3+ yrs) interview prep

Concept summary — all 6 steps

Web3 cohort framing
Sets the project context: build a production-grade staking dApp where users lock an ERC-20 to earn rewards, and where the contract logic must be changeable after launch. The interview angle is understanding *why* upgradeability is a first-class concern on-chain, where a deployed bug can permanently trap user funds.
Pre-requisites (EVM immutability)
You need Solidity fundamentals plus the EVM mental model: deployed bytecode is immutable and each contract's state lives at its own address, so there is no in-place patching. This is the whole motivation for the topic — interviewers probe whether you understand that 'upgrading' really means redirecting calls, not editing code.
Initialize the staking contract
Write the v1 staking logic (stake, unstake, rewards accounting) as a normal contract, then hit the core problem: if there's a bug or you need a new feature, the address is fixed, the code can't change, and all staked balances are tied to that address. This makes the pain concrete before introducing the fix — a common interview setup ('you shipped v1, now what?').
Fixing the problem (why naive redeploy fails)
The obvious fix — deploy a new contract and migrate state — breaks everything: the address changes (all integrations/UIs/allowances point at the old one), and migrating every staker's balance is gas-heavy, error-prone, and needs downtime. This step also surfaces the constructor problem, motivating separation of *storage* from *logic* so state can persist across upgrades.
Proxy contract (delegatecall pattern)
A proxy holds the storage and funds at a stable address and uses `delegatecall` to run an implementation contract's code in the proxy's storage context; upgrading just repoints the proxy to a new implementation. Because logic executes against the proxy's storage, constructors don't work (they'd run against the implementation), so you use an `initializer` function guarded against re-entry. Expect deep questions on delegatecall semantics, storage layout preservation, and EIP-1967 slots.
Moving further (production hardening)
Real deployments use OpenZeppelin's upgradeable contracts and Upgrades plugin: `__init` chains instead of constructors, `__gap` storage reserves, and a choice of Transparent vs UUPS proxy. The senior-level angle is governance and security — upgrade rights behind a multisig + timelock, initializing the implementation to block takeover, and validating storage-layout compatibility on every upgrade.

Real-world example

Patching a live staking pool without moving user funds

You ship StakingV1: users deposit an ERC-20, and rewards accrue per block. A week and $2M TVL later, QA finds the reward math under-pays anyone who unstakes partially. If StakingV1 were a plain contract you'd be stuck — you'd deploy StakingV2 at a new address and have to migrate every staker's balance and reward checkpoint, forcing users to re-approve and re-stake while the pool is paused. Instead you deployed V1 behind a UUPS proxy: the proxy address holds all the staked tokens and the balance/reward-index mappings, and delegatecalls into the V1 implementation for logic. To fix the bug you write StakingV2 (same storage layout, appended variables only), and the proxy admin — a Gnosis Safe multisig wired to a 48-hour timelock — calls `upgradeToAndCall` to repoint the proxy at V2 and run a small `reinitializer` migration. Users keep the same address, the same balances, and never touch their funds; the debugging work is verifying storage compatibility (no reordered/removed slots, gaps respected) and confirming the implementation itself was initialized so no attacker can hijack it.

Interview Q&A (SDE-2)

Q1. Why can't you just edit or redeploy a deployed staking contract to fix a bug?
Contract bytecode is immutable once deployed, and each contract's state (staked balances, reward indexes) lives at its own address. Redeploying gives you a new address, which breaks every UI, integration, and token allowance pointing at the old one, and forces a full state migration of every staker — gas-heavy, slow, and risky. Upgradeability solves this by keeping a stable address and stable storage while swapping only the logic.
Q2. How does a proxy contract actually enable upgrades — what's the role of delegatecall?
The proxy stores all state and holds the funds, but forwards calls to an implementation contract via `delegatecall`, which executes the implementation's code in the *proxy's* storage context (msg.sender and msg.value are also preserved). So logic lives in the implementation but reads/writes the proxy's storage. Upgrading is just changing the stored implementation address (the EIP-1967 logic slot), after which the same proxy runs new code against the same state.
Q3. Why do upgradeable contracts use an initializer instead of a constructor?
A constructor runs once at deployment against the deploying contract's own storage. With a proxy, the implementation is deployed separately, so its constructor would set state in the *implementation's* storage, not the proxy's — the proxy would never see it. Instead you expose an `initialize()` function that the proxy calls via delegatecall so it writes to proxy storage, and you guard it with an `initializer`/`reinitializer` modifier so it can only run once (and once more per version on upgrades).
Q4. Compare Transparent proxy vs UUPS — when would you pick each?
In a Transparent proxy the upgrade logic lives in the proxy, and it routes admin vs user calls to avoid selector clashes, which makes each proxy heavier and slightly more gas per call. In UUPS the upgrade function (`upgradeTo`) lives in the *implementation*, so proxies are lighter and cheaper, but if you ship an implementation that omits or mis-guards the upgrade function you can permanently brick upgradeability. UUPS is now the common default for its lower cost, provided you protect `_authorizeUpgrade` with access control and never lose the upgrade path.
Q5. What is a storage collision in proxy upgrades and how do you prevent it?
Because the implementation runs against the proxy's storage, slot N must mean the same variable in every version; if V2 reorders, inserts, or changes the type of an earlier variable, it corrupts existing state (e.g., a staker's balance reading from a slot now holding a reward rate). Prevent it by only *appending* new variables, never removing or reordering, reserving a `uint256[__gap]` at the end of base contracts, and using EIP-1967's pseudo-random slots for proxy-internal fields (implementation/admin) so they never collide with logic variables. OpenZeppelin's Upgrades plugin validates layout compatibility on each upgrade.
Q6. What are the main security risks unique to upgradeable staking contracts?
The biggest is centralization of the upgrade key — whoever controls upgrades can swap in malicious logic and drain all staked funds, so upgrade rights belong behind a multisig plus a timelock that gives users time to exit. A second is the uninitialized-implementation attack: if the implementation contract is left uninitialized, an attacker can call its initializer, become owner, and (in UUPS) trigger a `delegatecall`/`selfdestruct` path to brick the logic — mitigated by calling `_disableInitializers()` in the implementation's constructor. Storage-layout mistakes and forgetting to re-guard the upgrade function round out the list.
Q7. During an upgrade, how do you add a new state variable or run a one-time data migration?
Append the new variable after all existing ones (consuming reserved gap slots) so no prior slot shifts, and never insert it in the middle. For a one-time migration, add a `reinitializer(version)` function that runs new setup logic exactly once for that version and call it atomically via `upgradeToAndCall` so the upgrade and migration land in a single transaction. Always dry-run the layout check (e.g., OZ `validateUpgrade`) before deploying.
Q8. How does the proxy know which function to route, and what is the function-selector clash problem?
The proxy's `fallback` catches any call whose selector it doesn't itself implement and delegatecalls it to the implementation. A clash happens when a function the proxy defines (e.g., an admin `upgradeTo`) shares the 4-byte selector with a legitimate implementation function, so the proxy intercepts a call meant for the logic. Transparent proxies solve this by splitting behavior by caller (admin vs everyone else); UUPS sidesteps it by keeping upgrade logic in the implementation so the proxy has essentially no selectors of its own.
