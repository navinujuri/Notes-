End to end contract with FE

Cohort 3.0 · 5 lessons · SDE-2 (3+ yrs) interview prep

Concept summary — all 5 steps

Web3 Cohort: the full-stack dApp loop
Sets the frame for the whole topic: a dApp is a smart contract (on-chain state + logic) plus a frontend that reads/writes it through a node RPC and a wallet. The interview angle is being able to trace the request path end to end: FE -> library (ethers/viem) -> JSON-RPC -> node -> EVM -> contract, and back.
Step 1 - Writing the contract
Author the Solidity contract: state variables, functions marked view/pure vs state-changing, events, and access modifiers (onlyOwner, require checks). Compiling it (Hardhat/Foundry/Remix) produces two artifacts you must remember for interviews: the bytecode (deployed to the chain) and the ABI (the JSON interface the FE needs to encode calls); deploying returns the contract address.
Writing the FE
Build the React frontend that talks to the deployed contract: connect a wallet (window.ethereum / MetaMask), instantiate a provider for reads and a signer for writes, and construct a Contract object from address + ABI. The key point is the read/write split: view functions go through a provider via eth_call (free, synchronous-ish), while state changes require a signer, cost gas, and are asynchronous transactions.
Add the ABI + address wiring (glue layer)
Wire the compiled ABI JSON and the per-network contract address into the FE (typically imported from artifacts or an env/config keyed by chainId). This is the load-bearing glue: a stale ABI silently breaks call encoding/decoding, and a wrong address or chain means you're calling the wrong (or nonexistent) contract. Interviewers probe how you keep ABI and address in sync across redeploys and networks.
Assignment
Hands-on task: implement, deploy to a testnet, and drive a contract from the UI to internalize the full loop (connect wallet -> read state -> send tx -> await receipt -> reflect updated state). It matters because the failure modes only show up when you actually deal with pending transactions, reverts, user rejection, and network switching.

Real-world example

A decentralized crowdfunding dApp (contract -> React) on a testnet

You build a Crowdfund contract in Solidity: it stores a goal and totalRaised, exposes a payable contribute() that increments state and emits Contributed(address,uint256), and a view getStatus() returning progress. You compile with Hardhat, deploy to Sepolia, and note the address. On the frontend, users click Connect Wallet (MetaMask); you create a provider for reads (rendering totalRaised via getStatus through eth_call) and, once connected, a signer for writes. contribute() is called through the signer, returns a tx hash immediately, and you await tx.wait() to get the receipt before showing a success state and refreshing the total. You subscribe to the Contributed event (or poll) so other users' contributions update the progress bar live. Debugging this surfaces the real end-to-end concerns: a redeploy changes the address and sometimes the ABI (both must be re-wired), the user may reject the signature, the tx may revert if the goal is already met, and a wallet on the wrong chainId must be prompted to switch to Sepolia before any call succeeds.

Interview Q&A (SDE-2)

Q1. What exactly is the ABI, and why can't the frontend talk to a contract without it?
The ABI (Application Binary Interface) is a JSON description of the contract's functions, their input/output types, and events. Libraries like ethers/viem use it to ABI-encode a function call into the calldata bytes the EVM expects, and to decode the raw return bytes and event logs back into JS values. Without the ABI the client has no idea how to serialize arguments or interpret results, so the address alone is useless; a stale ABI (after changing a signature) causes silent decode failures or reverts.
Q2. Explain the difference between a provider and a signer, and how it maps to reads vs writes.
A provider is a read-only connection to a node over JSON-RPC; it executes view/pure functions via eth_call, which run locally on the node, cost no gas, and return a value synchronously. A signer wraps a wallet/private key and can sign transactions, which is required for any state-changing function because that produces an on-chain transaction that costs gas and must be mined. In practice you read with a provider even when disconnected, but you can only write after the user connects a wallet and you obtain a signer.
Q3. A state-changing call returns a tx hash, not the new value. How do you handle that in the UI?
State changes are asynchronous: the send resolves with a transaction hash once it's in the mempool, but the state isn't updated until the tx is mined. You await tx.wait() (or wait for N confirmations) to get the receipt, then re-read state or trust an emitted event to update the UI. You must also handle the intermediate pending state, user rejection of the signature, reverts (out-of-gas or failed require), and reorg risk for low-confirmation counts; optimistic UI is possible but must roll back if the tx fails.
Q4. How do you keep the contract address and ABI correct across multiple networks and redeploys?
Key the address by chainId in a config/env map (e.g. one address for Sepolia, another for mainnet) and read the wallet's current chainId at runtime to pick the right one, prompting a network switch on mismatch. The ABI should come straight from the build artifacts and be regenerated/re-imported on every redeploy, ideally as part of the build pipeline, so it never drifts from the deployed bytecode. Hardcoding a single address or committing a hand-edited ABI is the classic source of 'works on testnet, broken on mainnet' bugs.
Q5. Why should you never trust the frontend for validation in a dApp, and where do checks belong?
The frontend and even the RPC layer are fully controllable by an attacker who can craft raw transactions bypassing your UI entirely, so any invariant (ownership, limits, balances, deadlines) must be enforced on-chain with require/revert in the contract. FE validation is only a UX convenience to fail fast and save gas. This is why interviewers care about on-chain access control (onlyOwner), reentrancy guards on functions that send ETH, and checks-effects-interactions ordering.
Q6. How do you reflect contract state changes in the UI in near real time?
You subscribe to contract events via contract.on(...) over a WebSocket provider (or filter/query logs), which pushes updates when transactions emit events, and update local state accordingly. Alternatives are polling read functions on an interval, or for richer/historical queries using an indexer like a subgraph. Events are cheaper and more scalable than polling, but you should still reconcile on load by reading current state, since events can be missed during disconnects or reorgs.
Q7. ethers.js vs wagmi/viem — when would you reach for each in the frontend?
ethers/viem are the low-level libraries that handle providers, signers, ABI encoding, and transaction sending; viem is the newer, more typesafe and tree-shakeable option. wagmi is a React layer on top (usually of viem) that gives you hooks like useAccount, useReadContract, and useWriteContract plus wallet-connection state, caching, and re-render handling. For a plain script or non-React context you'd use ethers/viem directly; for a React dApp, wagmi removes a lot of boilerplate around connection lifecycle and query invalidation.
