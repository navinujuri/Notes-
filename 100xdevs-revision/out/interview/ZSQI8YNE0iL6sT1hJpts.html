Code along - Vercel

Cohort 2.0 · 4 lessons · SDE-2 (3+ yrs) interview prep

Concept summary — all 4 steps

Upload Service
An HTTP API that takes a git repo URL, clones it, generates a unique deployment id, uploads every source file to object storage (S3/Cloudflare R2) under that id, then LPUSHes the id onto a Redis queue and writes an initial status. It matters because it decouples the fast client-facing request from the slow, resource-heavy build — the API returns immediately while work is queued for async processing. Interview angle: producer side of a queue, unique/non-enumerable id generation, and treating storage as the source of truth.
Deploy Service
A worker that blocks on the Redis queue (BRPOP / RPOP loop), pulls the popped id's source from storage, runs the build (npm install && npm run build), then uploads the build output (dist/) back to storage and flips status to deployed. It matters because it is the consumer half of the queue and the place untrusted code actually executes. Interview angle: worker/consumer pattern, at-least-once delivery, idempotency, and sandboxing untrusted builds.
Request Handler
An HTTP server that reads the Host header, extracts the subdomain as the deployment id, maps the request path to the corresponding storage key (defaulting to index.html), fetches the artifact, sets the right Content-Type, and streams it back. It matters because it is how a built deployment becomes a live URL. Interview angle: wildcard-DNS subdomain routing, static serving, MIME handling, and where a CDN/cache belongs.
Frontend
A React app where the user pastes a repo URL, calls the upload service, polls deploy status, and surfaces the final live URL (id.yourdomain.com). It matters as the glue that wires the three backend pieces into a usable flow. Interview angle: async status UX (polling vs SSE/WebSockets), optimistic/loading states, and end-to-end integration.

Real-world example

Building a mini-Vercel that deploys a Vite React repo

A developer pastes https://github.com/user/my-vite-app into your dashboard. The upload service clones it, mints id `a1b2c3`, uploads the source to `s3://builds/a1b2c3/*`, LPUSHes `a1b2c3` to Redis, and returns the id. A deploy worker BRPOPs the id, downloads the source, runs `npm install && npm run build` inside an ephemeral container, uploads `dist/` to `s3://dist/a1b2c3/*`, and sets status=deployed. Wildcard DNS routes `a1b2c3.yourdomain.com` to the request handler, which resolves the subdomain to the id, fetches `dist/a1b2c3/index.html`, and serves the live site. Realistic debugging you'd hit: the build fails because a required env var is missing (surface exit code + stderr as status=failed), the site loads but assets 404 because the handler returns the wrong Content-Type or doesn't fall back to index.html for SPA routes, and a worker crash mid-build re-runs the job — which is fine only because the build is idempotent and keyed by id.

Interview Q&A (SDE-2)

Q1. Why put a Redis queue between the upload service and the deploy service instead of building synchronously in the upload request?
Builds are slow and CPU/memory-heavy, so running them inline would block the request and couple your API's throughput to build capacity. A queue lets the upload API return instantly, absorbs bursts, and lets you scale build workers horizontally and retry failures independently. Trade-off: a naive LPUSH/RPOP loop gives at-least-once semantics but loses in-flight jobs if a worker crashes — for durability you'd want a broker with visibility timeouts and a dead-letter queue (SQS/RabbitMQ) or Redis Streams with consumer groups.
Q2. How do you safely run untrusted user code during the build?
Never run npm install/build directly on the host — a malicious postinstall script could exfiltrate secrets or attack your infra. Run each build in an ephemeral, throwaway sandbox: a Docker container or Firecracker microVM with no host mounts, dropped capabilities, CPU/memory/pid limits, restricted network egress, and a hard timeout, destroyed after every build. This contains blast radius and guarantees a clean environment per deployment.
Q3. How does the request handler map a URL like a1b2c3.yourdomain.com to the right files?
A wildcard DNS record (*.yourdomain.com) points every subdomain at the request handler. The handler reads the Host header, takes the subdomain as the deployment id, maps the request path to a storage key such as dist/<id>/<path> (defaulting to index.html), fetches from S3/R2, sets Content-Type from the extension, and streams the body. In production you front this with a CDN keyed on host+path so most requests never hit your origin or storage.
Q4. The queue is at-least-once — how do you keep builds correct when a job can be processed twice?
Make builds idempotent and keyed by deployment id so reprocessing overwrites the same deterministic artifact path rather than creating duplicates. Track status (queued/building/deployed/failed) with atomic transitions, or use a broker with a visibility timeout so only one worker owns a job at a time. Then a worker crash simply makes the job visible again and it re-runs cleanly, without corrupting state or serving a half-built artifact.
Q5. Polling vs WebSockets/SSE for showing deploy status on the frontend — what would you choose?
Polling every few seconds is stateless, trivially survives reconnects, and is perfectly fine for short-lived builds — it just wastes a few requests. WebSockets/SSE give lower-latency updates but add connection state and scaling cost (sticky sessions or a Redis pub/sub fan-out) plus reconnection logic. For an MVP I'd poll; at scale, SSE backed by Redis pub/sub is the pragmatic middle ground and streams build logs well.
Q6. Why store both the raw source and the built output in object storage instead of just building on upload?
Separation lets the deploy worker — a different machine from the uploader — fetch and build independently, and lets you rebuild without re-uploading. The source is the reproducible input; the built dist is what the request handler serves. Keeping artifacts per deployment id also gives you versioning, caching, and instant rollback by simply pointing serving at a previous id.
Q7. How do you generate deployment ids, and why not use an auto-increment integer?
Use a short random id (nanoid, or base62 of a UUID) rather than a sequential int. Sequential ids let anyone enumerate and access other users' deployments by guessing subdomains, and can collide under concurrent uploads. A random id doubles as both the subdomain and the storage key prefix; if you want human-friendly URLs, combine a slug with a random suffix.
Q8. What are the main failure modes across these services and how do you surface them?
Build failures (missing deps or wrong build command), storage failures (partial/failed uploads), and serving 404s (missing file or SPA route). Capture the build's stdout/stderr and exit code, set status=failed, and stream the logs to the frontend so the user can self-diagnose. For serving, fall back to index.html for client-side routes but return a real 404 otherwise; add health checks, timeouts, and a dead-letter queue with alerting so stuck jobs don't hang silently.
