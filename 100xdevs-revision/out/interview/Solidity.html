Solidity

Cohort 3.0 · 4 lessons · SDE-2 (3+ yrs) interview prep

Concept summary — all 4 steps

Web3 Cohort / context
Framing lesson: Web3 apps (dApps) put business logic in smart contracts deployed to a blockchain (e.g. Ethereum), with a JS/TS frontend talking to them via a provider (MetaMask/RPC) and a library like ethers.js/viem. The interview angle is understanding the architecture split: on-chain code is immutable, costs gas, and is the source of truth; the frontend is just a thin client that reads state and submits signed transactions.
What is Solidity
Solidity is a statically-typed, contract-oriented, compiled language for the EVM; a contract is like a class whose fields are persisted in on-chain storage and whose public functions are the callable interface. It matters because its execution model is unusual: code runs deterministically across all nodes, every state write costs gas, there is no floating point, and reverting undoes all state changes in the transaction. Expect questions on storage vs memory vs calldata, visibility (public/external/internal/private), and view/pure vs state-changing functions.
ABIs and Bytecode
The compiler emits bytecode (the EVM instructions deployed on-chain) and the ABI (a JSON description of functions/events/types used to encode calls and decode results off-chain). This is the single most common integration gotcha: the frontend needs the ABI plus the contract address to build a typed interface, function calls are dispatched by a 4-byte selector (first 4 bytes of keccak256 of the signature), and events are read from indexed logs — knowing this explains 'why can't my JS see this function' and 'why is my decoded value garbage.'
Solidity vs JavaScript
Contrast lesson: despite similar syntax, Solidity is statically typed, deterministic, and gas-metered with no runtime GC, no async/promises, no floating point, and integer overflow that reverts (since 0.8.0). The interview point is the mental model shift — you optimize for gas and safety rather than developer ergonomics, all 'I/O' is via transactions/calls, msg.sender/msg.value replace request context, and there is no console/network access from within a contract.

Real-world example

Wiring a React dApp to an ERC-20 token contract

You are building the frontend for a token-gated feature. A teammate deploys an ERC-20 contract in Solidity; you receive its deployed address and its compiled ABI (the artifact JSON from the build). In your React app you instantiate a contract object with ethers.js using address + ABI + a provider, call the view function balanceOf(user) (a free read, no gas, no signature) to gate access, and for a transfer you build a transaction that the user signs in MetaMask — which costs gas and only mutates state if it doesn't revert. When balances come back wrong, you debug the classic pitfalls: the token has 18 decimals so you must scale with parseUnits/formatUnits (Solidity has no floats), the ABI you bundled is stale after a redeploy so the function selector doesn't match, and you listen for the Transfer event via indexed logs to update the UI rather than polling. This one flow exercises the whole topic: contracts as the on-chain source of truth, the ABI/bytecode split, and how Solidity's execution model differs from the JS you write around it.

Interview Q&A (SDE-2)

Q1. What is the difference between the ABI and the bytecode, and which does the frontend need?
Bytecode is the compiled EVM instructions that get deployed and executed on-chain; the ABI is a JSON schema describing the contract's functions, parameters, return types, and events. The frontend needs the ABI (plus the deployed address) to ABI-encode calls and decode responses — it never touches raw bytecode. Bytecode matters at deploy time and for verification; the ABI matters at every interaction, which is why a stale ABI after a redeploy is a frequent bug.
Q2. Explain storage, memory, and calldata in Solidity and their cost implications.
Storage is persistent per-contract state that lives on-chain and is the most expensive to write (SSTORE), memory is a temporary, mutable, zero-initialized scratch space that exists only for the duration of a call, and calldata is a read-only region holding function arguments. You use calldata for external function parameters to avoid copy costs, memory for local temporary data, and storage only when you must persist. Getting the data location wrong on reference types (arrays/structs/strings) is both a common compile error and a gas footgun.
Q3. How does the EVM dispatch a function call, and what is a function selector?
A call's calldata begins with a 4-byte function selector, which is the first 4 bytes of keccak256 of the canonical function signature (e.g. transfer(address,uint256)), followed by ABI-encoded arguments. The contract's dispatcher compares this selector to its known functions and routes accordingly, falling back to receive/fallback if none match. This is why the exact signature (types, no parameter names) matters and why two functions can't share a selector — collisions are a real concern in proxy patterns.
Q4. Why does Solidity have no floating-point numbers, and how do you represent fractional values like token amounts or percentages?
Floating point is non-deterministic across platforms, and the EVM requires bit-for-bit deterministic execution across all nodes, so Solidity only has integers. You represent fractions using fixed-point scaling: tokens use decimals (commonly 18, so 1 token = 1e18 base units) and percentages/rates use basis points (e.g. 250 = 2.5%). The rule of thumb is multiply before you divide to preserve precision, and be explicit about the scaling factor in your interfaces.
Q5. What are the function visibility levels and the meaning of view/pure?
Visibility is public (callable internally and externally, generates a getter for state vars), external (only via transactions/other contracts, cheaper for large args via calldata), internal (this contract and derivatives), and private (this contract only). Orthogonally, view functions read but don't modify state and pure functions neither read nor write state — both are free when called off-chain (eth_call, no gas) but still cost gas if invoked inside a state-changing transaction. Note that visibility is not a security boundary for confidentiality: all on-chain data is publicly readable regardless of 'private'.
Q6. How is error handling in Solidity different from exceptions in JavaScript?
Solidity has no try/catch around your own logic in the JS sense; instead require/revert/assert abort execution and revert all state changes made in the transaction, refunding remaining gas (assert consumes all gas and signals an invariant violation). Since 0.8.0 you get custom errors (cheaper than string messages) and automatic overflow/underflow reverts. You can catch reverts of external calls with try/catch on that specific call, but there is no general-purpose exception propagation, and a revert is all-or-nothing rather than a recoverable throw.
Q7. Since syntax looks JS-like, what execution-model differences actually bite you when moving from JS to Solidity?
There is no event loop or async — everything is synchronous within a single transaction, and cross-contract calls are synchronous message calls, not promises. Every state write costs gas so you optimize storage access aggressively, integers are fixed-width and revert on overflow, and there is no runtime environment: no filesystem, network, timers, or console — 'time' is block.timestamp and 'randomness' must come from an oracle. Context that a JS server gets from a request comes instead from msg.sender, msg.value, and tx metadata, and code is immutable once deployed unless you design an upgrade proxy.
Q8. What does 'the frontend is a thin client' mean in a dApp, and where does trust live?
The smart contract on-chain is the authoritative source of truth and enforces all rules; the frontend just reads state via RPC and constructs transactions the user signs in their wallet. You can never trust client-side validation for correctness because anyone can call the contract directly, bypassing your UI. That means all access control, invariants, and value checks must be enforced in Solidity (modifiers, require), and the JS layer is purely for UX — a critical distinction from traditional web apps where the server you control mediates every request.
