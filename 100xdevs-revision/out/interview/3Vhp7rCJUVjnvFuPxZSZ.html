Custom Hooks

Cohort 2.0 · 5 lessons · SDE-2 (3+ yrs) interview prep

Concept summary — all 5 steps

React hooks fundamentals
Hooks are functions that let function components subscribe to React features (useState, useEffect, useContext, useRef, useReducer) without classes. The Rules of Hooks — call only at the top level and only from React functions — exist because React has no names for hooks; it tracks them positionally as an ordered linked list on the component's fiber, so a conditional or looped call would shift indices and corrupt state. Interview angle: be able to explain the by-call-order mechanism, not just recite the rule.
What a custom hook is
A custom hook is just a function whose name starts with `use` and which composes other hooks to extract reusable stateful logic. The key nuance: it shares logic, not state — every component that calls it gets its own isolated state/effects instance. Interview angle: contrast with HOCs/render props (no wrapper hell, better typing, composable) and stress the `use` prefix is what lets the linter enforce the Rules of Hooks.
Data-fetching hooks
A useFetch/useQuery-style hook encapsulates loading/error/data state plus a dependency-driven request in useEffect, with cancellation via AbortController and cleanup on unmount or dep change. It matters because it centralizes race-condition handling (ignoring stale in-flight responses), retries, and caching in one place. Interview angle: explain the fast-typing race where an earlier request resolves last, and why this is exactly the problem React Query/SWR solve with request dedup + cache.
Browser-API hooks
Hooks like useLocalStorage, useMediaQuery, useOnlineStatus, useEventListener, and useWindowSize wrap imperative browser APIs and subscriptions, owning the add/removeEventListener lifecycle so callers don't leak listeners. Interview angle: prefer useSyncExternalStore for subscribing to external mutable sources (avoids tearing in concurrent React), and handle SSR — guard `window`/`localStorage` access and avoid hydration mismatches by not reading client-only values during the first render.
Performance / timer hooks
useDebounce, useThrottle, useInterval, and useTimeout manage timers with correct clearTimeout/clearInterval cleanup and stable references. The classic trap is the stale closure: an interval captures the callback from the render it was created in; Dan Abramov's pattern stores the latest callback in a ref and reads ref.current inside the tick so the interval isn't torn down and recreated every render. Interview angle: know when to debounce vs throttle and why useRef beats putting the callback in the effect deps.

Real-world example

A responsive product-search dashboard

You're building the search view of an admin dashboard. A useDebounce hook holds back the query ~300ms so you don't fire a request per keystroke. The debounced value feeds a useFetch hook that issues the API call inside useEffect keyed on the query, aborting the previous request with AbortController and setting an `ignore` flag in cleanup so a slow earlier response can't overwrite the latest results (the race you'll actually get a bug report for). A useLocalStorage hook persists recent searches across reloads, and a useMediaQuery hook swaps the results grid to a single column on mobile. Each hook is independently testable and reusable, the components stay declarative, and all timer/listener/request cleanup lives inside the hooks rather than being duplicated across pages — which is precisely the maintainability win an SDE-2 is expected to deliver.

Interview Q&A (SDE-2)

Q1. Why must hooks be called at the top level, never inside conditions or loops?
React doesn't identify hooks by name; it stores them as an ordered list on the component's fiber and matches each call to its slot by call order across renders. If you call a hook conditionally, the order changes between renders and the indices misalign, so state from one hook gets read by another. The top-level rule guarantees a stable call sequence every render, which is what makes positional tracking safe.
Q2. What's the difference between a custom hook and a plain utility function?
A custom hook is a function that itself calls React hooks, so it's bound by the Rules of Hooks and gets linted for them via the `use` prefix; a utility function is pure JS and can be called anywhere. Because it can use useState/useEffect/useRef, a custom hook carries component-scoped state and lifecycle behavior, whereas a util just transforms inputs to outputs. The `use` naming is not cosmetic — it's the signal the linter and React tooling rely on.
Q3. If two components call the same custom hook, do they share state?
No. A custom hook shares logic, not state — each call site instantiates its own independent state and effects, just as if you'd inlined the hook body into that component. If you actually need shared state across components, you need a shared store: Context, a state library, or useSyncExternalStore over an external source. This 'logic not state' distinction is a very common interview probe.
Q4. How do you prevent race conditions in a data-fetching hook?
Inside the fetching useEffect, either abort the prior request with an AbortController or keep a local `let ignore = false` and set `ignore = true` in the cleanup function, then only commit results when not ignored/aborted. This handles the case where the query changes rapidly and an older request resolves after a newer one — without it, stale data overwrites fresh data. In production you'd usually delegate this to React Query/SWR, which add dedup, caching, and revalidation on top.
Q5. Explain the stale-closure problem in a setInterval hook and how to fix it.
If you pass a callback to setInterval inside an effect that runs once, the interval permanently closes over the state/props from that first render, so it always sees stale values. Adding the callback to the effect deps 'fixes' correctness but tears down and recreates the timer on every render, breaking the cadence. The idiomatic fix (Dan Abramov's useInterval) stores the latest callback in a ref, updates ref.current in a separate effect each render, and calls ref.current() inside a single long-lived interval — fresh values without resetting the timer.
Q6. When would you debounce versus throttle, and how do you implement debounce as a hook?
Debounce waits until activity stops (fire once after N ms of silence) — ideal for search-as-you-type; throttle guarantees at most one call per interval — ideal for scroll/resize/mousemove where you want steady sampling. A useDebounce hook typically keeps the debounced value in state and runs a useEffect that setTimeout-s an update keyed on the raw value, clearing the timer in cleanup so each new keystroke cancels the pending update. The clearTimeout in the effect's cleanup is what makes it actually debounce rather than schedule N updates.
Q7. What are the SSR and concurrency pitfalls in browser-API hooks?
On the server there's no `window`, `document`, `matchMedia`, or `localStorage`, so unguarded access throws; and if the first client render produces a different value than the server HTML, you get a hydration mismatch warning and visual flicker. Use lazy state initializers guarded by `typeof window`, defer client-only reads to an effect, and for subscribing to external mutable sources use useSyncExternalStore, which is tear-safe under concurrent rendering and provides a getServerSnapshot for SSR. Rolling your own subscribe+useState can tear (show inconsistent values) during concurrent renders.
Q8. Why did the community move from HOCs and render props to custom hooks?
HOCs and render props both share logic by nesting components, which produces deeply nested 'wrapper hell', clashing prop names, and awkward typing. Custom hooks share logic through plain function composition — no extra tree nodes, explicit inputs/outputs, and clean TypeScript inference. The trade-off is that hooks can't be used conditionally and only work in function components, whereas HOCs can still be useful for cross-cutting concerns that wrap rendering itself (e.g., error boundaries, which must remain class-based).
