SQL databases

Cohort 2.0 · 11 lessons · SDE-2 (3+ yrs) interview prep

Concept summary — all 11 steps

Types of databases
The landscape splits into relational (Postgres, MySQL), document (MongoDB), key-value (Redis, DynamoDB), wide-column (Cassandra), graph (Neo4j), and search (Elasticsearch) engines. The interview angle is knowing that the choice is driven by access patterns and consistency needs, not popularity — you pick the store whose data model matches how you read/write, then justify the trade-off.
Why not NoSQL
NoSQL trades ACID guarantees and ad-hoc joins for horizontal scale and flexible schemas, which becomes a liability when your data is highly relational and you need strong consistency and cross-entity queries. The angle: NoSQL isn't 'schemaless,' it just pushes the schema into application code and denormalizes for a fixed query pattern, so unplanned queries and integrity enforcement get expensive.
Why SQL?
SQL databases give you a declarative query language, enforced schemas, foreign-key integrity, and ACID transactions — invaluable when correctness and complex relationships matter (payments, inventory, users). Modern Postgres/MySQL also scale vertically and via read replicas far further than most teams need, so 'SQL doesn't scale' is usually premature.
Creating a database
A database is the top-level container/namespace holding tables, indexes, and other objects, created with CREATE DATABASE and typically owned by a role with its own connection/permissions. Practically it matters for isolation and access control — separate services or environments should not share one database, and connection strings target a specific db.
Connecting via a driver/library
Applications talk to the DB through a client library (e.g. node-postgres `pg`, `mysql2`) that opens TCP connections, handles the wire protocol, and — critically — manages a connection pool so you reuse a bounded set of connections instead of opening one per request. The interview point is pooling: opening connections is expensive and databases cap max connections, so an unpooled or leaking app will exhaust the server under load.
Defining a table schema
CREATE TABLE declares columns with types, plus constraints — PRIMARY KEY, NOT NULL, UNIQUE, DEFAULT, CHECK, and FOREIGN KEY. The schema is your first and cheapest line of defense for data integrity; pushing invariants into constraints (rather than trusting app code) prevents whole classes of corrupt-data bugs.
Interacting with the database (CRUD)
The four DML operations — INSERT, SELECT, UPDATE, DELETE — plus WHERE/ORDER BY/LIMIT form the core of interacting with data. The SDE-2 nuance is that UPDATE/DELETE without a WHERE clause hit every row, and reads should lean on indexes and pagination rather than pulling full tables.
Querying from a Node.js app
From Node you send SQL through the driver using parameterized queries (`pool.query('... WHERE id = $1', [id])`) and await the result set. Always parameterize — string-concatenating user input is the classic SQL-injection vector, and prepared statements also let the DB cache the query plan.
Building a simple Node.js app
A minimal service wires an HTTP layer (Express/Fastify) to a shared connection pool created once at startup, with route handlers issuing queries and mapping rows to JSON. Key concerns: initialize the pool once (not per request), handle async errors so a failed query doesn't crash the process, and close the pool on graceful shutdown.
Relationships
Relational modeling expresses one-to-one, one-to-many, and many-to-many links via foreign keys — the many-to-many case needs a join/junction table holding both foreign keys. This is where normalization lives: model entities once and reference them, so an interviewer expects you to design a schema (e.g. users, orders, products) and place the FKs correctly.
Joins
JOINs combine rows across tables on a matching condition — INNER (only matches), LEFT/RIGHT (keep all rows from one side, NULLs for the other), and FULL. The depth question is performance: joins should run on indexed keys, and you should be able to reason about why an unindexed join or a join fanning out a many-to-many can explode row counts and latency.

Real-world example

An order-management API for a small e-commerce backend

You're building a Node.js/Express service backed by Postgres for a store. You model users, products, orders, and order_items, where an order belongs to one user (one-to-many) and links to many products through order_items (many-to-many with quantity and price-at-purchase). At startup you create a single `pg` connection pool from an env-based connection string. A `POST /orders` handler wraps an INSERT into orders plus multiple INSERTs into order_items inside a transaction (BEGIN/COMMIT), so a partial failure rolls back and never leaves an order with no items. A `GET /users/:id/orders` handler runs a parameterized query joining orders, order_items, and products to return each order with its line items. When it gets slow in production, you EXPLAIN ANALYZE the query, discover a sequential scan on order_items.order_id, add an index there, and watch the join drop from hundreds of milliseconds to single digits — the concrete payoff of schema design, foreign keys, transactions, parameterization, and indexed joins all in one flow.

Interview Q&A (SDE-2)

Q1. When would you choose a NoSQL document store over a relational database, and what do you give up?
Choose a document store when your access pattern is a single aggregate fetched/written by key (e.g. a user's whole profile document), the schema varies per record, and you need easy horizontal sharding for write scale. You give up cross-entity joins, foreign-key integrity, and multi-document ACID transactions (or get weaker/limited versions), so any query the data wasn't denormalized for becomes an expensive application-side join. The rule of thumb: relational when data is highly connected and correctness-critical; document when it's self-contained and scale/flexibility dominate.
Q2. Why must application queries be parameterized, and what does the driver actually do?
Parameterization separates the SQL text from the data values, so user input can never alter the query structure — this is the primary defense against SQL injection. The driver sends the query with placeholders and the arguments separately; the database treats arguments strictly as values, and can also cache the prepared plan for reuse. String-concatenating input is unsafe even with escaping because edge cases and encoding issues slip through.
Q3. Explain the difference between INNER JOIN and LEFT JOIN with a concrete failure case.
INNER JOIN returns only rows with a match on both sides; LEFT JOIN returns all rows from the left table, filling NULLs where the right has no match. A classic bug: querying users and their orders with INNER JOIN silently drops users who have zero orders, so a 'customers report' undercounts. If you also put a right-table condition in the WHERE clause of a LEFT JOIN, you accidentally turn it back into an inner join because NULLs fail the predicate — the condition belongs in the ON clause instead.
Q4. How do you model a many-to-many relationship, and why not just use an array column?
You introduce a junction (join) table holding a foreign key to each side, often with a composite primary key on the pair plus any relationship attributes (e.g. quantity, added_at). This keeps referential integrity in both directions, lets you index and join efficiently, and supports querying from either entity. An array/JSON column can't enforce that referenced IDs exist, can't be joined or indexed as cleanly, and forces read-modify-write updates that race under concurrency.
Q5. What is a connection pool and why does it matter in a Node.js service?
A pool maintains a fixed set of open database connections that request handlers borrow and return, instead of opening a new TCP connection and re-authenticating per query. It matters because connection setup is expensive and databases enforce a hard max_connections limit, so an unpooled or connection-leaking app exhausts the server and stalls under load. You size the pool to the DB's capacity divided across app instances, and you must release clients back (especially after errors) to avoid leaks.
Q6. How do ACID transactions help, and when would you reach for one?
A transaction groups statements so they commit all-or-nothing (atomicity), leave the DB in a valid state (consistency), don't interfere via isolation, and survive crashes once committed (durability). You reach for one whenever a logical operation spans multiple writes that must not partially apply — creating an order plus its line items, or transferring a balance between two accounts. Without it, a mid-operation failure or concurrent writer can leave orphaned or double-counted data.
Q7. A query joining two large tables is slow. How do you diagnose and fix it?
Run EXPLAIN ANALYZE to see the actual plan and row counts; a sequential scan on the join key or a huge estimated-vs-actual row gap points at the problem. The most common fix is adding an index on the foreign-key/join column (and on WHERE-filter columns) so the planner can use an index or hash join instead of scanning. Also check for accidental row fan-out from a one-to-many join inflating results, stale table statistics, and whether you're selecting more columns/rows than needed — add pagination and select only what you use.
Q8. What does normalization buy you, and when would you deliberately denormalize?
Normalization stores each fact once and references it, which eliminates update anomalies and keeps writes consistent — change a product name in one row, not thousands. You deliberately denormalize for read-heavy paths where joins are too costly, precomputing or duplicating data (e.g. a cached order_total or a materialized view) and accepting the burden of keeping copies in sync. It's a conscious read-vs-write and consistency trade-off, not a default.
