Rust for Solana contracts

Cohort 3.0 · 26 lessons · SDE-2 (3+ yrs) interview prep

Concept summary — all 26 steps

Transactions & IDLs in Solana
A Solana transaction is an atomic bundle of instructions; each instruction names a target program, the exact set of accounts it will touch, and opaque instruction data. The IDL (JSON, generated by Anchor) is the schema describing a program's instructions, accounts, and types so off-chain clients can serialize calls and decode account data. Interview angle: accounts must be declared up front (no dynamic account discovery inside a program), and the transaction either fully succeeds or fully reverts.
Course orientation (Web3 Cohort)
Framing lesson establishing why Rust is the language for Solana programs: no GC pauses, deterministic execution, tight memory control, and compilation to BPF/SBF bytecode that runs in the on-chain VM. The mental shift for a backend engineer is that a Solana program is stateless code; all mutable state lives in externally-owned accounts.
Solana account model & program context
Programs are stateless; state is held in accounts that the runtime loads and passes into the handler as an execution context (program_id, the ordered accounts, and instruction data; in Anchor this is Context<T>). Interview angle: because the caller supplies every account, programs must validate ownership, signer, and PDA-derivation constraints — trusting an account blindly is the classic security bug.
Toolchain installation
Set up rustup (toolchain manager), cargo (build + package manager), the Solana CLI, and Anchor. Know that cargo handles builds, dependency resolution, and tests, and that Solana programs cross-compile to the SBF target rather than a native binary.
Data types in Rust
Scalars (i8..i128/u8..u128, usize, f32/f64, bool, char) and compounds (tuples, fixed-size arrays). Types are fixed-width and known at compile time; integer overflow panics in debug and wraps in release unless you use checked/saturating ops. Solana angle: lamports and token amounts are u64, and fixed widths make binary serialization deterministic.
Conditionals & loops
if/else and match are expressions that return values; loop/while/for cover iteration, and loop can return a value via break. for iterates over anything implementing IntoIterator, which is idiomatic over manual index loops.
Mutable vs immutable variables
let bindings are immutable by default; mut opts into mutation. Distinguish shadowing (a new binding, can change type) from mutation, and let vs const (compile-time constant). Immutability-by-default is what makes the borrow-checker's aliasing rules tractable and code concurrency-safe.
Functions
fn with explicitly typed parameters and return types; Rust is expression-based, so the last expression without a semicolon is the return value. Understand the statement-vs-expression distinction — a stray semicolon changes a return into a unit ().
Ownership of heap variables
Every value has exactly one owner and is dropped (freed) when the owner goes out of scope — no GC, no manual free. Heap types (String, Vec, Box) move on assignment/pass (invalidating the source), while Copy stack types are duplicated. This is the single most-tested Rust concept: it gives memory safety without a runtime, which is why Solana can rely on deterministic, GC-free execution.
Borrowing & references
&T is a shared/immutable borrow, &mut T an exclusive/mutable borrow. The borrow checker enforces many shared XOR one mutable at any time, eliminating data races and use-after-free at compile time. Interview angle: this is aliasing control, not just pointers — the exclusivity rule is what prevents iterator invalidation and concurrent mutation bugs.
Structs
Custom aggregate types with named-field, tuple, or unit forms; they model domain data. In Solana/Anchor, account state is defined as a struct that is serialized into an account's data buffer.
Implementing structs (impl)
impl blocks attach methods and associated functions. Method receivers &self / &mut self / self encode borrow vs mutable-borrow vs consuming access; associated functions like Type::new act as constructors. This cleanly separates data layout from behavior.
Enums
Sum types (tagged unions) whose variants can each carry different data. They model state machines, instruction variants, and mutually-exclusive states precisely — making illegal states unrepresentable rather than validated at runtime.
Pattern matching
match must be exhaustive (compiler enforces every variant is handled), with destructuring, guards, bindings, and _ catch-alls; if let / while let are ergonomic single-pattern forms. Exhaustiveness means adding an enum variant forces you to update every match — a compile-time safety net.
Error handling
No exceptions: recoverable errors use Result<T, E>, absence uses Option<T>, and unrecoverable bugs use panic!. The ? operator propagates errors up by early-returning the Err. In Solana you define a custom error enum mapped to error codes and return Result from handlers so failures cleanly abort the transaction.
Option enum
Option<T> = Some(T) | None is Rust's type-safe replacement for null, forcing callers to handle absence. Combinators (map, and_then, unwrap_or, ok_or) and ? let you compose without unwrap() panics. Interview angle: it moves null-pointer errors from runtime to compile time.
External crates
Dependencies live on crates.io and are declared in Cargo.toml with semver ranges; feature flags gate optional functionality and workspaces group multiple crates. Know how features can bloat or trim a build (e.g., enabling only what a no_std/on-chain target needs).
Generics & trait bounds
Type parameters <T> plus bounds (T: Trait, or where clauses) write code once over many types while constraining what operations are allowed. Rust monomorphizes generics — it generates a specialized copy per concrete type at compile time — so they are zero-cost at runtime, trading binary size for speed.
Generics over structs
struct Wrapper<T> parameterizes data types (e.g., a container). impl<T> blocks add methods generically, and you can add bounded impls (impl<T: Display>) so some methods exist only when T meets a bound.
Generics over enums
Option<T> and Result<T, E> are the canonical generic enums; you can define your own to build reusable, type-safe variant types. Same monomorphization benefits apply.
Traits
Traits define shared behavior (interfaces) with required and default methods and associated types. Dispatch is static via generics (monomorphized, inlined) or dynamic via dyn Trait objects (vtable, runtime cost, but object-safe and heterogeneous). The orphan rule (you must own the trait or the type to impl it) is a common gotcha. Anchor's account handling is built on trait implementations.
Implementing & deriving traits (macros)
You implement traits manually with impl Trait for Type, or auto-generate them with #[derive(Debug, Clone, PartialEq, ...)] via derive macros. Solana/Anchor lean heavily on derives like #[derive(Accounts)] and BorshSerialize/BorshDeserialize to eliminate serialization boilerplate. Interview angle: derive macros are compile-time code generation, not reflection.
Serde
Serde is the general serialization framework: derive Serialize/Deserialize once and target many formats (JSON, YAML, etc.) via format crates. It is primarily an off-chain/client concern (APIs, config, RPC payloads), not the on-chain wire format.
Borsh
Borsh (Binary Object Representation Serializer for Hashing) is a compact, deterministic binary format — the same input always yields the same bytes — which is why Solana uses it for on-chain account and instruction serialization. Determinism is essential where bytes are hashed or must match across validators; contrast with Serde/JSON, which prioritizes flexibility over canonical output.
Iterators
The Iterator trait exposes next() and a large set of lazy adapters (map, filter, take, collect, fold). Adapters do no work until consumed, and the whole chain compiles down to code as fast as a hand-written loop (zero-cost). Distinguish iter() (&T), iter_mut() (&mut T), and into_iter() (owned) by what they borrow or consume.
Lifetimes
Lifetime parameters ('a) are compile-time annotations telling the borrow checker how long references must remain valid, preventing dangling references. Most are inferred via elision rules; you write them explicitly when a function returns a reference tied to an input or when a struct holds a reference. Interview angle: lifetimes describe existing relationships — they never change how long data actually lives.

Real-world example

Building an SPL-token escrow program in Anchor

You're an SDE-2 implementing a token-swap escrow on Solana: Alice deposits token A into a program-controlled vault (a PDA), and the trade settles atomically only when Bob deposits token B. You model the on-chain state as an EscrowAccount struct (maker, mint, amount: u64, plus a State enum: Pending | Completed | Cancelled) and derive BorshSerialize/BorshDeserialize so it round-trips deterministically into the account's data buffer. Each instruction handler receives a Context<T> whose #[derive(Accounts)] struct declares and constraint-checks every account (signer, ownership, PDA seeds) before your code runs. Inside handlers you work with borrows (&mut ctx.accounts.escrow) under the borrow checker, return Result with a custom error enum (e.g., InvalidAmount, WrongState) propagated via ?, and match on the State enum to enforce the state machine so illegal transitions won't compile-past your exhaustiveness check. Off-chain, your TypeScript client uses the generated IDL to build the transaction, while any Rust tooling uses Serde for JSON config and Borsh to encode instruction data. When you later hit a 'reference does not live long enough' error refactoring a helper that returns a slice of an account's data, lifetimes are what the compiler is asking you to annotate.

Interview Q&A (SDE-2)

Q1. Why does Solana use Rust, and how do ownership and borrowing help on-chain?
Solana programs need deterministic, low-latency execution with tight memory control and no GC pauses, and they compile to SBF bytecode. Rust's ownership model frees memory deterministically when owners go out of scope (no runtime, no collector), and the borrow checker guarantees memory safety and no data races at compile time. That means safety without a runtime cost, which matters when every compute unit is metered and execution must be reproducible across validators.
Q2. Why does Solana serialize on-chain data with Borsh instead of Serde/JSON?
Borsh is deterministic and canonical: identical input always produces identical bytes, which is required when data is hashed or must match byte-for-byte across validators for consensus. It's also compact and cheap to (de)serialize, saving account space and compute. Serde is format-agnostic and flexible but doesn't guarantee a single canonical encoding, so it's used off-chain (clients, config, RPC) rather than for on-chain account/instruction data.
Q3. Explain move vs Copy. What happens with String vs i32 when passed to a function?
Copy types (i32, bool, char, small fixed types) are bitwise-duplicated, so the original stays usable after being passed. Non-Copy heap-owning types like String move: ownership transfers to the callee and the original binding is invalidated, preventing double-free. To keep using the value you either borrow (&s), clone (explicit deep copy with a cost), or have the function return ownership back.
Q4. Static vs dynamic dispatch in Rust — when would you choose each?
Static dispatch via generics/trait bounds is monomorphized: the compiler emits a specialized, inlinable copy per concrete type, giving maximum speed at the cost of binary size and no runtime polymorphism. Dynamic dispatch via dyn Trait uses a vtable, adding a pointer indirection but allowing heterogeneous collections and smaller code. On-chain you generally prefer static dispatch for performance and predictability; use dyn only when you genuinely need runtime-varied types behind one interface.
Q5. How does error handling work, and how does ? interact with Result?
Rust has no exceptions: recoverable failures are values of type Result<T, E>, absence is Option<T>, and only true bugs panic!. The ? operator unwraps an Ok/Some or early-returns the Err/None, propagating errors up the call stack cleanly. In a Solana handler you return Result with a custom error enum; returning Err aborts and reverts the entire transaction atomically.
Q6. What problem do lifetimes solve, and why don't you usually write them?
Lifetimes are compile-time annotations that tell the borrow checker how long a reference stays valid, preventing dangling references (use-after-free). They describe relationships between references and the data they point to — they never extend how long data actually lives. Elision rules infer them in the common cases, so you only annotate explicitly when a function returns a reference derived from an input or a struct stores a reference.
Q7. Why are enums plus exhaustive pattern matching powerful for modeling program state?
Enums are sum types whose variants can carry data, letting you make illegal states unrepresentable (e.g., an Escrow can only be Pending, Completed, or Cancelled). match is checked for exhaustiveness at compile time, so adding a new variant forces every match site to be updated. This turns whole classes of state-machine bugs into compile errors instead of runtime failures — ideal for instruction dispatch and account state transitions.
Q8. What does 'zero-cost abstraction' mean for generics and iterators?
It means high-level constructs compile down to code as efficient as a hand-written low-level version, with no runtime overhead for the abstraction itself. Generics are monomorphized into type-specialized code, and iterator adapter chains (map/filter/collect) are lazy and get inlined into a tight loop. You get expressive, safe code without paying for it at runtime — critical where compute is metered like on Solana.
