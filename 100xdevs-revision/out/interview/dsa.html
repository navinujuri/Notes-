Intro Class - DSA

Cohort 2.0 · 7 lessons · SDE-2 (3+ yrs) interview prep

Concept summary — all 7 steps

DSA vs CP
DSA is the study of data organization and algorithms to solve problems with good time/space complexity; Competitive Programming layers speed, math tricks, and contest heuristics on top of that foundation. For SDE interviews you optimize for clean, correct, well-reasoned solutions and complexity analysis, not raw contest ranking, so build DSA depth first and treat CP as optional pattern practice.
Choice of Language
Pick one language you can write fluently under pressure: C++ for fastest execution and rich STL, Java for verbose but predictable enterprise-style code, Python for shortest solve time but slower runtime and hidden costs (big-int, dynamic typing). It matters because in an interview you should never fight your language's syntax or standard library; know your containers' complexities cold.
Arrays / Lists
Contiguous, index-addressable storage with O(1) random access and O(1) amortized append (dynamic arrays double capacity), but O(n) insert/delete in the middle and O(n) search when unsorted. Cache locality makes them the fastest real-world container; most interview problems reduce to array traversal plus an auxiliary hashmap or two pointers.
Strings
Strings are effectively immutable arrays of characters in Java/Python/C#, so naive concatenation in a loop is O(n^2) — use StringBuilder / lists / join instead. Interview angle: know the immutability model of your language, character-frequency counting, and the sliding-window / two-pointer patterns that dominate string questions.
Problem 1: Good Pairs
Count index pairs (i<j) with nums[i]==nums[j] (LeetCode 1512): the brute force is O(n^2), but grouping by value and summing C(freq,2) with a frequency hashmap gives O(n). It teaches the core lesson that a hashmap turns 'find matching elements' from quadratic into linear, plus the combinatorics of counting pairs within equal groups.
Problem 2: Chef and Happy Strings
A CodeChef-style problem where a string is 'happy' when no two adjacent characters are equal; you count happy substrings (or minimum edits to make it happy). Solve in one linear pass by tracking the length L of the current maximal alternating run and adding L to a running total, which reinforces run-length thinking and avoiding the O(n^2) substring enumeration.
Problem 3: Practice Consolidation
An additional practice problem (title unspecified) meant to combine the toolkit — array traversal, frequency maps, and string scanning — into pattern recognition under time pressure. The interview takeaway is to classify the problem to a known template (hashing, two pointers, prefix sums) before coding rather than inventing an ad-hoc approach.

Real-world example

Deduplicating and validating a log-analytics ingestion pipeline

You are an SDE-2 building the ingestion path for a request-log analytics service. Raw log lines arrive as strings, so you parse and normalize each into a request signature (endpoint + status + user tier). To detect retries and duplicate emissions you need to count 'good pairs' — pairs of log entries in a window sharing the same signature — which is exactly the frequency-map plus C(freq,2) technique that keeps the count linear instead of comparing every pair. During parsing you hit a latency regression and trace it to naive string concatenation building each normalized key in a loop (O(n^2)); switching to a builder/join fixes it. You store parsed signatures in a dynamic array for O(1) amortized append and cache-friendly scans, and reuse the adjacency/run-length idea from the happy-strings problem to flag malformed lines with repeated delimiter characters. The whole feature is arrays + strings + a hashmap — the exact toolkit this intro class establishes.

Interview Q&A (SDE-2)

Q1. How would you explain the difference between DSA and Competitive Programming to a junior, and which should they prioritize for SDE-2 interviews?
DSA is the fundamentals: knowing data structures, algorithms, and how to reason about time and space complexity to produce correct, maintainable solutions. CP adds contest constraints — extreme speed, number theory, and template-heavy tricks tuned to squeeze problems in minutes. For interviews prioritize DSA, because interviewers score clarity, correctness, complexity analysis, and communication far more than the exotic techniques CP rewards; CP is useful mainly as extra pattern drilling.
Q2. What trade-offs drive the choice of C++ vs Java vs Python for interviews, and how does it affect complexity?
C++ gives the fastest runtime and the richest STL but more manual memory/pointer care; Java is verbose but predictable with a strong standard library; Python is fastest to write but has slower execution and hidden costs like O(n) list slicing and per-object overhead. The choice rarely changes asymptotic complexity but changes constant factors and, critically, your speed and error rate under pressure — so pick the one whose containers and idioms you know cold.
Q3. Explain the amortized O(1) append of a dynamic array and when arrays beat linked lists.
A dynamic array doubles its capacity when full, so a single append occasionally costs O(n) to copy, but averaged over n appends each is O(1) amortized because the total copy work is bounded by 2n. Arrays beat linked lists for almost all real workloads because contiguous memory gives cache locality and O(1) random access; linked lists only win when you need O(1) splicing at a known node or true stable pointers, which is rare in interview problems.
Q4. Why is building a string with repeated concatenation in a loop a problem, and what's the fix?
In Java, Python, and C# strings are immutable, so each concatenation allocates a new string and copies all existing characters, making a loop of n appends O(n^2) time and heavy on allocations. The fix is a mutable buffer — StringBuilder in Java, a list plus ''.join in Python, or reserve+append on std::string in C++ — giving amortized O(n) overall. It's a classic hidden-cost gotcha interviewers probe.
Q5. Give the optimal solution and complexity for counting Good Pairs (nums[i]==nums[j], i<j).
Count the frequency of each value in one pass with a hashmap, then for each value with frequency f add f*(f-1)/2, since any two of the f equal elements form exactly one good pair. This is O(n) time and O(k) space for k distinct values, versus the O(n^2) brute force of checking all pairs. The key insight is converting 'find all matching pairs' into 'count within equal groups' using combinatorics.
Q6. How do you count 'happy' substrings (no two adjacent characters equal) in linear time?
Scan left to right maintaining the length L of the current maximal run where each character differs from its predecessor; whenever it extends, L increases and you add L to the answer (each new character forms L new happy substrings ending at it), resetting L to 1 when two adjacent characters are equal. This is O(n) time and O(1) space and avoids enumerating all O(n^2) substrings. It's a run-length pattern that generalizes to many 'count substrings with an adjacency property' problems.
Q7. When you see a new array/string problem, how do you decide which pattern to apply?
Classify by the operation the problem needs: 'find/match/count duplicates or complements' points to a hashmap; 'contiguous window with a constraint' points to sliding window; 'sorted or pair-from-both-ends' points to two pointers; 'range-sum queries' points to prefix sums. Committing to a template before coding prevents ad-hoc quadratic solutions, and stating the pattern and its complexity out loud signals structured thinking to the interviewer.
Q8. What are the complexity costs of common Python list operations an SDE-2 should not get wrong?
append and pop from the end are O(1) amortized, but pop(0) or insert(0, x) are O(n) because elements shift; the 'in' membership test on a list is O(n) whereas on a set/dict it's O(1); and slicing like a[1:] creates a full O(k) copy. Getting these wrong silently turns an intended O(n) solution into O(n^2), so prefer collections.deque for front operations and sets/dicts for membership.
