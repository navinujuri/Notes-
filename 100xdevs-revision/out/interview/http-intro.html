Intro to HTTP

Cohort 3.0 · 14 lessons · SDE-2 (3+ yrs) interview prep

Concept summary — all 14 steps

Context
Frames HTTP as the application-layer protocol that lets a frontend (or any client) talk to a backend over the network. The interview angle: HTTP sits on top of TCP (and TLS for HTTPS) at L7 — knowing where it lives in the stack signals you understand what HTTP does vs. what the transport layer guarantees.
Why the HTTP Protocol?
A protocol is an agreed-upon contract for message format so heterogeneous clients and servers (any language, any OS) can interoperate. HTTP standardizes how requests/responses are structured so a Python server and a JS browser can communicate without custom wire formats. Interviewers probe whether you can distinguish a 'protocol' (rules) from an 'implementation' (a specific server).
Request-Response model
HTTP is stateless and client-driven: the client always initiates a request, the server replies with exactly one response, and the server never speaks unprompted. Statelessness is the key point — each request carries all context it needs (auth token, cookies), which is why servers scale horizontally but also why you need tokens/sessions to simulate state. Contrast with WebSockets/SSE when full-duplex or server-push is needed.
Domain name / IP
Humans use domain names; machines route on IP addresses, so DNS resolves the name to an IP before any HTTP request is sent. The request itself travels to that IP:port. Common interview follow-up: DNS resolution happens before the TCP handshake, is cached at multiple layers (browser, OS, resolver), and one domain can map to many IPs (load balancing / CDN).
Ports
A port disambiguates which process on a host receives the traffic — HTTP defaults to 80, HTTPS to 443. An IP identifies the machine; the port identifies the service on it, so one server can run many services on distinct ports. Know the well-known defaults and that the browser hides :80/:443 in the URL bar.
Methods
HTTP verbs express intent: GET (read), POST (create/action), PUT (replace), PATCH (partial update), DELETE (remove). The properties that matter: GET is safe (no side effects) and cacheable; GET/PUT/DELETE are idempotent while POST is not. Interviewers care that you choose verbs by semantics (idempotency, safety, caching) rather than treating everything as POST.
Response
Every response has a status line (protocol version + status code), headers, and an optional body. It mirrors the request structure. The point to internalize: the client branches its logic on the status code first, then parses the body — a 200 with an error-shaped body is a common API design smell worth calling out.
Status codes
Grouped by class: 2xx success, 3xx redirection, 4xx client error, 5xx server error. Specifics matter — 200 vs 201 (created) vs 204 (no content); 301 vs 302 (permanent vs temporary redirect); 400 vs 401 (unauthenticated) vs 403 (unauthorized) vs 404; 429 rate limit; 500 vs 502/503/504 (gateway/unavailable/timeout). Using the right code drives client retry logic, caching, and monitoring.
Body
The payload carrying the actual data — JSON is the common format, indicated via the Content-Type header. GET requests conventionally have no body (data goes in query params); POST/PUT/PATCH carry it. Key nuance: the server must be told how to parse the body (Content-Type), which is why body-parser/express.json() middleware exists and why a mismatched header yields a parse failure.
Routes
The path portion of the URL (e.g., /users/42/orders) that maps a request to a specific handler on the server. Combined with the method, (method + route) forms the addressable endpoint. REST design uses nouns and hierarchy in routes and verbs for actions; interviewers look for clean resource modeling and awareness of path params vs query params.
Headers
Key-value metadata sent on both request and response — Content-Type, Authorization, Accept, Cookie, Cache-Control, CORS headers, User-Agent. They carry cross-cutting concerns (auth, content negotiation, caching, CORS) separate from the body. Strong answers cite specific headers for a scenario, e.g., Authorization: Bearer <token> for auth or Access-Control-Allow-Origin for CORS.
Checkpoint (do we understand this now?)
A consolidation step: given a full URL like https://api.example.com:443/users?id=5, you should be able to decompose it into protocol, domain (→IP via DNS), port, route, and query — plus the method, headers, and body that accompany the request. This end-to-end decomposition is exactly the whiteboard exercise interviewers use to check depth.
Clients (Postman / curl / browser)
Anything that can send an HTTP request is a client — the browser (GET on navigation, fetch/XHR for the rest), curl for scripting/CI, Postman for manual API exploration. The point: the server is client-agnostic, so you can test/debug an endpoint with curl before any frontend exists. Being fluent in curl flags (-X, -H, -d) is a practical expectation.
Writing HTTP code in JS
Implement a server with Node's http module or Express: define routes by (method + path), read headers/body, and send back a status code + body. On the client side, use fetch. Interview value: you can articulate the request lifecycle in code — parsing the body, setting Content-Type, returning correct status codes — rather than just describing HTTP abstractly.

Real-world example

Debugging a broken 'Add to Cart' call in an e-commerce app

Your React frontend calls POST https://api.shop.com/carts/123/items with a JSON body {"sku":"ABC","qty":2}, but items silently fail to add. Walking the HTTP path: the browser resolves api.shop.com via DNS to an IP, opens a TLS connection on port 443, and sends the request with headers Authorization: Bearer <jwt> and Content-Type: application/json. You reproduce it outside the UI with `curl -X POST -H 'Content-Type: application/json' -H 'Authorization: Bearer …' -d '{"sku":"ABC","qty":2}' https://api.shop.com/carts/123/items` and get back 415 Unsupported Media Type. The root cause: the Express server was missing app.use(express.json()), so the body was never parsed and Content-Type validation rejected it. After adding the middleware, the endpoint returns 201 Created with the updated cart in the body. This single bug touches nearly every concept in the topic — DNS/IP, port, method, route, headers, body, status codes, and the client tools used to isolate frontend vs backend fault.

Interview Q&A (SDE-2)

Q1. HTTP is described as stateless. What does that mean and how do real apps maintain user sessions?
Stateless means the server retains no memory of prior requests; each request must carry everything needed to be understood, and the server can handle any request on any instance. Apps simulate state by having the client resend identity on every request — typically a session cookie (server-side session store) or a self-contained JWT in the Authorization header. This is what lets you load-balance across many stateless server instances, at the cost of moving state into tokens or an external store like Redis.
Q2. Which HTTP methods are idempotent and safe, and why does it matter in practice?
Safe methods (GET, HEAD) have no side effects. Idempotent methods (GET, PUT, DELETE, HEAD) produce the same server state whether called once or N times; POST is neither safe nor idempotent. It matters because idempotency lets clients, proxies, and load balancers safely retry on network failures or timeouts without duplicating effects — retrying a POST could create duplicate orders, which is why real APIs add idempotency keys to make POST safely retryable.
Q3. A client gets a 401 and a 403 in different cases. What's the distinction, and how would you handle each?
401 Unauthorized means the request is not authenticated — no valid credentials — so the client should (re)authenticate, e.g., refresh the token or redirect to login. 403 Forbidden means the client is authenticated but not permitted to access that resource, so re-auth won't help; it's an authorization/permissions problem. Conflating them leads to bad UX like infinite login loops on a 403.
Q4. Walk me through what happens between typing a URL and the server receiving the HTTP request.
The browser parses the URL into protocol, host, port, path, and query. It resolves the host to an IP via DNS (checking browser/OS/resolver caches first). It opens a TCP connection to that IP on the port (80 or 443), and for HTTPS performs a TLS handshake. Only then does it send the HTTP request line (method + path + version), headers, and optional body to the server process listening on that port.
Q5. When should you use query parameters vs a request body vs path parameters?
Path params identify a resource and belong in the route hierarchy (/users/42). Query params are for optional, non-hierarchical modifiers like filtering, sorting, and pagination (?status=active&page=2) and travel on GET requests, which have no body. The request body carries the payload for create/update operations (POST/PUT/PATCH) and can hold large, structured, or sensitive data — unlike the URL, it isn't logged in access logs or browser history by default.
Q6. What role do Content-Type and Accept headers play, and what breaks if they're wrong?
Content-Type tells the receiver how to parse the body being sent (e.g., application/json vs multipart/form-data); Accept is content negotiation — the client stating which response formats it can handle. If Content-Type is missing or wrong, the server may fail to parse the body or reject it with 415, which is a classic cause of 'my POST body is empty' bugs. These headers decouple the wire format from the data and let one endpoint serve multiple representations.
Q7. Why would you reach for curl or Postman instead of just testing through the browser UI?
Because the server is client-agnostic, you can exercise an endpoint in isolation — before the frontend exists or to determine whether a bug is frontend or backend. curl and Postman let you control the method, headers, and body precisely, inspect raw status codes and response headers, and script/automate calls in CI. The browser mainly issues GETs on navigation and applies CORS and caching that can mask the true server behavior.
Q8. HTTP runs on top of TCP. What does TCP give HTTP, and when would you not want HTTP's request-response model?
TCP provides a reliable, ordered, connection-oriented byte stream, so HTTP doesn't handle packet loss, retransmission, or ordering itself. HTTP's strict one-request/one-response, client-initiated model is a poor fit when the server needs to push data or you need low-latency bidirectional communication — e.g., live chat, notifications, or collaborative editing. There you'd use WebSockets (full-duplex) or Server-Sent Events (server→client stream), which is a common follow-up interviewers use to test protocol trade-offs.
