Horizontal and vertical scaling, Indexes in DBs

Cohort 2.0 · 11 lessons · SDE-2 (3+ yrs) interview prep

Concept summary — all 11 steps

Scope & problem framing
Sets up the two axes of scaling a stateless API tier plus its stateful database, and why read/write latency and throughput eventually force a scaling decision. Interview angle: know that app servers are usually easy to scale (stateless), while the database is the real bottleneck, which is why indexing and data modeling dominate this topic.
Vertical scaling
Scaling up: give one machine more CPU, RAM, IOPS, or faster disk. It's the simplest lever (no code change, no distributed-systems complexity) but hits hard ceilings, gets exponentially expensive, and leaves a single point of failure. Interview angle: it buys time and is often the right first move, but it cannot give you HA on its own.
Horizontal scaling in a Node.js app
Run multiple identical stateless instances behind a load balancer (or use the Node `cluster`/PM2 to fork workers per core, since a single Node process is single-threaded for JS). The key constraint: instances must share no in-process state — sessions, rate-limit counters, and caches move to Redis/DB so any instance can serve any request. Interview angle: sticky sessions vs. stateless JWT, and why `cluster` only scales one box while a load balancer scales many.
Capacity estimation
Back-of-envelope sizing: estimate DAU, RPS (peak ≈ several× average), payload sizes, storage growth/day, and memory per connection to decide instance count and DB size. Interview angle: system-design rounds expect you to derive RPS from users and reason about read:write ratio, connection limits, and headroom rather than guess.
Horizontal scaling (general)
Scaling out across many commodity machines gives near-linear capacity and fault tolerance, but introduces load balancing, data partitioning/sharding, replication, and consistency concerns. Interview angle: stateless tiers scale trivially; the hard part is the data layer — read replicas for read-heavy loads, sharding for write-heavy/large datasets, and the CAP trade-offs that follow.
Indexing in Postgres
An index (default B-tree) is a separate sorted structure that turns an O(n) sequential scan into an O(log n) lookup for equality and range queries on the indexed column(s). The trade-off: every index adds write/update cost and storage, so you index columns used in WHERE/JOIN/ORDER BY, not everything. Interview angle: use EXPLAIN ANALYZE to confirm the planner actually uses the index, and know B-tree vs. Hash vs. GIN/GiST use cases.
Complex / composite indexes
Multi-column indexes serve queries that filter on a prefix of the columns; column order matters (leftmost-prefix rule), so `(a,b)` helps `WHERE a` and `WHERE a AND b` but not `WHERE b` alone. Covering indexes (INCLUDE) and partial indexes (`WHERE` clause) further optimize hot queries. Interview angle: be able to explain why `(user_id, created_at)` beats two single-column indexes for a paginated per-user feed.
Indexes in Prisma
Declare indexes in schema.prisma with `@@index([...])`, `@unique`/`@@unique`, and `@id`; Prisma emits the DDL via migrations so indexes are version-controlled with the schema. Interview angle: ORMs don't auto-create indexes on foreign keys or frequent filters — you must add them explicitly, and unique constraints double as indexes.
Normalization
Organizing data into related tables to eliminate redundancy and update/insert/delete anomalies (1NF→2NF→3NF removes repeating groups, partial, and transitive dependencies). Trade-off: normalized schemas need JOINs (read cost) while denormalization trades storage/consistency for read speed. Interview angle: normalize for write-heavy/OLTP integrity, denormalize selectively for read-heavy paths and analytics.
Types of relationships
One-to-one, one-to-many, and many-to-many, modeled with foreign keys — the many-to-many case requires a join/junction table holding both FKs. Interview angle: know where the FK lives (on the 'many' side), and that a junction table can carry its own attributes (e.g., role/joined_at on a user↔team link).
Normalizing data (applied)
Practical walkthrough: take a flat/denormalized table, split repeating and dependent attributes into their own tables, and wire them with FKs and the right relationship type. Interview angle: recognize a normalization smell (e.g., comma-separated values or duplicated customer data per order row) and refactor it into properly keyed tables.

Real-world example

Scaling a course platform's video-progress and comments service

You run a Node.js/Postgres API for an online course platform. It starts on one box; as signups climb you first vertically scale the DB (more RAM so the working set and indexes stay in memory). When the API tier saturates CPU, you go horizontal: run the API as several stateless instances behind a load balancer, move sessions/rate-limits to Redis so any instance can serve any request, and add a read replica for the read-heavy 'watch progress' and 'course comments' endpoints. A capacity estimate (say 500k DAU, ~5k peak RPS, 90:10 read:write) drives instance count and connection-pool sizing. The comments feed query WHERE course_id = ? ORDER BY created_at DESC LIMIT 20 is slow on a sequential scan, so you add a composite index (course_id, created_at) — declared in Prisma as @@index([courseId, createdAt]) — and confirm with EXPLAIN ANALYZE that the planner uses it. Finally you normalize the schema: instead of storing course title and instructor name on every progress row, you split users, courses, and enrollments into their own tables with FKs, modeling user-to-course as many-to-many via an enrollments junction table that also carries progress_pct and enrolled_at.

Interview Q&A (SDE-2)

Q1. When would you choose vertical scaling over horizontal scaling, and vice versa?
Vertical scaling is the right first move when the bottleneck fits on one bigger machine and you want zero code/architecture change — it's fast and avoids distributed-systems complexity, which is why it's common for the primary database early on. You switch to horizontal scaling when you hit the ceiling of a single machine, need high availability (removing the single point of failure), or need near-linear capacity growth. In practice you scale the stateless app tier horizontally early and defer horizontal DB scaling (replicas/sharding) as long as possible because it's the hardest part.
Q2. What does it take to make a Node.js service horizontally scalable?
The instances must be stateless — no session data, counters, or caches held in process memory — so any request can hit any instance. You externalize that state to Redis or the database, use stateless auth like signed JWTs (or a shared session store), and put a load balancer in front. Within a single box you can also use the Node `cluster` module or PM2 to fork one worker per core (since JS execution is single-threaded), but that only scales one machine; a load balancer across boxes is what gives you true horizontal scale and fault tolerance.
Q3. How does a B-tree index speed up queries, and what does it cost?
A B-tree keeps the indexed column(s) in sorted order, so equality and range lookups become O(log n) tree traversals instead of an O(n) sequential scan, and it can also satisfy ORDER BY without a separate sort. The cost is on writes: every INSERT/UPDATE/DELETE that touches an indexed column must also update the index, plus extra storage. So you index columns used in WHERE, JOIN, and ORDER BY on hot paths, and avoid indexing everything — especially low-selectivity columns where a scan is cheaper than an index traversal.
Q4. Explain composite index column ordering with a concrete example.
A composite index follows the leftmost-prefix rule: `(course_id, created_at)` can serve `WHERE course_id = ?`, `WHERE course_id = ? AND created_at > ?`, and `WHERE course_id = ? ORDER BY created_at`, but it cannot efficiently serve `WHERE created_at = ?` alone because created_at isn't a leading column. So you put the equality/most-selective filter first and the range/sort column second. For a per-course comment feed this single composite index beats two separate single-column indexes because Postgres can walk one structure to both filter and return rows in sorted order.
Q5. Prisma manages the schema — do you still need to think about indexes?
Yes. Prisma only creates indexes you declare: `@id`, `@unique`/`@@unique`, and `@@index([...])`. It does not automatically index foreign-key columns or your frequent filter/sort columns, so those queries will do sequential scans until you add explicit `@@index` entries. The upside is that once declared they live in schema.prisma and ship via migrations, so indexes are code-reviewed and version-controlled rather than applied ad hoc in production.
Q6. What problems does normalization solve, and when would you denormalize?
Normalization (up to 3NF) removes redundancy and the resulting update/insert/delete anomalies — e.g., storing a customer's address on every order means an address change must update many rows and risks inconsistency. Splitting into keyed tables linked by FKs makes each fact live in exactly one place. You denormalize deliberately on read-heavy paths where JOIN cost hurts (e.g., a precomputed feed, analytics/OLAP tables, or a cached aggregate), accepting extra storage and the burden of keeping copies consistent in exchange for faster reads.
Q7. How do you model a many-to-many relationship, and why can't a plain FK do it?
A foreign key on one row can only point to a single parent, so it natively expresses one-to-many (the FK lives on the 'many' side). For many-to-many — e.g., students enrolled in many courses and courses having many students — you introduce a junction table holding both foreign keys (student_id, course_id), typically with a composite unique/primary key on the pair. That table can also carry relationship attributes like enrolled_at or progress_pct, which is exactly why you can't collapse it into either parent.
Q8. How would you approach capacity estimation for a new read-heavy API?
Start from users: derive average RPS from DAU and requests-per-user-per-day, then assume peak is several times average (traffic isn't uniform). Combine that with the read:write ratio, average payload size, and per-request DB work to size instance count, DB connections (mind Postgres's connection limit — use a pooler like PgBouncer), and read replicas for reads. Add storage growth per day to plan disk, and always leave headroom (target well under 100% utilization) so you can absorb spikes and node failures without cascading.
