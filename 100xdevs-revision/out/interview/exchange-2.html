Building an Exchange - Part 2 (Backend)

Cohort 2.0 · 5 lessons · SDE-2 (3+ yrs) interview prep

Concept summary — all 5 steps

Recap: what was built in Part 1
Part 1 built the client side — the Next.js UI, order book / depth view, and the shape of the REST + WebSocket contract the frontend expects. It matters because it fixes the API surface the backend must now satisfy; interviewers use this framing to check you can design a backend against an already-agreed contract rather than in a vacuum.
Scope for today: the backend services
Today's goal is the server side: an API server that accepts orders, an in-memory matching engine, a durable queue between them, a DB persistence path, and a real-time market-data fan-out. The interview angle is decomposition — being able to name the discrete services and their responsibilities before writing code.
Backend architecture (queue + engine + pub/sub)
The canonical design is: API server LPUSHes orders onto a Redis list, a single-process Engine BRPOPs them, mutates an in-memory order book, matches, then publishes results — to a per-request channel for the API's synchronous response, to a DB queue for persistence, and to market channels for the WS server. This separates the low-latency deterministic matching hot path from slow, scalable, best-effort concerns (DB writes, client fan-out); expect to defend why the engine is single-threaded and in-memory.
Pub/sub for request-reply and market-data fan-out
Redis pub/sub is used two ways: a request/response bridge where the API subscribes to a channel keyed by a unique order id and the engine publishes the result to it, and a fan-out where a WebSocket server subscribes to market channels (depth@MARKET, trade@MARKET) and pushes to browsers. Key point: pub/sub is fire-and-forget with no persistence, so it's right for ephemeral market data but orders must travel via the durable list, not pub/sub.
Singletons for shared managers/state
RedisManager (one connection for pushing/publishing), the Engine's order book, and the SubscriptionManager (channel → sockets map) are singletons so there is exactly one connection and one source of truth for in-memory state per process. Interview angle: know the pitfalls — hidden global state hurts testability, and 'singleton' is per-process, so it does not give you a cluster-wide single instance.

Real-world example

Placing a limit order and watching the book update in real time

A user submits a BUY 0.5 BTC @ 60,000 limit order. The Express API assigns a unique id, LPUSHes {order, id} onto the Redis `orders` list, and subscribes to the pub/sub channel `id`. The single Engine process BRPOPs the order, inserts it into the in-memory order book at the 60,000 price level (price-time FIFO), runs matching against the best asks, and produces fills. It publishes the execution result back to channel `id` (the API resolves the pending HTTP response), pushes trade + balance events onto a `db` queue for a DB processor to write to TimescaleDB asynchronously, and publishes the updated depth and trade ticks to `depth@BTC_USDC` / `trade@BTC_USDC`. A WebSocket server subscribed to those channels fans the update out to every connected browser, so the depth chart moves within milliseconds. When you build this, the bugs you actually hit are: the API request hanging because the engine crashed mid-order (needs a timeout + snapshot/replay recovery), and duplicate DB rows on engine restart (needs idempotent, sequenced writes).

Interview Q&A (SDE-2)

Q1. Why run the matching engine as a single-threaded, in-memory process instead of a multi-threaded service backed by a database?
Matching is inherently sequential — price-time priority requires a total order of events — so a single thread eliminates lock contention and race conditions on the book and gives deterministic, replayable behavior. Keeping the book in memory keeps per-order latency in microseconds; a DB round-trip per match would be orders of magnitude slower. The trade-off is a throughput ceiling of one core and a durability problem, which you solve with snapshots plus an event log rather than by adding threads.
Q2. If the order book lives in memory, how do you make the engine crash-recoverable?
Combine periodic snapshots of the full book with a durable, ordered event source. Every order is on the Redis list (durable) before processing, and you snapshot the in-memory state to disk/Redis at intervals; on restart you load the latest snapshot and replay events since that snapshot's sequence number. The key invariants are that no accepted order is lost and that replay is deterministic, which is exactly why the engine is single-threaded and why writes downstream must be idempotent.
Q3. Why send orders through a Redis list but market data through Redis pub/sub?
A list with LPUSH/BRPOP is a durable, at-least-once work queue: messages persist until a consumer pops them, it provides natural backpressure, and it load-balances across consumers — orders must never be silently dropped. Pub/sub is fire-and-forget fan-out with no persistence; if no subscriber is connected the message is gone. That is unacceptable for orders but perfect for depth/trade updates, where subscribers are many, delivery is best-effort, and only the latest state matters.
Q4. How does a synchronous REST call get a response from an asynchronous engine?
The API generates a unique id, subscribes to a pub/sub channel named after that id, then pushes the order onto the queue and holds the HTTP response open (a pending Promise). The engine processes the order and publishes the result to that id-channel; the API's subscriber resolves the Promise and returns the response, then unsubscribes. You must add a timeout so a crashed or slow engine surfaces as a 5xx/timeout instead of a hung connection, and be careful to unsubscribe to avoid channel leaks.
Q5. Why are RedisManager, the Engine, and SubscriptionManager implemented as singletons, and what are the downsides?
They wrap resources and state that must be unique per process: one Redis connection avoids exhausting connections and interleaving issues, one Engine is the single source of truth for the in-memory book, and one SubscriptionManager owns the channel-to-socket map. The downsides are testability (global mutable state is hard to mock/reset) and a common misconception — a singleton is per-process, so running multiple API or WS instances gives you multiple singletons, which is fine for stateless managers but would be catastrophic if you accidentally ran two matching engines against the same market.
Q6. How do you scale the WebSocket fan-out layer without duplicating or dropping updates?
Run multiple stateless WS servers that each subscribe to the relevant Redis channels; Redis handles the fan-out, so horizontal scaling is cheap. Within a server, the SubscriptionManager should subscribe to each Redis channel once and ref-count the connected clients, adding sockets to an existing subscription rather than re-subscribing per user, and unsubscribing only when the last client for that channel disconnects. This avoids duplicate deliveries and leaked subscriptions as users open and close market views.
Q7. Persisting trades to the database is slow — how do you keep it off the matching hot path safely?
The engine emits trade, order, and balance events onto a separate DB queue and moves on; a dedicated DB processor consumes that queue and writes to the time-series/relational store asynchronously. This keeps matching latency independent of disk I/O and lets the persistence layer scale on its own. The costs are eventual consistency (reported balances can briefly lag the in-memory truth) and the need for ordered, idempotent writes — carry a monotonic sequence number so replays or retries after a crash don't double-apply.
