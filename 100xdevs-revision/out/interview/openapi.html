OpenAPI Spec

Cohort 2.0 · 7 lessons · SDE-2 (3+ yrs) interview prep

Concept summary — all 7 steps

Why OpenAPI Spec
It provides a single machine-readable contract for an HTTP API so docs, server, and clients never drift apart. Without it teams hand-write fetch code and types, docs go stale, and integration bugs surface only at runtime. Interview angle: enables contract-first workflows that decouple frontend/backend teams and let both build in parallel against a stable interface.
What is OpenAPI
A language-agnostic JSON/YAML standard (formerly Swagger) that describes REST endpoints, parameters, request/response schemas, auth, and servers. It is a specification, not a tool: a large ecosystem (docs UIs, validators, code/client generators) consumes it. Current major version is 3.x; know that 3.1 realigned with full JSON Schema Draft 2020-12 while 3.0 used a modified subset.
Parts of the spec file
Top-level keys: openapi (version), info (title/version), servers, paths (each path -> HTTP methods -> operation with parameters/requestBody/responses), and components (reusable schemas, securitySchemes, parameters, responses referenced via $ref). security applies auth globally or per-operation, and tags group operations for docs. Understanding $ref-based reuse and where auth is declared is the common interview probe.
How to create a spec
Two approaches: design-first (author YAML/JSON by hand or in an editor, then implement against it) vs code-first (derive the spec from annotated code or typed schemas). Design-first gives a stable contract for parallel work but risks implementation drift; code-first guarantees the spec matches running code but couples the contract to your framework. Mature setups add contract tests so neither approach silently diverges.
Hono + Zod + OpenAPI
Using @hono/zod-openapi you define routes with Zod schemas that do double duty: runtime request/response validation AND automatic OpenAPI generation. This makes one Zod schema the single source of truth for static TS types, runtime checks, and the published spec, eliminating the classic drift between validation and docs. Interview angle: this is code-first done well because the validating schema is the contract.
Create a swagger page
Serve interactive API docs by pointing a renderer (Swagger UI, Scalar, or Redoc) at your generated /openapi.json. It gives a browsable, always-current reference with a try-it-out console that fires real requests against the running server. In Hono, Scalar (@scalar/hono-api-reference) or Swagger UI is mounted as a route that reads the same spec the framework emits.
Auto generated clients
Tools like openapi-typescript, orval, or openapi-generator turn the spec into typed client SDKs, replacing hand-written fetch calls and keeping request/response types in sync with the backend. Regeneration on spec change surfaces breaking changes at compile time in the consumer. Caveat: generated clients are only as correct as the spec, so a wrong or under-specified spec produces confidently-typed but wrong code.

Real-world example

Shipping a typed payments API for a web + mobile frontend

You are building a payments service in Hono and two teams (a React web app and a React Native app) need to call it. You define each route with @hono/zod-openapi: a Zod schema like CreatePaymentBody validates the incoming request at runtime, doubles as the TS type in your handler, and auto-generates the OpenAPI 3.1 document served at /openapi.json. You mount Scalar at /docs so QA and the frontend devs get a live, always-current reference with a try-it-out console. The frontend teams run openapi-typescript (or orval) against your spec in CI to generate a fully typed client, so when you add a required idempotencyKey field, their build fails immediately instead of erroring in production. Later, when debugging a 422, you can see the exact Zod-derived schema in the spec, confirm the client sent a malformed amount, and know the validation, docs, and generated client all came from one source of truth.

Interview Q&A (SDE-2)

Q1. What changed between OpenAPI 3.0 and 3.1, and why does it matter?
3.1 is a full superset of JSON Schema Draft 2020-12, while 3.0 used a modified, incompatible subset. Practical consequences: nullability moves from nullable: true to type: ["string", "null"], you gain features like proper type arrays and top-level webhooks, and you can reuse standard JSON Schema tooling directly. It matters because tool compatibility differs by version, so generators and validators must target the version you emit.
Q2. Design-first vs code-first: when would you choose each?
Design-first (author the spec, then implement) is best when multiple teams need a stable contract to build against in parallel or when the API is a public product; the risk is the server drifting from the spec. Code-first (generate the spec from typed schemas like Zod) guarantees the spec reflects running code and reduces duplication, but couples the contract to your framework and can leak implementation detail. Strong teams pick one and add contract testing (e.g., schemathesis or a validation middleware) so drift is caught mechanically rather than by review.
Q3. In the Hono + Zod setup, how does one schema become a single source of truth?
The Zod schema is compiled to a runtime validator (rejecting bad requests with structured errors), inferred into a static TypeScript type used inside the handler, and converted to an OpenAPI schema object embedded in the generated document. Because all three derive from the same definition, you cannot validate one shape while documenting another. This closes the classic gap where hand-written validation, TS types, and docs each say something slightly different.
Q4. Why is $ref / components important, and what breaks without it?
components lets you define a schema, parameter, response, or securityScheme once and reference it with $ref everywhere it is used. Without it you duplicate the same object across dozens of operations, so a field change must be edited in many places and generated clients emit many near-identical types instead of one shared model. It also keeps the document readable and makes diffs meaningful when the contract evolves.
Q5. How do required, optional, and nullable map from schema to spec, and why do people confuse them?
Required means the key must be present (controlled by the object's required array, not the field itself); optional means the key may be absent; nullable means the value may be null when present. They are orthogonal: a field can be required-and-nullable (must be sent, may be null) or optional-and-non-nullable. Confusion arises because 3.0 expressed null via nullable: true while 3.1 uses type: ["...", "null"], and because Zod's .optional() vs .nullable() vs .nullish() each map differently.
Q6. What are the risks of auto-generated clients and how do you mitigate them?
A generated client is only as correct as the spec, so an under-specified or wrong schema yields confidently-typed but incorrect code, and different generators produce different ergonomics and error handling. Mitigations: treat the spec as reviewed source, pin the generator version and commit generated output (or regenerate in CI) so diffs are visible, and add a smoke/contract test that hits the real API through the generated client. Also validate the spec itself in CI to catch drift before clients are built from it.
Q7. How would you handle a breaking API change with an OpenAPI-driven workflow?
First distinguish breaking (removing/renaming a field, making an optional field required, tightening types) from additive changes. Use a spec-diff tool (e.g. oasdiff) in CI to classify the change and fail the pipeline on unintended breaks. For real breaks, version the API (path or header based), keep the old operations in the spec during a deprecation window using the deprecated flag, and let consumers regenerate clients against the new version on their own schedule.
Q8. How do you ensure the running implementation actually matches the published spec?
In code-first setups (Zod-generated), the spec is emitted from the same schemas that validate requests, so drift is structurally limited but response bodies can still lie if handlers return unvalidated data. Add response validation in non-prod, and run contract/property-based tests such as schemathesis or Dredd that exercise every operation against the spec. In design-first setups this testing is mandatory because the server is written independently of the contract.
