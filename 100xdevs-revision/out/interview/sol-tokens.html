Programs, Accounts and the Token Program

Cohort 3.0 · 11 lessons · SDE-2 (3+ yrs) interview prep

Concept summary — all 11 steps

Why Solana
Solana is a high-throughput L1 combining Proof of History (a verifiable clock that orders events before consensus), Tower BFT PoS, the Sealevel parallel runtime, and Gulf Stream tx forwarding, giving ~400ms slots and sub-cent fees. Interview angle: the parallel runtime is why transactions must declare every account they read/write upfront - non-overlapping accounts execute concurrently, unlike Ethereum's sequential EVM.
Accounts on Solana
Everything on Solana is an account: a record with a lamports balance, an owner (a program), a data byte-array, and an executable flag. Programs are stateless executable accounts; their state lives in separate data accounts they own. Interview angle: this code/state separation contrasts with EVM contracts that bundle logic and storage, and it is what enables account-level parallelism.
Install Solana CLI
Set up solana-cli plus spl-token CLI, configure the cluster (solana config set --url devnet), generate a keypair (solana-keygen), and airdrop devnet SOL. Foundational tooling for creating mints and inspecting accounts before touching client SDKs.
Web2 data model
In Web2 a trusted server owns a central database; rows are keyed by a user id, freely mutable, and access is gated by app-layer auth. Set up as the contrast baseline: state ownership and mutation authority sit entirely with the operator.
Data model on Solana
State is spread across many accounts owned by programs rather than rows in one DB; there is no ambient trust, only account ownership and signatures. Token balances live in per-holder token accounts, not in the mint - the mint holds supply/decimals/authorities while balances are separate accounts. Interview angle: understand mint account vs token account (ATA) separation.
Creating a token
Use the SPL Token program: create a mint account specifying decimals, mint authority, and optional freeze authority, then create associated token accounts and mint_to them. The spl-token CLI wraps these instructions; the mint defines the asset, ATAs hold balances.
Equivalent code in JS
With @solana/web3.js + @solana/spl-token you call createMint, getOrCreateAssociatedTokenAccount, and mintTo, which assemble instructions into a transaction you sign and send. Interview angle: these helpers are just instruction builders - a transaction is an ordered list of instructions targeting program ids with explicit account metas and data.
Equivalent code in Rust/Python/Go
The same mint/ATA/mint_to instructions can be built from Rust, Python (solana-py/solders), or Go SDKs because the on-chain program is language-agnostic. Interview angle: clients only serialize the same instruction data and accounts; the SDK language never changes on-chain behavior.
PDAs
Program Derived Addresses are addresses deliberately pushed off the ed25519 curve, derived deterministically from seeds + program id + a bump byte, so they have no private key. The owning program 'signs' for a PDA via invoke_signed, making PDAs ideal for deterministic state accounts and program-controlled authorities - Associated Token Accounts are themselves PDAs. High-frequency interview topic.
Token-2022 program
Token-2022 (Token Extensions) is a newer SPL token program at a distinct program id that is a superset of the original, adding opt-in extensions like transfer fees, transfer hooks, confidential transfers, non-transferable/soulbound, and interest-bearing tokens. Interview angle: it is not automatically backward compatible - the program id differs, so wallets/clients must explicitly support it, and extensions change account sizing and transfer semantics.
Token-2022 with metadata
The metadata (and metadata-pointer) extension stores name/symbol/uri directly on the mint via the Token Metadata interface, avoiding a separate Metaplex metadata PDA. Interview angle: contrast on-mint metadata (self-contained, resizable account) with the legacy Metaplex approach where metadata lives in a separate program-owned account.

Real-world example

Launching an on-chain rewards token for a consumer app

You are the SDE-2 shipping 'GRAV points', a loyalty token users earn in-app on Solana. You create a mint on Token-2022 with 6 decimals, set the mint authority to a PDA derived from your program (so only your program can mint, not a leaked keypair), and attach the metadata extension so the name, symbol, and logo URI live directly on the mint. When a user earns points, your backend derives their Associated Token Account (itself a PDA), creates it if missing via getOrCreateAssociatedTokenAccount, and calls mintTo with the program signing for the mint-authority PDA through invoke_signed. Two bugs you will actually debug: (1) 'incorrect program id for instruction' because you passed the classic Token program id to a Token-2022 mint, and (2) transfers failing because you added a transfer-fee extension and the recipient's ATA was sized for the base token layout. When finance later asks to freeze compromised balances, you use the freeze authority you set at mint creation; when they ask to cap supply, you set the mint authority to null to make it immutable.

Interview Q&A (SDE-2)

Q1. What is the difference between a mint account and a token account, and where does a user's balance live?
The mint account is the asset definition - it stores decimals, total supply, and the mint/freeze authorities, but no per-user balances. Each holder has a separate token account (typically an Associated Token Account, a PDA) that references the mint and stores that holder's amount. So one mint has many token accounts; balances live in the token accounts, never in the mint itself.
Q2. What is a PDA and how can a program authorize actions from an address with no private key?
A Program Derived Address is derived from seeds + the program id + a bump byte, chosen so the result falls off the ed25519 curve - meaning no private key can exist for it. A program signs on the PDA's behalf using invoke_signed, passing the same seeds and bump; the runtime verifies the derivation and treats the PDA as a signer only within that program's CPI. This gives you deterministic, program-controlled accounts and authorities without managing secret keys.
Q3. Why must a Solana transaction list every account it will read or write, and what does that enable?
The Sealevel runtime executes transactions in parallel by inspecting their declared account access lists: transactions with no overlapping writable accounts run concurrently. Declaring accounts upfront lets the scheduler detect conflicts statically instead of discovering them mid-execution. The trade-off is developer friction - you must know and pass all accounts, including PDAs, in advance.
Q4. How does Solana's account model differ from Ethereum's contract storage model?
On Ethereum a contract bundles its bytecode and its storage in one address, and state is mutated internally. On Solana programs are stateless executable accounts; all mutable state lives in separate data accounts the program owns. This separation enables the same program to operate over many independent state accounts in parallel and makes upgrade/state boundaries explicit, but requires passing state accounts into every instruction.
Q5. When would you choose Token-2022 over the original SPL Token program, and what is the main risk?
Choose Token-2022 when you need capabilities the base program lacks - transfer fees, transfer hooks, confidential transfers, non-transferable tokens, interest-bearing, or on-mint metadata. The main risk is ecosystem compatibility: it lives at a different program id, so wallets, DEXs, and clients must explicitly support it, and extensions like transfer hooks can inject custom logic (or fees) into every transfer, which is both a feature and an attack surface you must audit.
Q6. What is rent and rent-exemption on Solana, and why does it matter for token accounts?
Every account consumes on-chain storage, so it must hold a minimum lamport balance proportional to its data size to be rent-exempt; below that it would be purged. Practically all accounts today are created rent-exempt by funding that minimum at creation. It matters because creating a mint or an ATA costs SOL, and Token-2022 extensions increase account size, raising the rent-exempt minimum - a common source of 'insufficient funds' errors when creating extended accounts.
Q7. What do the mint authority and freeze authority control, and how do you create a fixed-supply token?
The mint authority is the only signer allowed to mint new tokens; the freeze authority can freeze/thaw individual token accounts, blocking transfers. To make supply permanently fixed, set the mint authority to null (SetAuthority to None) after minting the desired amount - after that no more tokens can ever be created. Leaving the freeze authority set is a centralization signal holders often check, so many projects disable it too.
Q8. A createMint call in @solana/web3.js succeeds but mintTo fails with an owner/program mismatch. How do you diagnose it?
This almost always means the mint was created under one token program but the ATA or mintTo call used the other. Token-2022 and the classic Token program have distinct program ids, and helpers default to the classic one unless you pass the TOKEN_2022_PROGRAM_ID. Verify the mint's owner field matches the program id you pass to getOrCreateAssociatedTokenAccount and mintTo; aligning all three (mint owner, ATA program, mintTo program) resolves it.
