ESLint, Prettier and Pre-commit hooks

Cohort 2.0 · 9 lessons · SDE-2 (3+ yrs) interview prep

Concept summary — all 9 steps

What is linting
Linting is static analysis: a tool parses source without running it and flags bugs, suspicious constructs (unused vars, unreachable code, floating promises), and anti-patterns. The interview angle is understanding it as a compile-time quality gate distinct from tests (behavior) and formatting (whitespace) — it catches a class of errors before they reach runtime or review.
ESLint
ESLint is the de-facto pluggable JS/TS linter: it parses code into an AST (espree by default, @typescript-eslint/parser for TS), traverses nodes, and runs configurable rules that register listeners on node types and emit reports, many with autofixers. Know that its power is extensibility — rules, plugins, and shareable configs — and that config comes in legacy `.eslintrc` (extends/overrides) or the newer flat `eslint.config.js` model.
Code formatting in ESLint
ESLint historically shipped stylistic rules (semi, quotes, indent), but formatting concerns have been deprecated from core and moved to @stylistic; the modern guidance is to NOT format with ESLint and delegate that to Prettier. The interview point is separation of concerns: linting decides code correctness/quality, formatting decides layout, and mixing them causes rule conflicts and slower runs.
VSCode ESLint plugin
The `dbaeumer.vscode-eslint` extension surfaces violations as inline squiggles as you type and can auto-fix on save via `editor.codeActionsOnSave: { source.fixAll.eslint }`. It matters because it shifts feedback left to the editor, but you must still enforce the same rules in CI/pre-commit since not every developer configures their editor identically.
ESLint in a React project
React linting relies on `eslint-plugin-react` (JSX/props correctness), `eslint-plugin-react-hooks` (rules-of-hooks + exhaustive-deps), and often `eslint-plugin-jsx-a11y` for accessibility. The high-value rule to discuss is `exhaustive-deps`, which catches stale-closure bugs in useEffect/useMemo/useCallback dependency arrays — the most common source of subtle React bugs.
Popular ESLint configurations
Shareable configs bundle opinionated rule sets: Airbnb (strict, comprehensive), StandardJS (zero-config, no semicolons), and framework configs like eslint-config-next; `typescript-eslint` provides the recommended TS presets. Know the distinction — a shareable config is consumed via `extends` and sets rules, whereas a plugin ships the rule implementations the config references.
ESLint with Next.js
Next.js ships `eslint-config-next` (invoked via `next lint` or its flat-config export), which bundles the React, React-hooks, and a Next-specific plugin plus the `core-web-vitals` ruleset that flags perf/SEO issues like using `<img>` instead of `next/image` or misusing `<Head>`. It gives sensible defaults out of the box; the discussion point is layering your own rules on top without fighting the preset.
Prettier
Prettier is an opinionated formatter that discards your original styling and reprints the AST to a consistent layout, with a deliberately tiny config surface (printWidth, semi, singleQuote, trailingComma). Run it alongside ESLint and use `eslint-config-prettier` to switch off ESLint's conflicting stylistic rules; avoid `eslint-plugin-prettier` in hot paths since running Prettier as a lint rule is markedly slower and noisier.
Pre-commit hooks
Git pre-commit hooks run a script before a commit is created; teams manage them with Husky and pair them with lint-staged so ESLint/Prettier run only on staged files rather than the whole repo. The nuance to raise: hooks are a fast local guard but are bypassable (`git commit --no-verify`), so CI must re-run the same checks as the real enforcement boundary.

Real-world example

Standardizing code quality across a Next.js + TypeScript team monorepo

You're the SDE-2 setting up tooling for a 12-person Next.js + TypeScript app where PRs are full of formatting noise and recurring useEffect stale-closure bugs. You start from `eslint-config-next` for framework/core-web-vitals rules, layer on `typescript-eslint`'s recommended preset, and enable `react-hooks/exhaustive-deps` as an error. You add Prettier for all formatting and wire `eslint-config-prettier` last in the extends chain so ESLint stops fighting Prettier over quotes and semicolons. Developers get instant editor feedback via the VSCode ESLint plugin with fix-on-save. To stop bad code before it's committed, you install Husky + lint-staged so a pre-commit hook runs `eslint --fix` and `prettier --write` only on staged files, keeping commits fast. Finally, because `--no-verify` can skip the hook, your CI pipeline runs `next lint` and `prettier --check` as the authoritative gate that blocks merges — so local hooks are convenience, and CI is enforcement.

Interview Q&A (SDE-2)

Q1. What's the difference between linting and formatting, and why run both ESLint and Prettier?
Linting is static analysis for code correctness and quality — unused variables, floating promises, rules-of-hooks violations — and can be semantically aware. Formatting is purely about layout (indentation, line width, quotes) and doesn't judge correctness. They're separate concerns, so the modern setup lets ESLint handle quality and Prettier own formatting, wiring `eslint-config-prettier` to disable ESLint's stylistic rules that would otherwise conflict with Prettier.
Q2. How does ESLint work under the hood?
ESLint parses source into an AST using a configurable parser (espree by default, @typescript-eslint/parser for TypeScript). It then walks the tree, and each enabled rule subscribes to specific node types (e.g. `CallExpression`, `VariableDeclarator`); when a listener fires and detects a violation it reports a message and optionally provides a fixer that produces text edits. This AST-based design is why rules are precise and why some can autofix while others can't.
Q3. Explain the react-hooks/exhaustive-deps rule and the risk of disabling it.
It statically checks that every reactive value referenced inside useEffect/useMemo/useCallback appears in the dependency array. Missing dependencies cause stale-closure bugs where the callback captures an outdated value and effects don't re-run when they should. Blindly disabling it with an eslint-disable comment usually hides a real bug; the correct fixes are adding the dep, using a functional state updater, or a ref — not silencing the warning.
Q4. eslint-config-prettier vs eslint-plugin-prettier — which do you use and why?
`eslint-config-prettier` is a config that turns OFF ESLint rules that conflict with Prettier; you add it last in the extends chain and run Prettier as a separate step. `eslint-plugin-prettier` runs Prettier as an ESLint rule so formatting issues appear as lint errors. The recommended approach is eslint-config-prettier plus running Prettier separately, because running Prettier through ESLint is slower and floods the lint output with formatting errors.
Q5. What is the flat config, and how does it differ from the legacy .eslintrc setup?
Flat config (`eslint.config.js`) is ESLint's current configuration model where you export an ordered array of config objects, each with `files`, `plugins` (imported as real JS modules), `languageOptions`, and `rules`. It replaces `.eslintrc`'s cascading/`extends` string-resolution and `overrides` with plain JavaScript composition and explicit imports, which is more predictable and removes the magic plugin-name resolution. Flat config is the default in ESLint 9+, so knowing the migration is increasingly expected.
Q6. How do pre-commit hooks work, why pair them with lint-staged, and why is CI still needed?
A pre-commit hook is a script Git runs before creating a commit; tools like Husky make these hooks version-controlled and easy to install. lint-staged runs linters/formatters only against files in the Git staging area, so commits stay fast instead of scanning the whole repo. CI is still the real gate because hooks run locally and can be skipped with `git commit --no-verify` or an unconfigured machine, so the pipeline must re-run the same checks to actually enforce them.
Q7. What's the difference between a shareable config and a plugin in ESLint?
A plugin (e.g. eslint-plugin-react) is a package that implements and exports rules, processors, and sometimes configs. A shareable config (e.g. eslint-config-airbnb) is a package that sets which rules are enabled and at what severity, typically referencing rules from one or more plugins. In short: plugins provide the rules, configs decide how to use them — and many packages ship both.
Q8. How do you lint TypeScript with ESLint, and what's the performance trade-off?
You use `typescript-eslint`, which provides the parser (@typescript-eslint/parser) and rules, plus recommended presets. Basic rules work off the AST alone, but type-aware rules (like no-floating-promises) require `parserOptions.project` so the parser loads the TypeScript type information. That type-checking makes linting significantly slower and more memory-heavy, so the trade-off is enabling type-aware rules only where their bug-catching value justifies the cost, and tuning the tsconfig scope.
