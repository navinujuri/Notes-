PayTM Part-2

Cohort 2.0 · 5 lessons · SDE-2 (3+ yrs) interview prep

Concept summary — all 5 steps

Get Comfortable with the Repo
Orient yourself in the Turborepo monorepo before writing code: apps/user-app (Next.js, App Router, server actions + NextAuth), apps/bank-webhook (a small Express server), and shared packages/db (the Prisma client + schema) and packages/ui. The interview angle is understanding why a payments product uses a shared DB package (one Prisma client, one schema, imported by every app) and how workspace boundaries keep the bank simulator decoupled from the wallet UI.
Finish onramps (Add Money)
Complete the money-in flow: a server action creates an OnRampTransaction row with status Processing plus a unique token, the user is sent to a mock bank, and the bank calls the bank-webhook server, which credits the user's Balance and flips the row to Success. The point is that crediting is asynchronous and webhook-driven (you never trust the client to say 'I paid') — the key discussion is idempotency and webhook authenticity.
Add transfers (P2P send money)
Implement a sendMoney server action that atomically debits the sender and credits the receiver inside a single Prisma $transaction, using a row lock (SELECT ... FOR UPDATE) on the sender's balance so the sufficient-funds check and the debit can't be interleaved by a concurrent request. This is the crux of the topic: preventing double-spend/lost-update race conditions, and the choice between pessimistic locking and optimistic (version-column) locking.
Add P2P transactions table
Add a p2pTransfer model (fromUserId, toUserId, amount, timestamp) and write that ledger row inside the same transaction that moves the balances, so the money movement and its audit record commit or roll back together. Interview angle: this is the beginning of a double-entry/ledger design — the transactions table is the source of truth for reconciliation and history, while Balance is a derived cache.
Assignment: frontend for P2P transactions
Build the Next.js page (ideally a server component that queries Prisma directly, or a server action) that lists a user's p2pTransfers, distinguishing sent vs received by comparing fromUserId/toUserId against the session user id. The takeaway is scoping every query to the authenticated user (no client-supplied user id), correct amount formatting, and ordering by timestamp.

Real-world example

Concurrency-safe wallet transfers under load

You own the wallet service. A user with a 100 rupee balance fires two 100 rupee transfers nearly simultaneously (double-tap, retry, or a scripted attack). With a naive read-check-write server action, both requests read balance=100, both pass the 'balance >= 100' check, and both debit — the account goes to -100 and money is created out of thin air (lost update / double-spend). The SDE-2 fix is to wrap debit+credit+ledger-insert in a single Prisma $transaction and take a pessimistic row lock via a raw `SELECT balance FROM "Balance" WHERE "userId" = $1 FOR UPDATE` so the second request blocks until the first commits, then re-reads 100→0 and correctly fails the check. In the same transaction you insert the p2pTransfer ledger row, so history and balances stay consistent. Separately, the onramp webhook must be idempotent: the bank retries on timeout, and without deduping on the transaction token you'd credit the deposit twice — so the webhook updates only rows still in Processing (a conditional update) and returns 200 on the already-Success case.

Interview Q&A (SDE-2)

Q1. Why is 'Add Money' (onramp) implemented via a webhook from the bank instead of just crediting the balance when the user clicks pay?
Because the wallet never actually holds the funds until the bank confirms settlement, and settlement is asynchronous — it can succeed, fail, or be delayed after the user leaves your page. The webhook is the bank's authoritative, out-of-band callback saying money really moved, so you credit only on that signal. Crediting on the client click would let anyone fake a deposit; the OnRampTransaction sits in Processing until the trusted server-to-server webhook flips it to Success.
Q2. How do you prevent a double-spend when two transfers hit the same account concurrently?
Do the whole operation in one DB transaction and serialize access to the sender's balance row with a pessimistic lock — in this repo a raw `SELECT ... FOR UPDATE` inside Prisma's $transaction. The lock forces the second request to wait until the first commits, so the balance check reads the post-debit value and correctly rejects the overdraft. Without the lock, both requests read the stale balance, both pass the check, and you get a lost update. The alternative is optimistic locking (a version/updatedAt column with a conditional update and retry), better under low contention.
Q3. What's the difference between Prisma's array-form $transaction and the interactive (callback) form, and why does this project need the interactive one?
The array form (`prisma.$transaction([...])`) just batches independent queries into one atomic commit but gives you no intermediate results or branching. The interactive form (`prisma.$transaction(async tx => {...})`) lets you run a query, read its result, make decisions (like the balance check), and issue further writes — all on the same connection/transaction. Transfers need the interactive form because you must lock and read the balance, branch on sufficiency, and then conditionally debit/credit; you also drop to `tx.$queryRaw` there because Prisma has no first-class SELECT FOR UPDATE.
Q4. The bank webhook can be called more than once for the same deposit. How do you make it safe?
Make it idempotent: key on the unique transaction token and only apply the credit if the OnRampTransaction is still in Processing, ideally via a single conditional/atomic update (update ... where status = Processing) rather than read-then-write, and return 200 on an already-processed token so the bank stops retrying. Wrap the balance increment and the status flip in one transaction so they can't partially apply. Separately, the endpoint must be authenticated — a signature/HMAC or shared secret from the bank — because the reference implementation trusting an unauthenticated JSON body is a real vulnerability.
Q5. Why write the p2pTransfer ledger row inside the same transaction as the balance updates instead of after?
Atomicity: the money movement and its record must both commit or both roll back. If you insert the ledger row after the transaction commits and that insert fails (crash, deploy, DB blip), balances change with no history — you can't reconcile or show the user what happened; the reverse leaves a phantom transaction that never moved money. Putting debit, credit, and the ledger insert in one transaction guarantees the audit trail always matches the balances.
Q6. How should the balance/amount be stored — float, decimal, or integer?
Never floating point — binary floats can't represent decimal currency exactly and accumulate rounding errors. Store money as an integer in the smallest unit (paise/cents), so 100.00 rupees is 10000, which makes arithmetic exact and comparisons safe; the 100xDevs schema uses an Int amount for exactly this reason. If you need fractional units or large ranges, use a fixed-precision DECIMAL/NUMERIC type, not double.
Q7. How do you correctly scope the P2P transactions page so a user only sees their own history?
Derive the user id from the server-side session (NextAuth's getServerSession), never from a client-supplied parameter, and query `p2pTransfer where fromUserId = me OR toUserId = me` on the server (server component or server action) ordered by timestamp desc. You then label each row sent vs received by comparing fromUserId to the session id. Doing the query on the client with a user id from props/query-string would be an IDOR — anyone could read others' transactions.
Q8. What database isolation level matters here, and what anomaly are you defending against?
The core anomaly is the lost update (and a read-then-write race on the balance). At the default Read Committed level a plain read-check-write is still vulnerable, which is why you add explicit row locking (SELECT FOR UPDATE) rather than relying on the isolation level alone. You could instead raise the transaction to Serializable and handle serialization-failure retries, but explicit pessimistic locking on the single hot row is simpler and more predictable for this write pattern.
