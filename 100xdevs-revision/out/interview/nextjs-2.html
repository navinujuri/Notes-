NextJS (Server side)

Cohort 2.0 · 12 lessons · SDE-2 (3+ yrs) interview prep

Concept summary — all 12 steps

Next.js as a fullstack framework
Next.js is not just a React renderer; it ships a backend runtime so a single deployable serves UI, API endpoints, and server-rendered pages. The interview angle is knowing the three backend surfaces (Server Components for reads, Route Handlers for public HTTP APIs, Server Actions for mutations) and when each is appropriate versus reaching for a separate Express/Nest service.
React client-side data fetching recap
Classic CSR fetches data in useEffect after mount, forcing loading/error state machines, request waterfalls, and exposing API keys to the browser. This is the baseline problem Next's server model solves; be ready to articulate why useEffect fetching hurts SEO, TTFB, and creates client-server round-trips.
Data fetching in Server Components
App-router Server Components are async functions that await data (fetch or a DB call) on the server before HTML is produced, so there is no client loading spinner and no exposed credentials. Interview points: fetch is deduped/cached per request, secrets stay server-side, and you push the fetch to where the data lives instead of round-tripping through the browser.
Loading UI with loading.tsx and Suspense
A loading.tsx (or explicit <Suspense>) wraps a route segment so Next streams a fallback instantly while the async Server Component resolves, then swaps in the real content. Know that this is React Suspense + streaming SSR under the hood, letting the shell paint fast while slow data streams in, avoiding an all-or-nothing blocking render.
Route Handlers (API routes)
app/api/*/route.ts exports named functions (GET, POST, etc.) that receive a Web Request and return a Response/NextResponse — the way to build actual HTTP endpoints for third-party clients, webhooks, or mobile apps. Contrast with the old pages/api handler signature and know these run on the server (Node or Edge runtime) with full access to secrets and DB.
Consolidating the backend into the app
Instead of a standalone API server, you move endpoints into Next Route Handlers so frontend and backend deploy together, share types, and skip CORS. The trade-off worth stating: simpler DX and colocated types, but coupled scaling/deploy and a serverless execution model that complicates long-lived connections and background jobs.
Signup frontend (client form)
A Client Component ('use client') holds form state and POSTs credentials to the signup endpoint, handling validation feedback and pending state. The angle is the client/server boundary: only interactive, stateful pieces need 'use client', and you should never trust client-side validation alone.
Signup backend (validation + hashing)
The POST handler validates input (e.g. zod), checks for existing users, hashes the password with bcrypt/argon2, and persists — never storing plaintext. Interview depth: why hashing (not encryption) with a salt and adaptive cost, returning generic errors to avoid user enumeration, and doing all validation server-side.
Databases with Prisma
Prisma provides a typed schema, migrations, and a generated client so DB access is type-safe end-to-end. Be ready on migrate dev vs migrate deploy, how the schema maps to SQL, and that Prisma is an ORM/query builder you call directly from Server Components and handlers.
Better fetches: skip the HTTP hop
From a Server Component you should call the DB (or a shared service function) directly rather than fetching your own /api route, because self-fetching adds a needless HTTP round-trip and serialization overhead. The principle: Route Handlers are for external callers; internal server code should invoke the data layer directly.
Singleton Prisma client
In dev, hot-reload re-imports modules and instantiates a new PrismaClient each time, exhausting DB connections; the fix is caching the instance on globalThis in non-production. Interview relevance: this generalizes to any expensive connection-holding client under serverless/HMR, and to connection pooling limits (pgBouncer/Data Proxy) in production.
Server Actions for mutations
Server Actions ('use server' functions) let you invoke server code straight from a form or event without hand-writing an API route, and pair with revalidatePath/revalidateTag to refresh cached data. Know the internals: Next generates a hidden POST endpoint per action, so treat inputs as untrusted and enforce auth/validation just like any endpoint.

Real-world example

Building auth + a dashboard for a SaaS app in a single Next.js app

You are building the onboarding and dashboard for a B2B SaaS. The marketing pages and the app live in one Next.js (app router) deployment backed by Postgres via Prisma. The signup page is a Client Component that POSTs to a POST Route Handler (app/api/auth/signup/route.ts), which validates with zod, checks for an existing email, hashes the password with bcrypt, and inserts the user. After login, the dashboard route is an async Server Component that reads the user's projects by calling Prisma directly (no self-fetch to /api), so the HTML arrives populated with no client spinner and no exposed DB URL. Slow widgets (usage charts) are wrapped in Suspense with a loading.tsx so the shell streams instantly. 'Create project' and 'rename' are handled by Server Actions that mutate via Prisma and call revalidatePath to refresh the list. In dev you hit 'too many connections' from Postgres because HMR keeps making new PrismaClients — you fix it with the globalThis singleton pattern, and before production you front Postgres with a pooler since each serverless invocation opens its own connection. The mobile app team later needs the same data, so you keep the JSON Route Handlers as the public, versioned API while the web app keeps calling the data layer directly.

Interview Q&A (SDE-2)

Q1. When do you use a Server Component, a Route Handler, and a Server Action?
Use Server Components for reads that render into a page — await the DB/fetch directly so data is server-rendered with no client waterfall. Use Route Handlers for public HTTP endpoints consumed by external clients (mobile, webhooks, third parties) or when you need fine control over the Request/Response. Use Server Actions for mutations triggered from the UI (form submit, button click) where you want to skip authoring an endpoint and get progressive enhancement plus cache revalidation. The rule of thumb: reads in components, mutations in actions, and explicit HTTP contracts in handlers.
Q2. Why is fetching your own /api route from a Server Component considered an anti-pattern?
The Server Component already runs on the server with direct access to your DB and service functions, so calling your own API means an extra HTTP round-trip, JSON serialization/deserialization, and a second execution context for no benefit. It adds latency and can even deadlock in constrained serverless setups. Instead, extract a plain async function (the data layer) and call it from both the Server Component and any Route Handler that external clients need.
Q3. Explain the Prisma singleton problem in Next.js and how you fix it.
In development, Next's hot module replacement re-evaluates modules on each change, and if you `new PrismaClient()` at import time you spawn a new client — and a new connection pool — every reload, quickly hitting Postgres's connection limit. The fix is to store the client on `globalThis` and reuse it, only creating a fresh instance in production where modules aren't reloaded. In production serverless, the deeper issue is that each function instance opens its own pool, so you add an external pooler (pgBouncer, Prisma Accelerate/Data Proxy) to cap total connections.
Q4. How does loading.tsx / Suspense streaming actually work?
Next uses React's streaming SSR: the route shell renders immediately and any subtree wrapped in Suspense (which loading.tsx creates automatically for a segment) emits a fallback in the initial HTML stream. As each async Server Component resolves on the server, Next streams the finished HTML chunk plus a small script that swaps out the fallback in place. This gives fast first paint and avoids blocking the whole page on the slowest query, without shipping a client-side loading state machine.
Q5. What are the security considerations with Server Actions?
A Server Action compiles to a hidden POST endpoint that anyone can invoke with crafted arguments, so you must treat its inputs as fully untrusted — validate with something like zod and enforce authentication/authorization inside the action, never relying on the fact that it's 'only' called from your UI. Next includes origin/CSRF protections for actions, but that doesn't replace authz checks. Also avoid closing over sensitive data in the action's bound arguments, since those are serialized to the client.
Q6. How does caching and revalidation work for data fetching in the app router?
Next extends fetch with caching semantics: by default many fetches are cached (force-cache) and you opt into freshness with `{ next: { revalidate: N } }` for time-based ISR or `{ cache: 'no-store' }` for always-dynamic. Tag-based invalidation (`next: { tags: [...] }` plus `revalidateTag`) and `revalidatePath` let mutations surgically bust the cache after a Server Action writes data. The mental model is a request-scoped dedupe layer plus a persistent data cache, and you should be explicit rather than relying on defaults, which have shifted across Next versions.
Q7. What are the trade-offs of running your backend inside Next.js versus a separate service?
Colocating gives shared TypeScript types, no CORS, one deploy, and simpler local dev, which is ideal for product teams shipping a web app. The costs: you inherit the serverless/edge execution model (short-lived, stateless), which is awkward for long-running jobs, websockets, or holding DB connections, and you couple frontend and backend scaling and release cadence. For heavy async workloads, streaming pipelines, or multiple consumers, a dedicated service (or offloading to queues/workers) is usually the better boundary.
Q8. How should password signup be implemented securely in a Route Handler?
Validate and normalize input server-side (never trust the client), then check for an existing account and hash the password with an adaptive algorithm like bcrypt or argon2 using a per-user salt and a sensible cost factor — hashing, not reversible encryption. Return generic, uniform responses and timing to avoid leaking whether an email is registered (user enumeration). Store only the hash, issue a session/JWT on success, and put the handler behind rate limiting to blunt credential-stuffing.
