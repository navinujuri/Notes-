AWS Deploying (EC2)

Cohort 2.0 · 8 lessons · SDE-2 (3+ yrs) interview prep

Concept summary — all 8 steps

What is AWS
AWS is a cloud provider offering on-demand, pay-per-use infrastructure (compute, storage, networking, managed services) across globally distributed Regions and Availability Zones. The interview angle is understanding the shared responsibility model (AWS secures the cloud, you secure what's in it) and why you'd pick IaaS like EC2 over PaaS/serverless — control and flexibility vs. operational overhead.
EC2 servers
EC2 provides resizable virtual machines (instances) launched from an AMI, sized by instance type (vCPU/RAM/network, e.g. t3.micro vs c7g), and billed on-demand, reserved, or spot. Know the trade-offs: burstable T-series (CPU credits) vs fixed-performance families, EBS-backed persistence vs instance-store ephemerality, and spot's ~70-90% savings at the cost of interruption.
Creating a new EC2 server
Provisioning means choosing an AMI, instance type, key pair, VPC/subnet, and a security group (stateful virtual firewall controlling inbound/outbound by port/CIDR). The critical interview point is security-group hygiene: open only 22 (ideally to your IP, not 0.0.0.0/0) and your app/HTTP(S) ports, and understand public vs private IP and the role of an Internet Gateway.
SSH into server
You connect with `ssh -i key.pem ec2-user@<public-ip>` using the private half of the key pair; the public key lives in the instance's authorized_keys. Interview-worthy details: key files must be chmod 400 or SSH refuses them, password auth is disabled by default, and prefer SSM Session Manager in production to avoid opening port 22 or managing keys at all.
Install the repo
Provision the runtime by cloning the code and installing dependencies (git, Node/Python, `npm install`, build, then run under a process manager like pm2 or a systemd unit so it survives reboots and crashes). The angle here is repeatability — doing this by hand doesn't scale, which is why real teams bake AMIs, use user-data/cloud-init bootstrap scripts, or containerize.
Try hitting the server
You verify the app is reachable by curling `http://<public-ip>:<port>` from outside. When it fails, debug systematically down the layers: process actually listening (ss/netstat, bound to 0.0.0.0 not 127.0.0.1?), security group inbound rule, NACL/subnet routing, then the app itself — this layered diagnosis is exactly what interviewers probe.
nginx
nginx sits in front of your app as a reverse proxy, terminating client connections on 80/443 and forwarding to the app on localhost. It matters because it handles TLS termination, gzip, static-file serving, load balancing across multiple app processes, and rate limiting — so your Node/app process never faces the raw internet. Know why a reverse proxy beats exposing the app port directly (security, buffering slow clients, unified TLS).
Certificate management
HTTPS requires a TLS certificate; the common free path is Let's Encrypt via certbot, which issues 90-day certs and auto-renews, wired into nginx to terminate TLS. Interview depth: the ACME challenge (HTTP-01/DNS-01) proving domain ownership, the trust chain (leaf → intermediate → root CA), and that in production you'd often offload TLS to an ALB/ACM instead of managing certbot on the box.

Real-world example

Deploying a Node.js API for a side project to production on a single EC2 box

You've built a Node/Express API and want it live at api.myapp.com. You launch a t3.small EC2 instance from an Ubuntu AMI in a public subnet, create a key pair, and set a security group allowing 22 (from your IP only), 80, and 443. You SSH in with the .pem key, install Node and git, clone the repo, run `npm ci && npm run build`, and start it under pm2 listening on localhost:3000. A quick `curl localhost:3000/health` works, but hitting the public IP on port 3000 fails until you realize the app should sit behind nginx — so you install nginx as a reverse proxy forwarding :80 to :3000, point the api.myapp.com A record at the instance's Elastic IP, then run certbot to issue a Let's Encrypt cert and enable HTTPS with auto-renewal. Now traffic flows client → nginx (TLS on 443) → Node on localhost, and you've got a repeatable mental model for the next debugging session: check the process, the security group, then nginx, then the cert. The natural follow-up an SDE-2 anticipates is that this single box is a SPOF, so the next iteration moves to an ALB + Auto Scaling Group with ACM-managed certs.

Interview Q&A (SDE-2)

Q1. Why put nginx in front of your Node app instead of exposing the app port directly to the internet?
nginx as a reverse proxy handles TLS termination, serves static assets efficiently, buffers slow clients (protecting your app's event loop from slowloris-style stalls), and can load-balance across multiple app processes. It also lets you run the app on an unprivileged localhost port while nginx owns 80/443, and centralizes concerns like gzip, rate limiting, and access logging. Exposing the app directly means it faces raw untrusted traffic and you'd have to reimplement all of that in application code.
Q2. Your app runs fine with `curl localhost:3000` on the box but is unreachable from the public IP. How do you debug?
Work up the layers. First confirm what the process binds to — `ss -tlnp` should show 0.0.0.0:3000, not 127.0.0.1:3000, since localhost-only binds are unreachable externally. Next check the security group has an inbound rule for that port and source, then verify subnet routing (Internet Gateway, public IP/Elastic IP) and NACLs. If all networking is open and it still fails, it's the reverse proxy config or the app itself.
Q3. What's the difference between a security group and a network ACL?
A security group is stateful and attached to an ENI/instance: return traffic for an allowed inbound connection is automatically permitted, and it only supports allow rules. A NACL is stateless and operates at the subnet level: you must explicitly allow both directions, it evaluates numbered rules in order, and it supports both allow and deny. Security groups are your primary instance firewall; NACLs are a coarser subnet-wide backstop, often used to blanket-block specific IPs.
Q4. How does Let's Encrypt / certbot actually prove you own the domain and keep certs valid?
certbot uses the ACME protocol with a challenge — typically HTTP-01, where the CA asks you to serve a specific token at a well-known URL on port 80, or DNS-01, where you publish a TXT record (required for wildcard certs). Once ownership is proven, it issues a 90-day cert. A systemd timer or cron runs `certbot renew` periodically, which reissues and reloads nginx before expiry; the short lifetime is intentional to limit exposure from key compromise.
Q5. When would you move TLS termination off the EC2 box to an ALB with ACM instead of running certbot?
Once you have more than one instance or want zero-downtime scaling, an Application Load Balancer with ACM-managed certs is better: ACM auto-renews certs with no box-level cron, terminates TLS at the edge, and spreads traffic across an Auto Scaling Group. It removes the single point of failure and the per-instance cert management. You'd keep certbot-on-the-box only for simple single-node deployments where you don't want the ALB's cost and complexity.
Q6. Why is SSM Session Manager often preferred over SSH with key pairs in production?
Session Manager gives you a shell through the AWS API using IAM permissions, so you can close port 22 entirely and don't distribute or rotate .pem files — a major reduction in attack surface and key-sprawl risk. Sessions are logged to CloudTrail/S3/CloudWatch for audit, and access is governed by IAM policies rather than filesystem key ownership. It works on instances in private subnets with no bastion, as long as the SSM agent and an appropriate instance role are present.
Q7. What are the trade-offs between on-demand, reserved, and spot instances?
On-demand is pay-as-you-go with no commitment — best for unpredictable or short-lived workloads. Reserved instances (or Savings Plans) give up to ~72% discount for a 1- or 3-year commitment, ideal for steady baseline load. Spot uses spare capacity at up to ~90% off but can be reclaimed with a two-minute warning, so it's suited to fault-tolerant, interruptible work like batch jobs or stateless workers behind an ASG — never for a stateful single node you can't afford to lose.
Q8. How do you make your deployed app survive reboots and crashes, and why does that matter?
Run it under a process manager — pm2 or, more robustly, a systemd unit with `Restart=always` and `WantedBy=multi-user.target` — so it restarts on failure and starts on boot. Running `npm start` in your SSH session dies the moment the session closes, so a supervised service is essential. This is also where you'd wire in log rotation and health checks; the deeper point is that manual setup doesn't scale, which pushes teams toward baked AMIs, cloud-init bootstrap scripts, or containers for reproducible provisioning.
