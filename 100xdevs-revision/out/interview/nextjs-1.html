NextJS (Client side)

Cohort 2.0 · 14 lessons · SDE-2 (3+ yrs) interview prep

Concept summary — all 14 steps

NextJS Intro & prerequisites
Next.js is a React meta-framework that bakes in routing, server rendering, API routes and bundling. You reach for it over CRA/Vite when you need SEO, server data-fetching or a single codebase for front + back. Prereq is solid React + JS fundamentals.
SEO Optimisation
A client-only React app ships an empty <div id=root> — crawlers and link-preview bots see nothing. Next server-renders real HTML with correct <title>/<meta>, so pages are indexable and shareable. Use the Metadata API for per-route SEO.
Waterfalling problem
In client React, a parent renders → fetches → then the child renders → fetches, creating sequential round-trips (waterfalls). Server Components fetch on the server, close to the DB, and can parallelise — eliminating client waterfalls and spinner chains.
Next.js offerings
File-based routing, SSR/SSG/ISR, API routes, image/font optimisation, middleware, and zero-config bundling. Know these as the 'why Next' checklist in interviews.
Bootstrap a Next app
create-next-app scaffolds the App Router: app/ holds routes, page.tsx is the UI for a route, layout.tsx wraps it. Understand the app/ vs pages/ router distinction.
Routing in Next
Routing is the filesystem: a folder = a URL segment, page.tsx = that route's UI, nested folders = nested routes, [id] = dynamic segments. No react-router config.
Prettify the signin page
Composing UI from components (often + Tailwind). Reinforces that presentational structure is just React; interactivity is where the server/client split starts to matter.
Server side rendering
In the App Router components are Server Components by default — they render to HTML on the server, get streamed to the browser, then hydrate. Faster first paint, better SEO, secrets stay server-side.
Layouts
layout.tsx wraps its route subtree and PERSISTS across navigation — shared nav/sidebar/footer don't unmount or re-fetch when you move between child pages. Key for perf and UX.
Layouts in sub routes
Layouts nest: each segment can have its own layout, composing top-down (root → section → page). Lets you scope shared UI to just part of the app.
Merging routes (route groups)
Route groups (folder) organise files and share layouts WITHOUT adding a URL segment — useful for grouping (marketing) vs (app) sections cleanly.
components directory
Colocating reusable, non-route components outside the routing tree keeps app/ focused on routes. A structure/organisation concern interviewers probe for maintainability.
onClick handler (interactivity)
The moment you add an event handler or a hook, that component must be a Client Component ('use client') — Server Components can't have interactivity or browser APIs.
Client and server components
The core mental model: Server Components (default) ship zero JS, can fetch and touch secrets; Client Components ('use client') run in the browser for state/effects/handlers. You can import client into server and pass server-rendered children down, but not the reverse.

Real-world example

Real-world: the signin + dashboard of a SaaS app

Picture the cohort's own app. The /dashboard route is a Server Component: it fetches the user's data directly from the DB on the server (no client waterfall, DB creds never reach the browser) and streams ready HTML — so it's fast and SEO-clean. The persistent sidebar lives in layout.tsx, so navigating between dashboard tabs never re-renders or re-fetches it. The signin FORM, however, needs state and an onClick submit, so it's a Client Component ('use client'). This split — server for data + shell, client for islands of interactivity — is the pattern you defend in an interview.

Interview Q&A (SDE-2)

Q1. When do you use a Server Component vs a Client Component, and what happens if you fetch data in a Client Component?
Default to Server Components for data-fetching, static content and anything touching secrets/DB — they ship no JS. Use Client Components only for interactivity (state, effects, event handlers, browser APIs). Fetching in a Client Component works but moves the request to the browser (extra round-trip, exposes endpoints, re-introduces waterfalls and loading states) — usually a smell unless the data is user-triggered/real-time.
Q2. How does Next.js eliminate the request waterfall that plain React suffers from?
By moving fetches to the server. Server Components run on the server next to the data source and can fire fetches in parallel before any HTML is sent, so the client receives a fully-populated page instead of rendering, discovering it needs data, fetching, then re-rendering down the tree.
Q3. What is hydration and what's its cost?
Hydration is React attaching event listeners and reconstructing component state on top of the server-rendered HTML in the browser. Cost: the JS bundle for client components must download and execute before the page is interactive (TTI lag), and a server/client markup mismatch throws hydration errors. Minimising client components keeps hydration cheap.
Q4. SSR vs SSG vs ISR — how do you choose?
SSG (build-time HTML) for content that rarely changes — fastest, cacheable at the edge (docs, marketing). SSR (per-request) for personalised or always-fresh data (dashboards). ISR (static + background revalidation) for mostly-static content that updates periodically (product pages, blogs) — you get SSG speed with a freshness window.
Q5. Does 'use client' mean the component only runs in the browser?
No — a common misconception. Client Components are still server-rendered once to HTML for the initial load (good for SEO/first paint); 'use client' marks them as also needing to run in the browser (hydrate) so they can be interactive. They run in both places, not client-only.
Q6. How do nested layouts render, and why do they help performance?
Layouts compose from the root down and preserve state across navigation within their subtree — a parent layout doesn't unmount when a child page changes. So shared chrome (nav, sidebar) and its data survive route transitions, avoiding needless re-renders/re-fetches and giving instant-feeling navigation.
Q7. Why is SEO fundamentally better in Next than in a CRA/Vite SPA?
A SPA serves an empty HTML shell and builds the DOM in JS after load — crawlers/preview bots that don't execute JS (or execute it partially) see nothing, and meta tags aren't present at request time. Next sends complete HTML with route-specific metadata (via the Metadata API) on the first response, so content and social/OG tags are there immediately.
