Authentication

Cohort 3.0 · 12 lessons · SDE-2 (3+ yrs) interview prep

Concept summary — all 12 steps

Context: authentication vs authorization
Authentication answers "who are you" (proving identity via credentials); authorization answers "what can you do" (permissions/roles). Interviewers probe this distinction constantly — auth failures are 401, authz failures are 403, and conflating them is a common design smell.
Auth workflow — bank analogy
The bank teller checks your ID (login/credentials), then hands you a token/passbook that proves you're allowed to transact without re-proving identity each time. The mental model: prove identity once, then carry a bearer credential for subsequent stateless requests. It frames why HTTP (stateless) needs a token passed on every call.
Auth workflow — technical flow
Signup stores a hashed password (bcrypt/argon2, never plaintext); login verifies the hash and issues a credential (session id or token); every protected request presents that credential. Key interview point: passwords are salted+hashed, and the login response is where the trust boundary is established.
Creating an Express app
Bootstrap an Express server with express.json() body-parsing, route handlers, and a listen port — the substrate for auth endpoints. The interview-relevant detail is middleware ordering: body parsers and CORS must run before route handlers, and auth middleware sits between the two.
Creating an authenticated endpoint
An endpoint that returns data only when the caller presents a valid credential, initially tracked with an in-memory/opaque token map. This exposes the core problem: the server must look up the token to know who's calling, which motivates the stateful-vs-stateless tradeoff next.
Tokens vs JWTs
Opaque tokens are random strings the server must store and look up (stateful — easy to revoke, needs a store/DB per request); JWTs are self-contained signed payloads the server verifies with a secret (stateless — no lookup, but revocation is hard). The classic tradeoff: server-side session store vs. stateless scalability, and revocation is the JWT weakness.
JWTs — structure and signing
A JWT is base64url(header).base64url(payload).signature, where the signature is HMAC/RSA over header+payload using a server secret (HS256) or private key (RS256). Verification recomputes the signature to guarantee integrity — the payload can't be tampered without invalidating it. Know the difference between HS256 (shared secret) and RS256 (asymmetric, good for multi-service verification).
Replacing token logic with JWT
Swap the in-memory token map for jwt.sign(payload, secret) on login and jwt.verify(token, secret) on protected routes — the server no longer stores per-user session state. This is the concrete refactor from stateful to stateless auth and demonstrates why JWTs scale horizontally without a shared session store.
JWTs can be decoded by anyone
The payload is base64-encoded, NOT encrypted — anyone with the token can read its claims at jwt.io. The signature protects integrity (tamper detection), not confidentiality, so never put secrets/passwords in a JWT; if you need confidentiality use JWE or keep sensitive data server-side. This is a frequent gotcha question.
Assignment: auth middleware
A reusable Express middleware that extracts the token from the Authorization: Bearer header, verifies it, attaches req.userId, and calls next() or returns 401. This is the DRY pattern for protecting many routes and the standard place interviewers expect verification and error handling to live.
Writing the frontend
On login the client stores the JWT (localStorage/sessionStorage or cookie) and sends it on subsequent requests via the Authorization header. Interview angle: localStorage is vulnerable to XSS, httpOnly cookies are vulnerable to CSRF — the storage choice is a security tradeoff, not a default.
Assignment: TODO app
A full-stack build tying signup/login/JWT/middleware together with per-user data isolation — every TODO query is scoped to req.userId from the verified token. It forces the authorization step: authenticated is not enough, you must ensure users only access their own resources (prevents IDOR/broken object-level authorization).

Real-world example

A multi-tenant TODO/notes API for a small SaaS

You're building the backend for a notes app used by thousands of users. On POST /signup you bcrypt-hash the password and store it; on POST /login you compare the hash and return a signed JWT containing { userId } (HS256, 24h expiry). The React frontend saves the token and sends it as Authorization: Bearer <token> on every call. An auth middleware verifies the signature, rejects expired/tampered tokens with 401, and sets req.userId. Every notes query is then scoped by req.userId so user A can never read user B's notes. In production you hit the classic issues: a leaked token can't be revoked before expiry (so you add short TTLs plus a refresh-token flow and a small server-side denylist), you realize you accidentally put the user's email in the JWT payload which is publicly decodable, and you debug a 403-vs-401 mixup where a valid token was hitting an admin-only route. This scenario exercises hashing, stateless verification, per-user authorization, and the real operational tradeoffs of JWTs.

Interview Q&A (SDE-2)

Q1. What's the difference between authentication and authorization, and which HTTP status codes map to each?
Authentication verifies identity — proving you are who you claim (login, credentials, token validity); its failure is 401 Unauthorized. Authorization verifies permissions — whether an authenticated principal may perform an action or access a resource; its failure is 403 Forbidden. A request can be authenticated (valid token) yet unauthorized (a normal user hitting an admin route).
Q2. Opaque tokens vs JWTs — when would you choose each?
Opaque (random) tokens are stateful: the server stores them and looks each one up, which makes revocation trivial (delete the row) but requires a shared session store and a DB hit per request. JWTs are stateless and self-verifying via a signature, so they scale horizontally with no lookup, but you can't easily revoke one before expiry. Choose JWTs for scalable, short-lived access with a refresh flow; choose opaque tokens when instant revocation and central session control matter more than avoiding lookups.
Q3. If a JWT can be decoded by anyone, what actually makes it secure?
The payload is base64url-encoded, not encrypted, so anyone can read the claims — security comes from the signature, not secrecy. The server signs header+payload with a secret (HS256) or private key (RS256); verification recomputes the signature, so any tampering with the claims invalidates it. It guarantees integrity and authenticity, not confidentiality — so never store secrets in the payload.
Q4. How do you handle JWT revocation and logout given tokens are stateless?
Since a valid JWT stays valid until it expires, you can't truly "log out" server-side without adding state. Common approaches: keep access tokens short-lived (minutes) plus a long-lived refresh token that can be revoked in the DB; maintain a denylist/blocklist of revoked token IDs (jti) checked on each request; or bump a per-user token version that's embedded as a claim and compared. Each reintroduces some server state, which is the fundamental tradeoff against stateless auth.
Q5. Where should the frontend store a JWT, and what are the security tradeoffs?
localStorage/sessionStorage is simple and immune to CSRF but readable by any injected script, so it's exposed to XSS. An httpOnly cookie is invisible to JavaScript (mitigates XSS token theft) but is sent automatically, exposing you to CSRF unless you add SameSite and/or CSRF tokens. There's no free lunch — pick based on your threat model; httpOnly + SameSite=strict/lax cookies is often the safer default for browser apps.
Q6. How do you securely store passwords, and why not just hash with SHA-256?
Use a slow, salted, adaptive hash like bcrypt, scrypt, or argon2 with a per-user salt and a tunable work factor. Fast hashes like SHA-256 are the problem: they let attackers try billions of guesses per second and enable rainbow-table/precomputation attacks. Adaptive functions are deliberately slow and memory-hard, and the per-user salt ensures identical passwords produce different hashes, defeating precomputed tables.
Q7. What is broken object-level authorization (IDOR) and how does it apply to the TODO app?
IDOR happens when an endpoint authenticates the user but doesn't check that the requested resource belongs to them — e.g. GET /todos/123 returns todo 123 regardless of owner. Being authenticated is not being authorized; every data access must be scoped to req.userId derived from the verified token (WHERE userId = req.userId), not just to "any logged-in user." It's OWASP's top API risk and a very common interview/design bug.
Q8. Why put auth logic in Express middleware instead of each route handler?
A single middleware extracts and verifies the token, handles 401s, and attaches req.userId in one place, so protected routes stay DRY and can't accidentally skip verification. It centralizes the trust boundary — one audited code path for token parsing and error handling — and lets you compose it (e.g. authMiddleware then adminMiddleware). Ordering matters: it must run after body/CORS parsing and before the protected handlers.
