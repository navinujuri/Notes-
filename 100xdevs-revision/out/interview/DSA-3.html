Concept summary — all 6 steps

Overall Plan / Roadmap
Sets the study arc for DSA: master how to measure cost (complexity) before touching data structures, so every later choice (array vs. hashmap, sort vs. scan) is justified by asymptotics rather than intuition. Interview angle: interviewers expect you to state and defend the time/space complexity of every solution unprompted — this module is the vocabulary for that.
Complexity Analysis
Quantifies how an algorithm's running time and memory scale with input size n, independent of hardware or language, by counting dominant operations. It matters because the same problem has solutions differing by orders of magnitude at scale (n=10^6 makes O(n^2) ~10^12 ops, infeasible); interviewers judge whether you can reason about scalability, not just correctness.
Asymptotic Notations
Big-O is the asymptotic upper bound (worst case), Big-Omega the lower bound (best case), and Big-Theta the tight bound when upper and lower match; drop constants and lower-order terms. The common interview trap is conflating O with worst case and average case — O(n^2) quicksort is worst case while its Theta on random input is n log n, so state which case you mean.
Problems / Applied Analysis
Hands-on derivation: count loop iterations, nested loops (multiply), sequential blocks (add, keep max), and log factors from halving. The skill tested is spotting hidden costs — e.g., string concatenation in a loop is O(n^2), or a .contains() on a list inside a loop turns O(n) into O(n^2).
Common Complexity Classes
Memorize the ladder: O(1) < O(log n) < O(n) < O(n log n) < O(n^2) < O(2^n) < O(n!), and map each to a pattern — hashmap lookup, binary search, single scan, comparison sort, nested loop, subset enumeration, permutations. Interviewers want you to instantly recognize which class a candidate solution falls into and whether it clears the input constraints.
Master Theorem
A formula for recurrences of the form T(n)=aT(n/b)+f(n) (divide-and-conquer), comparing f(n) against n^(log_b a) across three cases to get the closed-form bound without unrolling. It's the fast way to answer 'why is merge sort O(n log n)?' (a=2,b=2,f=n → case 2) or binary search O(log n) (a=1,b=2,f=1), a frequent follow-up when you propose a recursive solution.

Real-world example

Choosing a lookup strategy for a 50M-row user-permissions service

You own an authorization service that checks, per request, whether a user has a given permission across ~50M user-role rows. The v1 implementation loads roles into a List and does a linear .contains() scan per check — O(n) per lookup, and under load with n in the millions this dominates p99 latency. Profiling confirms the hot path scales linearly with tenant size. You migrate the membership check to a HashSet (O(1) average lookup, O(n) build amortized at load), and for range-style queries ('all permissions granted before time T') you keep a sorted array plus binary search (O(log n)). When a teammate proposes a recursive tree-merge to reconcile role hierarchies, you use the Master Theorem to show the merge-based approach is O(n log n) versus the O(n^2) pairwise reconciliation, justifying the design in the review. Every decision here — the data structure swap, the sort-then-binary-search, the recurrence analysis — is the complexity vocabulary from this module applied to a real latency bug.

Interview Q&A (SDE-2)

Q1. What's the difference between Big-O, Big-Theta, and Big-Omega, and why does saying 'O(n^2)' alone underspecify an algorithm?
Big-O is an asymptotic upper bound, Big-Omega a lower bound, and Big-Theta a tight bound that holds only when the upper and lower bounds coincide. Big-O says the algorithm is no worse than the bound but doesn't claim it's achieved — O(n^2) is technically a correct (if loose) bound for an O(n) algorithm. It also doesn't specify which input distribution you mean; you should pair the notation with a case, e.g. 'O(n^2) worst case, Theta(n log n) on average,' as with quicksort.
Q2. Why do we drop constants and lower-order terms, and when does that abstraction bite you in practice?
Asymptotic analysis describes growth as n approaches infinity, where constants and lower-order terms become negligible relative to the dominant term, giving a hardware-independent comparison. It bites you at small or bounded n and in constant-factor-sensitive hot paths: an O(n) algorithm with a huge constant (cache misses, syscalls) can lose to an O(n log n) one for realistic sizes, and two O(n log n) sorts can differ 3x in wall-clock. Big-O guides algorithm selection; profiling settles constant factors.
Q3. State the Master Theorem and use it to explain merge sort's complexity.
For T(n)=aT(n/b)+f(n) with a>=1, b>1, compare f(n) to n^(log_b a): case 1, if f is polynomially smaller, T=Theta(n^(log_b a)); case 2, if f=Theta(n^(log_b a)), T=Theta(n^(log_b a) * log n); case 3, if f is polynomially larger (and regularity holds), T=Theta(f(n)). Merge sort is T(n)=2T(n/2)+O(n), so a=2,b=2, n^(log_2 2)=n, and f(n)=n matches — case 2 — giving Theta(n log n).
Q4. Give an example of code that looks O(n) but is actually O(n^2), and how to fix it.
Building a string by repeated concatenation in a loop (s += token) is O(n^2) in languages with immutable strings because each concatenation copies the whole accumulated string; the fix is a StringBuilder/list-join for O(n). Similarly, calling list.contains(x) inside a loop over the list is O(n^2) — replace the inner membership test with a HashSet for O(1) average, making the whole thing O(n). The tell is a hidden linear operation nested inside another linear loop.
Q5. How does amortized complexity differ from worst-case, using a dynamic array as the example?
Amortized complexity is the average cost per operation over a sequence, guaranteeing the total, even though individual operations may spike. A dynamic array's push is O(n) worst case on the resize that copies all elements, but because capacity doubles, resizes happen geometrically rarely, so the total cost of n pushes is O(n) — amortized O(1) per push. It differs from average-case analysis: amortized makes no probabilistic assumption about inputs, it's a guaranteed bound over the worst sequence.
Q6. Given constraints like n <= 10^5 or n <= 20 in a problem statement, how do you back out the intended complexity?
Assume roughly 10^8 simple operations per second as the feasibility budget. n <= 10^5 permits about O(n log n) or O(n sqrt n) (~10^6–10^7 ops) but rules out O(n^2) at 10^10; n <= 10^3 tolerates O(n^2); n <= 20 signals exponential O(2^n) subset enumeration or O(n! ) permutations is expected. Reading constraints backwards is a fast way to eliminate solution classes before you start coding — interviewers give constraints deliberately.
Q7. When is O(log n) better than O(1) in real systems, or when would you accept worse asymptotics on purpose?
Asymptotically O(1) always wins at scale, but you accept worse asymptotics when constant factors, memory, or simplicity dominate for your actual n. A balanced BST's O(log n) ordered lookup can beat a hash table's O(1) when you also need range queries, sorted iteration, or predictable (non-amortized, no rehash-spike) latency. Similarly a linear scan over a small contiguous array often beats a hashmap due to cache locality, so 'better Big-O' is a starting heuristic, not the final answer.
