installing-node-assignments

Cohort 2.0 · 7 lessons · SDE-2 (3+ yrs) interview prep

Concept summary — all 7 steps

Installing Node.js
Node is the V8 engine plus libuv (the event loop and thread pool) packaged as a server-side runtime, installed via the official installer, a version manager (nvm/fnm/volta), or a package manager. Interview angle: know why a version manager beats a global install (per-project versions via .nvmrc, no sudo, painless upgrades) and the LTS-vs-Current distinction — pin LTS for production.
Running code with Node
You execute scripts with `node file.js`, experiment in the REPL, and pin dependencies through npm/package.json; this is the shift from browser JS (DOM, no filesystem) to a runtime with `fs`, `process`, `require`/`import`, and CommonJS-vs-ESM module systems. Interview angle: explain the difference between the browser and Node globals (`window` vs `global`/`globalThis`), and how `type: module` toggles ESM.
Easy JS assignments (week 2)
Fundamentals drills: string/array manipulation, loops, and pure functions that take input and return output without side effects. Interview angle: these map directly to the warm-up round — demonstrate clean naming, edge-case handling (empty input, off-by-one), and choosing the right built-in (`map`/`filter`/`reduce`) over manual loops.
Medium JS assignments (week 2)
Composite problems combining object manipulation, higher-order functions, and small algorithms (counting, grouping, transforming datasets). Interview angle: this is where time/space complexity starts to matter — recognize when a `Map`/`Set` turns an O(n²) nested scan into O(n), and reason about mutation vs immutability.
Hard JS assignments
Problems requiring data-structure design, recursion, or non-trivial state (e.g. an in-memory store, parsing, or a class-based abstraction like a calculator/todo engine). Interview angle: expect to defend your design — encapsulation, invariants, and how you'd extend it — plus handle tricky JS semantics like closures, `this` binding, and reference vs value.
Easy async assignment
First contact with asynchrony: reading a file or firing a timer, then handling the result with a callback, a Promise, or `async/await`. Interview angle: articulate why Node is single-threaded but non-blocking — the event loop offloads I/O to libuv so the main thread never waits — and why blocking calls (`fs.readFileSync`, tight CPU loops) are harmful in a server.
Medium async
Coordinating multiple async operations: sequential vs parallel execution, `Promise.all`/`allSettled`/`race`, error propagation through `try/catch` around `await`, and avoiding the callback-pyramid/unhandled-rejection traps. Interview angle: know when to parallelize independent I/O (`Promise.all`) vs when order matters, and how one rejected promise in `all` fails the whole batch while `allSettled` does not.

Real-world example

Building a small file-processing CLI batch job

You're an SDE-2 asked to write a Node CLI that reads a directory of JSON order files, validates and transforms each record, enriches them by calling a pricing API, and writes a consolidated report. You start by pinning the runtime (an .nvmrc set to LTS via nvm so teammates and CI match). The synchronous JS core — parsing, validating, grouping orders by customer with a Map — is your 'easy/medium/hard' JS work. The async layer is where it gets real: you read files with fs/promises, fan out the API enrichment calls with Promise.all for the independent requests (parallel, fast), but keep the final write sequential. You wrap awaits in try/catch, switch to Promise.allSettled so one failed pricing lookup doesn't abort the entire batch, and confirm you never used a blocking readFileSync in the hot path so the event loop stays responsive. This one script exercises the entire topic: install, run, pure-JS logic, and both easy and coordinated async.

Interview Q&A (SDE-2)

Q1. Node is single-threaded — how does it handle thousands of concurrent I/O operations without blocking?
The JavaScript executes on one main thread, but I/O is delegated to libuv, which uses the OS async facilities (epoll/kqueue/IOCP) and a small thread pool for operations like filesystem and DNS. When an operation completes, its callback is queued and the event loop picks it up when the call stack is clear. So the thread is never idle waiting on I/O; it's only blocked by synchronous CPU-bound work, which is why you offload heavy computation to worker threads or a separate service.
Q2. What actually breaks if you call fs.readFileSync in a request handler on a busy server?
The synchronous call blocks the single main thread until the disk read finishes, so every other in-flight request — even cheap ones — stalls behind it, tanking throughput and latency. The event loop cannot process queued callbacks during that window. Sync APIs are fine at startup (loading config once) but should never sit in a per-request hot path; use fs.promises or the callback form there.
Q3. Compare Promise.all, Promise.allSettled, and Promise.race and when you'd pick each.
Promise.all runs promises concurrently and resolves with all results, but rejects immediately if any one rejects — use it when you need every operation to succeed and want fail-fast. allSettled always waits for all and returns a status/value/reason per promise — use it for batch jobs where partial success is acceptable and you want to report failures. race settles as soon as the first promise settles (resolve or reject) — useful for timeouts (race a work promise against a delayed reject).
Q4. CommonJS vs ES Modules in Node — what are the practical differences?
CommonJS uses require/module.exports, loads synchronously, and resolves at runtime, so you can require conditionally. ESM uses import/export, is statically analyzable (enabling tree-shaking), loads asynchronously, and is the standard the ecosystem is moving to. You opt into ESM with "type": "module" in package.json or the .mjs extension; interop gotchas include no __dirname in ESM and the fact that you can't require() an ESM-only package — you need dynamic import().
Q5. Why use a Node version manager like nvm/fnm instead of the system-wide installer?
A version manager lets you run different Node versions per project (pinned via .nvmrc) without root, so upgrading one service doesn't break another and CI can match local exactly. It sidesteps EACCES permission issues from installing global packages under a system-owned prefix. The system installer is fine for a single-purpose box, but on a dev machine juggling multiple codebases, per-project pinning prevents 'works on my machine' drift.
Q6. How does async/await relate to Promises, and what's the risk of awaiting sequentially?
async/await is syntactic sugar over Promises — an async function always returns a Promise, and await pauses the function until the awaited Promise settles without blocking the event loop. The common mistake is awaiting independent operations one after another (await a; await b), which serializes them needlessly; if they don't depend on each other, kick them off first and await Promise.all([a, b]) to run them concurrently. Also wrap awaits in try/catch, since a rejected awaited Promise throws.
Q7. What is an unhandled promise rejection and how do you guard against it?
It's a rejected Promise with no .catch or surrounding try/catch; in modern Node it emits an unhandledRejection event and by default crashes the process. You guard against it by always handling rejections at the boundary — try/catch around awaits, .catch() on floating promises, and allSettled for batches. A process-level process.on('unhandledRejection') handler is a last-resort safety net for logging, not a substitute for handling errors where they occur.
Q8. When would you reach for worker threads or clustering in Node?
Worker threads are for CPU-bound work (image processing, crypto, large parsing) that would otherwise block the event loop — they run JS in parallel on separate threads with message passing. Clustering (or a process manager like PM2) forks multiple processes to use all CPU cores for handling I/O-bound HTTP load, since one Node process only uses one core for JS. Rule of thumb: worker threads for heavy computation, cluster/multiple processes for scaling concurrent request handling.
