Node.js Runtime

Cohort 3.0 · 9 lessons · SDE-2 (3+ yrs) interview prep

Concept summary — all 9 steps

What is Node.js?
A server-side JavaScript runtime that embeds Google's V8 engine and pairs it with libuv, which provides the event loop and a thread pool for async I/O. It is not a language or framework: the key mental model is a single-threaded event loop that offloads blocking work (file, network, DNS, crypto) to libuv/OS so JS code stays non-blocking. Interview angle: explain why CPU-bound work stalls the loop while I/O-bound work scales well.
What is bun?
An alternative JS runtime built on Apple's JavaScriptCore (not V8) and written in Zig, bundling a runtime, package manager, bundler, and test runner into one binary. Its selling points are fast cold start and much faster installs, plus first-class TypeScript/JSX execution without a build step. Interview angle: know the trade-off — speed and DX vs Node's larger ecosystem maturity and npm-compatibility edge cases.
Starting a Node.js project
`npm init` (or `-y`) scaffolds a package.json, the manifest that declares name, version, entry point (`main`), module system (`type`), `scripts`, and dependency ranges. It is the contract that tooling and the resolver read. Interview angle: know what `type: module` vs default CommonJS changes, and that `scripts` run in a shell with node_modules/.bin on PATH.
npm
Node's default package manager: a CLI plus the public registry plus semver-based dependency resolution that materializes a node_modules tree. Understand `install` vs `--save-dev`, semver ranges, and that install runs lifecycle scripts (postinstall) — a common supply-chain risk. Interview angle: contrast npm with pnpm (content-addressed store, symlinks) and yarn on disk layout and speed.
Internal packages
Node's built-in core modules (fs, path, http, crypto, os, stream, events) shipped with the runtime — no install, no node_modules entry. Best practice is the `node:` prefix (e.g. `node:fs`) to disambiguate from userland packages and avoid shadowing. Interview angle: know which are sync vs async APIs and that most core modules expose both callback and promise (`fs/promises`) forms.
External packages
Third-party libraries pulled from the npm registry into node_modules and located at runtime by Node's resolution algorithm (walk up node_modules directories, honor package.json `exports`/`main`). These carry transitive dependency trees, so the real concern is bloat, versioning, and supply-chain trust. Interview angle: explain how the resolver finds a module and why duplicate versions can coexist in the tree.
package-lock.json
The lockfile that pins the exact resolved version, source URL, and integrity hash of every package in the full tree, making installs deterministic across machines and CI. Commit it. Interview angle: `npm install` may mutate the lock to satisfy manifest ranges, while `npm ci` installs strictly from the lock (failing on drift) and is the correct choice for CI/reproducible builds.
Assignment #1 — Create a CLI
Build an executable Node script: add a `#!/usr/bin/env node` shebang, parse `process.argv` (index 2 onward for user args), and expose it via the package.json `bin` field so `npm link`/`npx`/global install create a PATH shim. Interview angle: know how `npx` resolves and runs a bin, and why the shebang + executable bit matter on POSIX.
Assignment #2
A follow-on exercise applying the same runtime primitives — typically reading/writing files with `node:fs`, doing async work, or calling an API with `fetch` (global since Node 18) and printing formatted output. The point is composing core modules, argument parsing, and proper exit codes/error handling into a small real tool. Interview angle: use `process.exitCode`/non-zero exits and handle rejections rather than letting the process crash silently.

Real-world example

Shipping an internal `deploy-notes` CLI to the team via npm

Your team needs a small CLI that reads a service's changelog, calls an internal release API, and posts a summary. You start with `npm init -y`, set `type: module`, and add a `bin` entry pointing at `cli.js` with a `#!/usr/bin/env node` shebang. Core work uses internal packages — `node:fs/promises` to read the changelog and `node:path` to resolve paths — while `process.argv` parsing selects the target environment. For HTTP you use the built-in global `fetch` (Node 18+) instead of adding axios, keeping the dependency tree lean; the one external package you do add (say `chalk` for colored output) lands in `dependencies`, and `vitest` goes in `devDependencies`. You commit `package-lock.json` so CI runs `npm ci` for byte-identical, reproducible installs, and teammates run the tool via `npx` after `npm link`. When someone reports the tool runs slowly on cold start in a scratch container, you evaluate running it under bun for faster startup and installs — and confirm your reliance only on standard APIs keeps it portable across both runtimes.

Interview Q&A (SDE-2)

Q1. Node.js is single-threaded, so how does it handle thousands of concurrent connections?
JavaScript executes on one thread via the event loop, but I/O is delegated to libuv, which uses OS async primitives (epoll/kqueue/IOCP) and a small worker thread pool for operations that can't be async at the OS level (file I/O, DNS, crypto). While an I/O request is outstanding the loop is free to process other callbacks, so I/O-bound concurrency scales cheaply. The catch is CPU-bound work: a heavy synchronous computation blocks the single thread and stalls all connections, which is why you offload to worker_threads or a separate process.
Q2. What's the difference between dependencies, devDependencies, and peerDependencies?
`dependencies` are needed at runtime and are installed for anyone who installs your package. `devDependencies` are only needed during development/build/test (bundlers, linters, test runners) and are skipped for consumers and with `npm install --production`. `peerDependencies` declare a package your code expects the host project to provide (e.g. a React plugin expecting `react`), avoiding duplicate/mismatched copies; npm v7+ auto-installs them by default, which can surface version conflicts.
Q3. Why commit package-lock.json, and when do you use `npm ci` vs `npm install`?
The lockfile records exact versions and integrity hashes for the entire transitive tree, so committing it guarantees every machine and CI run resolves an identical node_modules. `npm install` reads the manifest ranges and may update the lockfile to satisfy them, so it can drift. `npm ci` ignores the manifest for resolution, installs strictly from the lockfile, errors if the two are out of sync, and wipes node_modules first — making it the deterministic, faster choice for CI and reproducible builds.
Q4. Explain semver ranges: what do `^1.2.3` and `~1.2.3` allow?
`^1.2.3` allows any version compatible under the same major — `>=1.2.3 <2.0.0` — so it picks up minor and patch updates. `~1.2.3` is narrower, allowing patch updates only — `>=1.2.3 <1.3.0`. The premise is that maintainers follow semver (breaking changes bump major), but that trust plus floating ranges is exactly why the lockfile exists to pin the actual installed version. For `0.x` versions the caret behaves more conservatively, treating the minor as breaking.
Q5. CommonJS vs ES Modules in Node — what actually differs?
CommonJS uses `require`/`module.exports`, loads synchronously, and resolves at runtime, so you get dynamic requires and a live `__dirname`. ESM uses `import`/`export`, is statically analyzable (enabling tree-shaking), loads asynchronously, and top-level `await` is allowed; `__dirname` is gone in favor of `import.meta.url`. Node selects the system via the file extension (`.mjs`/`.cjs`) or the package.json `type` field, and interop is asymmetric — ESM can import CJS, but CJS can only reach ESM via dynamic `import()`.
Q6. How does Node resolve `require('lodash')` or an import of a bare specifier?
For a bare specifier (no `./` or `/`), Node walks up from the current file's directory checking each `node_modules` folder until it finds a matching package. Inside that package it consults the `exports` map first (which can gate and remap entry points), falling back to `main` or `index.js`. Because each dependency can have its own nested node_modules, multiple versions of the same package can legitimately coexist in one tree, which explains duplicate installs and version-specific bugs.
Q7. When would you choose bun over Node.js, and what are the risks?
Bun is attractive when startup latency, install speed, and zero-config TypeScript/JSX matter — short-lived CLIs, scripts, CI, and fast local dev — and it consolidates runtime, package manager, bundler, and test runner into one tool. The risks are ecosystem maturity: it runs on JavaScriptCore rather than V8, and some npm packages relying on Node-specific internals or native addons may behave differently or break. For long-lived production services the safer default is still Node, given its operational track record and broader tooling/observability support.
Q8. How do you turn a Node script into an installable CLI command?
Add a `bin` field in package.json mapping a command name to your script, and start that script with a `#!/usr/bin/env node` shebang so the OS runs it with Node. On install (global, `npm link`, or via `npx`) npm creates a shim on PATH that points at your file. Inside, read arguments from `process.argv` (real args start at index 2), and set a non-zero `process.exitCode` on failure so shells and CI can detect errors.
