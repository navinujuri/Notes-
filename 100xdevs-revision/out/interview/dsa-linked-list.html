Linked List

Cohort 2.0 · 6 lessons · SDE-2 (3+ yrs) interview prep

Concept summary — all 6 steps

Intro to Linked Lists
A linear structure of nodes where each node holds a value plus a pointer to the next node, so elements live at scattered heap addresses rather than one contiguous block. The trade-off vs arrays is the whole interview: O(1) insert/delete at a known position vs O(n) random access, and poor cache locality (each hop is a potential cache miss) that makes them slower in practice than Big-O suggests.
Core operations
Traverse, search (O(n)), insert, and delete — all reduce to careful pointer rewiring while handling the head, tail, and empty-list edge cases. Insert/delete at the head is O(1); at an arbitrary position it's O(n) to reach the node but O(1) to splice. A dummy/sentinel head node is the standard trick to avoid special-casing head mutations.
Reversing a list
The canonical answer is the iterative three-pointer walk (prev, curr, next) that flips each next pointer in place — O(n) time, O(1) space. Recursion is elegant but costs O(n) stack. Interviewers extend this to reverse-in-groups-of-k or reverse a sublist [m,n], which test whether you actually understand the pointer bookkeeping.
Cycle detection
Floyd's tortoise-and-hare: advance one pointer by 1 and another by 2; if they ever meet there is a cycle, in O(n) time and O(1) space (beats the O(n)-space hash-set approach). Follow-up: reset one pointer to head and advance both by 1 — they meet at the cycle's entry node, which is the real gotcha to be able to prove.
Add two numbers
Two numbers stored as linked lists of digits (typically least-significant first); walk both lists simultaneously summing digit+carry and building the result node by node. It matters because it drills carry propagation, unequal lengths, and a trailing carry that creates a final node — the exact edge cases interviewers watch for. Same pattern powers big-integer arithmetic beyond native int width.
Doubly linked list
Each node adds a prev pointer, enabling O(1) deletion given only a node reference and bidirectional traversal, at the cost of extra memory and two pointers to keep consistent on every mutation. This is the backbone of an LRU cache and of many language runtimes' ordered collections — the O(1) 'unlink a known node' property is exactly what a singly linked list can't give you.

Real-world example

Building an LRU cache for a service layer

You're adding an in-memory LRU cache in front of a slow datastore, needing O(1) get and put with eviction of the least-recently-used entry. The standard design is a hash map (key -> node) combined with a doubly linked list ordered by recency: the map gives O(1) lookup, and the doubly linked list lets you unlink a touched node and move it to the front in O(1) — impossible with a singly linked list because you'd need O(n) to find the predecessor. On every get/put you splice the node to the head; on overflow you drop the tail. A sentinel head and tail node eliminate null-checks at the boundaries. This single scenario exercises node structure, O(1) insert/delete, pointer rewiring, and precisely why 'doubly' beats 'singly' — and debugging it usually means chasing a mis-updated prev/next pointer that corrupts the list ordering.

Interview Q&A (SDE-2)

Q1. When would you pick a linked list over a dynamic array, given arrays usually win in practice?
Choose a linked list when you need frequent O(1) insertion/deletion at positions you already hold a reference to (e.g., splicing nodes in an LRU cache or intrusive OS queues), or when you can't tolerate the amortized O(n) copy of an array resize. Otherwise arrays almost always win: contiguous memory means cache-friendly iteration and no per-node pointer overhead. The honest SDE-2 answer is that linked lists are rarely the right default and their O(1) claims are dominated by cache misses on real hardware.
Q2. Explain Floyd's cycle detection and how to find the node where the cycle begins.
Move a slow pointer 1 step and a fast pointer 2 steps per iteration; if they meet, a cycle exists, else fast reaches null. It works because once both are in the cycle the fast pointer gains one step per iteration on the slow one, so it must eventually land on it. To find the entry, keep one pointer at the meeting point and reset the other to head, then advance both by 1 — the distance math (head-to-entry equals meeting-point-to-entry around the loop) guarantees they meet exactly at the cycle start. It's O(n) time, O(1) space.
Q3. Walk through reversing a singly linked list iteratively and its complexity.
Maintain prev=null and curr=head; each step save next=curr.next, point curr.next=prev, then advance prev=curr and curr=next. When curr is null, prev is the new head. It's O(n) time and O(1) space, which is why it's preferred over the recursive version that consumes O(n) stack and can overflow on long lists. The one-liner insight: you're flipping each pointer exactly once while carrying three references so you never lose the rest of the list.
Q4. Why does an LRU cache need a doubly linked list rather than a singly linked one?
On a cache hit you must move the touched node to the most-recently-used end in O(1), which requires unlinking it from its current position — and unlinking needs the predecessor. A doubly linked list stores prev, so given the node (found via the hash map) you rewire node.prev.next and node.next.prev in constant time. A singly linked list would force an O(n) scan to find the predecessor, breaking the O(1) guarantee that's the entire point of the structure.
Q5. What are the edge cases in the 'add two numbers as linked lists' problem?
Unequal list lengths (keep consuming the longer one), a final carry that must create a new most-significant node (e.g., 5+5 -> 0->1), and one list being empty. If digits are stored most-significant-first instead of least-significant-first you must reverse both first or use a stack. Using a dummy head for the result list avoids special-casing the first appended node.
Q6. How do you remove the nth node from the end in a single pass?
Use two pointers off a dummy head: advance a lead pointer n+1 steps ahead, then move both until lead hits null; the trailing pointer now sits just before the target, so you splice it out with trailing.next = trailing.next.next. The dummy head cleanly handles removing the actual head node. It's O(n) time, O(1) space, and one pass instead of the naive length-then-index two-pass approach.
Q7. How would you detect whether two singly linked lists intersect, and find the intersection node?
Two-pointer trick: pA walks list A then continues onto B, pB walks B then A; after at most len(A)+len(B) steps they either meet at the intersection node or both reach null (no intersection). It equalizes the path lengths so the pointers align, giving O(m+n) time and O(1) space. Note intersection means shared nodes by identity, not equal values — after the shared node the tails are identical objects.
Q8. What's the memory trade-off of doubly vs singly linked lists, and what's an XOR linked list?
A doubly linked list costs an extra pointer per node (16 bytes on 64-bit) to gain O(1) deletion-by-reference and backward traversal. An XOR linked list stores a single field holding prev XOR next, recovering both neighbors during a walk using the previously visited address, halving pointer memory. It's a clever trick but rarely used: it breaks garbage-collected languages, defeats debuggers and memory-safety tooling, and the savings almost never justify the fragility.
