Basics of DOM

Cohort 2.0 · 8 lessons · SDE-2 (3+ yrs) interview prep

Concept summary — all 8 steps

What is the DOM?
The DOM is a live, in-memory tree representation of an HTML document that the browser builds during parsing; each tag, attribute, and text node becomes an object the browser exposes to JavaScript. It matters because it is the contract between JS and rendered UI — the DOM is not the HTML source string, it is a stateful object graph you mutate, and understanding that distinction explains why re-serializing (innerHTML) differs from reading the live tree.
Static HTML
Static HTML is markup shipped fully-formed from the server (or a file) and rendered as-is, with no client-side mutation after parse. The interview angle is understanding its trade-offs: fast first paint, SEO-friendly, cacheable at the CDN — but no interactivity without added JS, which motivates the move to dynamic manipulation.
Dynamic HTML
Dynamic HTML is content the browser generates or mutates at runtime via JavaScript touching the DOM (in response to events, fetches, or timers) rather than being present in the initial payload. The key insight for an SDE-2 is that this shifts work to the client and affects SEO/first-contentful-paint, and that every mutation risks triggering layout/reflow, so batching and minimizing DOM writes matters for performance.
Fetching / selecting elements
You obtain node references with querySelector/querySelectorAll (CSS-selector based) or the older getElementById/getElementsByClassName/getElementsByTagName. The interview nuance is the difference between a static NodeList (querySelectorAll — a snapshot) and a live HTMLCollection (getElementsBy* — auto-updates as the DOM changes), plus that getElementById is fastest and querySelector returns only the first match.
Updating elements
You change existing nodes by setting textContent/innerHTML, mutating attributes (setAttribute, or properties like .value/.className), or editing .style/classList. The critical distinction is textContent (safe, plain text, no parsing) vs innerHTML (parses markup and is an XSS vector with untrusted input), and that property assignment (el.className) differs from attribute assignment (setAttribute) for form/state properties like value and checked.
Deleting elements
Remove nodes with element.remove() (modern) or parent.removeChild(child) (older, returns the detached node). The interview angle is memory/leak awareness: detaching a node does not automatically clear event listeners or references held elsewhere, so a removed subtree still reachable via a JS variable or a listener closure will not be garbage-collected.
Adding elements
Create nodes with document.createElement, populate them, then attach via appendChild/prepend/append or the more flexible insertBefore/insertAdjacentElement. The performance point is that inserting nodes one-by-one into the live tree forces repeated reflows, so you build in a DocumentFragment (or set innerHTML once) and insert in a single operation to reflow only once.
More complex elements / composition
Building real components means composing subtrees — nested elements, attributes, data-* datasets, and event handlers wired before insertion — and often delegating events to a stable ancestor rather than binding per-child. The staff-level takeaway is event delegation (one listener on a container using event.target and closest()) for dynamic lists, plus preferring createElement/textContent over string concatenation to stay XSS-safe and avoid re-parsing existing markup.

Real-world example

Rendering and mutating a live comments feed

You are building the comments section of a product page. On load, static server-rendered HTML shows the first page of comments for fast FCP and SEO. Then JS fetches newer comments and appends them dynamically: you createElement each comment, set textContent (never innerHTML, since comment bodies are untrusted user input), and batch them into a DocumentFragment so the list reflows once instead of N times. A single delegated click listener on the <ul> container handles Edit/Delete for every row via event.target.closest('.comment') — so newly added comments work without rebinding. Edit updates the node's textContent in place; Delete calls comment.remove() and you also drop any cached reference to it so it can be garbage-collected. This one feature exercises the whole topic: static vs dynamic rendering, selecting, updating, deleting, adding, and composing complex nodes safely and performantly.

Interview Q&A (SDE-2)

Q1. What is the difference between the DOM and the HTML source, and between the DOM and what the browser paints?
The HTML source is a text string; the browser parses it once to construct the DOM, a live object tree that then holds the document's state — subsequent JS mutations change the DOM, not the original source. The render tree (and pixels painted) is derived from the DOM plus CSSOM after layout, so the DOM can hold nodes that aren't visible (display:none) and the painted result can lag DOM changes until the next reflow/repaint. Practically, reading innerHTML re-serializes the current live DOM, which will differ from the bytes the server sent.
Q2. querySelectorAll vs getElementsByClassName — what's the difference and when does it bite you?
querySelectorAll returns a static NodeList — a snapshot taken at call time — while getElementsByClassName/getElementsByTagName return a live HTMLCollection that updates automatically as the DOM changes. The classic bug is looping over a live collection while removing matching nodes: the collection shrinks under you and you skip elements. Live collections can also be a subtle perf trap because re-reading .length forces the browser to re-query the DOM.
Q3. textContent vs innerHTML vs innerText — trade-offs?
textContent gets/sets raw text without parsing markup, is fastest, and is XSS-safe, but returns text of hidden nodes too. innerHTML parses its input as HTML — powerful for building markup but a direct XSS vector if the string contains untrusted data, and it destroys and recreates the subtree (dropping listeners). innerText is layout-aware (reflects visible, rendered text and triggers reflow to compute), so it's slower and should be avoided when you just want the text content.
Q4. Why can naively adding many elements in a loop hurt performance, and how do you fix it?
Each insertion into the live tree can invalidate layout, and interleaving reads (offsetHeight, getBoundingClientRect) with writes forces synchronous layout thrashing where the browser reflows repeatedly. The fix is to batch: build the subtree in a DocumentFragment or a detached parent and insert once, so only one reflow occurs; alternatively construct an innerHTML string and assign it in a single write. Also separate all DOM reads from writes to avoid forced synchronous reflow.
Q5. What is event delegation and why is it important for dynamic DOM content?
Event delegation attaches a single listener to a stable ancestor and uses event bubbling with event.target/closest() to handle events from many descendants. It matters because dynamically added children automatically work without rebinding listeners, it reduces memory from thousands of individual handlers, and it survives re-renders. The caveats are that it only works for events that bubble (focus/blur don't, though focusin/focusout do) and you must guard target matching so unrelated inner elements don't trigger the handler.
Q6. Does removing an element from the DOM free its memory? What causes DOM-related memory leaks?
remove()/removeChild detaches the node from the tree, but the memory is only reclaimed once nothing references it. Leaks happen when a JS variable, a closure inside an event listener, or an external structure (a Map/array cache) still holds the detached node — it stays reachable and uncollectable, along with its whole subtree. Detached-node leaks are common with SPAs; mitigate by nulling references, using event delegation instead of per-node listeners, and leveraging removeEventListener or AbortController for cleanup.
Q7. Attribute vs property: what's the difference between setAttribute('value', x) and element.value = x?
Attributes are the initial values parsed from HTML markup; properties are the live state on the DOM object. For form controls they diverge: the value attribute reflects the default/initial value (what a form reset restores), while the .value property holds the current user-edited value. So setAttribute('value',...) won't visibly update an input the user has typed into, and reading getAttribute('value') won't return the user's current text — you need the .value property. Similar divergence applies to checked, and some like id stay in sync.
Q8. Why is directly injecting user input via innerHTML dangerous, and what are safer alternatives?
innerHTML parses the string as HTML, so attacker-controlled input can inject <script>, event-handler attributes, or <img onerror> payloads — a stored/reflected XSS. Safer options: use textContent for plain text, createElement plus setAttribute/property assignment to build nodes, and sanitize with a vetted library (DOMPurify) or the Sanitizer API when you must render rich HTML. Framework binding (React/Vue) escapes by default, which is why dangerouslySetInnerHTML/v-html are explicitly named escape hatches.
