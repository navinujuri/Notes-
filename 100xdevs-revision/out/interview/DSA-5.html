C++ Part 1

Cohort 2.0 · 8 lessons · SDE-2 (3+ yrs) interview prep

Concept summary — all 8 steps

C++ Setup & Toolchain
Installing a compiler (g++/clang++/MSVC), invoking it with a standard flag (e.g. -std=c++17), and understanding the four-stage pipeline: preprocess → compile → assemble → link. Interview angle: know why an 'undefined reference' is a linker error (not a compile error) and why the chosen -std flag changes which language features and overload resolutions are available.
Anatomy of a C++ Program
main() is the entry point returning int (0 = success), #include pulls in declarations via the preprocessor, and `using namespace std` / `std::` controls name lookup. Interview angle: static/global objects are constructed before main runs and destroyed after it returns, and dynamic initialization order across translation units is unspecified (the 'static init order fiasco').
I/O with Streams
std::cin/std::cout are buffered stream objects; >> and << are overloaded operators, and >> skips whitespace and does type-checked extraction. Interview angle: `std::endl` flushes the buffer (expensive in loops — prefer '\n'), and `std::ios::sync_with_stdio(false)` + `cin.tie(nullptr)` is the standard trick to make cin/cout competitive with scanf/printf in hot input loops.
Data Types & Representation
Fixed set of primitives (int, long, char, bool, float, double) whose sizes are only minimum-guaranteed, plus signed/unsigned variants and fixed-width aliases (int32_t, size_t). Interview angle: signed integer overflow is undefined behavior while unsigned wraps modulo 2^n; implicit signed↔unsigned conversions and float equality/precision are classic bug sources — prefer fixed-width types when layout or range matters.
Conditional Statements
if/else, the ternary ?:, and switch for dispatching on an integral/enum value. Interview angle: switch cases fall through unless you break (intentional fallthrough is a bug magnet), a dense switch can compile to a jump table (O(1)) vs an if-else chain's sequential O(n) comparisons, and && / || short-circuit — relied on for guards like `if (p && p->x)`.
Looping Constructs
for, while, do-while (runs at least once), range-based for over containers, plus break/continue. Interview angle: reason about loop invariants and off-by-one boundaries, know that range-based for uses iterators under the hood so mutating the container during iteration can invalidate them, and prefer range-for when you don't need the index.
Functions
Declaration vs definition, parameter passing by value / reference (&) / const-reference / pointer, overloading, default arguments, and recursion with its stack-frame cost. Interview angle: pass large objects by const& to avoid copies, use non-const& or pointer to signal mutation/optional, and lean on const-correctness; understand recursion depth risks (stack overflow) and when to convert to iteration.
Arrays
Fixed-size, contiguous, zero-indexed blocks with no bounds checking; a C-array decays to a pointer to its first element when passed to a function, losing its size. Interview angle: out-of-bounds access is undefined behavior, sizeof(arr) gives the byte length only in the declaring scope (not after decay), and production code should prefer std::array (stack, size in type) or std::vector (heap, dynamic) which carry their size and support .at() bounds checking.

Real-world example

A CLI sensor-log summarizer

You're asked to write a small command-line tool that reads temperature readings from stdin (piped from a sensor log) and prints min, max, and average. You set up the build with `g++ -std=c++17 -O2 summarize.cpp`. In main() you read a count then loop with a range/index loop pulling values via std::cin into a fixed std::array<double, N> (or a std::vector when N is unknown at compile time). You factor the statistics into small functions — `double average(const std::vector<double>&)` taking a const-reference to avoid copying the buffer — and use conditional checks to skip malformed lines and guard against an empty input (which would divide by zero). This one tool exercises every step in the topic: toolchain setup, the main() lifecycle, buffered stream I/O (turning off sync and using '\n' instead of endl for speed), correct numeric data types (double for the mean, size_t for the count to avoid signed/unsigned mismatch), conditionals, loops, functions with const-ref passing, and array/contiguous storage — and the classic bug you'll debug is either an off-by-one on the read loop or integer overflow when summing counts into the wrong type.

Interview Q&A (SDE-2)

Q1. Walk me through what happens from source file to running executable in C++.
Four stages: the preprocessor expands #includes and macros into a single translation unit; the compiler turns that into assembly for the target; the assembler produces an object file (.o/.obj) with possibly unresolved symbols; the linker resolves those symbols across object files and libraries into an executable. This is why a missing function definition surfaces as an 'undefined reference' at link time rather than a compile error, and why each .cpp compiles independently — headers provide declarations so units can be compiled separately and linked together.
Q2. Why is `std::endl` often a performance problem, and how do you speed up cin/cout?
`std::endl` writes '\n' and then flushes the stream buffer to the OS; doing that inside a tight loop forces a syscall per line and kills throughput — use '\n' and let the buffer flush naturally (or on program exit). To speed up standard streams further, call `std::ios::sync_with_stdio(false)` to decouple C++ streams from C stdio, and `std::cin.tie(nullptr)` so cout isn't flushed before every cin read. Together these make iostream competitive with scanf/printf in I/O-heavy problems.
Q3. What is array decay, and how does it affect sizeof?
When a C-style array is passed to a function (or assigned to a pointer), it 'decays' to a pointer to its first element, losing the length information carried by its type. So `sizeof(arr)` returns the full array size in bytes only in the scope where it's declared; inside a function receiving it as a parameter, sizeof yields the pointer size (e.g. 8), not the array size. That's why you must pass the length separately for C-arrays, or use std::array/std::vector/std::span which retain their size.
Q4. How do you decide between pass-by-value, const reference, non-const reference, and pointer?
Pass by value for small, cheap-to-copy types (int, pointers) or when you need an independent copy anyway. Pass by const reference for large read-only objects to avoid the copy while guaranteeing you won't mutate. Use a non-const reference when the function must modify the caller's object and the argument is always required. Use a pointer when the argument is optional (can be nullptr) or when you want to make 'this may be reassigned/null' explicit at the call site. Const-correctness on references also lets the compiler bind temporaries and communicates intent.
Q5. Explain the difference between signed and unsigned overflow in C++ and why it matters.
Signed integer overflow is undefined behavior — the compiler may assume it never happens and optimize accordingly, so it's not safe to rely on wraparound. Unsigned overflow is well-defined and wraps modulo 2^n. The practical trap is mixed signed/unsigned arithmetic and comparisons: expressions like `for (int i = 0; i < vec.size(); ++i)` compare a signed int to an unsigned size_t, and a subtraction like `a - b` on unsigned types where b > a yields a huge positive number instead of a negative. Prefer consistent types and fixed-width integers when range and behavior must be predictable.
Q6. When is a switch statement better than an if-else chain, and what's the fallthrough gotcha?
A switch over a dense set of integral/enum values can be compiled into a jump table giving effectively O(1) dispatch, whereas an if-else chain evaluates conditions sequentially (O(n)). switch is also clearer for exhaustive enum handling and lets the compiler warn on unhandled cases. The gotcha is that cases fall through to the next unless you break (or return), so a missing break silently executes subsequent cases — modern C++ offers [[fallthrough]] to mark deliberate fallthrough and silence warnings.
Q7. Give concrete examples of undefined behavior a developer can hit with just these basics.
Reading or writing past the end of an array (no bounds checking); signed integer overflow; using an uninitialized local variable's value; dividing by zero; returning a reference/pointer to a local that goes out of scope; and reading an object as the wrong type. UB means the program may crash, produce garbage, or appear to work — and optimizers can transform code assuming UB never occurs, so bugs can vanish in debug builds and manifest only under -O2. This is the core reason to prefer bounds-checked containers (.at()) and initialize variables at declaration.
Q8. Compare C-style arrays, std::array, and std::vector — when would you use each?
A C-array is a raw fixed-size block with no size tracking, decays to a pointer, and offers no bounds checking — avoid in modern code except for interop or tiny local buffers. std::array<T,N> is a zero-overhead stack-allocated fixed-size array that keeps its size in the type, supports .size()/.at(), and is copyable/assignable — use it when the size is a compile-time constant. std::vector<T> is a heap-backed dynamic array that can grow/shrink (amortized O(1) push_back) and owns its memory — the default choice when the size is runtime-determined, at the cost of a heap allocation and a level of indirection.
