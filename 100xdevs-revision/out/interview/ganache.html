Ganache, Trufle, Hardhat and Foundry

Cohort 3.0 · 12 lessons · SDE-2 (3+ yrs) interview prep

Concept summary — all 12 steps

Web3 Cohort framing
Sets the mental model for the whole track: Solidity source compiles to EVM bytecode, gets deployed on-chain, and is interacted with off-chain via JSON-RPC using the contract's ABI. Interview angle: be able to describe this pipeline (source -> bytecode + ABI -> deployed address -> RPC calls) because every tool below just automates one part of it.
Local ETH development
You never iterate against mainnet; you run a local EVM node (Ganache, Anvil, or Hardhat Network) that mines instantly, pre-funds deterministic accounts, and can fork live mainnet state at a block. Why it matters: fast, gas-free, reproducible tests, plus the ability to fork and impersonate accounts to integration-test against real protocols (Uniswap, Aave) without touching real funds.
Ganache + Truffle (legacy)
The original JS toolchain: Ganache is a local blockchain (GUI/CLI) with prefunded accounts and configurable block times; Truffle handled compilation, migrations, and Mocha/Chai tests. Interview angle: ConsenSys sunset both in 2023, so treat them as legacy you may inherit in old repos; the modern answer is Hardhat or Foundry.
Hardhat
A JS/TS development environment built around Hardhat Network, which adds Solidity console.log, readable revert stack traces, and mainnet forking. Strength: deep JS/npm ecosystem and plugins (ethers, typechain) that suit full-stack teams; weakness: tests run in JS so they are slower and force a context switch between JS test logic and Solidity contract logic.
Foundry
A Rust-based toolkit where you write tests in Solidity itself, giving very fast compile/run cycles plus first-class fuzzing and invariant testing. Four binaries: forge (build/test), cast (CLI for RPC and tx encoding), anvil (local node), and chisel (Solidity REPL). Interview angle: preferred today for speed and Solidity-native tests using cheatcodes via the vm object.
Installing Foundry
Install via the foundryup manager, which fetches/updates forge, cast, anvil, and chisel; foundryup lets you pin nightly or a specific version. `forge init` scaffolds a project with foundry.toml (compiler/config), src/, test/, and lib/ for dependencies.
Your first contract
Write Solidity in src/ with an SPDX license identifier and pragma, then `forge build` compiles it into out/ (bytecode + ABI). Tests live in test/ as contracts inheriting forge-std's Test, with functions prefixed test... that forge auto-discovers and runs.
Installing external packages
`forge install <org>/<repo>` vendors dependencies as git submodules under lib/ (pinned by commit), and remappings in foundry.toml/remappings.txt map import prefixes to those paths, e.g. `@openzeppelin/=lib/openzeppelin-contracts/`. Interview angle: contrast the git-submodule model with Hardhat's npm node_modules approach, and know that remappings are how the compiler resolves imports.
Creating an ERC-20
Inherit OpenZeppelin's audited ERC20, pass name/symbol to its constructor, and _mint an initial supply. Interview angle: know the ERC-20 surface (totalSupply, balanceOf, transfer, approve, allowance, transferFrom), the Transfer/Approval events, and the two-step approve + transferFrom pattern that lets contracts pull tokens on your behalf.
Testing emits
Use `vm.expectEmit(checkTopic1, checkTopic2, checkTopic3, checkData)` immediately before the call, emit the exact event you expect, then invoke the function; forge asserts the indexed topics and/or data byte-for-byte match. Why it matters: events are the contract's observable log output that off-chain indexers and frontends depend on, so testing them protects that implicit interface.
Std libraries (forge-std)
forge-std provides Test.sol (assertions like assertEq plus the Vm cheatcode interface), console/console2 for logging during runs, and helpers like StdCheats (deal, prank, hoax) and StdStorage. Cheatcodes such as vm.prank (spoof msg.sender), vm.warp (set timestamp), vm.expectRevert, and vm.deal (set balance) are what make Solidity-native testing expressive.
Deploying contracts
Use `forge create` for a single contract or, preferably, `forge script` with a Script contract wrapping the deploy in vm.startBroadcast()/stopBroadcast(), run with --rpc-url and --broadcast. Interview angle: never hardcode private keys - use an encrypted keystore or env var, keep scripts versioned in Solidity for reproducible deploys, and pass --verify to publish source to the block explorer.

Real-world example

Shipping and integration-testing a rewards token with Foundry

You are an SDE-2 building an ERC-20 rewards token for a dApp. You scaffold with `forge init`, `forge install OpenZeppelin/openzeppelin-contracts`, add a remapping, and write RewardToken by inheriting OpenZeppelin's ERC20 and minting an initial supply to the treasury. In test/ you write Solidity tests using forge-std: vm.prank to act as different users, assertEq on balances, vm.expectEmit to assert the Transfer event fires with the right from/to/amount, and vm.expectRevert to prove a transfer beyond balance fails. To be sure rewards can actually be swapped, you run against a mainnet fork via anvil (or --fork-url), impersonate a whale with vm.deal, and route a swap through the real Uniswap router to catch decimals/allowance bugs that a bare local chain would hide. Finally you write a Deploy.s.sol script that broadcasts the deployment, dry-run it against a testnet with an env-loaded key, then `forge script --broadcast --verify` to deploy and publish the source. If you had inherited an older repo, the same flow would have run through Ganache + Truffle migrations and JS/Mocha tests instead.

Interview Q&A (SDE-2)

Q1. Foundry vs Hardhat - when would you pick each?
Foundry writes tests in Solidity, compiles/runs far faster, and gives native fuzzing and invariant testing, which makes it strong for protocol/contract-heavy work and tight test loops. Hardhat is JS/TS-based with a rich npm plugin ecosystem (ethers, typechain) and Solidity console.log/stack traces, which suits full-stack teams whose deploy scripts, frontend, and tests all live in TypeScript. Many teams now use Foundry for contract testing and keep Hardhat or a JS layer for tooling/integration; the two can coexist.
Q2. Why develop against a local node, and what does mainnet forking add?
A local EVM node (Anvil/Hardhat Network/Ganache) mines instantly, pre-funds accounts, and costs no gas, giving deterministic, fast, repeatable tests. Forking spins up that local node seeded with real mainnet state at a chosen block, so you can integration-test against deployed protocols like Uniswap or Aave and impersonate any account. This catches real-world issues (token decimals, existing allowances, oracle values) that a blank local chain never exposes.
Q3. How does vm.expectEmit work and why test events at all?
You call vm.expectEmit with booleans for which indexed topics and the data to check, then emit the exact expected event, then make the real call; forge asserts the next emitted log matches on the fields you enabled. Events are the contract's canonical off-chain log output that indexers, subgraphs, and frontends consume, so they are effectively part of the public interface. Testing them prevents silently breaking downstream consumers even when core state logic is correct.
Q4. Explain the approve/transferFrom pattern and the allowance front-running risk.
transfer moves your own tokens, but to let a contract (e.g. a DEX) pull tokens you first approve it for an allowance, and it later calls transferFrom up to that limit. The classic risk: changing an allowance from N to M is two transactions, and a spender watching the mempool can spend the old N before the new M lands, then spend M too. Mitigations are setting allowance to 0 first, or using increaseAllowance/decreaseAllowance / EIP-2612 permit.
Q5. How do you deploy safely with forge script, and how are keys handled?
You write a Script contract that wraps the deployment between vm.startBroadcast() and vm.stopBroadcast(), then run it with --rpc-url and --broadcast so forge builds, signs, and sends the real transactions. Keys should come from an encrypted keystore (cast wallet) or an env var, never hardcoded in source, and you dry-run without --broadcast first. Keeping the script in Solidity makes deploys reproducible and reviewable, and --verify publishes source to the explorer.
Q6. How does Foundry manage dependencies and imports compared to Hardhat?
forge install vendors dependencies as git submodules under lib/, pinned to a specific commit, and remappings (in foundry.toml or remappings.txt) map an import prefix like @openzeppelin/ to that lib path so the compiler resolves imports. Hardhat instead uses npm/node_modules with semver ranges. The submodule model gives exact commit reproducibility without a package registry, at the cost of git submodule ergonomics.
Q7. What cheatcodes from forge-std do you use most and what do they do?
vm.prank/startPrank spoof msg.sender for the next (or a span of) calls; vm.deal sets an address's ETH balance; vm.warp and vm.roll set block.timestamp and block.number; vm.expectRevert asserts a call reverts with a given reason; vm.expectEmit asserts event emission. These let you simulate access control, time-based logic, and failure paths entirely in Solidity without deploying helper mocks.
Q8. What is fuzz and invariant testing in Foundry and why does it matter?
A fuzz test takes typed parameters and forge runs it hundreds of times with randomized inputs, surfacing edge cases (overflows, boundary amounts) that hand-picked cases miss. Invariant testing declares a property that must always hold (e.g. sum of balances equals totalSupply) and forge drives random sequences of calls trying to break it. Both push you past example-based testing toward property-based guarantees, which is critical for value-holding contracts where a single edge case can be an exploit.
