ETH on the frontend.

Cohort 3.0 · 10 lessons · SDE-2 (3+ yrs) interview prep

Concept summary — all 10 steps

Web3 cohort intro
Framing step for the Web3 module: the goal is to build a browser dApp frontend that reads chain state and lets users sign/send transactions. The mental model to lock in early is that your React app is just a client to Ethereum, not a custodian of funds or keys.
Context: how a frontend talks to a chain
A web app never holds private keys or runs a node; it communicates with Ethereum over JSON-RPC (via an RPC endpoint for reads) and delegates all signing to the user's wallet. Interview angle: separate the two channels clearly, reads go through an RPC transport (public client), writes get signed by the wallet (wallet client) and broadcast.
Wallets in ETH
A wallet (MetaMask, Rabby, Coinbase Wallet) manages the user's private keys and exposes an injected provider on window.ethereum following the EIP-1193 standard (request/on events). Know the difference between an EOA (externally owned account, key-controlled) and a smart-contract wallet, and that the frontend only ever asks the wallet to sign — it never sees the key.
Wagmi library
Wagmi is a React hooks library that turns wallet connection, account state, reads, and writes into declarative, reactive hooks (useAccount, useConnect, useBalance, useSendTransaction). It matters because it eliminates the boilerplate of manually wiring window.ethereum, managing connection state, and handling reconnection/chain-change events.
The stack: TanStack + Viem + Wagmi
Viem is the low-level, tree-shakeable, type-safe TS library that actually talks to the chain (replacing ethers/web3.js); TanStack Query provides async caching, deduplication, and refetching; Wagmi composes the two into React hooks. Interview angle: this is clean separation of concerns — viem = transport/encoding, TanStack = async state, wagmi = React glue.
Hookifying viem
Wagmi wraps viem's imperative actions (getBalance, readContract, sendTransaction) into hooks backed by TanStack Query, so each call gets isLoading/isError/data/refetch and automatic caching keyed by args. This replaces the error-prone pattern of calling viem inside a useEffect with manual loading flags and no cache invalidation.
Connecting to the wallet
You configure a WagmiProvider with chains, transports, and connectors, then use useConnect to trigger connection and useAccount to read address/isConnected/chainId reactively. Key details to discuss: connector selection, persisted reconnection on reload, and reacting to the user switching accounts or networks in the wallet.
Showing the user their balance
useBalance fetches the account's native (or token) balance, which comes back as a BigInt in wei; you format it for display with viem's formatEther/formatUnits using the correct decimals. Interview angle: never use floats for balances — arithmetic stays in BigInt and formatting is only a display concern to avoid precision loss.
Initiating a transaction
useSendTransaction (or useWriteContract) prompts the wallet to sign and broadcasts; you build value with parseEther and then wait for inclusion with useWaitForTransactionReceipt. Cover the full lifecycle: hash returned immediately vs. mined receipt, waiting for N confirmations, and handling user rejection (code 4001) and gas/revert errors distinctly.
Assignments / practice
Consolidate the flow end-to-end: connect, display balance, send ETH, and ideally extend to an ERC-20 transfer via useWriteContract with an ABI. The practical goal is fluency in the read-vs-write distinction, unit conversion, and transaction-state UX (pending/confirmed/failed).

Real-world example

A donation/tip widget on a creator dashboard

You are an SDE-2 adding a 'Tip in ETH' widget to a creator's public page. The user clicks Connect, wagmi's useConnect opens MetaMask and useAccount exposes their address and chainId. You render their balance with useBalance, formatting the wei BigInt via formatEther so it doesn't show 18 zeros. When they enter an amount and hit Send, useSendTransaction calls parseEther(amount) and asks the wallet to sign; you show a spinner keyed off useWaitForTransactionReceipt until it mines, then a success toast with the tx hash linking to Etherscan. Debugging you would actually hit: the send silently fails because the user is on Sepolia instead of mainnet (guard on chainId and prompt a switchChain), a rejection throws with code 4001 that you must catch and show as 'cancelled' rather than a scary error, and the balance appearing stale after a tip because you forgot that TanStack caches it — so you invalidate/refetch the balance query on receipt.

Interview Q&A (SDE-2)

Q1. Why use viem over ethers.js or web3.js?
Viem is designed for a modern TS toolchain: strong end-to-end type inference from ABIs, a tree-shakeable modular API that keeps bundles small, and clear separation of public client (reads over RPC) vs. wallet client (writes/signing). It also has predictable BigInt-based number handling and is the native substrate wagmi is built on, so you get consistent behavior between low-level calls and hooks. Ethers is still fine, but viem's typing and DX are the reason the ecosystem shifted.
Q2. How does wagmi leverage TanStack Query, and why does that matter?
Every wagmi read hook is a TanStack Query under the hood, with a query key derived from the call's arguments (address, chainId, args). That gives you caching, request deduplication across components, background refetching, and stale-while-revalidate for free, so ten components reading the same balance make one request. It also means you invalidate or refetch chain data through the query client — e.g., refreshing a balance after a transaction confirms — instead of manual state juggling.
Q3. How can a frontend send transactions without ever holding the private key?
The wallet injects an EIP-1193 provider (window.ethereum) that exposes a request interface. Reads go through your own RPC transport via a public client, but for writes the app builds an unsigned transaction and calls the wallet to sign it; the private key never leaves the wallet's process. The wallet returns a signed tx or broadcasts it, and the app only ever receives a hash — this is the core security boundary of a dApp.
Q4. Why are balances and amounts handled as BigInt, and how do you display them?
Ether has 18 decimals and token amounts routinely exceed JavaScript's safe integer range, so representing them as Number risks silent precision loss. Viem/wagmi keep all on-chain values as BigInt in the smallest unit (wei), and you only convert at the display boundary with formatUnits/formatEther, going the other way with parseEther/parseUnits for input. Rule of thumb: do all math in BigInt with the token's decimals, and treat formatting purely as a view concern.
Q5. Walk through the full lifecycle of sending a transaction with wagmi.
useSendTransaction (or useWriteContract for a contract call) prompts the wallet to sign and broadcasts, returning a transaction hash almost immediately — that does not mean it's mined. You then pass that hash to useWaitForTransactionReceipt to await inclusion in a block and get success/revert status, optionally waiting for extra confirmations for finality. UX-wise you distinguish three states — signing (can be rejected, error code 4001), pending (broadcast, not mined), and confirmed/failed — and only mutate app state or refetch balances on the receipt.
Q6. How do you handle the user being on the wrong network?
Read chainId from useAccount and compare it to the chain your contracts are deployed on; if it mismatches, disable actions and offer useSwitchChain to request a network switch (which prompts the wallet, and may prompt to add the chain if unknown). Never assume the connected chain — users can switch networks in the wallet at any time, and wagmi will reactively update chainId, so your guard should be reactive too. Sending to the wrong chain is a common cause of 'my transaction did nothing' bugs.
Q7. What's the difference between viem's public client and wallet client?
A public client is read-only and talks to an RPC endpoint you configure (transport) to fetch balances, call view functions, read receipts, and estimate gas — no signer required. A wallet client wraps the user's account/provider and handles anything requiring a signature: sending transactions, signing messages, writing to contracts. Wagmi maps this split onto hooks — read hooks use the public client, write hooks use the wallet client — so mixing them up (e.g., trying to write with only an RPC transport) is a frequent conceptual error.
Q8. Why not just call window.ethereum directly inside a useEffect?
You can, but you'd re-implement connector management, persisted reconnection, account- and chain-change event subscriptions, request deduplication, caching, and loading/error state by hand — and get it subtly wrong (stale closures, race conditions, no cache invalidation). Wagmi + TanStack give you all of that declaratively and reactively, plus viem's typed encoding/decoding. Direct provider access is reserved for edge cases the hooks don't cover, not the default path.
