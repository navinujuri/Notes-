HTTP Deep dive

Cohort 3.0 · 9 lessons · SDE-2 (3+ yrs) interview prep

Concept summary — all 9 steps

Recap: the HTTP request/response model
HTTP is a stateless, text-based, request/response protocol over TCP: a client sends a method (GET/POST/PUT/PATCH/DELETE) + path + headers + optional body, and the server replies with a status line, headers, and body. Interview angle: statelessness is why we need cookies/tokens for sessions, and knowing status code families (2xx success, 3xx redirect, 4xx client error, 5xx server error) plus idempotency (GET/PUT/DELETE idempotent, POST not) is table stakes.
Headers: metadata that drives behavior
Headers carry metadata separate from the body — Content-Type/Content-Length describe the payload, Authorization carries credentials, and Cache-Control/ETag/If-None-Match drive caching and conditional requests. Interview angle: be able to explain content negotiation (Accept), the difference between Authorization vs Cookie auth, and how caching validators (ETag + 304 Not Modified) save bandwidth.
Fetch API: making HTTP calls from the client
fetch() is the promise-based browser (and modern Node) HTTP client; it resolves to a Response object and does NOT reject on 4xx/5xx — you must check response.ok / response.status yourself. Interview angle: know that the body is a stream consumed once (await res.json()), that fetch only rejects on network failures/CORS, and how to attach headers, method, body, credentials, and an AbortController for timeouts/cancellation.
Create an HTTP server
A server binds to a port and handles incoming requests; Node's core http.createServer((req,res)=>…) is the primitive, while Express wraps it with routing and middleware ergonomics. Interview angle: understand that req is a readable stream (body arrives in chunks), res needs an explicit status + headers + end(), and that a single Node process handles concurrency via the event loop, not threads.
Middlewares: the request pipeline
Middleware are functions (req, res, next) that run in order for each request, forming a pipeline where each can read/mutate req/res, short-circuit by sending a response, or call next() to pass control (or next(err) to jump to error handlers). Interview angle: explain ordering matters, that forgetting next() hangs the request, and that this is the composition pattern behind auth, logging, body parsing, and error handling.
Route-specific middleware
Middleware can be scoped: app.use(mw) runs globally, app.use('/admin', mw) runs for a path prefix, and app.get('/x', mw, handler) runs only for that route/method. Interview angle: use this to apply auth or validation only where needed (e.g. protect /admin routes) instead of gating every request, and know that error-handling middleware has a distinct 4-arg signature (err, req, res, next).
Middleware assignments (hands-on)
Practice building real middleware: a request logger, an auth/JWT verifier that attaches req.user, a rate limiter using an in-memory or Redis counter, and input validation. Interview angle: the takeaway is reusable cross-cutting concerns — interviewers probe whether you can write one correctly, including calling next() on the happy path and responding with the right status (401 vs 403, 429 for rate limits) on rejection.
Commonly used middleware
Production apps compose well-known middleware: express.json()/urlencoded() to parse bodies, cors for cross-origin access, helmet for security headers, morgan for request logging, cookie-parser, and compression. Interview angle: know why express.json() is needed (raw body is a stream, req.body is undefined without it) and that order matters — body parsers before handlers, error handler last.
CORS: Cross-Origin Resource Sharing
CORS is a browser-enforced mechanism where the server opts in to cross-origin requests via Access-Control-Allow-Origin/Methods/Headers; the browser sends a preflight OPTIONS request for non-simple requests and blocks the response if the headers don't permit it. Interview angle: CORS is enforced by the browser, not the server (curl/Postman ignore it), Allow-Origin '*' cannot be combined with credentials, and preflight caching via Access-Control-Max-Age reduces overhead.

Real-world example

Building and debugging a React SPA + Node/Express API split across two origins

You build a Node/Express JSON API on api.myapp.com and a React SPA served from app.myapp.com. The SPA uses fetch() with credentials:'include' to hit the API, sending a JWT in an Authorization header. On the server you compose a middleware pipeline: express.json() to parse bodies, morgan for logging, a global auth middleware that verifies the token and sets req.user, and route-specific middleware that gates /admin behind a role check — plus a final (err,req,res,next) error handler that maps failures to 401/403/500. The first thing that breaks in the browser is a CORS error: the preflight OPTIONS request fails because the server never sent Access-Control-Allow-Origin/Allow-Headers, so you add the cors middleware configured with the specific origin (not '*', since you use credentials) and allow the Authorization header. Then you notice fetch isn't throwing on a 401, so you fix the client to check res.ok before calling res.json(). Finally you add ETag/Cache-Control on a GET endpoint so repeat requests return 304 and save bandwidth. This single flow exercises headers, fetch semantics, the server, the middleware chain, route-scoped auth, common middleware, and CORS end to end.

Interview Q&A (SDE-2)

Q1. Why doesn't fetch() reject the promise on a 404 or 500, and how do you handle it correctly?
fetch only rejects on network-level failures (DNS, connection refused, CORS block, aborted request); any completed HTTP response — including 4xx/5xx — is considered a success at the transport level, so the promise resolves. You must inspect response.ok (true for 200–299) or response.status and throw/branch yourself. This trips people up because axios throws on 4xx/5xx by default, so migrating code between them changes error-handling flow.
Q2. Walk through what happens in an Express middleware chain when a request comes in, and what breaks if you forget next().
Middleware run in registration order; each gets (req, res, next) and can mutate req/res, end the response, or call next() to pass control to the next matching middleware/handler. If you neither send a response nor call next(), the request hangs until the client times out — a common bug. Calling next(err) skips remaining normal middleware and jumps to the error-handling middleware (the 4-arg signature), which should be registered last.
Q3. Explain CORS and the preflight request. Why can a request work in Postman but fail in the browser?
CORS is a browser security mechanism that restricts cross-origin requests unless the server explicitly opts in via Access-Control-Allow-* response headers. For non-simple requests (custom headers, methods like PUT/DELETE, or certain content types) the browser first sends a preflight OPTIONS request and only proceeds if the response permits the origin, method, and headers. Postman/curl aren't browsers and don't enforce the same-origin policy, so they bypass CORS entirely — meaning a CORS failure is a client-browser enforcement issue, not something the server 'blocks.'
Q4. Why can't you use Access-Control-Allow-Origin: * with credentialed requests, and what's the fix?
When a request includes credentials (cookies or Authorization via credentials:'include'), the browser forbids the wildcard '*' for Allow-Origin and also requires Access-Control-Allow-Credentials: true. You must echo back the specific requesting origin instead of '*'. Practically this means reflecting an allowlisted origin dynamically, which is exactly what the cors middleware does when you pass it a concrete origin and credentials:true.
Q5. What does express.json() actually do, and what happens if you omit it?
The raw HTTP request body arrives as a readable stream of chunks; Express does not buffer or parse it by default, so req.body is undefined. express.json() is middleware that reads the stream, parses it as JSON based on Content-Type, and populates req.body. If you omit it (or the client omits Content-Type: application/json), req.body stays undefined and you get 'cannot read property of undefined' errors in handlers. It also must be registered before the routes that need it.
Q6. How do you implement a request timeout / cancellation with fetch?
fetch has no built-in timeout, so you use an AbortController: pass controller.signal in the fetch options and call controller.abort() after a setTimeout; the fetch promise then rejects with an AbortError. Modern runtimes also offer AbortSignal.timeout(ms) as shorthand. This is important for preventing hung requests and for cancelling stale in-flight requests (e.g. type-ahead search) to avoid race conditions where an older response overwrites a newer one.
Q7. When would you use route-specific middleware versus global (app.use) middleware?
Use global middleware for concerns every request needs — body parsing, logging, security headers, CORS. Use route- or path-scoped middleware for concerns that apply to a subset, like requiring authentication only on /api or a role check only on /admin, applied via app.use('/admin', authz) or per-route app.get('/x', authz, handler). Scoping avoids running expensive or restrictive logic on public routes (like a healthcheck or login endpoint) and keeps the pipeline efficient and correct.
Q8. How do ETag / If-None-Match enable caching, and what status code signals a cache hit?
The server sends an ETag header (a hash/version of the resource) with a response; on the next request the client sends If-None-Match with that ETag. If the resource is unchanged, the server responds 304 Not Modified with no body, letting the client reuse its cached copy and saving bandwidth. This is a validation/conditional-request model (freshness confirmed with the server), distinct from Cache-Control max-age which lets the client skip the server entirely until the TTL expires.
