Building an Exchange - Part 1

Cohort 2.0 · 9 lessons · SDE-2 (3+ yrs) interview prep

Concept summary — all 9 steps

What is an exchange
A marketplace that matches buyers and sellers of an asset and records the resulting trades; at its core it is a matching engine that maintains order books and enforces fair ordering (price-time priority). Interview angle: an exchange is not a CRUD app — the defining component is a deterministic in-memory matching engine, and everything else (API, DB, websockets) exists to feed and observe it.
Core concepts
The order book (sorted bids and asks), the best bid/ask, the spread, market depth, and the base/quote asset pair (e.g. SOL/USDC where SOL is base, USDC is quote). These are the shared vocabulary; knowing that a bid is a buy at-or-below a price and an ask is a sell at-or-above lets you reason about how a new order will match against the opposite side.
Types of orders
Market (execute now at best available price, no price guarantee), Limit (execute only at your price or better, no fill guarantee), plus modifiers/variants like Stop-Loss/Stop-Limit, IOC (immediate-or-cancel), FOK (fill-or-kill), and GTC (good-till-cancelled). The interview point is the trade-off axis: execution certainty vs price certainty — market orders take liquidity and risk slippage, limit orders provide liquidity but may never fill.
Jargon
Maker vs taker (adding liquidity to the book vs removing it, usually with asymmetric fees), liquidity, slippage, spread, ticker, volume, and partial fills. These terms recur in fee models and matching logic; e.g. a market order is always a taker, and knowing maker/taker economics explains why exchanges incentivize resting limit orders.
How orders are made
Order lifecycle: client submits -> API validates and locks funds -> order enters the matching engine -> engine walks the opposite side of the book by price-time priority producing full/partial fills -> unmatched remainder rests on the book -> balances settle and fills/book updates are broadcast. The key insight is fund locking at placement time to prevent double-spend, and that matching is a deterministic sequential pass over sorted price levels.
Backend Routes
REST endpoints for the request/response operations: place order, cancel order, get open orders, get order-book depth, get balances, and get trade/ticker history. These are stateless HTTP calls that mutate or read engine/DB state; place/cancel are writes routed to the engine (often via a queue), while depth/balances are reads that can be served from cache or the engine snapshot.
Websocket streams
Real-time push channels for data that changes continuously — order-book depth diffs, trade prints, ticker, and per-user fill notifications — because REST polling can't deliver low-latency market updates at scale. Interview angle: clients subscribe to channels (e.g. depth@SOL_USDC), the server pushes deltas rather than full snapshots, and you typically bootstrap with a REST snapshot then apply websocket diffs by sequence number to avoid gaps.
Architecture
Separation of concerns: stateless API servers accept orders and push them onto a queue (e.g. Redis); a single-threaded matching engine consumes the queue, mutates in-memory order books deterministically, and publishes results to a pub/sub channel; a websocket server fans those out to clients, while a persistence layer (DB + snapshots/event log) records fills and enables recovery. This decouples request ingress from matching, lets you scale API/websocket tiers horizontally, and keeps the engine authoritative and replayable.
Coding the API Server
Build the HTTP layer (Express/Node) that validates payloads, authenticates the user, serializes the order, and pushes it onto the Redis queue keyed for the engine, then awaits the engine's response (via a response queue or pub/sub) to return synchronously to the client. The practical lessons are request validation, idempotency, and treating the API server as a thin, stateless adapter that never mutates order-book state directly.

Real-world example

Placing a limit buy on SOL/USDC and watching it fill

You're an SDE-2 building a crypto exchange. A user submits a limit buy for 10 SOL at 150 USDC. The API server authenticates them, validates the payload, and locks 1,500 USDC of their balance to prevent double-spend, then pushes the order onto a Redis queue instead of touching any order book directly. The single-threaded matching engine pops the order, walks the ask side of the SOL/USDC book by price-time priority, and finds 4 SOL resting at 149 and 6 SOL at 150 — it produces two fills, fully matching the order, updates both users' locked/available balances, and emits the results to a pub/sub channel. The websocket server picks up those messages and pushes a depth diff and trade prints to every subscriber, plus a private fill notification to the buyer, while the persistence layer writes the trades to the DB. When you later debug a report that 'the order book on the frontend drifts from the engine,' the root cause is a client that applied websocket depth diffs without first anchoring to a REST snapshot by sequence number — so it missed an update and its book desynced.

Interview Q&A (SDE-2)

Q1. Why is the matching engine typically single-threaded and in-memory rather than a multi-threaded service backed by a database transaction per order?
Matching must be deterministic and enforce strict price-time priority; a single thread over in-memory sorted books gives you sequential, lock-free, reproducible ordering with microsecond-level operations that a per-order DB transaction can't match. Concurrency would require locking each price level and risks reordering, and RDBMS round-trips add milliseconds. You get durability separately via an append-only event log plus periodic snapshots, so the engine can be rebuilt by replaying events — you trade horizontal scalability of the engine for correctness and speed, then scale by sharding per trading pair.
Q2. Explain price-time priority and how a new limit order matches against the book.
Orders are ranked first by best price (highest bid / lowest ask) and, within the same price level, by arrival time (FIFO). A new buy limit order matches against the lowest asks first, consuming resting quantity in time order until either the order is filled or the next ask price exceeds the order's limit price; any remainder rests on the bid side at its limit price. This guarantees the best-priced and earliest orders execute first, which is the fairness property exchanges are legally and reputationally expected to uphold.
Q3. What's the difference between a maker and a taker, and why do exchanges charge them differently?
A maker posts a resting order that adds liquidity to the book (a limit order that doesn't immediately match); a taker submits an order that removes liquidity by matching against resting orders (any market order, or a marketable limit). Exchanges usually charge takers a higher fee and makers a lower or negative (rebate) fee to incentivize deep, tight books, because liquidity is what attracts traders. This directly shapes order-type economics: a market order is always a taker and pays for immediacy plus slippage.
Q4. How do you keep the API server and matching engine consistent given they're decoupled by a queue?
Treat the engine as the single source of truth and the API server as a stateless adapter: the API locks funds and enqueues an order with a client-supplied idempotency key, then correlates the engine's async response (via a response queue or pub/sub keyed by request id) before replying. Idempotency keys make retries safe against duplicate submission, and because the engine processes the queue sequentially, ordering is well-defined. Balances shown to users should reflect engine-confirmed state, not the API's optimistic view, to avoid drift.
Q5. Why use websockets for market data instead of REST polling, and how do clients stay in sync?
Order books and trades change continuously; polling forces a latency-vs-load trade-off (poll fast and you hammer the server, poll slow and data is stale), while websockets let the server push deltas the instant they happen. Clients bootstrap correctly by fetching a REST depth snapshot with a sequence number, then applying websocket diffs whose sequence continues from it, discarding older diffs and reconnecting/re-snapshotting if they detect a gap. This snapshot-plus-diff pattern is what prevents the desynced-book bug that plagues naive clients.
Q6. How do you prevent a user from spending the same balance twice across concurrent orders?
Lock funds at order-placement time, not at fill time: when a buy limit for 10 SOL at 150 is accepted, move 1,500 USDC from available to locked so it can't back another order. Since the matching engine is the authoritative single-threaded owner of balances (or balance checks happen atomically before enqueue), there's no race — the second order sees reduced available funds and is rejected if insufficient. On cancel or partial fill, the unused locked amount is released back to available.
Q7. How do you make the in-memory engine durable and recoverable after a crash?
Use event sourcing: append every accepted order/cancel/fill to a durable, ordered log before or as it's applied, and periodically write a full snapshot of the order books and balances. On restart you load the latest snapshot and replay the log from that point to reconstruct exact state, because the engine is deterministic. This gives you crash recovery, an audit trail for compliance, and the ability to spin up a hot standby that consumes the same event stream.
Q8. How would you scale this architecture as trading volume grows?
Scale the stateless tiers horizontally — multiple API servers behind a load balancer and multiple websocket servers subscribed to pub/sub — since they hold no authoritative state. The engine is the bottleneck, so you shard by trading pair (one engine instance owns SOL/USDC, another owns BTC/USDC), which parallelizes across markets while keeping each book single-threaded and deterministic. Redis (or Kafka for higher throughput/durability) handles order ingress and result fan-out, and reads like depth/ticker are served from caches or snapshots to keep load off the engines.
