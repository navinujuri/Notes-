Paytm Project

Cohort 2.0 · 20 lessons · SDE-2 (3+ yrs) interview prep

Concept summary — all 20 steps

How to approach building any project
A meta-lesson: don't jump to code. Move top-down — features -> UX flows -> architecture -> stack -> implementation — so ambiguous requirements get scoped before you commit to a design. Interview angle: this is exactly the structured decomposition an SDE-2 is expected to show in a system-design or take-home round.
Feature planning / scoping the MVP
Define the concrete feature set: P2P transfers, on-ramp (add money from a bank), merchant receive-payments, and transaction history, then cut to an MVP. The signal being tested is prioritization and saying no — shipping wallet balance + on-ramp + P2P before KYC, refunds, or notifications.
End-user UX flows
Map the consumer screens and flows: view balance, add money, send to another user, see transactions. Doing this first surfaces the data you must persist (balances, transactions) and the states you must render (processing/success/failure), which de-risks the schema and API design later.
Merchant UX flows
Design the merchant-side surface separately: receiving payments, revenue/settlement view, and a different auth identity. The point is that merchant and consumer are distinct personas with different data and permissions — a classic reason to split into two apps rather than role-flagging one.
System architecture
A Turborepo monorepo with two Next.js apps (user-app, merchant-app), shared packages (db/Prisma, ui, store), and a separate bank-webhook server for on-ramp callbacks. Interview angle: justify the split — shared domain code with independently deployable frontends, and an external webhook boundary you don't control the timing of.
Tech stack selection
Next.js + TypeScript (App Router), Prisma + PostgreSQL, NextAuth, Recoil, Tailwind, Turborepo, and an Express webhook service. Be able to defend each choice: Postgres for ACID money operations, Prisma for typed schema/migrations, NextAuth to avoid rolling your own session layer.
Bootstrap the monorepo
Scaffold the Turborepo, create the app and package workspaces, and wire the shared tsconfig/eslint/build pipeline. This is where you set up workspace dependencies (apps import @repo/db, @repo/ui, @repo/store) so code is shared, not copy-pasted.
Add Prisma as a shared db package
Model User, Merchant, Balance, OnRampTransaction, and p2pTransfer; run migrations; export a singleton PrismaClient. Two interview-critical details: store money as integers (paise), never floats, and reuse one client instance to avoid exhausting the connection pool (especially in serverless).
Create a shared Recoil store package
Put atoms/selectors (e.g. balance state) in a @repo/store package so both apps share client state logic. The reason for a separate package rather than per-app state is a single source of truth for derived values like locked vs. unlocked balance.
Wire Recoil into the Next.js apps
Wrap each app in a RecoilRoot. The App Router gotcha: RecoilRoot uses React context so it must live in a 'use client' provider component imported into the server layout — you cannot make the root layout itself a client component.
Install and configure NextAuth
Add the [...nextauth] route handler and a SessionProvider, establishing the auth plumbing. The point of this step is infrastructure: a single place that issues/validates sessions so every protected route and server action can call getServerSession.
Implement the auth logic
Configure a Credentials provider (phone + password) with bcrypt-hashed passwords, JWT session strategy, and a callback that injects the user id into the token/session. Interview angle: know why you add userId in the jwt/session callbacks — without it the session has an email but no stable key to scope DB queries.
Add an Appbar component
A first shared UI component rendering the signed-in user and sign-in/out actions. It's the vehicle for wiring auth state into the UI and proving the shared @repo/ui package is consumable from both apps.
Checkpoint
A consolidation pause to verify auth, Prisma, and the monorepo wiring all work end-to-end before building features on top. Practically it's a smoke test — sign up, sign in, session persists, DB reads work — so later bugs aren't ambiguous.
On-ramping (add money from bank)
The core money flow: create an OnRampTransaction in Processing state with a token, then a bank webhook hits your Express server to confirm, which credits the balance. This must be atomic and idempotent — update balance and mark the txn Success inside one DB transaction with a row lock, and reject duplicate webhook deliveries. This is the highest-signal step in the whole project.
Refactor into a generic Appbar
Extract the appbar into a presentational component in @repo/ui that takes props (user, onSignin, onSignout) instead of reaching into auth directly. The lesson is dependency inversion: shared UI stays framework/auth-agnostic and each app injects behavior.
Create the dashboard sidebar
Build the navigation sidebar (Home, Transfer, Transactions, P2P) with active-route highlighting. Minor but it establishes the authenticated app-shell layout and client-side navigation pattern.
Create the transfer / add-money page
The page where a user picks a bank and amount and kicks off the on-ramp: it inserts the OnRampTransaction and redirects to the (simulated) bank. This is the client entry point to the webhook flow built earlier and where you validate input server-side.
Add seed data
A Prisma seed script that creates test users, balances, and merchants so the app is demoable and testable without manual setup. Interview angle: seeds make local dev and CI deterministic and give integration tests a known fixture.
Landing page redirect based on auth
Make / redirect to /dashboard when authenticated (and to sign-in otherwise) using a server-side redirect in the layout/page. Doing it server-side via getServerSession avoids a flash of the wrong page that a client-side check would cause.

Real-world example

Debugging a double-credited wallet during on-ramp

You built the add-money flow: the user-app creates an OnRampTransaction (status Processing, unique token), the user is sent to a mock bank, and the bank calls your Express webhook to confirm payment. In production you notice some wallets get credited twice. The cause is the bank retrying the webhook (network timeout) plus your handler reading the balance, adding the amount, and writing it back in separate queries — a classic read-modify-write race. The fix is twofold: make the credit atomic and idempotent. Wrap it in prisma.$transaction, use an atomic increment (balance.update with { amount: { increment } }) or SELECT ... FOR UPDATE so concurrent handlers serialize, and guard on the transaction token — only credit if the OnRampTransaction is still Processing, flipping it to Success inside the same transaction so a retried delivery finds it already Success and no-ops. You'd also verify the webhook signature so an attacker can't forge credits, and confirm amounts are stored as integer paise so no rounding drift creeps in. This single flow exercises the whole project: Prisma schema, ACID transactions, idempotency, webhook security, and the app/webhook-server architectural split.

Interview Q&A (SDE-2)

Q1. Why split this into a Turborepo monorepo with separate user-app and merchant-app instead of one app with roles?
Consumers and merchants are distinct personas with different UX, auth identities, and permission surfaces, so separating them keeps each app simple and independently deployable. The monorepo lets them still share domain code — the Prisma db package, UI components, and the Recoil store — through workspace dependencies, avoiding copy-paste drift. Turborepo adds cached, parallel builds across the workspaces. The trade-off is more upfront tooling setup versus a single app.
Q2. How do you make the on-ramp balance update correct under concurrency and webhook retries?
Do it in a single DB transaction and make it idempotent. Use an atomic increment (or SELECT ... FOR UPDATE) so two concurrent handlers can't do a stale read-modify-write, and gate the credit on the OnRampTransaction still being in Processing, flipping it to Success in the same transaction. A retried delivery then finds the row already Success and safely no-ops. Postgres's ACID guarantees are exactly why you don't do this in application memory.
Q3. Why Recoil here, and what's the App Router pitfall when adding it?
Recoil gives fine-grained atom/selector state (e.g. balance and derived locked balance) shared via a package across apps, with less boilerplate than Redux for this scale. The pitfall: RecoilRoot relies on React context and hooks, so it must be a client component — you create a 'use client' provider wrapper and mount it inside the server root layout rather than marking the whole layout as client, which would opt your entire tree out of server rendering.
Q4. NextAuth with a Credentials provider — JWT sessions vs database sessions, and how do you get the user id into the session?
With the Credentials provider you use the JWT strategy (database sessions aren't supported for credentials without extra work), so the session lives in a signed cookie with no per-request DB hit — at the cost of not being able to instantly revoke a token server-side. NextAuth by default exposes name/email but no stable id, so you add it in the callbacks: the jwt callback copies user.id onto the token, and the session callback copies it onto session.user.id, giving you a key to scope all DB queries.
Q5. Why store money as integers, and how would a bug here manifest?
Floating-point can't represent many decimal fractions exactly, so repeated additions accumulate rounding error — balances drift by fractions of a rupee and reconciliation fails. You store amounts as integer paise (or cents) and format for display only. The bug is insidious because small transfers look fine and it only surfaces at scale or during audits, which is why it's a common interview probe.
Q6. How do you implement a P2P transfer so it can't lose or duplicate money?
Run the debit of the sender and credit of the receiver inside one prisma.$transaction so both succeed or both roll back — never a debit without a matching credit. Lock the sender's balance row (SELECT ... FOR UPDATE / atomic decrement) and assert sufficient funds inside the transaction, not before it. To avoid deadlocks when two users transfer to each other simultaneously, acquire row locks in a consistent order (e.g. by ascending user id).
Q7. How do you secure the bank webhook endpoint?
Treat it as untrusted input on a public URL. Verify an HMAC signature (or shared secret) the bank sends so nobody can forge a credit, enforce idempotency via the unique transaction token so retries don't double-credit, and validate that the amount and token match a real Processing transaction. Optionally restrict by source IP. The threat model is that anyone who can hit the URL could otherwise mint money.
Q8. Why redirect the landing page server-side, and how?
You call getServerSession in the server component/layout and issue a redirect() to /dashboard (or to sign-in) before anything renders. Doing it client-side would render the landing page first, then flash and bounce after the auth check hydrates — bad UX and a brief exposure of the wrong screen. Server-side the user only ever sees the correct destination, and it avoids shipping the redirect logic to the client.
