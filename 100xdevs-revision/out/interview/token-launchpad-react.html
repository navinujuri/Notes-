Token Launchpad in React

Cohort 3.0 · 10 lessons · SDE-2 (3+ yrs) interview prep

Concept summary — all 10 steps

Context: what a launchpad is
A token launchpad is a client-only React app that lets a user create their own fungible SPL token on Solana without deploying any on-chain program. The interview angle is the Solana mental model: unlike Ethereum where each ERC-20 is its own deployed contract, on Solana you create a Mint account owned by the single pre-deployed Token (or Token-2022) program, so 'making a token' is really just creating and initializing an account.
Transactions vs instructions
An instruction is the atomic unit of work (program id + ordered accounts + opaque data byte array); a transaction is an ordered list of instructions signed by all required signers and executed atomically all-or-nothing. This matters because creating a token requires several steps (create account, init mint, init metadata) that must land together in one transaction so the mint is never left half-initialized; also worth knowing the ~1232-byte size cap and the single recent blockhash used for expiry/dedup.
Initializing the project (Vite + React)
Scaffold a React app, in this era with Vite rather than the deprecated Create React App. The relevant detail for later: Vite serves native browser ESM and, unlike CRA's webpack 4, does not auto-polyfill Node core modules, which is exactly why step 7 is needed.
Add the Solana wallet adapter
Wrap the app in ConnectionProvider (RPC endpoint), WalletProvider (supported wallets like Phantom/Backpack), and WalletModalProvider (connect UI), then consume useWallet()/useConnection() for publicKey and sendTransaction. The key point is security separation: the app never sees a private key — it builds unsigned transactions and delegates signing to the wallet extension.
Installing dependencies
Add @solana/web3.js (Connection, Transaction, Keypair, SystemProgram), @solana/spl-token (mint helper instructions and program constants), and the wallet-adapter packages. Interview-relevant nuance: spl-token ships both high-level helpers (createMint) and low-level instruction builders (createInitializeMintInstruction) — the launchpad needs the low-level ones.
Inspecting the createMint call
The convenient spl-token createMint() helper takes a payer Keypair and internally calls sendAndConfirmTransaction, so it only works server-side / in a script. In a browser you have a wallet that signs but never exposes its secret key, so you must decompose createMint into its underlying instructions and send them via wallet.sendTransaction — this is the central 'why the tutorial is built this way' moment.
Adding Node polyfill
web3.js and spl-token depend on Node globals (Buffer, process, crypto, stream) that don't exist in the browser; under Vite this surfaces as a runtime 'Buffer is not defined'. Fix it with vite-plugin-node-polyfills (or manually alias the buffer package and define global). The interview point is understanding bundler differences: CRA/webpack 4 auto-polyfilled these, Vite and webpack 5 do not.
Creating the mint function
Manually build the transaction: generate a fresh mint Keypair, compute rent via getMinimumBalanceForRentExemption(getMintLen(...)), SystemProgram.createAccount for the mint address, then createInitializeMintInstruction with decimals, mint authority and freeze authority. Crucially the transaction needs two signers — the wallet (fee payer) and the new mint Keypair — so you partialSign with the mint keypair before wallet.sendTransaction, because SystemProgram.createAccount requires the new account to sign for itself.
Attaching metadata (Token-2022)
With Token-2022 you store metadata inline on the mint using the MetadataPointer extension (pointing the mint at itself) plus the TokenMetadata extension carrying name/symbol/uri, rather than a separate Metaplex PDA. Ordering and sizing are exam-worthy: size the account with getMintLen([ExtensionType.MetadataPointer]) + pack(metadata) length + TYPE_SIZE/LENGTH_SIZE, initialize the pointer before initializeMint, and initialize the metadata after initializeMint.
Actually minting the tokens
Creating the mint yields zero supply; to mint you derive the user's Associated Token Account (getAssociatedTokenAddressSync), add createAssociatedTokenAccountInstruction, then createMintToInstruction for amount * 10^decimals signed by the mint authority. The distinction to articulate: the mint account defines the token, the ATA holds a specific owner's balance, and base-unit math (decimals) is applied at the boundary between UI and chain.

Real-world example

Shipping a self-serve memecoin/creator-token launchpad for a client

You're an SDE-2 building a self-serve page where a non-technical creator connects Phantom, types a name, symbol, image URI, decimals and initial supply, clicks 'Launch', and gets a live Token-2022 token on devnet/mainnet — no backend key custody. You wire up the wallet adapter providers, then discover the app crashes with 'Buffer is not defined' the moment you import spl-token, so you add vite-plugin-node-polyfills. Next, your first attempt reuses createMint() and fails because there's no payer Keypair in the browser, forcing you to hand-assemble the transaction: SystemProgram.createAccount for a new mint keypair, initialize the MetadataPointer extension, initializeMint, initialize the on-chain TokenMetadata, then create the creator's ATA and mintTo the full supply. The bug you spend the most time on is a 'signature verification failed' error — you forgot to partialSign with the mint keypair before calling sendTransaction — and a rent error because you sized the account with MINT_SIZE instead of getMintLen plus the packed metadata length. This scenario exercises every step: provider setup, polyfills, instruction decomposition, multi-signer transactions, extension sizing, and the mint-vs-ATA split.

Interview Q&A (SDE-2)

Q1. Why can't you use spl-token's createMint() helper directly in a browser token launchpad?
createMint() expects a payer Keypair and a mint-authority Keypair and internally calls sendAndConfirmTransaction, meaning it must hold and sign with raw secret keys. In a browser the wallet adapter only signs transactions and never exposes the private key, so there's no Keypair to pass. The fix is to decompose it into SystemProgram.createAccount + createInitializeMintInstruction and submit via wallet.sendTransaction, letting the wallet supply the fee-payer signature.
Q2. Explain the difference between an instruction and a transaction, and why atomicity matters here.
An instruction is a single program invocation carrying a program id, the list of accounts it touches, and a data payload. A transaction bundles one or more instructions, is signed by every required signer, and executes atomically — if any instruction fails, the entire transaction reverts. That atomicity is why create-account, initialize-mint and initialize-metadata are packed into one transaction: the mint can never end up created-but-uninitialized. Transactions are also capped near 1232 bytes and carry one recent blockhash for expiry and replay protection.
Q3. Why do you need a Node polyfill in a Vite React app, and what changed from CRA?
@solana/web3.js and @solana/spl-token rely on Node built-ins like Buffer, process, crypto and stream. Create React App used webpack 4, which shimmed these automatically, whereas Vite (and webpack 5) intentionally do not, so you hit runtime errors such as 'Buffer is not defined'. You resolve it with vite-plugin-node-polyfills, or by aliasing the buffer package and defining a global shim in the Vite config.
Q4. What is the difference between a mint account and an associated token account (ATA)?
A mint account holds token-wide state: total supply, decimals, mint authority and freeze authority — it defines the token. A token account holds one owner's balance of one specific mint, and the ATA is the canonical, deterministically derived token account for a given (owner, mint, token program) so each owner has exactly one per mint. Creating the mint produces zero supply; you must create an ATA and mintTo it to give someone a balance.
Q5. Which signers does the token-creation transaction require, and why the mint keypair?
It needs two: the connected wallet as fee payer/mint authority, and the newly generated mint Keypair. The mint keypair must sign because SystemProgram.createAccount is creating a brand-new account at that public key, and Solana requires a new account to authorize its own creation. In practice you partialSign the transaction with the mint keypair, then call wallet.sendTransaction so the wallet adds the fee-payer signature.
Q6. How does Token-2022 metadata differ from the older Metaplex approach?
Token-2022 stores metadata inline on the mint via extensions: a MetadataPointer extension (typically pointing the mint at itself) plus a TokenMetadata extension holding name, symbol and URI — no separate Metaplex metadata PDA or program is required. You must pre-size the account for those extensions and respect ordering: initialize the metadata pointer before initializeMint and initialize the metadata after. The trade-off is that Token-2022 is newer with historically thinner wallet/explorer support, while Metaplex metadata is more universally recognized.
Q7. Why must you compute rent with getMintLen and getMinimumBalanceForRentExemption instead of a fixed size?
Solana accounts must be rent-exempt, holding a minimum lamport balance proportional to their byte size, or they get garbage-collected. A plain mint is MINT_SIZE bytes, but enabling extensions makes the account larger, so you compute getMintLen([ExtensionType.MetadataPointer]) and add the packed metadata length plus the type/length headers, then pass that to getMinimumBalanceForRentExemption. Using the fixed MINT_SIZE underfunds the account and the transaction fails or the account can't hold its extension data.
Q8. How are decimals handled when minting a supply, and who must sign?
On-chain amounts are unsigned integers denominated in the token's smallest unit, so a token with decimals=9 and a desired display supply of 100 requires minting 100 * 10^9 base units; the UI applies this scaling at the read/write boundary. The mintTo instruction must be signed by the mint authority set during initializeMint — in the launchpad that's the connected wallet — and it deposits the base-unit amount into the target ATA.
