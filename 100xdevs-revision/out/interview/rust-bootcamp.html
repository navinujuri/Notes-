Rust Bootcamp

Cohort 2.0 · 23 lessons · SDE-2 (3+ yrs) interview prep

Concept summary — all 23 steps

Preface
Orientation for the bootcamp: sets expectations that Rust's learning curve is front-loaded (the borrow checker) but pays off in runtime safety and performance. Interview framing: articulate Rust's core promise — memory safety without a garbage collector, enforced at compile time.
Syllabus
Roadmap from tooling and syntax basics to the ownership model, then type modeling (structs/enums), error handling, collections, and advanced topics. The mental model to carry through: everything downstream is a consequence of the ownership/borrowing rules.
Why Rust? Isn't Node.js enough?
Rust targets CPU-bound, latency-sensitive, and systems work where Node's single-threaded event loop and GC pauses fall short — think proxies, parsers, WASM, embedded, and high-throughput services. Interview angle: Node wins on iteration speed and I/O-bound glue; Rust wins when you need predictable memory, real multithreading, and zero-cost abstractions. It's not either/or — Rust often ships as native addons (napi-rs) or CLIs alongside Node.
Initializing Rust locally
Install via rustup (the toolchain manager), which manages rustc, cargo, and toolchains (stable/beta/nightly) and targets. Know that rustup lets you pin per-project toolchains and cross-compile targets, unlike a single global compiler.
Initializing a Rust project
`cargo new` scaffolds a project with Cargo.toml (manifest: metadata, dependencies, edition) and src/main.rs (binary) or src/lib.rs (library). Cargo is build system + package manager + test runner in one — the reason Rust's project ergonomics feel cohesive versus the fragmented npm/webpack/jest chain.
VSCode setup + Hello world
rust-analyzer is the LSP that provides inline types, borrow-checker diagnostics, and refactors — non-negotiable for productivity. `fn main()` is the entrypoint and `println!` is a macro (the `!` matters: macros are compile-time code generation, not functions, which is how format strings get type-checked).
Simple variables
`let` bindings are immutable by default; types are inferred but statically checked. Shadowing (re-declaring `let x`) lets you reuse a name with a new type/value without mutation. Scalar types are fixed-width (i32/u64/f64/bool/char) — no implicit numeric coercion, so casts are explicit with `as`.
Conditionals & loops
`if`, `match`, and `loop` are expressions that return values (`let y = if cond { 1 } else { 2 }`). Three loop forms: `loop` (infinite, can `break` with a value), `while`, and `for` over iterators. Prefer iterator `for` over index loops — it sidesteps bounds bugs and is idiomatic.
Functions
Declared with `fn`, explicit typed parameters and return type after `->`. Distinguish statements (no value) from expressions (the trailing expression without a semicolon is the return value). This expression-oriented design is why omitting a semicolon meaningfully changes behavior.
Memory management in Rust
No garbage collector and no manual malloc/free — memory is freed deterministically when its owner goes out of scope (RAII / the `Drop` trait). This gives predictable, pause-free reclamation; the compiler statically guarantees no use-after-free, double-free, or leaks-by-dangling. The whole ownership system exists to make this safe at zero runtime cost.
Jargon #0 — Mutability
Bindings are immutable unless declared `let mut`. Mutability is a property of the binding/reference, not just the data, and it composes with borrowing (you can only take a `&mut` if the binding is `mut`). Immutable-by-default is a correctness lever: it makes shared state and aliasing bugs visible at compile time.
Jargon #1 — Stack vs heap
Fixed-size, `Copy` types (integers, bools, arrays of known size) live on the stack and are cheap to copy; dynamically-sized or owned data (String, Vec, Box<T>) stores its buffer on the heap with a stack-side pointer/len/capacity. Understanding this predicts what moves vs copies and where allocation cost lives — a common interview probe.
Jargon #2 — Ownership
Every value has exactly one owner; assigning or passing a non-Copy value moves ownership (the source is invalidated), and when the owner drops, the value is freed. This single-owner rule is what makes deterministic cleanup and data-race freedom decidable at compile time. Expect to explain move-vs-clone and why `let b = a;` can invalidate `a`.
Jargon #3 — Borrowing & references
Instead of moving, you can borrow via references: any number of shared `&T` immutable borrows, XOR exactly one exclusive `&mut T` — never both simultaneously. This aliasing-XOR-mutability rule is the borrow checker's core invariant and is precisely what prevents data races and iterator invalidation without a runtime. Lifetimes (later) exist to prove references never outlive their data.
Structs
Product types for modeling domain data: named-field structs, tuple structs, and unit structs. Derive traits (`#[derive(Debug, Clone, PartialEq)]`) to opt into printing/cloning/equality. They're the primary building block; combined with enums they give you algebraic data types.
Implementing structs
`impl` blocks attach behavior: methods take `self`/`&self`/`&mut self` (borrowing rules apply to the receiver), and associated functions (no self, e.g. `Struct::new`) act as constructors/factories. The `&self` vs `&mut self` vs `self` choice encodes whether a method reads, mutates, or consumes the value.
Enums
Sum types where each variant can carry different data (`enum Msg { Quit, Move{x:i32,y:i32}, Write(String) }`) — far richer than C enums or TS string unions. They make illegal states unrepresentable and pair with `match` for exhaustive handling. `Option` and `Result` are just enums from the standard library.
Pattern matching
`match` destructures values and is exhaustive — the compiler forces you to handle every variant (or add `_`), so adding an enum variant surfaces every site that must change. `if let`/`let else`/`while let` are ergonomic single-pattern forms. Exhaustiveness is a major reliability feature versus switch-with-fallthrough languages.
Error handling
Recoverable errors use `Result<T, E>`; unrecoverable bugs use `panic!` (unwinds/aborts). The `?` operator propagates errors up by early-returning the `Err`, replacing verbose try/catch nesting. Idiomatic APIs return `Result`; libraries often use `thiserror`/`anyhow` — reserve `unwrap()`/`expect()` for cases you can prove can't fail.
Option enum
`Option<T>` (`Some`/`None`) is Rust's null replacement — absence is encoded in the type, so you can't dereference a 'null' by accident. Combinators (`map`, `and_then`, `unwrap_or`, `?`) let you chain without unwrapping. This is the language-level fix for the billion-dollar null mistake.
Collections
Heap-backed growable containers: `Vec<T>` (dynamic array), `HashMap<K,V>`, and `String` (UTF-8 growable); plus slices (`&[T]`, `&str`) as borrowed views. Ownership and borrowing govern them — e.g. you can't push to a Vec while holding a reference into it (that's iterator invalidation, caught at compile time).
Cargo, packages & external deps
Dependencies (crates) are declared in Cargo.toml with semver ranges and resolved via Cargo.lock; crates.io is the registry. Feature flags enable conditional compilation to keep builds lean, and workspaces group multiple crates. Know the crate-vs-module distinction and how `cargo add`/features affect compile time and binary size.
Leftovers — Traits, Generics, Lifetimes, Concurrency, Macros, Async
Traits = shared behavior/interfaces enabling static (generics, monomorphized) or dynamic (`dyn Trait`) dispatch; generics + trait bounds give zero-cost polymorphism. Lifetimes are compile-time annotations proving references stay valid. `Send`/`Sync` make 'fearless concurrency' — the borrow checker prevents data races across threads — and async/`Future`s (with runtimes like Tokio) provide cooperative, poll-based concurrency for I/O. These are the topics an SDE-2 must eventually master to write real Rust.

Real-world example

Building a concurrent web scraper / ingestion service in Rust

You're replacing a Node.js scraper that pegs one CPU and stalls under GC pressure. In Rust you model each scraped record as a struct with derived Debug/Clone, and represent a page's outcome as an enum (Fetched(String), Redirected(Url), Failed(StatusCode)) matched exhaustively. Network and parse errors return Result<Page, ScrapeError> and propagate with `?`; missing optional fields (e.g. a page's canonical URL) are Option<String> so you never dereference a null. Results accumulate into a Vec and are deduped via a HashMap keyed by URL — and when you try to hold a reference into that Vec while also pushing, the borrow checker stops you at compile time, killing a whole class of iterator-invalidation bugs. For throughput you go async with Tokio: an HTTP client trait lets you swap real vs mock fetchers (generics for tests, dyn for a plugin registry), and you fan out with tasks. Because shared state (the dedup map) must cross threads, you wrap it in Arc<Mutex<..>> — Send/Sync bounds force you to make sharing explicit, so there are no data races. Ownership guarantees each response buffer is freed the moment its task finishes, giving flat, GC-pause-free memory instead of Node's sawtooth. The result ships as a single static binary via `cargo build --release`.

Interview Q&A (SDE-2)

Q1. Explain the borrow checker's core rule and why it prevents data races.
At any point a value can have either any number of shared immutable references (&T) or exactly one exclusive mutable reference (&mut T), never both. Since a data race requires two threads accessing the same memory where at least one writes, and the rule forbids simultaneous aliasing-plus-mutation, races become impossible to express in safe code. The check is fully static, so it costs nothing at runtime, and it also incidentally prevents single-threaded bugs like iterator invalidation.
Q2. What's the difference between move, Copy, and Clone?
Move is the default for heap-owning types: assigning or passing the value transfers ownership and invalidates the source, preventing double-frees. Copy is an opt-in marker for cheap, fixed-size stack types (integers, bool, char) where a bitwise duplicate is made and the original stays valid. Clone is an explicit, potentially expensive deep copy you call with `.clone()` for types like String or Vec — the language never inserts it implicitly, so allocation cost is always visible in the source.
Q3. When do you reach for lifetimes, and what problem do they solve?
Lifetimes are compile-time annotations that let the borrow checker prove a reference never outlives the data it points to, eliminating dangling pointers without a runtime. You need explicit annotations when a function returns a reference derived from its inputs or when a struct stores a reference — the compiler can't always infer which input the output borrows from. They generate no code; they're purely a proof obligation, which is why elision rules handle most cases automatically.
Q4. Result vs Option, and what does the `?` operator actually do?
Option<T> models presence/absence (a value that may be missing), while Result<T, E> models success/failure with an error payload. The `?` operator unwraps the Ok/Some case or, on Err/None, early-returns that variant from the enclosing function — so it only compiles where the return type is compatible. This replaces nested match/try-catch with linear code while keeping error handling explicit in the type signature, unlike exceptions which are invisible in signatures.
Q5. Static vs dynamic dispatch with traits — trade-offs?
Generics with trait bounds (`fn f<T: Trait>`) are monomorphized: the compiler generates a specialized copy per concrete type, giving inlinable, zero-cost calls at the price of larger binaries and longer compiles. `dyn Trait` (trait objects) uses a vtable for runtime dispatch — one indirection per call, smaller code, and it lets you store heterogeneous types in one collection (e.g. Vec<Box<dyn Handler>>). Reach for generics on hot paths and dyn when you need runtime polymorphism or to break monomorphization bloat.
Q6. How does Rust achieve 'fearless concurrency' without a GC?
Two auto-derived marker traits gate sharing: Send means a type can be moved to another thread, Sync means &T can be shared across threads. The borrow checker plus these bounds mean the compiler rejects code that would share mutable state without synchronization — e.g. you must wrap shared mutable data in Arc<Mutex<T>>, and a plain Rc won't compile across threads because it isn't Send. So data races are caught at compile time rather than debugged at 3am in production.
Q7. Box vs Rc vs Arc vs RefCell — when do you use each?
Box<T> is single-owner heap allocation (for recursive types, trait objects, or large values). Rc<T> is single-threaded shared ownership via reference counting; Arc<T> is its atomic, thread-safe counterpart (more expensive due to atomic ops). RefCell<T> moves borrow checking to runtime, enabling interior mutability (mutating through a shared reference) and panicking on violated borrow rules. Common combos: Rc<RefCell<T>> for shared mutable single-threaded graphs, Arc<Mutex<T>> for shared mutable state across threads.
Q8. Why choose Rust over Node.js for a service, and when would you not?
Choose Rust for CPU-bound work, low and predictable latency (no GC pauses), true multithreading, tight memory footprints, or when memory-safety bugs are unacceptable — parsers, proxies, data pipelines, WASM. Stick with Node when the workload is I/O-bound glue, iteration speed and the npm ecosystem matter more than raw performance, or the team lacks Rust expertise and the borrow-checker ramp isn't worth it. A pragmatic middle ground is keeping Node for orchestration and dropping to Rust via native addons (napi-rs) for the hot path.
