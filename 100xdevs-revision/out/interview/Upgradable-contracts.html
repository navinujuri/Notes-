Why upgradability matters (Context)
A deployed contract is just bytecode living at an address, and you often need to fix bugs or ship features after launch. This sets up the core tension: EVM code is frozen once deployed, yet real protocols must evolve. Interview angle: framing why 'upgradeability' is even a problem worth solving on an immutable chain.
Contracts are immutable
Once bytecode is deployed to an address, it can never be changed — this is a deliberate trust/security property (users can audit code that can't be swapped). The downside is that bugs are permanent; TheDAO and the Parity multisig freeze are the canonical cautionary tales. This immutability is exactly what proxy patterns work around without breaking the guarantee entirely.
Goal of upgradability
Upgradeability means keeping a stable public address and persistent on-chain state while being able to change the executing logic. You can't mutate bytecode, so the trick is indirection: separate 'where state and identity live' from 'which code runs'. Interview angle: state the three invariants you want to preserve — address, storage, and user approvals/balances.
Naive approaches (and why they fail)
Naive option 1: deploy a fresh contract and tell everyone to use the new address — breaks integrations, wallets, and existing token approvals. Naive option 2: a registry/lookup contract users query for the current address — still forces every caller to add a hop and doesn't migrate existing state or funds. Both fail the 'stable address + persistent state' requirement, motivating delegatecall-based proxies.
delegatecall opcode
delegatecall executes another contract's code in the caller's context: the callee's logic runs against the caller's storage, and msg.sender / msg.value are preserved from the original call. This is the primitive that lets a proxy 'borrow' logic while state stays put. Interview angle: contrast with regular call (callee's own storage, msg.sender becomes the proxy) — mixing these up is the #1 conceptual error.
delegatecall worked example
A minimal demo where contract A delegatecalls contract B's setter, and the variable that changes is A's storage slot, not B's — proving code comes from B but state belongs to A. It also shows the danger: A and B must agree on slot layout or you write to the wrong variable. This example is the mental model every proxy is built on.
Proxy pattern
Split the system into a Proxy (holds state + the permanent address, and delegatecalls everything) and an Implementation/Logic contract (holds code only). Upgrading = pointing the proxy's stored implementation address at a new logic contract; users keep calling the same proxy address. This is the foundational pattern; Transparent/UUPS/Beacon are all variations of it.
onlyOwner on upgrade
The function that sets a new implementation address must be access-controlled (onlyOwner/admin), otherwise anyone could repoint the proxy to malicious logic and drain everything. This introduces the admin key as the trust anchor — and the question of who holds it (multisig, timelock, DAO). Interview angle: upgradeability reintroduces centralization/trust that immutability removed.
Storage layout constraint
Because the implementation runs in the proxy's storage, both must share a compatible slot layout, and across versions you may only append new variables — never reorder, remove, or change the type of existing ones. Violating this causes storage collisions where new code reads/writes the wrong slot, silently corrupting balances. This is the single most common source of upgrade bugs and a favorite interview trap.
Solution: fixed/unstructured storage slots
To stop the proxy's own bookkeeping (implementation address, admin) from colliding with the implementation's sequential slots, store them at pseudo-random fixed slots derived from a keccak hash rather than at slot 0/1. This 'unstructured storage' idea is what ERC-1967 later standardizes. It cleanly separates proxy metadata from application state.
Problem: proxy can't forward unknown calls
A hand-written proxy only forwards the specific functions you coded, so calls to any other implementation function (or new functions added in v2) hit nothing and revert. You need a generic mechanism to forward arbitrary calldata rather than enumerating every selector. This motivates the fallback function.
fallback (and receive) forwarding
The proxy's fallback function catches any call whose selector it doesn't explicitly define and delegatecalls it to the implementation, forwarding raw calldata and bubbling up returndata/reverts (via assembly). This makes the proxy transparently expose the implementation's entire ABI without knowing it in advance. receive() handles plain ETH transfers similarly.
Simple proxy assembled
A complete minimal proxy: a stored implementation address plus a fallback that uses inline assembly to delegatecall, copy calldata in, and return/revert with the returndata. This is essentially OpenZeppelin's Proxy.sol core. Knowing why assembly is needed (to forward arbitrary-length data and preserve exact return/revert behavior) is a good depth signal.
ERC-1967 storage slots
ERC-1967 standardizes the exact slots for the implementation address (bytes32(uint256(keccak256('eip1967.proxy.implementation')) - 1)), the admin, and the beacon, plus Upgraded/AdminChanged/BeaconUpgraded events. Standardization lets explorers like Etherscan auto-detect proxies and show the real implementation ABI. It's the storage-layout 'Solution' formalized into a shared convention.
Overview of common proxy patterns
Sets up the three production patterns built on the ERC-1967 proxy: Transparent, UUPS, and Beacon. The axis of comparison is where the upgrade logic lives and how many proxies one implementation serves. Interview angle: be able to pick the right one per scenario and state its cost.
Transparent proxy
Upgrade/admin logic lives in the proxy, which routes by caller: the admin address hits the proxy's own admin functions, everyone else gets delegatecalled to the implementation. This resolves the selector-clash problem where an implementation function could shadow an admin function. Trade-off: extra SLOAD + branch on every call makes it more gas-expensive, and it needs a separate ProxyAdmin contract.
UUPS proxy
The upgrade function (upgradeTo + _authorizeUpgrade) lives in the implementation, not the proxy, so the proxy stays minimal and cheaper per call. The catch: if a new implementation forgets to include the upgrade logic, the contract becomes permanently un-upgradeable ('bricked'). It's the currently recommended default (cheaper, cleaner) as long as you guard the upgrade path carefully.
Beacon proxy
Many proxies delegate to a single Beacon contract that stores the one implementation address, so upgrading the beacon upgrades every proxy at once. Ideal for factory scenarios — thousands of identical wallets/vaults that must all move to v2 together. Trade-off: an extra external call to the beacon per invocation, and a single beacon becomes a high-value point of control/failure.

Real-world example

Shipping v2 of a UUPS-upgradeable staking vault — and debugging the balances that broke

You maintain a DeFi staking vault deployed behind a UUPS/ERC-1967 proxy that custodies user deposits; users interact only with the proxy address, and its storage holds every stakedBalance. For v2 you add a rewards feature. During QA on a fork, users report their balances are suddenly wrong after the upgrade. Root cause: in v2 you inserted a new state variable (rewardRate) *before* the existing stakedBalances mapping instead of appending it, shifting every subsequent storage slot — a classic storage-layout collision, since the implementation runs against the proxy's untouched storage. The fix is to only append new variables (or reserve a __gap) and keep the layout identical up to v1. Along the way you rely on the whole stack: the proxy's fallback delegatecalls into the logic, the implementation address sits in the ERC-1967 slot so Etherscan shows the right ABI, upgradeTo is access-controlled via _authorizeUpgrade (and you double-check v2 still includes it so you don't brick the vault), and because it's UUPS you also disable initializers on the implementation and use an initializer instead of a constructor so the proxy's storage — not the logic contract's — gets set up. If instead you were running a fleet of per-user vaults from a factory, you'd reach for a Beacon proxy so one beacon upgrade moves all of them to v2 atomically.

Interview Q&A (SDE-2)

Q1. Why can't you just edit a deployed contract, and what exactly does a proxy keep constant vs. change?
Deployed EVM bytecode at an address is immutable by design — that's the trust guarantee that lets users audit code that can't be swapped underneath them. A proxy works around this without breaking the address/state guarantees: the proxy contract keeps the permanent address and all storage, and delegatecalls into a separate implementation contract that holds the logic. 'Upgrading' just repoints the stored implementation address, so the address and state persist while the executing code changes.
Q2. Explain delegatecall and its context semantics, and name the classic exploit it enabled.
delegatecall runs the callee's code in the caller's context: it uses the caller's storage, and msg.sender / msg.value stay as they were in the original call (unlike a normal call, where the callee uses its own storage and sees the proxy as msg.sender). That's what lets logic live elsewhere while state stays in the proxy. The classic pitfall is the second Parity multisig freeze: a shared library contract was left uninitialized, someone took ownership and called selfdestruct, and because dependents delegatecalled into it, ~$150M+ was permanently frozen.
Q3. What is a storage collision in a proxy, and what are the rules for evolving storage safely across upgrades?
Because the implementation executes against the proxy's storage, both must agree on which variable maps to which slot; if v2 reorders, removes, retypes, or inserts a variable, existing data is read/written from the wrong slot and silently corrupts (e.g., balances). The rule is append-only: never change existing declarations, only add new variables at the end. Standard mitigations are reserving a __gap uint256 array for future variables, using ERC-1967 fixed slots for proxy metadata, and increasingly namespaced/diamond storage (ERC-7201) to isolate layouts.
Q4. Transparent proxy vs. UUPS — how do they differ and when would you pick each?
In a Transparent proxy the upgrade logic lives in the proxy and it routes by caller (admin → admin functions, others → implementation) to avoid selector clashes; this costs extra gas on every call and needs a ProxyAdmin. In UUPS the upgrade function lives in the implementation, so the proxy is smaller and cheaper per call, but a new implementation that omits the upgrade authorization permanently bricks upgradeability. UUPS is the modern default for gas reasons; Transparent is fine when you want upgrade logic firmly outside the implementation and don't mind the overhead.
Q5. Why can't upgradeable contracts use constructors, and what replaces them?
A constructor only runs once, at the moment the implementation is deployed, and it initializes the *implementation's* storage — but the proxy has its own separate storage, so the proxy would be left uninitialized. Instead you use an initializer function protected by an `initializer` modifier so it runs exactly once against the proxy's storage. You should also call _disableInitializers() in the implementation's constructor so the logic contract itself can't be initialized and hijacked directly, and be aware initialization must be done atomically (or guarded) to avoid front-running.
Q6. What is the 'bricking' risk unique to UUPS, and how do you prevent it?
In UUPS the upgrade capability lives in the implementation, so if you deploy a new implementation that forgets to inherit/include the upgrade function (and its _authorizeUpgrade access control), there's no longer any code path to upgrade again — the contract is frozen at that version forever. Prevent it by always inheriting OpenZeppelin's UUPSUpgradeable, keeping _authorizeUpgrade present and correctly access-controlled in every version, and testing the upgrade path itself on a fork before shipping. OZ's upgrade tooling also runs safety checks that catch a missing upgrade function.
Q7. When is a Beacon proxy the right choice, and what's the cost?
Use a Beacon proxy when you have many instances of the same contract — e.g., a factory minting thousands of identical wallets or vaults — and you want to upgrade them all at once. Each proxy reads the implementation address from a single shared Beacon, so one setImplementation call on the beacon atomically upgrades every instance. The costs are an extra external call to the beacon on each invocation (more gas) and concentration risk: the beacon (and whoever controls it) is a single high-value point of control for the entire fleet.
Q8. What does ERC-1967 standardize and why does it matter operationally?
ERC-1967 fixes the storage slots for the implementation address, admin, and beacon at hash-derived locations (e.g., keccak256('eip1967.proxy.implementation') - 1) and defines Upgraded/AdminChanged/BeaconUpgraded events. Putting this metadata at pseudo-random slots guarantees it won't collide with the implementation's sequentially-assigned application storage. Operationally it means block explorers and tooling can automatically detect a proxy and surface the real implementation's ABI and verified source, which is essential for auditing and interacting with the contract.
