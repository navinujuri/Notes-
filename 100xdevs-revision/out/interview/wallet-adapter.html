Client side apps in Solana, Wallet adapter

Cohort 3.0 · 8 lessons · SDE-2 (3+ yrs) interview prep

Concept summary — all 8 steps

Recap of Solana primitives
Quick review of the core model the client sits on: accounts, lamports (1 SOL = 1e9 lamports), ed25519 keypairs, RPC endpoints (devnet/testnet/mainnet-beta), and programs. Interview angle: a browser dApp never holds a private key — it only holds a public key and delegates all signing to the wallet, so you must be fluent in which operations are on-chain (cost fees, need a blockhash) vs off-chain (message signing).
Scaffold the React app
Bootstrap with Vite (or CRA) and install the adapter stack: @solana/web3.js, @solana/wallet-adapter-react, @solana/wallet-adapter-react-ui, and @solana/wallet-adapter-wallets. Why it matters: the packages split responsibilities — web3.js is the RPC/transaction layer, the adapter packages abstract dozens of wallets (Phantom, Solflare, Backpack) behind one interface so you don't integrate each wallet by hand.
Wire up the provider tree
The 'things to do' is wrapping the app in three nested providers: ConnectionProvider (holds the RPC Connection + endpoint), WalletProvider (manages selected wallet, autoConnect, event listeners), and WalletModalProvider (the connect-wallet UI). Interview angle: order and nesting matter — useConnection and useWallet are context hooks, so any component reading them must live inside these providers, and the endpoint is where you pick devnet vs mainnet.
Request an airdrop (devnet faucet)
Use connection.requestAirdrop(publicKey, amountInLamports) to fund a wallet on devnet/testnet for testing, then confirm the returned signature. Why it matters: airdrops only work on non-mainnet clusters and are aggressively rate-limited (often ~1-2 SOL per request), so production funding flows can't rely on this — it's purely a test faucet, and you should handle 429/'airdrop limit reached' errors gracefully.
Display the user's balance
Read the connected wallet's SOL via connection.getBalance(publicKey), which returns an integer number of lamports; divide by LAMPORTS_PER_SOL for display. Interview angle: this is an async RPC call tied to publicKey — re-fetch in a useEffect keyed on the key (and after transactions), keep amounts in lamports internally to avoid floating-point drift, and consider commitment level since balances differ across processed/confirmed/finalized.
Sign a message (off-chain auth)
Use the signMessage function from useWallet: encode the string with new TextEncoder().encode(...), get back an ed25519 signature, and verify it server-side. Why it matters: this is authentication (Sign-In With Solana / proof of key ownership) with no transaction, no fee, and no on-chain footprint — the classic use case is logging a user in by proving they control the wallet, and the interviewer wants you to distinguish this from signing a transaction.
Send SOL (build + send a transaction)
Construct a Transaction with SystemProgram.transfer({fromPubkey, toPubkey, lamports}), then call sendTransaction(tx, connection) from useWallet, which asks the wallet to sign and broadcasts it. Interview angle: know the moving parts — the wallet (connected pubkey) is the fee payer, the transaction needs a recentBlockhash and feePayer (the adapter/wallet fills these), and you must await confirmation on the returned signature rather than assuming instant finality.
Assignment: full client dApp
Ties it together: build a working dApp that connects a wallet, shows balance, airdrops on devnet, signs a message, and transfers SOL to another address. The goal is demonstrating the end-to-end pattern — provider setup, the useConnection/useWallet hooks, error/loading states, and confirmation handling — which is exactly the skeleton every real Solana frontend reuses.

Real-world example

A devnet-first token-tipping dApp for a creator platform

You're an SDE-2 building the tipping widget for a creator platform. The React app wraps everything in ConnectionProvider (pointed at devnet during dev, mainnet-beta in prod), WalletProvider with autoConnect, and WalletModalProvider so users click one 'Connect Wallet' button and pick Phantom or Backpack. On connect you read connection.getBalance(publicKey) to show their SOL balance, and during QA you expose a 'Fund test wallet' button that calls requestAirdrop so testers aren't blocked on real funds. To gate access to a creator's private page, you require the user to signMessage a nonce your backend issued — no fee, no transaction — and your server verifies the ed25519 signature against their public key to establish a session. When a fan tips, you build a SystemProgram.transfer transaction, hand it to sendTransaction, and only show 'Tip sent!' after you confirm the signature. The bugs you actually hit are instructive: airdrop 429s on devnet, a stale balance because you forgot to re-fetch after the transfer, and treating sendTransaction's returned signature as final before confirmation — the transaction was still only 'processed', not 'finalized'.

Interview Q&A (SDE-2)

Q1. What is the responsibility split between ConnectionProvider, WalletProvider, and WalletModalProvider?
ConnectionProvider owns the web3.js Connection object and the RPC endpoint (which cluster you talk to), exposed via useConnection. WalletProvider manages wallet state — the list of supported wallets, the currently selected/connected wallet, autoConnect, and connect/disconnect events — exposed via useWallet. WalletModalProvider is purely the UI layer providing the wallet-selection modal. They're nested (Connection outermost) because the hooks are React context, so any component using useConnection/useWallet must render inside the relevant provider.
Q2. Why can't you use requestAirdrop on mainnet, and what should a production funding flow do instead?
Airdrops are a faucet feature of devnet/testnet validators that mint test SOL out of thin air; mainnet has real economic value and no faucet, so the call fails there. Faucets are also rate-limited (typically a small amount per IP/address per window), so even on devnet you must handle 429/'airdrop limit reached' errors and back off. In production, users fund wallets by buying/transferring SOL from an exchange or on-ramp — the app never mints tokens for them.
Q3. getBalance returns what, and why keep amounts in lamports internally?
It returns an integer number of lamports (1 SOL = 1,000,000,000 lamports), not SOL. You convert for display by dividing by LAMPORTS_PER_SOL, but you should keep and compute with lamports as integers because SOL amounts as JS floats introduce rounding errors — 0.1 + 0.2 style drift is unacceptable for money. It's also an async RPC call, so it's affected by the commitment level and needs re-fetching after state-changing operations.
Q4. When would you use signMessage versus sendTransaction, and how do they differ?
signMessage is off-chain: it produces an ed25519 signature over arbitrary bytes, costs no fee, touches no ledger, and is used to prove key ownership — authentication, Sign-In With Solana, or authorizing an API call. sendTransaction is on-chain: it signs and broadcasts an actual transaction that mutates state (e.g., transferring SOL), requires a recent blockhash and a fee payer, and must be confirmed. Confusing the two is a common mistake — you don't 'send a transaction' just to log someone in.
Q5. Walk through sending SOL with the wallet adapter. Who pays the fee and what's the blockhash for?
You build a Transaction containing a SystemProgram.transfer instruction with fromPubkey (the connected wallet), toPubkey, and lamports, then call sendTransaction(tx, connection) from useWallet, which prompts the wallet to sign and then broadcasts. The connected wallet is the fee payer by default. The transaction needs a recentBlockhash — a recent slot's hash that bounds the transaction's validity window and prevents replay; the adapter/wallet typically fetches and sets it. After sending you get a signature and must await confirmation rather than assuming success.
Q6. How does the wallet adapter protect private keys, and what's the trust model?
The dApp never sees or handles the private key. The adapter communicates with the wallet extension/app over a defined interface; the wallet holds the secret key, shows a confirmation prompt, signs internally, and returns only the public key and signatures. The security model is that the user trusts the wallet (not the dApp) with their key, and they review each signing request — which is why phishing that tricks users into approving malicious transactions, not key theft, is the realistic attack surface.
Q7. How do you verify a signed message on the backend?
The client sends the original message bytes, the signature, and the public key. The server re-encodes the message the same way (TextEncoder) and verifies the ed25519 signature using a library like tweetnacl (nacl.sign.detached.verify(messageBytes, signature, publicKeyBytes)) or @noble/ed25519. To prevent replay, the message should embed a server-issued nonce and/or timestamp that the backend tracks and expires, so a captured signature can't be reused.
Q8. What do the commitment levels processed, confirmed, and finalized mean, and why do they matter here?
They describe how much of the cluster has acknowledged a slot. 'processed' means a single validator has processed it (fastest, can be rolled back), 'confirmed' means a supermajority of the cluster has voted on it (a good default for UX), and 'finalized' means it's rooted and effectively irreversible (~31+ confirmations). They matter because getBalance and transaction confirmation depend on the level — showing 'success' at 'processed' can lie to the user if the block is dropped, so money-moving flows should wait for at least 'confirmed', ideally 'finalized' for high-value actions.
