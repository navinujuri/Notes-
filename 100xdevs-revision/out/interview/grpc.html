GRPC

Cohort 2.0 · 8 lessons · SDE-2 (3+ yrs) interview prep

Concept summary — all 8 steps

What is RPC
Remote Procedure Call is a paradigm where calling a function on a remote server looks like a local call — the client stub serializes args, ships them over the network, and the server executes and returns a result. It matters because it abstracts away transport/serialization so you think in terms of methods and messages, not HTTP verbs and URLs; the interview angle is contrasting RPC (action/verb-oriented, tightly coupled contracts) with REST (resource/noun-oriented, HATEOAS).
Implementing a dumb RPC
A hand-rolled RPC: agree on a wire format (e.g. JSON {method, params}), have the client POST it, and dispatch on the method name server-side to invoke the real function. The point is to expose what a framework hides — manual serialization, method routing, error propagation, and the lack of a typed contract — which is exactly the pain gRPC/protobuf solve.
Common RPC protocols
Know the landscape: JSON-RPC and XML-RPC (simple, text-based, HTTP), SOAP (heavy, WSDL-defined, enterprise), Thrift (Facebook), and gRPC (Google, HTTP/2 + protobuf). Interview angle: pick based on schema/typing needs, streaming support, cross-language codegen, and performance — gRPC wins on binary efficiency and streaming, JSON-RPC on human-readability and browser reach.
Protocol Buffers
Protobuf is Google's language-neutral IDL and binary serialization format: you define messages/services in a .proto file and compile it to typed code in many languages. It matters because it gives you a single source-of-truth contract, compact tag-based binary encoding (much smaller/faster than JSON), and forward/backward compatibility when fields are added — key for evolving microservice APIs.
Types in protobuf
Covers scalar types (int32/int64, sint/fixed variants, bool, string, bytes, double), enums, nested messages, repeated (lists), map<k,v>, and oneof. The nuance to know: every field has a unique tag number (the wire identity — never reuse/renumber), all fields are optional in proto3 with type defaults, and choosing sint/varint vs fixed affects encoding size for negative/large numbers.
Implementing services
In protobuf you declare a `service` with `rpc Method(Request) returns (Response)` entries, defining the RPC surface purely from the contract. This matters because the service block is what the compiler turns into client stubs and server interfaces — the contract drives codegen, so API design happens in the .proto, not in handwritten glue.
Implementing services using gRPC
gRPC takes the compiled .proto and generates a server base class you implement and a client stub you call; it runs over HTTP/2 giving multiplexing, and supports four call types: unary, server-streaming, client-streaming, and bidirectional streaming. Interview angle: know why HTTP/2 matters (multiplexed streams over one connection, header compression), that gRPC needs a proxy (grpc-web/Envoy) for browsers, and how deadlines, cancellation, and metadata work.
Adding types
Extending a live API means adding new fields/messages/RPCs while preserving compatibility: give new fields fresh tag numbers, never change or reuse existing ones, and use `reserved` for removed fields. This is the crux of schema evolution in interviews — old clients ignore unknown fields and new clients see defaults for missing ones, so additive changes stay backward/forward compatible.

Real-world example

Order + Payment services in an e-commerce backend

You own an Order service that must call a Payment service to charge a card. Instead of a REST call with hand-written JSON DTOs on both sides, you define a payment.proto with a `PaymentService` exposing `rpc Charge(ChargeRequest) returns (ChargeResponse)`, where ChargeRequest has typed fields (int64 amount_cents, string currency, string idempotency_key with fixed tag numbers). Both teams compile the same .proto to generate the Go server skeleton and the Java client stub, so the contract is shared and type-safe across languages. Payment runs over HTTP/2, letting Order set a 2s deadline and propagate cancellation if the checkout request is abandoned. When Payments later needs to return a fraud score, they add `float fraud_score = 7;` — older Order builds simply ignore the new field, so the rollout is non-breaking. If you ever needed a stream of settlement events, you'd switch that RPC to server-streaming rather than polling.

Interview Q&A (SDE-2)

Q1. When would you choose gRPC over REST, and when not?
Choose gRPC for internal service-to-service traffic where you want strong typed contracts, cross-language codegen, low latency, compact binary payloads, and streaming (e.g. high-throughput microservices, real-time feeds). Avoid it for public/browser-facing APIs where human-readable JSON, caching, and broad tooling matter — browsers can't speak raw gRPC without grpc-web plus a proxy like Envoy. REST is also better when loose coupling and easy debuggability outweigh performance.
Q2. Why does gRPC use HTTP/2, and what does that buy you?
HTTP/2 provides request/response multiplexing over a single TCP connection, so many concurrent RPCs share one connection without head-of-line blocking at the HTTP layer, plus header compression (HPACK) and binary framing. This is what makes long-lived bidirectional streaming and high concurrency efficient. It also enables server push semantics for streaming RPCs, which HTTP/1.1 could only fake with long-polling or chunked hacks.
Q3. How do protobuf field tag numbers work and why can't you reuse them?
Each field's tag number is its identity on the wire — encoded values are keyed by (tag << 3 | wire_type), not by field name, so names are purely for code generation. If you reuse a retired tag for a new field of a different type, an old serialized message could be misinterpreted by a new client, silently corrupting data. That's why you mark removed tags/names as `reserved` to prevent accidental reuse.
Q4. Explain protobuf's backward/forward compatibility model.
In proto3 all fields are optional and additive changes are safe: unknown fields encountered by an older reader are preserved/ignored, and fields missing for a newer reader take type defaults (0, empty string, etc.). This means new clients can read old data (backward compat) and old clients can read new data (forward compat) as long as you only add fields with new tags and never change existing types or renumber. Removing a field is fine if you reserve its tag.
Q5. What are the four gRPC call types and give a use case for each?
Unary (one request, one response) for a normal call like Charge. Server-streaming (one request, stream of responses) for pushing a feed of updates or paginated results. Client-streaming (stream of requests, one response) for uploading chunks/metrics that aggregate into one result. Bidirectional streaming (both stream independently) for chat, live telemetry, or interactive sessions — over a single HTTP/2 connection.
Q6. How do deadlines/timeouts and cancellation work in gRPC, and why prefer deadlines over timeouts?
A client sets a deadline (an absolute point in time) that propagates through the call as metadata; the server sees remaining time and can abort work, and cancellation flows downstream to nested RPCs. Deadlines are preferred over per-hop timeouts because they compose across a call chain — each service subtracts elapsed time rather than each restarting its own clock, preventing the total latency budget from ballooning. Without deadlines, hung dependencies leak resources and cause cascading failures.
Q7. Compared to a hand-rolled JSON-RPC, what concrete problems does gRPC + protobuf solve?
It removes hand-written serialization and method-dispatch glue by generating typed stubs from one contract, so client and server can't drift out of sync and you get compile-time type checking across languages. Binary protobuf is smaller and faster to encode/decode than JSON, and the schema gives you disciplined evolution via tag numbers and reserved fields. You also get built-in streaming, deadlines, metadata, and status codes instead of reinventing them.
Q8. What does `oneof` do and how does it differ from just having multiple optional fields?
`oneof` declares a group of fields where at most one can be set at a time; setting one clears the others, and it saves memory since only the active field is stored. It's ideal for tagged-union / variant payloads (e.g. a result that is either a Success or an Error). Unlike several independent optional fields, it enforces mutual exclusivity in the generated code and lets you switch on which case is set, but note oneof fields can't be `repeated`.
