CI/CD

Cohort 2.0 · 8 lessons · SDE-2 (3+ yrs) interview prep

Concept summary — all 8 steps

What CI and CD actually mean
CI (Continuous Integration) means every push is automatically built and tested so integration bugs surface within minutes instead of at a big-bang merge; CD is the overloaded term — Continuous Delivery keeps main always release-ready with a manual promote gate, while Continuous Deployment auto-ships every green commit to prod. The interview angle is precision: know that CI is about fast feedback on a shared mainline (small frequent merges, trunk-based dev) and be able to distinguish Delivery vs Deployment rather than using 'CD' loosely.
Continuous Deployment via GitHub Actions
GitHub Actions is the event-driven runner: workflows are YAML in .github/workflows triggered by events (push, pull_request, workflow_dispatch, schedule), composed of jobs (each on a fresh runner VM) and steps (shell commands or reusable actions). It matters because it co-locates pipeline-as-code with the repo, and interviewers probe on triggers, runner isolation, secrets, and why jobs are parallel-by-default while steps are sequential.
The monorepo setup
A monorepo holds multiple deployable apps/packages (frontend, backend, worker) in one repo with shared tooling and a single commit history. The CI/CD implication that matters most: you should scope pipelines with path filters (paths:/paths-ignore:) or a build tool's affected-graph so a change to one service doesn't rebuild and redeploy everything — this is the classic monorepo interview trade-off (atomic cross-cutting changes vs. build blast radius).
Anatomy of a CI/CD pipeline
A pipeline is an ordered graph of stages — typically checkout, install, lint/test, build artifact (often a Docker image), push to a registry, then deploy — where later stages gate on earlier ones passing. The key design idea is 'build once, promote the same immutable artifact through environments'; interviewers want you to reason about stage ordering, fail-fast, caching, and where approval gates live.
Creating the CI pipeline
The CI job wires up the basics: trigger on push/PR, actions/checkout, set up the runtime (setup-node etc.), restore a dependency cache, install, then run lint and tests — failing the check status so a broken commit can't merge. This is the fast, cheap feedback loop that should run on every PR and gate merges via branch protection / required checks.
Adding the build pipeline
The build stage turns source into a deployable artifact — here a Docker image built from a Dockerfile and tagged (usually with the immutable commit SHA, not just :latest) then pushed to a registry like Docker Hub / GHCR / ECR. It matters because a containerized, SHA-tagged image is the reproducible unit you ship and roll back to; interview angle: multi-stage builds and layer caching to keep images small and builds fast.
Adding the deploy step
The deploy stage takes the pushed image and updates the running environment — commonly SSH into a server (or trigger a k8s/ECS rollout) to pull the new tag and restart the container, gated behind the build succeeding and often a branch condition (deploy only on main). Discussion points: managing secrets/SSH keys via GitHub Secrets, environment protection rules, and how you achieve zero-downtime and rollback.
Pulling the Docker image on the target
On the server the deploy authenticates to the registry, docker pull the new SHA-tagged image, then stops the old container and starts the new one (docker run / compose up). The crux is that pulling by immutable tag guarantees the box runs exactly the artifact CI built, and rollback is just re-pulling the previous SHA — versus the anti-pattern of rebuilding on the server or trusting a mutable :latest.

Real-world example

Auto-deploying a backend service from a monorepo on every merge to main

You own a monorepo with a React frontend and a Node/Express API. You want: every PR runs lint + tests, and every merge to main that touches the API rebuilds and redeploys only the API to a single EC2 box — without disturbing the frontend. You write a GitHub Actions workflow triggered on push/PR with a paths filter of apps/api/**. The CI job checks out, sets up Node, restores an npm cache, installs, and runs eslint + jest as a required status check behind branch protection. On merge to main a build job runs docker build against apps/api/Dockerfile (multi-stage to keep the image lean), tags it with the commit SHA plus latest, and docker push to GHCR using a GITHUB_TOKEN. A dependent deploy job (needs: build, if: github.ref == 'refs/heads/main') uses an SSH action with a private key from GitHub Secrets to connect to the server, docker login to the registry, docker pull the SHA tag, and docker compose up -d to swap the container. When a bad deploy ships, you don't panic-rebuild — you re-run the deploy step pinned to the previous SHA, and because the image is immutable the box is back to a known-good state in seconds.

Interview Q&A (SDE-2)

Q1. Continuous Delivery vs Continuous Deployment — what's the actual difference?
Both keep main in a releasable state via an automated pipeline. Continuous Delivery stops at a manual promote/approval gate — a human clicks to ship to prod — so you can align releases with business timing. Continuous Deployment removes that gate: every commit that passes the pipeline goes straight to production automatically, which demands very strong test coverage, monitoring, and fast rollback because there's no human in the loop.
Q2. In GitHub Actions, what's the relationship between workflows, jobs, and steps, and how do they execute?
A workflow is a YAML file triggered by an event. It contains jobs, which by default run in parallel, each on its own fresh isolated runner VM. Within a job, steps run sequentially and share the same filesystem/workspace. Because jobs are isolated, to enforce order you use needs: to create dependencies, and to pass data between jobs you use artifacts or job outputs rather than the filesystem.
Q3. Why tag Docker images with the commit SHA instead of just using :latest?
:latest is a mutable pointer — two deploys can resolve to different images and you lose the mapping between what's running and which commit produced it, which breaks reproducibility and rollback. A SHA (or semver) tag is immutable: it names exactly one artifact forever, so you can deploy, audit, and roll back to a precise build. Best practice is push both — SHA for traceability/rollback and latest as a convenience pointer — but deploy by SHA.
Q4. How do you keep CI/CD efficient in a monorepo so one change doesn't rebuild everything?
Scope the trigger and build to what changed. In GitHub Actions use paths:/paths-ignore: filters so a workflow only fires for its service's directory; for finer control use a tool that computes the affected dependency graph (Nx, Turborepo, Bazel) to build/test/deploy only impacted projects. Combine with a matrix strategy for parallelism and aggressive layer/dependency caching. The trade-off is added pipeline complexity in exchange for avoiding the full-repo blast radius on every commit.
Q5. How should secrets like SSH keys and registry credentials be handled in the pipeline?
Store them in GitHub Actions encrypted secrets (repo/org/environment scope) and inject them as env vars at runtime — never commit them or bake them into images. Prefer short-lived, least-privilege credentials, and where possible use OIDC to assume a cloud role instead of long-lived static keys. Use environment-scoped secrets plus environment protection rules so prod credentials are only available to jobs targeting prod, and remember secrets are masked in logs but a careless echo can still leak them.
Q6. Your deploy SSHes in, stops the old container, and starts the new one — how do you make that zero-downtime and safe to roll back?
The stop-then-start pattern has a gap of downtime. To avoid it, run the new container alongside the old and cut traffic over only after a health check passes — blue/green or rolling via a reverse proxy/load balancer (or an orchestrator like ECS/Kubernetes that does this natively). Because images are immutable SHA tags, rollback is just redeploying the previous SHA. Add a health/readiness endpoint the pipeline polls before considering the deploy successful, and automate rollback on failure.
Q7. Why build the Docker image in CI and pull it on the server, rather than building on the server itself?
Building once in CI produces a single immutable artifact that is tested and then promoted unchanged through every environment — guaranteeing prod runs exactly what you tested. Building on the server is non-reproducible (differs by host state, network, base-image drift), wastes prod resources, and means dev/staging/prod can silently diverge. Pulling a pre-built, registry-stored image also makes deploys fast and rollbacks trivial.
Q8. How do you make CI itself fast and reliable as the codebase grows?
Cache dependencies and Docker layers, use multi-stage builds to shrink images, and parallelize with matrix jobs and job-level fan-out. Fail fast by running cheap checks (lint, unit tests) before expensive ones (integration, build). Keep tests deterministic — flaky tests destroy trust in the pipeline — and enforce green CI as a required status check via branch protection so nothing merges on red. For monorepos, only run what the change affects.
