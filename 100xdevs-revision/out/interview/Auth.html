Authentication

Cohort 2.0 · 8 lessons · SDE-2 (3+ yrs) interview prep

Concept summary — all 8 steps

What is authentication
Authentication proves who a caller is (identity), distinct from authorization which decides what they can do. HTTP is stateless, so after a successful login the server must issue a token/credential the client resends on every request; the whole topic is about how that credential is stored and transmitted safely. Interview angle: be able to cleanly separate authn vs authz and explain why statelessness forces per-request credentials.
JWT + localStorage
On login the server signs a JWT (header.payload.signature, HMAC or RSA) containing claims like userId/exp; the client stores it in localStorage and manually attaches it as `Authorization: Bearer <token>` on each request. The server verifies the signature and expiry without a DB/session lookup, making it stateless and horizontally scalable. Interview angle: localStorage is readable by any JS on the page, so this pattern is vulnerable to XSS token theft, and stateless JWTs are hard to revoke before expiry.
Authentication using cookies (Part 1)
Alternative to manual token handling: the server sends `Set-Cookie` on login and the browser automatically returns the cookie on every same-origin request, so the client code never touches the credential. The cookie can hold a session id (server-side session store) or a JWT. Interview angle: automatic transmission is the key win over localStorage, but it introduces CSRF risk precisely because the browser sends it without app involvement.
Authentication with cookies (Part 2)
Covers the mechanics of reading cookies server-side (e.g. cookie-parser), how the browser scopes cookies by domain/path, and CORS with credentials — `credentials: 'include'` on fetch plus `Access-Control-Allow-Credentials: true` and a specific (non-wildcard) origin on the server. Interview angle: the classic bug is cookies silently not being sent cross-origin because the credentials flags or SameSite settings weren't configured.
Properties of cookies
Key attributes: HttpOnly (blocks JS access, mitigates XSS theft), Secure (HTTPS only), SameSite (Lax/Strict/None — CSRF mitigation, None requires Secure), Domain/Path (scope), Expires/Max-Age (lifetime). Interview angle: HttpOnly + Secure + SameSite together are the standard hardening; know that SameSite=Lax is the modern browser default and that SameSite=None;Secure is required for legitimate cross-site cookies.
Example in Express (backend)
A minimal backend exposes /signup and /signin that verify credentials and respond with either a JWT (Bearer flow) or `res.cookie(name, token, {httpOnly, secure, sameSite})` (cookie flow), plus auth middleware that reads the token, verifies it, and attaches req.userId. Interview angle: passwords must be hashed (bcrypt/argon2, never plaintext), the JWT secret comes from env config, and the verify step belongs in reusable middleware, not per-route.
Frontend in React
A separate React app (different origin/port) calls the backend via fetch/axios; with JWT it stores and manually sends the Bearer header, with cookies it must set `withCredentials`/`credentials:'include'` and the backend must whitelist that origin with credentials enabled. Interview angle: this is where CORS-with-credentials and cookie SameSite issues surface, and where you decide token storage strategy (memory vs localStorage vs HttpOnly cookie).
Frontend served from Express
Serving the React build as static assets from the same Express server (express.static) makes frontend and API same-origin, which eliminates CORS entirely and lets HttpOnly SameSite=Strict cookies work seamlessly. Interview angle: same-origin deployment is the simplest and most secure setup for cookie auth; know the trade-off against a separately deployed/CDN-hosted SPA.

Real-world example

Session auth for a dashboard SPA behind a shared API

You are building the login flow for an internal analytics dashboard: a React SPA and an Express API. In dev, React runs on :5173 and the API on :3000, so you hit CORS — cookies aren't sent until you set `credentials:'include'` on fetch and, on the server, `cors({origin:'http://localhost:5173', credentials:true})` (wildcard origin is illegal with credentials). On signin, Express hashes-compares the password with bcrypt, signs a JWT, and sends it via `res.cookie('token', jwt, {httpOnly:true, secure:true, sameSite:'lax'})` so JS can't read it (XSS-resistant) and the browser auto-attaches it. Auth middleware reads the cookie, verifies the JWT, and sets req.userId. For production you drop the React build into express.static on the same origin, which makes CORS moot and lets you tighten to sameSite:'strict'. The debugging story an SDE-2 lives through: 'why is my cookie set in the response but never sent back?' — usually a missing credentials flag, a SameSite/Secure mismatch on http, or a cross-origin/domain scope problem.

Interview Q&A (SDE-2)

Q1. JWT in localStorage vs a session/JWT in an HttpOnly cookie — what's the security and operational trade-off?
localStorage is accessible to any JavaScript on the page, so a single XSS flaw leaks the token; you also send it manually via the Authorization header, which sidesteps CSRF. HttpOnly cookies can't be read by JS (XSS-resistant) and are sent automatically, but that auto-send is exactly what exposes you to CSRF, mitigated with SameSite and/or CSRF tokens. Cookies also give you easy server-controlled expiry/rotation, while stateless JWTs are hard to revoke early. The common recommendation is HttpOnly+Secure+SameSite cookies for browser apps.
Q2. How do you revoke a JWT before it expires?
You can't invalidate a self-contained signed JWT directly because verification is stateless — the token is valid until exp regardless. Practical options: keep tokens short-lived (minutes) with refresh tokens, maintain a server-side denylist/allowlist (which reintroduces state and a lookup), or bump a per-user token version/`tokenVersion` claim that you compare against the DB. This is the core trade-off: pure statelessness costs you revocation, so most real systems add some server state back.
Q3. What is CSRF, why do cookies enable it, and how do you defend against it?
CSRF tricks a logged-in user's browser into making an unwanted authenticated request; it works because the browser auto-attaches cookies to requests regardless of which site initiated them. Defenses: SameSite=Lax/Strict cookies (browser won't send on cross-site navigations/requests), anti-CSRF tokens (synchronizer/double-submit), and verifying Origin/Referer. Token-in-header auth (Bearer from localStorage) is inherently CSRF-safe because the attacker's site can't read or set that header, which is one reason people accept the XSS trade-off.
Q4. Walk through configuring CORS so cookies work between a React app and an Express API on different origins.
On the client, set `credentials:'include'` (fetch) or `withCredentials:true` (axios). On the server, set `Access-Control-Allow-Credentials: true` and reflect a specific origin — the wildcard `*` is rejected when credentials are included. You must also whitelist the needed methods/headers for preflight, and the cookie needs `SameSite=None; Secure` to be sent cross-site at all. Missing any one of these produces the classic 'cookie is set but never sent back' symptom.
Q5. Explain the JWT structure and what the signature actually guarantees.
A JWT is three base64url parts: header (alg/typ), payload (claims like sub, exp, iat), and signature. The signature is computed over header+payload with a secret (HMAC) or private key (RSA/ECDSA) and guarantees integrity and authenticity — that the claims weren't tampered with and came from someone holding the key. It does NOT provide confidentiality: the payload is only encoded, not encrypted, so never put secrets in it. Also validate `alg` server-side to avoid the `alg:none` and algorithm-confusion attacks.
Q6. What do HttpOnly, Secure, and SameSite each protect against?
HttpOnly blocks document.cookie access from JavaScript, so XSS can't exfiltrate the cookie. Secure ensures the cookie is only sent over HTTPS, preventing interception on plaintext connections. SameSite controls cross-site sending: Strict never sends on cross-site requests, Lax (the modern default) sends only on top-level GET navigations, and None sends always but requires Secure — SameSite is your primary in-cookie CSRF mitigation.
Q7. Why does serving the React build from the same Express server simplify auth?
Same-origin means the browser treats the SPA and API as one origin, so CORS disappears entirely and you avoid the credentials/preflight configuration. Cookies are first-party, letting you use the strictest SameSite=Strict without breaking requests, and there's no cross-site cookie concern. The trade-off is deployment coupling — you give up independently scaling/CDN-hosting the frontend, so large teams often still split origins and pay the CORS cost.
Q8. Where should an SPA store an access token, and why not localStorage by default?
The most XSS-resistant place is an HttpOnly cookie, so JS never touches the token; if you must use header-based auth, keeping the token in memory (a JS variable/closure) is safer than localStorage because it dies on tab close and isn't persisted for theft. localStorage persists and is fully readable by any script, so one XSS gives long-lived token access. A common pattern is a short-lived access token in memory plus a refresh token in an HttpOnly cookie.
