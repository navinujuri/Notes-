Promises and async, await

Cohort 2.0 · 10 lessons · SDE-2 (3+ yrs) interview prep

Concept summary — all 10 steps

Course intro / shoutouts
Non-technical housekeeping and acknowledgements; no engineering content. Skip in revision — nothing here affects how you reason about async code.
Classes in JS
ES6 `class` is syntactic sugar over JS's prototype-based inheritance: `constructor`, instance fields, and methods that actually live on `Class.prototype`. It matters here because `Promise` is itself a class you instantiate with `new`, and understanding `this`-binding pitfalls (methods lose `this` when passed as bare callbacks) is a recurring interview trap.
Inheritance in classes
`extends` sets up the prototype chain and `super()` calls the parent constructor / `super.method()` calls the overridden parent method. Interview angle: explain that `extends` links prototypes (not a copy), method resolution walks the chain, and you must call `super()` before touching `this` in a subclass constructor.
Some more classes
Covers static methods (called on the class, not instances — e.g. `Promise.all`, `Promise.resolve`), getters/setters, and private fields (`#field`). Knowing static vs instance members clarifies why `Promise.resolve()` is static while `.then()` is on the instance.
Promise class
A Promise is an object representing the eventual result of an async operation with three states — pending, fulfilled, rejected — that settle exactly once and are then immutable. You construct it with an executor `(resolve, reject) => {}` and consume it with `.then`/`.catch`/`.finally`; its callbacks run on the microtask queue, which drains before the next macrotask (e.g. `setTimeout`). This is the core abstraction the whole topic builds on.
Callback hell
Deeply nested callbacks ("pyramid of doom") for sequential async work — hard to read, hard to error-handle, and it cedes control to the callee (inversion of control). Promises fix this by flattening nesting into a linear `.then` chain with a single terminal `.catch`, and async/await flattens it further into straight-line code.
Async await syntax
`await` suspends an `async` function until the awaited promise settles, returning its fulfilled value or throwing its rejection reason, so you write async logic as if it were synchronous and use normal `try/catch`. It's sugar over `.then` chains; the key interview point is that `await` in a loop serializes requests — use `Promise.all` when the operations are independent.
Defining your own async function
Any function marked `async` always returns a Promise (a returned value is auto-wrapped, a `throw` becomes a rejection). You also 'promisify' legacy callback APIs by wrapping them in `new Promise((resolve, reject) => …)` and calling resolve/reject inside the callback — the standard way to bridge old code into async/await.
Err-first callback vs rejects in promises
Node's classic convention is `callback(err, data)` where you must inspect `err` on every call, versus Promises which surface failures through the rejection channel handled by `.catch`/`try-catch`. The trade-off: err-first is easy to ignore silently and doesn't compose, whereas promise rejections propagate down the chain automatically and can be caught in one place.
Assignments
Hands-on practice: promisify a callback API, chain vs parallelize with `Promise.all`, and add proper error handling. The point is to internalize sequencing, parallelism, and rejection propagation by writing them, not just reading.

Real-world example

Order-checkout service in a Node backend

You're building the `placeOrder(userId, cart)` handler for an e-commerce service. It must: (1) load the user and validate each cart item's inventory — independent reads, so you run them concurrently with `await Promise.all([...])` instead of awaiting them one-by-one in a loop (which would serialize the round-trips); (2) charge the payment provider and create the order row — these are sequential because the order needs the charge ID; (3) fire a confirmation SMS through a legacy library that only exposes an err-first callback `sms.send(to, body, (err, res) => …)`, which you wrap once in a `sendSms` helper that returns `new Promise((resolve, reject) => …)` so it slots into your async/await flow. The whole handler is one `async` function wrapped in `try/catch`: a rejected payment throws into the `catch`, where you roll back and return a 402, and the non-critical SMS is wrapped in its own `Promise.allSettled` (or an inner try/catch) so a failed text doesn't fail the paid order. This one function exercises classes (`new Promise`, `Promise.all`), rejection propagation, sequential-vs-parallel awaiting, and bridging an err-first callback into promises.

Interview Q&A (SDE-2)

Q1. What are the three states of a Promise, and what does it mean that a promise is 'settled'?
A promise is `pending` until it either fulfills (with a value) or rejects (with a reason). 'Settled' means it has moved to fulfilled or rejected; this transition happens exactly once and is immutable — calling `resolve` or `reject` again is a no-op. This is why you can safely attach `.then` handlers even after a promise has already settled: they'll still fire on the microtask queue with the stored result.
Q2. Compare Promise.all, Promise.allSettled, Promise.race, and Promise.any.
`all` resolves with an array of all values but rejects immediately (fail-fast) if any input rejects — good when you need every result and want to abort on the first failure. `allSettled` never rejects; it resolves with a `{status, value|reason}` array, ideal when you want every outcome regardless of failures (e.g. best-effort side jobs). `race` settles as soon as the first input settles, fulfilled or rejected — used for timeouts. `any` resolves with the first fulfillment and only rejects (with an `AggregateError`) if all inputs reject — used for redundancy/failover.
Q3. Why does `console.log` after a resolved promise's `.then` sometimes run before the `.then` callback, but a `setTimeout(…,0)` callback runs even later?
Promise callbacks are scheduled on the microtask queue, which the event loop drains completely after the current synchronous call stack finishes and before rendering or the next macrotask. `setTimeout` schedules a macrotask, which runs only after all pending microtasks are done. So synchronous code runs first, then all microtasks (promise handlers, including chained ones), then macrotasks like timers — a common ordering question to probe event-loop understanding.
Q4. Does `await` make code slower, and how do you run independent async operations concurrently?
`await` itself isn't slow, but awaiting operations one at a time serializes them — if you `await` inside a `for` loop, each iteration waits for the previous network call to finish. When the operations are independent, kick them all off first and await together: `await Promise.all(items.map(fetchItem))`. Only keep sequential awaits when a later call genuinely depends on an earlier result. This latency difference (sum vs max of durations) is a frequent real-world performance bug.
Q5. How do you convert an err-first callback API into a promise, and why is rejection often better than err-first?
Wrap it: `const readFileP = (p) => new Promise((resolve, reject) => fs.readFile(p, (err, data) => err ? reject(err) : resolve(data)))`, or use `util.promisify`. Rejection is generally better because errors propagate automatically down a `.then` chain to a single `.catch` (or are caught by `try/catch` with await), whereas err-first requires you to manually check `err` at every call site and is silently ignorable. Promises also compose with combinators like `Promise.all`, which raw callbacks don't.
Q6. What does an async function return, and what happens if you return a value versus throw inside it?
An async function always returns a Promise. A plain `return value` fulfills that promise with the value (a returned promise is adopted/flattened, not double-wrapped), and a `throw` inside it rejects the promise with the thrown reason. This is why callers must still `.then`/`await` the result and why you can use ordinary `try/catch` around `await` — the thrown rejection is turned back into a synchronous-looking exception at the await point.
Q7. What is an unhandled promise rejection, and how does forgetting to await cause bugs?
If a promise rejects and no `.catch` or surrounding `try/catch` handles it, Node emits an `unhandledRejection` (and crashes by default in modern versions). Forgetting `await` is a classic cause: a fire-and-forget async call whose rejection nobody catches, or a function that returns before its inner async work finishes, so errors surface after the request has already responded. Always await (or explicitly `.catch`) promises you start, and use `try/finally` to guarantee cleanup like releasing a DB connection.
Q8. What's the difference between `return somePromise` and `return await somePromise` inside a try block?
Functionally the resolved value is the same, but timing and error handling differ. `return await p` awaits the promise inside the function, so if `p` rejects the error is caught by the local `try/catch`; a bare `return p` hands the still-pending promise to the caller, so a rejection escapes the local `catch` entirely. Inside a `try/finally`, `return await` also ensures the `finally` runs only after the promise settles. Outside a try block the extra `await` is usually a redundant microtask tick.
