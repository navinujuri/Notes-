Building Codeforces (Project track)

Cohort 2.0 · 10 lessons · SDE-2 (3+ yrs) interview prep

Concept summary — all 10 steps

Project demo
A walkthrough of the finished product: users browse problems, submit code in multiple languages, the code is compiled and run against hidden test cases in an isolated environment, and a verdict (AC / WA / TLE / MLE / RE / CE) comes back — plus contests and leaderboards. Seeing the end-to-end flow first matters because it frames the whole thing as a distributed compute problem wearing a web-app costume; the hard part is not the CRUD, it's safely running untrusted code at scale.
What you'll learn
The load-bearing skills: sandboxed/remote code execution, async job queues (submit → enqueue → worker → verdict), a deterministic judging pipeline, contest and leaderboard modeling, and monorepo structure across platform + setter + worker. Interview angle: name these as the reason the project is resume-worthy — it exercises systems concerns (isolation, backpressure, fairness) that plain full-stack CRUD never touches.
Good-to-haves (stretch scope)
Beyond MVP: real-time verdict streaming (WebSocket/SSE), an Elo/Glicko-style rating system, virtual contests, editorials, plagiarism detection, and partial/subtask scoring. The interview value is scoping discipline — being able to say what is MVP (submit → judge → verdict) versus what is a fast-follow, and defending the cut line under a deadline.
Extra discussions (edge cases & threats)
The uncomfortable questions: how do you stop malicious submissions (fork bombs, memory bombs, filesystem/network access, reading other test data), how do you make time limits fair across heterogeneous hardware, and how do you cap compute cost when judging is the expensive part. These adversarial/fairness concerns are exactly what a senior interviewer probes; the MVP demo hides them.
Tech stack
Typical choice: Next.js (React) frontend, a Node/Express (or Next API) backend, Postgres via Prisma for durable state, Redis as the submission queue, and Docker containers as the sandboxed judge/worker. The point is fitting tools to a compute-heavy async workload — a durable relational store for submissions/contests, a fast in-memory broker to decouple the web tier from judging, and containers for per-submission isolation.
System design — Platform
The user-facing architecture and the submission lifecycle: the web/API tier writes a Submission row (status=PENDING) and pushes a job to Redis, decoupling the request from execution; judge workers pull jobs, run code in a container, and write back the verdict; the client learns the result via polling or a push channel. Interview angle: the queue is what gives you backpressure, horizontal worker scaling, and a fast HTTP response instead of holding a connection open for seconds.
System design — Setter
The problem-authoring side: authors define a problem (statement, constraints, time/memory limits), upload test inputs and expected outputs, and often a boilerplate/driver generator produces language-specific function stubs from a single problem definition. This service is the source of truth the judge trusts — its output format, test-case storage layout, and limit metadata are what the worker reads at judge time, so its correctness directly determines verdict correctness.
Database design
Core entities: User, Problem, TestCase (or test data referenced by object store), Submission (code, language, verdict, runtime, memory, timestamps), Contest, ContestParticipant, and a rating/leaderboard model. Discussion points: indexing submissions by (userId, problemId) and (contestId), computing rankings without full-table scans, and whether to store large test data as BLOBs vs in S3-style storage with only pointers in Postgres.
Code walkthrough — Problem setter
Reading the actual setter implementation: how a problem folder/record is structured, how test cases are parsed and stored, and how the boilerplate generator turns a problem's input/output signature into runnable driver code per language. Understanding this end-to-end matters because a bug here (wrong signature, mismatched I/O parsing) surfaces as bogus WA verdicts that look like platform bugs.
Code walkthrough — Website/platform
Reading the platform code: the submit endpoint that persists a submission and enqueues the job, the worker that spins up a container with resource limits and captures stdout/exit status/timings, the diff/verdict logic, and how results propagate back to the UI. This is where the async architecture and sandbox config become concrete — trace one submission from button click to rendered verdict.

Real-world example

Judging a contest submission under a 1-second time limit

You're an SDE-2 on the judge team. A contestant submits a C++ solution to Problem D during a live round. The Next.js frontend POSTs the code to the API, which writes a Submission row (status=PENDING) and pushes {submissionId, language} onto a Redis queue — the HTTP call returns in ~50ms, and the UI starts polling the submission's status. A judge worker pulls the job, fetches the problem's test cases and limits from the setter's store, wraps the code with the generated boilerplate driver, and runs it inside a fresh Docker container hardened with no network, a read-only filesystem, a memory cgroup cap, a process/pids limit (to kill fork bombs), and a wall-clock timeout slightly above the 1s CPU budget. It streams each test input, diffs stdout against expected output, and records runtime/memory. On the third hidden test the process exceeds 1s, so the worker writes verdict=TLE, and the next poll flips the UI to 'Time Limit Exceeded on test 3.' Later a contestant reports 'my AC solution got TLE' — you debug by checking whether the time limit is measured as CPU time vs wall time, whether a cold container/JIT startup ate into the budget, and whether the judge box was overloaded running too many concurrent containers. That triage — fairness, isolation, and queue saturation — is the real job.

Interview Q&A (SDE-2)

Q1. How do you safely execute arbitrary untrusted user code, and what specifically stops a malicious submission?
Run each submission in a short-lived, isolated sandbox — typically a Docker container (or firecracker/gVisor for stronger isolation) started fresh per submission and destroyed after. Harden it: no network, read-only or ephemeral filesystem, a non-root user, a memory cgroup limit (kills memory bombs → MLE/RE), a pids/process limit (kills fork bombs), dropped Linux capabilities and a seccomp profile to block dangerous syscalls, and a wall-clock timeout as a backstop to the CPU limit. Never run untrusted code in the API process or share a container across submissions, since that leaks state and test data.
Q2. Why judge submissions through an async queue instead of running the code synchronously in the request handler?
Judging takes seconds and is CPU-bound, so doing it inline would tie up a web worker/connection, collapse under a contest's submission spike, and give a terrible client experience. A queue (Redis/SQS/etc.) decouples the tiers: the API returns immediately after persisting PENDING and enqueuing, judge workers scale horizontally and pull at their own pace, and the queue absorbs bursts as backpressure instead of dropping requests. It also makes retries, rate limiting, and priority (e.g., contest vs practice) natural.
Q3. How do you enforce fair, deterministic time limits across different hardware and languages?
Measure CPU time, not just wall-clock, so an overloaded host or scheduler jitter doesn't unfairly TLE a correct solution, and pin/limit concurrency per judge box so containers aren't starved. Normalize by hardware (Codeforces publishes a reference machine) and give slower languages a multiplier or higher limit for the same problem. Warm the runtime where possible to avoid cold-start/JIT costs counting against the budget, and always keep a wall-clock backstop so an infinite loop still gets killed.
Q4. How would you model submissions and contest leaderboards in the database for efficient ranking?
Store each Submission with userId, problemId, contestId, language, verdict, runtimeMs, memoryKb, and createdAt, indexed on (contestId, userId) and (userId, problemId). For rankings, don't scan all submissions on every request — maintain a denormalized standings/score table (per contest: solved count, penalty/time, last-AC timestamp) updated as verdicts land, or use a Redis sorted set keyed by score for live leaderboards. Large test data lives in object storage (S3) with only pointers in Postgres to keep rows small.
Q5. What does the problem setter's boilerplate/driver generator do, and why decouple it from the languages?
The setter defines a problem once — its input/output signature, constraints, and limits — and a generator emits language-specific driver code (stubs plus the I/O harness that reads stdin, calls the user's function, and prints output in the expected format). Decoupling means an author adds a problem or a new language without hand-writing a harness per pair, and the judge trusts one canonical I/O contract. It's a load-bearing correctness surface: a mismatch between the generated parser and the test files shows up as spurious WA verdicts that look like platform bugs.
Q6. How do you deliver the verdict back to the user in near real-time — polling, SSE, or WebSockets?
Polling the submission status endpoint every 1-2s is the simplest and perfectly adequate for MVP — it's stateless and trivially scalable, at the cost of some redundant requests. SSE (server-sent events) is a clean one-way upgrade for pushing status changes without the client hammering the API. WebSockets are warranted when you also want live leaderboard/contest updates and bidirectional traffic; the trade-off is connection state and needing a pub/sub layer (e.g., Redis) so the worker that computed the verdict can notify the web tier holding the socket.
Q7. How does a rating system like Elo/Glicko work here, and when do you update it?
After a rated contest, each participant's rating changes based on their rank versus the field's expected performance: you beat higher-rated users → gain more, lose to lower-rated → drop more, roughly like Elo generalized to a multiplayer ranking (Codeforces uses a custom variant). Update it as a batch job once final standings are frozen (after system tests / anti-cheat), not live during the round, because ratings must reflect the settled result and be recomputable. Store rating history per user so you can display graphs and re-run if a contest is invalidated.
Q8. A user reports their previously-accepted solution now gets TLE intermittently. How do you triage?
First separate submission-level from system-level causes: check whether the limit is measured as CPU vs wall time (wall time makes it sensitive to host load), and whether judge boxes are running too many concurrent containers so submissions starve for CPU. Look at container cold-start/JIT overhead counting toward the budget, and whether the solution is genuinely borderline (near the limit on the largest test). Confirm determinism by re-judging on an idle reference machine; if it passes there, the fix is capacity/concurrency control or moving to CPU-time measurement, not the user's code.
