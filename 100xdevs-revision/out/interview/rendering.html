CSR vs SSR vs SSG

Cohort 2.0 · 3 lessons · SDE-2 (3+ yrs) interview prep

Concept summary — all 3 steps

Client-Side Rendering (CSR)
Server ships a near-empty HTML shell plus a JS bundle; the browser downloads, parses, and executes JS to build the DOM and fetch data at runtime (the classic SPA model, e.g. plain React/Vite). It matters because it gives cheap-to-host, highly interactive apps but pays with slow first paint (blank screen until JS hydrates), weak SEO for crawlers that don't run JS, and TTI that degrades with bundle size — the interview angle is recognizing when a heavy client bundle is the root cause of poor Core Web Vitals.
Server-Side Rendering (SSR)
For each request the server runs the app, fetches data, and returns fully-formed HTML, which the client then hydrates to attach interactivity (Next.js getServerSideProps / React Server Components with dynamic rendering). It matters for fast, SEO-friendly first paint on personalized or frequently-changing pages; the trade-offs interviewers probe are per-request server cost/latency (TTFB), the need for caching/CDN, and hydration mismatches or the 'hydration is slow' problem.
Static Site Generation (SSG)
HTML is rendered once at build time and served as static files from a CDN, optionally refreshed via revalidation/ISR (Next getStaticProps, Gatsby, Astro). It matters for the best possible TTFB and scalability on content that changes infrequently (docs, blogs, marketing, catalogs); the key interview nuance is handling dynamic-ish data with Incremental Static Regeneration or on-demand revalidation, and knowing that build times explode as page count grows.

Real-world example

Choosing a rendering strategy for an e-commerce site

You're building a storefront on Next.js with three page types and must pick a rendering mode per route rather than one for the whole app. Product listing and detail pages use SSG with ISR: they're SEO-critical and mostly static, so you pre-render them at build time and set revalidate (or on-demand revalidation triggered by a webhook when inventory/price changes) to keep them fresh without a full rebuild. The logged-in cart, checkout, and 'recommended for you' pages use SSR because they're per-user and can't be cached publicly, so you render on request and lean on edge caching only where safe. The interactive parts inside an already-rendered page — image zoom, add-to-cart, filters — are CSR/hydration on top of the server HTML. A common real debugging task here is a hydration mismatch warning on product pages caused by rendering a locale-dependent price or Date.now() on the server that differs from the client, which you fix by making that value deterministic or deferring it to a client-only effect.

Interview Q&A (SDE-2)

Q1. When would you choose SSR over SSG, and vice versa?
Use SSG when content is the same for all users and changes infrequently (blogs, docs, product catalogs) — you get CDN-level TTFB and near-infinite scalability. Use SSR when the page is personalized per user or depends on data that must be fresh on every request (dashboards, authenticated views, live pricing). The decision hinges on data freshness requirements and cacheability, not on whether the page is 'dynamic' in a UI sense. If content changes often but isn't per-user, ISR/on-demand revalidation is the middle ground that keeps SSG's performance.
Q2. What is hydration, and why can it hurt performance?
Hydration is the client-side process where React (or another framework) attaches event listeners and reconstructs component state on top of server-rendered HTML so the static markup becomes interactive. It hurts because the browser must download and execute the full JS bundle and walk the entire tree before the page is interactive, so a page can look ready (fast FCP) but have a slow TTI — the 'uncanny valley' where clicks don't work yet. Mitigations include reducing bundle size, code-splitting, streaming SSR with Suspense, partial/selective hydration, islands architecture, and React Server Components which ship zero JS for non-interactive parts.
Q3. How does CSR affect SEO, and how do you mitigate it?
With pure CSR the initial HTML is an empty shell, so crawlers that don't execute JavaScript (or crawl with a limited render budget) may index a blank page, and social/link-preview scrapers that never run JS see no metadata. Modern Googlebot does render JS but on a deferred, best-effort second pass, which delays and can degrade indexing. Mitigations: use SSR or SSG for crawlable content, add dynamic rendering/prerendering for bots, or at minimum server-render the meta tags. For most content-driven sites, SSG/SSR is the correct answer rather than trying to patch CSR.
Q4. What is Incremental Static Regeneration (ISR) and what problem does it solve?
ISR lets you keep SSG's static, CDN-served output while updating individual pages after deployment, either on a time interval (revalidate: N seconds) or on-demand via a revalidation API/webhook. It solves SSG's core weakness — stale content and long full rebuilds — so a 100k-page catalog can update a single changed product without rebuilding everything. The typical model is stale-while-revalidate: the first request after expiry serves the old page, triggers a background regeneration, and subsequent requests get the fresh version. The trade-off is eventual consistency and a small window of stale data, which is fine for most content but not for strongly consistent needs.
Q5. How do these rendering modes map to Core Web Vitals like FCP, LCP, and TTI?
SSG generally wins FCP/LCP because HTML arrives instantly from a CDN with no server compute; SSR has good FCP/LCP too but adds TTFB from per-request rendering and data fetching. CSR has the worst FCP/LCP since content only appears after JS downloads, parses, and fetches data. TTI is dominated by JS bundle size and hydration cost, so a large CSR or heavily-hydrated SSR app can have poor TTI even when the paint metrics look acceptable — which is why bundle size and hydration strategy matter as much as the rendering mode.
Q6. Can you mix rendering strategies in one application? How?
Yes, and it's the recommended approach — modern frameworks let you choose per route or even per component. In Next.js you can make marketing pages SSG, dashboards SSR, and docs SSG with ISR, all in one app; with the App Router and React Server Components you mix server components (no client JS) with client components ('use client') on the same page. The mental model is: pick the strategy per route based on that route's data freshness, personalization, and SEO needs, then use client components/islands only for the interactive slices. This avoids the false premise that an app must commit to a single rendering mode.
Q7. What are the trade-offs of SSR at scale, and how do you handle them?
SSR runs application code and data fetching on every request, so it increases server CPU, latency (TTFB), and infrastructure cost, and makes you vulnerable to slow upstream data sources or traffic spikes. You handle it with layered caching: full-page or fragment caching with stale-while-revalidate at a CDN/edge, per-route cache-control headers, request-level data caching/deduplication, and moving compute to the edge to cut round-trips. For pages that don't truly need per-request freshness, converting them to SSG/ISR removes the load entirely — the biggest SSR win is often deciding a route doesn't need SSR.
