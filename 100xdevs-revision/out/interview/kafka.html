Kafka

Cohort 2.0 · 9 lessons · SDE-2 (3+ yrs) interview prep

Concept summary — all 9 steps

What is Kafka?
A distributed, append-only commit log used as a durable, high-throughput pub/sub and event-streaming backbone. Unlike a traditional queue, messages are retained (by time/size, not by consumption) and reread by offset, which is why it fits event sourcing, log aggregation, and decoupling services. Interview angle: it's a replicated log, not a message broker that deletes on ack.
Jargon
Core vocabulary: broker (a Kafka server), topic (named log), partition (the unit of ordering/parallelism), offset (per-partition message index), producer/consumer, consumer group, replica/leader/ISR, and the controller. Being precise here matters because most Kafka trade-offs (ordering, parallelism, delivery guarantees) are stated in terms of partitions and offsets.
Start Kafka locally
Run a broker locally via Docker (or KRaft mode / ZooKeeper-backed in older versions) to develop against a real broker. Practically you expose a bootstrap server (e.g. localhost:9092) and create topics with a chosen partition count. Interview angle: modern Kafka (2.8+/3.x) can run in KRaft mode without ZooKeeper, since the metadata is itself a Raft-managed log.
Kafka in a Node.js process
Use a client like KafkaJS: construct a Kafka instance with the broker list, then create producer/consumer objects, connect them, and send/subscribe. Key point is connections are long-lived and the client handles metadata discovery, leader routing, and retries. Interview angle: producers batch and are async; you must await connect() and handle graceful disconnect on shutdown.
Splitting into producer and consumer scripts
Separate the producing and consuming code into independent processes to mirror real deployments where producers and consumers scale and fail independently. This makes the decoupling concrete: producers don't know or wait for consumers, and consumers can be down and catch up later via retained offsets. It also sets up testing consumer-group behavior with multiple consumer processes.
Consumer groups and partitions
A consumer group is a set of consumers sharing a groupId; Kafka assigns each partition to exactly one consumer in the group, so parallelism is capped by partition count. Multiple groups each get their own full copy of the stream (independent offsets). Interview angle: to scale a workload you add partitions AND consumers; extra consumers beyond partition count sit idle.
Partitions in action
Observing that messages with the same key land on the same partition and are consumed in order, while different partitions are processed concurrently and have no cross-partition ordering. Rebalancing reassigns partitions when consumers join/leave, briefly pausing consumption. Interview angle: ordering is guaranteed only within a partition, never across the topic.
Three cases to discuss
The canonical consumer-to-partition scenarios: (a) consumers < partitions — some consumers own multiple partitions; (b) consumers == partitions — one-to-one, ideal balance; (c) consumers > partitions — surplus consumers are idle. This frames capacity planning and why over-provisioning consumers without enough partitions wastes resources.
Partitioning strategy
How the producer maps a message to a partition: default is hash(key) % partitions, round-robin when no key, or a custom partitioner. The choice trades off ordering (same key => same partition => ordered) against load balance (skewed/hot keys cause partition hotspots). Interview angle: partition count is hard to reduce later and re-keying reshuffles ordering, so choose the key and count deliberately up front.

Real-world example

Order-processing pipeline for an e-commerce backend

You build an 'orders' topic with 6 partitions, keyed by customerId, feeding downstream services. The checkout service is a KafkaJS producer that emits OrderPlaced events; keying by customerId guarantees all events for one customer stay ordered on the same partition (so 'placed' is always processed before 'cancelled'). Two consumer groups read the same stream independently: a 'fulfillment' group (6 consumers, one per partition) that reserves inventory and ships, and an 'analytics' group (2 consumers) that aggregates metrics — each group keeps its own offsets, so a slow analytics consumer never blocks fulfillment. When you deploy a 7th fulfillment consumer during a sale, it sits idle because there are only 6 partitions, which teaches you to provision partitions for peak parallelism up front. During a rolling deploy you observe a rebalance pause, and you debug a hot partition when one wholesale customer's huge order volume hashes to a single partition — the classic key-skew problem.

Interview Q&A (SDE-2)

Q1. Why does Kafka use partitions, and what do they trade off?
Partitions are the unit of parallelism and ordering. Splitting a topic into partitions lets many producers and consumers work concurrently and lets data exceed a single machine, since each partition is an independent log that can live on a different broker. The trade-off is that ordering is only guaranteed within a partition, so you sacrifice global ordering for throughput; you recover per-entity ordering by keying related messages to the same partition.
Q2. How do consumer groups relate to partitions, and what happens if you have more consumers than partitions?
Within a single consumer group, each partition is assigned to exactly one consumer, so the group's effective parallelism is min(consumers, partitions). If consumers exceed partitions, the surplus consumers get no assignment and stay idle as hot standbys. Different consumer groups are fully independent — each maintains its own offsets and receives the entire stream — which is how you fan the same data out to multiple applications.
Q3. What delivery guarantees does Kafka provide, and how do you achieve exactly-once?
By default Kafka gives at-least-once (consumer can reprocess after a crash before committing offsets) or at-most-once (commit before processing). Exactly-once within Kafka is achieved with the idempotent producer (dedupes retries via producer id + sequence number) plus transactions that atomically write messages and commit consumer offsets. End-to-end exactly-once to an external system still requires idempotent writes or the transactional outbox pattern, since Kafka can't make an arbitrary sink atomic.
Q4. How does the default partitioner assign messages, and why can that cause a hot partition?
With a key, the default partitioner uses hash(key) % partitionCount so the same key always lands on the same partition (preserving order); with no key it round-robins (or uses sticky batching in newer clients). A hot partition arises when key distribution is skewed — e.g. one tenant or one popular product generates most traffic — so its partition gets far more load than others, bottlenecking the single consumer that owns it. Fixes include a composite/salted key, more partitions, or a custom partitioner.
Q5. What is a rebalance and why is it disruptive?
A rebalance is the redistribution of partitions among a group's consumers when membership changes (join, leave, crash, or timeout). During a classic (eager) rebalance the group stops-the-world: all consumers revoke their partitions and pause processing until reassignment completes, which adds latency and can cause duplicate processing around the boundary. Cooperative/incremental rebalancing and static group membership reduce this by only moving the partitions that must move and avoiding churn on brief restarts.
Q6. How does Kafka achieve durability and what do replicas, leader, and ISR mean?
Each partition has replicas across brokers; one is the leader that handles all reads/writes and the others are followers that replicate it. The ISR (in-sync replica set) is the followers caught up with the leader. Durability is tuned via replication.factor, min.insync.replicas, and producer acks=all — with acks=all a write is acknowledged only after all in-sync replicas have it, so you can tolerate broker failures without data loss, at the cost of higher latency.
Q7. Why is choosing partition count up front important, and what makes it hard to change?
Partition count caps consumer parallelism and is baked into the key-to-partition mapping. Increasing partitions is possible but changes hash(key) % N, so existing keys get remapped to different partitions and per-key ordering is broken for the transition; decreasing partitions is not supported at all. Because of this you size partitions for anticipated peak throughput early, accepting some over-provisioning rather than reshuffling a live keyed stream later.
Q8. How does a consumer track progress, and what's the risk of auto-commit?
A consumer commits offsets (per partition) to record how far it has read, stored in the internal __consumer_offsets topic. With auto-commit, offsets are committed on a timer regardless of whether processing finished, so a crash between commit and completing work causes message loss (at-most-once) or, depending on timing, reprocessing. For correctness you typically disable auto-commit and commit manually after the work is durably done, accepting at-least-once and making handlers idempotent.
