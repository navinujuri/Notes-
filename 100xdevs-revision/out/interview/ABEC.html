Advance BE communication

Cohort 2.0 · 8 lessons · SDE-2 (3+ yrs) interview prep

Concept summary — all 8 steps

What is backend communication?
The set of mechanisms by which a client and server (or server-to-server) exchange data — request/response, push, and streaming — over transports like HTTP, TCP, and WebSockets. It matters because choosing the wrong pattern (e.g., polling where you need push) is the root cause of latency, cost, and scaling problems; interviewers probe whether you can map a product requirement (chat, notifications, dashboards) to the right communication model.
Types of communication
The main models are synchronous request/response (HTTP REST, gRPC), short/long polling, Server-Sent Events (unidirectional server→client push over HTTP), and WebSockets (full-duplex bidirectional). The interview angle is trade-offs: latency vs overhead, directionality, firewall/proxy friendliness, and reconnection/backpressure — e.g., SSE is simpler and auto-reconnects but is one-way, while WebSockets are bidirectional but need custom heartbeat/reconnect logic.
WebSockets
A protocol (RFC 6455) that starts as an HTTP request with an `Upgrade: websocket` header, then switches the same TCP connection to a persistent full-duplex channel using a lightweight framing protocol. It matters for low-latency, high-frequency bidirectional workloads (chat, live trading, multiplayer, collaborative editing); expect questions on the handshake, framing, `ping/pong` heartbeats, and how it differs from HTTP/2 streams and SSE.
WebSockets in Node.js
Node's single-threaded event loop and non-blocking I/O make it well-suited to holding thousands of concurrent socket connections cheaply, which is why `ws` and Socket.IO are common choices. The angle: understand that each connection is an in-memory object tied to one process, so connection state doesn't automatically share across instances — a fact that dictates the scaling story later.
ws in Node.js (code)
The `ws` library gives a bare `WebSocketServer` where you handle `connection`, `message`, `close`, and `error` events and call `socket.send()`; you own message parsing (typically JSON), heartbeats via `ping`/`pong` + `terminate` on dead sockets, and any rooms/broadcast logic. Socket.IO layers reconnection, rooms, and fallbacks on top — knowing when the raw `ws` primitive suffices vs when Socket.IO's abstractions earn their overhead is the practical judgment tested.
Client-side code
The browser `WebSocket` API (`new WebSocket(url)`, `onopen/onmessage/onclose/onerror`, `send()`) plus a client-owned reconnect-with-backoff loop and a message/event protocol agreed with the server. The interview point is resilience: browsers don't auto-reconnect raw WebSockets, so you must handle exponential backoff, resubscribing after reconnect, and buffering or dropping messages sent while disconnected.
Next.js
Next.js is oriented around serverless/edge request-response and does not host long-lived WebSocket connections in its standard API routes or on typical serverless platforms; you run WebSockets on a separate long-running Node server (or use a managed service like Pusher/Ably) and connect from the client. The angle is architectural: recognizing that stateful persistent connections don't fit a stateless serverless deployment model and separating the real-time tier from the SSR/API tier.
Scaling WebSocket servers
Because connections and room state live in a single process's memory, scaling horizontally requires sticky sessions at the load balancer plus a shared backplane (usually Redis pub/sub) so a message published on one instance reaches subscribers connected to another. Expect deep questions on the Redis pub/sub adapter, sticky routing (or a stateless token approach), handling reconnect storms, and observability of connection counts — this is the highest-signal part of the topic for an SDE-2.

Real-world example

Live order-tracking + support chat for a food-delivery app

You're building the customer app's real-time layer: a map that updates the courier's position every few seconds and a support chat, both needing sub-second server→client push. You reject long-polling (too much HTTP overhead at scale) and pick WebSockets for the bidirectional chat while noting SSE would suffice for one-way courier updates. The app is deployed on Next.js/Vercel for SSR and REST APIs, but since serverless can't hold persistent connections, you stand up a dedicated Node `ws` server behind a load balancer. The browser client opens a WebSocket, authenticates with a JWT on connect, subscribes to its order channel, and runs exponential-backoff reconnect that re-subscribes on reopen. As traffic grows you run multiple `ws` instances with sticky sessions and a Redis pub/sub backplane, so a courier-location event ingested on instance A is fanned out to the customer connected to instance B. You add `ping/pong` heartbeats to reap dead connections and export a live connection-count metric to catch reconnect storms after a deploy.

Interview Q&A (SDE-2)

Q1. When would you choose Server-Sent Events over WebSockets, and vice versa?
Choose SSE when you only need server→client push (live feeds, notifications, progress) because it runs over plain HTTP, auto-reconnects with `Last-Event-ID`, and is proxy/firewall friendly with almost no client code. Choose WebSockets when you need low-latency bidirectional traffic (chat, multiplayer, collaborative editing) or binary frames. SSE is limited by browser per-domain HTTP/1.1 connection caps (mitigated under HTTP/2) and is text-only, while WebSockets require you to build heartbeats and reconnection yourself.
Q2. Walk me through the WebSocket handshake.
The client sends a normal HTTP GET with `Connection: Upgrade`, `Upgrade: websocket`, a `Sec-WebSocket-Key` (random base64), and `Sec-WebSocket-Version: 13`. The server responds `101 Switching Protocols` with `Sec-WebSocket-Accept`, which is the SHA-1 of the client key concatenated with a fixed GUID, base64-encoded — proving it understood the protocol. After that the same TCP connection is reused for full-duplex, length-prefixed frames rather than HTTP messages. Because it begins as HTTP, it can share port 443 and pass through most proxies.
Q3. Why can't you just run WebSockets inside Next.js API routes on Vercel?
Next.js API routes on serverless/edge platforms are request-scoped: the function spins up, handles one HTTP request, and is torn down, so it can't hold a long-lived, stateful socket. WebSockets need a persistent process that owns connection state in memory for the connection's lifetime. The standard pattern is to keep Next.js for SSR/REST and run a separate long-running Node WebSocket server (or a managed real-time service like Ably/Pusher), with the client connecting to that endpoint directly.
Q4. How do you scale a WebSocket service horizontally across multiple instances?
Connections are pinned to whichever process accepted them and their room state is in that process's memory, so you need two things: sticky sessions (or connection-level routing) at the load balancer so a client's traffic keeps hitting the same instance, and a shared backplane — typically Redis pub/sub — so an event produced on one instance is published and delivered to subscribers connected to any other instance. Socket.IO's Redis adapter implements exactly this fan-out. You also plan for reconnect storms on deploy and monitor per-instance connection counts.
Q5. How do you detect and clean up dead WebSocket connections?
TCP can keep a socket 'open' long after the peer vanished (network drop, sleeping laptop), so you implement application-level heartbeats: the server periodically sends a `ping`, marks the socket alive on the corresponding `pong`, and on the next interval `terminate()`s any socket that didn't respond. In the `ws` library this is the classic `isAlive` flag pattern on an interval. Without this you leak memory and file descriptors and may broadcast to sockets that will never receive the data.
Q6. How should the client handle reconnection, and what are the pitfalls?
The browser `WebSocket` does not auto-reconnect, so the client must reopen on `close`/`error` using exponential backoff with jitter to avoid a thundering-herd reconnect storm after a server restart. On reconnect it must re-authenticate and re-subscribe to any channels/rooms, and decide how to handle messages produced while offline — either buffer-and-flush with server-side resumption (sequence numbers / last-event-id) or accept a gap and resync state. Assuming the socket is always up is the most common bug.
Q7. What are the trade-offs between the raw `ws` library and Socket.IO?
`ws` is a thin, fast, standards-compliant WebSocket implementation — you get raw frames and must build reconnection, heartbeats, rooms, and any fallback yourself, which is ideal when you want control and minimal overhead. Socket.IO is a higher-level framework adding automatic reconnection, rooms/namespaces, acknowledgements, a Redis adapter for scaling, and an HTTP long-polling fallback, at the cost of a custom (non-WebSocket) wire protocol requiring its client. Pick `ws` for lean, custom protocols or non-browser clients; pick Socket.IO when its batteries-included features would otherwise be reinvented.
Q8. How do you authenticate and authorize a WebSocket connection?
Authenticate at connect time — pass a token (e.g., JWT) via a query param, subprotocol header, or an initial message, and validate it in the server's `connection`/`upgrade` handler before accepting; reject with a close code if invalid. Since the connection is long-lived, also handle token expiry by re-validating periodically or forcing reconnect, and enforce per-message authorization for room joins/actions rather than trusting the initial auth forever. Never rely on cookies alone cross-origin, and validate origin to mitigate cross-site WebSocket hijacking.
