Monitoring, Logging and Newrelic

Cohort 2.0 · 14 lessons · SDE-2 (3+ yrs) interview prep

Concept summary — all 14 steps

Observability pillars: logging, monitoring, alerts, status pages
Logging = discrete timestamped event records, monitoring = continuous metric collection + visualization, alerts = automated notification on threshold breach, status pages = external communication of system health during incidents. Interview angle: articulate the three pillars (logs, metrics, traces), why metrics are cheap-to-aggregate while logs are high-cardinality/expensive, and how status pages cut support load and preserve trust during outages.
Services as the unit of monitoring
A service is a logically deployable unit (microservice/API) that you monitor as a first-class entity with its own golden signals (latency, traffic, errors, saturation). Observability is organized around service boundaries because SLIs/SLOs, ownership, and on-call rotations are defined per service, and cross-service dependencies drive distributed tracing.
New Relic platform overview
A SaaS observability platform that unifies APM, infrastructure, logs, browser, and synthetics on a single telemetry store (NRDB) queried by NRQL. Language agents auto-instrument apps and ship telemetry with minimal code. Interview angle: know agent-based instrumentation vs OpenTelemetry (vendor-neutral), and the consumption pricing model (per-GB ingest + per-user seats) which makes log/metric cardinality a real cost lever.
Entities and the entity model
Everything New Relic observes — application, host, container, DB, queue, Lambda — is an entity with a unique GUID, tags, a health status, and relationships to other entities. The entity-centric model is what lets you pivot from a slow service to the host it runs on to the DB it calls. Interview angle: a consistent tagging strategy (team, env, service) is what makes filtering, grouping, and access control tractable at scale.
Dashboards
Custom, NRQL-backed views combining time-series charts, billboards, and tables across many entities for operational and SLO reporting. They matter for shared situational awareness and for executive/SLO summaries; best practice is dashboards-as-code (Terraform) so they are version-controlled and reproducible rather than hand-built and drift-prone.
Alerts (conditions, policies, workflows)
An alert condition is a NRQL threshold evaluated over a window; conditions group into policies, and workflows route incidents to notifiers. Key knobs are static vs baseline (anomaly) thresholds, evaluation window, and a for-duration to prevent flapping. Interview angle: prefer symptom-based alerts (user-facing SLO breach) over cause-based ones, and tune thresholds/aggregation to fight alert fatigue.
NRQL query language
A SQL-like language over NRDB: SELECT ... FROM EventType WHERE ... FACET ... SINCE ... TIMESERIES. It powers both dashboards and alert conditions. Interview angle: know FACET (group-by), percentile()/rate()/latest(), and that TIMESERIES buckets data into time windows — the same query drives a chart and the alert that watches it.
APM (Application Performance Monitoring)
Per-service transaction traces, throughput, error rate, Apdex, database and external-call timing, and distributed tracing across service hops. It answers 'is this service healthy and where is the time going.' Interview angle: explain Apdex (satisfied/tolerating/frustrated ratio against a target T) and how distributed tracing propagates a trace/span context via headers (e.g. W3C traceparent) to stitch a request across services.
Winston structured logging (Node.js)
A Node logging library with configurable levels, multiple transports (console, file, HTTP/log-shipper), and JSON output. It matters because machine-parseable structured logs with correlation/trace IDs are what make logs searchable and linkable to traces. Interview angle: use JSON not plaintext, attach request/trace IDs, pick levels deliberately, and avoid synchronous file I/O that blocks the event loop under load.
Metrics derived from logs
Log-based metrics aggregate log data into counts/rates (e.g. error-level events per minute via NRQL FACET/rate) to bridge logging and monitoring. Interview angle: it is convenient but ties monitoring cost to log ingest volume and can explode cardinality — for hot, high-frequency signals prefer emitting a purpose-built metric directly rather than reconstructing it from logs.
p95 / p99 tail latency
The value under which 95%/99% of requests complete — the metric that captures worst-case user experience the mean hides. Interview angle: you alert on p99, not average, because a healthy mean can mask a painful tail; and percentiles are NOT averageable across hosts or time buckets, so you must compute them over the raw distribution (e.g. histogram) rather than averaging per-host p99s.
Infrastructure tab
Host, container, and cloud-integration monitoring: CPU, memory, disk, network, and process-level metrics per host. Its value is correlation — linking an app-level slowdown seen in APM to resource saturation (CPU throttling, memory pressure, disk I/O) on the underlying host. Interview angle: this is how you distinguish a code regression from an infra/capacity problem.
Status pages and notifiers
Status pages communicate incident state to users/stakeholders (public or internal), while notifiers are the delivery channels — Slack, PagerDuty, email, webhook — wired to alert workflows. Together they close the loop from detection to human response to external communication. Interview angle: route by severity, dedupe/group incidents to avoid paging storms, and keep the status page authoritative during an outage.
Triage workflow: debugging a real problem
The capstone that ties the stack together: alert fires on an SLI breach, you open the dashboard to scope blast radius, drill into an APM distributed trace to find the slow hop, read correlated structured logs by trace ID for the root cause, and check the Infrastructure tab to rule in/out host saturation. Interview angle: be able to narrate this alert-to-root-cause path fluently — it is the practical point of all the preceding tooling.

Real-world example

Checkout service p99 latency spike during a flash sale

You own a Node.js checkout API instrumented with New Relic APM and Winston JSON logs (each line carrying the distributed-trace ID). During a flash sale, a baseline-anomaly alert on checkout p99 latency fires and routes to PagerDuty via a notifier workflow. You open the team dashboard (NRQL: SELECT percentile(duration, 95, 99) FROM Transaction WHERE appName = 'checkout' FACET name TIMESERIES) and see p99 jumped from 300ms to 4s while the mean barely moved — a classic tail-latency problem. Drilling into an APM distributed trace shows the time is spent in a downstream inventory service call, and following the trace ID into the correlated logs reveals lock contention on a hot SKU row. The Infrastructure tab confirms the inventory host is CPU-saturated, so it is a capacity + contention issue, not a checkout code regression. You scale the inventory tier, post an update to the public status page, and afterward add a log-derived error-rate metric plus a tighter p99 SLO alert so the next occurrence is caught earlier.

Interview Q&A (SDE-2)

Q1. What are the three pillars of observability, and when do you reach for each?
Metrics are cheap, pre-aggregated numeric time series ideal for dashboards, alerting, and trend/capacity analysis. Logs are high-cardinality discrete events you use for detailed forensic root-cause once you know roughly where to look. Traces stitch a single request across services to localize which hop is slow or failing. In practice you alert on metrics, use a trace to find the culprit service, then read that service's logs (linked by trace ID) for the exact cause.
Q2. Why alert on p99 rather than average latency, and why can't you average p99 across hosts?
The mean is dominated by the bulk of fast requests and hides the painful tail that real users hit, so a healthy average can coexist with many users experiencing multi-second latency; p99 surfaces that worst-case. You cannot average per-host or per-bucket p99 values because percentiles are not linear/additive — the 99th percentile of a combined distribution is not the mean of the parts' 99th percentiles. Correct aggregation requires the underlying distribution, typically merging histograms and then computing the percentile.
Q3. How does distributed tracing propagate context across services?
On the first request a trace ID and a root span ID are generated; each service creates a child span and injects trace context into outbound calls via headers (commonly the W3C traceparent header, or a vendor format). Downstream services extract that context, continue the same trace, and report spans back to the backend, which reassembles them by parent-child relationship into one waterfall. Sampling decisions are also carried in the header so the whole trace is kept or dropped consistently.
Q4. Static vs baseline (anomaly) alert thresholds — trade-offs, and how do you reduce alert fatigue?
Static thresholds are simple and predictable but need constant tuning and fail for workloads with strong daily/weekly seasonality. Baseline/anomaly thresholds adapt to expected patterns and catch relative deviations, but can be noisy during legitimate traffic shifts and are harder to reason about. To cut fatigue, alert on user-facing symptoms (SLO breach) rather than every cause, add a for-duration so transient blips don't page, group related conditions into one incident, and route by severity so only actionable, urgent issues page a human.
Q5. What is Apdex and how is it computed?
Apdex is a normalized 0-1 satisfaction score built around a target response time T. Requests are bucketed as satisfied (<= T), tolerating (> T and <= 4T), or frustrated (> 4T or errored), and the score is (satisfied + tolerating/2) / total. It gives a single user-centric health number that is easier to reason about and alert on than a raw latency distribution, though it is a coarse summary and you still need percentiles for tail detail.
Q6. Structured logging with Winston: what practices matter, and when should you derive metrics from logs vs emit them directly?
Emit JSON not plaintext, attach a correlation/trace ID and consistent fields, choose log levels deliberately, and avoid synchronous file writes that block Node's event loop under load. Deriving metrics from logs (e.g. counting error-level events via NRQL) is convenient and needs no new instrumentation, but it couples monitoring cost to log ingest volume and risks cardinality blowups. For hot, high-frequency signals you want in dashboards/alerts, emit a purpose-built metric directly; reserve log-derived metrics for lower-volume or ad-hoc needs.
Q7. Agent-based instrumentation vs OpenTelemetry — how do you choose?
Vendor agents (like New Relic's) auto-instrument common frameworks with near-zero code and deep out-of-the-box detail, at the cost of vendor lock-in. OpenTelemetry is a vendor-neutral standard (SDKs + collector) that lets you instrument once and export anywhere, but often needs more setup and may lag agents on some auto-instrumentation. Choose OTel when you want portability, multi-backend flexibility, or already standardize on it; choose the agent for fastest time-to-value and richest single-vendor experience — many teams run OTel and export to New Relic to get both.
Q8. Walk through how you'd triage a production latency incident using this tooling.
Start from the firing alert to confirm which SLI degraded and its blast radius, then open the dashboard to see whether it's global or scoped to certain endpoints/hosts/regions. Pull an APM distributed trace for a slow request to localize the offending hop (DB, external call, or a specific service), then jump to that service's structured logs by trace ID for the concrete error or slow query. Check the Infrastructure tab to decide whether it's code (regression) or capacity (CPU/memory/disk saturation), apply the fix, communicate via the status page, and afterward add or tighten an alert so it's detected earlier next time.
