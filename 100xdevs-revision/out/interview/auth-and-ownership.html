Ownership And Authorities and PDAs

Cohort 3.0 · 6 lessons · SDE-2 (3+ yrs) interview prep

Concept summary — all 6 steps

Context: Solana's account model
On Solana everything is an account — a blob of bytes with metadata — and the runtime cleanly separates code (stateless, executable program accounts) from state (data accounts a program operates on). This is the opposite of Ethereum's contract-holds-its-own-storage model, and it's what enables Sealevel's parallel execution (transactions declaring non-overlapping accounts run concurrently). Interview angle: understanding that state lives in separate accounts is the prerequisite for every ownership/PDA question.
Accounts: fields and the owner rule
Every account has lamports (balance; 1 SOL = 1e9 lamports), a data byte array, an owner (a program's pubkey), an executable flag, and rent metadata, all keyed by a 32-byte pubkey. The hard runtime invariant is: only the owner program may mutate an account's data or debit its lamports — no one else can, even the fee payer. Accounts must also be rent-exempt (hold a minimum balance for their size) or they get purged, which is a common footgun when sizing accounts.
System Program: the account factory
The System Program (address all-ones, 111...111) owns every ordinary wallet/SOL account and is the only program that can create accounts, allocate their space, assign their ownership to another program, and transfer lamports out of system-owned accounts. To give your program a data account you CPI into System Program's CreateAccount (fund + allocate + assign) — your program can't conjure an account itself because it doesn't own the source lamports or the new address. Interview angle: 'who creates the account and how does ownership get transferred' is a classic setup question.
BPF Loader: how programs are stored and executed
A deployed program is just an account whose data is SBF/eBPF bytecode, marked executable=true, and owned by a BPF loader (today the BPFLoaderUpgradeable program). The loader is what the runtime actually invokes to run your code, and the upgradeable variant splits the program into a Program account plus a ProgramData account holding the bytes. Key point: programs aren't special magic — they're loader-owned executable accounts, and the loader is what governs upgradeability.
Authority: program-enforced privilege vs runtime ownership
An 'authority' is a convention/application-level pubkey (upgrade authority, mint/freeze authority, update authority) that a program's logic — or the loader — requires to be a transaction signer before allowing a privileged action. It is NOT the same as owner: owner is the runtime rule for who can write bytes, while authority is who the program decides can do X. The upgrade authority is the sharpest example: whoever holds it can redeploy new bytecode, and setting it to None makes the program permanently immutable — a security/trust decision teams make deliberately.
Program Derived Addresses (PDAs)
A PDA is an address derived deterministically from seeds + the program ID that is intentionally pushed OFF the Ed25519 curve, so it has no private key and no one can produce a normal signature for it. findProgramAddress hashes the seeds, program ID, and a bump byte, decrementing the bump from 255 until the result is off-curve (the first success is the 'canonical bump'). This gives programs deterministic, discoverable addresses for state (e.g. per-user accounts) AND lets a program 'sign' for its own accounts via invoke_signed by supplying the seeds — the foundation of vaults, escrows, and program-owned token accounts.

Real-world example

A token-staking vault where a PDA custodies user deposits

You're building a staking program. A user deposits SPL tokens and the program derives a PDA from seeds ["vault", user_pubkey, program_id] whose canonical bump you store in the account. During deposit, your program CPIs into the System Program to create the vault data account (fund for rent-exemption, allocate space, assign ownership to your program), then initializes it — now your program is the runtime owner and the only thing that can mutate the balance record. When the user withdraws, the PDA itself must authorize moving tokens out of the program-owned token account, so you call invoke_signed passing the same seeds + bump; the runtime re-derives the PDA, sees it matches, and treats it as a valid signer even though no private key exists. Separately, 'authority' shows up twice: your program logic enforces that only the original depositor's signature can trigger a withdrawal, and at the deployment level your team holds the BPF upgrade authority so you can patch bugs — which you later revoke (set to None) to make the vault immutable and trustless. A real debugging session here usually ends at one of three things: a 'ConstraintSeeds'/wrong-bump error (you didn't use the canonical bump), a rent-exemption failure (account underfunded for its size), or a missing-signer error (forgot to pass seeds to invoke_signed, so the PDA wasn't recognized as signer).

Interview Q&A (SDE-2)

Q1. What's the difference between an account's `owner` and an `authority`?
`owner` is a runtime primitive: it's the program pubkey that is the only entity allowed to mutate the account's data and debit its lamports, enforced by the Solana runtime itself. `authority` is an application/convention concept — a pubkey that a program's own logic (or the loader) requires as a transaction signer before permitting a privileged operation like upgrading, minting, or withdrawing. In short: owner = who can write the bytes; authority = who the program decides is allowed to do a specific action. A single account can be owned by one program yet reference several distinct authorities.
Q2. What is a PDA and why must it be off the Ed25519 curve?
A Program Derived Address is deterministically derived from seeds plus the program ID and is deliberately chosen to lie off the Ed25519 curve, which means no corresponding private key exists. That's the whole point: because no one can hold a key for it, no external party can forge a normal signature for the address, so the program alone can authorize actions on it via invoke_signed. This lets a program own and 'sign' for accounts without managing secrets, giving you secure, deterministic addresses for state and program-controlled funds.
Q3. Explain bump seeds and why the canonical bump matters for security.
Since roughly half of candidate addresses land on the curve, findProgramAddress appends a 1-byte 'bump' and decrements it from 255 until the derived address is off-curve; the first (highest) value that works is the canonical bump. Security matters because if your program accepts any bump the caller supplies, an attacker can grind a different valid bump to derive an alternate PDA for the same logical seeds and bypass your account checks. The fix is to always compute or store the canonical bump and validate against it — Anchor's `bump` constraint does this for you.
Q4. How does a program actually 'sign' for a PDA if it has no private key?
It uses invoke_signed for the cross-program invocation, passing the exact seeds (including the bump) used to derive the PDA. The runtime re-derives the address from those seeds plus the calling program's ID, confirms it matches the PDA in the instruction, and then treats that PDA as an authorized signer for the CPI. This works only for PDAs derived from the caller's own program ID — a program can never sign for a PDA belonging to a different program.
Q5. Why must you go through the System Program to create an account, and what does it do?
The System Program owns all system accounts and SOL and is the only program permitted to create accounts, allocate their data space, assign their ownership to another program, and move lamports out of system-owned accounts. To set up a data account for your program you CPI CreateAccount, which funds it (typically to rent-exemption), allocates bytes, and assigns ownership to your program in one step. Your program can't do this itself because the source lamports live in a system-owned account that only the System Program can debit.
Q6. How are programs stored and executed on Solana, and how do upgrades work?
A program is an account whose data is compiled SBF/eBPF bytecode, flagged executable, and owned by a BPF loader — the loader is what the runtime invokes to run it. With the upgradeable loader, the deployment is split into a Program account and a ProgramData account holding the bytes, plus an upgrade authority. Whoever holds the upgrade authority can redeploy new bytecode; setting that authority to None makes the program permanently immutable, which is the standard way to make a program trustless.
Q7. What's the difference between findProgramAddress and createProgramAddress?
createProgramAddress (create_program_address) takes an exact set of seeds — including a specific bump — and returns the derived address, erroring if the result happens to be on-curve; it does no searching. findProgramAddress wraps it in a loop, decrementing the bump from 255 until it finds an off-curve address, and returns both the address and that canonical bump. You use find off-chain or once to discover the bump, then store it and validate on-chain with the cheaper create call to avoid re-running the search every instruction (it costs compute).
Q8. Where does rent-exemption fit into ownership and PDAs, and what breaks if you get it wrong?
Every account must hold a minimum lamport balance proportional to its data size to be rent-exempt, otherwise the runtime can purge it. When you create a PDA or data account via the System Program you must fund it to that threshold up front, and if you later reallocate to a larger size you must top it up. Getting this wrong shows up as account-creation failures or, worse, silent data loss when an underfunded account is garbage-collected — so account sizing and funding are part of correctly designing any program-owned state.
