Kubernetes Part 2

Cohort 2.0 · 23 lessons · SDE-2 (3+ yrs) interview prep

Concept summary — all 23 steps

Today's agenda
Part 2 moves past raw pods/deployments into how traffic actually reaches your app and how state survives a pod restart: Services, Ingress, Namespaces, ConfigMaps/Secrets, TLS via cert-manager, and Volumes/PersistentVolumes. The through-line is decoupling — networking, config, and storage all get abstracted away from the pod so pods stay stateless and disposable.
Recap: core workloads
Recap of Part 1: Pods are the smallest deployable unit; Deployments manage ReplicaSets which keep N pod replicas running and enable rolling updates/rollbacks. The key mental model to carry forward is that pods are ephemeral and get rescheduled with new IPs, which is precisely why Services and PersistentVolumes need to exist.
Recap: running locally
Local clusters via minikube or kind (Kubernetes-in-Docker) let you run a full control plane + node on your laptop. Interview angle: local clusters don't have a real cloud load balancer, so `type: LoadBalancer` stays Pending unless you use `minikube tunnel` or MetalLB — a common source of confusion when moving between local and cloud.
Running on a cloud provider
Managed offerings (EKS, GKE, AKS) run the control plane for you; you provision node pools (worker VMs) and the cloud controller manager wires `type: LoadBalancer` Services to real cloud LBs and PVCs to cloud disks. The trade-off is you give up control-plane tuning in exchange for not operating etcd/apiserver yourself.
Services
A Service gives a stable virtual IP and DNS name in front of a set of pods selected by labels, load-balancing across them so clients never track individual pod IPs. Types: ClusterIP (internal only, default), NodePort (opens a port on every node), LoadBalancer (provisions a cloud LB). kube-proxy programs iptables/IPVS rules to route the VIP to backing pods.
Downsides of Services
`type: LoadBalancer` gives you one cloud LB (and one public IP + cost) per service, and Services operate at L4 — no host- or path-based routing, no shared TLS termination. Exposing 10 microservices this way means 10 load balancers; this pain is exactly what Ingress solves.
Ingress & Ingress Controller
Ingress is just an API object holding L7 routing rules (host/path → Service). It does nothing on its own — you need an Ingress Controller running in the cluster to actually implement those rules. This separation of the declarative spec from the implementation is a recurring Kubernetes pattern worth naming in interviews.
Ingress Controller
The controller is the real reverse proxy (nginx, Traefik, HAProxy, cloud-native ones) that watches Ingress resources and reconfigures itself to route traffic. It typically sits behind a single LoadBalancer Service, so one LB fans out to many backends — collapsing the per-service-LB cost from the previous step.
Namespaces
Namespaces logically partition one physical cluster into virtual sub-clusters for scoping names, applying RBAC, and enforcing ResourceQuotas/LimitRanges (e.g. dev/staging/prod or per-team). Note that namespaces isolate objects but not the network by default — pods across namespaces can still talk unless you add NetworkPolicies.
Install nginx ingress controller
Install via Helm chart or manifest; it deploys the controller pods plus a LoadBalancer Service that becomes your cluster's single public entrypoint. Interview nuance: the ingress-nginx project (community) is distinct from NGINX Inc.'s controller — same proxy, different codebase and annotations.
Adding routing rules
Create an Ingress resource mapping hosts/paths to backend Services (e.g. `api.example.com/users` → users-svc). `pathType` (Prefix vs Exact) and `ingressClassName` (which controller handles it) are the fields people get wrong; rewrite/annotation behavior is controller-specific.
Trying Traefik
Traefik is an alternative controller with native dynamic config, dashboards, and its own CRDs (IngressRoute, Middleware) beyond the standard Ingress API. The takeaway is that swapping controllers is low-cost if you stick to the standard Ingress object, but you get locked in once you adopt a controller's proprietary CRDs.
Secrets & ConfigMaps
Both externalize configuration out of the container image so the same image runs across environments — the 12-factor 'config in the environment' principle. They can be consumed as environment variables or mounted as files; volume mounts have the advantage of hot-reloading on update, whereas env vars are fixed at pod start.
ConfigMaps
Store non-sensitive key/value config (feature flags, URLs, whole config files) up to ~1MB. Injected via `envFrom`, individual `valueFrom.configMapKeyRef`, or a volume mount. Gotcha: pods do not automatically restart when a ConfigMap changes — env-var consumers need a rollout, and even mounted files can lag by the kubelet sync period.
Secrets
Same shape as ConfigMaps but for sensitive data (passwords, tokens, TLS keys); values are base64-encoded, which is encoding, NOT encryption. Anyone with `get secret` RBAC or etcd access can read them unless you enable encryption-at-rest (EncryptionConfiguration) or an external KMS/Vault provider — a frequent security-interview point.
ConfigMaps vs Secrets
Functionally similar APIs; the real differences are intent and handling — Secrets get base64 encoding, can be encrypted at rest, are held in tmpfs when mounted, and are treated more carefully by RBAC and tooling. Rule of thumb: anything you wouldn't paste into a Slack channel goes in a Secret.
HTTPS via cert-manager
cert-manager is a controller that automates X.509 certificate lifecycle — issuing and auto-renewing certs, typically from Let's Encrypt via the ACME protocol (HTTP-01 or DNS-01 challenge). You define an Issuer/ClusterIssuer, annotate the Ingress, and it provisions a TLS Secret the ingress controller uses to terminate HTTPS. Kills manual cert renewal toil.
Volumes in Docker
Baseline for the k8s discussion: Docker offers bind mounts (host path mapped in) and named volumes (Docker-managed storage), both letting data outlive the container. This matters because a container's writable layer is destroyed on removal — the same problem k8s volumes solve at cluster scale.
Volumes in Kubernetes
A k8s Volume is attached to a Pod (not a single container) and is shareable across containers in that pod, with a lifecycle that depends on the volume type. The abstraction spans dozens of backends (emptyDir, hostPath, configMap, CSI cloud disks) behind one `volumes`/`volumeMounts` interface — CSI being the modern plugin standard.
Ephemeral volumes
emptyDir is created empty when the pod is scheduled and deleted when the pod is removed — great for scratch space, caches, or sharing files between sidecar containers. It survives container crashes/restarts within the pod but NOT rescheduling to another node, so never use it for data you need to keep.
Persistent Volumes
PV/PVC decouples storage from pods: a PersistentVolume is a cluster storage resource, and a PersistentVolumeClaim is a pod's request for storage of a given size/access mode. The pod references the PVC, k8s binds it to a matching PV — so the pod can die and reschedule while the data (and its backing disk) persists independently.
Static Persistent Volumes
An admin manually pre-provisions PV objects pointing at existing storage (an NFS export, a pre-created cloud disk); PVCs then bind to whichever PV matches their capacity/accessModes/storageClass. This is operationally heavy and doesn't scale — you must create disks ahead of demand — which motivates dynamic provisioning.
Dynamic (automatic) PVs
With a StorageClass, creating a PVC triggers a provisioner (CSI driver) to create the backing disk and PV on the fly — no admin pre-work. Key knobs: `reclaimPolicy` (Delete vs Retain — Delete will wipe the cloud disk when the PVC is deleted) and `volumeBindingMode` (Immediate vs WaitForFirstConsumer, which delays binding until a pod is scheduled so the disk lands in the right zone).

Real-world example

Exposing a 3-tier app on EKS with one load balancer, TLS, and a durable database

You're deploying a web app on EKS: a React frontend, a Node API, and Postgres, each in its own Deployment inside a `prod` namespace. Instead of three `type: LoadBalancer` Services (three cloud LBs, three IPs, three bills), you give each a ClusterIP Service and install the nginx ingress controller — it gets the single public ELB. One Ingress routes `app.example.com/` to the frontend and `app.example.com/api` to the API. cert-manager, wired to a Let's Encrypt ClusterIssuer, auto-issues the TLS cert into a Secret and nginx terminates HTTPS. The API's non-secret settings (log level, feature flags) live in a ConfigMap; the Postgres DSN and JWT signing key live in a Secret, both injected as env vars. Postgres is the stateful piece: it uses a PVC against a `gp3` StorageClass, so a CSI-provisioned EBS volume is created automatically and, thanks to `WaitForFirstConsumer`, lands in the same AZ as the pod. When you push a new API image and the pod reschedules to another node, the ClusterIP keeps its stable DNS, the Ingress routing is untouched, and Postgres re-attaches its EBS volume — no data loss. A classic debug during this build: your Ingress returns 404 because `pathType`/`ingressClassName` is wrong, or Postgres data 'disappears' because someone used an emptyDir instead of a PVC, or the PV's `reclaimPolicy: Delete` silently deleted the EBS volume when a PVC was recreated.

Interview Q&A (SDE-2)

Q1. What problem does an Ingress solve that a LoadBalancer Service doesn't, and what actually enforces the rules?
A `type: LoadBalancer` Service is L4 and gives you one cloud load balancer (plus IP and cost) per service, with no host- or path-based routing or shared TLS termination. Ingress is an L7 API object that lets many services share a single entrypoint with host/path routing and centralized TLS. Critically, the Ingress object is inert — an Ingress Controller (nginx, Traefik) running in the cluster watches those objects and configures the actual reverse proxy that enforces them.
Q2. Are Kubernetes Secrets encrypted? How would you actually secure them?
No — by default Secret values are only base64-encoded and stored in plaintext in etcd, so anyone with `get secret` RBAC or etcd access can read them. To genuinely secure them you enable encryption-at-rest via an EncryptionConfiguration (ideally backed by a KMS provider), tighten RBAC so few subjects can read Secrets, and often use an external secret store (Vault, cloud Secrets Manager) synced in via a CSI driver or External Secrets Operator. Base64 is for binary-safe transport, not confidentiality.
Q3. Walk through how a PVC ends up bound to real storage with dynamic provisioning, and the dangerous defaults.
A pod references a PVC; the PVC names a StorageClass. With dynamic provisioning, that StorageClass's provisioner (a CSI driver) creates the backing disk and a matching PV on demand, then binds the PVC to it — no admin pre-provisioning. The two settings that bite people: `reclaimPolicy: Delete` (default for many classes) destroys the cloud disk and its data when the PVC is deleted, so use `Retain` for anything precious; and `volumeBindingMode` should be `WaitForFirstConsumer` so the disk is provisioned in the same zone the pod schedules to, avoiding cross-AZ attach failures.
Q4. If a ConfigMap or Secret changes, do the consuming pods pick it up automatically?
It depends on how it's consumed. Values injected as environment variables are read once at container start and never change until the pod is recreated (a rollout). Values mounted as a volume are updated by the kubelet within its sync period, so mounted files eventually reflect the new content — but only if your app watches/re-reads the file; most apps don't, so you still trigger a rollout. A common pattern is hashing the config into a pod annotation so any change forces a rolling restart.
Q5. What's the difference between emptyDir and a PersistentVolume, and when would you reach for each?
emptyDir is an ephemeral volume created when the pod is scheduled and deleted when the pod is removed; it survives container restarts within the pod but not rescheduling to another node — good for scratch space, caches, or sharing files between sidecars. A PersistentVolume (via PVC) is independent of any pod lifecycle: the pod can die and reschedule while the underlying disk and data persist and re-attach. Use PVs for databases, uploads, and anything that must outlive the pod; use emptyDir for disposable working data.
Q6. Do Namespaces provide security isolation between teams?
Only partially. Namespaces isolate the object/name scope and are the boundary for RBAC roles and ResourceQuotas/LimitRanges, so they're great for organizational and resource isolation. But by default they do NOT isolate the network — pods in one namespace can reach Services and pods in another. For real tenant isolation you add NetworkPolicies (and note that even those need a CNI that enforces them), plus separate ServiceAccounts and quotas; strong isolation often means separate clusters.
Q7. How does cert-manager get and keep a TLS certificate valid, end to end?
You define an Issuer or ClusterIssuer (e.g. a Let's Encrypt ACME issuer). When you create a Certificate — or annotate an Ingress with a TLS block — cert-manager requests a cert via ACME, proving domain ownership through an HTTP-01 challenge (served through the ingress) or DNS-01 challenge (creating a TXT record). On success it stores the signed cert and key in a Kubernetes Secret that the ingress controller uses to terminate HTTPS, and it auto-renews before expiry, eliminating manual cert rotation.
Q8. How does a Service actually route traffic to pods, and how does it stay correct as pods churn?
A Service selects pods by label; the endpoints controller continuously maintains an EndpointSlice listing the IPs of ready matching pods. kube-proxy on each node watches those and programs iptables or IPVS rules so the Service's stable ClusterIP is DNAT'd to a backing pod, load-balancing across them. Because the pod set is reconciled from labels and readiness, pods can be added, removed, or rescheduled with new IPs while the Service IP and DNS name stay constant — which is the whole point of the abstraction.
