Serverless Backends

Cohort 2.0 · 12 lessons · SDE-2 (3+ yrs) interview prep

Concept summary — all 12 steps

Backend servers
A long-running process (Node/Java/Go) bound to a port, listening for requests behind a load balancer on a VM/container that you keep alive 24/7. The interview angle is cost and ops: you pay for idle capacity, own scaling/patching/health-checks, but get full control over the runtime, in-memory state, and long-lived connections.
Serverless backends
Code deployed as stateless functions/handlers that a provider invokes on-demand, auto-scaling from zero to N and billing per request/CPU-ms rather than per uptime hour. It matters because it removes server management and idle cost, but forces a stateless, short-lived, event-driven model — the classic trade-off SDE-2s are asked to reason about.
Serverless providers
AWS Lambda (+ API Gateway) is the incumbent container-based FaaS; Cloudflare Workers, Vercel/Netlify Functions, GCP Cloud Functions/Run, and Azure Functions are the main alternatives. The key distinction to know: Lambda runs a full micro-VM (Firecracker) per function, while Cloudflare Workers run on the lighter V8 isolate model at the edge — this drives the cold-start and pricing differences.
When to go serverless
Good fit for spiky/unpredictable traffic, event-driven glue, cron jobs, low-to-medium steady load, and teams wanting zero ops; poor fit for sustained high throughput (idle-hour billing beats per-request at scale), long-running/CPU-heavy jobs, workloads needing persistent connections or large in-memory caches, and latency-sensitive paths hurt by cold starts. Interviewers want the cost-crossover reasoning, not a blanket 'serverless is better'.
Cloudflare Workers setup
Install Wrangler (the CLI), authenticate against your Cloudflare account, and it becomes your build/dev/deploy tool. Worth knowing that Workers run on the V8 isolate runtime (workerd), not Node — so you get Web-standard APIs (fetch, Request/Response, Web Crypto) rather than the full Node stdlib, though Node compat shims exist via a flag.
How Workers work
Each Worker runs as a V8 isolate rather than a container/VM: thousands of isolates share one process, spun up in ~5ms with near-zero cold start, and executed at Cloudflare's edge POP closest to the user. The interview point is why cold starts are negligible — isolates avoid booting an OS/runtime per invocation — plus the constraints this imposes (CPU-time limits, no filesystem, no raw TCP historically).
Initializing a Worker
`wrangler init` (or `npm create cloudflare`) scaffolds a project with `wrangler.toml`/`wrangler.jsonc` config and an entry module exporting a default `fetch(request, env, ctx)` handler. Know the handler signature: `env` carries bindings (secrets, KV, D1, R2), and `ctx.waitUntil()` lets you run work after the response is returned.
Deploying a Worker
`wrangler dev` runs it locally against the real runtime; `wrangler deploy` pushes it globally to every edge POP in seconds, with instant rollbacks and gradual/versioned rollouts available. The takeaway is deploy latency and reach: one command distributes to 300+ locations, and there's no infra to provision or scale groups to manage.
Adding Express
Express assumes Node's http server and long-lived process, so it doesn't run natively on the isolate runtime; you either use it only on Node-based serverless (Lambda via a serverless-http adapter) or shim it. The interview lesson is recognizing why a framework built around `req`/`res` streams and `listen()` is a poor fit for a fetch-handler, edge environment.
Using Hono
Hono is a tiny, Web-standard router (built on Request/Response) designed for edge/serverless runtimes, giving Express-like ergonomics — routing, params, middleware — that run on Workers, Deno, Bun, Lambda, and Node alike. It matters because it's the idiomatic replacement for Express on Workers: same DX, but portable across runtimes and zero heavy dependencies.
Middlewares
Hono/edge middleware use the onion model: `app.use(async (c, next) => { ...; await next(); ... })`, wrapping the handler to add auth, logging, CORS, and error handling before/after the route runs. Know that ordering matters, `await next()` is what continues the chain, and code after it runs on the way out — the same pattern as Koa, distinct from Express's callback-style `(req,res,next)`.
Connecting to a DB
The hard part in serverless is connection management: many concurrent short-lived invocations exhaust a Postgres/MySQL connection pool, so you use HTTP/serverless drivers (Neon, PlanetScale, Prisma Accelerate, D1) or an external pooler (PgBouncer, RDS Proxy) instead of raw TCP pools. On Workers specifically, prefer HTTP-based drivers or Hyperdrive for pooling+caching, since isolates historically couldn't hold persistent DB connections.

Real-world example

A webhook + API backend for a SaaS billing integration

You're an SDE-2 building the backend that ingests Stripe webhooks and serves a lightweight read API for a SaaS dashboard. Traffic is bursty (quiet most of the day, spikes at billing cycles), so you deploy it as a Cloudflare Worker instead of an always-on Node box: Wrangler scaffolds the project, you write a Hono app with routes for `/webhooks/stripe` and `/api/subscriptions/:id`, and add middleware for signature verification, request logging, and CORS. The Worker runs at the edge with near-zero cold start, and `ctx.waitUntil()` fires off an async audit-log write without delaying the webhook 200. For persistence you hit Postgres on Neon over its HTTP serverless driver (fronted by Hyperdrive for pooling) — deliberately avoiding a raw TCP pool, because hundreds of concurrent isolate invocations would otherwise blow past Postgres's connection limit. When you later debug elevated p99 latency, the root cause isn't cold starts but the DB round-trip from a distant edge POP, which you fix with Hyperdrive connection caching and a nearer read replica — exactly the kind of serverless-specific trade-off this topic is about.

Interview Q&A (SDE-2)

Q1. What actually causes cold starts, and why are Cloudflare Workers' cold starts so much smaller than Lambda's?
A cold start is the latency of provisioning execution context for a new instance before your code runs. Lambda boots a Firecracker micro-VM and initializes a language runtime per instance (tens to hundreds of ms, worse for JVM/.NET), whereas Workers use V8 isolates — lightweight sandboxes within a shared process — so there's no OS/runtime boot, giving ~5ms starts. The trade-off is that isolates impose tighter limits (CPU time, memory, no arbitrary native binaries) and a Web-standard rather than full-Node runtime.
Q2. Why is database connectivity a known pain point in serverless, and how do you solve it?
Traditional DBs like Postgres cap concurrent connections (often ~100s), and each connection is expensive; serverless scales to many concurrent short-lived instances that each want a connection, so you quickly exhaust the pool and see connection errors under load. Fixes: put a pooler in front (PgBouncer, RDS Proxy) so DB connections are reused across invocations, or use HTTP-based serverless drivers (Neon, PlanetScale, Prisma Accelerate, Cloudflare D1) that avoid persistent TCP. On Workers specifically, Hyperdrive adds pooling plus query caching at the edge.
Q3. When would you deliberately NOT choose serverless for a backend?
Sustained high, predictable throughput — per-request/CPU-ms billing eventually costs more than a reserved VM/container running near full utilization, so there's a cost crossover point. Also avoid it for long-running or CPU-heavy jobs (execution time limits), workloads needing persistent connections (WebSockets, long-poll) or large in-memory state/caches, and ultra-low-latency paths where cold starts or vendor cold-path variance hurt. In those cases a container platform (ECS/K8s/Cloud Run with min instances) is usually better.
Q4. Why doesn't Express run natively on Cloudflare Workers, and what do you use instead?
Express is built on Node's `http` module — it expects a long-lived process, `server.listen()`, and Node `req`/`res` stream objects — none of which exist in the V8 isolate runtime, which instead invokes a `fetch(request)` handler with Web-standard Request/Response. So you use Hono (or itty-router), which is built directly on the Fetch API and gives Express-like routing/middleware while staying portable across Workers, Deno, Bun, Node, and Lambda. On Node-based serverless like Lambda you can still run Express via an adapter such as `serverless-http`.
Q5. Explain the onion/middleware model in Hono and how it differs from Express middleware.
Hono middleware is `async (c, next) => {}`: code before `await next()` runs on the way in, `next()` invokes the rest of the chain, and code after it runs on the way out — so a single middleware can both set up and tear down (timing, error wrapping, response headers). Express uses `(req, res, next)` where `next()` just passes control forward and post-processing is awkward. The onion model (shared with Koa) makes ordering explicit and makes cross-cutting concerns like error handling and logging cleaner because you wrap the downstream handler.
Q6. What is the significance of the `fetch(request, env, ctx)` handler signature on Workers, especially `env` and `ctx`?
`request` is a standard Fetch Request and you return a Response — that's the whole contract, which is what makes the code portable. `env` holds bindings injected at runtime: secrets, environment vars, and resource handles like KV, D1, R2, and Durable Objects — you never read them from a global `process.env` at import time. `ctx.waitUntil(promise)` lets you extend the invocation to finish background work (logging, cache writes) after the response is sent, and `ctx.passThroughOnException()` controls failure behavior.
Q7. How does Cloudflare's edge execution model affect where your code runs and what latency you should expect?
Workers execute at the Cloudflare POP nearest the user, so static/compute-only responses are very low latency globally without you managing regions. But if your Worker calls a database or origin in a single region, requests from far-away POPs pay the round-trip to that region — so edge compute doesn't magically make a centralized DB fast. You mitigate with edge caching, read replicas / distributed data (D1, KV, Durable Objects placed near users), or Hyperdrive connection caching, and you keep latency-critical state close to compute.
Q8. How do you handle secrets, config, and environments across `wrangler dev` and `wrangler deploy`?
Non-sensitive config and bindings go in `wrangler.toml`/`jsonc` (including per-environment sections), while secrets are set via `wrangler secret put` and stored encrypted, surfaced only through the `env` argument at runtime — never committed. Locally, `wrangler dev` runs the real workerd runtime with a `.dev.vars` file for local secrets, so dev parity is high. Deploys are versioned with instant rollback and support gradual/percentage rollouts, so you promote a build rather than mutating a running server.
