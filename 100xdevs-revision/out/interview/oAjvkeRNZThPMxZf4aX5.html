PayTM project

Cohort 2.0 · 21 lessons · SDE-2 (3+ yrs) interview prep

Concept summary — all 21 steps

Clone the repo / project scaffold
Start from the boilerplate mono-structure (separate backend Express app and frontend React app, plus package.json per side). The interview angle is recognizing a full-stack project layout and being able to bootstrap dependencies, env config, and the MongoDB connection string quickly.
User Mongoose schema
Define the User model (firstName, lastName, username/email, password) with validation, unique index on username, and trim/lowercase normalization. Matters because schema design (indexes, required fields, field constraints) is your first line of data integrity and query performance in MongoDB.
Create routers
Modularize routes with express.Router() instead of hanging everything off the app, e.g. a user router file exporting a router object. The point is separation of concerns and scalability — routers keep route definitions, middleware, and handlers grouped by domain.
Route user requests (mounting)
Mount routers under a versioned prefix like app.use('/api/v1/user', userRouter) in the entry file. API versioning and a consistent base path matter for backward compatibility and clean client integration.
External libraries
Pull in the core stack: express, mongoose, jsonwebtoken (JWT), bcrypt (hashing), zod (input validation), and cors. Knowing what each solves — and that you never hand-roll auth/crypto — is a common interview signal of production maturity.
Backend auth routes (signup/signin)
Implement /signup (validate with zod, hash password with bcrypt, create user, issue JWT) and /signin (verify credentials, return JWT). The interview depth is on password hashing (salt + cost factor), never storing plaintext, and returning a signed token rather than a session.
Auth middleware
Write middleware that reads the Authorization: Bearer <token> header, verifies the JWT with the secret, and attaches userId to req before protected handlers run. Middleware is the reusable gate for every authenticated route — expect questions on where it sits in the chain and how it short-circuits with 401/403.
User routes (update + search)
Add authenticated endpoints: update profile (PUT, zod-validated partial fields, re-hash password if changed) and a bulk user search using a filter with $regex/$or on first/last name. Tests your grasp of query building and protecting mutations behind auth.
Account schema
Introduce a separate Account collection holding userId (ObjectId ref to User) and balance. Splitting balance out of User is a deliberate normalization choice — it isolates the frequently-mutated, transactionally-sensitive money field from mostly-static profile data.
Transactions (atomic transfer)
Implement money transfer inside a Mongoose session/transaction: debit sender and credit receiver as one atomic unit, aborting on insufficient funds or error. This is the marquee interview topic — MongoDB multi-document transactions require a replica set, and the transaction prevents partial transfers under failure.
Initialise balances
Seed each new user with a random starting balance on signup (create the Account doc alongside the User). Simple, but it teaches you to keep the User and Account creation consistent — ideally the pair should also be atomic.
Create a new router (account)
Add a dedicated account router mounted at /api/v1/account, separate from the user router. Reinforces domain-based routing — money operations live apart from identity operations.
New endpoints (/balance, /transfer)
Expose GET /balance (return the authenticated user's balance) and POST /transfer (amount + recipient, runs the transaction). These are the business-critical endpoints; expect questions on validating amount > 0, checking sufficient funds, and idempotency.
Checkpoint (backend review)
Consolidate and smoke-test the backend end to end: signup, signin, protected balance read, and a transfer. The value is verifying the auth-to-transaction flow works as a whole before pivoting to the frontend.
Routing in React
Set up react-router-dom with BrowserRouter and a Routes tree for /signup, /signin, and /dashboard. Client-side routing matters for SPA navigation without full reloads and sets up where protected/redirect logic will live.
Writing reusable components
Build primitives — Button, InputBox, Heading, SubHeading, BottomWarning — as props-driven, composable pieces. The interview angle is component reuse and a lightweight design system rather than duplicated markup per page.
Signin page
Compose the Signin page from the primitives, manage controlled inputs with useState, and call the backend signin on submit. Demonstrates assembling a screen from components plus local form state handling.
Main component (Dashboard)
Build the post-login Dashboard: an AppBar, a Balance display fed by GET /balance, and a Users list with search. This is the app's home screen and shows how you fetch and render authenticated data.
Auth components
Group the shared building blocks used by both Signup and Signin (input boxes, headings, warning links). Keeps the two auth screens visually and behaviorally consistent through shared components.
Signed-in components
Build the components only visible after auth — Users list, Send-Money view, balance card. These consume the token and hit protected endpoints, tying UI state to authentication.
Wiring it all up
Connect frontend to backend with axios: send the JWT in the Authorization header, persist the token (localStorage), redirect on auth, and execute a real end-to-end transfer. The integration step is where CORS, token storage trade-offs, and error handling all surface.

Real-world example

Building and debugging the money-transfer flow in a Paytm-style wallet

You're an SDE-2 shipping the peer-to-peer transfer feature. A user signs in, the React app stores the JWT and calls GET /api/v1/account/balance with a Bearer header; the auth middleware verifies the token and sets req.userId. On the Send Money screen the user picks a recipient and an amount, and the client POSTs to /transfer. On the server you open a Mongoose session, start a transaction, re-read the sender's account inside the session, reject if balance < amount, then $inc the sender down and the recipient up and commit — aborting on any error so you never leave money debited-but-not-credited. During load testing you notice two concurrent transfers from the same account can both pass the balance check and overdraw it; you fix it by moving the check-and-decrement into the atomic update (a conditional $inc guarded by balance >= amount) inside the transaction, and confirm your MongoDB is a replica set so multi-document transactions are even supported. This one feature exercises the whole stack: Mongoose schemas, JWT auth middleware, zod validation of the amount, transactional writes, and the axios-wired React UI.

Interview Q&A (SDE-2)

Q1. Why store the balance in a separate Account collection instead of a field on the User document?
It separates a hot, transactionally-sensitive, frequently-mutated value (money) from mostly-static profile data, which keeps write contention and document churn off the User document. It also lets the account/money domain evolve independently — you can add fields like currency, frozen status, or per-account indexes without bloating User. In interviews, frame it as normalization plus isolating the field that participates in transactions.
Q2. MongoDB is often called schemaless — how do multi-document transactions work here and what do they require?
Mongoose transactions use a client session: you startSession, startTransaction, run all reads/writes with { session }, then commitTransaction or abortTransaction on error. They give ACID semantics across multiple documents, so a debit and credit either both apply or neither does. Critically, multi-document transactions require MongoDB to run as a replica set (even a single-node replica set locally) — they do not work on a standalone mongod.
Q3. A naive transfer reads the balance, checks it, then decrements. What's the race condition and how do you fix it?
Two concurrent transfers can both read the same balance, both pass the check, and both decrement — overdrawing the account (a classic check-then-act TOCTOU bug). The fix is to make the check and decrement atomic: use a conditional update like updateOne({ _id, balance: { $gte: amount } }, { $inc: { balance: -amount } }) and treat a modifiedCount of 0 as insufficient funds. Running this inside the transaction gives both atomicity of the single account update and all-or-nothing across both accounts.
Q4. Why bcrypt for passwords, and what's the trade-off in the cost factor?
bcrypt is a deliberately slow, salted, adaptive hash designed to resist brute-force and rainbow-table attacks — unlike fast hashes like SHA-256 which are unsuitable for passwords. The per-password salt defeats precomputation, and the cost factor (rounds) lets you scale difficulty as hardware improves. The trade-off: higher cost hardens against attackers but adds latency to every signin, so you tune it to a few hundred milliseconds rather than maxing it out.
Q5. JWT vs server-side sessions — why does this project use JWT and what are the downsides?
JWTs are stateless: the server signs a token, the client sends it on each request, and the server just verifies the signature without a session store — which scales horizontally with no shared session state. The downsides are revocation and expiry: you can't easily invalidate a valid unexpired token, so logout/ban needs short TTLs plus refresh tokens or a denylist. You also can't store sensitive data in the payload since it's only signed, not encrypted, and readable by anyone.
Q6. The frontend stores the JWT in localStorage. What's the security trade-off versus httpOnly cookies?
localStorage is simple and immune to CSRF (nothing is auto-sent), but it's readable by any JavaScript, so an XSS vulnerability leaks the token directly. httpOnly cookies are invisible to JS (mitigating XSS token theft) but are sent automatically, so they need CSRF protection (SameSite, CSRF tokens). For a real wallet you'd lean toward httpOnly cookies with SameSite plus strict XSS hygiene; localStorage is acceptable for a learning project but worth calling out as a weakness.
Q7. Why validate inputs with zod on the backend when the React form already validates?
Client-side validation is purely UX — it can be bypassed by hitting the API directly with curl or Postman, so it is never a security boundary. Zod on the server enforces the real contract: types, required fields, string lengths, and amount > 0, rejecting malformed requests with a 400 before they touch business logic or the database. The rule is validate on the client for feedback, validate on the server for trust.
Q8. Walk through what the auth middleware does and where it sits in the request lifecycle.
It runs before the protected route handlers: it reads the Authorization header, extracts the Bearer token, and calls jwt.verify with the secret. On success it attaches the decoded userId to req and calls next() to pass control down the chain; on a missing, malformed, or invalid/expired token it short-circuits with a 401/403 and never reaches the handler. Mounting it selectively (e.g., on the account router but not on signup/signin) is what distinguishes public from protected endpoints.
