Kubernetes Part 3 (Scaling)

Cohort 2.0 · 9 lessons · SDE-2 (3+ yrs) interview prep

Concept summary — all 9 steps

Scope: what scaling in K8s means
Overview of the three scaling axes you get in Kubernetes: horizontal pod scaling (more replicas), vertical scaling (bigger pods), and node/cluster scaling (more machines). The interview angle is knowing these are distinct control loops that interact — HPA adds pods, but pods only schedule if the Cluster Autoscaler has added nodes.
Horizontal Pod Autoscaler (HPA)
The step title 'Horizontal pod accelerator' means the Horizontal Pod Autoscaler — a controller that watches a metric (CPU/memory or custom/external via the metrics API) and adjusts a Deployment/ReplicaSet's replica count. It requires metrics-server (or a custom metrics adapter) to be installed; without a metrics pipeline the HPA reports <unknown> and does nothing.
Sample app for the lab
A deliberately CPU-burnable demo app (e.g. a small HTTP server that does busy work per request, like the classic php-apache image) used to trigger autoscaling under load. Point of choosing this app: it makes CPU utilization move predictably so you can watch the HPA react in real time.
Authoring Deployment + Service manifests
Writing the YAML for a Deployment (pod template, replicas, container image) and a Service to expose it, which are the objects the HPA scales and load-balances across. Interview relevance: the HPA scales the Deployment's replica count, and the Service's endpoints update automatically as pods come and go — you don't manually wire new pods into load balancing.
Triggering a scale-up
Driving load (e.g. a busybox wget loop) so measured CPU exceeds the target, then watching replicas increase via `kubectl get hpa` / `kubectl get pods`. Key nuance: scale-up is fast but bounded by stabilization windows and pod readiness — new pods must pass readiness probes before they take traffic, so effective capacity lags the replica count.
The HPA desired-replicas formula
desiredReplicas = ceil(currentReplicas * (currentMetricValue / desiredMetricValue)), clamped to [minReplicas, maxReplicas] and with a small tolerance (default 10%) so it doesn't flap on tiny deltas. Interviewers probe this: e.g. 3 replicas at 90% CPU with a 50% target → ceil(3 * 90/50) = ceil(5.4) = 6 replicas. Requests are the denominator for utilization, so wrong requests break the math.
Resource requests vs limits
Requests are what the scheduler reserves and what CPU-utilization-based HPA is measured against; limits are the hard ceiling (memory over-limit → OOMKill, CPU over-limit → throttling). The trade-off worth articulating: setting no requests makes HPA CPU% meaningless and lets the scheduler overcommit nodes, while limits set too low cause silent throttling/OOM that looks like an app bug.
Cluster Autoscaler
A separate controller that adds/removes nodes when pods are Pending due to insufficient capacity (scale-up) or when nodes are underutilized and their pods can be rescheduled (scale-down). It operates on the infra layer via the cloud provider's node groups, and is the necessary counterpart to HPA — HPA can request pods that will sit Pending forever unless the Cluster Autoscaler provisions nodes for them.
Hands-on scaling lab
End-to-end exercise: deploy the app, install metrics-server, create an HPA, generate load, and observe pods and nodes scale, then scale back down. The takeaway is seeing the full loop — metric → HPA → new pods → (if needed) Cluster Autoscaler → new nodes — and the delays at each hop.

Real-world example

Autoscaling a checkout service for a flash sale

You run a stateless checkout API on EKS. Baseline traffic needs 3 pods, but marketing schedules a flash sale expecting a 10x spike. You set CPU requests to 250m and a limit to 500m per pod, then create an HPA targeting 60% CPU utilization with minReplicas=3, maxReplicas=40. When the sale starts, CPU jumps to ~95%; the HPA applies ceil(3 * 95/60) = 5, then keeps stepping up each cycle until pods stall in Pending because the three m5.large nodes are full. The Cluster Autoscaler sees the Pending pods and adds nodes from the node group, the scheduler places the new pods, readiness probes pass, and the Service's endpoints pick them up. The debugging this exercises: why HPA said <unknown> before metrics-server was healthy, why replicas plateaued (node capacity, not the HPA), and why the 500m limit caused throttled tail latency under burst even though average CPU looked fine. After the sale, scale-down is gentler — HPA respects the stabilization window (default 5 min) before removing pods, and the Cluster Autoscaler only drains a node once its pods fit elsewhere.

Interview Q&A (SDE-2)

Q1. How does the HPA decide how many replicas to run, and what's the exact formula?
It uses desiredReplicas = ceil(currentReplicas * (currentMetricValue / desiredMetricValue)), clamped to [minReplicas, maxReplicas]. A tolerance (default 10%) prevents flapping when the ratio is close to 1, and stabilization windows smooth out scale-down. For CPU-utilization targets the denominator is the sum of container CPU requests, which is why the HPA silently misbehaves if requests aren't set.
Q2. What's the difference between resource requests and limits, and how does each affect scheduling and runtime?
Requests are the amount the scheduler reserves on a node and are used for bin-packing and for HPA CPU-utilization math; a pod won't schedule unless a node has its requests free. Limits are the runtime ceiling — exceeding a memory limit triggers an OOMKill, while exceeding a CPU limit causes CFS throttling rather than termination. A common production mistake is a large gap between requests and limits, which lets nodes get overcommitted and causes noisy-neighbor throttling under load.
Q3. HPA and Cluster Autoscaler — how do they relate, and what happens if you only have one?
HPA scales pods (application layer); the Cluster Autoscaler scales nodes (infra layer) by reacting to Pending pods and underutilized nodes. They're complementary control loops: HPA can keep raising the replica count, but those extra pods sit Pending forever if there's no node capacity and no Cluster Autoscaler. Conversely, a Cluster Autoscaler with no HPA just right-sizes nodes for a fixed pod count. You typically want both, plus sane resource requests so the autoscaler can reason about capacity.
Q4. Your HPA shows TARGET as <unknown> and never scales. How do you debug it?
<unknown> means the HPA can't read the metric. First check that metrics-server (or your custom/external metrics adapter) is installed and healthy — `kubectl top pods` should return data. Then confirm the pods have CPU requests defined, since utilization is a percentage of requests and is undefined without them. Also check the HPA's target references the right Deployment and that RBAC allows the controller to read the metrics API.
Q5. Why can effective capacity lag behind the replica count during a scale-up, and how do you mitigate it?
A new pod counts as a replica immediately but doesn't receive traffic until it passes its readiness probe, and the pod itself may wait on image pull, node provisioning (via Cluster Autoscaler), and app warm-up. So there's a multi-stage delay: HPA decision → pod scheduled → node added if needed → readiness. Mitigations include pre-pulling images, tuning readiness probes, keeping a warm buffer via minReplicas, using overprovisioning (pause pods) so nodes are ready ahead of demand, and setting startupProbes for slow-booting apps.
Q6. What are the trade-offs between scaling on CPU utilization versus custom/external metrics?
CPU utilization is easy and built in but is a poor proxy for many workloads — an I/O-bound or queue-consuming service can be saturated while CPU stays low. Custom metrics (e.g. requests-per-second, p99 latency) or external metrics (e.g. SQS queue depth, Kafka lag) scale on what actually reflects load, giving tighter SLO control. The cost is operational complexity: you need a metrics adapter (like Prometheus Adapter or KEDA), and badly chosen or noisy metrics can cause aggressive flapping.
Q7. How does the HPA avoid thrashing (rapid scale up/down), and what knobs control it?
It has a built-in tolerance so small metric deviations don't trigger changes, and behavior policies with stabilization windows — by default scale-down waits 300s and considers the max recommendation over that window, while scale-up is more responsive. In the autoscaling/v2 API you can set `behavior.scaleUp` and `behavior.scaleDown` with policies capping how many pods or what percentage can change per period. This lets you make scale-up aggressive for spikes while keeping scale-down conservative to avoid dropping capacity prematurely.
Q8. Can HPA and VPA (Vertical Pod Autoscaler) be used together, and what's the conflict?
They conflict if both act on the same resource metric: VPA changes pod CPU/memory requests, and HPA scales replicas based on CPU utilization relative to those same requests, so they fight and cause instability. The safe pattern is HPA on CPU/custom metrics with VPA in recommendation-only mode, or HPA on a custom metric (like RPS) while VPA manages CPU/memory requests. VPA also historically required pod restarts to apply new requests, which matters for availability.
