Concept summary — all 7 steps

Prerequisites: generics + utility-type mental model
These are all built-in generic utility types (Pick, Partial, Readonly, Record, Exclude) that transform one type into another at compile time only — zero runtime cost. The prereq is being comfortable with generics (<T>), keyof, indexed access (T[K]), union types, and mapped/conditional types, because every utility type is just a thin wrapper over those primitives. Interview angle: expect to be asked to re-implement one from scratch, which tests whether you understand mapped types rather than memorizing names.
Pick<T, K>
Pick constructs a new type by selecting a subset of keys K (a union of literal keys) from T, e.g. Pick<User, 'id' | 'name'>. It matters for building narrow DTOs/view models from a large domain type without duplicating fields, so the derived type stays in sync when the source changes. Interview angle: contrast with Omit (which is the complement, Pick<T, Exclude<keyof T, K>>) and know K is constrained by keyof T so invalid keys are a compile error.
Partial<T>
Partial makes every property of T optional (adds `?` to each key via a mapped type with the `?` modifier). It matters for PATCH/update payloads, config objects with defaults, and builder patterns where not all fields are supplied. Interview angle: it's shallow (nested objects stay required) so a DeepPartial needs a recursive conditional type; also know its inverse is Required<T> which strips optionality with `-?`.
Readonly<T>
Readonly marks every property immutable at the type level (`readonly` modifier via mapped type), preventing reassignment after construction. It matters for enforcing immutability contracts on shared state, Redux-style stores, and function params you don't want mutated. Interview angle: stress it is compile-time only — nothing freezes at runtime (use Object.freeze for that) — it's shallow, and `as const` produces deep readonly plus literal narrowing which Readonly<T> does not.
Record<K, V>
Record<K, V> builds an object type whose keys are K (typically a union of string/number literals or a broad string/number) and whose values are all V, e.g. Record<'admin'|'user', Permissions>. It matters for typing dictionaries, lookup maps, and exhaustive config keyed by a union. Interview angle: when K is a finite literal union, Record enforces every key is present (exhaustiveness); when K is `string`, it's an index signature with no such guarantee — know that trade-off.
Exclude<T, U>
Exclude is a conditional type that removes from union T all members assignable to U (T extends U ? never : T, distributed over the union). It matters for narrowing unions — e.g. dropping 'loading' from a status union or removing null. Interview angle: it operates on union members not object keys (that's Omit), its counterpart is Extract<T, U> which keeps the matching members, and the distributive `never` collapse is what makes it work.
Type inference in zod (z.infer)
z.infer<typeof schema> extracts the static TypeScript type from a zod runtime schema, giving you a single source of truth: define the schema once and derive the type instead of hand-writing a parallel interface that can drift. It matters because zod validates at runtime (API boundaries, form input, env vars) while z.infer keeps the compile-time type in lockstep. Interview angle: know z.infer vs z.input/z.output (they differ when transforms/defaults/coercion change the parsed shape), and that this is the reverse direction of the utility types — deriving a type from a value rather than a type from a type.

Real-world example

Building a typed user-settings PATCH endpoint

You own a User service. The canonical domain type is a large `User` interface (id, email, passwordHash, role, profile, createdAt...). For the public API you derive a `PublicUser = Omit<User, 'passwordHash'>` and a settings view with `Pick<User, 'email' | 'role' | 'profile'>` so the DTO can never accidentally leak the hash. The PATCH handler accepts `Partial<Pick<User, 'email' | 'profile'>>` so clients send only the fields they're changing. Role-based access is a `Record<Role, Permission[]>` map, where `Role` is a literal union — so adding a new role produces a compile error until you fill in its permissions (built-in exhaustiveness). The request status union is narrowed with `Exclude<Status, 'internal-error'>` before it's serialized to the client. Crucially, the request body isn't trusted, so you define a zod schema `updateUserSchema` and use `type UpdateUserBody = z.infer<typeof updateUserSchema>` as the handler's param type: zod `.parse()` validates at runtime and z.infer guarantees the compile-time type matches the validator exactly, so the two can't drift. Once frozen, the resolved settings object is passed around as `Readonly<Settings>` to guarantee downstream code doesn't mutate shared state.

Interview Q&A (SDE-2)

Q1. How would you implement Pick<T, K> and Partial<T> from scratch?
Both are one-line mapped types. `type MyPick<T, K extends keyof T> = { [P in K]: T[P] }` iterates the selected keys and copies their value types via indexed access. `type MyPartial<T> = { [P in keyof T]?: T[P] }` maps over all keys adding the optional `?` modifier. The K constraint `extends keyof T` is what makes invalid keys a compile error, and being able to write these shows you understand mapped types rather than memorizing utility names.
Q2. Pick/Omit vs Exclude/Extract — what's the difference and when do you use each?
Pick and Omit operate on the *keys* of an object type (Omit<T,K> = Pick<T, Exclude<keyof T, K>>). Exclude and Extract operate on the *members of a union*. So you Omit a field from an interface, but you Exclude a variant from a status union like `Exclude<'idle'|'loading'|'error', 'error'>`. Mixing them up is a common mistake; the giveaway is whether you're reshaping an object or filtering a union.
Q3. Readonly<T> is compile-time only — what does that mean in practice and how do you get real immutability?
Readonly<T> and the `readonly` modifier are erased at compile time; the emitted JavaScript has no protection, so a value cast or `any` can still mutate the object at runtime. It's also shallow — nested objects remain mutable. For runtime immutability use Object.freeze (also shallow), and for deep compile-time immutability plus literal narrowing use `as const`, which is strictly stronger than Readonly<T>.
Q4. When does Record<K, V> guarantee all keys exist and when doesn't it?
When K is a finite literal union (e.g. `Record<'a'|'b', number>`), TypeScript requires every member key to be present, giving you free exhaustiveness — add a union member and the object errors until updated. When K is a wide type like `string` or `number`, Record degrades to an index signature with no presence guarantee, and accessing a key returns V even if it's actually undefined (unless noUncheckedIndexedAccess is on). Prefer literal-union keys when you want the compiler to enforce completeness.
Q5. What is z.infer and why prefer it over writing a matching interface by hand?
z.infer<typeof schema> statically extracts the type a zod schema produces after parsing, making the schema the single source of truth. Hand-writing a parallel interface means two definitions that silently drift when one changes; z.infer eliminates that class of bug because the type is derived from the validator. It also bridges runtime and compile time: zod validates untrusted input at the boundary, and z.infer ensures the value you get back is typed exactly as validated.
Q6. When do z.input and z.output differ from each other?
They differ whenever the schema transforms the data — via .transform(), .default(), coercion (z.coerce), or .pipe(). z.input is the type accepted *before* parsing; z.output (what z.infer aliases) is the type produced *after*. For example a field with `.default('x')` is optional on input but required on output, and `z.coerce.number()` accepts unknown/string on input but yields number on output. Using z.infer for a function's input parameter is a bug in those cases — you'd want z.input there.
Q7. How does Exclude actually work under the hood?
Exclude<T, U> is defined as `T extends U ? never : T`. Because conditional types distribute over naked union type parameters, this is evaluated per-member of T: each member that's assignable to U becomes `never`, and `never` is absorbed when the union is reassembled, so those members drop out. That distributive behavior over unions plus the `never` collapse is the whole mechanism — the same pattern underpins NonNullable = Exclude<T, null | undefined>.
Q8. How would you compose these utilities to type a safe update payload that excludes immutable fields?
Layer them: `Partial<Omit<User, 'id' | 'createdAt' | 'passwordHash'>>`. Omit (built on Pick + Exclude) strips the fields clients must never change, then Partial makes the remainder optional so a PATCH can send any subset. This is fully derived from the source User type, so adding a field to User automatically flows through, and dropping an immutable field is a one-line change. It demonstrates that utility types compose cleanly because each just returns another type.
