ETH intro, EVM

Cohort 3.0 · 10 lessons · SDE-2 (3+ yrs) interview prep

Concept summary — all 10 steps

Web3 orientation
Web3 is the shift from Web2's centralized read/write servers to user-owned, decentralized state living on public blockchains. The mental model to keep for interviews: Web1 = read, Web2 = read/write (platform owns your data), Web3 = read/write/own (assets and identity are cryptographically owned by the user, not a company).
Ethereum vs Ether (ETH)
Ethereum is a decentralized, programmable blockchain — a globally-replicated 'world computer' that runs Turing-complete smart contracts; ETH (Ether) is its native asset used to pay for computation (gas) and secure the network via staking. Interview angle: contrast with Bitcoin, which is primarily a value-transfer ledger (UTXO, script is intentionally limited), whereas Ethereum uses an account model and general-purpose compute.
Blockchain fundamentals
A blockchain is an append-only, tamper-evident ledger where each block commits to the previous block's hash, so altering old history invalidates every subsequent block. Consensus (Ethereum uses Proof-of-Stake post-Merge) lets thousands of untrusting nodes agree on one canonical state without a central authority. The core trade-off to articulate: decentralization + security come at the cost of throughput and latency (every node re-executes every transaction).
Account types (EOA vs contract)
Two account types share an address, balance, and nonce: Externally Owned Accounts (EOAs), controlled by a private key, and Contract Accounts, controlled by their deployed code and owning persistent storage. Key interview fact: only an EOA can originate a transaction — contract code executes only when called, and contract-to-contract calls all trace back to an initiating EOA (until account-abstraction / EIP-4337 blurs this).
EVM as a virtual machine
The EVM is an abstract, stack-based virtual machine specified by the protocol; it runs identically on top of whatever physical/OS hardware each node uses. Virtualization is what guarantees deterministic, sandboxed, platform-independent execution so that every node computes the exact same post-transaction state — a hard requirement for consensus that raw native execution couldn't provide.
Single-threaded execution
The EVM processes transactions sequentially — no threads, no concurrency in the global state machine — because every node must reproduce byte-identical results. This is why sources of nondeterminism (wall-clock time, real randomness, floating point) are banned or restricted, and it's the fundamental reason base-layer Ethereum has limited TPS, motivating rollups/L2s for scale.
Memory, storage, and contract data
The EVM has distinct data locations with very different cost/lifetime profiles: the stack (1024 x 32-byte words, cheapest), memory (volatile, cleared after the call, linear byte array), calldata (read-only input), and storage (persistent 32-byte key→value slots written to chain). Interview angle: storage is orders of magnitude more expensive because every node persists it forever, so gas optimization is largely about minimizing SSTORE/SLOAD.
Gas and the gas limit
Gas meters computational and storage work; the gas limit caps how much a transaction (and a block) may consume, which solves the halting problem — an infinite loop simply runs out of gas and halts. Know the distinction between the per-transaction gas limit set by the sender and the per-block gas limit set by the protocol, plus that out-of-gas still charges for work done.
Opcodes
Solidity compiles to EVM bytecode, a sequence of ~140 low-level opcodes (PUSH, ADD, MLOAD, SSTORE, CALL, JUMP, etc.) executed against the stack. Understanding that contracts are ultimately opcodes explains debugging with disassembly/traces, why some Solidity patterns are cheaper than others, and how tools like the ABI and function selectors map onto raw bytecode.
Opcode gas costs
Each opcode has a gas price reflecting its real resource cost: arithmetic like ADD is 3 gas, while state-changing SSTORE can be up to ~20,000 gas and cold SLOAD is 2,100 (EIP-2929 warm/cold access). This pricing is the lever for gas optimization — batching storage writes, using memory over storage, and packing variables into single slots directly reduce transaction cost.

Real-world example

Debugging an out-of-gas revert while optimizing an on-chain reward distributor

You're an SDE-2 shipping a smart contract that pays out staking rewards to a list of users. In testing, calling distribute() with 50 addresses works, but on mainnet it reverts with 'out of gas' for 500 addresses. Tracing the transaction, you see the loop performs one SSTORE per user to update each balance in persistent storage — at up to ~20,000 gas each, the loop blows past the block gas limit, so the whole transaction reverts and consumes the gas without doing the work. Because the EVM is single-threaded and deterministic, you can't parallelize the loop; instead you redesign around its cost model: switch to a pull-based Merkle-claim pattern so each user pays their own SLOAD/SSTORE at claim time, cache the reward-per-token accumulator in a memory variable inside the loop instead of re-reading storage, and pack related struct fields into one 32-byte slot to turn multiple SSTOREs into one. This exercise touches every concept in the topic: EOA-initiated transactions, contract storage vs memory, opcodes and their gas prices, and the gas limit as the hard ceiling that forces the redesign.

Interview Q&A (SDE-2)

Q1. What is the difference between an EOA and a contract account, and why does it matter that only EOAs can initiate transactions?
An EOA is controlled by a private key and has a balance and nonce but no code; a contract account is controlled by its deployed bytecode and additionally owns persistent storage. Only an EOA can originate a transaction — contract code is purely reactive and runs only when invoked. This matters because every execution trace on-chain, including deep contract-to-contract call chains, ultimately roots at an externally-signed transaction from an EOA (account abstraction / EIP-4337 introduces smart-contract wallets that relax this UX-wise but still settle via the protocol's transaction rules).
Q2. Why is the EVM single-threaded and deterministic, and what are the consequences?
Consensus requires that every independent node re-executing a block arrives at the exact same resulting state, so execution must be fully deterministic — which rules out threads, real randomness, floating point, and wall-clock dependence. Sequential single-threaded execution is the simplest way to guarantee identical ordering and results everywhere. The consequence is limited base-layer throughput, which is why scaling is pushed to L2 rollups that execute off-chain and post compressed proofs/data back to L1.
Q3. Compare memory, storage, and calldata in terms of lifetime and gas cost.
Storage is persistent key→value state written to the chain and replicated by all nodes forever, making it by far the most expensive (SSTORE up to ~20k gas). Memory is a volatile linear byte array that exists only for the duration of a single call and is comparatively cheap (though it grows quadratically in cost as it expands). Calldata is the read-only, immutable input to a call — cheapest to read and ideal for function arguments you don't need to mutate. Good contract design keeps hot data in memory/calldata and touches storage as little as possible.
Q4. What problem does the gas limit solve, and what happens when a transaction runs out of gas?
Gas metering with a hard limit solves the halting problem on a public VM: since you can't statically prove arbitrary contract code terminates, you cap execution so a runaway or malicious infinite loop simply exhausts gas and stops, preventing network DoS. When a transaction runs out of gas it reverts all state changes it made (as if it never ran), but the sender is still charged for the gas consumed up to that point, since nodes already did the work. There are two limits to know: the per-transaction limit set by the sender and the per-block limit set by the protocol.
Q5. Explain cold vs warm storage access (EIP-2929) and why it affects gas optimization.
EIP-2929 made the first (cold) access to a given storage slot or account within a transaction more expensive than subsequent (warm) accesses — a cold SLOAD is 2,100 gas versus 100 for a warm one. The intent is to price in the disk I/O and state-access cost that a cold read actually incurs on nodes, and to mitigate certain DoS vectors. For optimization it means repeatedly reading the same slot is cheap after the first touch, so caching a storage value in a local (stack/memory) variable and reusing it, rather than re-reading storage in a loop, meaningfully lowers cost.
Q6. How does the EIP-1559 fee model work, and what happens to the base fee?
Under EIP-1559 each block has a protocol-computed base fee per gas that adjusts up or down based on how full the previous block was relative to a target, plus a user-specified priority fee (tip) that incentivizes validators to include the transaction. The total paid is (base fee + priority fee) x gas used, capped by the max fee the sender sets. Critically, the base fee is burned rather than paid to the validator, which removes an incentive to manipulate fees and makes ETH partially deflationary under high demand.
Q7. How does Solidity source relate to opcodes, and why should an SDE-2 care about the bytecode level?
The Solidity compiler lowers your source into EVM bytecode — a linear sequence of stack-based opcodes (PUSH, ADD, SSTORE, CALL, JUMP, etc.) — and the ABI/function selectors are just conventions layered on top of that bytecode. Caring about this level pays off when debugging reverts via execution traces, reasoning about why one code pattern costs more gas than another, and auditing for security issues like unexpected delegatecall or reentrancy that are clearer in the opcode flow than in high-level source.
Q8. At a high level, how does Ethereum's model differ from Bitcoin's, beyond 'smart contracts'?
Bitcoin uses a UTXO model where the ledger is a set of unspent outputs and script is intentionally limited (not Turing-complete) to keep validation simple and safe. Ethereum uses an account/balance model with global mutable state and a Turing-complete VM, so contracts can hold arbitrary state and logic — at the cost of needing gas metering to bound execution. That design choice is exactly what enables programmable applications (tokens, DeFi, NFTs) but also introduces the attack surface and gas-cost concerns Bitcoin largely avoids.
