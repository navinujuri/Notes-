Typescript

Cohort 2.0 · 11 lessons · SDE-2 (3+ yrs) interview prep

Concept summary — all 11 steps

Language typing spectrum
Languages sit on axes of static vs dynamic typing (when types are checked: compile-time vs runtime) and strong vs weak typing (how strictly type rules are enforced). TS adds a static, structural type layer over JS, which is dynamically and weakly typed; the interview angle is knowing TS types are erased at compile time and provide zero runtime guarantees — validation at trust boundaries (API responses, user input) still needs runtime checks like zod.
What TypeScript is
TS is a strict syntactic superset of JavaScript that adds optional static typing and compiles down to plain JS — all valid JS is valid TS. It matters because it catches type errors, enables safe refactoring, and powers editor tooling (autocomplete, go-to-def) without changing runtime behavior; the key nuance is that its type system is structural (duck-typed), not nominal.
The tsc compiler
tsc transpiles TS to JS for a chosen target (ES version) and performs type checking; the two jobs are separable — you can emit JS even with type errors (unless noEmitOnError) or type-check without emitting (--noEmit for CI). Interview angle: in real builds tsc is often used only for type-checking while esbuild/swc/Babel do the actual (type-stripping) transpile, because they're far faster but do NOT type-check.
Basic types & inference
Core primitives (string, number, boolean, null, undefined, bigint, symbol) plus TS-specific any, unknown, never, and void. The discriminating knowledge is any (opts out of checking — avoid) vs unknown (type-safe top type requiring narrowing before use) vs never (bottom type for unreachable/exhaustiveness), and leaning on inference rather than annotating everything.
tsconfig configuration
tsconfig.json defines compiler options, target/module, and file includes for a project. The interview-relevant flags are the strict family — especially strictNullChecks (separates null/undefined from other types) and noImplicitAny — plus target, module/moduleResolution, and paths; knowing why strict: true is the sane default and what each strict sub-flag buys you signals maturity.
Interfaces
Interfaces describe the shape of objects and classes and support declaration merging and extends-based inheritance. They matter for defining contracts (API DTOs, class implements); the classic interview point is interface vs type — interfaces are open (mergeable, better for public API surfaces and OOP), types are closed but more expressive (unions, intersections, mapped/conditional types).
Type aliases
type aliases name any type: primitives, unions, intersections, tuples, and computed/mapped/conditional types that interfaces can't express. They're the tool for union modeling (e.g. discriminated unions) and type-level programming; unlike interfaces they cannot be reopened/merged, which is often a feature (prevents accidental augmentation).
Arrays & tuples
Arrays typed as T[] or Array<T>, with readonly T[] for immutability and fixed-length tuples [string, number] for positional data. The nuance is tuples model heterogeneous positional structures (like React useState returns) and support labels and rest elements; also knowing that TS arrays are not bounds-checked so arr[i] is typed T even when the index is out of range (unless noUncheckedIndexedAccess).
Enums
Enums define named constant sets; numeric enums are bidirectional (reverse-mapped) while string enums are not. The important trade-off: enums emit real runtime objects (increasing bundle size and breaking type-only erasure), so many teams prefer union string literal types or `as const` objects — const enums inline values but interact poorly with isolatedModules/bundlers.
Generics
Generics parameterize types over other types for reusable, type-safe abstractions (containers, functions, higher-order utilities) without resorting to any. Depth points: generic constraints (T extends X), default type params, and inference from arguments; mastery shows in reading/writing utility types (Partial, Pick, Record) which are generics plus mapped/conditional types.
ES modules: import/export
TS uses ES module syntax (import/export, named and default) and resolves modules per moduleResolution. Interview-relevant: import type / export type for type-only imports (elided at compile so no runtime cost and safe under isolatedModules), the CommonJS vs ESM interop story (esModuleInterop), and how module vs the older namespace differs.

Real-world example

Typing a payments API client end-to-end

You're building a shared TypeScript SDK that other teams import to call your payments service. You model the wire contracts with interfaces for DTOs (PaymentRequest, PaymentResponse) and a discriminated union type for results — type Result = { status: 'succeeded'; id: string } | { status: 'failed'; error: string } | { status: 'pending' } — so a switch on status gives compile-time exhaustiveness (the default branch assigns to a never to catch newly added states). A string-literal union (or `as const` object) replaces an enum for currency codes to keep the bundle lean and erasable. The core request function is generic — request<TReq, TRes>(path, body: TReq): Promise<TRes> — reusing built-ins like Pick and Partial for update payloads. Crucially, because tsc types are erased at runtime, you validate the fetched JSON with a runtime schema (zod) at the network boundary and treat parsed data as unknown until narrowed, rather than trusting an as cast. tsconfig runs with strict: true and noUncheckedIndexedAccess; CI runs tsc --noEmit for type-checking while esbuild does the fast transpile, and all cross-file DTO imports use import type so nothing leaks into the emitted JS.

Interview Q&A (SDE-2)

Q1. interface vs type — when do you reach for each?
Both describe object shapes and are largely interchangeable there, but interfaces support declaration merging and are conventionally better for public/extensible API surfaces and class contracts (implements/extends). type aliases are closed (no merging) and strictly more expressive: unions, intersections, tuples, and mapped/conditional types can only be written as types. A common rule is interfaces for object/class contracts, types for unions and type-level computation; the merging behavior of interfaces is the deciding factor when you need — or want to forbid — later augmentation.
Q2. any vs unknown vs never — explain the difference.
any disables type checking entirely and is assignable both ways, so it silently propagates unsafety — avoid it. unknown is the type-safe top type: anything is assignable to it, but you can't use it until you narrow (typeof/instanceof/guards), making it the correct type for untrusted input like JSON. never is the bottom type — no value inhabits it — used for functions that never return (throw/infinite loop) and for compile-time exhaustiveness checks in discriminated unions.
Q3. Are TypeScript types available at runtime? What's the implication?
No — types are fully erased during compilation; the emitted JS has no type information (interfaces, type aliases, generics all vanish). The implication is you cannot make runtime decisions based on types and must not trust `as` casts on external data. At trust boundaries (API responses, form input, message queues) you need runtime validation (zod, io-ts, manual guards) to actually enforce shapes; TS only guarantees consistency of your own code, not the shape of data crossing the boundary.
Q4. Why might a team avoid enums in favor of union literal types or `as const`?
Enums emit a real runtime object, which adds bundle size and breaks the otherwise clean type-erasure model, and numeric enums add reverse mappings you rarely want. const enums inline values but are disallowed/awkward under isolatedModules and many bundlers (swc/esbuild/Babel transpile files in isolation). String-literal unions ('a' | 'b') or `as const` objects give the same type safety with zero or minimal runtime footprint and full erasability, which is why they're often preferred; enums still shine when you want a single named namespace of related constants with runtime iteration.
Q5. What does strictNullChecks change, and why is it important?
With it off, null and undefined are assignable to every type, so the compiler can't catch null-dereference bugs — the biggest source of runtime errors. With it on, null and undefined are their own types that must be explicitly included in a union (string | null) and narrowed before use, forcing you to handle the empty case. It's the single most valuable strict flag; enabling it on a legacy codebase is the hard part of adopting strict mode and typically surfaces many latent bugs.
Q6. Explain generic constraints and give a case where you need one.
A constraint (T extends SomeShape) restricts what types can be substituted for a type parameter, letting you access members of T safely inside the generic. For example, function getProp<T, K extends keyof T>(obj: T, key: K): T[K] — the K extends keyof T constraint guarantees key is a valid property and lets the return type be precisely T[K]. Without the constraint you'd only have an opaque T and couldn't index into it or would fall back to any, losing the type link between input and output.
Q7. What is a discriminated union and why is it powerful?
It's a union of object types that share a common literal-typed field (the discriminant), e.g. { kind: 'circle'; r: number } | { kind: 'square'; side: number }. Switching on the discriminant narrows the type in each branch so you get the correct members with no casting, and adding a `const _: never = x` in the default branch makes the compiler error whenever a new variant is added but not handled. It's the idiomatic TS pattern for modeling state machines, API results, and events with compile-time exhaustiveness.
Q8. How do you type-check and transpile in a modern build, and why separate them?
tsc does two jobs — type-checking and emitting JS — but they can be decoupled. Fast transpilers (esbuild, swc, Babel) strip types per-file without any type-checking, so they're used for the actual build/bundle, while tsc --noEmit runs in CI/editor purely for type safety. The separation matters because per-file transpilers can't understand cross-file type info (hence isolatedModules constraints like requiring `export type`), so you rely on them for speed and on tsc as the source of truth for correctness.
