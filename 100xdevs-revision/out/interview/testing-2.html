Integration and End to End testing

Cohort 2.0 · 10 lessons · SDE-2 (3+ yrs) interview prep

Concept summary — all 10 steps

Unit testing
Exercises a single unit (function/module) in isolation with all external dependencies (DB, HTTP, clock, other services) mocked or stubbed. Fast, deterministic, and cheap, so it forms the base of the test pyramid. Interview angle: unit tests catch logic/branch bugs early but prove nothing about whether components actually wire together correctly.
Integration tests
Verify that multiple units work together across a real boundary, e.g. route handler + service + real database, catching serialization, query, migration, and wiring bugs that mocks hide. The core trade-off is fidelity vs. speed: they need real infrastructure and run slower, but give far more confidence than unit tests alone.
Prerequisites for integration tests
You need an isolated, production-like environment: a dedicated test database (never dev/prod), applied migrations, deterministic seed data, and env/config that points the app at the test infra. The governing concern is isolation and repeatability so runs don't leak into real data or contaminate each other.
Starting the DB
Spin up a disposable database before the suite (a Docker/docker-compose service or Testcontainers), apply schema/migrations, then tear it down after. Ephemeral, containerized DBs give production-parity behavior (real SQL dialect, constraints, transactions) without the shared-state flakiness of a long-lived shared instance.
Bootstrapping integration tests in Vitest
Wire Vitest via vitest.config with globalSetup and/or setupFiles to bring up infra, run migrations, and expose a ready test app/client before specs execute. globalSetup runs once per whole run (ideal for starting the container/DB), while setupFiles run per worker/file (connections, globals) — mixing them up causes duplicated or missing setup.
Adding integration tests
Write tests that drive real endpoints (e.g. supertest against the Express app) and assert both the HTTP response and the resulting DB state, not just the status code. Cover happy path, validation/edge cases, and error/failure handling so the test reflects real client behavior end-to-end within the service.
beforeEach and beforeAll
beforeAll runs once per file for expensive one-time setup (connect DB, boot server); beforeEach runs before every test for cheap idempotent resets (truncate tables, reseed) that guarantee isolation, with afterEach/afterAll for cleanup. Interview angle: putting per-test reset in beforeEach is what prevents order-dependent, flaky failures.
CI/CD pipeline
Run the full suite automatically on every push/PR: the pipeline provisions a DB service, applies migrations, runs unit + integration + E2E, and gates merge/deploy on green. Tests are only trustworthy when they run in CI on a clean environment, not just on someone's laptop; this is also where you enforce coverage and prevent 'works on my machine' regressions.
End to end tests
Exercise the entire system through the UI as a real user would — a browser drives the frontend, which calls the real backend and DB. Highest fidelity but slowest and most brittle, so they sit at the top of the pyramid and are kept few, focused on critical user journeys (login, checkout) rather than exhaustive coverage.
Cypress
A browser-based E2E framework offering real-browser execution, time-travel debugging, automatic waiting/retries, and network stubbing (cy.intercept). Interview angle: contrast with Playwright — Cypress historically ran in-browser with same-origin/single-tab limits and per-test parallelism caveats, while Playwright offers true multi-browser, multi-tab, and faster cross-process parallelism.

Real-world example

Testing the checkout API of an e-commerce backend

You own POST /orders in a Node/TypeScript service backed by Postgres. Unit tests cover the pure price/discount/tax logic with mocked inputs so branch bugs surface instantly. Integration tests bring up a real Postgres via Testcontainers (started in Vitest globalSetup), apply migrations, then use supertest to POST a cart and assert both the HTTP 201 body and that an orders row was created and inventory was decremented; beforeAll connects and migrates once, while beforeEach truncates and reseeds so tests are order-independent and non-flaky. GitHub Actions runs unit + integration on every PR with a postgres service container and blocks merge on red. Finally, one Cypress E2E test drives a real browser through add-to-cart to checkout to the confirmation page against a deployed staging build, validating the whole journey — frontend, API, and DB — the way a customer actually experiences it.

Interview Q&A (SDE-2)

Q1. Explain the test pyramid and where unit, integration, and E2E tests fit.
The pyramid prescribes many fast, cheap unit tests at the base, fewer integration tests in the middle, and very few slow, brittle E2E tests at the top. As you move up, fidelity and confidence increase but speed, determinism, and maintainability decrease. The rationale is cost-efficiency: catch most defects cheaply at the bottom and reserve expensive full-system tests for a handful of critical user journeys.
Q2. Should integration tests use a real database or a mock/in-memory substitute?
Prefer a real instance of the same engine (e.g. Postgres via Testcontainers) because the whole point is to catch bugs in real SQL, constraints, transactions, and migrations that mocks and dialect-swapping (like SQLite-for-Postgres) silently hide. In-memory or mocked DBs are faster but give false confidence and let dialect-specific bugs reach production. The trade-off is runtime and setup complexity, which containerization largely neutralizes.
Q3. How do you keep integration tests isolated and avoid flakiness?
Give each run a clean, dedicated database and reset state between tests — typically truncate/reseed or wrap each test in a transaction that rolls back in beforeEach. Avoid shared mutable state, real time (inject a clock), and reliance on test execution order. For parallel workers, isolate by schema/database per worker or serialize DB-touching suites so they don't stomp on each other.
Q4. In Vitest, what is the difference between globalSetup, setupFiles, beforeAll, and beforeEach?
globalSetup runs once for the entire run — ideal for starting a container or DB and exposing teardown. setupFiles run once per test file/worker for things like registering globals or opening a connection. beforeAll runs once per describe/file for expensive per-file setup, and beforeEach runs before every individual test for cheap resets that guarantee isolation. Choosing the wrong scope causes either duplicated expensive setup or leaked state between tests.
Q5. How do you run integration tests that need a database in CI?
Provision the DB as part of the pipeline — a service container (GitHub Actions services:) or Testcontainers started by the test bootstrap — then apply migrations before running the suite against it. Pass connection details via environment variables, run on every PR, and fail the build (block merge) on any failure. Keeping infra ephemeral per run avoids cross-build contamination and matches production behavior.
Q6. Why are E2E tests flaky, and how do you reduce it?
They span network, rendering, async timing, and third-party dependencies, so timing races and environmental noise cause intermittent failures. Reduce it by relying on the framework's auto-waiting/retry and asserting on stable selectors (data-testid) instead of arbitrary sleeps, stubbing unreliable external calls (cy.intercept), seeding state via API rather than clicking through setup, and keeping the suite small and focused on critical paths. Run them against a stable, dedicated environment.
Q7. Cypress vs Playwright — how would you choose?
Cypress has an excellent DX — time-travel debugging, an interactive runner, and auto-retrying assertions — but historically runs inside the browser with same-origin/single-tab constraints and weaker native parallelism. Playwright runs out-of-process, supports true multi-browser (Chromium/Firefox/WebKit), multi-tab/multi-origin, and fast built-in parallelism, making it stronger for broad cross-browser coverage and CI scale. Pick Cypress for polished single-app debugging ergonomics; pick Playwright for cross-browser breadth, parallel throughput, and complex multi-context flows.
Q8. A feature spans a React form, an API, and a DB write. How do you decide what to test at each layer?
Push logic-heavy, branch-rich pieces (validation rules, price calc) into unit tests where they're cheapest to exhaust. Cover the API-plus-DB contract — request in, correct response and persisted state out, including error paths — with integration tests. Reserve one or two E2E tests for the critical end-to-end journey through the UI. The heuristic: test each behavior at the lowest layer that can meaningfully verify it, and don't duplicate the same assertion at multiple layers.
