DOM Part 2

Cohort 3.0 · 8 lessons · SDE-2 (3+ yrs) interview prep

Concept summary — all 8 steps

Context for today
Recaps DOM Part 1 (selecting nodes, textContent, basic event listeners) and frames today's goal: building a full TODO app to graduate from ad-hoc DOM pokes to a repeatable rendering pattern. The interview angle is recognizing that manually mutating individual nodes doesn't scale, which motivates the state-driven model that React later formalizes.
Complex DOM manipulation
Programmatically building nested element trees with document.createElement, element.append/appendChild, setAttribute, dataset, and classList, versus the string shortcut innerHTML. Key trade-off to articulate: createElement is verbose but safe and keeps node references/listeners intact; innerHTML is concise but re-parses/replaces the subtree (dropping existing listeners) and is an XSS vector when interpolating user input.
Finishing the TODO App - 2
Wires the remaining pieces of the app end-to-end: capturing input, updating the in-memory list, and reflecting it in the DOM so the UI stays consistent. The point is a complete round-trip (event -> mutate data -> render) rather than isolated snippets, which is the mental model an interviewer probes.
TODO app
The canonical CRUD-on-a-list exercise: an array of todo objects plus a render loop that paints each item with add/delete controls. It matters because it compresses selection, creation, events, and list rendering into one problem that mirrors real feature work (comment lists, cart items, notifications).
State derived frontends
The paradigm shift: treat a plain JS data structure (e.g. an array of {id, text, done}) as the single source of truth, and make the DOM a pure projection of it — UI = f(state). This matters because it eliminates the class of bugs where the screen and the data disagree, and it's the exact mental model behind React/Vue/Svelte.
State derived rendering
Implementing the projection: on every state change, run a render() that rebuilds the list from state (typically clear container + rebuild, or targeted updates). Discuss the trade-off — naive full re-render is simple and always correct but causes excess reflow/repaint and destroys focus/scroll/input state; this cost is precisely what React's virtual-DOM diffing and keyed reconciliation exist to avoid.
Add TODO functionality
Read the input value, push a new item (ideally with a stable unique id, not just the array index) into state, clear the input, then call render(). The subtlety worth naming: mutate state first and let render be the only thing that touches the DOM, rather than appending a node imperatively AND updating the array (two sources of truth drift).
Delete functionality
Remove the item from state (filter by stable id) and re-render, rather than removing the DOM node directly. Classic interview traps: using array index as the key breaks after other deletes, and attaching a per-row listener that closes over a loop variable declared with var captures the wrong id — solved with let/closured id or, better, event delegation from the parent container reading data-id.

Real-world example

Debugging a 'delete removes the wrong task' bug in a vanilla-JS task widget

You own an embeddable task-list widget (no framework — it ships on a marketing page) backed by an array of todos. A support ticket says clicking delete on the 2nd item sometimes removes the 3rd. You find the render loop attaches a delete handler per row using the array index as the identifier, so after any deletion the indices shift and stale handlers point at the wrong slot; a colleague's earlier attempt also used `var i` in the loop, so every handler closed over the final index. You refactor to the state-derived pattern: each todo gets a stable `id` (crypto.randomUUID()), the container uses a single delegated click listener that reads `e.target.closest('[data-id]').dataset.id`, delete does `todos = todos.filter(t => t.id !== id)` followed by one `render()`, and add does `todos.push(...)` then `render()`. You also swap an `innerHTML = userText` line for `textContent` after noticing a task titled `<img onerror=...>` executed — an XSS hole. The DOM is now a pure function of `todos`, the index bug is structurally impossible, and the input's focus loss on re-render becomes your next optimization (targeted updates or keys).

Interview Q&A (SDE-2)

Q1. What does 'state-derived rendering' mean, and why is it preferred over directly mutating the DOM on each user action?
It means keeping application data in a JS structure as the single source of truth and rendering the DOM as a pure projection of that state (UI = f(state)); every interaction mutates state, then a render step reconciles the DOM. Directly mutating the DOM on each action creates two sources of truth (the data and the screen) that inevitably drift, producing bugs where the UI shows something the data doesn't reflect. Centralizing all DOM writes in one render function makes behavior predictable and is the same model frameworks like React are built on.
Q2. A naive render() clears the container and rebuilds every item on each change. What are the downsides and how do frameworks address them?
Full rebuilds are always correct but expensive: they trigger layout/reflow and repaint for the whole list, and they destroy transient DOM state like input focus, text selection, scroll position, and CSS transitions. They also detach and recreate event listeners each time. React and similar libraries avoid this with a virtual DOM diff plus keyed reconciliation, updating only the nodes that actually changed and preserving the rest — which is why stable keys matter.
Q3. Why should you delete a todo by a stable id rather than by its array index?
An index is positional, not identity: after any insertion or deletion the indices of other items shift, so a handler or key bound to index 2 now refers to a different item. Using a stable unique id (UUID, timestamp, or a monotonic counter) means `todos.filter(t => t.id !== id)` always targets the intended item regardless of list mutations. This is the same reason React warns against using array index as a `key` for dynamic lists.
Q4. Explain the closure bug when attaching delete handlers in a loop, and how to fix it.
If you loop with `var i` and write `btn.onclick = () => remove(i)`, all handlers share one function-scoped `i`, which after the loop equals its final value — so every button deletes the last item. Fixes: declare the loop variable with `let` (block-scoped, a fresh binding per iteration), capture the id in the closure explicitly, or avoid per-row handlers entirely by using event delegation on the parent and reading a `data-id` attribute. Delegation is usually best because it's one listener regardless of list size.
Q5. When is innerHTML acceptable and when is it dangerous?
innerHTML is fine for injecting trusted, static, or developer-controlled markup and is convenient for building a subtree in one string. It's dangerous whenever the string contains user-supplied content, because the browser parses it as HTML and will execute injected scripts/handlers (e.g. `<img onerror>`), an XSS vulnerability. For user text use textContent (or createElement + textContent), reserve innerHTML for sanitized/known-safe content, and note it also replaces the subtree, discarding existing nodes and their listeners.
Q6. What is event delegation and why use it for a dynamic list?
Event delegation attaches a single listener to a stable parent and relies on event bubbling; inside the handler you inspect e.target (often via closest()) to find which child was interacted with. For a dynamic list it means you don't add/remove a listener per row as items come and go — one listener covers all current and future children, reducing memory and avoiding the need to rebind after every re-render. You typically pair it with data-id attributes to map the clicked element back to a state item.
Q7. How would you add basic performance and correctness improvements to a full-re-render TODO app without adopting a framework?
Preserve identity with stable ids and reuse nodes: instead of clearing and rebuilding, diff the state against existing rows (add new, remove missing, update changed text) so focus/scroll survive. Batch DOM writes using a DocumentFragment when you do rebuild, so you touch the live DOM once and trigger a single reflow. Keep all mutations flowing through state -> render, debounce high-frequency inputs, and read layout properties before writing to avoid layout thrashing.
Q8. Why does building this vanilla TODO app matter as preparation for React?
It makes the framework's value concrete: you've felt the pain of manual DOM sync, index-based bugs, listener rebinding, and full re-render cost, so React's declarative render, keys, and reconciliation read as solutions rather than magic. The core loop — state change triggers re-render, UI is a function of state — is identical; React just automates the efficient DOM diffing and gives you components and hooks on top. Understanding the underlying imperative model also helps you debug when the abstraction leaks (stale closures in hooks, key warnings, unnecessary re-renders).
