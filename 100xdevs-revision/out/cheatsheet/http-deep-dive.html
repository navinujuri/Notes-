HTTP Deep dive — Cheat Sheet

Cohort 3.0 · 9 lessons · quick reference

HTTP Recap (methods, status, anatomy)

Methods (verbs)
GET     // read, no body, idempotent, cacheable
POST    // create / non-idempotent action
PUT     // full replace (idempotent)
PATCH   // partial update
DELETE  // remove (idempotent)
OPTIONS // preflight / capabilities
HEAD    // GET headers only, no body
Idempotent = same result if repeated. Only GET/HEAD are safe (no side effects).
Status code families
1xx info    2xx success   3xx redirect
4xx client  5xx server
200 OK  201 Created  204 No Content
301 Moved  302 Found  304 Not Modified
400 Bad Request  401 Unauthorized  403 Forbidden
404 Not Found    409 Conflict     429 Too Many
500 Internal  502 Bad Gateway  503 Unavailable
401 = not authenticated; 403 = authenticated but not allowed.
Request line + anatomy
POST /api/users?active=1 HTTP/1.1
Host: api.example.com
Content-Type: application/json

{"name":"jae"}
Blank line separates headers from body. Query string is part of the URL, not the body.

Headers

Content negotiation
Content-Type: application/json; charset=utf-8
Accept: application/json
Accept-Encoding: gzip, br
Content-Length: 348
Server MUST set Content-Type on responses so the client parses correctly.
Auth
Authorization: Bearer <jwt>
Authorization: Basic base64(user:pass)
WWW-Authenticate: Bearer   // sent with 401
Never log Authorization headers.
Caching
Cache-Control: no-store | no-cache | max-age=3600
ETag: "abc123"
If-None-Match: "abc123"   // -> 304 if unchanged
no-store = never cache; no-cache = revalidate before use.
Cookies
Set-Cookie: sid=xyz; HttpOnly; Secure; SameSite=Strict; Max-Age=3600
Cookie: sid=xyz   // sent back by browser
HttpOnly blocks JS access (XSS defense); SameSite mitigates CSRF.
Read / set in Express
app.get('/x', (req, res) => {
  const ct = req.headers['content-type'];  // lowercased keys
  res.set('X-Custom', 'v');
  res.status(200).json({ ct });
});
req.headers keys are always lowercase in Node.

Fetch API (client)

GET + parse JSON
const res = await fetch('/api/users');
if (!res.ok) throw new Error(res.status);  // fetch does NOT reject on 4xx/5xx
const data = await res.json();
fetch only rejects on network failure, never on HTTP error status.
POST JSON
await fetch('/api/users', {
  method: 'POST',
  headers: { 'Content-Type': 'application/json' },
  body: JSON.stringify({ name: 'jae' })
});
Must JSON.stringify the body AND set Content-Type manually.
Response helpers
res.ok        // true if status 200-299
res.status    // 200
res.headers.get('content-type')
await res.text() / .json() / .blob() / .arrayBuffer()
Body can only be consumed once.
Timeout / abort
const c = new AbortController();
const t = setTimeout(() => c.abort(), 5000);
await fetch(url, { signal: c.signal });
clearTimeout(t);
Also: AbortSignal.timeout(5000) as a one-liner signal.
Credentials & mode
fetch(url, {
  credentials: 'include',  // send cookies cross-origin
  mode: 'cors'
});
credentials 'include' requires the server to send Access-Control-Allow-Credentials: true (and a non-wildcard origin).

Create an HTTP Server

Raw Node http
const http = require('http');
const server = http.createServer((req, res) => {
  res.writeHead(200, { 'Content-Type': 'application/json' });
  res.end(JSON.stringify({ ok: true }));
});
server.listen(3000, () => console.log('up on 3000'));
res.end() must be called or the request hangs.
Express setup
npm init -y && npm i express
---
const express = require('express');
const app = express();
app.use(express.json());         // parse JSON bodies
app.get('/', (req, res) => res.send('hi'));
app.listen(3000);
Without express.json() req.body is undefined for JSON requests.
Route params & query
app.get('/users/:id', (req, res) => {
  req.params.id;      // path param
  req.query.active;   // ?active=1
  res.json({ id: req.params.id });
});
Send responses
res.status(201).json({...});
res.send('text or buffer');
res.sendStatus(204);
res.redirect(302, '/login');
res.set('X-Foo','bar');
Set status BEFORE json/send; sending twice throws ERR_HTTP_HEADERS_SENT.

Middlewares

Signature & next()
function logger(req, res, next) {
  console.log(req.method, req.url);
  next();            // pass control on
}
app.use(logger);     // global, runs on every request
Forget next() (and don't send a response) => request hangs forever.
Order matters
app.use(express.json());   // 1st
app.use(logger);           // 2nd
app.get('/x', handler);    // runs after both
Middleware runs top-to-bottom; register body parsers before routes.
Short-circuit (auth gate)
function auth(req, res, next) {
  if (!req.headers.authorization)
    return res.status(401).json({ err: 'no token' });
  next();
}
return the res call so next() is never reached after responding.
Error-handling middleware
app.use((err, req, res, next) => {   // 4 args = error handler
  console.error(err);
  res.status(500).json({ err: err.message });
});
Must have exactly 4 params and be registered LAST. In Express 4, async errors need next(err); Express 5 auto-forwards rejected promises.

Route-specific middleware

Per-route inline
app.get('/dashboard', auth, (req, res) => {
  res.json({ secret: true });
});
Middleware args between path and handler run only for that route.
Chain multiple
app.post('/admin', auth, isAdmin, rateLimit, handler);
Executed left to right; each must call next() to continue the chain.
Path-mounted
app.use('/api', apiMiddleware);   // runs for any /api/*
app.use with a path prefix scopes middleware to that subtree.
Router-level
const router = express.Router();
router.use(auth);                 // applies to all router routes
router.get('/me', handler);
app.use('/users', router);
Great for grouping and mounting a feature under one prefix.

Commonly used middlewares

Body parsers (built-in)
app.use(express.json());                       // JSON
app.use(express.urlencoded({ extended: true })); // form posts
app.use(express.static('public'));             // serve files
No more separate body-parser package; it's built into Express 4.16+.
morgan — request logging
npm i morgan
app.use(require('morgan')('dev'));
helmet — security headers
npm i helmet
app.use(require('helmet')());
Sets sane defaults (CSP, X-Frame-Options, etc.).
express-rate-limit
npm i express-rate-limit
const rl = require('express-rate-limit');
app.use(rl({ windowMs: 60_000, max: 100 }));   // 100 req/min
Returns 429 when exceeded.
cookie-parser
npm i cookie-parser
app.use(require('cookie-parser')());
// then: req.cookies.sid

CORS — Cross-Origin Resource Sharing

Why
// Origin = scheme + host + port
// http://a.com:3000  vs  http://a.com:4000  => cross-origin
Enforced by the BROWSER, not the server. Server-to-server calls ignore CORS.
cors package
npm i cors
const cors = require('cors');
app.use(cors());                        // allow all origins
app.use(cors()) globally; or cors(opts) per-route.
Restrict origin + credentials
app.use(cors({
  origin: 'https://app.example.com',
  credentials: true,
  methods: ['GET','POST']
}));
With credentials:true you CANNOT use origin '*' — must echo a specific origin.
Manual headers
res.setHeader('Access-Control-Allow-Origin', 'https://app.com');
res.setHeader('Access-Control-Allow-Methods', 'GET,POST,PUT,DELETE');
res.setHeader('Access-Control-Allow-Headers', 'Content-Type,Authorization');
Preflight (OPTIONS)
// Browser auto-sends OPTIONS before non-simple requests
// (custom headers, PUT/DELETE, JSON content-type)
app.options('*', cors());   // answer preflight
Preflight response needs 2xx + Allow-* headers, or the real request is blocked.

Quick tips & gotchas

  • fetch never throws on 4xx/5xx — always check res.ok before parsing.
  • A middleware that neither calls next() nor sends a response hangs the request; always do one or the other.
  • Register body parsers (express.json) BEFORE routes, and the error handler (4-arg) LAST.
  • Set res.status() before .json()/.send(); calling res twice throws ERR_HTTP_HEADERS_SENT.
  • CORS is browser-enforced — curl/Postman/server-to-server calls bypass it, so 'works in Postman' doesn't mean CORS is configured.
  • credentials:'include' (fetch) requires a specific Allow-Origin (no '*') plus Access-Control-Allow-Credentials:true.
  • Node lowercases all req.headers keys — use req.headers['content-type'], not 'Content-Type'.
  • 401 = who are you (unauthenticated); 403 = I know you, still not allowed (unauthorized).
