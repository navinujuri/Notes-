Kubernetes Part 2 — Cheat Sheet

Cohort 2.0 · 23 lessons · quick reference

Cluster: run locally & on cloud

Local cluster (kind)
kind create cluster --name dev
kubectl cluster-info --context kind-dev
kind runs nodes as Docker containers; ports must be mapped via cluster config for ingress.
Local cluster (minikube)
minikube start --driver=docker
minikube tunnel   # needed for type=LoadBalancer + ingress
Without `minikube tunnel`, LoadBalancer EXTERNAL-IP stays <pending>.
Cloud cluster (EKS / GKE)
eksctl create cluster --name prod --nodes 3
gcloud container clusters create prod --num-nodes 3
aws eks update-kubeconfig --name prod   # writes kubeconfig
Cloud gives you real LoadBalancer + dynamic PV provisioning out of the box.
Contexts & apply loop
kubectl config get-contexts
kubectl config use-context kind-dev
kubectl apply -f manifest.yaml
kubectl get pods,svc,ingress -o wide
Everything is `apply -f` (declarative). `kubectl get all -n <ns>` for a quick snapshot.

Services & their downsides

ClusterIP (default, internal only)
apiVersion: v1
kind: Service
metadata: { name: web }
spec:
  selector: { app: web }   # must match pod labels
  ports:
    - port: 80          # service port
      targetPort: 3000  # container port
Reachable in-cluster only. DNS: web.<namespace>.svc.cluster.local
NodePort (exposes on every node)
spec:
  type: NodePort
  ports:
    - port: 80
      targetPort: 3000
      nodePort: 30080   # 30000-32767
Hit any nodeIP:30080. Ugly high ports; fine for dev/testing.
LoadBalancer (cloud external IP)
spec:
  type: LoadBalancer
  ports: [{ port: 80, targetPort: 3000 }]
Provisions a real cloud LB. On bare-metal/kind stays <pending> without MetalLB.
Quick expose
kubectl expose deployment web --port=80 --target-port=3000 --type=LoadBalancer
Downside → why Ingress exists
One LoadBalancer per service = one cloud LB each = $$ + a public IP each. Services are L4 only: no host/path routing, no TLS termination, no single entrypoint.

Ingress, controllers & namespaces

Namespaces (isolate resources)
kubectl create namespace app
kubectl apply -f x.yaml -n app
kubectl get pods -n app
kubectl config set-context --current --namespace=app
Controllers usually live in their own ns (e.g. ingress-nginx, cert-manager).
Install NGINX ingress controller
helm repo add ingress-nginx https://kubernetes.github.io/ingress-nginx
helm install ingress-nginx ingress-nginx/ingress-nginx -n ingress-nginx --create-namespace
# kind/manifest alt:
kubectl apply -f https://raw.githubusercontent.com/kubernetes/ingress-nginx/main/deploy/static/provider/cloud/deploy.yaml
Controller = the actual pod (a reverse proxy) that watches Ingress objects. Ingress object = just the routing config.
Ingress routing rules
apiVersion: networking.k8s.io/v1
kind: Ingress
metadata: { name: app-ing }
spec:
  ingressClassName: nginx
  rules:
    - host: app.example.com
      http:
        paths:
          - path: /api
            pathType: Prefix
            backend:
              service: { name: api, port: { number: 80 } }
          - path: /
            pathType: Prefix
            backend:
              service: { name: web, port: { number: 80 } }
One LB fronts many services via host/path. pathType: Prefix | Exact | ImplementationSpecific.
Try Traefik instead
helm repo add traefik https://traefik.github.io/charts
helm install traefik traefik/traefik -n traefik --create-namespace
Same Ingress API works; set ingressClassName: traefik. Swappable controllers, one spec.

ConfigMaps & Secrets

Create ConfigMap
kubectl create configmap app-cfg \
  --from-literal=LOG_LEVEL=info \
  --from-file=config.json
Non-sensitive config only. Stored in plaintext in etcd.
Create Secret
kubectl create secret generic db-sec \
  --from-literal=PASSWORD=s3cr3t
# YAML: use stringData (plain) or data (base64)
kind: Secret
type: Opaque
stringData: { PASSWORD: s3cr3t }
`data:` values must be base64. Decode: kubectl get secret db-sec -o jsonpath='{.data.PASSWORD}' | base64 -d
Consume as env vars
envFrom:
  - configMapRef: { name: app-cfg }
  - secretRef:    { name: db-sec }
# or single key:
env:
  - name: PASSWORD
    valueFrom:
      secretKeyRef: { name: db-sec, key: PASSWORD }
Consume as mounted volume
volumes:
  - name: cfg
    configMap: { name: app-cfg }
containers:
  - volumeMounts:
      - { name: cfg, mountPath: /etc/app }
Mounted values update live (env vars do NOT — pod needs restart).
ConfigMap vs Secret
Same API shape. Secrets are base64 (NOT encryption), can be RBAC-restricted + encrypted-at-rest, mounted as tmpfs. Use Secret for tokens/passwords/certs; ConfigMap for everything else.

HTTPS with cert-manager

Install cert-manager
helm repo add jetstack https://charts.jetstack.io
helm install cert-manager jetstack/cert-manager \
  -n cert-manager --create-namespace --set crds.enabled=true
ClusterIssuer (Let's Encrypt)
apiVersion: cert-manager.io/v1
kind: ClusterIssuer
metadata: { name: letsencrypt }
spec:
  acme:
    server: https://acme-v02.api.letsencrypt.org/directory
    email: you@example.com
    privateKeySecretRef: { name: le-account }
    solvers:
      - http01: { ingress: { class: nginx } }
Use staging server first to avoid LE rate limits.
Auto-TLS via Ingress annotation
metadata:
  annotations:
    cert-manager.io/cluster-issuer: letsencrypt
spec:
  tls:
    - hosts: [app.example.com]
      secretName: app-tls   # cert-manager fills this
cert-manager watches the tls block, solves ACME challenge, writes the cert Secret, and auto-renews.

Volumes: ephemeral → persistent

Docker volumes (baseline)
docker run -v mydata:/var/lib/data img      # named volume
docker run -v $(pwd):/app img               # bind mount
Container FS is ephemeral; volumes outlive the container. k8s generalizes this.
emptyDir (ephemeral)
volumes:
  - name: cache
    emptyDir: {}
Lives & dies with the POD (survives container restart, NOT pod deletion/reschedule). Good for scratch/cache.
PersistentVolumeClaim (what pods use)
apiVersion: v1
kind: PersistentVolumeClaim
metadata: { name: data }
spec:
  accessModes: [ReadWriteOnce]
  resources: { requests: { storage: 5Gi } }
  storageClassName: standard
Pod references the CLAIM, never the PV directly: volumes: [{ name: d, persistentVolumeClaim: { claimName: data } }]
Static PV (admin pre-provisions)
apiVersion: v1
kind: PersistentVolume
metadata: { name: pv1 }
spec:
  capacity: { storage: 5Gi }
  accessModes: [ReadWriteOnce]
  hostPath: { path: /mnt/data }   # or nfs/csi
PVC binds to a matching PV. No StorageClass → manual binding.
Dynamic / automatic PV (StorageClass)
kind: StorageClass
metadata: { name: fast }
provisioner: kubernetes.io/aws-ebs
parameters: { type: gp3 }
reclaimPolicy: Delete
PVC names the StorageClass → PV is provisioned on the fly. This is the default in cloud clusters.

Quick tips & gotchas

  • Service selector labels MUST match pod template labels exactly, or EXTERNAL/ENDPOINTS stay empty — debug with `kubectl get endpoints <svc>`.
  • One entrypoint rule: use ONE LoadBalancer for the ingress controller, then route everything else through Ingress objects (ClusterIP services behind it).
  • Ingress does nothing without a controller running — installing the Ingress object alone is a common newbie trap.
  • Secrets are base64-encoded, NOT encrypted. Don't commit them to git; enable etcd encryption-at-rest and restrict via RBAC.
  • env-var-injected ConfigMaps/Secrets need a pod restart to pick up changes; volume-mounted ones update live (with a short kubelet sync delay).
  • Test cert-manager against Let's Encrypt STAGING first — the prod endpoint rate-limits ~5 certs/domain/week.
  • emptyDir data is lost when the pod is deleted or rescheduled to another node — only PVCs survive that. For DBs, always use a PVC.
  • accessModes matter: ReadWriteOnce = one node at a time (most block storage); use ReadWriteMany (NFS/CSI) only if multiple pods must write simultaneously.
