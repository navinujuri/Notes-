Asynchronous Javascript, Callbacks and more — Cheat Sheet

Cohort 2.0 · 9 lessons · quick reference

Functions in JS (declaration, expression, arrow)

Function declaration (hoisted)
function add(a, b) {
  return a + b;
}
add(2, 3); // 5
Hoisted to top of scope — callable before its line.
Function expression (not hoisted)
const add = function (a, b) {
  return a + b;
};
The const binding is TDZ; calling before definition throws.
Arrow function
const add = (a, b) => a + b;
const sq = x => x * x;      // single arg, no parens needed
const noop = () => {};      // empty body
No own `this`/`arguments`; inherits from enclosing scope. Can't be a constructor.
Functions are first-class values
const fns = [add, sq];
const m = { op: add };
fn = add; // pass around, store, return
Functions are objects — assign, store in arrays/objects, return them.

Synchronous / blocking code

Sync = runs top-to-bottom, one line blocks the next
console.log(1);
const x = heavyCompute(); // everything waits here
console.log(2);           // runs only after heavyCompute returns
Single-threaded: while a sync call runs, nothing else executes.
Blocking the main thread (bad)
function sleepBlock(ms) {
  const end = Date.now() + ms;
  while (Date.now() < end) {} // spins CPU, freezes everything
}
Busy-wait blocks the event loop; UI/other requests stall.
Synchronous file read (blocks)
const fs = require('fs');
const data = fs.readFileSync('a.txt', 'utf8');
console.log(data); // nothing else runs until read completes
`*Sync` APIs block the whole process — fine for CLI startup, bad in servers.

I/O-bound vs CPU-bound

I/O-bound task
// waiting on disk / network / DB
fs.readFile(...)
fetch('https://api...')
db.query('SELECT ...')
Bottleneck is external wait, not the CPU — async wins here.
CPU-bound task
// heavy pure computation
hash(bigBuffer);
sortMillionItems(arr);
for (…) { crunch(); }
Bottleneck is the processor; async doesn't help — offload to Worker threads / child processes.
Why async matters for I/O
During I/O the thread would otherwise sit idle; async lets it serve other work while the OS handles the wait.
Handling CPU-bound work off the main thread
const { Worker } = require('worker_threads');
new Worker('./crunch.js');
Callbacks/promises don't parallelize CPU work — you need real threads/processes.

I/O in the real world (Node fs)

Async read with callback (non-blocking)
const fs = require('fs');
fs.readFile('a.txt', 'utf8', (err, data) => {
  if (err) throw err;
  console.log(data);
});
console.log('runs FIRST'); // logs before file contents
Node convention: callback signature is (err, result) — error first.
Async write
fs.writeFile('out.txt', 'hi', (err) => {
  if (err) return console.error(err);
  console.log('written');
});
Sync variant for comparison
try {
  const d = fs.readFileSync('a.txt', 'utf8');
} catch (err) {
  console.error(err);
}
Sync uses try/catch; async passes err into the callback.
setTimeout — the canonical async primitive
setTimeout(() => console.log('later'), 1000);
console.log('now'); // 'now' then 'later'
Delay is a minimum, not a guarantee — callback runs after the stack clears AND the timer fires.

Functions as arguments (callbacks)

Pass a function into another function
function withLogging(fn, a, b) {
  console.log('calling');
  return fn(a, b);
}
withLogging(add, 2, 3); // 5
A callback is just a function passed as an argument to be invoked later.
Named vs inline callback
arr.forEach(function (x) { console.log(x); }); // inline
function log(x) { console.log(x); }
arr.forEach(log);                              // named ref
Pass the reference `log`, not `log()` — the latter calls it immediately.
Higher-order array methods
[1,2,3].map(x => x * 2);        // [2,4,6]
[1,2,3].filter(x => x % 2);     // [1,3]
[1,2,3].reduce((a, x) => a + x, 0); // 6
Callback that takes its own args
function repeat(n, cb) {
  for (let i = 0; i < n; i++) cb(i);
}
repeat(3, i => console.log(i)); // 0 1 2

Async code & callbacks

Deferred execution via callback
function fetchUser(id, cb) {
  setTimeout(() => cb(null, { id, name: 'Alice' }), 500);
}
fetchUser(1, (err, user) => {
  if (err) return console.error(err);
  console.log(user.name);
});
Result isn't returned — it's delivered later through the callback.
Sequencing dependent async ops (nesting)
fetchUser(1, (e, u) => {
  fetchPosts(u.id, (e, posts) => {
    fetchComments(posts[0], (e, c) => {
      // ...callback hell
    });
  });
});
Nested callbacks = 'pyramid of doom'; promises/async-await flatten this.
Error-first pattern (Node standard)
function op(cb) {
  const err = null;
  const result = 42;
  cb(err, result);
}
Always check `err` first before touching the result.
You cannot `return` an async result
function bad() {
  let v;
  setTimeout(() => { v = 1; }, 0);
  return v; // undefined — timer hasn't fired yet
}
The function returns before the callback runs. Use the callback (or a promise) to get the value out.

JS async architecture (event loop)

The moving parts
Call Stack (runs sync code) + Web/Node APIs (handle timers, I/O off-thread) + Callback/Task Queue + Event Loop (moves queued callbacks onto the stack when it's empty).
How a callback gets executed
console.log('A');
setTimeout(() => console.log('B'), 0);
console.log('C');
// Output: A, C, B
Even with 0ms, B waits until the stack is empty — the loop drains sync code first.
Microtasks run before macrotasks
setTimeout(() => console.log('timeout'), 0); // macrotask
Promise.resolve().then(() => console.log('promise')); // microtask
// Output: promise, then timeout
Promise `.then`/`await` continuations (microtasks) drain fully before the next timer/I/O callback.
Single-threaded, non-blocking
One call stack. Concurrency comes from offloading waits to the runtime (libuv/browser), not from parallel JS execution.

Quick tips & gotchas

  • Node callbacks are error-first: `(err, result) => {}` — handle `err` before using `result`.
  • Pass a function reference (`cb`) not a call (`cb()`); the latter executes it right away and passes the return value.
  • `setTimeout(fn, 0)` does NOT run immediately — it defers `fn` until the current stack clears.
  • Async doesn't speed up CPU-bound work; it only helps you not block during I/O waits. Use Worker threads for heavy compute.
  • You can't `return` a value produced inside an async callback — the outer function has already returned. Deliver it via the callback.
  • Deeply nested callbacks (callback hell) are hard to read and error-handle — reach for Promises / async-await for sequential async.
  • Microtasks (Promises) always drain before the next macrotask (timers, I/O), regardless of delay.
  • Prefer async `fs.readFile` over `fs.readFileSync` in servers — the Sync version blocks every other request.
