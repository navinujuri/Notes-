Integration and End to End testing — Cheat Sheet

Cohort 2.0 · 10 lessons · quick reference

Test pyramid & install

Unit test — pure fn / single module, mock all I/O (fast, most numerous)
import { describe, it, expect } from 'vitest'
describe('add', () => {
  it('sums', () => expect(add(2,3)).toBe(5))
})
No network/DB/filesystem. Milliseconds. Run on every save.
Integration test — multiple real units wired together (route + service + real DB)
Hits a REAL test database, not mocks. Verifies SQL, migrations, serialization.
E2E test — full stack through the browser as a user (frontend+backend+DB)
Slowest, fewest. Cypress/Playwright drive a real browser against a running app.
Install stack
npm i -D vitest supertest
npm i -D cypress
npx vitest        # watch
npx cypress open  # interactive runner

Prerequisites for integration tests

Isolated test DB (never your dev/prod DB)
# .env.test
DATABASE_URL="postgresql://postgres:postgres@localhost:5433/appdb_test"
Different PORT/DB name than dev so a test wipe can't nuke real data.
Migrations applied to the test DB before the suite runs
npx prisma migrate deploy   # or: prisma db push for throwaway schema
Deterministic seed / clean slate per run
Tests must not depend on ordering or leftover rows.
App exportable without auto-listening
// server.ts
export const app = express() // ...routes
// index.ts (only prod entry)
app.listen(3000)
Export `app` so supertest can bind it without an open port.

Starting the DB (Docker)

One-off Postgres container
docker run -d --name pg-test -e POSTGRES_PASSWORD=postgres \
  -e POSTGRES_DB=appdb_test -p 5433:5432 postgres:16
docker-compose for repeatable local + CI (docker compose -f docker-compose.test.yml up -d)
# docker-compose.test.yml
services:
  db:
    image: postgres:16
    environment:
      POSTGRES_PASSWORD: postgres
      POSTGRES_DB: appdb_test
    ports: ['5433:5432']
    healthcheck:
      test: ['CMD-SHELL','pg_isready -U postgres']
      interval: 2s
      retries: 10
Wait until ready before migrating
docker compose -f docker-compose.test.yml up -d --wait
npx prisma migrate deploy
--wait blocks until healthcheck passes; avoids race on first connect.
Testcontainers (spin DB from inside the test process)
const pg = await new PostgreSqlContainer('postgres:16').start()
process.env.DATABASE_URL = pg.getConnectionUri()
Ephemeral, port randomised — great for CI parallelism.

Bootstrapping Vitest for integration

Separate config so unit and integration run independently
// vitest.config.integration.ts
import { defineConfig } from 'vitest/config'
export default defineConfig({
  test: {
    include: ['**/*.int.test.ts'],
    globalSetup: './test/global-setup.ts',
    hookTimeout: 30_000,
    fileParallelism: false, // shared DB -> run files serially
  },
})
globalSetup — start DB + migrate ONCE for the whole run
// test/global-setup.ts
export async function setup() {
  execSync('docker compose -f docker-compose.test.yml up -d --wait')
  execSync('npx prisma migrate deploy')
}
export async function teardown() {
  execSync('docker compose -f docker-compose.test.yml down -v')
}
globalSetup runs in a separate context — you can't share JS objects, only side effects/env.
npm scripts
"test": "vitest run",
"test:int": "vitest run --config vitest.config.integration.ts",
"test:e2e": "cypress run"
Load .env.test
import 'dotenv/config'
// or: vitest --config ... with test.env / setupFiles
Set NODE_ENV=test and point DATABASE_URL at the test DB before Prisma client instantiates.

Writing integration tests + hooks

beforeAll — expensive one-time setup (connect client, create server)
import { beforeAll, afterAll, beforeEach, describe, it, expect } from 'vitest'
const prisma = new PrismaClient()
beforeAll(async () => { await prisma.$connect() })
afterAll(async () => { await prisma.$disconnect() })
beforeEach — reset state so tests are independent
beforeEach(async () => {
  await prisma.$executeRawUnsafe('TRUNCATE users, posts RESTART IDENTITY CASCADE')
})
beforeAll runs ONCE per file; beforeEach runs before EVERY it(). Clean in beforeEach, not after — a crashed test still leaves a clean slate for the next.
Assert through the HTTP layer with supertest (route → DB round trip)
import request from 'supertest'
import { app } from '../src/server'

it('creates + persists a user', async () => {
  const res = await request(app)
    .post('/users').send({ email: 'a@b.com' })
  expect(res.status).toBe(201)
  const row = await prisma.user.findUnique({ where: { email: 'a@b.com' } })
  expect(row).not.toBeNull()  // verify it actually hit the DB
})
Hook order (nested describe)
beforeAll(outer) → beforeAll(inner) → beforeEach(outer) → beforeEach(inner) → test → afterEach(inner→outer) → afterAll. Return a fn from beforeEach to auto-run as afterEach cleanup.

CI/CD pipeline (GitHub Actions)

Postgres as a service container
jobs:
  test:
    runs-on: ubuntu-latest
    services:
      postgres:
        image: postgres:16
        env: { POSTGRES_PASSWORD: postgres, POSTGRES_DB: appdb_test }
        ports: ['5432:5432']
        options: >-
          --health-cmd pg_isready --health-interval 2s --health-retries 10
    env:
      DATABASE_URL: postgresql://postgres:postgres@localhost:5432/appdb_test
    steps:
      - uses: actions/checkout@v4
      - uses: actions/setup-node@v4
        with: { node-version: 20, cache: npm }
      - run: npm ci
      - run: npx prisma migrate deploy
      - run: npm run test:int
`services` are managed by the runner (started/healthchecked before steps) — no docker-compose needed in CI.
Gate merges on green tests
Make the job a required status check on the protected branch so failing integration/e2e blocks the PR.
Run E2E in CI
- run: npm run build && npm start &
- run: npx wait-on http://localhost:3000
- run: npx cypress run
Start the app, wait for the port, then run headless Cypress.

Cypress E2E

Config
// cypress.config.ts
import { defineConfig } from 'cypress'
export default defineConfig({
  e2e: {
    baseUrl: 'http://localhost:3000',
    supportFile: 'cypress/support/e2e.ts',
  },
})
A spec — cypress/e2e/login.cy.ts
describe('login', () => {
  it('logs in', () => {
    cy.visit('/login')
    cy.get('[data-cy=email]').type('a@b.com')
    cy.get('[data-cy=password]').type('pw{enter}')
    cy.url().should('include', '/dashboard')
    cy.contains('Welcome').should('be.visible')
  })
})
Prefer stable data-cy attributes over CSS/text selectors.
Commands don't return — chain, don't await
cy.get('.btn').click()          // queued, retries automatically
// WRONG: const el = await cy.get(...)
Cypress commands are async but not Promises; use .then() to read values. Assertions auto-retry until timeout.
Stub network / seed via API for speed
cy.intercept('GET', '/api/users', { fixture: 'users.json' }).as('users')
cy.request('POST', '/test/seed', { email: 'a@b.com' })  // reset backend state
cy.wait('@users')
Reusable steps as custom commands
// support/commands.ts
Cypress.Commands.add('login', (u,p) => {
  cy.request('POST','/api/login',{u,p})
})
// spec: cy.login('a@b.com','pw')
Log in via API not the UI in setup — faster and less flaky than clicking through the form every test.
Run modes
npx cypress open   # GUI, watch, time-travel debugger
npx cypress run    # headless, for CI (records video/screenshots on fail)

Quick tips & gotchas

  • Isolation first: each test must set up and tear down its own data. A shared, mutating DB with order-dependent tests is the #1 source of flake.
  • Clean in beforeEach (TRUNCATE ... RESTART IDENTITY CASCADE), not afterEach — a crashed test then can't poison the next one.
  • beforeAll = once per file (connect DB, boot server); beforeEach = before every it() (reset rows). Mixing these up causes cross-test bleed or slow suites.
  • Set fileParallelism:false (or maxWorkers:1) when integration test files share one DB, or give each worker its own schema/DB to keep parallelism.
  • Never point tests at dev/prod: guard with a separate DATABASE_URL and refuse to run if NODE_ENV !== 'test'. TRUNCATE on the wrong DB is unrecoverable.
  • Bump hook/test timeouts (hookTimeout, testTimeout ~30s) — Docker start + migrate is slow and the default 5s will falsely fail the first run.
  • In CI use `services:` (managed containers with healthchecks) and run `prisma migrate deploy` before the suite; wait-on the port before Cypress.
  • Cypress commands are queued, not awaited — never `await cy.get()`. Assert with .should() (auto-retries) and select via data-cy attributes to survive markup changes.
