AWS Deploying (EC2) — Cheat Sheet

Cohort 2.0 · 8 lessons · quick reference

AWS & EC2 Fundamentals

What is AWS
Cloud provider renting compute/storage/network on demand. EC2 = Elastic Compute Cloud = virtual servers (VMs) you rent by the second/hour.
Instance = a rented server
Pick a Region (physical location, e.g. ap-south-1) close to users. Instance stays in one AZ (availability zone).
AMI (Amazon Machine Image)
OS template. Ubuntu Server LTS is the common default; Amazon Linux 2023 uses dnf/yum + user ec2-user.
Instance types / free tier
t2.micro / t3.micro  # 1 vCPU, 1GB RAM — free tier eligible
Family + size. t = burstable/general. Free tier: 750 hrs/month of micro for 12 months.
Key pair
Download the .pem ONCE at launch — you cannot re-download it. It's your SSH private key. Lose it = lose SSH access.
Security Group = firewall
Controls inbound/outbound ports. Nothing is reachable unless you open the port here. Default: only SSH open, or nothing.

Create & Configure an Instance

Launch flow (console)
EC2 → Launch instance → Name → AMI (Ubuntu) → t2.micro → create/select key pair → configure Security Group → Launch.
Open ports in Security Group
SSH    TCP  22    Source: My IP (or 0.0.0.0/0)
HTTP   TCP  80    Source: 0.0.0.0/0
HTTPS  TCP  443   Source: 0.0.0.0/0
Custom TCP 3000  Source: 0.0.0.0/0   # only while testing app directly
0.0.0.0/0 = the whole internet. Lock SSH to your IP. Remove the app port (3000) once nginx fronts it.
Public IP vs Elastic IP
Public IP changes on every stop/start. Allocate an Elastic IP and associate it for a stable address (free while attached to a running instance).
Get the public address
Instance details → 'Public IPv4 address' or 'Public IPv4 DNS'. This is what you SSH to and point DNS at.

SSH Into the Server

Fix key permissions (Linux/macOS)
chmod 400 my-key.pem
SSH refuses keys that are group/world-readable ('UNPROTECTED PRIVATE KEY FILE').
Connect
ssh -i my-key.pem ubuntu@<public-ip>
User = ubuntu (Ubuntu AMI), ec2-user (Amazon Linux), admin (Debian). Wrong user is the #1 'Permission denied' cause.
Windows key permission fix
icacls my-key.pem /inheritance:r
icacls my-key.pem /grant:r "%USERNAME%:R"
Or use the pem directly from PowerShell/OpenSSH; PuTTY needs a .ppk (convert via PuTTYgen).
Connection timeout vs refused
Timeout = Security Group not allowing port 22 (or wrong IP). Refused = sshd down / wrong port.

Install & Run the Repo

Update packages first
sudo apt update && sudo apt upgrade -y
Install git + clone
sudo apt install git -y
git clone https://github.com/user/repo.git
cd repo
Private repo? Use a deploy key or a PAT in the HTTPS URL.
Install Node (via nvm)
curl -o- https://raw.githubusercontent.com/nvm-sh/nvm/v0.40.1/install.sh | bash
source ~/.bashrc
nvm install --lts
nvm avoids permission issues vs apt's node; keeps versions per-user.
Install deps & run
npm install
node index.js   # or: npm start
Foreground process dies when you log out. Use pm2/systemd for persistence (below).
Keep it alive with pm2
npm i -g pm2
pm2 start index.js --name app
pm2 startup    # generates a boot service — run the printed command
pm2 save
pm2 restarts on crash and on reboot. pm2 logs / pm2 restart app to manage.

Hit the Server (Verify)

Test locally on the box first
curl http://localhost:3000
Works locally but not from browser = firewall/Security Group, not your app.
Test from your machine
curl http://<public-ip>:3000
# browser: http://<public-ip>:3000
Requires that port open in the Security Group. This step is only pre-nginx.
Bind to 0.0.0.0, not 127.0.0.1
App listening on localhost only is unreachable externally even with the port open. Listen on 0.0.0.0 (or let nginx proxy to localhost).

nginx Reverse Proxy

Install & start
sudo apt install nginx -y
sudo systemctl enable --now nginx
Now port 80 serves the default nginx page — confirms 80 is reachable.
Server block → proxy to app
# /etc/nginx/sites-available/default  (server { ... })
server_name your-domain.com;
location / {
    proxy_pass http://localhost:3000;
    proxy_set_header Host $host;
    proxy_set_header X-Real-IP $remote_addr;
    proxy_set_header X-Forwarded-For $proxy_add_x_forwarded_for;
    proxy_set_header X-Forwarded-Proto $scheme;
}
nginx on 80/443 (public) forwards to your app on localhost:3000. Now you can close 3000 in the Security Group.
WebSocket / SSE upgrade
proxy_http_version 1.1;
proxy_set_header Upgrade $http_upgrade;
proxy_set_header Connection "upgrade";
Required for socket.io / ws; without it connections hang or drop.
Test config + reload
sudo nginx -t
sudo systemctl reload nginx
Always nginx -t before reload — a bad config won't reload but a restart on bad config leaves nginx down.

HTTPS / Certificate Management

Point DNS first
Create an A record for your domain → Elastic IP. Certbot's HTTP-01 challenge needs the domain resolving to this box on port 80.
Install certbot (Let's Encrypt)
sudo apt install certbot python3-certbot-nginx -y
Issue + auto-configure nginx
sudo certbot --nginx -d your-domain.com -d www.your-domain.com
Edits your server block to add 443/SSL and redirect 80→443 automatically. Free 90-day cert.
Auto-renewal
sudo certbot renew --dry-run
sudo systemctl status certbot.timer
Renewal is installed as a systemd timer/cron by default. Certs expire in 90 days — the timer renews at ~60.
Verify
curl -I https://your-domain.com
Ensure 443 is open in the Security Group, else HTTPS times out even with a valid cert.

Quick tips & gotchas

  • Security Group is the top cause of 'can't connect' — the OS may be fine but the port is closed. Check it first.
  • Public IP changes on stop/start; allocate an Elastic IP before pointing DNS or you'll re-edit records constantly.
  • chmod 400 the .pem and SSH as the AMI's user (ubuntu/ec2-user) — wrong perms or wrong user = 'Permission denied'.
  • Never bake the .pem into a repo or share it; you can't re-download it and it grants full server access.
  • A raw 'node index.js' dies on logout — use pm2 (or a systemd unit) so the app survives disconnects and reboots.
  • Front the app with nginx on 80/443 and proxy to localhost:PORT; then close the app port in the Security Group.
  • Run 'sudo nginx -t' before every reload; reload (not restart) on a bad config so a live server isn't taken down.
  • DNS must resolve to the box and port 80 must be open before running certbot — the HTTP-01 challenge needs both.
