The problem (pre-requisites: EVM immutability)

Deployed bytecode is frozen
You cannot patch code at an address. Each contract's state (staked balances, reward index) lives at its own address — there is no in-place edit.
Naive redeploy breaks everything
New address ⇒ every UI/integration/token allowance points at the old contract, and you'd have to migrate every staker's balance (gas-heavy, needs downtime). Registry/lookup adds a hop and still doesn't move state.
delegatecall = run their code, touch MY storage
// logic code runs, but writes to caller's (proxy's) storage;
// msg.sender & msg.value are preserved from the original call
(bool ok, ) = impl.delegatecall(
    abi.encodeWithSignature("stake(uint256)", amt)
);
// vs. plain call(): impl's OWN storage changes, msg.sender = proxy
This is the primitive the whole pattern rests on. Mixing up call vs delegatecall is the #1 conceptual error.
Why a constructor cannot initialize a proxy
A constructor runs once at IMPLEMENTATION deploy, against the implementation's storage. The proxy has separate storage and never sees it ⇒ proxy state stays zero. Fix: an initialize() called via delegatecall.

Initialize the staking contract (initializer, not constructor)

UUPS-upgradeable StakingV1 skeleton
contract StakingV1 is Initializable, OwnableUpgradeable, UUPSUpgradeable {
    using SafeERC20 for IERC20;
    IERC20 public stakingToken;
    mapping(address => uint256) public staked;
    uint256 public rewardRate;

    /// @custom:oz-upgrades-unsafe-allow constructor
    constructor() { _disableInitializers(); }   // lock the impl itself

    function initialize(address token, uint256 rate) public initializer {
        __Ownable_init(msg.sender);      // OZ v5: takes initialOwner
        __UUPSUpgradeable_init();
        stakingToken = IERC20(token);
        rewardRate   = rate;
    }

    function _authorizeUpgrade(address) internal override onlyOwner {}
    uint256[50] private __gap;           // reserve slots for future vars
}
`initializer` modifier lets the body run exactly once. Every parent must be init'd via its __X_init() (no constructors).
Stake / unstake logic
function stake(uint256 amt) external {
    staked[msg.sender] += amt;
    stakingToken.safeTransferFrom(msg.sender, address(this), amt);
}
function unstake(uint256 amt) external {
    staked[msg.sender] -= amt;               // reverts on underflow (0.8+)
    stakingToken.safeTransfer(msg.sender, amt);
}
User must approve() the PROXY address first. Effects-before-interaction ordering guards reentrancy.
initializer vs onlyInitializing vs reinitializer
modifier initializer          // top-level init, runs once (version 1)
function __Ownable_init() onlyInitializing  // parent init, callable during init
modifier reinitializer(uint64 v)  // one-time migration for version v
Parent __X_init() functions use onlyInitializing so they can be chained from your initializer without tripping the once-guard.

Proxy contract internals

Minimal ERC-1967 proxy (fallback + delegatecall)
contract Proxy {
    // keccak256('eip1967.proxy.implementation') - 1
    bytes32 constant SLOT =
      0x360894a13ba1a3210667c828492db98dca3e2076cc3735a920a3ca505d382bbc;
    constructor(address impl){ assembly { sstore(SLOT, impl) } }

    fallback() external payable {
        assembly {
            let impl := sload(SLOT)
            calldatacopy(0, 0, calldatasize())
            let ok := delegatecall(gas(), impl, 0, calldatasize(), 0, 0)
            returndatacopy(0, 0, returndatasize())
            switch ok
            case 0 { revert(0, returndatasize()) }
            default { return(0, returndatasize()) }
        }
    }
    receive() external payable {}
}
fallback catches any selector the proxy doesn't define and forwards raw calldata — that's how one proxy exposes the whole impl ABI, including v2 functions.
ERC-1967 reserved slots
impl   = keccak256('eip1967.proxy.implementation') - 1
admin  = keccak256('eip1967.proxy.admin')          - 1
beacon = keccak256('eip1967.proxy.beacon')         - 1
Pseudo-random slots keep proxy bookkeeping from colliding with the impl's sequential slots 0,1,2... Also what lets Etherscan auto-detect proxies.
Upgrade = repoint the slot (guarded)
function upgradeToAndCall(address newImpl, bytes memory data)
    external onlyOwner {
    assembly { sstore(SLOT, newImpl) }
    if (data.length > 0) newImpl.delegatecall(data);  // run migration
}
In UUPS this lives in the IMPLEMENTATION (via _authorizeUpgrade); in Transparent it lives in the proxy/ProxyAdmin.

Deploy & upgrade (OZ Upgrades plugin)

Deploy behind proxy
const F = await ethers.getContractFactory("StakingV1");
const proxy = await upgrades.deployProxy(
    F, [tokenAddr, rate],
    { kind: "uups", initializer: "initialize" }
);
await proxy.waitForDeployment();
console.log("proxy:", await proxy.getAddress());
Deploys impl + proxy and calls initialize() atomically. kind: "transparent" | "uups" | "beacon".
Upgrade + run migration atomically
const V2 = await ethers.getContractFactory("StakingV2");
await upgrades.upgradeProxy(proxyAddr, V2, {
    call: { fn: "reinitialize", args: [newRate] }
});
Proxy address, balances and approvals are unchanged. reinitialize runs once via upgradeToAndCall.
Validate layout before shipping (dry run)
await upgrades.validateUpgrade(proxyAddr, V2);   // throws on incompatible layout
await upgrades.forceImport(proxyAddr, F);        // adopt an existing proxy
Foundry equivalent
import {Upgrades} from "openzeppelin-foundry-upgrades/Upgrades.sol";
address p = Upgrades.deployUUPSProxy(
    "StakingV1.sol",
    abi.encodeCall(StakingV1.initialize, (token, rate)));
Upgrades.upgradeProxy(p, "StakingV2.sol",
    abi.encodeCall(StakingV2.reinitialize, (newRate)));
Beacon (fleet of identical vaults)
const beacon = await upgrades.deployBeacon(F);
const v = await upgrades.deployBeaconProxy(beacon, F, args);
await upgrades.upgradeBeacon(beacon, V2);  // upgrades ALL proxies at once

Storage layout & migration (fixing the problem)

Golden rule: append-only
Never reorder, remove, or change the type of an existing variable — slot N must mean the same thing in every version, or you silently corrupt balances (storage collision).
Add v2 vars AFTER existing ones (consume the gap)
contract StakingV2 is StakingV1 {
    uint256 public rewardPerBlock;   // appended = safe
    // WRONG: inserting before stakingToken shifts every later slot
    uint256[49] private __gap;        // shrink gap by #vars added
}
One-time data migration on upgrade
function reinitialize(uint256 newRate) public reinitializer(2) {
    rewardRate = newRate;   // runs exactly once, for version 2
}
reinitializer(n) must strictly increase across versions; called via upgradeToAndCall so upgrade + migration land in one tx.
Namespaced storage (ERC-7201) to isolate layout
/// @custom:storage-location erc7201:staking.main
struct MainStorage { mapping(address=>uint256) staked; uint256 rate; }
Modern alternative to __gap: puts all state under one hashed slot so inheritance order can't collide. Used by OZ v5 internals.

Security & production hardening (moving further)

Disable initializers on the implementation
constructor() { _disableInitializers(); }
Blocks the uninitialized-impl takeover (Parity multisig freeze): otherwise an attacker calls initialize() on the impl, becomes owner, and can selfdestruct/brick it.
Guard the upgrade path (UUPS)
function _authorizeUpgrade(address newImpl)
    internal override onlyOwner {}
If a new impl OMITS this / UUPSUpgradeable, upgradeability is permanently bricked. Always re-inherit UUPSUpgradeable and test the upgrade on a fork.
Upgrade key = drain key ⇒ multisig + timelock
// transfer proxy admin / owner to a Gnosis Safe behind a TimelockController
await proxy.transferOwnership(timelockAddr);
Whoever can upgrade can swap in malicious logic and drain all staked funds. Timelock gives users a window to exit before a live upgrade.
Transparent vs UUPS vs Beacon
Transparent: upgrade logic in proxy, routes by caller (admin vs user)
             + no bricking  - extra SLOAD/branch gas, needs ProxyAdmin
UUPS:        upgrade logic in impl (upgradeTo + _authorizeUpgrade)
             + cheapest/lightest proxy  - bricks if impl omits it
Beacon:      N proxies -> 1 beacon impl slot
             + upgrade whole fleet at once  - extra call/gas, SPOF
UUPS is the modern default. Beacon for factory-minted fleets (thousands of identical vaults).

Quick tips & gotchas

  • Constructors run against the IMPLEMENTATION's storage, so proxy state stays zero — always use an initialize() + `initializer` modifier and chain parents via __X_init().
  • Storage is append-only forever: never reorder/insert/retype existing vars. Reserve `uint256[50] __gap;` (or use ERC-7201 namespaced storage) and shrink the gap by the count of vars you add.
  • Call `_disableInitializers()` in the implementation's constructor — an uninitialized impl is the Parity-style takeover that permanently bricks the logic.
  • UUPS can be bricked: if a new implementation forgets to inherit UUPSUpgradeable / _authorizeUpgrade there's no upgrade path left. Test the upgrade itself on a mainnet fork.
  • Users must approve() the PROXY address, not the implementation — all funds and balances live in the proxy's storage.
  • The upgrade key can drain everything. Put it behind a multisig + TimelockController, never an EOA.
  • Run `upgrades.validateUpgrade()` (or the Foundry safety check) before every deploy to catch layout-incompatible changes.
  • Use reinitializer(n) with a strictly increasing version for one-time migrations, and call it atomically via upgradeToAndCall so upgrade + migration are one tx.
