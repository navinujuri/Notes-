Kubernetes Part 3 (Scaling) — Cheat Sheet

Cohort 2.0 · 9 lessons · quick reference

Scaling Basics & Manual Scaling

Two axes of scaling
Horizontal = more pod replicas (stateless, preferred). Vertical = bigger CPU/mem per pod (VPA, restarts pods). This lesson = horizontal.
Set replicas declaratively in the Deployment
spec:
  replicas: 3   # kubectl apply -f deploy.yaml
Source of truth. HPA overrides this field at runtime — don't fight it by re-applying replicas.
Scale imperatively (quick, ad-hoc)
kubectl scale deployment myapp --replicas=5
Watch replicas roll out
kubectl get deploy myapp -w
kubectl get pods -l app=myapp -o wide
'Horizontal pod accelerator' = Horizontal Pod Autoscaler (HPA)
Controller that auto-adjusts replicas based on observed metrics (CPU/mem/custom). Reconciles every ~15s.

Prerequisite: metrics-server

HPA + `kubectl top` need metrics-server running
kubectl apply -f https://github.com/kubernetes-sigs/metrics-server/releases/latest/download/components.yaml
No metrics-server => HPA shows targets as <unknown> and never scales.
Local clusters (minikube/kind/k3d) need --kubelet-insecure-tls
# minikube shortcut
minikube addons enable metrics-server
On kind, patch the deployment args to add --kubelet-insecure-tls or it CrashLoops on cert errors.
Verify it works
kubectl top nodes
kubectl top pods

HorizontalPodAutoscaler (HPA)

Create imperatively
kubectl autoscale deployment myapp \
  --cpu-percent=50 --min=2 --max=10
Declarative HPA (autoscaling/v2 — current API)
apiVersion: autoscaling/v2
kind: HorizontalPodAutoscaler
metadata:
  name: myapp-hpa
spec:
  scaleTargetRef:
    apiVersion: apps/v1
    kind: Deployment
    name: myapp
  minReplicas: 2
  maxReplicas: 10
  metrics:
  - type: Resource
    resource:
      name: cpu
      target:
        type: Utilization
        averageUtilization: 50
averageUtilization is a % of the pod's CPU *request*, NOT the limit. No request set => HPA can't compute % => won't scale.
Multiple / memory metrics
  - type: Resource
    resource:
      name: memory
      target:
        type: AverageValue
        averageValue: 200Mi
With several metrics, HPA computes desiredReplicas for each and takes the MAX.
Tune reaction speed (v2 behavior block)
  behavior:
    scaleDown:
      stabilizationWindowSeconds: 300
    scaleUp:
      stabilizationWindowSeconds: 0
Default scale-down window is 300s (prevents flapping). Scale-up is fast by default.
Inspect
kubectl get hpa
kubectl describe hpa myapp-hpa   # see events + current metric

The Scaling Formula

Core formula
desiredReplicas =
  ceil[ currentReplicas * (currentMetric / desiredMetric) ]
Worked example
3 pods, avg CPU = 90%, target = 50%
= ceil[ 3 * (90/50) ] = ceil[5.4] = 6 pods
Result is clamped to [minReplicas, maxReplicas].
Tolerance dead-zone (default 10%)
If |currentMetric/desiredMetric - 1| <= 0.1, HPA does nothing — avoids thrashing on tiny fluctuations.
Ratio scale-down example
6 pods, avg CPU = 20%, target = 50%
= ceil[ 6 * (20/50) ] = ceil[2.4] = 3 pods

Resource Requests & Limits

Set in the container spec (per container)
resources:
  requests:
    cpu: 100m
    memory: 128Mi
  limits:
    cpu: 250m
    memory: 256Mi
requests = guaranteed / used for scheduling + HPA %. limits = hard ceiling.
CPU units
1000m = 1 vCPU core   (100m = 0.1 core)
CPU is compressible: over limit => throttled, not killed.
Memory units
128Mi = 128 * 2^20 B   |   128M = 128 * 10^6 B
Memory is incompressible: exceed limit => OOMKilled. Use Mi/Gi (binary), not M/G.
QoS classes (drives eviction order)
Guaranteed = requests==limits on all containers. Burstable = requests<limits. BestEffort = none set (evicted first under pressure).
Requests power the scheduler AND the autoscaler
Scheduler sums requests vs node allocatable. Cluster autoscaler adds nodes when requests can't fit; HPA % is relative to requests.

Cluster Autoscaler (node scaling)

What it does
HPA scales pods; Cluster Autoscaler scales NODES. Distinct controller, usually cloud-managed (EKS/GKE/AKS node groups).
Scale-up trigger
Fires when pods are stuck Pending due to insufficient CPU/mem (unschedulable). Adds a node from the node group.
Scale-down trigger
Removes a node underutilized (default <50%) for ~10min if its pods can reschedule elsewhere. Blocked by pods with no controller, local storage, or restrictive PDBs.
Diagnose why a pod won't schedule
kubectl get pods --field-selector=status.phase=Pending
kubectl describe pod <pod>   # look at Events: 0/3 nodes available
No requests set => CA can't reason about capacity and may not add nodes correctly.
Full elastic stack
HPA (pods) + Cluster Autoscaler (nodes) + requests/limits together = end-to-end elastic scaling.

Lab: Load Test & Debug

Generate CPU load against a Service
kubectl run -it --rm load --image=busybox:1.28 --restart=Never -- \
  /bin/sh -c "while true; do wget -q -O- http://myapp; done"
Watch HPA react in real time
kubectl get hpa myapp-hpa -w
Live resource usage
kubectl top pods -l app=myapp --containers
Expose the deployment (target for HPA/load)
kubectl expose deployment myapp --port=80 --target-port=8080
Reset an HPA experiment
kubectl delete hpa myapp-hpa
kubectl scale deployment myapp --replicas=1

Quick tips & gotchas

  • No resource *requests* = HPA can't compute CPU % and stays <unknown>. Always set requests before enabling HPA.
  • HPA utilization % is measured against the request, never the limit — a common source of confusion.
  • HPA needs metrics-server; on kind/minikube add --kubelet-insecure-tls or it CrashLoops on TLS.
  • Don't `kubectl apply` a Deployment with a hardcoded `replicas:` while an HPA manages it — they fight. Omit replicas or use apply server-side with the field ignored.
  • Default 10% tolerance means small metric drifts won't trigger scaling — expected, not a bug.
  • Memory over limit => OOMKilled (incompressible); CPU over limit => throttled (compressible). Size memory limits carefully.
  • Scale-down is deliberately slow (300s stabilization window) to prevent flapping; scale-up is fast.
  • HPA scales pods, Cluster Autoscaler scales nodes — you usually need both; nodes only get added when pods are Pending for lack of capacity.
