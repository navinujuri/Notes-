Prisma Schema (core models)

Balance — money is Int (paise), never float
model Balance {
  id     Int  @id @default(autoincrement())
  userId Int  @unique
  amount Int
  locked Int
  user   User @relation(fields: [userId], references: [id])
}
userId @unique = one balance per user. Store smallest currency unit as Int; floats corrupt money.
OnRampTransaction + status enum
model OnRampTransaction {
  id        Int          @id @default(autoincrement())
  status    OnRampStatus
  token     String       @unique
  provider  String
  amount    Int
  startTime DateTime
  userId    Int
  user      User         @relation(fields: [userId], references: [id])
}
enum OnRampStatus { Success Failure Processing }
token is @unique — it's the idempotency key the webhook matches on.
p2pTransfer — two relations to User
model p2pTransfer {
  id         Int      @id @default(autoincrement())
  amount     Int
  timestamp  DateTime
  fromUserId Int
  fromUser   User     @relation("FromUserRelation", fields:[fromUserId], references:[id])
  toUserId   Int
  toUser     User     @relation("ToUserRelation",   fields:[toUserId],   references:[id])
}
Two FKs to the same table REQUIRE named relations, else prisma validate errors.
Back-relations on User
model User {
  // ...
  Balance           Balance[]
  OnRampTransaction OnRampTransaction[]
  sentTransfers     p2pTransfer[] @relation("FromUserRelation")
  receivedTransfers p2pTransfer[] @relation("ToUserRelation")
}

Onramp (Add Money) flow

Server action — create Processing txn + token
"use server"
export async function createOnRampTransaction(provider: string, amount: number) {
  const session = await getServerSession(authOptions);
  const userId = session?.user?.id;
  if (!userId) return { message: "User not logged in" };
  const token = Math.random().toString();   // real world: from bank
  await prisma.onRampTransaction.create({
    data: { userId: Number(userId), amount, status: "Processing", startTime: new Date(), provider, token }
  });
  return { message: "On ramp transaction added" };
}
Never credit Balance here — the client is untrusted. Credit only on the bank webhook.
Bank webhook — credit + flip status atomically
app.post("/hdfcWebhook", async (req, res) => {
  const { token, user_identifier: userId, amount } = req.body;
  await db.$transaction([
    db.balance.updateMany({
      where: { userId: Number(userId) },
      data:  { amount: { increment: Number(amount) } }
    }),
    db.onRampTransaction.updateMany({
      where: { token, status: "Processing" },   // idempotency guard
      data:  { status: "Success" }
    })
  ]);
  res.json({ message: "Captured" });
});
Array $transaction = atomic batch. Guard on status:Processing so a retried webhook can't double-credit.
increment / decrement atomic ops
data: { amount: { increment: n } }   // or { decrement: n }
Runs UPDATE ... SET amount = amount + n in the DB — no read-modify-write in JS.

P2P Transfer (send money) — the crux

Interactive $transaction with row lock
"use server"
export async function p2pTransfer(to: string, amount: number) {
  const session = await getServerSession(authOptions);
  const from = session?.user?.id;
  if (!from) return { message: "Error while sending" };
  const toUser = await prisma.user.findFirst({ where: { number: to } });
  if (!toUser) return { message: "User not found" };

  await prisma.$transaction(async (tx) => {
    await tx.$queryRaw`SELECT * FROM "Balance" WHERE "userId" = ${Number(from)} FOR UPDATE`;
    const fromBalance = await tx.balance.findUnique({ where: { userId: Number(from) } });
    if (!fromBalance || fromBalance.amount < amount) throw new Error("Insufficient funds");
    await tx.balance.update({ where: { userId: Number(from) },  data: { amount: { decrement: amount } } });
    await tx.balance.update({ where: { userId: toUser.id },     data: { amount: { increment: amount } } });
  });
}
SELECT ... FOR UPDATE takes a pessimistic lock so a concurrent transfer blocks until commit — kills the lost-update/double-spend race.
Array form vs interactive form
// array: atomic batch, no intermediate reads / branching
await prisma.$transaction([q1, q2]);
// interactive: read -> decide -> write on one connection
await prisma.$transaction(async (tx) => { /* ... */ });
Transfers need interactive: you must lock, read balance, branch on sufficiency, then write.
Why $queryRaw for the lock
Prisma has no first-class SELECT FOR UPDATE, so drop to tx.$queryRaw. Default Read Committed alone does NOT stop a read-check-write race — you need the explicit lock (or Serializable + retry).

P2P transactions table (ledger write)

Insert ledger row inside the SAME transaction
await prisma.$transaction(async (tx) => {
  await tx.$queryRaw`SELECT * FROM "Balance" WHERE "userId" = ${Number(from)} FOR UPDATE`;
  const fromBalance = await tx.balance.findUnique({ where: { userId: Number(from) } });
  if (!fromBalance || fromBalance.amount < amount) throw new Error("Insufficient funds");
  await tx.balance.update({ where:{ userId:Number(from) }, data:{ amount:{ decrement: amount } } });
  await tx.balance.update({ where:{ userId: toUser.id },   data:{ amount:{ increment: amount } } });
  await tx.p2pTransfer.create({
    data: { fromUserId: Number(from), toUserId: toUser.id, amount, timestamp: new Date() }
  });
});
Debit + credit + ledger insert commit or roll back together — balances and history can never diverge.
Migrate after adding the model
cd packages/db && npx prisma migrate dev --name add_p2p_transfer && npx prisma generate

Assignment — Frontend for P2P transactions

Server-side fetch scoped to session user
async function getP2PTransactions() {
  const session = await getServerSession(authOptions);
  const me = Number(session?.user?.id);
  const txns = await prisma.p2pTransfer.findMany({
    where: { OR: [ { fromUserId: me }, { toUserId: me } ] },
    orderBy: { timestamp: "desc" }
  });
  return txns.map(t => ({
    time: t.timestamp,
    amount: t.amount,
    type: t.fromUserId === me ? "Sent" : "Received"
  }));
}
Get the user id from getServerSession — NEVER from props/query string, or it's an IDOR. Scope with OR fromUserId/toUserId.
Async server component page
export default async function Page() {
  const transactions = await getP2PTransactions();
  return <P2PTransactions transactions={transactions} />;
}
App Router server component can await Prisma directly — no API route needed.
Render sent vs received
transactions.map(t => (
  <div key={t.time.toString()}>
    <span>{t.type}</span>
    <span>{t.type === "Sent" ? "- " : "+ "}{t.amount / 100} INR</span>
    <span>{t.time.toDateString()}</span>
  </div>
))
Divide by 100 for display since amount is stored in paise.

Quick tips & gotchas

  • Money is an Int in paise/cents — never float/double (binary floats can't represent decimal currency). Divide by 100 only at display time.
  • Onramp: credit Balance ONLY in the bank webhook, never in the click-time server action — the client is untrusted.
  • Make the webhook idempotent: match on the unique token and update only rows still in Processing; the bank retries on timeout and would otherwise double-credit.
  • Transfers MUST be an interactive $transaction with SELECT ... FOR UPDATE on the sender's balance row — array-form $transaction and default Read Committed don't stop the lost-update race.
  • Two FKs from p2pTransfer to User need named @relation("FromUserRelation"/"ToUserRelation") or prisma validate fails.
  • Write the p2pTransfer ledger row inside the same $transaction as the balance moves so audit trail and balances always agree.
  • Re-run npx prisma generate (and migrate dev) after every schema change, from packages/db — the apps import that generated client.
  • Scope the P2P history query to the session user id (getServerSession), never a client-supplied id — otherwise it's an IDOR leaking others' transactions.
