Docker Swarm — Cheat Sheet

Cohort 2.0 · 5 lessons · quick reference

Core Concepts & Architecture

Swarm
A cluster of Docker Engines running in 'swarm mode'. Built into the Docker CLI — no extra install.
Manager node
Maintains cluster state via Raft, schedules tasks, exposes the swarm API. Odd count recommended (1/3/5). Quorum = (N/2)+1. 3 managers tolerate 1 failure, 5 tolerate 2.
Worker node
Only runs tasks (containers). Cannot manage the swarm. Managers are also workers by default unless drained.
Service
The desired state definition: image + replicas + config. You declare intent; the swarm reconciles reality to match.
Task
A single scheduled slot that runs exactly one container. A service with 3 replicas = 3 tasks. Tasks are immutable — a failed task is replaced, not restarted in place.
Container
The actual running instance backing a task. Chain: Service -> Tasks -> Containers.
Declarative + self-healing
If a container dies or a node drops, the manager reschedules tasks to converge back to the declared replica count.
Raft consensus
Manager quorum stores cluster state. Losing quorum freezes management (existing tasks keep running, but you can't schedule changes).

Init & Join (Create a 2-Node Swarm)

Initialize swarm on node1 (manager)
docker swarm init --advertise-addr <MANAGER_IP>
--advertise-addr is required when the host has multiple IPs. Prints the worker join command with a token.
Join node2 as worker
docker swarm join --token SWMTKN-1-xxxx <MANAGER_IP>:2377
Run on node2. Copy-paste the exact command that `swarm init` printed.
Re-print join tokens
docker swarm join-token worker
docker swarm join-token manager
Worker and manager tokens differ. Use the manager token to add another manager.
Verify cluster
docker node ls
Only works on a manager. Leader shows `*` in MANAGER STATUS as 'Leader'.
Rotate a compromised token
docker swarm join-token --rotate worker
Leave the swarm
docker swarm leave           # worker
docker swarm leave --force   # last manager

Node Management

List nodes
docker node ls
Inspect a node
docker node inspect <NODE> --pretty
Promote / demote
docker node promote <NODE>   # worker -> manager
docker node demote <NODE>    # manager -> worker
Change availability
docker node update --availability drain <NODE>
active | pause | drain. `drain` evicts tasks and stops scheduling — use before maintenance. Common on managers to keep them workload-free.
Label nodes for placement
docker node update --label-add region=us-east <NODE>
Reference in constraints: --constraint 'node.labels.region==us-east'.
Remove a down node
docker node rm <NODE>          # node must have left first
docker node rm --force <NODE>

Deploy & Manage a Service

Create a service
docker service create --name web --replicas 3 -p 8080:80 nginx
-p 8080:80 publishes via the routing mesh — reachable on EVERY node's :8080, not just where a task runs.
List services & their replica health
docker service ls
REPLICAS column shows running/desired, e.g. 2/3 means one task not yet up.
List tasks of a service (where they run)
docker service ps web
Shows NODE, CURRENT STATE, and error/exit info for failed tasks.
Inspect service definition
docker service inspect web --pretty
Stream logs across all tasks
docker service logs -f web
Remove a service
docker service rm web
Stops and removes all its tasks/containers cluster-wide.
Global service (one task per node)
docker service create --mode global --name node-exporter prom/node-exporter
Use for agents/monitoring. No --replicas with global mode.

Scale, Update & Rollback

Scale replicas
docker service scale web=5
docker service update --replicas 5 web
Both work; `scale` can set multiple services at once: `docker service scale web=5 api=3`.
Rolling image update
docker service update --image nginx:1.27 web
Replaces tasks gradually per update policy — zero-downtime if configured well.
Tune the rollout policy
docker service update \
  --update-parallelism 2 \
  --update-delay 10s \
  --update-failure-action rollback \
  web
parallelism = tasks updated at once; delay = wait between batches.
Manual rollback
docker service rollback web
Reverts to the previous spec.
Change published port / env / mounts
docker service update --env-add LOG=debug --publish-add 9090:90 web

Stacks (Multi-Service Deploy via Compose)

Deploy a stack from a compose file
docker stack deploy -c docker-compose.yml myapp
Requires compose file version '3.x'. `docker run`/`build`-only options (build, depends_on ordering) are ignored — swarm needs pre-built images.
deploy block (swarm-specific config)
services:
  web:
    image: nginx:1.27
    ports: ["8080:80"]
    deploy:
      replicas: 3
      placement:
        constraints: ["node.role==worker"]
      restart_policy:
        condition: on-failure
      update_config:
        parallelism: 1
        delay: 10s
The `deploy:` key is only honored by `stack deploy`, ignored by `docker compose up`.
List / inspect stack
docker stack ls
docker stack services myapp
docker stack ps myapp
Tear down a stack
docker stack rm myapp

Networking, Secrets & Configs

Create an overlay network (multi-host)
docker network create --driver overlay backend
Overlay spans all nodes. Attach with `--network backend` on service create.
Service discovery via DNS + VIP
Services on the same overlay reach each other by service name (e.g. http://api:3000). Swarm load-balances across tasks behind a virtual IP.
Routing mesh vs host mode
--publish mode=host,target=80,published=8080
Default (mesh) exposes the port on every node. host mode binds only on nodes actually running a task.
Secrets
echo "s3cr3t" | docker secret create db_pass -
docker service create --secret db_pass mysql
Mounted at /run/secrets/db_pass (in-memory tmpfs). Managers store them encrypted in Raft.
Configs (non-sensitive files)
docker config create nginx_conf ./nginx.conf
docker service create --config source=nginx_conf,target=/etc/nginx/nginx.conf nginx

Quick tips & gotchas

  • Most swarm commands (node ls, service *, stack *) only run on a MANAGER. Workers can't query cluster state.
  • Run managers in odd numbers. Quorum is (N/2)+1 — 2 managers is worse than 1 (any single loss breaks quorum and freezes management).
  • --advertise-addr matters on multi-NIC/cloud hosts; picking the wrong IP means workers can't reach the manager on :2377.
  • Required ports between nodes: TCP 2377 (cluster mgmt), TCP+UDP 7946 (node discovery), UDP 4789 (overlay/VXLAN data). Open these in firewalls/security groups.
  • `docker service ps <svc> --no-trunc` reveals full error messages for stuck/failed tasks (image pull errors, constraint mismatches).
  • Swarm needs pre-built, pushed images — there is no build step across nodes. Push to a registry every node can pull from.
  • The `deploy:` key in a compose file is ignored by `docker compose up`; it only applies under `docker stack deploy`.
  • Drain your managers (`--availability drain`) in production so control-plane nodes don't also carry app workload.
