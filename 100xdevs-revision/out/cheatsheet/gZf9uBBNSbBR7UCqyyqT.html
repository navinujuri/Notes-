Concepts: ORMs & Prisma

What is an ORM
Object-Relational Mapper: maps DB tables <-> code objects/models. Write type-safe queries in JS/TS instead of raw SQL.
Why use one
Type safety, autocomplete, migrations, no hand-written SQL, DB-agnostic queries, guards against injection & schema drift.
What Prisma is
Next-gen Node/TS ORM = 3 parts: Prisma Schema (declarative model), Prisma Migrate (SQL migrations), Prisma Client (generated, type-safe query builder).
Two ORM styles
Prisma is schema-first (define models in schema.prisma). Contrast: code-first / Active-Record (TypeORM, Sequelize).

Install & Init (fresh app)

New project
mkdir app && cd app
npm init -y
npm i -D prisma typescript ts-node @types/node
npx tsc --init
Prisma CLI + init
npx prisma init
Creates prisma/schema.prisma and .env with DATABASE_URL. Default datasource is postgresql.
Pick DB at init
npx prisma init --datasource-provider postgresql
# or: sqlite | mysql | sqlserver | mongodb
SQLite is the zero-setup choice for learning.
Install client runtime
npm i @prisma/client
Runtime dep; prisma CLI stays a dev dep.

Config: datasource, generator, .env

Datasource block
datasource db {
  provider = "postgresql"
  url      = env("DATABASE_URL")
}
Generator block
generator client {
  provider = "prisma-client-js"
}
Postgres URL
DATABASE_URL="postgresql://user:pass@localhost:5432/mydb?schema=public"
SQLite URL
datasource db {
  provider = "sqlite"
  url      = "file:./dev.db"
}
Relative to the schema file location.
.env is git-ignored
prisma init auto-adds .env to .gitignore. Never commit real credentials.

Data model (schema.prisma)

Basic model
model User {
  id       Int    @id @default(autoincrement())
  email    String @unique
  name     String?
  age      Int    @default(0)
  createdAt DateTime @default(now())
}
? = nullable/optional. No ? = required (NOT NULL).
Field attributes
@id  @unique  @default(...)  @updatedAt
@map("col_name")  @db.VarChar(255)
@map renames the DB column; @@map renames the table.
Default helpers
@default(autoincrement())
@default(now())
@default(uuid())    // or cuid()
@default(false)
Enums
enum Role { USER ADMIN }
model User { role Role @default(USER) }
Composite / block attrs
@@unique([firstName, lastName])
@@index([email])
@@id([a, b])   // composite PK

Relationships

1-to-many
model User {
  id    Int    @id @default(autoincrement())
  todos Todo[]
}
model Todo {
  id       Int  @id @default(autoincrement())
  title    String
  user     User @relation(fields: [userId], references: [id])
  userId   Int
}
FK field (userId) lives on the 'many' side; the scalar + relation field both required.
1-to-1
model User    { id Int @id @default(autoincrement()) profile Profile? }
model Profile { id Int @id @default(autoincrement()) user User @relation(fields:[userId], references:[id]) userId Int @unique }
@unique on the FK enforces one-to-one.
Many-to-many (implicit)
model Post { id Int @id @default(autoincrement()) tags Tag[] }
model Tag  { id Int @id @default(autoincrement()) posts Post[] }
Prisma manages the hidden join table automatically.
Referential actions
user User @relation(fields:[userId], references:[id], onDelete: Cascade, onUpdate: Cascade)
Self-relation
model User {
  id       Int    @id @default(autoincrement())
  invitedBy   User?  @relation("invites", fields:[inviterId], references:[id])
  inviterId   Int?
  invited     User[] @relation("invites")
}

Migrate, generate, explore

Create + apply migration (dev)
npx prisma migrate dev --name init
Generates SQL in prisma/migrations, applies it, and regenerates the client.
Prototype without migration files
npx prisma db push
Pushes schema straight to DB (no migration history). Great for early iteration/SQLite.
(Re)generate client
npx prisma generate
Run after every schema change if you didn't use migrate dev. Outputs to node_modules/@prisma/client.
Prisma Studio (GUI)
npx prisma studio
Visual DB browser at localhost:5555 to explore/edit rows.
Introspect existing DB
npx prisma db pull
Reverse-engineers schema.prisma from an existing database.
Deploy migrations (prod)
npx prisma migrate deploy
Applies pending migrations non-interactively; no schema changes generated.
Reset DB (dev only)
npx prisma migrate reset
Drops + recreates + re-seeds. Destroys all data.

Prisma Client — CRUD & Todo functions

Instantiate
import { PrismaClient } from "@prisma/client";
const prisma = new PrismaClient();
Create ONE instance and reuse it — never new PrismaClient() per request.
Create
const todo = await prisma.todo.create({
  data: { title: "Learn Prisma", userId: 1 }
});
Read many + filter
await prisma.todo.findMany({
  where: { done: false, userId: 1 },
  orderBy: { createdAt: "desc" },
  take: 10, skip: 0
});
Read one
await prisma.user.findUnique({ where: { id: 1 } });
await prisma.user.findFirst({ where: { name: "Alice" } });
select vs include
// select: pick specific fields
{ select: { id: true, title: true } }
// include: pull relations
{ include: { user: true, todos: true } }
Can't use select and include at the same top level.
Update
await prisma.todo.update({
  where: { id: 1 },
  data: { done: true }
});
Delete
await prisma.todo.delete({ where: { id: 1 } });
await prisma.todo.deleteMany({ where: { userId: 1 } });
Upsert
await prisma.user.upsert({
  where: { email: "a@b.com" },
  update: { name: "Alice" },
  create: { email: "a@b.com", name: "Alice" }
});
Nested write (create w/ relation)
await prisma.user.create({
  data: {
    email: "a@b.com",
    todos: { create: [{ title: "first" }] }
  }
});
Transaction
await prisma.$transaction([
  prisma.todo.create({ data: { title: "a", userId: 1 } }),
  prisma.user.update({ where:{id:1}, data:{ name:"b" } })
]);

Expressify it (REST API)

Setup
npm i express
import express from "express";
import { PrismaClient } from "@prisma/client";
const app = express();
const prisma = new PrismaClient();
app.use(express.json());
GET todos
app.get("/todos", async (req, res) => {
  const todos = await prisma.todo.findMany();
  res.json(todos);
});
POST todo
app.post("/todos", async (req, res) => {
  const { title, userId } = req.body;
  const todo = await prisma.todo.create({ data: { title, userId } });
  res.json(todo);
});
PUT mark done
app.put("/todos/:id", async (req, res) => {
  const todo = await prisma.todo.update({
    where: { id: Number(req.params.id) },
    data: { done: true }
  });
  res.json(todo);
});
Listen
app.listen(3000, () => console.log("http://localhost:3000"));
Params are strings
req.params/query are strings — Number(req.params.id) before passing to an Int where clause or Prisma throws a type error.

Quick tips & gotchas

  • Regenerate the client after EVERY schema change: `prisma migrate dev` does it automatically, but `db push` / manual edits need `npx prisma generate`.
  • Use `db push` for fast prototyping (no history); switch to `migrate dev` once the schema stabilizes so you get versioned SQL migrations.
  • Instantiate PrismaClient once and reuse. In dev with hot-reload (Next.js/nodemon), cache it on globalThis to avoid connection-pool exhaustion.
  • `findUnique` only accepts fields that are @id or @unique; use `findFirst` for arbitrary where filters.
  • Optional (`?`) fields map to nullable columns; adding a required field to a table with existing rows needs a @default or the migration fails.
  • The FK scalar (e.g. userId) AND the relation field both live on the 'many' side — forgetting the scalar is the #1 relation error.
  • Never build queries via string concatenation; Prisma parameterizes everything, so pass values through `where`/`data` to stay injection-safe.
  • Keep DATABASE_URL in .env (git-ignored) and use `env("DATABASE_URL")` in the datasource — hardcoding the URL leaks credentials.
