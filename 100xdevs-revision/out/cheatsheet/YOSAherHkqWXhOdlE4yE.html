SQL databases — Cheat Sheet

Cohort 2.0 · 11 lessons · quick reference

SQL vs NoSQL — pick & justify

Relational (SQL): Postgres, MySQL, SQLite
Structured tables + fixed schema, ACID transactions, strong consistency. Default choice for most CRUD apps.
NoSQL: MongoDB (document), Redis (KV), Cassandra (wide-col), Neo4j (graph)
Flexible/denormalized schema, horizontal scale. Good for caching, huge write volume, unstructured data.
Why SQL over NoSQL
Enforced schema catches bad data at insert; JOINs avoid duplicating data; transactions give all-or-nothing correctness (money, inventory). Relationships are first-class.
When NoSQL wins
Rapidly changing/unknown schema, extreme horizontal scale, or key-value/cache workloads. Trade consistency for availability/speed.
ACID (why relational is safe)
Atomicity, Consistency, Isolation, Durability — the guarantee that a transaction fully commits or fully rolls back.

Setup — Postgres CLI

Run Postgres fast (Docker)
docker run -d --name pg -e POSTGRES_PASSWORD=mypass -p 5432:5432 postgres
Connect with psql
psql "postgresql://postgres:mypass@localhost:5432/postgres"
Full connection string format: postgresql://USER:PASS@HOST:PORT/DBNAME
Create / drop a database
CREATE DATABASE myapp;
DROP DATABASE myapp;
Or from shell: createdb myapp / dropdb myapp
psql meta-commands
\l          -- list databases
\c myapp    -- connect to db
\dt         -- list tables
\d users    -- describe table
\q          -- quit
Backslash commands are psql-only, not SQL.

DDL — tables & schema

CREATE TABLE with common types + constraints
CREATE TABLE users (
  id       SERIAL PRIMARY KEY,
  username VARCHAR(50) UNIQUE NOT NULL,
  email    VARCHAR(255) NOT NULL,
  age      INT CHECK (age >= 0),
  active   BOOLEAN DEFAULT true,
  created  TIMESTAMP DEFAULT NOW()
);
SERIAL = auto-increment int. Prefer GENERATED ALWAYS AS IDENTITY on PG10+. Use BIGSERIAL for large tables.
Core types
INT/BIGINT, SERIAL, VARCHAR(n)/TEXT, BOOLEAN, TIMESTAMP/TIMESTAMPTZ, NUMERIC(10,2) for money, DATE, JSONB, UUID.
Constraints
PRIMARY KEY, UNIQUE, NOT NULL, DEFAULT, CHECK(...), REFERENCES (FK). Constraints reject bad rows at write time.
Alter schema
ALTER TABLE users ADD COLUMN bio TEXT;
ALTER TABLE users DROP COLUMN bio;
ALTER TABLE users RENAME COLUMN username TO handle;
Index for fast lookups
CREATE INDEX idx_users_email ON users(email);
PK/UNIQUE are auto-indexed. Index columns you filter/JOIN on; each index slows writes.

DML — CRUD queries

INSERT (return the new row)
INSERT INTO users (username, email, age)
VALUES ('alice', 'a@x.com', 30)
RETURNING id;
RETURNING avoids a second SELECT to get the generated id.
SELECT with filter, sort, limit
SELECT id, username FROM users
WHERE active = true AND age > 18
ORDER BY created DESC
LIMIT 10 OFFSET 0;
UPDATE (always scope with WHERE)
UPDATE users SET active = false WHERE id = 5;
No WHERE = updates EVERY row. Same trap on DELETE.
DELETE
DELETE FROM users WHERE id = 5;
Aggregates + GROUP BY
SELECT age, COUNT(*) FROM users
GROUP BY age
HAVING COUNT(*) > 1;
WHERE filters rows before grouping; HAVING filters after aggregation.
Upsert
INSERT INTO users (username, email) VALUES ('alice','a@x.com')
ON CONFLICT (username) DO UPDATE SET email = EXCLUDED.email;

Node.js with the pg library

Install
npm install pg
Connect with a Pool (preferred)
const { Pool } = require('pg');
const pool = new Pool({
  connectionString: process.env.DATABASE_URL
});
Pool reuses connections across requests. A single Client is fine for scripts, not for a server.
Parameterized query (async/await)
const res = await pool.query(
  'SELECT * FROM users WHERE email = $1',
  ['a@x.com']
);
console.log(res.rows);
res.rows is an array of objects; res.rowCount is the count.
NEVER string-concat user input (SQL injection)
// BAD: `SELECT * FROM users WHERE email='${email}'`
// GOOD: pool.query('... WHERE email = $1', [email])
$1, $2 placeholders are the fix — the driver escapes safely.
Minimal Express endpoint
app.post('/users', async (req, res) => {
  const { username, email } = req.body;
  const q = `INSERT INTO users (username, email)
             VALUES ($1, $2) RETURNING id`;
  const { rows } = await pool.query(q, [username, email]);
  res.json({ id: rows[0].id });
});
Transaction (all-or-nothing)
const client = await pool.connect();
try {
  await client.query('BEGIN');
  await client.query('UPDATE acct SET bal=bal-100 WHERE id=1');
  await client.query('UPDATE acct SET bal=bal+100 WHERE id=2');
  await client.query('COMMIT');
} catch (e) {
  await client.query('ROLLBACK'); throw e;
} finally {
  client.release();
}
Must use a single checked-out client for a transaction, not pool.query.

Relationships

One-to-Many (user has many posts)
CREATE TABLE posts (
  id      SERIAL PRIMARY KEY,
  title   TEXT NOT NULL,
  user_id INT NOT NULL REFERENCES users(id) ON DELETE CASCADE
);
FK lives on the 'many' side. ON DELETE CASCADE removes child rows with the parent.
One-to-One
CREATE TABLE profiles (
  user_id INT PRIMARY KEY REFERENCES users(id),
  bio     TEXT
);
FK is also the PK (or UNIQUE) → at most one row per user.
Many-to-Many (join/junction table)
CREATE TABLE student_courses (
  student_id INT REFERENCES students(id),
  course_id  INT REFERENCES courses(id),
  PRIMARY KEY (student_id, course_id)
);
Composite PK prevents duplicate enrollments.
FK referential integrity
You can't insert a post with a user_id that doesn't exist, and can't delete a referenced user unless CASCADE/SET NULL is set.

Joins

INNER JOIN — only matching rows
SELECT u.username, p.title
FROM users u
JOIN posts p ON p.user_id = u.id;
Rows with no match on either side are dropped.
LEFT JOIN — keep all left rows
SELECT u.username, p.title
FROM users u
LEFT JOIN posts p ON p.user_id = u.id;
Users with zero posts still appear; p.title is NULL for them.
Many-to-many via junction
SELECT s.name, c.title
FROM students s
JOIN student_courses sc ON sc.student_id = s.id
JOIN courses c        ON c.id = sc.course_id;
JOIN types cheat
INNER = intersection. LEFT/RIGHT = keep all of one side. FULL = keep both. CROSS = every combination (cartesian).
Aggregate across a join
SELECT u.username, COUNT(p.id) AS post_count
FROM users u
LEFT JOIN posts p ON p.user_id = u.id
GROUP BY u.id, u.username;
LEFT JOIN + COUNT(child.id) correctly yields 0 for empty parents (COUNT(*) would give 1).

Quick tips & gotchas

  • Always parameterize ($1, $2) — never interpolate user input into SQL. This is the #1 way to prevent SQL injection.
  • UPDATE/DELETE without a WHERE clause hits every row. Test the WHERE with a SELECT first.
  • Use a connection Pool in servers, not a single Client — and always client.release() in a finally block.
  • Index columns you filter or JOIN on frequently, but not everything: each index adds write cost and storage.
  • Use TIMESTAMPTZ (not TIMESTAMP) and NUMERIC (not FLOAT) for money to avoid timezone bugs and rounding errors.
  • Transactions require one checked-out client with BEGIN/COMMIT/ROLLBACK — pool.query() runs each statement on a possibly different connection.
  • Prefer LEFT JOIN when you must keep parent rows with no children; INNER JOIN silently drops them.
  • Store DATABASE_URL in env vars, never hardcode credentials; load with dotenv in dev.
