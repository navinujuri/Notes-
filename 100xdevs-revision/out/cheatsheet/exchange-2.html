Backend Architecture & Data Flow

Order path (write)
client → API (LPUSH orders) → Engine (BRPOP)
     → match in-memory book
     → PUBLISH result to channel `id`   (API resolves HTTP)
     → LPUSH events to `db` queue        (async persist)
     → PUBLISH depth@X / trade@X         (WS fan-out)
One hot path (deterministic matching) separated from slow/best-effort concerns (DB, client fan-out).
Why single-threaded + in-memory engine
Matching needs a total order (price-time priority) → single thread kills lock contention/races + is deterministically replayable. In-mem = microsecond latency vs a DB round-trip per match.
Cost of that design
Throughput ceiling = 1 core; durability problem. Solve with snapshots + ordered event log, NOT more threads.
Crash recovery
Every order is on the durable Redis list before processing; snapshot the book periodically + replay events since the snapshot's sequence number. Downstream writes must be idempotent.
DB kept off hot path
// engine emits, never awaits a DB write
RedisManager.getInstance().pushMessage({ type: TRADE_ADDED, data });
Dedicated db/ processor drains the queue → TimescaleDB. Trade-off: eventual consistency; carry a monotonic seq for idempotent, ordered writes.

Redis List = Durable Order Queue

Why a LIST not pub/sub for orders
LPUSH/BRPOP is a durable, at-least-once work queue: messages persist until popped, gives backpressure, load-balances across consumers. Orders must never be silently dropped.
API side — enqueue order
await client.lPush("orders", JSON.stringify({ clientId: id, message }));
clientId is a unique per-request id used as the reply channel.
Engine side — blocking drain loop
while (true) {
  const res = await client.brPop("orders", 0); // blocks, no busy-spin
  if (res) engine.process(JSON.parse(res.element));
}
Prefer BRPOP (blocks) over a while-true RPOP busy-loop that pegs CPU. LPUSH + BRPOP = FIFO.
node-redis v4 client
import { createClient } from "redis";
const client = createClient();
await client.connect();

Redis Pub/Sub: Request-Reply + Fan-out

Sync REST over async engine (sendAndAwait)
sendAndAwait(message: MessageToEngine) {
  return new Promise((resolve) => {
    const id = getRandomClientId();
    this.client.subscribe(id, (msg) => {
      this.client.unsubscribe(id);        // avoid channel leak
      resolve(JSON.parse(msg));
    });
    this.client.lPush("orders", JSON.stringify({ clientId: id, message }));
  });
}
API subscribes to channel = unique id, then enqueues. Engine PUBLISHes result to `id`; promise resolves, HTTP responds.
Engine replies to the API
this.publisher.publish(clientId, JSON.stringify(payload));
Market-data fan-out channels
publish("depth@BTC_USDC", JSON.stringify(depth));
publish("trade@BTC_USDC",  JSON.stringify(trade));
Fire-and-forget, no persistence. Perfect for depth/trade (many subs, best-effort, only latest matters).
Pub/Sub is NOT for orders
No subscriber connected = message gone forever. Unacceptable for orders (use the list); fine for ephemeral market data.
Separate pub vs sub connection
A node-redis client in subscribe mode can't run normal commands — use one connection to publish/enqueue and a second to subscribe.

Matching Engine (in-memory Orderbook)

Engine.process — command dispatch
process({ message, clientId }) {
  switch (message.type) {
    case CREATE_ORDER:  /* match, reply, emit */ break;
    case CANCEL_ORDER:  break;
    case GET_DEPTH:     break;
    case ON_RAMP:       break;   // add balance
  }
}
Orderbook.addOrder
addOrder(order) {
  if (order.side === "buy") {
    const { executedQty, fills } = this.matchBid(order);
    order.filled = executedQty;
    if (executedQty < order.quantity) this.bids.push(order); // rest on book
    return { executedQty, fills };
  } /* else matchAsk + this.asks.push */
}
Fully filled → don't rest. Partial → remainder joins the book.
matchBid core
for (const ask of this.asks) {
  if (ask.price <= order.price && executedQty < order.quantity) {
    const q = Math.min(order.quantity - executedQty, ask.quantity - ask.filled);
    executedQty += q; ask.filled += q;
    fills.push({ price: ask.price, qty: q, tradeId: this.lastTradeId++, otherUserId: ask.userId, markerOrderId: ask.orderId });
  }
}
Price-time priority. Then drop fully-filled asks: this.asks = this.asks.filter(a => a.filled < a.quantity).
Balance check + lock
On buy: verify quote balance ≥ price*qty, lock it; on sell: lock base asset. Update both users' available/locked after each fill. Reject if insufficient.

Singletons

Explicit getInstance (lazy init)
export class RedisManager {
  private static instance: RedisManager;
  private constructor() { this.client = createClient(); this.client.connect(); }
  static getInstance() {
    if (!this.instance) this.instance = new RedisManager();
    return this.instance;
  }
}
Lazy = created on first use, can wait for config/env. Private constructor blocks `new`.
Free module-level singleton
// redis.ts
export const redisManager = new RedisManager();  // import cache dedupes per process
import/require caches the module → every importer gets the same instance. Eager init at import time. Enough for most cases.
What is a singleton here
RedisManager (one connection, avoids exhausting/interleaving), the Engine (one source of truth for the book), SubscriptionManager (one channel→socket map).
Downsides (say these in interview)
Controlled global → hidden coupling, hard to mock/reset in tests, init-order/circular-dep issues. DI is often cleaner.
CRITICAL: per-process only
Singleton ≠ cluster-wide single instance. PM2 cluster / k8s replicas each get their OWN heap + singleton. Fine for stateless managers; CATASTROPHIC if you run two matching engines on the same market.

WebSocket Fan-out Layer

SubscriptionManager (singleton) — maps
private subscriptions: Map<string, string[]> = new Map();        // userId → channels
private reverseSubscriptions: Map<string, string[]> = new Map(); // channel → userIds
reverse map lets a Redis message find every socket for that channel in O(1).
Subscribe once per channel, ref-count users
subscribe(userId, channel) {
  // ...push into both maps...
  if (this.reverse.get(channel)?.length === 1)   // first subscriber
    this.redisClient.subscribe(channel, this.onRedisMessage);
}
Add socket to an existing Redis sub instead of re-subscribing per user → no duplicate delivery.
Redis → local sockets (last-hop delivery)
onRedisMessage = (message, channel) => {
  this.reverse.get(channel)?.forEach(userId =>
    UserManager.getInstance().getUser(userId)?.emit(JSON.parse(message)));
};
Redis broker fans out cross-instance; the in-mem singleton is only the last hop to locally-connected sockets.
Unsubscribe on disconnect
// on ws 'close': remove from both maps
if (this.reverse.get(channel)?.length === 0)  // last client gone
  this.redisClient.unsubscribe(channel);
Skipping this leaks closures + dead-socket refs → heap grows until the process falls over.
Scale WS horizontally
Run N stateless WS servers, each subscribes to the Redis channels. Registry stays process-local; shared truth is in Redis. In-process EventEmitter instead of Redis = publish on A never reaches a client on B.

Message Types & Config at a Glance

API → Engine (MessageToEngine)
CREATE_ORDER | CANCEL_ORDER | GET_DEPTH | GET_OPEN_ORDERS | ON_RAMP
Engine → API (MessageFromOrderbook)
ORDER_PLACED | ORDER_CANCELLED | OPEN_ORDERS | DEPTH
Engine → db queue (DbMessage)
TRADE_ADDED | ORDER_UPDATE   // LPUSH to `db` list
Engine → ws channels (WsMessage)
depth@<MARKET> | trade@<MARKET>   // PUBLISH
Channel / queue names
orders          # Redis list (durable work queue)
<clientId>      # pub/sub reply channel per request
db              # Redis list (persistence)
depth@BTC_USDC  # pub/sub market data
trade@BTC_USDC
Persistence store
TimescaleDB (Postgres + time-series) for klines/trades; the db/ processor writes it, materialized views build candles.

Quick tips & gotchas

  • Orders travel via the durable LIST (LPUSH/BRPOP); market data via PUB/SUB. Never send an order over pub/sub — no subscriber = message lost forever.
  • Always add a timeout to sendAndAwait: a crashed engine mid-order leaves the HTTP request hanging. Surface it as a 5xx, and unsubscribe to avoid channel leaks.
  • Keep exactly ONE matching engine per market. A singleton is per-process, so two engine instances on the same market = double-spend / corrupted book.
  • Engine writes DB events fire-and-forget to a queue — it must never await disk I/O on the matching path. Carry a monotonic sequence number so replays/retries are idempotent (no duplicate rows on restart).
  • Prefer BRPOP over a while(true){ RPOP } busy-loop — the latter pegs a CPU core spinning on an empty queue.
  • In node-redis, a connection in subscribe mode can't issue normal commands — keep separate publisher and subscriber clients.
  • Unsubscribe from Redis + prune the maps on socket 'close'. Missing unsubscribe-on-disconnect leaks dead sockets and grows the heap until the process dies.
  • Wrap each pub/sub subscriber callback in try/catch so one throwing subscriber doesn't abort the whole fan-out; ref-count so you subscribe once per channel, not per user.
