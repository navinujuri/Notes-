Data Fetching: Server Components & Loaders

Server Component (default) — fetch directly, no useEffect
// app/users/page.tsx  (Server Component by default)
export default async function Page() {
  const res = await fetch("https://api.example.com/users", { cache: "no-store" });
  const users = await res.json();
  return <div>{users.map((u:any) => <p key={u.id}>{u.name}</p>)}</div>;
}
async components run on the server; secrets/DB calls stay off the client. No loading state needed — it streams.
React recap — client fetch needs useEffect + state
"use client";
import { useEffect, useState } from "react";
export default function Users() {
  const [data, setData] = useState<any[]>([]);
  useEffect(() => { fetch("/api/users").then(r=>r.json()).then(setData); }, []);
  return <>{data.map(u => <p key={u.id}>{u.name}</p>)}</>;
}
Runs after mount → waterfall + spinner flash. Prefer server fetch when possible.
fetch cache control (App Router extends fetch)
fetch(url)                              // cached (default, SSG-like)
fetch(url, { cache: "no-store" })       // never cache (SSR every req)
fetch(url, { next: { revalidate: 60 }})// ISR: revalidate every 60s
fetch(url, { next: { tags: ["users"] }})// tag → revalidateTag("users")
Next 15 default changed to uncached (no-store) unless you opt in.
loading.tsx — automatic route-level loader
// app/users/loading.tsx
export default function Loading() {
  return <p>Loading users...</p>;
}
Rendered instantly while the async page resolves — wraps the segment in a Suspense boundary for you.
Granular loading with <Suspense>
import { Suspense } from "react";
<Suspense fallback={<Spinner/>}>
  <SlowServerComponent />
</Suspense>
Stream part of the page while the rest loads. fallback shows until the child's await resolves.
Parallel vs sequential fetches
// sequential (slow): await a; await b;
// parallel (fast):
const [a, b] = await Promise.all([getUser(), getPosts()]);

API Routes (Route Handlers)

File convention — one route.ts per path
app/api/users/route.ts        // -> /api/users
app/api/users/[id]/route.ts   // -> /api/users/:id
Export named functions per HTTP verb: GET, POST, PUT, PATCH, DELETE.
GET handler + NextResponse
import { NextResponse } from "next/server";
export async function GET() {
  return NextResponse.json({ users: [] }, { status: 200 });
}
POST — read JSON body
export async function POST(req: Request) {
  const body = await req.json();          // { username, password }
  // ...validate & persist
  return NextResponse.json({ id: 1 }, { status: 201 });
}
req is the standard Web Request. Use req.formData() for form posts.
Dynamic params (Next 15: params is a Promise)
export async function GET(req: Request,
  { params }: { params: Promise<{ id: string }> }) {
  const { id } = await params;
  return NextResponse.json({ id });
}
Next 14: params is a plain object (no await). Next 15 made it async.
Query / search params
export async function GET(req: Request) {
  const { searchParams } = new URL(req.url);
  const q = searchParams.get("q");
}
Reading cookies / headers
import { cookies, headers } from "next/headers";
const token = (await cookies()).get("token")?.value; // async in Next 15
cookies()/headers() are async in Next 15, sync in 14.

Sign-up Flow (Frontend + Backend in one app)

Why co-locate — no CORS, call /api/* directly
Moving the backend into the Next app means same-origin requests; no separate Express server, no CORS config.
Frontend form (client component)
"use client";
import { useState } from "react";
import axios from "axios";
export default function Signup() {
  const [username, setU] = useState("");
  const [password, setP] = useState("");
  async function submit() {
    await axios.post("/api/signup", { username, password });
  }
  return (<div>
    <input onChange={e=>setU(e.target.value)} placeholder="email"/>
    <input type="password" onChange={e=>setP(e.target.value)}/>
    <button onClick={submit}>Sign up</button>
  </div>);
}
Any component with onClick/useState/hooks must start with "use client".
Backend signup handler
// app/api/signup/route.ts
import { NextResponse } from "next/server";
import { prisma } from "@/lib/prisma";
export async function POST(req: Request) {
  const { username, password } = await req.json();
  const user = await prisma.user.create({
    data: { username, password }   // hash in real apps (bcrypt)!
  });
  return NextResponse.json({ id: user.id });
}
Never store plaintext passwords in production — bcrypt.hash before create.

Databases with Prisma

Install & init
npm install prisma --save-dev
npm install @prisma/client
npx prisma init        # creates prisma/schema.prisma + .env
schema.prisma — datasource + model
datasource db {
  provider = "postgresql"
  url      = env("DATABASE_URL")
}
generator client { provider = "prisma-client-js" }

model User {
  id       Int    @id @default(autoincrement())
  username String @unique
  password String
}
Migrate & generate client
npx prisma migrate dev --name init   # create + apply migration, regen client
npx prisma generate                  # regen client only
npx prisma studio                    # GUI to browse data
Use `migrate deploy` in production/CI (non-interactive).
CRUD queries
await prisma.user.create({ data: { username, password } });
await prisma.user.findUnique({ where: { username } });
await prisma.user.findMany();
await prisma.user.update({ where: { id }, data: { password } });
await prisma.user.delete({ where: { id } });
.env — connection string
DATABASE_URL="postgresql://user:pass@host:5432/db"
Never commit .env. Prefix client-exposed vars with NEXT_PUBLIC_ (never for secrets).

Singleton Prisma Client

Problem — dev hot-reload spawns many clients
Next.js hot-reloads modules; `new PrismaClient()` per import exhausts DB connections ("too many clients").
Fix — cache on globalThis
// lib/prisma.ts
import { PrismaClient } from "@prisma/client";
const globalForPrisma = globalThis as unknown as {
  prisma?: PrismaClient;
};
export const prisma =
  globalForPrisma.prisma ?? new PrismaClient();
if (process.env.NODE_ENV !== "production")
  globalForPrisma.prisma = prisma;
Import { prisma } everywhere. In prod one instance per lambda; the global guard only matters in dev.

Server Actions

"use server" — run server code from a form, no API route
// app/actions.ts
"use server";
import { prisma } from "@/lib/prisma";
export async function signup(formData: FormData) {
  const username = formData.get("username") as string;
  await prisma.user.create({ data: { username, password: "..." } });
}
Directive at top of file (or inline in an async fn). Args/return are serialized over the wire.
Wire to a form (works without JS)
import { signup } from "./actions";
export default function Page() {
  return (<form action={signup}>
    <input name="username"/>
    <button type="submit">Go</button>
  </form>);
}
action={fn} passes FormData automatically — progressive enhancement.
Call from a client component
"use client";
import { signup } from "./actions";
<button onClick={() => signup(fd)}>Save</button>
You can import a server action into a client component and await it like a function.
Pending state with useFormStatus
"use client";
import { useFormStatus } from "react-dom";
function Submit() {
  const { pending } = useFormStatus();
  return <button disabled={pending}>{pending?"...":"Submit"}</button>;
}
Revalidate / redirect after mutation
import { revalidatePath } from "next/cache";
import { redirect } from "next/navigation";
// inside the action:
revalidatePath("/users");   // refresh cached data
redirect("/dashboard");     // navigate after success

Quick tips & gotchas

  • Server Components can't use hooks/state/onClick — add "use client" the moment you need interactivity or browser APIs.
  • Next 15 breaking changes: cookies(), headers(), params, and searchParams are now async (await them); fetch defaults to uncached.
  • Never expose secrets to the client: DB calls, API keys, and password hashing belong in Server Components, route handlers, or server actions — only NEXT_PUBLIC_* vars reach the browser.
  • Always use the singleton Prisma client from lib/prisma — importing `new PrismaClient()` per module crashes dev with connection-limit errors.
  • route.ts and page.tsx can't coexist in the same folder — a segment is either a page or an API route.
  • Server actions are the lightweight alternative to writing an /api route for form mutations — but return only serializable data and call revalidatePath to refresh stale caches.
  • Prefer fetching in Server Components over useEffect: it avoids client waterfalls, ships less JS, and needs no loading spinner (use loading.tsx / Suspense instead).
  • Hash passwords with bcrypt before storing — the raw-password examples are for learning only, never production.
