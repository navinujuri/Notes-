Intro to HTTP — Cheat Sheet

Cohort 3.0 · 14 lessons · quick reference

Fundamentals & Request/Response Model

What HTTP is
HyperText Transfer Protocol: stateless, text-based, application-layer protocol (layer 7) riding on TCP. Each request is independent — server keeps no memory between requests (that's why we bolt on cookies/tokens for state).
Why a protocol at all
A shared contract so any client (browser, curl, mobile app) and any server (Node, Go, nginx) can talk. Defines the exact bytes: method + path + version, headers, blank line, body.
Request/Response cycle
Client  --- REQUEST  --->  Server
        <--- RESPONSE ---
Client always initiates; server only replies. One request => exactly one response.
Raw HTTP/1.1 request wire format
GET /users/42 HTTP/1.1\r\nHost: api.example.com\r\nAccept: application/json\r\n\r\n
Request line, headers, blank line (CRLF CRLF), then optional body. Blank line separates headers from body.
Raw HTTP response wire format
HTTP/1.1 200 OK\r\nContent-Type: application/json\r\nContent-Length: 27\r\n\r\n{"id":42,"name":"Alice"}
Status line, headers, blank line, body.
URL anatomy
https://api.example.com:443/users?id=42#top
  |      |               |    |      |     |
scheme  host           port  path  query  fragment
Fragment (#top) is client-only — never sent to the server.

Domain, IP & Ports

Domain -> IP (DNS)
nslookup example.com
dig +short example.com
Humans use names (example.com); machines route by IP (93.184.x.x). DNS resolves name -> IP before the TCP connection opens.
localhost / loopback
127.0.0.1  ==  localhost  ==  ::1 (IPv6)
Traffic never leaves your machine.
Ports = which process on the host
IP finds the machine; port finds the app on it. 0-65535; <1024 are privileged/well-known.
Common ports
80    HTTP
443   HTTPS
3000  Node/React dev (convention)
5432  Postgres
27017 MongoDB
6379  Redis
80 and 443 are implied by http:// and https:// so you rarely type them.
Check / free a port (Windows)
netstat -ano | findstr :3000
taskkill /PID <pid> /F
macOS/Linux: lsof -i :3000 then kill -9 <pid>.

HTTP Methods (Verbs)

GET — read, no body
curl https://api.example.com/users
Safe + idempotent. Params go in the query string, not the body. Cacheable.
POST — create / non-idempotent action
curl -X POST -H 'Content-Type: application/json' -d '{"name":"Alice"}' https://api.example.com/users
Sends a body. Calling twice creates two resources.
PUT — replace/upsert (idempotent)
curl -X PUT -d '{"name":"Bob"}' https://api.example.com/users/42
Full replacement of the resource. Same call N times = same end state.
PATCH — partial update
curl -X PATCH -d '{"name":"Bob"}' https://api.example.com/users/42
Only send the fields you change.
DELETE — remove
curl -X DELETE https://api.example.com/users/42
Idempotent — deleting an already-deleted resource still ends deleted.
OPTIONS — capabilities / CORS preflight
Browsers auto-fire OPTIONS before non-simple cross-origin requests to check CORS headers. You don't call it by hand.
Idempotency at a glance
Safe:       GET, HEAD, OPTIONS
Idempotent: GET, PUT, DELETE, HEAD
Neither:    POST, PATCH
The method is a convention — the server must actually enforce the semantics.

Status Codes

Class meanings (first digit)
1xx  informational
2xx  success
3xx  redirect
4xx  client error (your fault)
5xx  server error (their fault)
2xx success
200 OK              GET/PUT/PATCH ok

204 No Content      success, empty body (e.g. DELETE)
3xx redirect
301 Moved Permanently
302 Found (temp)

4xx client error

404 Not Found

401 = who are you? 403 = I know you, no.
5xx server error

503 Service Unavailable     down/overloaded
504 Gateway Timeout

Headers, Body & Routes

What headers are
Key:value metadata about the request/response — not the payload itself. Case-insensitive keys.
Common request headers
Host: api.example.com
Content-Type: application/json
Accept: application/json
Authorization: Bearer <token>
User-Agent: curl/8.4.0
Cookie: session=abc123
Common response headers
Content-Type: application/json; charset=utf-8
Content-Length: 27
Set-Cookie: session=abc123; HttpOnly
Cache-Control: max-age=3600
Location: /users/42   (with 201/3xx)
Body — the actual payload
Present on POST/PUT/PATCH requests and most responses. Format is declared by Content-Type. GET/DELETE typically have no request body.
Content-Type examples
application/json
application/x-www-form-urlencoded
multipart/form-data      (file uploads)
text/html
Routes = method + path
GET    /users        list
GET    /users/:id    read one
POST   /users        create
PUT    /users/:id    replace
DELETE /users/:id    delete
Same path, different method = different route. Path params (:id) vs query (?sort=asc) vs body serve different jobs.

Clients: curl, Postman, Browser

curl — verbose, see full exchange
curl -v https://api.example.com/users
-v prints request (>) and response (<) lines including headers.
curl — JSON POST with auth
curl -X POST https://api.example.com/users \
  -H 'Content-Type: application/json' \
  -H 'Authorization: Bearer TOKEN' \
  -d '{"name":"Alice"}'
curl — headers only / status only
curl -I https://example.com                       # HEAD, headers only
curl -s -o /dev/null -w '%{http_code}\n' URL      # just the code
On Windows use -o NUL instead of /dev/null.
Browser as a client
URL bar / links fire GET; <form method="post"> fires POST. Inspect everything in DevTools > Network (see method, status, headers, payload, timing). fetch() from console for ad-hoc calls.
Postman
GUI client: pick method, set URL, add Headers/Params/Body tabs, save into Collections, use {{variables}} for envs. Great for auth flows and sharing; curl is faster for scripting/CI.

Writing HTTP in JavaScript

Bare Node.js server (no deps)
import http from 'node:http';
const server = http.createServer((req, res) => {
  if (req.method === 'GET' && req.url === '/health') {
    res.writeHead(200, { 'Content-Type': 'application/json' });
    return res.end(JSON.stringify({ ok: true }));
  }
  res.writeHead(404).end('Not Found');
});
server.listen(3000, () => console.log('http://localhost:3000'));
req is a readable stream — body arrives in chunks, not as a ready string.
Reading a JSON request body (raw Node)
let data = '';
req.on('data', c => data += c);
req.on('end', () => {
  const body = JSON.parse(data || '{}');
  // ...use body
});
Frameworks/body-parsers do this for you.
Express server (idiomatic)
import express from 'express';
const app = express();
app.use(express.json());              // parse JSON bodies

app.get('/users/:id', (req, res) => {
  res.json({ id: req.params.id });     // sets Content-Type + 200
});
app.post('/users', (req, res) => {
  res.status(201).json(req.body);
});
app.listen(3000);
req.params (path), req.query (?a=b), req.body (parsed). res.json() serializes + sets header. Order of app.use matters.
Client: fetch (browser + Node 18+)
const res = await fetch('https://api.example.com/users', {
  method: 'POST',
  headers: { 'Content-Type': 'application/json' },
  body: JSON.stringify({ name: 'Alice' }),
});
if (!res.ok) throw new Error(`HTTP ${res.status}`);
const data = await res.json();
fetch does NOT reject on 4xx/5xx — always check res.ok. body must be a string; JSON.stringify it yourself.

Quick tips & gotchas

  • HTTP is stateless: the server forgets you between requests. State (login) is faked via cookies/tokens sent on every request.
  • fetch() only rejects on network failure — a 404 or 500 still resolves. Guard with `if (!res.ok)` before parsing.
  • Query string = filtering/sorting/pagination (GET); request body = the resource payload (POST/PUT/PATCH). Don't put secrets in the query string — it lands in logs and history.
  • 401 vs 403: 401 = not authenticated (login), 403 = authenticated but not permitted. 404 is often used to hide existence of a resource from unauthorized users.
  • Set Content-Type correctly or the other side mis-parses the body; a POST with JSON but no `Content-Type: application/json` often arrives as undefined/empty on the server.
  • The method (GET/POST/...) is only a convention — the server decides what actually happens. Keep GET side-effect-free and POST non-idempotent to match caches, proxies, and retries.
  • Port already in use (EADDRINUSE) = a previous process is still bound. Find and kill it: `netstat -ano | findstr :3000` then `taskkill /PID <pid> /F`.
  • CORS is a browser rule, not an HTTP error: cross-origin requests need the server to return Access-Control-* headers. curl/Postman ignore CORS, so 'works in Postman, fails in browser' is almost always CORS.
