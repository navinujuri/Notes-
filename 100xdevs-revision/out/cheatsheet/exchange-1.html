Building an Exchange - Part 1 — Cheat Sheet

Cohort 2.0 · 9 lessons · quick reference

Core Concepts

Exchange = order matching engine that pairs buyers with sellers of an asset at an agreed price
It never trades; it only matches counterparties and settles balances.
Market / trading pair — always base_quote
SOL_USDC  ->  base = SOL (what you buy/sell), quote = USDC (what you price it in)
price 120.5 means 1 SOL = 120.5 USDC
Buy = spend quote to get base. Sell = give base to get quote.
Order book — resting bids (buys) and asks (sells)
asks (sell)  121.0 x 1.5   <- lowest ask = best sell
             120.8 x 4.0
----------- spread -----------
bids (buy)   120.5 x 3.2   <- highest bid = best buy
             120.4 x 10
Bids sorted price DESC, asks price ASC. Best prices sit at the top.
Spread & mid price
spread   = bestAsk - bestBid          // 121.0 - 120.5 = 0.5
midPrice = (bestAsk + bestBid) / 2
Depth — cumulative quantity available at each price level
Deep book = lots of resting liquidity = small price impact per trade.

Jargon

Bid
Highest price a buyer will pay.
Ask / Offer
Lowest price a seller will accept.
Maker
Order that rests on the book and adds liquidity (usually a limit order that doesn't cross). Lower fees.
Taker
Order that matches an existing resting order and removes liquidity. Higher fees.
Fill
A (partial or full) execution of an order against a counterparty at a price.
Slippage
Diff between expected price and actual avg fill price when eating through the book.
Ticker
24h rolling stats for a market: last price, high, low, volume, % change.
Kline / Candlestick
OHLCV bucketed by interval (1m, 1h, 1d). Open/High/Low/Close/Volume.
Liquidity
How much can be traded without moving price. Provided by makers.

Order Types

Limit — execute at a set price or better; rests on book if it doesn't cross
{ side: "buy", price: "120.5", quantity: "2", type: "limit" }
Buy limit fills at <= price; sell limit fills at >= price. Maker if it rests.
Market — fill immediately at best available price(s), no price guarantee
{ side: "buy", quantity: "2", type: "market" }  // no price field
Always a taker. Walks the book -> slippage. Lock funds by worst-case, not price.
IOC (Immediate-Or-Cancel) — fill whatever it can now, cancel the remainder
{ side: "buy", price: "120.5", quantity: "5", type: "ioc" }
Never rests on the book. Partial fills allowed.
Stop / Stop-limit — dormant until trigger price hit, then becomes market/limit
Not held in the main book; kept in a trigger set and injected when price crosses.

Order Lifecycle (how an order is made)

End-to-end flow
1. Client -> POST /api/v1/order (REST)
2. API validates, LPUSH job to Redis queue with a unique clientId
3. Engine RPOP job -> checkAndLockFunds -> match against book
4. Engine creates fills, updates balances, adjusts book
5. Engine PUBLISH depth/trade/ticker to pubsub channels
6. Engine PUBLISH result to channel `clientId`
7. API (subscribed to clientId) resolves promise -> HTTP response
8. DB processor consumes pubsub -> persists to TimescaleDB
Lock funds before matching
// buy  -> lock quote:  price * qty  (USDC)
// sell -> lock base:   qty          (SOL)
if (side==="buy" && bal[quote].available < price*qty) throw "insufficient funds";
bal[quote].available -= price*qty;
bal[quote].locked    += price*qty;
Reject early on insufficient balance so the engine never produces an invalid state.
Matching (price-time priority)
matchBid(order){
  for (const ask of this.asks){                 // asks sorted price ASC
    if (ask.price > order.price) break;          // no cross -> stop
    const qty = Math.min(order.qty - filled, ask.qty - ask.filled);
    filled += qty; ask.filled += qty;
    fills.push({ price: ask.price, qty, tradeId: ++this.tid, markerOrderId: ask.id });
  }
}
Same price -> FIFO (earliest order first). Remove fully-filled resting orders.
Fill shape
interface Fill { price: string; qty: number; tradeId: number;
                 otherUserId: string; markerOrderId: string; }

Architecture

Service split — API is stateless, Engine is the single source of truth
Frontend(Next.js)
   |  REST + WS
   v
API Server (Express, stateless, N replicas)
   |  LPUSH "messages"          ^ SUBSCRIBE clientId
   v                            |
[ Redis ]  --queue--> Engine (single instance, in-mem order book)
   ^  PUBLISH depth@ / trade@ / ticker@   |
   |                                      v
WS Server (fan-out)              DB Processor -> TimescaleDB
Engine MUST be a single process so the in-memory book stays consistent.
Redis as queue (durable, ordered, backpressure)
// API produces
redis.lPush("messages", JSON.stringify({ clientId, message }));
// Engine consumes (blocking pop)
const job = await redis.brPop("messages", 0);
Redis pub/sub for broadcast (trades, depth deltas, ticker, RPC replies)
redis.publish(`depth@${market}`, JSON.stringify(depthDelta));
redis.publish(clientId, JSON.stringify(orderResult)); // request/response over pubsub
Snapshot & recovery
// engine periodically flushes state
fs.writeFileSync("./snapshot.json", JSON.stringify({ orderbooks, balances }));
On boot, load snapshot then replay any queued messages. Keeps restarts safe.

GET /api/v1/klines?symbol=SOL_USDC&interval=1m&startTime=..&endTime=..

Klines materialized view (TimescaleDB)
CREATE MATERIALIZED VIEW klines_1m
WITH (timescaledb.continuous) AS
SELECT time_bucket('1 minute', time) AS bucket,
       first(price,time) AS open, max(price) AS high,
       min(price)  AS low,        last(price,time) AS close,
       sum(quantity) AS volume
FROM trades GROUP BY bucket;
Continuous aggregate auto-refreshes; don't compute OHLCV per-request in the API.

WebSocket Streams

Subscribe / unsubscribe protocol
ws.send(JSON.stringify({ method: "SUBSCRIBE",
  params: ["depth@SOL_USDC", "trade@SOL_USDC", "ticker@SOL_USDC"] }));
ws.send(JSON.stringify({ method: "UNSUBSCRIBE", params: ["depth@SOL_USDC"] }));
depth@ stream — incremental book deltas
{ e:"depth", s:"SOL_USDC",
  b:[["120.5","3.2"],["120.4","0"]],   // qty 0 => remove level
  a:[["121.0","1.5"]] }
Send deltas, not full book, every tick. qty "0" means delete that price level.
trade@ stream — each execution
{ e:"trade", s:"SOL_USDC", p:"120.5", q:"2", t:9931, m:false }
m = buyer is maker (side that was resting).
ticker@ stream — rolling 24h stats push
{ e:"ticker", s:"SOL_USDC", c:"120.5", h:"125", l:"118", v:"40213" }
Server maps Redis pubsub -> subscribed sockets
redis.subscribe(`trade@${market}`, (msg) =>
  subscribers[`trade@${market}`].forEach(s => s.send(msg)));
WS server holds subscription map only; it never touches the order book.

Coding the API Server

Setup
npm init -y
npm i express redis uuid
npm i -D typescript @types/express @types/node ts-node
Message type constants (shared with engine)
export const CREATE_ORDER = "CREATE_ORDER";
export const CANCEL_ORDER = "CANCEL_ORDER";
export const GET_DEPTH    = "GET_DEPTH";
export const GET_OPEN_ORDERS = "GET_OPEN_ORDERS";
export const ON_RAMP      = "ON_RAMP";
RedisManager — request/response over queue + pubsub (singleton)
class RedisManager {
  private client; private publisher;
  static getInstance(){ return this.instance ??= new RedisManager(); }
  constructor(){ this.client=createClient(); this.publisher=createClient();
    this.client.connect(); this.publisher.connect(); }

  sendAndAwait(message){
    return new Promise((resolve)=>{
      const id = randomUUID();
      this.client.subscribe(id, (msg)=>{
        this.client.unsubscribe(id);
        resolve(JSON.parse(msg));
      });
      this.publisher.lPush("messages", JSON.stringify({ clientId:id, message }));
    });
  }
}
One channel per request keyed by uuid; engine publishes the reply to that channel.
Order route wiring
app.post("/api/v1/order", async (req,res)=>{
  const { market, price, quantity, side, userId } = req.body;
  const r = await RedisManager.getInstance().sendAndAwait({
    type: CREATE_ORDER,
    data: { market, price, quantity, side, userId }
  });
  res.json(r.payload);           // { orderId, executedQty, fills }
});
Depth route (read-only query to engine)
app.get("/api/v1/depth", async (req,res)=>{
  const r = await RedisManager.getInstance().sendAndAwait({
    type: GET_DEPTH, data: { market: req.query.symbol }
  });
  res.json(r.payload);
});
Engine consumer loop (separate process)
while (true) {
  const res = await redis.rPop("messages");
  if (!res) continue;
  const { clientId, message } = JSON.parse(res);
  Engine.getInstance().process({ clientId, message });
}

Quick tips & gotchas

  • Engine is single-threaded and single-instance — never horizontally scale it, or the in-memory order book forks. Scale the stateless API/WS layers instead.
  • Store price & quantity as strings and do integer math in the smallest unit (lamports/cents). Floats silently corrupt balances.
  • Match price-time priority: best price first, FIFO within a price level. Break the match loop the moment the resting price stops crossing.
  • Lock funds up front (buy locks quote = price*qty, sell locks base = qty) and reject insufficient balance before matching — the engine must never emit an invalid state.
  • Push depth as incremental deltas over WS (qty "0" = remove level), not the full book each tick — full snapshots kill bandwidth on active markets.
  • Keep the engine off the DB. It publishes trades/fills to pub/sub; a separate DB processor persists them so matching stays microsecond-fast.
  • Use a Redis list as the job queue (LPUSH from API, BRPOP in engine) for ordering + durability + natural backpressure; use pub/sub only for broadcast and RPC replies.
  • Snapshot the order book + balances to disk periodically and reload on boot; otherwise an engine restart loses every resting order.
