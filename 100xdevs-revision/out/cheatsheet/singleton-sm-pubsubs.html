Singleton Pattern, Backend State management and Pub Subs — Cheat Sheet

Cohort 2.0 · 6 lessons · quick reference

Stateful vs Stateless Backends

Stateless — no memory between requests; every request self-contained
// REST handler: state lives in DB/Redis, not RAM
app.get('/balance/:id', async (req, res) => {
  const bal = await db.balance(req.params.id); // fetch every time
  res.json({ bal });
});
Scales horizontally trivially — any instance can serve any request. Default choice for HTTP APIs.
Stateful — holds data in process memory across requests
// WebSocket game server keeps rooms in RAM
const rooms = new Map<string, Game>();
wss.on('connection', ws => rooms.get(roomId)?.addPlayer(ws));
Used for real-time (WS), low-latency, ephemeral data. WS connection object CANNOT be serialized to a DB — must stay in memory.
The scaling problem: N instances don't share memory
Instance A's `rooms` Map is invisible to instance B. Two players on different pods can't see each other → need sticky sessions or a shared bus (Redis pub/sub).
Sticky sessions — pin a client to one instance
# nginx: hash on client IP so a socket always hits same backend
upstream ws { ip_hash; server a:8080; server b:8080; }
Band-aid for stateful WS. Breaks on pod restart; doesn't help cross-room broadcast. Real fix = pub/sub layer.

State in the JS Process

Module-level variable = persists for process lifetime
// counter.ts
let count = 0;
export const inc = () => ++count; // survives across requests
Node keeps the module loaded; the closure variable lives as long as the process.
Single-threaded event loop → no locks for simple state
let seats = 100;
app.post('/book', (req, res) => {
  if (seats > 0) { seats--; res.send('ok'); } // no race: JS is single-threaded
  else res.send('sold out');
});
Safe because no two handlers run truly in parallel. BUT an `await` in the middle yields the loop — check-then-act across an await CAN race.
Restart / redeploy wipes everything
In-memory state is ephemeral. Never keep durable data (payments, user records) only in RAM.
worker_threads / cluster do NOT share memory
Each worker has its own heap → its own copy of your 'singleton'. Memory sharing needs SharedArrayBuffer or an external store.

Classes & Singleton Pattern (TS)

Class basics — fields, private, constructor, methods
class UserManager {
  private users: Map<string, User> = new Map();
  addUser(u: User) { this.users.set(u.id, u); }
  getUser(id: string) { return this.users.get(id); }
}
Classic Singleton — private ctor + static getInstance()
class Store {
  private static instance: Store;
  private state = new Map<string, number>();
  private constructor() {}          // blocks `new Store()`
  static getInstance(): Store {
    if (!Store.instance) Store.instance = new Store();
    return Store.instance;
  }
  set(k: string, v: number) { this.state.set(k, v); }
}
const s = Store.getInstance();      // always same object
Lazy init. Guarantees one instance app-wide within a single process.
Idiomatic Node singleton — export a live instance
// store.ts
class Store { /* ... */ }
export const store = new Store();

// anywhere.ts
import { store } from './store'; // Node module cache → same instance
Simplest and most common in Node/TS. Module resolution caches the module, so every importer gets the one instance.
Why singleton here: one shared source of truth for state
Connection pool, config, in-memory store, pub/sub client — things that must be exactly one per process.

Pub/Sub — Concept & In-Process

Model: publishers → topic/channel → subscribers (decoupled)
Publisher doesn't know who listens. Many subscribers per channel. Fire-and-forget, no reply.
In-process bus — Node EventEmitter
import { EventEmitter } from 'node:events';
const bus = new EventEmitter();
bus.on('order:created', (o) => sendEmail(o));   // subscribe
bus.on('order:created', (o) => updateStats(o)); // 2nd subscriber
bus.emit('order:created', { id: 42 });          // publish
Only works within ONE process. Listeners run synchronously in registration order.
Cross-process pub/sub — Redis (the scaling fix)
import { createClient } from 'redis';
const pub = createClient();  const sub = createClient();
await pub.connect(); await sub.connect();
await sub.subscribe('room:1', (msg) => broadcast(JSON.parse(msg)));
await pub.publish('room:1', JSON.stringify({ move: 'e4' }));
MUST use two clients: a connection in subscribe mode can't run normal commands. This lets instance A's publish reach subscribers on instance B.

Pub/Sub + Singleton — Implementation

Singleton Redis manager (one pub, one sub per process)
class PubSubManager {
  private static instance: PubSubManager;
  private pub = createClient();
  private sub = createClient();
  // channel -> local sockets interested in it
  private subs = new Map<string, WebSocket[]>();
  private constructor() { this.pub.connect(); this.sub.connect(); }
  static getInstance() {
    if (!this.instance) this.instance = new PubSubManager();
    return this.instance;
  }
}
Singleton so the whole process shares ONE Redis pub/sub pair + one subscription registry.
Subscribe local client, and subscribe Redis only on first interest
subscribe(userSock: WebSocket, channel: string) {
  const list = this.subs.get(channel) ?? [];
  if (list.length === 0)                 // first subscriber on THIS node
    this.sub.subscribe(channel, (msg) => this.fanout(channel, msg));
  this.subs.set(channel, [...list, userSock]);
}
Only call Redis SUBSCRIBE once per channel per instance; track local sockets in the Map.
Fan out Redis message to local sockets
private fanout(channel: string, raw: string) {
  this.subs.get(channel)?.forEach(ws => ws.send(raw));
}
publish(channel: string, payload: unknown) {
  this.pub.publish(channel, JSON.stringify(payload));
}
Any instance publishes → Redis broadcasts → every instance's fanout pushes to its own connected clients. Cross-node broadcast solved.
Unsubscribe + cleanup on disconnect
unsubscribe(userSock: WebSocket, channel: string) {
  const rest = (this.subs.get(channel) ?? []).filter(w => w !== userSock);
  this.subs.set(channel, rest);
  if (rest.length === 0) this.sub.unsubscribe(channel); // last one left
}
Leak guard: drop the Redis subscription when no local socket cares, else channels accumulate forever.

Quick tips & gotchas

  • `export const x = new X()` is a de-facto singleton via Node's module cache — but it's one-per-process, NOT one-per-cluster. Horizontal scaling breaks the illusion.
  • Redis pub/sub needs TWO connections: a client in subscribe mode rejects normal commands. Reuse a single publisher client for all publishes.
  • In-memory singleton state diverges across instances the moment you scale past 1 pod. Sync via Redis pub/sub, or keep truth in a shared DB/Redis.
  • Lazy `getInstance()` isn't thread-safe in general, but Node's single thread makes the null-check fine — unless you use worker_threads/cluster, where each worker gets its own instance.
  • A check-then-act (`if (seats>0) { await ...; seats-- }`) with an `await` between IS racy even in single-threaded Node — the event loop yields at await. Decrement before awaiting, or use atomic Redis ops.
  • Singletons leak state between tests. Add a `reset()` method or prefer dependency injection so test cases start clean.
  • Duplicate module instances (mixed ESM/CJS, symlinked deps, different import paths/casing on Windows) create TWO 'singletons'. Import from one canonical path.
  • Never store durable data (orders, balances) only in a process-memory singleton — a deploy or crash wipes it. RAM state = ephemeral cache/session/room only.
