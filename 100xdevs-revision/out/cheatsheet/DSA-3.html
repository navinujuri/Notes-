Arrays — Cheat Sheet

Cohort 2.0 · 6 lessons · quick reference

Arrays: declare & operation costs

Create / init (JS)
const a = [1, 2, 3];
const b = new Array(5).fill(0);      // [0,0,0,0,0]
const c = Array.from({length: 5}, (_, i) => i); // [0..4]
new Array(5) alone makes 5 empty slots, not zeros — always .fill().
Access by index — O(1)
a[2]; // random access via base + i*size
Contiguous memory is what makes indexing constant-time.
Push / pop (end) — O(1) amortized
a.push(4);  // append
a.pop();    // remove last
Amortized: occasional resize copies all n elements but averages to O(1).
Shift / unshift (front) — O(n)
a.unshift(0); // insert front, shifts all right
a.shift();    // remove front, shifts all left
Every element re-indexed → avoid in hot loops; use a deque instead.
Insert / delete middle — O(n)
a.splice(i, 0, x); // insert x at i
a.splice(i, 1);    // delete at i
Cost is from shifting the n-i trailing elements.
Linear search — O(n)
a.indexOf(x);
a.find(v => v > 10);

Asymptotic notations

Big-O — upper bound (worst case)
f(n) = O(g(n))  iff  f(n) <= c*g(n) for all n >= n0
"At most this fast-growing." The one you almost always quote.
Big-Omega Ω — lower bound (best case)
f(n) = Ω(g(n))  iff  f(n) >= c*g(n) for n >= n0
"At least this much work."
Big-Theta Θ — tight bound
f(n) = Θ(g(n))  iff  f = O(g) AND f = Ω(g)
Upper and lower match → the true growth rate.
little-o / little-ω — strict bounds
f = o(g): f grows strictly slower (ratio -> 0)
o/ω exclude the tight case; O/Ω allow equality.
Growth ranking
1 < log n < n < n log n < n^2 < n^3 < 2^n < n!
Compare by dominant term as n -> infinity; constants don't affect the class.

Complexity analysis rules

Drop constants
O(2n)   -> O(n)
O(n/2)  -> O(n)
Big-O ignores multiplicative constants.
Drop lower-order terms
O(n^2 + n + 100) -> O(n^2)
Keep only the fastest-growing term.
Sequential code → ADD
loop A: O(n)
loop B: O(m)
// total O(n + m)
Nested loops → MULTIPLY
for i in n:
  for j in n:   // O(n * n) = O(n^2)
Watch inner bounds: for j in i gives n(n+1)/2 = O(n^2) still.
Halving the input → log n
while (n > 1) n = n / 2;  // O(log n)
Doubling/halving each step ⇒ log2(n) iterations.
Separate variables stay separate
// two different arrays sizes
O(a + b), NOT O(n)
Don't collapse independent inputs into one n.
Analyze worst case by default
Best/average also exist, but O() interviews mean worst case unless stated.

Popular complexity classes

O(1) constant
arr[i]; map.get(k); push/pop
Independent of n.
O(log n) logarithmic
binary search; balanced BST op
Input cut by a constant factor each step.
O(n) linear
single pass / scan
One touch per element.
O(n log n) linearithmic
merge sort, heap sort, Array.sort()
Best possible for comparison sorting.
O(n^2) quadratic
nested loop, bubble/insertion sort, all pairs
n up to ~10^4 is usually the ceiling for O(n^2).
O(2^n) exponential
naive subsets, recursive fib without memo
Feasible only for n ~ 20-25.
O(n!) factorial
permutations, brute-force TSP
n ~ 10-12 max.

Master Theorem (divide & conquer)

Recurrence form
T(n) = a*T(n/b) + f(n)
// a >= 1 subproblems, each size n/b, f(n) = split+merge cost
Compare f(n) against n^(log_b a) = the "watershed" term.
Case 1 — leaves dominate
if f(n) = O(n^(log_b a - e)),  e>0
=> T(n) = Θ(n^(log_b a))
Work grows toward the bottom of the recursion tree.
Case 2 — balanced
if f(n) = Θ(n^(log_b a))
=> T(n) = Θ(n^(log_b a) * log n)
Every level does equal work; +log n from the level count.
Case 3 — root dominates
if f(n) = Ω(n^(log_b a + e)) and regularity a*f(n/b) <= c*f(n)
=> T(n) = Θ(f(n))
Merge sort
T(n) = 2T(n/2) + O(n)
log_2 2 = 1, f = n = n^1  -> Case 2
=> Θ(n log n)
Binary search
T(n) = 1*T(n/2) + O(1)
log_2 1 = 0, f = 1 = n^0  -> Case 2
=> Θ(log n)
Doesn't apply when
a<1, non-polynomial gap between f and n^(log_b a), or uneven splits — use recursion-tree / Akra–Bazzi instead.

Core array problem patterns

Two pointers — O(n)
let l = 0, r = a.length - 1;
while (l < r) {
  if (a[l] + a[r] === t) return [l, r];
  a[l] + a[r] < t ? l++ : r--;
}
Requires sorted input for the sum/target variant.
Sliding window — O(n)
let sum = 0, best = 0, l = 0;
for (let r = 0; r < a.length; r++) {
  sum += a[r];
  while (sum > k) sum -= a[l++];
  best = Math.max(best, r - l + 1);
}
Turns an O(n^2) subarray scan into one pass.
Prefix sum — O(n) build, O(1) range query
const p = [0];
for (const x of a) p.push(p.at(-1) + x);
// sum(i..j) = p[j+1] - p[i]
Kadane — max subarray, O(n)
let cur = a[0], best = a[0];
for (let i = 1; i < a.length; i++) {
  cur = Math.max(a[i], cur + a[i]);
  best = Math.max(best, cur);
}
Binary search — O(log n)
let lo = 0, hi = a.length - 1;
while (lo <= hi) {
  const m = (lo + hi) >> 1;
  if (a[m] === t) return m;
  a[m] < t ? lo = m + 1 : hi = m - 1;
}
(lo+hi)>>1 avoids overflow vs (lo+hi)/2 in fixed-width langs.
Frequency / hashing — O(n)
const freq = new Map();
for (const x of a) freq.set(x, (freq.get(x) ?? 0) + 1);
Trades O(n) space to drop time from O(n^2) to O(n).

Quick tips & gotchas

  • Big-O measures growth rate, not runtime — a well-tuned O(n^2) can beat O(n log n) for tiny n; asymptotics only win as n grows.
  • unshift/shift/splice at the front are O(n) because every element is re-indexed — reach for a deque or index pointer in hot paths.
  • push/pop is O(1) amortized, not strictly O(1): a resize copies all n elements, but it averages out.
  • Map input size to a target complexity: n<=10 → O(n!)/O(2^n) ok; n<=10^4 → O(n^2); n<=10^6 → O(n log n)/O(n).
  • Master Theorem needs the a*T(n/b) shape; unequal splits (e.g. T(n)=T(n/3)+T(2n/3)+n) need a recursion tree or Akra–Bazzi.
  • In Master Theorem the pivot is n^(log_b a): compare f(n) against it — smaller ⇒ Case 1, equal ⇒ Case 2 (adds log n), larger ⇒ Case 3.
  • Space complexity counts too — recursion depth (call stack) and auxiliary arrays are part of the cost, not free.
  • Don't collapse independent inputs: two loops over different arrays are O(a+b), not O(n) — only merge variables that are actually the same size.
