CSR vs SSR vs SSG — Cheat Sheet

Cohort 2.0 · 3 lessons · quick reference

The 3 Strategies at a Glance

CSR — render in the browser
Server sends near-empty HTML + JS bundle; React builds DOM client-side. Fast nav after load, weak SEO/first paint. Use for auth'd dashboards, SPAs.
SSR — render per request on the server
HTML built on every request with fresh data, then hydrated. Good SEO + personalization; higher TTFB, needs a running server. Use for feeds, per-user pages.
SSG — render once at build time
HTML generated during `next build`, served from CDN. Fastest + cheapest; data is frozen until rebuild. Use for blogs, docs, marketing, product pages.
ISR — SSG that self-updates
Static page regenerated in background on a timer / on-demand. Static speed with periodic freshness. Best of SSG+SSR for semi-dynamic content.
Hydration
SSR/SSG ship HTML AND JS; the JS 'hydrates' static markup into an interactive React tree. Mismatched server/client output = hydration error.

CSR — Client Side Rendering

Plain React SPA scaffold
npm create vite@latest my-app -- --template react
cd my-app && npm i && npm run dev
Fetch data after mount (classic CSR)
function Users() {
  const [data, setData] = useState(null);
  useEffect(() => {
    fetch('/api/users').then(r => r.json()).then(setData);
  }, []);
  if (!data) return <Spinner/>;
  return <List items={data}/>;
}
User sees a loading spinner first — data is NOT in the initial HTML, so crawlers may miss it.
Force client rendering in Next.js App Router
'use client';
import { useState, useEffect } from 'react';
'use client' opts a component (and its imports) into the client bundle. Required for hooks, event handlers, browser APIs.
Client-only data with SWR / React Query
const { data, isLoading } = useSWR('/api/users', fetcher);
Handles caching, revalidation, dedupe — the idiomatic CSR data layer.

SSR — Server Side Rendering

Pages Router: getServerSideProps
export async function getServerSideProps(ctx) {
  const res = await fetch(`https://api/x/${ctx.params.id}`);
  const data = await res.json();
  return { props: { data } }; // -> page props
}
export default function Page({ data }) { return <div>{data.name}</div>; }
Runs on the server for EVERY request. Access ctx.req, ctx.res, ctx.query, ctx.params. Never bundled to the client.
Redirect / 404 from SSR
return { redirect: { destination: '/login', permanent: false } };
// or
return { notFound: true };
App Router: dynamic by default via no-store
export default async function Page() {
  const data = await fetch('https://api/x', { cache: 'no-store' })
    .then(r => r.json());
  return <div>{data.name}</div>;
}
Server Components are async. `cache: 'no-store'` makes the route render per request (SSR).
App Router: force whole segment dynamic
export const dynamic = 'force-dynamic';
Also triggered implicitly by using cookies(), headers(), or searchParams.

SSG — Static Site Generation

Pages Router: getStaticProps
export async function getStaticProps() {
  const posts = await fetch('https://api/posts').then(r => r.json());
  return { props: { posts } };
}
Runs ONLY at build time (and during ISR). No access to req/res. Data is baked into the HTML.
Pages Router: getStaticPaths for dynamic routes
// pages/posts/[id].js
export async function getStaticPaths() {
  const posts = await fetch('https://api/posts').then(r => r.json());
  return {
    paths: posts.map(p => ({ params: { id: String(p.id) } })),
    fallback: false,
  };
}
getStaticPaths tells Next WHICH dynamic pages to pre-render. Pair with getStaticProps({ params }).
fallback modes
fallback: false   // unlisted paths -> 404
fallback: 'blocking' // SSR on first hit, then cached
fallback: true    // serve fallback UI, then swap in
App Router: static by default + generateStaticParams
// app/posts/[id]/page.js
export async function generateStaticParams() {
  const posts = await fetch('https://api/posts').then(r => r.json());
  return posts.map(p => ({ id: String(p.id) }));
}
App Router prerenders statically unless you opt out. Default fetch cache = 'force-cache'.

ISR — Incremental Static Regeneration

Pages Router: time-based revalidate
export async function getStaticProps() {
  const data = await fetch('https://api/x').then(r => r.json());
  return { props: { data }, revalidate: 60 }; // seconds
}
First request after 60s serves stale, triggers background regen; next request gets fresh HTML (stale-while-revalidate).
App Router: fetch-level revalidate
const data = await fetch('https://api/x', {
  next: { revalidate: 60 },
}).then(r => r.json());
Or set for the whole segment: `export const revalidate = 60;`
On-demand revalidation (webhook / server action)
import { revalidatePath, revalidateTag } from 'next/cache';
revalidatePath('/posts');
revalidateTag('posts');
Tag a fetch to invalidate it precisely: fetch(url, { next: { tags: ['posts'] } }).

App Router Rendering Control (Next 13+)

fetch cache = strategy switch
fetch(url)                              // cached -> SSG
fetch(url, { cache: 'no-store' })       // uncached -> SSR
fetch(url, { next: { revalidate: 60 }}) // ISR
Route segment config exports
export const dynamic = 'auto' | 'force-dynamic' | 'force-static';
export const revalidate = 60 | false;
export const fetchCache = 'auto' | 'force-cache' | 'force-no-store';
export const runtime = 'nodejs' | 'edge';
Server vs Client components
Everything is a Server Component by default (runs on server, 0 JS shipped). Add 'use client' only where interactivity/hooks are needed. Keep 'use client' at leaves.
Streaming SSR with Suspense
<Suspense fallback={<Skeleton/>}>
  <SlowServerComponent/>
</Suspense>
Shell streams instantly; slow parts stream in. Improves TTFB without going full CSR.

Full static export (no Node server)
// next.config.js
module.exports = { output: 'export' };
Pure SSG to static files. Disables SSR/ISR, API routes, and image optimization defaults.

Decision Cheat Sheet

Content same for all users, changes rarely
-> SSG (blog, docs, landing).
Content same for all, changes periodically
-> ISR (product listings, pricing, news).
Per-request / per-user / real-time fresh data
-> SSR (dashboards feed, personalized, search results).
Highly interactive, SEO irrelevant, behind auth
-> CSR (admin panels, editors, internal tools).
SEO + speed both critical
-> SSG/ISR first, SSR only if data must be per-request.
Mix per page
Modern default: static shell (SSG/ISR) + client islands (CSR) + a few dynamic (SSR) routes.

Quick tips & gotchas

  • CSR hurts SEO & first paint because initial HTML is empty — crawlers/slow devices see a blank shell until JS runs.
  • getServerSideProps (SSR) runs on EVERY request; getStaticProps (SSG) runs at BUILD time. Neither ships to the client bundle — safe for secrets/DB calls.
  • In App Router the mental model flips: pages are static by default; you opt INTO dynamic via cache:'no-store', cookies(), headers(), or dynamic='force-dynamic'.
  • ISR revalidate is stale-while-revalidate: the request that triggers regen still gets the OLD page; the NEXT request gets the fresh one.
  • Hydration errors come from server/client output mismatch — Date.now(), Math.random(), window checks, or locale formatting. Gate them behind useEffect or suppressHydrationWarning.
  • 'use client' is contagious downward but not upward: it marks a boundary; children become client too, but a client component can still receive server-rendered children as props.
  • output:'export' gives pure static hosting (S3/CDN) but kills SSR, ISR, API routes, and middleware — only choose it if the site is fully static.
  • Always run `next build` and read the route table (○ ● ƒ) to CONFIRM each page uses the strategy you expect — silent fallbacks to dynamic are common.
