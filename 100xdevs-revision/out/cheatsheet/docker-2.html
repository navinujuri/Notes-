Docker End to End — Cheat Sheet

Cohort 2.0 · 22 lessons · quick reference

Concepts & Setup

Why Docker
Kills "works on my machine": ships app + deps + runtime as one portable unit; fast, reproducible onboarding & deploys.
Containers vs VMs
Containers share the host kernel (MBs, boot in ms). VMs virtualize hardware w/ a full guest OS (GBs, slow boot).
History (why it exists)
dotCloud, 2013. Originally wrapped Linux LXC; now uses runc + containerd. Containers are Linux cgroups + namespaces.
Install check
docker --version
docker info
docker run hello-world
Docker Desktop on Win/Mac runs a Linux VM under the hood; `docker info` confirms the daemon is up.
Image vs Container
Image = read-only blueprint (built once, versioned by tag). Container = a running (or stopped) instance of an image. One image → many containers.

Core CLI Commands

Run a container
docker run -d --name web -p 8080:80 nginx
-d detached, --rm auto-delete on exit, -it interactive shell.
List containers
docker ps        # running
docker ps -a     # all (incl. stopped)
Shell into a running container
docker exec -it web sh   # or bash
Debug live containers here; alpine images only have `sh`.
Logs
docker logs -f web       # -f = follow/tail
Stop / start / remove
docker stop web
docker start web
docker rm web            # -f to force-remove running
Images
docker images
docker pull redis:7
docker rmi nginx
Reclaim space
docker system prune -a --volumes
Removes stopped containers, unused images/networks/volumes. Irreversible.

Port Mapping & Env Vars

Publish a port
docker run -p 3000:3000 myapp
Format is -p HOST:CONTAINER. Without -p the container port is unreachable from host.
Multiple / random ports
docker run -p 8080:80 -p 443:443 img
docker run -P img        # -P = auto-map all EXPOSEd ports
Single env var
docker run -e NODE_ENV=production -e PORT=3000 myapp
Env file
docker run --env-file .env myapp
File is KEY=VALUE per line, no quotes/spaces around =.
EXPOSE (Dockerfile)
EXPOSE 3000
Documentation only — does NOT publish. You still need -p at runtime.

Dockerfile

Typical Node app
FROM node:20-alpine
WORKDIR /app
COPY package*.json ./
RUN npm ci
COPY . .
EXPOSE 3000
CMD ["node", "index.js"]
FROM
FROM node:20-alpine
Base image + tag. Always pin a tag; `latest` drifts. alpine/slim = smaller.
WORKDIR / COPY / RUN
WORKDIR /app        # cwd for later cmds
COPY src ./src      # host -> image
RUN npm run build   # runs at build time
RUN executes during build (bakes into image); each RUN = a layer.
CMD vs ENTRYPOINT
ENTRYPOINT ["node"]
CMD ["index.js"]
CMD is the default, overridable by `docker run img <args>`. ENTRYPOINT is fixed; combine so run-args append to it. Use exec (JSON) form.
ARG vs ENV
ARG VERSION=1.0     # build-time only
ENV APP_VERSION=$VERSION  # persists in container
ARG values vanish after build; ENV is available at runtime.

Build, Run & Push to Docker Hub

Build & tag
docker build -t saichandan/myapp:v1 .
`.` is the build context (dir sent to daemon). -t names/tags it.
Run your image
docker run -d -p 3000:3000 saichandan/myapp:v1
Tag an existing image
docker tag myapp saichandan/myapp:latest
Login & push
docker login
docker push saichandan/myapp:v1
Image name MUST be prefixed with your Docker Hub username to push.
Pull elsewhere
docker pull saichandan/myapp:v1

Layers & Optimization

Every instruction = a layer
FROM/COPY/RUN each create a cached, read-only layer. Container adds a thin writable layer on top.
Inspect layers
docker history saichandan/myapp:v1
docker image inspect myapp
Cache = build speed
Docker reuses a layer if the instruction + its inputs are unchanged. First changed layer busts cache for all layers below it.
Order matters (key optimization)
COPY package*.json ./
RUN npm ci
COPY . .
Copy manifest & install deps BEFORE copying source. Code edits then skip the slow npm install layer.
.dockerignore
node_modules
.git
*.log
Dockerfile
Shrinks build context & final image; keeps secrets/junk out. Like .gitignore.
Multi-stage build
FROM node:20 AS build
WORKDIR /app
COPY . .
RUN npm ci && npm run build

FROM nginx:alpine
COPY --from=build /app/dist /usr/share/nginx/html
Ship only artifacts, drop build toolchain → far smaller, safer images.

Volumes, Networks & Compose

Named volume (persist data)
docker volume create pgdata
docker run -v pgdata:/var/lib/postgresql/data postgres:16
Container FS is ephemeral — data dies with `rm`. Volumes survive, managed by Docker.
Bind mount (live dev)
docker run -v "$(pwd)":/app node:20
Maps a host dir into the container — edits reflect instantly. On Windows PS use ${PWD}.
Volume commands
docker volume ls
docker volume rm pgdata
Custom network + DNS
docker network create mynet
docker run -d --name db --network mynet postgres
docker run --network mynet myapp   # reach db at host "db"
Containers on a user-defined network resolve each other by container name. Default bridge does NOT give name DNS.
Network drivers
docker network ls
bridge (default, isolated), host (share host net, no isolation), none (no net).
docker-compose.yml
services:
  app:
    build: .
    ports: ["3000:3000"]
    environment:
      - DATABASE_URL=postgres://db/app
    depends_on: [db]
  db:
    image: postgres:16
    environment:
      POSTGRES_PASSWORD: secret
    volumes:
      - pgdata:/var/lib/postgresql/data
volumes:
  pgdata:
Compose auto-creates a shared network so services reach each other by service name.
Compose lifecycle
docker compose up -d --build
docker compose logs -f
docker compose down -v
down -v also drops named volumes. Newer syntax is `docker compose` (space), not `docker-compose`.

Quick tips & gotchas

  • CMD is overridden by args after the image name (`docker run img echo hi`); ENTRYPOINT isn't — pair them for a fixed binary + default args.
  • Always COPY package.json + install deps before COPY-ing source, so code changes don't invalidate the dependency layer.
  • Add a .dockerignore (node_modules, .git) — it shrinks the build context and prevents leaking secrets into layers.
  • -p HOST:CONTAINER order trips everyone up; the container port is the right-hand side and must match what your app listens on.
  • Anything written inside a container is lost on `docker rm` — mount a named volume for databases and any state you must keep.
  • Only containers on the SAME user-defined network can resolve each other by name; the default bridge requires --link or IPs.
  • Prefer alpine/slim bases and multi-stage builds to cut image size (and attack surface) by an order of magnitude.
  • `docker exec -it <id> sh` is your go-to for debugging a live container; `docker logs -f` tails stdout/stderr.
