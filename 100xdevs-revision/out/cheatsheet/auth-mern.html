Authentication — Cheat Sheet

Setup & Express scaffold

Install deps
npm init -y
npm i express jsonwebtoken bcryptjs cors
# mongoose for persistence, nodemon --save-dev for reload
Minimal Express app
const express = require("express");
const app = express();
app.use(express.json());          // parse JSON body
app.listen(3000, () => console.log("up on 3000"));
Without express.json() req.body is undefined for JSON POSTs.
Enable CORS (frontend on another origin)
const cors = require("cors");
app.use(cors());  // dev only; lock origins in prod
Browser blocks cross-origin fetch to :3000 from :5173 without this.
In-memory store (course pattern)
const USERS = [];  // [{ username, password }] — reset on restart
Fine for the lesson; swap for MongoDB/mongoose in the TODO app.

Auth workflow (bank analogy)

Mental model
Signup = open account. Signin = teller checks ID, hands you a token. Token = your ID card; show it on every request instead of re-proving identity each time.
Flow
POST /signup  -> store {username, hash(password)}
POST /signin  -> verify creds, RETURN token
GET  /me      -> send token, server identifies you
Where the token lives (client)
// after signin
localStorage.setItem("token", data.token);
// on each call
headers: { Authorization: token }
Server stays stateless with JWT — no session table needed.

Signup / Signin endpoints

Hash password on signup
const bcrypt = require("bcryptjs");
app.post("/signup", async (req, res) => {
  const { username, password } = req.body;
  const hash = await bcrypt.hash(password, 5);
  USERS.push({ username, password: hash });
  res.json({ message: "created" });
});
NEVER store plaintext passwords. Second arg = salt rounds.
Verify + issue token on signin
app.post("/signin", async (req, res) => {
  const { username, password } = req.body;
  const user = USERS.find(u => u.username === username);
  if (!user) return res.status(403).json({ message: "bad creds" });
  const ok = await bcrypt.compare(password, user.password);
  if (!ok) return res.status(403).json({ message: "bad creds" });
  const token = jwt.sign({ username }, JWT_SECRET);
  res.json({ token });
});
Return SAME error for unknown user + wrong password (no user enumeration).
Naive random token (pre-JWT lesson)
const token = crypto.randomBytes(16).toString("hex");
tokenToUser[token] = username;  // server MUST remember mapping
Stateful: server must store token->user. JWT removes this need.

JWTs — sign, verify, decode

Sign (create)
const jwt = require("jsonwebtoken");
const JWT_SECRET = "mysecret";  // env var in prod
const token = jwt.sign({ username }, JWT_SECRET, { expiresIn: "1h" });
Payload should carry only an identifier — never the password.
Verify (trust + extract)
try {
  const decoded = jwt.verify(token, JWT_SECRET);
  const username = decoded.username;
} catch (e) {
  return res.status(401).json({ message: "invalid token" });
}
verify checks the signature. Throws on tamper/expiry — always wrap in try/catch.
Decode (NO signature check)
const payload = jwt.decode(token);  // anyone can do this
Use for reading, never for trust. Verification is what makes a JWT secure.
Anatomy
header.payload.signature  // 3 base64url parts split by '.'
header + payload are just base64 — paste into jwt.io to read. Only signature needs the secret.

Auth middleware

Reusable guard
function auth(req, res, next) {
  const token = req.headers.authorization;
  if (!token) return res.status(401).json({ message: "no token" });
  try {
    const decoded = jwt.verify(token, JWT_SECRET);
    req.username = decoded.username;   // attach for handlers
    next();
  } catch (e) {
    res.status(401).json({ message: "invalid token" });
  }
}
next() passes control on; forgetting it hangs the request.
Protect a route
app.get("/me", auth, (req, res) => {
  res.json({ username: req.username });
});
Middleware slots between path and handler; runs before the handler.
Bearer scheme (real-world)
const token = req.headers.authorization?.split(" ")[1]; // 'Bearer <tok>'
Course sends raw token; production convention is 'Bearer <token>'.

Frontend wiring (fetch)

Signin -> store token
const res = await fetch("http://localhost:3000/signin", {
  method: "POST",
  headers: { "Content-Type": "application/json" },
  body: JSON.stringify({ username, password })
});
const data = await res.json();
localStorage.setItem("token", data.token);
Must set Content-Type: application/json or express.json() ignores the body.
Authenticated request
const res = await fetch("http://localhost:3000/me", {
  headers: { Authorization: localStorage.getItem("token") }
});
Header key must match what the middleware reads (authorization).
Axios equivalent
axios.get("/me", { headers: { Authorization: token } });

Tokens vs JWTs

Opaque token
Random string; server stores token->user mapping (stateful). Revoke = delete row. Needs DB lookup per request.
JWT
Self-contained + signed; server stores nothing (stateless). Scales horizontally. Verify with secret, no DB hit.
Trade-off
JWTs are hard to revoke before expiry — keep expiresIn short or maintain a blocklist. Don't put secrets in the payload (it's readable).

Quick tips & gotchas

  • JWT payload is base64, NOT encrypted — anyone can decode and read it. Never put passwords or secrets in it.
  • jwt.verify (checks signature) vs jwt.decode (does not). Only trust data that came from verify.
  • Always hash with bcrypt before storing; use bcrypt.compare to check — never compare plaintext or store raw passwords.
  • Keep JWT_SECRET in an env var (process.env.JWT_SECRET), never hardcoded/committed. Anyone with the secret can forge tokens.
  • Return identical errors for 'user not found' and 'wrong password' to avoid leaking which usernames exist.
  • Forgetting next() in middleware silently hangs the request; forgetting express.json() makes req.body undefined.
  • Set expiresIn on tokens — JWTs can't be revoked server-side without extra state, so short lifetimes limit damage.
  • Header casing: req.headers keys are lowercased by Node, so read req.headers.authorization regardless of how the client cased it.
