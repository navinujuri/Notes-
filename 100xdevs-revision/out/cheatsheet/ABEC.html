Advance BE communication — Cheat Sheet

Cohort 2.0 · 8 lessons · quick reference

Communication models

Request/Response (HTTP)
Client initiates, server responds once. Stateless, short-lived. REST/GraphQL. No server push.
Short polling
setInterval(() => fetch('/updates'), 3000)
Client re-asks on a timer. Simple, but wasteful + high latency between polls.
Long polling
Server holds request open until data ready, then client re-requests. Fallback when WS unavailable.
Server-Sent Events (SSE)
const es = new EventSource('/stream');
es.onmessage = e => console.log(e.data);
One-way server→client over HTTP, auto-reconnect built in. No client→server channel.
WebSockets
Full-duplex, persistent, bidirectional over a single TCP connection. Use for chat, live cursors, games, real-time dashboards.
WebRTC
Peer-to-peer (browser↔browser) for media/data. Needs a signaling server (often WS) to establish.

WebSocket protocol

Upgrade handshake
GET /chat HTTP/1.1
Upgrade: websocket
Connection: Upgrade
Sec-WebSocket-Key: <base64>
Sec-WebSocket-Version: 13
Starts as HTTP GET; server replies 101 Switching Protocols. Same TCP socket is reused after.
Schemes
ws://host:port   // plain
wss://host:port  // TLS (use in prod)
Frames, not messages
Protocol is frame-based (text/binary/ping/pong/close). Long-lived and stateful — no per-message HTTP overhead.
Stays open
No new handshake per message = low latency. Cost: server holds a connection/memory per client — plan for scale.

ws server (Node.js)

Install
npm i ws            # runtime
npm i -D @types/ws  # TS types
Standalone server
import { WebSocketServer } from 'ws';
const wss = new WebSocketServer({ port: 8080 });

wss.on('connection', (socket) => {
  socket.on('message', (data) => {
    // data is a Buffer -> convert
    console.log(data.toString());
    socket.send('pong');
  });
  socket.on('close', () => console.log('gone'));
  socket.on('error', console.error);
});
Attach to existing HTTP server
import { createServer } from 'http';
import { WebSocketServer } from 'ws';
const server = createServer();
const wss = new WebSocketServer({ server });
server.listen(8080);
Lets WS and REST share one port. Use { noServer:true } + server.on('upgrade') for path/auth-based routing.
Broadcast to all clients
for (const client of wss.clients) {
  if (client.readyState === WebSocket.OPEN) client.send(msg);
}
JSON message protocol
socket.on('message', (raw) => {
  const { type, payload } = JSON.parse(raw.toString());
  if (type === 'join') rooms[payload.room].add(socket);
});
socket.send(JSON.stringify({ type:'ack' }));
Wrap everything in a { type, payload } envelope; parse defensively in try/catch.
Auth on upgrade
wss.on('connection', (socket, req) => {
  const token = new URL(req.url, 'ws://x').searchParams.get('token');
  if (!verify(token)) return socket.close(1008, 'unauthorized');
});
Browsers can't set custom headers on WS — pass tokens via query string or a post-connect auth message.

Client side

Browser WebSocket API
const ws = new WebSocket('wss://api.example.com/chat');
ws.onopen  = () => ws.send(JSON.stringify({ type:'join' }));
ws.onmessage = (e) => console.log(JSON.parse(e.data));
ws.onclose = () => console.log('closed');
ws.onerror = (e) => console.error(e);
Guard sends until OPEN
if (ws.readyState === WebSocket.OPEN) ws.send(msg);
readyState: 0 CONNECTING, 1 OPEN, 2 CLOSING, 3 CLOSED. Sending before OPEN throws.
React (useEffect cleanup)
useEffect(() => {
  const ws = new WebSocket(URL);
  ws.onmessage = (e) => setMsgs(m => [...m, e.data]);
  return () => ws.close();
}, []);
Always close on unmount. In StrictMode dev double-invoke, guard against duplicate connections.
Reconnect with backoff
function connect(retry = 0) {
  const ws = new WebSocket(URL);
  ws.onclose = () => setTimeout(
    () => connect(retry + 1),
    Math.min(1000 * 2 ** retry, 30000));
}
Native WS does NOT auto-reconnect. Cap the delay; reset retry counter on successful open.

Next.js integration

Can't run WS in route handlers
App Router / API routes are serverless/short-lived — no long-lived socket. Run the WS server as a separate process.
Separate ws service pattern
Deploy Next.js (frontend/HTTP) and a standalone Node `ws` server independently; client connects to the ws service URL via env var.
Custom server (self-hosted only)
// server.js
import next from 'next';
import { createServer } from 'http';
import { WebSocketServer } from 'ws';
const app = next({ dev });
const server = createServer(app.getRequestHandler());
const wss = new WebSocketServer({ server });
Only works when you control the Node process. NOT supported on Vercel serverless — use a dedicated host (Render/Railway/EC2).
Client URL via env
const ws = new WebSocket(process.env.NEXT_PUBLIC_WS_URL!);
NEXT_PUBLIC_ prefix exposes it to the browser bundle.

Scaling WS servers

The problem
Sockets are stateful + sticky to one instance. With N servers, a user on server A can't directly reach a user on server B.
Sticky sessions at LB
A client must keep hitting the same instance (the socket lives there). Enable session affinity / IP hash on the load balancer.
Redis Pub/Sub fan-out
import { createClient } from 'redis';
const pub = createClient(); const sub = pub.duplicate();
await pub.connect(); await sub.connect();

sub.subscribe('room:1', (msg) => broadcastLocal(msg));
// on incoming ws message:
pub.publish('room:1', JSON.stringify(payload));
Every WS instance subscribes; a message published on any node reaches all nodes, which relay to their local sockets.
Horizontal architecture
LB → [ws-1, ws-2, ws-3] all connected to a Redis (or Kafka/NATS) pub/sub bus. Add instances to scale connections.
Heartbeat / dead-connection reap
const iv = setInterval(() => {
  wss.clients.forEach(s => {
    if (!s.isAlive) return s.terminate();
    s.isAlive = false; s.ping();
  });
}, 30000);
wss.on('connection', s => {
  s.isAlive = true;
  s.on('pong', () => s.isAlive = true);
});
TCP can silently drop; ping/pong detects zombies so you free memory. Clear the interval on wss 'close'.
Managed alternatives
socket.io (rooms + @socket.io/redis-adapter), Ably, Pusher, or Redis for presence/state — skip hand-rolling scale.

Quick tips & gotchas

  • ws message events give you a Buffer, not a string — always .toString() (or JSON.parse(raw.toString())) before use.
  • Wrap JSON.parse in try/catch: a single malformed frame from a client shouldn't crash the whole server.
  • Browsers can't set custom headers (e.g. Authorization) on a WebSocket — authenticate via query param or a first post-connect message.
  • Native WebSocket has zero built-in reconnection — implement exponential backoff yourself or use reconnecting-websocket / socket.io.
  • Vercel and most serverless platforms can't host long-lived WS — run the socket server on a persistent host (Render, Railway, Fly, EC2).
  • One process = one CPU core; sockets don't share memory across instances. Use Redis pub/sub + sticky sessions to scale out.
  • Always ws.close() on client unmount and clean up server timers on 'close' — leaked sockets/intervals exhaust memory fast.
  • Use wss:// (TLS) in production; ws:// is blocked from https:// pages by mixed-content rules.
