Serverless Backends — Cheat Sheet

Cohort 2.0 · 12 lessons · quick reference

Servers vs Serverless

Traditional backend server
node server.js  # one process, listens on a port, runs 24/7
Always-on: you provision, scale, patch, and pay for the box whether traffic exists or not.
Serverless backend
You ship functions; provider runs them on demand, auto-scales to zero, bills per request/CPU. No server/OS to manage.
Core trade-off
Pay-per-use + zero ops vs cold starts, execution/CPU limits, statelessness, and vendor lock-in.
Cloudflare Workers model
Runs on V8 isolates (not containers) at the edge -> ~0ms cold start, global low latency, but limited Node APIs & strict CPU-time caps.

Providers & When to Use

Famous providers
AWS Lambda
Cloudflare Workers
Vercel / Netlify Functions
Google Cloud Functions
Azure Functions
Deno Deploy
Use serverless when
Spiky/unpredictable or low/intermittent traffic, small APIs / webhooks / cron, want autoscale + pay-per-use + global latency.
Avoid serverless when
Long-running jobs, websockets/stateful sockets, steady high throughput (a VM is cheaper), heavy compute, or need full Node + native libs.
Limits to watch
Cold starts (Lambda), CPU/wall-time caps (Workers ~10-50ms CPU on free tier), payload/memory limits, no persistent local state between requests.

Worker Anatomy & Deploy

Minimal worker (module syntax)
export default {
  async fetch(request, env, ctx) {
    return new Response('Hello World');
  }
};
env = bindings/secrets/vars; ctx = execution context. No app.listen() - the runtime calls fetch().
Route by path + method
const url = new URL(request.url);
if (url.pathname === '/api' && request.method === 'POST') { /* ... */ }
Background work after response
ctx.waitUntil(logToAnalytics(request));
Keeps the worker alive to finish async work after the Response is returned.
Deploy to production
npx wrangler deploy
# -> https://my-app.<subdomain>.workers.dev
Tail live logs
npx wrangler tail

Express vs Hono

Why Express doesn't fit Workers
Express targets Node's http server (req/res, app.listen). Workers use a fetch(Request)->Response event model with no long-running server.
Run Express anyway (compat)
# wrangler.toml
compatibility_flags = ["nodejs_compat"]
Partial Node polyfill; heavier bundle, not idiomatic. Prefer Hono on Workers.
Use Hono (Workers-native, Express-like API)
npm i hono
Minimal Hono app
import { Hono } from 'hono';
const app = new Hono();
app.get('/', (c) => c.text('Hello Hono!'));
export default app;
Tiny, zero-dependency; same code runs on Workers/Bun/Deno/Node.

Hono Routing & Middlewares

Path params
app.get('/user/:id', (c) => {
  const id = c.req.param('id');
  return c.json({ id });
});
Query + JSON body
app.post('/todo', async (c) => {
  const sort = c.req.query('sort');
  const body = await c.req.json();
  return c.json(body, 201);
});
Custom middleware
app.use('*', async (c, next) => {
  const start = Date.now();
  await next();
  console.log(`${c.req.method} ${Date.now() - start}ms`);
});
await next() continues the chain; return early (skip next) to short-circuit.
Auth guard (scoped path)
app.use('/api/*', async (c, next) => {
  const token = c.req.header('Authorization');
  if (!token) return c.json({ error: 'unauthorized' }, 401);
  await next();
});
Built-in middleware
import { cors } from 'hono/cors';
import { logger } from 'hono/logger';
import { jwt } from 'hono/jwt';
app.use('*', logger(), cors());

Config, Env & Bindings

wrangler.toml basics
name = "my-worker"
main = "src/index.ts"
compatibility_date = "2024-09-01"

[vars]
API_BASE = "https://api.example.com"
Secrets (encrypted)
npx wrangler secret put DATABASE_URL
# prompts for the value; never commit it
Use for keys/DB URLs. [vars] is plaintext and bundled - not for secrets.
Local secrets/vars
# .dev.vars  (gitignored)
DATABASE_URL="postgres://..."
Auto-loaded by wrangler dev.
Access env in Hono
app.get('/', (c) => c.text(c.env.API_BASE));
In a raw worker it's the 2nd fetch arg (env). There is no process.env at runtime.
KV / D1 bindings
[[kv_namespaces]]
binding = "CACHE"
id = "xxxx"

[[d1_databases]]
binding = "DB"
database_name = "prod"
database_id = "xxxx"

Connecting to a DB

Problem: no raw TCP
Workers can't open arbitrary TCP sockets, so classic pg/mysql drivers fail. Use HTTP-based drivers or a pooling proxy.
Prisma + Accelerate (edge)
npm i @prisma/client @prisma/extension-accelerate
npx prisma generate --no-engine
Instantiate per-request
import { PrismaClient } from '@prisma/client/edge';
import { withAccelerate } from '@prisma/extension-accelerate';

app.get('/users', async (c) => {
  const prisma = new PrismaClient({
    datasourceUrl: c.env.DATABASE_URL,
  }).$extends(withAccelerate());
  const users = await prisma.user.findMany();
  return c.json(users);
});
DATABASE_URL is the prisma:// Accelerate URL. Build the client INSIDE the handler, not at module top level.
D1 (native SQLite)
const { results } = await c.env.DB.prepare(
  'SELECT * FROM users WHERE id = ?'
).bind(id).all();
Other edge-friendly options
Neon serverless driver (@neondatabase/serverless over HTTP), Turso/libSQL, Supabase REST, Upstash Redis (HTTP).

Quick tips & gotchas

  • Create the Prisma client inside the request handler - Workers isolates don't persist DB connections across requests like a Node server does.
  • No process.env at runtime: read config from env (raw worker) or c.env (Hono), populated only from bindings/vars/secrets.
  • Workers bill CPU time, not wall time - avoid heavy sync loops; push slow/async work into ctx.waitUntil.
  • Secrets go through `wrangler secret put` (encrypted); [vars] is plaintext and shipped in the bundle - never put keys there.
  • Bump compatibility_date deliberately: it pins runtime behavior and changing it can alter available APIs.
  • Prefer Hono over Express on Workers; Express via nodejs_compat works but bloats the bundle and isn't idiomatic.
  • Type Hono bindings for autocomplete: `new Hono<{ Bindings: { DATABASE_URL: string } }>()`.
  • `wrangler dev` runs the actual workerd runtime, so local behavior mirrors prod far better than a Node mock server.
