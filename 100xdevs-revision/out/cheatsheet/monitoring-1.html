Monitoring, Logging and Newrelic — Cheat Sheet

Cohort 2.0 · 14 lessons · quick reference

Observability Foundations

Three pillars
Logs = discrete events (what happened). Metrics = numeric aggregates over time (how much/how fast). Traces = request path across services. NewRelic ingests all three.
Monitoring vs Alerting vs Status page
Monitoring = collect+visualize. Alerting = notify a human when a threshold breaks. Status page = public/internal uptime dashboard for stakeholders (e.g. statuspage.io, Better Uptime).
SLI / SLO / SLA
SLI = measured signal (latency, error rate). SLO = internal target (99.9% < 300ms). SLA = contractual promise with penalties. Alert on SLO burn, not raw spikes.
Golden signals to watch
Latency, Traffic (throughput), Errors, Saturation (CPU/mem/disk). Everything else is usually derivable from these.
Structured logs > plain strings
// bad
console.log("user " + id + " failed login");
// good — queryable JSON
logger.warn("login_failed", { userId: id, ip });
JSON logs let you filter/aggregate in NRQL (metrics on logs). Never log secrets/PII/tokens.

Winston Logger (Node.js)

Install & create logger
npm i winston

const winston = require('winston');
const logger = winston.createLogger({
  level: process.env.LOG_LEVEL || 'info',
  defaultMeta: { service: 'auth-api' },
  format: winston.format.combine(
    winston.format.timestamp(),
    winston.format.errors({ stack: true }),
    winston.format.json()
  ),
  transports: [
    new winston.transports.Console(),
    new winston.transports.File({ filename: 'error.log', level: 'error' }),
    new winston.transports.File({ filename: 'combined.log' })
  ]
});
Log levels (npm scale)
error:0  warn:1  info:2  http:3  verbose:4  debug:5  silly:6
Logger emits messages at its level and BELOW the number (i.e. more severe). level:'info' drops debug/silly.
Log with structured metadata
logger.info('order_created', { orderId, amount, userId });
logger.error('db_write_failed', { err });  // errors() format captures .stack
2nd arg merges into JSON output — this is what makes 'metrics on logs' possible.
Pretty console in dev, JSON in prod
format: process.env.NODE_ENV === 'production'
  ? winston.format.json()
  : winston.format.combine(winston.format.colorize(), winston.format.simple())
HTTP request logging (Express)
const morgan = require('morgan');
app.use(morgan('combined', { stream: { write: m => logger.http(m.trim()) } }));
Ship logs to NewRelic w/ trace context
npm i @newrelic/winston-enricher

const nrFormatter = require('@newrelic/winston-enricher');
format: winston.format.combine(winston.format.timestamp(), nrFormatter(winston))
Enricher injects trace.id/span.id so 'Logs in context' links a log line to its APM transaction.

NewRelic APM Setup (Node.js)

Install agent
npm i newrelic
# copy config template to project root
cp node_modules/newrelic/newrelic.js ./newrelic.js
Require FIRST — before anything else
require('newrelic');           // must be line 1 of entrypoint
const express = require('express');
Or use node --require newrelic app.js. Loading it late means the agent can't patch libraries → no data.
Config via env (12-factor, no file)
NEW_RELIC_LICENSE_KEY=xxxxx
NEW_RELIC_APP_NAME=auth-api
NEW_RELIC_NO_CONFIG_FILE=true
NEW_RELIC_LOG_LEVEL=info
NEW_RELIC_DISTRIBUTED_TRACING_ENABLED=true
app_name groups instances into ONE APM entity. Same name across pods = aggregated.
What APM auto-captures
Throughput (rpm), response time, error rate, Apdex, transaction traces, DB/external call breakdown, slow queries — zero manual instrumentation.
Custom instrumentation
const newrelic = require('newrelic');
newrelic.recordMetric('Custom/CacheHit', 1);
newrelic.addCustomAttribute('userId', id);
newrelic.noticeError(err, { orderId });

Entities & Infrastructure

Entity = anything NewRelic monitors
APM service, host, container, browser app, lambda, K8s cluster. Each has a GUID. Everything in the platform is scoped to an entity.
Tag entities for filtering
NEW_RELIC_LABELS="env:prod;team:payments;region:ap-south-1"
Tags power entity search, dashboard FACETs, and alert scoping. Tag consistently (env/team/service).
Infrastructure agent (host metrics)
curl -Ls https://download.newrelic.com/install/newrelic-cli/scripts/install.sh | bash
NEW_RELIC_API_KEY=xxx NEW_RELIC_ACCOUNT_ID=123 /usr/local/bin/newrelic install
Infra tab = CPU, memory, disk, network, processes per host. Separate agent from APM.
Infra data in NRQL
SELECT average(cpuPercent) FROM SystemSample FACET hostname TIMESERIES
SELECT latest(memoryUsedBytes/memoryTotalBytes)*100 FROM SystemSample FACET hostname

NRQL (NewRelic Query Language)

Shape
SELECT <func> FROM <EventType> WHERE <cond> FACET <attr> SINCE <time> TIMESERIES
SQL-like but time-series first. Common event types: Transaction, TransactionError, Log, SystemSample, PageView, Metric.
Throughput & error rate
SELECT rate(count(*), 1 minute) FROM Transaction WHERE appName='auth-api' TIMESERIES
SELECT percentage(count(*), WHERE error IS true) FROM Transaction SINCE 30 minutes ago
Percentiles (p95 / p99)
SELECT percentile(duration, 50, 95, 99) FROM Transaction
FACET name SINCE 1 hour ago TIMESERIES
duration is in seconds. FACET name = break down per endpoint/transaction.
Metrics on logs
SELECT count(*) FROM Log WHERE level='error' FACET service, message SINCE 1 day ago
SELECT count(*) FROM Log WHERE message LIKE '%timeout%' TIMESERIES 5 minutes
Once logs are JSON, any field becomes a filterable/aggregatable dimension.
Time & bucketing helpers
SINCE 1 hour ago UNTIL 30 minutes ago
TIMESERIES AUTO        -- auto bucket width
COMPARE WITH 1 week ago  -- overlay prior period
LIMIT 100
Aggregations
count(*)  average(x)  sum(x)  max(x)  min(x)
uniqueCount(userId)  latest(x)  histogram(duration, 50, 20)  apdex(duration, t:0.5)

p95 / p99 — Tail Latency

What a percentile means
p95 = 95% of requests were faster than this value; only 5% (the tail) were slower. p99 exposes worst-case UX that averages hide.
Why not average
avg is skewed by outliers and hides bimodal behavior. A 50ms avg can still mean p99=4s → 1% of users suffer. SLOs target p95/p99, never mean.
Percentiles don't add / average
You cannot average p99s across hosts or time buckets to get an overall p99. Query the raw distribution over the full window instead.

Dashboards, Alerts & Status Pages

Dashboard = saved NRQL widgets
Build charts by pasting NRQL; each widget is one query. Use FACET for breakdowns, TIMESERIES for trends, billboard for single-number KPIs. Add variables (template) to make dashboards reusable across services/env.
NRQL alert condition
-- signal query (must return one value / thin FACET)
SELECT percentile(duration, 95) FROM Transaction WHERE appName='auth-api'
-- threshold: static, > 0.5 for at least 5 minutes
Static (fixed number), Baseline (deviation from learned normal), or Anomaly. Set 'for at least N min' to avoid flapping.
Policy vs Condition vs Incident
Condition = one rule. Policy = group of conditions + how they roll up into incidents. Breach opens an Incident/Issue ('Problem'); notifiers fire; auto-closes on recovery.
Notifiers / channels
Route issues to Slack, email, PagerDuty, Opsgenie, or generic Webhook via a Workflow (filter issues → destination). Set severity + de-dup to avoid alert fatigue.
Webhook notifier payload
{
  "issueUrl": "{{issuePageUrl}}",
  "title": "{{annotations.title.[0]}}",
  "priority": "{{priority}}",
  "state": "{{state}}"
}
Status page
Public-facing uptime + incident history for users (statuspage.io / Better Stack / instatus). Feed it synthetic checks or alert webhooks. Separate concern from internal dashboards.

Quick tips & gotchas

  • require('newrelic') MUST be the very first line loaded — load it late and the agent can't monkey-patch http/express/pg, so you get zero data.
  • Give every instance the SAME NEW_RELIC_APP_NAME so replicas aggregate into one APM entity; a typo silently creates a second ghost service.
  • Alert and set SLOs on p95/p99, never on average — the mean hides the slow tail that actually hurts users.
  • Percentiles can't be averaged or summed across hosts/buckets; recompute over the full window or you'll report a wrong p99.
  • Log JSON, not string concatenation — unstructured logs can't be FACETed or aggregated into metrics in NRQL.
  • Winston level filters by severity: level:'info' emits error/warn/info but DROPS debug — bump LOG_LEVEL=debug to see them.
  • Add 'for at least N minutes' + recovery threshold on alert conditions to stop flapping and alert-fatigue.
  • Never log secrets, tokens, or PII — logs are queryable, retained, and widely readable; scrub before shipping to NewRelic.
