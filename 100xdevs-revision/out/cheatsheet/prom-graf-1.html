Prometheus and Grafana — Cheat Sheet

Cohort 2.0 · 16 lessons · quick reference

Setup & Instrumentation (Node/prom-client)

Install client lib
npm install prom-client
prom-client is the official Node exporter; Prometheus pulls (scrapes) — the app never pushes.
Import + default process metrics
const client = require('prom-client');
const register = new client.Registry();
register.setDefaultLabels({ app: 'my-service' });
client.collectDefaultMetrics({ register });
collectDefaultMetrics gives CPU, heap, event-loop lag, GC for free. Register scopes which metrics an endpoint exposes.
Expose /metrics endpoint
app.get('/metrics', async (req, res) => {
  res.set('Content-Type', register.contentType);
  res.end(await register.metrics());
});
register.metrics() is async and returns text/plain in the Prometheus exposition format. This is the URL Prometheus scrapes.

Metric Types

Counter — monotonically increasing (requests, errors)
const reqCounter = new client.Counter({
  name: 'http_requests_total',
  help: 'Total HTTP requests',
  labelNames: ['method', 'route', 'status_code'],
  registers: [register],
});
reqCounter.inc();                                  // +1
reqCounter.inc({ method: 'GET', route: '/', status_code: 200 });
reqCounter.labels('GET', '/', '200').inc(5);
Only .inc() — never decreases. End name with _total by convention. Query rate() not the raw value.
Gauge — value that goes up AND down (active conns, queue depth, temp)
const active = new client.Gauge({
  name: 'active_connections',
  help: 'Currently open connections',
  registers: [register],
});
active.set(10);
active.inc(); active.dec();
active.setToCurrentTime();
const stop = active.startTimer();  // in-progress timer; stop() sets elapsed seconds
Use for point-in-time snapshots. .set() overwrites; don't use rate() on gauges.
Histogram — distribution/latency, produces buckets + _sum + _count
const httpDur = new client.Histogram({
  name: 'http_request_duration_seconds',
  help: 'Request duration in seconds',
  labelNames: ['method', 'route', 'code'],
  buckets: [0.005, 0.01, 0.05, 0.1, 0.3, 0.5, 1, 3, 5],
  registers: [register],
});
const end = httpDur.startTimer();
// ...handle request...
end({ method: 'GET', route: '/', code: 200 });
httpDur.observe(0.42);  // manual observation
Emits _bucket{le="..."}, _sum, _count. Choose buckets to bracket your SLO. Buckets are cumulative (le = 'less than or equal').
Summary — client-side quantiles (rarely needed)
const sum = new client.Summary({
  name: 'resp_size_bytes', help: '...',
  percentiles: [0.5, 0.9, 0.99], registers: [register],
});
Prefer Histogram: summaries can't be aggregated across instances; histograms can via histogram_quantile().

Better Structure — Middleware & Labels

Single middleware instruments every route
app.use((req, res, next) => {
  const end = httpDur.startTimer();
  res.on('finish', () => {
    const route = req.route?.path ?? req.path;
    const labels = { method: req.method, route, code: res.statusCode };
    end(labels);            // observe duration
    reqCounter.inc(labels); // increment count
  });
  next();
});
res.on('finish') captures the real final status code. Use req.route.path (templated) not req.path to avoid high-cardinality label blowup.
Keep metric defs in one module
// metrics.js
module.exports = { register, reqCounter, httpDur, active };
Define each metric ONCE at module load. Re-registering the same name throws 'A metric with the name X has already been registered'.
Label cardinality discipline
Never label with user IDs, emails, timestamps, or raw URLs — each unique combo = a new time series and can OOM Prometheus. Keep labels to bounded enums (method, route template, status class).

Prometheus Config & Running

prometheus.yml — scrape config
global:
  scrape_interval: 15s
  evaluation_interval: 15s

scrape_configs:
  - job_name: 'node-app'
    metrics_path: /metrics   # default
    static_configs:
      - targets: ['host.docker.internal:3000']
From a container, reach the host app via host.docker.internal (Docker Desktop) — not localhost, which is the container itself.
Run Prometheus in Docker
docker run -p 9090:9090 \
  -v $(pwd)/prometheus.yml:/etc/prometheus/prometheus.yml \
  prom/prometheus
UI at http://localhost:9090. Check Status > Targets to confirm your app is UP and being scraped.
Reload config without restart
curl -X POST http://localhost:9090/-/reload
Requires the flag --web.enable-lifecycle at startup. Otherwise restart the container.

PromQL — Queries & Graphs

Raw + filtered selection
http_requests_total
http_requests_total{route="/user", code=~"2.."}
= exact, != not, =~ regex match, !~ regex no-match. Instant vector = one point per series.
rate() — per-second average over window (the go-to for counters)
rate(http_requests_total[5m])
sum(rate(http_requests_total[5m])) by (route)
ALWAYS wrap counters in rate()/increase() — raw counter numbers reset to 0 on restart and are meaningless alone. [5m] = range vector.
increase() / irate()
increase(http_requests_total[1h])   # total growth over 1h
irate(http_requests_total[5m])      # instant rate (last 2 samples)
irate for volatile/fast-moving graphs; rate for smoother alerting.
p95/p99 latency from histogram
histogram_quantile(
  0.95,
  sum(rate(http_request_duration_seconds_bucket[5m])) by (le)
)
Must aggregate by (le). Accuracy is bounded by your bucket boundaries.
Error ratio
sum(rate(http_requests_total{code=~"5.."}[5m]))
/
sum(rate(http_requests_total[5m]))
Graph tab renders any query over time; Table tab shows instant values.

Grafana (docker-compose + datasource)

docker-compose.yml — Prometheus + Grafana together
services:
  prometheus:
    image: prom/prometheus
    ports: ['9090:9090']
    volumes:
      - ./prometheus.yml:/etc/prometheus/prometheus.yml
  grafana:
    image: grafana/grafana
    ports: ['3001:3000']
    environment:
      - GF_SECURITY_ADMIN_PASSWORD=admin
    volumes:
      - grafana-data:/var/lib/grafana
volumes:
  grafana-data:
Map Grafana to 3001 to avoid clashing with a Node app on 3000 (Grafana listens on 3000 internally). The named volume persists dashboards across restarts.
Add Prometheus as a data source
URL: http://prometheus:9090
Use the compose SERVICE NAME (prometheus), not localhost — containers resolve each other by service name on the shared network. Connections > Data sources > Add > Prometheus > Save & test.
Provision datasource as code (optional)
# ./grafana/provisioning/datasources/ds.yml
apiVersion: 1
datasources:
  - name: Prometheus
    type: prometheus
    access: proxy
    url: http://prometheus:9090
    isDefault: true
Mount to /etc/grafana/provisioning/datasources so it's auto-configured on boot — no manual clicks.
Build a panel
New dashboard > Add visualization > pick Prometheus > paste a PromQL query (e.g. rate(...)). Import community dashboards by ID via Dashboards > Import (e.g. Node Exporter Full).

Alerting

Prometheus alert rule file
# alerts.yml
groups:
  - name: app-alerts
    rules:
      - alert: HighErrorRate
        expr: |
          sum(rate(http_requests_total{code=~"5.."}[5m]))
          / sum(rate(http_requests_total[5m])) > 0.05
        for: 5m
        labels: { severity: critical }
        annotations:
          summary: "5xx error rate > 5% for 5m"
for: makes the condition hold N minutes before firing (avoids flapping). State goes Pending -> Firing.
Wire rules + Alertmanager into prometheus.yml
rule_files:
  - 'alerts.yml'
alerting:
  alertmanagers:
    - static_configs:
        - targets: ['alertmanager:9093']
Prometheus evaluates rules and fires to Alertmanager, which handles routing/dedup/silencing and sends to Slack, email, PagerDuty, etc.
Grafana-native alerts (no Alertmanager needed)
Alerting > Alert rules > New: define a query + threshold, set evaluation interval + pending period, attach a contact point (Slack/email webhook) and notification policy. Simpler for a single-team setup.

Quick tips & gotchas

  • Prometheus PULLS metrics by scraping /metrics on an interval — your app just exposes an endpoint, it never pushes. For short-lived jobs use the Pushgateway.
  • Always wrap counters in rate()/increase() — a bare counter resets to 0 on restart and its absolute value is meaningless.
  • Label cardinality is the #1 way to kill Prometheus: never put user IDs, request IDs, timestamps, or raw URLs in labels. Use route templates and status classes.
  • Prefer Histograms over Summaries — histogram buckets aggregate across instances via histogram_quantile(); summary quantiles cannot be summed.
  • Choose histogram buckets around your SLO/expected latency; percentile accuracy is limited by bucket boundaries, and too many buckets inflate series count.
  • In Docker, use service names between containers (http://prometheus:9090) and host.docker.internal to reach an app on the host — never localhost across the network boundary.
  • Confirm scraping in Prometheus UI > Status > Targets (state UP) before debugging dashboards; a blank Grafana panel is usually a broken target or wrong datasource URL.
  • Metric naming: base unit suffix (_seconds, _bytes), _total for counters, and histograms auto-emit _bucket/_sum/_count. Define each metric exactly once or registration throws.
