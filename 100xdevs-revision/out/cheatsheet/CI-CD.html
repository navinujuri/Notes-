CI/CD — Cheat Sheet

Cohort 2.0 · 8 lessons · quick reference

CI vs CD — concepts

CI (Continuous Integration)
Every push/PR auto-runs lint + tests + build so broken code is caught before merge. Keep branches short-lived, merge often.
CD — Delivery vs Deployment
Continuous DELIVERY = every green build is release-ready but a human clicks deploy. Continuous DEPLOYMENT = green build ships to prod automatically, no gate.
Typical pipeline stages
checkout -> install -> lint -> test -> build -> dockerize -> push -> deploy -> pull & run
Where it runs
GitHub Actions spins up an ephemeral VM (ubuntu-latest) per job; filesystem/env do NOT persist across jobs unless you use artifacts/cache.
Trigger-on-merge model
CI on PRs (push to feature branch), CD on push to main. Gate prod behind branch protection + required checks.

GitHub Actions anatomy

File location
.github/workflows/ci.yml   # any *.yml under this dir is auto-detected
Repo must have Actions enabled. One file = one workflow.
Minimal workflow skeleton
name: CI
on:
  push:
    branches: [main]
  pull_request:
    branches: [main]
jobs:
  build:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4
Trigger events (on:)
on:
  push: { branches: [main] }
  pull_request:
  workflow_dispatch:        # manual 'Run workflow' button
  schedule: [{ cron: '0 2 * * *' }]
Checkout the repo (required first step)
- uses: actions/checkout@v4
Without this the runner has no source code. fetch-depth: 0 for full history/tags.
Setup a language runtime
- uses: actions/setup-node@v4
  with:
    node-version: 20
    cache: npm            # caches ~/.npm keyed on lockfile
Run a shell step
- name: Install
  run: npm ci             # ci = clean, lockfile-exact, deletes node_modules
job dependencies
jobs:
  test: { runs-on: ubuntu-latest, steps: [...] }
  deploy:
    needs: test           # deploy waits for test to pass
    runs-on: ubuntu-latest

Step 1 — the CI pipeline (lint/test/build)

Standard CI job
jobs:
  ci:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4
      - uses: actions/setup-node@v4
        with: { node-version: 20, cache: npm }
      - run: npm ci
      - run: npm run lint
      - run: npm run build
      - run: npm test
npm ci vs npm install
Use npm ci in CI: fails if package-lock.json is out of sync, reproducible, faster. Never npm install in pipelines.
Matrix (test across versions)
strategy:
  matrix:
    node: [18, 20, 22]
steps:
  - uses: actions/setup-node@v4
    with: { node-version: ${{ matrix.node }} }
Fail fast / continue
strategy:
  fail-fast: false        # let all matrix legs finish
Default fail-fast: true cancels siblings on first failure.
Manual cache (non-node)
- uses: actions/cache@v4
  with:
    path: ~/.cache
    key: ${{ runner.os }}-${{ hashFiles('**/lockfile') }}

Monorepo — build only what changed

Path filter — skip unaffected apps
on:
  push:
    paths:
      - 'apps/web/**'
      - 'packages/ui/**'
      - '.github/workflows/**'
Workflow only fires when matching files change — avoids rebuilding the whole monorepo.
Run in a subdir
- name: Build web
  working-directory: apps/web
  run: npm ci && npm run build
Or set defaults.run.working-directory at job level.
Matrix over packages
strategy:
  matrix:
    app: [web, admin, api]
steps:
  - run: npm run build --workspace=apps/${{ matrix.app }}
Turborepo / affected builds
- run: npx turbo run build --filter=...[origin/main]
Turbo/Nx compute the dependency graph and build only affected + downstream packages; caches unchanged ones.
npm workspaces
npm ci                          # installs ALL workspaces from root lockfile
npm run build -w apps/web       # target one workspace

Build & push the Docker image

Dockerfile (multi-stage, monorepo app)
FROM node:20-alpine AS build
WORKDIR /app
COPY package*.json ./
RUN npm ci
COPY . .
RUN npm run build

FROM node:20-alpine
WORKDIR /app
COPY --from=build /app/dist ./dist
COPY package*.json ./
RUN npm ci --omit=dev
CMD ["node", "dist/index.js"]
Login to registry (Docker Hub)
- uses: docker/login-action@v3
  with:
    username: ${{ secrets.DOCKERHUB_USERNAME }}
    password: ${{ secrets.DOCKERHUB_TOKEN }}
Build & push in one step
- uses: docker/build-push-action@v6
  with:
    context: .
    file: ./apps/web/Dockerfile
    push: true
    tags: |
      myuser/web:latest
      myuser/web:${{ github.sha }}
Tag with the commit SHA for immutable, rollback-able deploys; latest is a moving pointer.
Buildx + layer cache
- uses: docker/setup-buildx-action@v3
- uses: docker/build-push-action@v6
  with:
    cache-from: type=gha
    cache-to: type=gha,mode=max
Push to GHCR instead
- uses: docker/login-action@v3
  with:
    registry: ghcr.io
    username: ${{ github.actor }}
    password: ${{ secrets.GITHUB_TOKEN }}
# tags: ghcr.io/owner/web:${{ github.sha }}
GITHUB_TOKEN is auto-provided; add permissions: { packages: write } to the job.

Deploy step — SSH to server & pull the image

Deploy job (gated on CI)
deploy:
  needs: build
  runs-on: ubuntu-latest
  if: github.ref == 'refs/heads/main'
  steps:
    - uses: appleboy/ssh-action@v1
      with:
        host: ${{ secrets.SSH_HOST }}
        username: ${{ secrets.SSH_USER }}
        key: ${{ secrets.SSH_KEY }}
        script: |
          docker pull myuser/web:latest
          docker stop web || true
          docker rm web || true
          docker run -d --name web -p 80:3000 --restart unless-stopped myuser/web:latest
Pull the image on the box
docker pull myuser/web:latest
Server must be logged in (docker login) if the repo is private.
Zero-ish downtime swap
docker pull myuser/web:${SHA}
docker stop web || true && docker rm web || true
docker run -d --name web -p 80:3000 myuser/web:${SHA}
|| true so the run doesn't fail on first deploy when no container exists yet.
Prune old images
docker image prune -f
Servers fill up fast pulling every SHA tag — prune or cron it.
Compose-based deploy
docker compose pull && docker compose up -d
Pulls new images referenced in compose.yml and recreates changed services only.

Secrets, permissions & config

Reference a secret
${{ secrets.DOCKERHUB_TOKEN }}
Masked in logs. NEVER echo raw secrets. Repo/org level or per-environment.
Add secrets via gh CLI
gh secret set DOCKERHUB_TOKEN
gh secret set SSH_KEY < ~/.ssh/id_ed25519
gh secret list
Built-in GITHUB_TOKEN
permissions:
  contents: read
  packages: write
Auto-generated per run, scoped to the repo; grant least privilege explicitly.
Contexts / expressions
${{ github.sha }}      # commit
${{ github.ref }}      # refs/heads/main
${{ github.actor }}    # who triggered
${{ env.MY_VAR }}
Environment-scoped deploys
jobs:
  deploy:
    environment: production   # enables required reviewers / protection rules
Adds a manual approval gate — turns continuous delivery into deployment-with-a-gate.
Job-level env
env:
  NODE_ENV: production
  API_URL: ${{ secrets.API_URL }}

Quick tips & gotchas

  • Tag Docker images with ${{ github.sha }}, not just latest — you get immutable builds and can roll back to any commit instantly.
  • Use npm ci (never npm install) in pipelines: it's reproducible, respects the lockfile, and fails loudly on drift.
  • In monorepos, gate workflows with on.push.paths so touching apps/web doesn't rebuild apps/api — huge CI-time savings.
  • needs: creates the dependency graph; without it jobs run in parallel and a deploy can race ahead of its tests.
  • || true after docker stop/rm so the deploy script survives the first run when the container doesn't exist yet.
  • Cache aggressively: cache: npm in setup-node and type=gha for Docker layers cut minutes off every run.
  • Guard deploy with if: github.ref == 'refs/heads/main' so PR builds test but never ship to prod.
  • Runners are ephemeral — nothing persists between jobs. Pass data via actions/upload-artifact / download-artifact.
