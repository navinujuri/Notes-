Frontend Deployments on AWS — Cheat Sheet

Cohort 2.0 · 9 lessons · quick reference

1. Build the React bundle

Vite production build
npm run build
# outputs to ./dist  (index.html + assets/*.[hash].js|css)
Vite default output dir is dist/. Filenames are content-hashed for cache busting.
Create React App build
npm run build
# outputs to ./build
CRA uses build/ not dist/. Know which one your tool emits before uploading.
Set base path (S3 subfolder / non-root)
// vite.config.js
export default defineConfig({ base: '/' })  // '/' for root domain
// package.json (CRA)
"homepage": "https://cdn.example.com"
Wrong base = broken asset URLs (blank page, 404 on /assets/*). Keep '/' when served at domain root.
Verify the bundle locally before upload
npm run preview          # Vite serves dist/ on :4173
npx serve -s build       # CRA / any static dir
-s = SPA mode: serves index.html for unknown routes so client routing works.

2. CDN concepts (why CloudFront)

What a CDN does
Caches static assets at edge PoPs near users -> lower latency, less origin (S3) load, TLS termination + HTTP/2/3.
Origin vs edge
Origin = S3 bucket (source of truth). Edge = CloudFront cache copies. Cache miss -> fetch from origin, then serve cached.
Cache key & TTL
Object cached until TTL expires or you invalidate. Hashed asset filenames -> cache forever; index.html -> short/no cache.
Why not just S3 website URL
Raw S3 website endpoint = HTTP only, no custom SSL, single region latency. CloudFront adds HTTPS + global edge + custom domain.

3. Create S3 bucket & upload

Create bucket
aws s3 mb s3://my-app-frontend --region us-east-1
Bucket names are globally unique. For CloudFront-only access, keep it private (don't enable public site).
Enable static website hosting (public-endpoint path)
aws s3 website s3://my-app-frontend \
  --index-document index.html \
  --error-document index.html
error-document = index.html makes SPA deep links work on the S3 website endpoint.
Allow public read (only if using public S3 endpoint)
aws s3api put-public-access-block --bucket my-app-frontend \
  --public-access-block-configuration BlockPublicPolicy=false,RestrictPublicBuckets=false
# then attach bucket policy allowing s3:GetObject on arn:aws:s3:::my-app-frontend/*
Skip this entirely if using CloudFront + OAC (bucket stays fully private — preferred).
Upload the bundle (sync)
aws s3 sync ./dist s3://my-app-frontend --delete
--delete removes stale objects (old hashed assets) so the bucket mirrors dist/ exactly.
Cache headers per file type
# long cache for hashed assets
aws s3 sync ./dist/assets s3://my-app-frontend/assets \
  --cache-control "public,max-age=31536000,immutable"
# no cache for the HTML entrypoint
aws s3 cp ./dist/index.html s3://my-app-frontend/index.html \
  --cache-control "no-cache"
index.html must NOT be long-cached or users get stale bundles pointing at deleted asset hashes.

4. Access via S3 website endpoint

Website endpoint URL format
http://<bucket>.s3-website-<region>.amazonaws.com
# e.g. http://my-app-frontend.s3-website-us-east-1.amazonaws.com
HTTP only. Region-specific hostname. This is the quick smoke test before wiring CloudFront.
Website vs REST endpoint (don't mix them up)
s3-website-* = serves index/error docs, SPA-friendly. *.s3.amazonaws.com (REST) = returns XML/403 for a bare path, no index resolution.
403 / 404 troubleshooting
403 usually = block-public-access still on or missing bucket policy. 404 = object key wrong or index-document not set.

5. Put CloudFront in front

Create distribution (recommended: OAC + private bucket)
Origin = S3 REST domain (my-app-frontend.s3.us-east-1.amazonaws.com). Attach Origin Access Control (OAC) so only CloudFront can read the bucket.
OAC bucket policy
{
  "Effect": "Allow",
  "Principal": { "Service": "cloudfront.amazonaws.com" },
  "Action": "s3:GetObject",
  "Resource": "arn:aws:s3:::my-app-frontend/*",
  "Condition": { "StringEquals":
    { "AWS:SourceArn": "arn:aws:cloudfront::<acct>:distribution/<dist-id>" } }
}
OAC replaces the deprecated OAI. Bucket public access can stay fully blocked.
Default root object
aws cloudfront ... --default-root-object index.html
Without this, hitting https://d123.cloudfront.net/ returns a 403/AccessDenied instead of your app.
Redirect HTTP -> HTTPS
Set Viewer Protocol Policy = redirect-to-https on the default cache behavior.
Invalidate cache after each deploy
aws cloudfront create-invalidation \
  --distribution-id E123ABC \
  --paths "/index.html" "/"
Hashed assets need no invalidation. Only invalidate index.html (and /) — invalidations beyond the free 1000/mo cost money.

6. Custom domain + HTTPS

Request ACM cert — MUST be us-east-1
aws acm request-certificate \
  --domain-name app.example.com \
  --validation-method DNS \
  --region us-east-1
CloudFront ONLY accepts certs from us-east-1 (N. Virginia), regardless of where your bucket lives. Validate via DNS CNAME.
Attach domain to distribution
Add app.example.com to CloudFront 'Alternate domain names (CNAMEs)' and select the ACM cert. Cert domain must cover the CNAME.
Point DNS at CloudFront
# Route 53 (best: alias A/AAAA record)
app.example.com  ALIAS -> d123abc.cloudfront.net
# External DNS provider
app.example.com  CNAME -> d123abc.cloudfront.net
Route 53 alias lets you point an apex/root domain; a plain CNAME cannot be used on the zone apex.

7. Error pages & SPA routing

The SPA deep-link problem
/dashboard has no S3 object -> S3 returns 403/404. React Router never loads. Fix: rewrite errors to index.html with 200.
CloudFront custom error responses (OAC/REST origin)
Error code:        403   (and 404)
Response page path: /index.html
HTTP response code: 200
Error caching TTL:  0
With a private bucket + OAC, missing keys return 403 — map BOTH 403 and 404 to /index.html so client routing takes over.
S3 website-endpoint alternative
aws s3 website s3://my-app-frontend \
  --index-document index.html --error-document index.html
If origin is the S3 *website* endpoint (not REST), error-document=index.html handles routing — but you lose OAC/private bucket.
Real 404 page (optional)
Ship a 404.html and only map genuine not-found routes to it inside your app; keep the catch-all -> index.html for real routes.

Quick tips & gotchas

  • ACM certs for CloudFront MUST be in us-east-1 — a cert in your bucket's region will not appear in the distribution dropdown.
  • Never long-cache index.html. Cache hashed assets forever (immutable, 1yr) but serve HTML with no-cache so new deploys are picked up.
  • Prefer OAC over public buckets: keep Block Public Access ON, let only CloudFront read S3 via the OAC bucket policy.
  • Deploy = sync + invalidate. `aws s3 sync --delete` then `create-invalidation --paths /index.html` — forgetting the invalidation serves the old app.
  • Set --default-root-object index.html or the CloudFront root URL returns AccessDenied.
  • Map BOTH 403 and 404 CloudFront error responses to /index.html (HTTP 200) or React Router deep links break on refresh.
  • Match your build's `base` (Vite) / `homepage` (CRA) to the serving path — mismatch shows a blank page with 404s on /assets/*.
  • Use `aws s3 sync` not `cp` for deploys; --delete prunes old hashed files so the bucket exactly mirrors dist/.
