Turborepo and Monorepos — Cheat Sheet

Cohort 2.0 · 18 lessons · quick reference

Monorepo Concepts & Why

Monorepo
One repo holding many apps + packages, versioned together. NOT one giant app (that's a monolith).
Why
Atomic cross-package commits, shared code/config/tooling, single install, easy refactors, consistent deps. Cost: needs a task orchestrator to stay fast.
Polyrepo pain it solves
No more publishing internal libs to npm just to share; import via workspace protocol instead.
Typical layout
apps/       # deployables (web, docs, api)
packages/   # shared libs (ui, config, utils)

Node.js Monorepo Tooling

Workspaces (built-in)
npm/yarn/pnpm all support workspaces = symlink local packages into node_modules. The foundation everything else builds on.
Turborepo
Task orchestrator + cache layer over workspaces. Doesn't bundle — runs YOUR scripts smartly.
Nx
Heavier: orchestration + generators + plugins + graph. More opinionated than Turbo.
Lerna
Original JS monorepo tool; now focuses on versioning/publishing (delegates task running to Nx).
Changesets
Common companion for versioning + changelogs + publishing packages.

Build System vs Orchestrator

Build system / bundler
tsc, webpack, vite, next build, esbuild — transforms source into artifacts.
Orchestrator (Turborepo)
Decides WHAT to run, in WHAT ORDER, in PARALLEL, and skips work via cache. Calls the build systems; never replaces them.
History
Turborepo built by Jared Palmer, acquired by Vercel (Dec 2021). v2.0 (2024) renamed turbo.json `pipeline` -> `tasks`.

Internal package naming
Convention: @repo/* (or your org scope). private:true so it never publishes to npm.

Workspaces & package.json

Root package.json
{
  "name": "monorepo",
  "private": true,
  "packageManager": "pnpm@9.0.0",
  "scripts": {
    "build": "turbo run build",
    "dev": "turbo run dev",
    "lint": "turbo run lint"
  },
  "devDependencies": { "turbo": "^2.0.0" }
}
private:true is REQUIRED at root. No source here — only scripts + shared devDeps.
Declare workspaces (npm/yarn)
"workspaces": ["apps/*", "packages/*"]
Declare workspaces (pnpm)
# pnpm-workspace.yaml
packages:
  - "apps/*"
  - "packages/*"
pnpm ignores the package.json workspaces field — uses this YAML instead.
Consume a local package
// apps/web/package.json
"dependencies": {
  "@repo/ui": "workspace:*"
}
workspace:* = always link the local version, never fetch from npm.
packages/ui — expose entrypoints
// packages/ui/package.json
{
  "name": "@repo/ui",
  "exports": {
    "./button": "./src/button.tsx",
    "./card": "./src/card.tsx"
  }
}
Modern approach ships raw TS/TSX; the consuming app transpiles it (Next: transpilePackages: ['@repo/ui']).
Import shared code
import { Button } from "@repo/ui/button";

Running Tasks (turbo CLI)

Run a task across all packages
turbo run build
Run dev for everything
turbo run dev   # parallel, persistent
Filter to one app
turbo run build --filter=web
App + its dependencies
turbo run build --filter=web...   # ... = include deps
Only changed since a ref
turbo run build --filter='...[origin/main]'
Ignore cache / force rerun
turbo run build --force
See plan without executing
turbo run build --dry-run
Prune for Docker
turbo prune web --docker
Emits a minimal subset (out/) with only web + its deps — small, cacheable Docker layers.

turbo.json Config

Basic tasks pipeline
{
  "$schema": "https://turbo.build/schema.json",
  "tasks": {
    "build": {
      "dependsOn": ["^build"],
      "outputs": [".next/**", "!.next/cache/**", "dist/**"]
    },
    "lint": {},
    "dev": { "cache": false, "persistent": true }
  }
}
Key is `tasks` in v2+ (was `pipeline`). `dev` must set persistent:true (long-running) and cache:false.
^ prefix = topological dep
"dependsOn": ["^build"]
^build = build all internal DEPENDENCIES first. Bare "build" = a task in the SAME package must run first.
outputs = what to cache
Files Turbo saves/restores on a cache hit. Miss these and restore produces empty dirs. ! excludes.
inputs = what invalidates cache
"test": { "inputs": ["src/**", "test/**"] }
Narrow inputs so unrelated file changes don't bust the cache.
Env vars in hash
"build": { "env": ["API_URL"] }
// or global:
"globalEnv": ["NODE_ENV"]
Env vars affecting output MUST be listed or Turbo serves a stale cached build.
Global invalidators
"globalDependencies": [".env", "tsconfig.json"]
Changing these busts the cache for ALL tasks.
Quiet cached logs
"outputLogs": "new-only"

Caching, React & Node Apps

How caching works
Turbo hashes inputs (files + deps + env + config). Same hash -> skips task, restores outputs + replays logs. '>>> FULL TURBO' = everything hit.
Local cache location
node_modules/.turbo   (and .turbo/)
Remote cache (share across CI/team)
turbo login
turbo link
Free with Vercel; or self-host. CI reuses a teammate's/previous build's artifacts.
Add a React (Next.js) app
cd apps && npx create-next-app@latest web
# then add "@repo/ui": "workspace:*" and
# transpilePackages: ['@repo/ui'] in next.config
Add a Node.js app
mkdir apps/api
# package.json: name @repo/api, scripts build(tsc)/dev/start
# turbo picks up build/dev automatically via turbo.json tasks
Add a common shared module
packages/utils/
  package.json  -> name "@repo/utils", exports "./index"
  src/index.ts
# consume: "@repo/utils": "workspace:*"
Run pnpm install after adding a package so symlinks + graph update.

Quick tips & gotchas

  • v2 breaking change: turbo.json uses `tasks`, not `pipeline`. Old repos on v1 still use `pipeline`.
  • Root package.json MUST be private:true, and each shared package should be private:true unless you actually publish it.
  • Empty dirs after a cache hit? Your `outputs` glob is wrong/missing — Turbo only restores what you list.
  • Stale build after changing an env var? Add it to task `env` or `globalEnv` so it enters the cache hash.
  • cache:false + persistent:true is mandatory for `dev` and any watch/long-running task.
  • Use `--filter=app...` to include deps and `--filter='[HEAD^1]'` in CI to build only what changed.
  • pnpm reads pnpm-workspace.yaml, NOT the package.json `workspaces` field — set the right one for your PM.
  • Turborepo orchestrates and caches; it never bundles. Your tsc/next/vite scripts still do the actual building.
