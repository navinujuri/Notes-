Kafka — Cheat Sheet

Cohort 2.0 · 9 lessons · quick reference

Core concepts & jargon

Kafka = distributed, append-only commit log for event streaming
Pull-based, durable, replayable. NOT a queue that deletes on read — messages persist per retention.
Broker
One Kafka server. A cluster = many brokers sharing partitions.
Topic
Named stream/category of messages (e.g. 'orders').
Partition
Ordered, immutable log within a topic. Unit of parallelism. Ordering guaranteed ONLY within a partition, never across a topic.
Offset
Sequential id of a message inside a partition. Consumers track 'where they are' by committing offsets.
Producer / Consumer
Producers write; consumers read. Decoupled — neither knows the other.
Consumer group
Consumers sharing a groupId that SPLIT a topic's partitions among themselves.
Replication factor / retention
RF = copies of a partition across brokers. Retention = keep by time/size (log.retention.hours), not by consumption.
KRaft vs ZooKeeper
Modern Kafka runs metadata in KRaft mode (no separate ZooKeeper needed).

Start Kafka locally (Docker + CLI)

Run a broker (KRaft, no ZooKeeper)
docker run -p 9092:9092 apache/kafka:3.7.1
Shell into the container
docker ps                       # get container id
docker exec -it <id> /bin/bash
cd /opt/kafka/bin               # all *.sh tools live here
Create a topic with partitions
./kafka-topics.sh --create --topic quickstart \
  --partitions 3 --replication-factor 1 \
  --bootstrap-server localhost:9092
List / describe topics
./kafka-topics.sh --list --bootstrap-server localhost:9092
./kafka-topics.sh --describe --topic quickstart --bootstrap-server localhost:9092
--describe shows leader/replicas per partition.
Console producer / consumer (quick sanity check)
./kafka-console-producer.sh --topic quickstart --bootstrap-server localhost:9092
./kafka-console-consumer.sh --topic quickstart --from-beginning --bootstrap-server localhost:9092
Inspect a consumer group (offsets + LAG)
./kafka-consumer-groups.sh --describe --group group-1 \
  --bootstrap-server localhost:9092
CURRENT-OFFSET vs LOG-END-OFFSET gives LAG per partition — key debugging tool.

KafkaJS setup (Node.js)

Install
npm install kafkajs
Create client (shared across producer/consumer/admin)
const { Kafka } = require('kafkajs');
const kafka = new Kafka({
  clientId: 'my-app',
  brokers: ['localhost:9092'],
});
Admin: create topic with N partitions from code
const admin = kafka.admin();
await admin.connect();
await admin.createTopics({
  topics: [{ topic: 'quickstart', numPartitions: 3, replicationFactor: 1 }],
});
await admin.disconnect();
Idempotent — returns false if the topic already exists.
Graceful shutdown
process.on('SIGINT', async () => {
  await consumer.disconnect();
  await producer.disconnect();
  process.exit(0);
});
Disconnect triggers a clean group rebalance instead of a session timeout.

Producer script

Minimal producer
// producer.js
const producer = kafka.producer();
await producer.connect();
await producer.send({
  topic: 'quickstart',
  messages: [{ value: 'hello' }],
});
await producer.disconnect();
Send WITH a key (controls routing)
await producer.send({
  topic: 'quickstart',
  messages: [{ key: 'user-1', value: 'order-created' }],
});
Same key → same partition → per-key ordering preserved. Use key = entity id (userId, orderId).
Batch multiple messages / multiple topics
await producer.sendBatch({
  topicMessages: [
    { topic: 'a', messages: [{ value: '1' }, { value: '2' }] },
    { topic: 'b', messages: [{ value: '3' }] },
  ],
});
Pin to an explicit partition
messages: [{ value: 'x', partition: 0 }]
Overrides the partitioner — bypasses key hashing entirely.

Consumer script & consumer groups

Minimal consumer
// consumer.js
const consumer = kafka.consumer({ groupId: 'group-1' });
await consumer.connect();
await consumer.subscribe({ topic: 'quickstart', fromBeginning: true });
await consumer.run({
  eachMessage: async ({ topic, partition, message }) => {
    console.log(partition, message.offset, message.value.toString());
  },
});
fromBeginning only applies the FIRST time a group has no committed offset; ignored afterward.
Manual offset commit (at-least-once control)
await consumer.run({
  autoCommit: false,
  eachMessage: async ({ topic, partition, message }) => {
    await handle(message);
    await consumer.commitOffsets([{ topic, partition,
      offset: (Number(message.offset) + 1).toString() }]);
  },
});
Commit offset+1 = the NEXT offset to read. Default autoCommit can lose/replay on crash.
Batch processing
await consumer.run({
  eachBatch: async ({ batch, resolveOffset, heartbeat }) => {
    for (const m of batch.messages) {
      await handle(m);
      resolveOffset(m.offset);
      await heartbeat();
    }
  },
});
Concurrency across partitions
await consumer.run({ partitionsConsumedConcurrently: 3, eachMessage });
Runs eachMessage in parallel across partitions; still serial within one partition.

Partitions & partitioning strategy

Default routing: key present
partition = hash(key) % numPartitions → deterministic, sticky per key.
Default routing: no key
Round-robin / sticky across partitions → good load spread, NO ordering guarantee.
Custom partitioner
const MyPartitioner = () => ({ partitionMetadata, message }) => {
  const n = partitionMetadata.length;
  return Number(message.key) % n;
};
kafka.producer({ createPartitioner: MyPartitioner });
Match Java hashing (KafkaJS v2 gotcha)
const { Partitioners } = require('kafkajs');
kafka.producer({ createPartitioner: Partitioners.LegacyPartitioner });
KafkaJS v2 changed the default partitioner and warns. Use LegacyPartitioner to keep old key→partition mapping / match other clients.
Choosing a key
Pick a key with even cardinality (userId good; country bad → hot partition). Adding partitions later re-hashes keys and breaks existing per-key ordering.

Consumers vs partitions — the three cases

Case 1: consumers < partitions
Some consumers own multiple partitions. Works, but uneven load. E.g. 3 partitions, 2 consumers → 2+1 split.
Case 2: consumers == partitions
Ideal 1:1 mapping — max parallelism, each consumer owns exactly one partition.
Case 3: consumers > partitions
Extra consumers sit IDLE. Max useful consumers in a group = partition count. Add partitions to scale further.
Multiple consumer groups = broadcast (pub/sub)
kafka.consumer({ groupId: 'analytics' });
kafka.consumer({ groupId: 'billing' });
Each group tracks its own offsets, so each group receives a FULL copy of every message. Within a group = load-balanced; across groups = fan-out.

Quick tips & gotchas

  • Ordering is per-partition only — never across a topic. Need global order? Use 1 partition (kills parallelism) or a shared key.
  • Same key → same partition → same consumer. Route related events (same userId) with a key to keep them ordered and co-located.
  • A consumer group can't use more consumers than partitions — extras stay idle. Provision partitions for your peak consumer count up front.
  • Kafka does not delete on read; it's retention-based. Replay by resetting offsets: kafka-consumer-groups.sh --reset-offsets --to-earliest --execute.
  • KafkaJS v2 changed the default partitioner (emits a warning). Set Partitioners.LegacyPartitioner for consistent hashing with Java/other producers.
  • Commit offset = last processed + 1. Committing message.offset (not +1) causes the last message to be reprocessed on restart.
  • Rebalancing pauses ALL consumption in the group. Keep eachMessage fast and call heartbeat() in long loops, or bump sessionTimeout, to avoid being kicked.
  • Adding partitions to a live topic breaks key→partition stability (keys re-hash) — plan partition count ahead; you can't cleanly decrease it.
