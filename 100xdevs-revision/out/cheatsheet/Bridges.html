Bridges — Cheat Sheet

Cohort 3.0 · 9 lessons · quick reference

Mental model & setup

What a bridge actually is
Chains are isolated state machines — no chain can read another. A bridge = lock/burn on source + off-chain relayer + mint/unlock on destination. The relayer is the trust boundary.
Lock-and-mint invariant
minted(dest) == locked(source), always 1:1. Break it (double-mint, unbacked mint) and the peg collapses. This is the #1 thing to monitor.
One-way bridge scope
Value flows source→dest only (lock→mint). No burn→unlock reverse leg. Halves attack surface and state to track.

Source vault — lock the ERC-20

Vault that custodies tokens + emits event with nonce
// SPDX-License-Identifier: MIT
pragma solidity ^0.8.24;
import {IERC20} from "@openzeppelin/contracts/token/ERC20/IERC20.sol";
import {SafeERC20} from "@openzeppelin/contracts/token/ERC20/utils/SafeERC20.sol";

contract Vault {
    using SafeERC20 for IERC20;
    IERC20 public immutable token;
    uint256 public nonce;                     // monotonically increasing
    event Locked(address indexed from, uint256 amount, address indexed to, uint256 nonce);

    constructor(address _token) { token = IERC20(_token); }

    function deposit(uint256 amount, address to) external {
        token.safeTransferFrom(msg.sender, address(this), amount); // pull
        emit Locked(msg.sender, amount, to, nonce++);              // then emit
    }
}
safeTransferFrom, not transferFrom — handles non-standard (USDT-like) tokens that don't return bool.
Why approve + transferFrom (not raw transfer)
token.approve(vaultAddr, amount);   // tx 1 (from user's wallet)
vault.deposit(amount, l2Addr);      // tx 2 -> runs transferFrom
A plain transfer to the contract gives no execution hook — can't emit the structured event the relayer needs, funds can be stranded. EIP-2612 permit collapses the two txs into one.

Destination — mint contract

Wrapped ERC-20 with restricted mint + replay guard
// SPDX-License-Identifier: MIT
pragma solidity ^0.8.24;
import {ERC20} from "@openzeppelin/contracts/token/ERC20/ERC20.sol";
import {Ownable} from "@openzeppelin/contracts/access/Ownable.sol";

contract WrappedToken is ERC20, Ownable {
    address public relayer;
    mapping(uint256 => bool) public processed;   // nonce => minted?

    modifier onlyRelayer() { require(msg.sender == relayer, "not relayer"); _; }

    constructor(address _relayer)
        ERC20("Wrapped GEM", "wGEM") Ownable(msg.sender) { relayer = _relayer; }

    function mint(address to, uint256 amount, uint256 nonce) external onlyRelayer {
        require(!processed[nonce], "replayed");   // check
        processed[nonce] = true;                  // effect
        _mint(to, amount);                        // interaction
    }
}
mint signature mirrors the Locked event args (to, amount, nonce). Checks-effects-interactions: set processed BEFORE _mint.
Access control = the crown jewel
Whoever holds relayer/owner key can mint arbitrarily. Ronin (~$625M) = stolen validator keys. Production: multisig relayer set + rate limits + pausable circuit breaker.

Off-chain relayer

Watch source event, submit mint on destination (ethers v6)
const vault = new ethers.Contract(VAULT, vaultAbi, srcProvider);
const wrapped = new ethers.Contract(WGEM, wgemAbi, relayerWallet); // signer

vault.on("Locked", async (from, amount, to, nonce, ev) => {
  await ev.log.getTransactionReceipt();            // ensure mined
  const tx = await wrapped.mint(to, amount, nonce);
  await tx.wait();
  console.log(`minted for nonce ${nonce}`);
});
Wait for finality before minting
const CONFIRMATIONS = 12;
const rcpt = await srcProvider.getTransactionReceipt(ev.log.transactionHash);
await rcpt.wait?.(CONFIRMATIONS); // or poll block number
Reorg risk: if you mint on an event that later gets reorged out, you've minted unbacked tokens. Trade latency for safety.
Backfill missed events on restart
const logs = await vault.queryFilter("Locked", fromBlock, "latest");
for (const l of logs) { /* re-submit; processed[nonce] dedupes */ }
Messaging is at-least-once — relayer WILL crash and re-send. The nonce mapping makes minting exactly-once.

Nonces & replay protection

The core defense
Each deposit gets a unique, monotonically increasing nonce. Destination stores mapping(uint256=>bool) processed, set on first mint, checked on every mint. Enforces exactly-once over an at-least-once layer.
Attack it prevents
Without it, one legit deposit → relayer replays / attacker resubmits the same mint → unbacked supply printed. Classic double-mint exploit.
Source nonce++ is the sole issuer
emit Locked(msg.sender, amount, to, nonce++); // post-increment: emits current, then bumps
Two-way bridge needs a SEPARATE nonce space for the burn/unlock leg — don't share counters across directions.

Testing the one-way bridge

Setup: mock token + both contracts in one EVM
const { expect } = require("chai");
const { ethers } = require("hardhat");

describe("One-way bridge", () => {
  let token, vault, wrapped, owner, user, relayer;
  const AMT = ethers.parseEther("100");
  beforeEach(async () => {
    [owner, user, relayer] = await ethers.getSigners();
    token = await (await ethers.getContractFactory("MockERC20"))
      .deploy("GEM", "GEM", ethers.parseEther("1000"));
    vault = await (await ethers.getContractFactory("Vault"))
      .deploy(await token.getAddress());
    wrapped = await (await ethers.getContractFactory("WrappedToken"))
      .connect(owner).deploy(relayer.address);
  });
Simulate cross-chain in ONE chain: run lock, read emitted args, then call mint with those exact args as the relayer signer.
Happy path: lock emits event, relayer mints 1:1
it("locks and mints", async () => {
  await token.transfer(user.address, AMT);
  await token.connect(user).approve(await vault.getAddress(), AMT);
  await expect(vault.connect(user).deposit(AMT, user.address))
    .to.emit(vault, "Locked").withArgs(user.address, AMT, user.address, 0n);
  expect(await token.balanceOf(await vault.getAddress())).to.equal(AMT);
  await wrapped.connect(relayer).mint(user.address, AMT, 0);
  expect(await wrapped.balanceOf(user.address)).to.equal(AMT);
});
Negative tests (the high-value ones)
it("rejects replayed nonce", async () => {
  await wrapped.connect(relayer).mint(user.address, AMT, 0);
  await expect(wrapped.connect(relayer).mint(user.address, AMT, 0))
    .to.be.revertedWith("replayed");
});
it("rejects non-relayer mint", async () => {
  await expect(wrapped.connect(user).mint(user.address, AMT, 0))
    .to.be.revertedWith("not relayer");
});
it("reverts deposit without approval", async () => {
  await expect(vault.connect(user).deposit(AMT, user.address))
    .to.be.reverted;
});
Replay reverts, non-relayer mint reverts, deposit-without-approve reverts. Bugs hide in the failure paths, not the happy path.

Quick tips & gotchas

  • Follow checks-effects-interactions in mint(): set processed[nonce]=true BEFORE _mint to kill reentrancy, even though _mint itself is safe.
  • Wait for confirmations/finality before minting — minting on a soon-to-be-reorged block creates unbacked tokens on the destination.
  • Nonces enforce exactly-once minting over an at-least-once relayer. Assume the relayer will crash and re-send; the processed mapping is what saves you.
  • Never commit private keys. Use .env + dotenv, add .env to .gitignore, and treat the relayer/owner key as the single biggest risk (Ronin, Wormhole both key/validation failures).
  • Use SafeERC20 — tokens like USDT don't return a bool from transfer/transferFrom and will silently break a naive vault.
  • approve + transferFrom, not a raw transfer to the vault: only the pull pattern lets you atomically record sender/dest/nonce and emit the event the relayer consumes.
  • In prod, monitor the invariant minted(dest) == locked(source) and add a pausable circuit breaker — pause minting the instant it diverges.
  • Extending to two-way? The unlock/withdraw path releases REAL collateral — give it its own nonce space and its own access control; most real exploits target the withdraw leg.
