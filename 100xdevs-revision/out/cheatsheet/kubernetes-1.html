Kubernetes Part 1 — Cheat Sheet

Cohort 2.0 · 20 lessons · quick reference

Why k8s (before vs after)

Before k8s: manual ops
SSH into VMs, run containers by hand, manual scaling, manual restart on crash, load balancer configured by hand. No self-healing, drift between machines.
After k8s: declarative + self-healing
You declare desired state (YAML); the control loop continuously reconciles actual→desired. Auto-restart, auto-reschedule on node death, rolling updates, service discovery, horizontal scaling.
Cluster = control plane + worker nodes
Control plane: api-server (front door), etcd (state store), scheduler (picks node), controller-manager (runs reconcile loops). Workers run kubelet + container runtime + kube-proxy.
Smallest deployable unit = Pod, not container
A Pod wraps 1+ containers sharing network (same IP/localhost) and volumes. You rarely create Pods directly — a Deployment manages them.

Create a cluster (local)

kind (k8s in Docker) — fast local
kind create cluster --name dev
kind get clusters
kind delete cluster --name dev
minikube — single-node VM/driver
minikube start --nodes 1
minikube status
minikube stop && minikube delete
k3d (lightweight k3s in Docker)
k3d cluster create dev --agents 2
Check the cluster is up
kubectl cluster-info
kubectl get nodes -o wide
kubeconfig / contexts
kubectl config get-contexts
kubectl config use-context kind-dev
kubectl config current-context
Default kubeconfig: ~/.kube/config (Windows: %USERPROFILE%\.kube\config). KUBECONFIG env var can point elsewhere.

Kubernetes API & kubectl basics

kubectl is a REST client to the API server
Everything is an API object. kubectl get pods == GET /api/v1/namespaces/default/pods. See raw calls with -v=8.
Discover API resources / versions
kubectl api-resources          # kinds, shortnames, apiGroup
kubectl api-versions           # served group/versions
kubectl explain deployment.spec  # field docs
apiVersion cheat: core vs apps
Pod/Service/ConfigMap → apiVersion: v1 (core group). Deployment/ReplicaSet → apps/v1.
Read-only inspection
kubectl get pods -A -o wide
kubectl describe pod <name>
kubectl logs <pod> -f --tail=100
kubectl logs <pod> -c <container>   # multi-container
Exec / debug into a pod
kubectl exec -it <pod> -- sh
kubectl get events --sort-by=.lastTimestamp
Namespaces
kubectl get ns
kubectl get pods -n kube-system
kubectl config set-context --current --namespace=dev
Objects are namespaced by default; nodes and PVs are cluster-scoped.

Pods: create, inspect, stop

Run a pod imperatively (quick test)
kubectl run nginx --image=nginx:1.27 --port=80
kubectl get pod nginx -w
kubectl run creates a bare Pod (not managed) — deleting it does NOT recreate it.
Pod manifest (declarative)
apiVersion: v1
kind: Pod
metadata:
  name: nginx
  labels:
    app: nginx
spec:
  containers:
  - name: nginx
    image: nginx:1.27
    ports:
    - containerPort: 80
Apply / delete a manifest
kubectl apply -f pod.yaml
kubectl delete -f pod.yaml
Stop / delete the pod
kubectl delete pod nginx
kubectl delete pod nginx --grace-period=0 --force  # immediate
A bare Pod stays deleted. A Pod owned by a ReplicaSet/Deployment gets recreated — you must delete the owner instead.
Reach the pod locally (no service yet)
kubectl port-forward pod/nginx 8080:80
# then curl localhost:8080
port-forward is for debugging only, not for real traffic.

ReplicaSet & Deployment

ReplicaSet: keeps N identical pods alive
apiVersion: apps/v1
kind: ReplicaSet
metadata:
  name: nginx-rs
spec:
  replicas: 3
  selector:
    matchLabels:
      app: nginx
  template:
    metadata:
      labels:
        app: nginx
    spec:
      containers:
      - name: nginx
        image: nginx:1.27
selector.matchLabels MUST match template.metadata.labels or apply is rejected. Rarely authored directly — Deployments own ReplicaSets.
Deployment: manages ReplicaSets + rollouts
apiVersion: apps/v1
kind: Deployment
metadata:
  name: nginx-deploy
spec:
  replicas: 3
  selector:
    matchLabels:
      app: nginx
  template:
    metadata:
      labels:
        app: nginx
    spec:
      containers:
      - name: nginx
        image: nginx:1.27
Create a deployment imperatively
kubectl create deployment nginx --image=nginx:1.27 --replicas=3
kubectl get deploy,rs,pods
Scale
kubectl scale deployment nginx-deploy --replicas=5
Rolling update (change image) + rollback
kubectl set image deploy/nginx-deploy nginx=nginx:1.28
kubectl rollout status deploy/nginx-deploy
kubectl rollout history deploy/nginx-deploy
kubectl rollout undo deploy/nginx-deploy
Deployment spins up a new ReplicaSet and shifts pods over gradually — zero-downtime. ReplicaSet alone cannot do rolling updates or rollbacks.
Why Deployment over ReplicaSet?
Deployment adds versioned rollouts, rollback, and controlled surge/unavailable during updates. Use Deployment for stateless apps; use ReplicaSet almost never on its own.

Services: expose the app

ClusterIP (default) — internal only
apiVersion: v1
kind: Service
metadata:
  name: nginx-svc
spec:
  selector:
    app: nginx
  ports:
  - port: 80
    targetPort: 80
Stable virtual IP + DNS (nginx-svc.<ns>.svc.cluster.local) load-balancing across pods matching the selector. Not reachable from outside.
NodePort — expose on every node's port
spec:
  type: NodePort
  selector:
    app: nginx
  ports:
  - port: 80
    targetPort: 80
    nodePort: 30080   # 30000-32767
Reach via <nodeIP>:30080. Good for local/dev; ugly for prod.
LoadBalancer — external cloud LB
apiVersion: v1
kind: Service
metadata:
  name: nginx-svc
spec:
  type: LoadBalancer
  selector:
    app: nginx
  ports:
  - port: 80
    targetPort: 80
Cloud provider provisions a real external LB and fills EXTERNAL-IP. On bare kind/minikube it stays <pending> unless you run metallb / `minikube tunnel`.
Expose imperatively
kubectl expose deployment nginx-deploy \
  --type=LoadBalancer --port=80 --target-port=80
kubectl get svc nginx-deploy -w
port vs targetPort vs nodePort
port = the Service's port; targetPort = container port it forwards to; nodePort = port opened on nodes (NodePort/LB only).

Series of events (reconcile loop)

apply Deployment → running pods
kubectl → api-server validates & writes object to etcd → Deployment controller creates a ReplicaSet → ReplicaSet controller creates Pod objects → scheduler assigns each Pod to a node → kubelet on that node pulls image & starts container → kubelet reports status back to api-server.
Self-healing loop
If a pod/node dies, actual state < desired (replicas). The ReplicaSet controller notices and creates replacements — no human action needed.
Traffic path with a Service
Client → Service IP → kube-proxy (iptables/IPVS rules) → one of the pods matching the label selector. Endpoints object is auto-updated as pods come and go.

Quick tips & gotchas

  • Label selectors are the glue: a Service or ReplicaSet finds pods ONLY by matchLabels. A typo in labels = 0 endpoints and silent 'connection refused'.
  • Deleting a pod owned by a Deployment/ReplicaSet just triggers a recreate. To actually stop the app, delete or scale the owning Deployment (kubectl scale --replicas=0 / kubectl delete deploy).
  • selector.matchLabels is immutable on a Deployment after creation — pick labels carefully; changing them forces a delete+recreate.
  • kubectl apply (declarative, merges) vs kubectl create (imperative, fails if exists). Use apply -f for GitOps-style repeatable manifests.
  • On kind/minikube, LoadBalancer EXTERNAL-IP stays <pending> forever without metallb; use `minikube tunnel` or NodePort/port-forward for local access.
  • Always pin image tags (nginx:1.27), never :latest — :latest breaks rollout history/rollback and makes deploys non-reproducible.
  • kubectl describe <obj> and kubectl get events --sort-by=.lastTimestamp are your first stop for 'why is my pod not running' (ImagePullBackOff, CrashLoopBackOff, FailedScheduling).
  • Set --namespace once (kubectl config set-context --current --namespace=dev) to stop forgetting -n and 'losing' your resources in the wrong namespace.
