Ownership And Authorities and PDAs — Cheat Sheet

Cohort 3.0 · 6 lessons · quick reference

Account Model & Fields

On-chain account layout (every account has these 5 fields)
pub struct Account {
    pub lamports: u64,       // SOL balance
    pub data: Vec<u8>,       // raw state bytes
    pub owner: Pubkey,       // program allowed to mutate this
    pub executable: bool,    // true = program account
    pub rent_epoch: Epoch,   // legacy rent field
}
Everything on Solana is an account — wallets, programs, program state, token mints.
AccountInfo (what your program receives at runtime)
pub struct AccountInfo<'a> {
    pub key: &'a Pubkey,
    pub lamports: Rc<RefCell<&'a mut u64>>,
    pub data:     Rc<RefCell<&'a mut [u8]>>,
    pub owner: &'a Pubkey,
    pub is_signer: bool,
    pub is_writable: bool,
    pub executable: bool,
}
is_signer / is_writable come from the transaction, not from the account state.
The ownership rule (core invariant)
Only the OWNER program can mutate data or DEBIT lamports. ANYONE can CREDIT lamports (send SOL). Non-owners get read-only view.
Inspect accounts / rent from CLI
solana account <PUBKEY>        # dump owner, lamports, data
solana balance <PUBKEY>
solana rent 200                # rent-exempt minimum for 200 bytes

Context — instruction entrypoint

Native entrypoint: program_id + accounts + data
entrypoint!(process_instruction);
fn process_instruction(
    program_id: &Pubkey,
    accounts: &[AccountInfo],
    instruction_data: &[u8],
) -> ProgramResult {
    let acc_iter = &mut accounts.iter();
    let payer = next_account_info(acc_iter)?;
    ...
}
Accounts arrive in a fixed order defined by the client — you validate each one manually.
Anchor Context<T> — typed, validated accounts
pub struct Context<'a,'b,'c,'info, T> {
    pub program_id: &'a Pubkey,
    pub accounts: &'b mut T,          // your #[derive(Accounts)] struct
    pub remaining_accounts: &'c [AccountInfo<'info>],
    pub bumps: T::Bumps,              // generated PDA bumps struct
}
Access as ctx.accounts.vault, ctx.bumps.vault, ctx.remaining_accounts.
Accounts struct = declarative validation
#[derive(Accounts)]
pub struct Deposit<'info> {
    #[account(mut)]
    pub user: Signer<'info>,
    #[account(mut)]
    pub vault: Account<'info, Vault>,
    pub system_program: Program<'info, System>,
}

System Program — create / fund / assign

ID + role
11111111111111111111111111111111
Owns all new/empty accounts & wallets. Only it can allocate space, assign owners, and create accounts.
create_account: alloc space + fund rent + set owner (one op)
let rent = Rent::get()?;
let lamports = rent.minimum_balance(space);
let ix = system_instruction::create_account(
    payer.key, new_acc.key, lamports, space as u64,
    program_id,           // new owner = YOUR program
);
invoke(&ix, &[payer.clone(), new_acc.clone(), system_program.clone()])?;
new_acc must sign (it's a keypair) unless created as a PDA via invoke_signed.
transfer SOL
invoke(
  &system_instruction::transfer(from.key, to.key, lamports),
  &[from.clone(), to.clone(), system_program.clone()],
)?;
allocate + assign (two-step ownership handoff)
system_instruction::allocate(acc.key, space as u64);
system_instruction::assign(acc.key, &new_owner);
Alternative to create_account when the account already holds lamports.

BPF Loader & Deployment

Loader IDs
BPFLoaderUpgradeab1e11111111111111111111111  # upgradeable (default)
BPFLoader2111111111111111111111111111111111  # v2, immutable
A deployed program's OWNER is the loader. executable=true. Loader marks the account executable after deploy.
Program vs ProgramData account (upgradeable)
Program account is a thin pointer -> ProgramData account, which stores the actual bytecode + the upgrade_authority. Upgrading rewrites ProgramData.
Deploy / upgrade
solana program deploy target/deploy/prog.so
solana program deploy --program-id <ID> prog.so   # upgrade in place
solana program show <PROGRAM_ID>                   # authority, size, slot
Manage upgrade authority
solana program set-upgrade-authority <ID> \
    --new-upgrade-authority <PUBKEY>
solana program set-upgrade-authority <ID> --final  # immutable forever
--final is IRREVERSIBLE: no more upgrades, no reclaim. Only current authority can upgrade/close.
Reclaim rent / grow program
solana program close <ID> --recipient <PUBKEY>
solana program extend <ID> <ADDITIONAL_BYTES>

Authority & Ownership Checks

Signer check (native) — is this action authorized?
if !authority.is_signer {
    return Err(ProgramError::MissingRequiredSignature);
}
Owner check (native) — is this account really mine?
if data_acc.owner != program_id {
    return Err(ProgramError::IllegalOwner);
}
ALWAYS verify owner before trusting/deserializing account data — else attacker feeds fake accounts.
Anchor: types enforce checks automatically
pub user: Signer<'info>,               // must sign
pub cfg:  Account<'info, Config>,      // owner==program_id + discriminator
pub token_prog: Program<'info, Token>, // owner==Token program
Account<T> checks owner + 8-byte discriminator. Signer checks is_signer.
has_one — field-level authority match
#[account(mut, has_one = authority)]
pub config: Account<'info, Config>,
pub authority: Signer<'info>,
Asserts config.authority == authority.key(). Classic admin-gate pattern.
SPL Token authorities
mint_authority (who can mint), freeze_authority (who can freeze accounts), account owner (who can transfer). Set to None to permanently disable minting.

PDAs — Derivation

What a PDA is
Deterministic address OFF the ed25519 curve => no private key exists. Only the deriving program can 'sign' for it. Used for program-owned accounts (vaults, state, escrows).
find_program_address — get PDA + canonical bump
let (pda, bump) = Pubkey::find_program_address(
    &[b"vault", user.key.as_ref()],
    program_id,
);
Loops bump 255->0 until an off-curve address is found. Use off-chain or at init; it's compute-heavy.
create_program_address — derive with a known bump (cheap)
let pda = Pubkey::create_program_address(
    &[b"vault", user.key.as_ref(), &[bump]],
    program_id,
)?;
Store the bump on-chain and reuse it to skip the find loop and save CU.
Anchor: derive + init a PDA account
#[account(
    init, payer = user, space = 8 + 40,
    seeds = [b"vault", user.key().as_ref()],
    bump
)]
pub vault: Account<'info, Vault>,
On later calls use bump = vault.bump to re-check the same PDA. Access as ctx.bumps.vault.

PDAs — Signing via CPI

invoke_signed — program signs for its PDA (native)
let seeds: &[&[u8]] = &[b"vault", user.key.as_ref(), &[bump]];
invoke_signed(
    &system_instruction::transfer(vault.key, user.key, amount),
    &[vault.clone(), user.clone(), system_program.clone()],
    &[seeds],   // signer seeds MUST match the PDA derivation
)?;
Runtime re-derives the PDA from seeds; if it matches an account, that account is treated as a signer.
Anchor CPI with signer seeds
let bump = ctx.bumps.vault;
let seeds: &[&[u8]] = &[b"vault", user.key().as_ref(), &[bump]];
let signer = &[seeds];
token::transfer(
    CpiContext::new_with_signer(
        ctx.accounts.token_program.to_account_info(),
        cpi_accounts,
        signer,
    ),
    amount,
)?;
Seed constraints
Max 16 seeds, each <= 32 bytes. The bump is appended as the final seed when signing. Use the canonical (highest) bump only.

Quick tips & gotchas

  • Ownership asymmetry: only the owner program can debit lamports & mutate data, but ANYONE can send (credit) SOL to any account.
  • New accounts are owned by the System Program by default — call create_account (or allocate + assign) to transfer ownership to your program before writing state.
  • Always check BOTH is_signer and owner in native code. Anchor's Signer / Account<T> types do this for you; skipping owner checks is the #1 exploit vector.
  • PDAs have no private key (off-curve), so they can only be 'signed' by the owning program via invoke_signed with the exact derivation seeds + bump.
  • Use find_program_address off-chain / at init, then store the bump and use create_program_address (or Anchor's bump = state.bump) on-chain to save compute — find loops up to 256 times.
  • Anchor init space must include the 8-byte discriminator: space = 8 + <struct size>. Forgetting the 8 is a classic deserialization bug.
  • solana program set-upgrade-authority --final is irreversible: the program can never be upgraded or closed again.
  • Seeds: max 16, each <= 32 bytes, and the bump byte is appended last when signing — signer seeds must exactly match how the PDA was derived or the sign fails.
