Horizontal and vertical scaling, Indexes in DBs — Cheat Sheet

Cohort 2.0 · 11 lessons · quick reference

Scaling: Vertical vs Horizontal

Vertical (scale up) — add CPU/RAM/disk to one box
Simplest: no code change, no distributed complexity. Hard ceiling (biggest instance), single point of failure, downtime to resize, cost grows non-linearly.
Horizontal (scale out) — add more machines/instances behind a LB
Near-infinite headroom + HA/redundancy. Needs stateless app, load balancer, shared state store. Requires the app to NOT keep session/data in local memory.
Rule of thumb
Scale vertically first (cheap, easy) until you hit a single-node limit or need HA, then go horizontal. DB usually scales up + read-replicas before sharding.
Statelessness is the prerequisite for horizontal
// BAD: in-memory session -> breaks across instances
const sessions = {};
// GOOD: externalize state to Redis/DB
await redis.set(`sess:${id}`, JSON.stringify(data));
Move sessions, cache, rate-limit counters, uploads to Redis/DB/S3. Sticky sessions are a crutch, not a fix.
Load balancing strategies
round-robin (default), least-conn (uneven request cost), ip-hash/sticky (stateful fallback). Health checks evict dead nodes.

Horizontal Scaling in Node.js

cluster module — one worker per CPU core (Node is single-threaded)
const cluster = require('node:cluster');
const os = require('node:os');
if (cluster.isPrimary) {
  for (let i = 0; i < os.availableParallelism(); i++) cluster.fork();
  cluster.on('exit', () => cluster.fork()); // auto-restart
} else {
  require('./server'); // each worker binds same port (OS load-balances)
}
Uses all cores on ONE box (a form of scale-out within a machine). Workers share the listening socket; no shared memory.
PM2 cluster mode — production process manager
pm2 start app.js -i max      # one instance per core
pm2 scale app +3             # add 3 workers
pm2 reload app               # zero-downtime rolling restart
pm2 start app.js -i 0        # 0/max = all cores
`reload` (graceful) vs `restart` (hard). `pm2 startup` + `pm2 save` to survive reboots.
Nginx as reverse-proxy / load balancer across boxes
upstream app {
  least_conn;
  server 10.0.0.1:3000;
  server 10.0.0.2:3000;
}
server {
  location / { proxy_pass http://app; }
}
In front of N machines each running the Node app. Add `proxy_set_header X-Forwarded-For $remote_addr;` and trust proxy in Express.
Trust the proxy in Express (correct client IP / protocol)
app.set('trust proxy', 1);
Graceful shutdown (drain in-flight before exit)
process.on('SIGTERM', () => server.close(() => process.exit(0)));
Required so LB/PM2 can rotate instances without dropping requests.

Capacity Estimation (back-of-envelope)

Average RPS from daily traffic
RPS = daily_requests / 86400   # 86,400 s/day
# 100M req/day  ->  ~1,157 RPS avg
Peak ≈ 2–3× average. Size for peak, not average.
Little's Law — concurrency needed
concurrency = RPS * avg_latency_seconds
# 1000 RPS * 0.2s  ->  200 concurrent in flight
Drives worker count, thread/connection pool size, and instance count.
Instances needed
instances = ceil(peak_RPS / per_instance_RPS)
# add headroom (~30%) + N+1 for failover
Storage sizing
storage = rows * bytes_per_row * (1 + index_overhead)
# 1B rows * 200B * 1.5  ->  ~300 GB
Account for indexes (often 20–100% of table size), WAL, and growth rate.
DB connection budget
Postgres default max_connections ~100. Each connection ~5–10MB. Use PgBouncer/pooling; pool size ≈ cores*2, NOT one per app worker × instances.
Handy units
1 day ≈ 86.4k s; 1M/day ≈ 12 RPS. Read:write ratios drive caching + replica decisions.

Postgres Indexing — Basics

Create / drop a B-tree index (default)
CREATE INDEX idx_users_email ON users(email);
DROP INDEX idx_users_email;
B-tree serves =, <, >, BETWEEN, ORDER BY, IN, LIKE 'prefix%'. Not LIKE '%x'.
Build without locking writes (production)
CREATE INDEX CONCURRENTLY idx_users_email ON users(email);
Slower, can't run in a txn; a failed run leaves an INVALID index (DROP + retry). Plain CREATE INDEX takes a write lock on the table.
Unique index / constraint
CREATE UNIQUE INDEX ON users(email);
ALTER TABLE users ADD CONSTRAINT uq_email UNIQUE(email);
PKs and UNIQUE constraints auto-create a backing index. FKs do NOT — index FK columns yourself.
Inspect + verify usage
EXPLAIN ANALYZE SELECT * FROM users WHERE email='a@b.com';
\d users        -- list indexes (psql)
Look for 'Index Scan' vs 'Seq Scan'. Check pg_stat_user_indexes for unused indexes to drop.
Index type cheat sheet
B-tree  -- default, equality + range + sort
Hash    -- equality only (=)
GIN     -- jsonb, arrays, full-text (@>, ?, @@)
GiST    -- geometric, ranges, nearest-neighbor
BRIN    -- huge append-only tables (time-series)
GIN for containment/search; BRIN is tiny + fast on naturally-ordered columns like created_at.
Cost of indexes
Each index slows INSERT/UPDATE/DELETE and consumes disk. Index for read patterns you actually run; don't index everything.

Complex / Advanced Postgres Indexes

Composite (multi-column) — leftmost-prefix rule
CREATE INDEX idx ON orders(user_id, created_at);
Serves WHERE user_id=? , WHERE user_id=? AND created_at>? , and (user_id) alone. Does NOT serve created_at alone. Put the equality/most-selective column first.
Partial index — index a subset of rows
CREATE INDEX idx_active ON orders(user_id)
  WHERE status = 'active';
Smaller + faster. Query's WHERE must match the predicate to use it. Great for soft-deletes / hot subsets.
Expression (functional) index
CREATE INDEX idx_lower_email ON users(lower(email));
-- query MUST use the same expression:
SELECT * FROM users WHERE lower(email)=lower($1);
Enables case-insensitive / computed lookups. Column-only query won't hit it.
Covering index (INCLUDE) — index-only scan
CREATE INDEX idx ON orders(user_id) INCLUDE (total, status);
Payload columns live in the index so the query avoids heap fetches. Needs recent VACUUM (visibility map) for index-only scan.
GIN on JSONB / arrays
CREATE INDEX ON docs USING GIN (data jsonb_path_ops);
SELECT * FROM docs WHERE data @> '{"active":true}';
Full-text search
CREATE INDEX ON posts USING GIN (to_tsvector('english', body));
SELECT * FROM posts
 WHERE to_tsvector('english', body) @@ plainto_tsquery('scaling');

Indexes & Relations in Prisma

Single-field id / unique / index
model User {
  id    Int    @id @default(autoincrement())
  email String @unique          // unique index
  name  String
  @@index([name])               // non-unique index
}
@unique = unique index; @@index = plain index; @id = PK (indexed).
Composite index / composite unique
model Post {
  authorId Int
  slug     String
  @@index([authorId, createdAt])
  @@unique([authorId, slug])
}
@@unique also gives you a compound lookup key for `where: { authorId_slug: {...} }`.
Apply schema changes
npx prisma migrate dev --name add_indexes   # dev
npx prisma migrate deploy                    # prod
npx prisma generate                          # regen client
Prisma emits CREATE INDEX in the migration SQL; edit the SQL for CONCURRENTLY on big tables.
Index-type / DB-specific
@@index([data], type: Gin)          // Postgres GIN
@@fulltext([title, body])           // MySQL fulltext
1:1 relation
model User    { id Int @id  profile Profile? }
model Profile { id Int @id  userId Int @unique
  user User @relation(fields:[userId], references:[id]) }
FK column carries @unique to enforce one-to-one.
1:N relation
model User { id Int @id  posts Post[] }
model Post { id Int @id  authorId Int
  author User @relation(fields:[authorId], references:[id]) }
FK lives on the 'many' side. Prisma does NOT auto-index FKs on Postgres — add @@index([authorId]).
M:N relation (implicit vs explicit)
// implicit (Prisma manages join table)
model Post { id Int @id  tags Tag[] }
model Tag  { id Int @id  posts Post[] }

// explicit (when join table needs extra columns)
model PostTag {
  postId Int; tagId Int; addedAt DateTime @default(now())
  post Post @relation(fields:[postId],references:[id])
  tag  Tag  @relation(fields:[tagId], references:[id])
  @@id([postId, tagId])
}

Normalization & Relationship Modeling

1NF — atomic values, no repeating groups
-- BAD:  phones = '555-1, 555-2'
-- GOOD: separate phones table, one row per phone
Each cell holds a single value; each row is unique (has a key).
2NF — 1NF + no partial dependency on a composite PK
Every non-key column depends on the WHOLE composite key, not part of it. Split out columns that depend on only one key component.
3NF — 2NF + no transitive dependency
-- BAD: orders(id, userId, userEmail)  -- email depends on userId, not order
-- GOOD: orders(id, userId) + users(id, email)
Non-key columns depend only on the key. Everyday target for OLTP schemas.
Relationship → schema mapping
1:1  -> FK + UNIQUE on the FK
1:N  -> FK on the 'many' side
M:N  -> junction/join table (PK = both FKs)
Always index foreign keys
CREATE INDEX ON orders(user_id);
Speeds joins and cascades; without it, deleting a parent does a seq scan of children.
Normalize vs denormalize trade-off
Normalized = no update anomalies, less storage, more joins. Denormalize (duplicate/cache columns) only for proven read-hot paths; keep copies in sync deliberately.

Quick tips & gotchas

  • Scale vertically first (zero code change); go horizontal when you hit a single-node ceiling or need HA. The blocker is almost always in-memory state, not CPU.
  • Horizontal scaling requires stateless app servers: push sessions, cache, rate-limit counters, and uploads to Redis/DB/S3 before adding instances.
  • node cluster / PM2 -i max scales across cores on ONE machine; a load balancer (Nginx/ELB) scales across machines. You usually want both layers.
  • Composite index obeys the leftmost-prefix rule: (a,b) helps WHERE a and WHERE a AND b, never WHERE b alone. Order columns equality-first, then range/sort.
  • Always CREATE INDEX CONCURRENTLY on production tables — the plain form takes a write lock. On failure it leaves an INVALID index you must DROP and rebuild.
  • Verify every index with EXPLAIN ANALYZE — an unused index is pure write/disk overhead. Drop indexes with zero scans in pg_stat_user_indexes.
  • FKs are NOT auto-indexed in Postgres (or Prisma on Postgres). Add the index yourself or joins, cascades, and parent deletes will seq-scan.
  • Expression and partial indexes only fire when the query's WHERE matches exactly — lower(email) index needs WHERE lower(email)=…, and a partial index needs its predicate.
