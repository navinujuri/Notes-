Redis (Pub subs and Messaging queues) — Cheat Sheet

Cohort 2.0 · 9 lessons · quick reference

Setup & Starting Redis Locally

Run via Docker (fastest, recommended)
docker run -d --name redis -p 6379:6379 redis
# with persistence volume
docker run -d --name redis -p 6379:6379 -v redis-data:/data redis redis-server --appendonly yes
Default port 6379. -d = detached. Data lost on container rm unless volume mounted.
Install natively
# macOS
brew install redis && brew services start redis
# Ubuntu/Debian
sudo apt install redis-server && sudo systemctl start redis
# Windows: use Docker or WSL2 (no official native build)
Connect with the CLI
redis-cli                 # localhost:6379
redis-cli -h host -p 6379 -a <password>
redis-cli ping            # -> PONG
redis-cli -u redis://default:pass@host:6379
Managed / hosted
# Redis Cloud, AWS ElastiCache, Upstash
# connect via TLS URL:
redis-cli -u rediss://default:<pass>@<host>:6379
rediss:// (double s) = TLS. Managed services are the norm in production.
Quick health/inspect
redis-cli info
redis-cli monitor          # live stream of every command (debug only)
redis-cli keys '*'         # NEVER in prod - blocks; use SCAN

Redis as a DB — Core Commands

Strings (get/set + expiry)
SET user:1 "chan"
GET user:1
SET session:abc "data" EX 3600     # expire in 3600s
SETEX session:abc 3600 "data"       # same thing
TTL session:abc                     # seconds left, -1 none, -2 gone
INCR counter                        # atomic ++
INCRBY counter 10
SET ... NX = set only if not exists (great for locks). GET on missing key = nil.
Hashes (objects)
HSET user:1 name chan age 30
HGET user:1 name
HGETALL user:1
HINCRBY user:1 age 1
Store an object as one key instead of many string keys.
Sets & Sorted Sets
SADD tags a b c            # unique members
SMEMBERS tags
SISMEMBER tags a          # 1/0
ZADD leaderboard 100 chan 90 alice   # score member
ZRANGE leaderboard 0 -1 WITHSCORES REV  # top-first
Sorted sets = leaderboards, rate limiters, delayed queues (score = timestamp).
Keys, expiry & delete
EXPIRE key 60
PERSIST key                # remove TTL
DEL key
EXISTS key
SCAN 0 MATCH user:* COUNT 100   # cursor-based, non-blocking
When to use Redis as DB
In-memory = blazing fast but volatile. Ideal for caching, sessions, counters, rate-limiting, leaderboards. Not a durable primary store unless AOF/RDB persistence tuned — treat as cache-first.

Redis as a Queue (Lists)

Producer pushes, worker pops (FIFO)
# producer
LPUSH jobs '{"id":1,"task":"email"}'
# consumer (FIFO): push left, pop right
RPOP jobs
LLEN jobs                  # queue depth
LPUSH + RPOP = FIFO. LPUSH + LPOP = LIFO (stack). Pick one convention and stick to it.
Blocking pop (efficient worker loop)
BRPOP jobs 0               # block forever until an item
BRPOP jobs 5               # block up to 5s, then nil
# returns [listName, value]
Use BRPOP, NOT a RPOP polling loop — blocking avoids hammering the server. timeout 0 = wait indefinitely.
Reliable queue (survive worker crash)
BLMOVE jobs processing LEFT RIGHT 0
# ... do work ...
LREM processing 1 <value>   # ack: remove from processing
Plain BRPOP loses the message if the worker dies mid-job. BLMOVE atomically moves to a 'processing' list so you can recover/retry. (RPOPLPUSH/BRPOPLPUSH are the deprecated equivalents.)
Pub/Sub vs Queue — the key difference
Queue (list): message consumed by exactly ONE worker, persists until popped, work is load-balanced. Pub/Sub: fire-and-forget broadcast to ALL current subscribers, nothing stored — offline subscribers miss it.
Streams (durable queue upgrade)
XADD mystream '*' task email
XREAD BLOCK 0 STREAMS mystream $
XGROUP CREATE mystream grp1 0
XREADGROUP GROUP grp1 c1 COUNT 1 STREAMS mystream >
Redis Streams = persistent log + consumer groups + acks (XACK). Use over lists when you need replay, multiple consumer groups, or delivery guarantees.

Node.js — Connecting & Data Ops

node-redis (official) setup
// npm i redis
import { createClient } from 'redis';
const client = createClient({ url: 'redis://localhost:6379' });
client.on('error', e => console.error('Redis', e));
await client.connect();
v4+ is promise-based and requires explicit await client.connect(). Commands are camelCase (client.hSet).
Basic commands (async)
await client.set('user:1', 'chan', { EX: 3600 });
const v = await client.get('user:1');
await client.hSet('user:1', { name: 'chan', age: 30 });
await client.incr('counter');
await client.lPush('jobs', JSON.stringify(job));
Redis stores strings — JSON.stringify/parse objects yourself.
ioredis (popular alt) setup
// npm i ioredis
import Redis from 'ioredis';
const redis = new Redis('redis://localhost:6379');
await redis.set('k', 'v');
await redis.lpush('jobs', data);
ioredis auto-connects (no .connect()), uses lowercase method names, has built-in cluster/sentinel support.
Worker loop (blocking pop)
while (true) {
  const res = await client.brPop('jobs', 0); // {key, element}
  const job = JSON.parse(res.element);
  await handle(job);
}
Run the blocking loop on a DEDICATED client — the connection is busy while blocked and can't issue other commands.

Pub/Sub — CLI Commands

Subscribe & publish
# terminal 1
SUBSCRIBE news
# terminal 2
PUBLISH news "hello"        # -> returns # of subscribers that got it
PUBLISH returns the count of receivers. If 0 subscribers, the message vanishes — no storage, no replay.
Pattern subscribe
PSUBSCRIBE news.*          # matches news.sports, news.tech
PUBLISH news.sports "goal!"
PUNSUBSCRIBE news.*
Glob-style: * (any chars), ? (one char), [ae] (set).
Inspect channels
PUBSUB CHANNELS           # active channels with subscribers
PUBSUB NUMSUB news        # subscriber count per channel
PUBSUB NUMPAT             # active pattern subs

Pub/Sub — Node.js

Subscriber (needs its own connection)
const sub = client.duplicate();
await sub.connect();
await sub.subscribe('news', (message, channel) => {
  console.log(channel, message);
});
CRITICAL: a subscribed client cannot run normal commands. Use client.duplicate() so your main client stays free for get/set.
Publisher
await client.publish('news', JSON.stringify({ title: 'hi' }));
Publish from the main client; subscribe on the duplicated one.
Pattern subscribe (node-redis v4)
await sub.pSubscribe('news.*', (msg, channel) => {
  console.log(channel, msg);
});
await sub.unsubscribe('news');
ioredis pub/sub (event-based)
const sub = new Redis();
sub.subscribe('news');
sub.on('message', (channel, message) => {
  console.log(channel, message);
});
redis.publish('news', 'hi');
ioredis emits a 'message' event rather than a per-channel callback.

Assignment Patterns & Mental Model

Typical exercise: producer + worker via list queue
// producer.js
await client.lPush('submissions', JSON.stringify(payload));
// worker.js (long-running)
const { element } = await client.brPop('submissions', 0);
processAndPushResult(JSON.parse(element));
Classic 'code submission' style: HTTP handler pushes job, separate worker process blocks on BRPOP, pushes result to a per-user list/channel.
Return result to a specific client
// worker publishes to a user-scoped channel
await client.publish(`result:${userId}`, JSON.stringify(out));
// client subscribed to its own channel gets it
Combine queue (work distribution) + pub/sub (push result back) for request/response over Redis.
Decision cheat
One consumer, must-not-lose, load-balance -> List/Stream queue. Broadcast to many, real-time, OK to miss if offline -> Pub/Sub. Need replay + acks + groups -> Streams.

Quick tips & gotchas

  • Pub/Sub is fire-and-forget: no persistence, no delivery guarantee. Offline or slow subscribers miss messages entirely — use Streams or a list queue when you need durability.
  • A subscribed connection is 'locked' into subscriber mode and can't run GET/SET/etc. Always client.duplicate() (node-redis) or use a second client for subscribing.
  • Prefer blocking BRPOP/BLMOVE over polling RPOP in a loop — polling wastes CPU and network round-trips.
  • Plain BRPOP loses the job if the worker crashes mid-processing. Use BLMOVE into a 'processing' list (or Streams with XACK) for at-least-once delivery.
  • Never run KEYS * in production — it blocks the single-threaded server. Use SCAN with a cursor instead.
  • Redis is single-threaded: each command is atomic, but a slow command (KEYS, big SORT, large MULTI) stalls everyone. Keep operations small.
  • Redis is in-memory and volatile by default — enable AOF (appendonly yes) or RDB snapshots if you can't afford to lose data on restart.
  • Redis stores only strings/bytes. Always JSON.stringify on write and JSON.parse on read; set EX/TTL on cache keys so memory doesn't grow unbounded.
