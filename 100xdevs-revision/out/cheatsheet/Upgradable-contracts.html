Upgradable contracts — Cheat Sheet

Cohort 3.0 · 19 lessons · quick reference

Why upgrade + delegatecall core

The problem: deployed bytecode is immutable
You cannot patch a bug or add features once a contract is on-chain. Naive fixes are costly.
Naive approach 1: redeploy + migrate state
New address breaks every reference/integration; copying storage on-chain is gas-expensive and error-prone.
Naive approach 2: registry / data-separation
Logic contract CALLs a separate storage contract via a lookup registry. Works but clumsy, extra gas, no shared context.
Real solution: delegatecall — run OTHER code in THIS context
(bool ok, bytes memory ret) = impl.delegatecall(msg.data);
require(ok);
// storage, msg.sender, msg.value, address(this) all stay the PROXY's
call = callee's context/storage; delegatecall = caller's context/storage. This is the whole trick behind proxies.
delegatecall silent-fail gotcha
// delegatecall to an EOA / empty address returns success=true!
require(impl.code.length > 0, "no code");
Always check the target has code before delegatecalling — otherwise it 'succeeds' doing nothing.

Proxy = storage + fallback that delegates

Split into Proxy (holds state, permanent address) + Implementation (logic, swappable)
Users only ever interact with the proxy address. Upgrade = point proxy at new impl.
fallback() forwards every unknown call to the impl
fallback() external payable { _delegate(_implementation()); }
receive() external payable { _delegate(_implementation()); }
fallback catches any selector the proxy itself doesn't define, then delegates it.
Canonical assembly delegate (preserves calldata + return/revert data)
function _delegate(address impl) internal {
    assembly {
        calldatacopy(0, 0, calldatasize())
        let ok := delegatecall(gas(), impl, 0, calldatasize(), 0, 0)
        returndatacopy(0, 0, returndatasize())
        switch ok
        case 0 { revert(0, returndatasize()) }
        default { return(0, returndatasize()) }
    }
}
Must be raw assembly — Solidity can't return a dynamic value whose type isn't known at compile time.
Upgrade function (gate it!)
function upgradeTo(address newImpl) external onlyOwner {
    _setImplementation(newImpl);
}
Without onlyOwner ANYONE can repoint the proxy to malicious logic and drain it.

Storage layout — the #1 way to brick a proxy

Proxy & impl SHARE the same storage slots
delegatecall writes into the proxy's storage by slot index, using the IMPL's variable layout. They must agree.
Storage collision: naive `address public implementation` at slot 0
// PROXY:  slot0 = implementation
// IMPL:   slot0 = uint256 count   <-- COLLISION
// writing count overwrites the implementation address
This is 'Problem with current proxy' — proxy's own vars overlap the impl's vars.
Fix: unstructured storage — stash proxy vars at a random hashed slot
Pick a slot no normal variable will ever occupy. Standardized as ERC-1967 (below).
Upgrade rule: append-only
// V1: uint a; uint b;
// V2 OK:   uint a; uint b; uint c;   (append)
// V2 BAD:  uint b; uint a;          (reorder -> corrupt)
// V2 BAD:  bool a; uint b;          (retype  -> corrupt)
Never reorder, remove, or change the type of existing state vars between versions.
Reserve space in base contracts with a gap
uint256[50] private __gap;
Lets you add fields to an inherited base later without shifting child storage.

ERC-1967 standard storage slots

Implementation slot
// bytes32(uint256(keccak256("eip1967.proxy.implementation")) - 1)
bytes32 internal constant _IMPL_SLOT =
  0x360894a13ba1a3210667c828492db98dca3e2076cc3735a920a3ca505d382bbc;
Admin slot
// keccak256("eip1967.proxy.admin") - 1
0xb53127684a568b3173ae13b9f8a6016e243e63b6e8ee1178d6a717850b5d6103
Beacon slot
// keccak256("eip1967.proxy.beacon") - 1
0xa3f0ad74e5423aebfd80d3ef4346578335a9a72aeaee59ff6cb3582b35133d50
Read/write via assembly
function _getImplementation() internal view returns (address a) {
    assembly { a := sload(_IMPL_SLOT) }
}
function _setImplementation(address newImpl) internal {
    require(newImpl.code.length > 0);
    assembly { sstore(_IMPL_SLOT, newImpl) }
}
Block explorers & tooling detect proxies by reading these exact slots. -1 keeps the slot off any keccak-derived mapping.

The three common proxy patterns

Transparent Proxy (TPP)
// admin  -> calls handled by proxy (upgrade)
// non-admin -> everything delegates to impl
import "@openzeppelin/contracts/proxy/transparent/TransparentUpgradeableProxy.sol";
Upgrade logic in the PROXY. A separate ProxyAdmin owns it. Routing by msg.sender avoids selector clashes but adds an SLOAD to every call. Admin can't call impl functions.
UUPS (EIP-1822) — upgrade logic in the IMPLEMENTATION
import {UUPSUpgradeable} from "@openzeppelin/contracts-upgradeable/proxy/utils/UUPSUpgradeable.sol";
contract MyV1 is Initializable, OwnableUpgradeable, UUPSUpgradeable {
    function initialize(address o) public initializer {
        __Ownable_init(o);
        __UUPSUpgradeable_init();
    }
    function _authorizeUpgrade(address) internal override onlyOwner {}
}
Cheaper deploy + calls (proxy is minimal, no admin routing). DANGER: ship an impl without upgrade code and the proxy is permanently bricked.
Beacon Proxy — many proxies, one upgrade point
import "@openzeppelin/contracts/proxy/beacon/UpgradeableBeacon.sol";
import "@openzeppelin/contracts/proxy/beacon/BeaconProxy.sol";
// beacon holds impl address; each BeaconProxy reads it
new BeaconProxy(address(beacon), initData);
beacon.upgradeTo(newImpl); // upgrades ALL proxies at once
Best for factories deploying N identical instances you want to upgrade in one tx. Extra SLOAD to fetch beacon then impl.

Initializers — no constructors in logic contracts

Why: constructor runs in the IMPL's context, so its state never lands in the proxy
State set by a constructor lives in the implementation's own storage — the proxy never sees it.
Use an initializer instead (runs via delegatecall = proxy storage)
function initialize(address o) public initializer {
    __Ownable_init(o);
}
`initializer` modifier ensures it runs exactly once.
Lock the implementation's own initializers
/// @custom:oz-upgrades-unsafe-allow constructor
constructor() { _disableInitializers(); }
Prevents an attacker from initializing the bare impl contract directly and hijacking it.
Versioned re-init for new state added in an upgrade
function initializeV2(uint x) public reinitializer(2) { newVar = x; }
`reinitializer(n)` runs once per version; `onlyInitializing` for internal __init_ helpers of parents.

OpenZeppelin Upgrades tooling

Install
npm i @openzeppelin/contracts-upgradeable @openzeppelin/hardhat-upgrades
# foundry:
forge install OpenZeppelin/openzeppelin-contracts-upgradeable
Deploy proxy (Hardhat)
const { ethers, upgrades } = require("hardhat");
const Box = await ethers.getContractFactory("Box");
const box = await upgrades.deployProxy(Box, [42], { kind: "uups" });
await box.waitForDeployment();
Upgrade proxy
const BoxV2 = await ethers.getContractFactory("BoxV2");
await upgrades.upgradeProxy(await box.getAddress(), BoxV2);
// plugin auto-validates storage layout compatibility before upgrading
It aborts if you reordered/retyped storage or used unsafe delegatecall/selfdestruct.
Beacon deploy
const beacon = await upgrades.deployBeacon(Box);
const proxy  = await upgrades.deployBeaconProxy(beacon, Box, [42]);
await upgrades.upgradeBeacon(beacon, BoxV2);
Foundry: OZ Upgrades + validation
import {Upgrades} from "openzeppelin-foundry-upgrades/Upgrades.sol";
Upgrades.deployUUPSProxy("BoxV1.sol", abi.encodeCall(Box.initialize, (42)));
// build with: forge build --ast   (needed for layout checks)

Quick tips & gotchas

  • No constructors in logic contracts: use `initialize()` + `initializer`, and call `_disableInitializers()` in the impl's constructor so the bare implementation can't be seized.
  • Storage is append-only across versions — never reorder, remove, or retype existing vars. Add `uint256[50] private __gap;` to upgradeable base contracts to reserve room.
  • delegatecall to an address with no code returns success=true and does nothing. Always `require(impl.code.length > 0)` before setting/using an implementation.
  • Gate every upgrade path (`onlyOwner` / `_authorizeUpgrade`). An ungated upgrade lets anyone repoint the proxy to a drain contract.
  • UUPS bricking risk: if a new implementation omits UUPSUpgradeable / `_authorizeUpgrade`, the proxy loses its only upgrade mechanism forever. Transparent proxies keep upgrade logic in the proxy, so they can't self-brick this way.
  • Deploy + initialize atomically (single tx via proxy constructor data) — a two-step deploy lets an attacker front-run the initializer.
  • Explorers detect proxies via ERC-1967 slots; the `keccak(...) - 1` offset keeps those slots from ever colliding with a real mapping/array slot.
  • Transparent adds an SLOAD + admin check to every call; UUPS is cheapest at runtime; Beacon adds an extra hop but upgrades N proxies in one tx — pick per use case.
