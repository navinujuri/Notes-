Monorepo & stack setup

Workspace layout (pnpm + turbo)

  web/        # Next.js — problems, submit UI, contests, leaderboard
  setter/     # problem-authoring UI/CLI + boilerplate generator
  judge/      # worker: pulls queue, runs code in Docker

Split web / setter / judge so the judge (untrusted-code runner) scales & deploys independently.

Core stack
Next.js (React) FE + API, Node/Express or Next route handlers BE, Postgres+Prisma (durable state), Redis (submission queue), Docker (per-submission sandbox).

Database design (Prisma)

Core models
model Problem {
  id        String @id @default(cuid())
  slug      String @unique
  title     String
  statement String
  difficulty Int
  timeLimitMs Int   @default(1000)
  memoryKb    Int   @default(262144)
  hidden    Boolean @default(true)
  submissions Submission[]
  defaultCode DefaultCode[]  // per-language boilerplate
}

model Submission {
  id        String   @id @default(cuid())
  userId    String
  problemId String
  contestId String?
  code      String
  language  String
  status    SubmissionStatus @default(PENDING)
  runtimeMs Int?
  memoryKb  Int?
  failedTest Int?
  createdAt DateTime @default(now())
  @@index([userId, problemId])
  @@index([contestId])
}

enum SubmissionStatus { PENDING AC WA TLE MLE RE CE }
Contest / ContestParticipant / rating history are separate tables. Keep rows small.
Big test data → object store
Store test inputs/outputs in S3 (or a git-backed problems repo); keep only a pointer/path in Postgres — never BLOB megabytes of tests into a row.

Submit flow + Redis queue (Platform)

Submit endpoint (return fast)
// POST /submissions
const sub = await db.submission.create({
  data: { userId, problemId, code, language, status: 'PENDING' }
});
await redis.lpush('submissions', JSON.stringify({ id: sub.id, language }));
return Response.json({ id: sub.id }); // ~50ms, no judging inline
API writes PENDING + enqueues, then returns. Never run untrusted code in the request handler.
Worker pulls (blocking pop)
// judge worker loop
while (true) {
  const [, raw] = await redis.brpop('submissions', 0); // blocks
  const { id, language } = JSON.parse(raw);
  const verdict = await judge(id, language);            // Docker run
  await db.submission.update({ where: { id }, data: verdict });
  await pub.publish(`sub:${id}`, JSON.stringify(verdict)); // notify FE
}
Redis list = simplest broker. BullMQ (`new Queue/Worker`) adds retries, rate-limit, priority (contest > practice) for free.

Judge worker & Docker sandbox

Hardened container per submission
docker run --rm \
  --network none \                 # no internet / data exfil
  --read-only --tmpfs /tmp:size=64m \ # ro rootfs, scratch only
  --memory 256m --memory-swap 256m \  # MLE cap (swap==mem disables swap)
  --pids-limit 64 \                # kill fork bombs
  --cpus 1 --ulimit cpu=2 \        # CPU-TIME cap → SIGKILL
  --cap-drop ALL --security-opt no-new-privileges \
  -u 1000:1000 \                   # non-root
  judge-cpp /run.sh
memory-swap MUST equal memory or MLE never fires. gVisor/Firecracker for stronger isolation than plain Docker.
Compile once, then loop tests
// CE first — bail before running any test
if (compileExit !== 0) return { status: 'CE' };
for (const t of tests) {
  const { stdout, exitCode, timedOut, killedByOom } =
    await run(binary, t.input, { timeoutMs: timeLimitMs });
  if (timedOut)     return { status: 'TLE', failedTest: t.n };
  if (killedByOom)  return { status: 'MLE', failedTest: t.n };
  if (exitCode)     return { status: 'RE',  failedTest: t.n };
  if (norm(stdout) !== norm(t.expected))
    return { status: 'WA', failedTest: t.n };
}
return { status: 'AC', runtimeMs, memoryKb };
Compile once, reuse binary across all tests. norm() = trim trailing whitespace/newlines before diff.

Problem setter & boilerplate generator

Problem folder = source of truth
problems/two-sum/
  Problem.md        # statement (markdown)
  Structure.md      # I/O signature (parsed)
  tests/
    0.in  0.out
    1.in  1.out
  boilerplate/      # generated per language
    function.cpp function.js ...
Structure.md format
Problem Name: "Sum of Two"
Function Name: sumOfTwo
Input Structure:
Input Field: int a
Input Field: int b
Input Field: list<int> arr
Output Structure:
Output Field: int result
Generate stubs + I/O harness
const parser = new ProblemDefinitionParser();
parser.parse(readFileSync('Structure.md','utf8'));
// full = harness that reads stdin, calls fn, prints stdout
// partial = the stub shown to the user
new FullGenerator(parser).generateCpp()  -> boilerplate/function.cpp
new FullGenerator(parser).generateJs()   -> boilerplate/function.js
One canonical I/O contract → add a language or problem without hand-writing a harness per pair. A parser/test mismatch shows up as bogus WA that looks like a platform bug.

Verdict delivery (poll / SSE / WS)

Polling (MVP — do this first)
// FE polls until terminal status
while (true) {
  const { status } = await fetch(`/submissions/${id}`).then(r=>r.json());
  if (status !== 'PENDING') break;
  await sleep(1000);
}
Stateless, trivially scalable, good enough for MVP. Cost is redundant requests.
SSE (one-way push)
// GET /submissions/:id/stream
const sub = redis.duplicate();
await sub.subscribe(`sub:${id}`);
sub.on('message', (_c, msg) => controller.enqueue(`data: ${msg}\n\n`));
WebSocket + Redis pub/sub
Needed for live leaderboard/bidirectional. Worker publishes to Redis; the web node holding the socket subscribes — pub/sub bridges worker → correct socket across instances.

Fairness, limits & rating

CPU time, not wall time
Measure CPU time (ulimit cpu / cgroup cpuacct) so an overloaded host doesn't unfairly TLE a correct solution — keep a wall-clock backstop to kill infinite loops.
Per-language limit multiplier
const limit = base * ({ cpp:1, java:2, python:3 }[lang] ?? 1);
Slower runtimes get a higher/multiplied limit for the same problem. Warm the runtime to keep JIT/cold-start out of the budget.
Rating = post-contest batch
// after system tests + anti-cheat freeze standings:
for (const p of finalStandings) applyElo(p, field);
// store ratingHistory row per user (graphs + recompute)
Update once results are settled — NOT live during the round. Elo/Glicko-style: beat higher-rated → gain more.
Live leaderboard
redis.zadd(`contest:${cid}`, score, userId);
redis.zrevrange(`contest:${cid}`, 0, 49, 'WITHSCORES');
Redis sorted set for live standings; denormalized standings table for durable ranking — avoid full-table scans of submissions.

Quick tips & gotchas

  • --memory-swap MUST equal --memory or the OOM killer never fires and MLE submissions silently swap instead of failing.
  • Enforce limits with CPU time (ulimit cpu / cgroups), not wall-clock — wall time makes verdicts flaky under host load. Always keep a wall backstop.
  • Fresh, short-lived container per submission; never reuse one — sharing leaks state and hidden test data between users.
  • Never judge inline in the request handler: write PENDING + enqueue, return in ~50ms. Judging is CPU-bound and would collapse a web worker under a contest spike.
  • Compile once, reuse the binary across all tests; a compile failure is CE and must short-circuit before any test runs.
  • A boilerplate-generator / test-file I/O mismatch surfaces as spurious WA that looks like a platform bug — the setter's output is a load-bearing correctness surface.
  • Store test data in object storage with pointers in Postgres; keep Submission rows small and index (userId, problemId) + (contestId).
  • Compute leaderboards from a denormalized standings table / Redis sorted set — never rank by scanning the full submissions table per request.
