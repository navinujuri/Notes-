Rate limitting, DDoS and Captcha — Cheat Sheet

Cohort 2.0 · 10 lessons · quick reference

Why & Where to Rate Limit

Why
Stop brute-force/credential-stuffing, OTP guessing, scraping, spam signups, cost blow-ups (SMS/email/LLM), and app-layer DoS.
High-value endpoints to protect
/login, /signup, /forgot-password, /reset, /verify-otp, /payment, public search & any unauthenticated write. Auth + reset are #1 targets.
Defense layers (outer→inner)
Edge CDN/WAF (Cloudflare) → LB/API gateway → app middleware (express-rate-limit) → DB constraints. Never rely on a single layer.
Trust the proxy first
app.set('trust proxy', 1); // behind CF/nginx/ELB
// else req.ip = LB IP and every client shares one bucket
Without this, X-Forwarded-For is ignored and all users look like one IP.

express-rate-limit (app layer)

Basic global limiter
import rateLimit from 'express-rate-limit';
const limiter = rateLimit({
  windowMs: 60_000, limit: 100,
  standardHeaders: 'draft-7', legacyHeaders: false,
});
app.use('/api/', limiter);
Sends RateLimit-* headers; 429 on exceed.
Strict limiter on sensitive routes
const authLimiter = rateLimit({ windowMs: 15*60_000, limit: 5,
  message: 'Too many attempts, try later' });
app.post('/reset', authLimiter, resetHandler);
Tighter budget for OTP/reset/login than for read APIs.
Key by IP + identity (not IP alone)
keyGenerator: (req) => `${req.ip}:${req.body?.email ?? ''}`
Per-account budget so one shared NAT/office IP can't lock everyone out, and attacker can't spread guesses across accounts.
Distributed store (multi-instance)
import RedisStore from 'rate-limit-redis';
rateLimit({ windowMs: 60_000, limit: 100,
  store: new RedisStore({ sendCommand: (...a)=>redis.call(...a) }) });
In-memory limiter resets on restart and is per-process — useless behind >1 pod. Use Redis.
Slow-down instead of hard block
import slowDown from 'express-slow-down';
app.use(slowDown({ windowMs: 60_000, delayAfter: 20, delayMs: (h)=>h*200 }));
Adds latency progressively — friendlier for legit bursty users.

The Vulnerable Reset Endpoint (anti-pattern)

Naive forgot: 6-digit OTP in memory
const otpStore = {}; // email -> otp
app.post('/forgot', (req,res) => {
  const otp = Math.floor(100000 + Math.random()*900000);
  otpStore[req.body.email] = otp;      // no expiry
  sendEmail(req.body.email, otp);
  res.json({ ok: true });
});
Math.random() is not cryptographic; 6 digits = 10^6 space; never expires.
Naive reset: no attempt cap
app.post('/reset', (req,res) => {
  const { email, otp, newPassword } = req.body;
  if (otpStore[email] == otp) {        // == coercion, unlimited tries
    updatePassword(email, newPassword);
    return res.json({ ok: true });
  }
  res.status(400).json({ ok: false });
});
No rate limit + no lockout + loose == = brute-forceable.
Exploit: brute force all 6-digit OTPs
for (let otp = 100000; otp <= 999999; otp++) {
  const r = await fetch('/reset', { method:'POST',
    headers:{'content-type':'application/json'},
    body: JSON.stringify({ email:'victim@x.com', otp, newPassword:'pwned' }) });
  if (r.ok) { console.log('CRACKED', otp); break; }
}
At even 1k req/s, 10^6 codes fall in ~minutes. Production = same script + concurrency.

Saving the Endpoint (hardening)

Use a crypto token, not a 6-digit code
import crypto from 'crypto';
const token = crypto.randomBytes(32).toString('hex'); // 256-bit
const hash  = crypto.createHash('sha256').update(token).digest('hex');
// email link contains raw token; DB stores only hash + expiry
Unbrute-forceable search space; store the HASH so a DB leak can't reset accounts.
Persist with expiry + attempt counter
// resetTokens: { hash, email, expiresAt, attempts }
await db.resetTokens.create({ hash, email, attempts: 0,
  expiresAt: Date.now() + 15*60_000 });
Short TTL (10-15 min), single-use, invalidate on success/logout.
Timing-safe compare (if you keep short codes)
const ok = crypto.timingSafeEqual(Buffer.from(a), Buffer.from(b));
Prevents timing side-channel; buffers must be equal length. Prefer full tokens over OTPs.
Lockout after N attempts
if (row.attempts >= 5 || row.expiresAt < Date.now())
  return res.status(429).json({ error: 'Invalid or expired' });
await db.resetTokens.increment(row.id, 'attempts');
Combine with the rate limiter — layered, not either/or.
Prevent user enumeration
// /forgot: same 200 + generic message whether or not email exists
res.json({ message: 'If an account exists, a link was sent' });
Also keep response timing constant; don't leak 'user not found'.

The Problem: IP Limits Aren't Enough → CAPTCHA

Why IP rate limiting still fails
Attackers rotate IPs via botnets/residential proxies (thousands of IPs), so per-IP budgets never trip. Need a proof-of-humanity + edge protection.
Cloudflare Turnstile — client widget
<script src="https://challenges.cloudflare.com/turnstile/v0/api.js" async></script>
<div class="cf-turnstile" data-sitekey="0x4AA..."></div>
<!-- posts cf-turnstile-response with the form -->
Privacy-friendly, no image puzzles. reCAPTCHA is the Google equivalent.
Turnstile — server-side verify (required!)
const v = await fetch('https://challenges.cloudflare.com/turnstile/v0/siteverify', {
  method:'POST', headers:{'content-type':'application/x-www-form-urlencoded'},
  body: new URLSearchParams({ secret: process.env.TS_SECRET,
    response: req.body['cf-turnstile-response'], remoteip: req.ip }) });
if (!(await v.json()).success) return res.status(403).end();
Token is single-use and expires (~300s). Never trust the client widget alone.
reCAPTCHA v3 — score-based
const r = await (await fetch('https://www.google.com/recaptcha/api/siteverify', {
  method:'POST', body: new URLSearchParams({ secret, response: token }) })).json();
if (!r.success || r.score < 0.5) return res.status(403).end();
v3 returns 0.0-1.0; tune threshold. v2 is the checkbox/image challenge.

Cloudflare & DDoS Protection in Prod

Proxy DNS through Cloudflare
Set the record to Proxied (orange cloud). Traffic hits CF edge first — that's what enables WAF, rate limiting, and L3/4 DDoS scrubbing.
Hide / lock down the origin
If the real server IP leaks, attackers bypass CF entirely. Firewall origin to only accept CF IP ranges (or use cloudflared Tunnel — no public inbound port).
WAF rate limiting rule (edge)
# Security > WAF > Rate limiting rules
Expression: (http.request.uri.path eq "/reset")
Rate: 5 requests / 1 min  per  IP
Action: Block  (or Managed Challenge)
Blocks at the edge before load ever reaches origin — cheaper and faster than app-layer.
Under Attack Mode / Bot Fight Mode
# Overview > Under Attack Mode: ON  (JS challenge for all)
# Security > Bots > Bot Fight Mode: ON
Flip Under Attack on during an active L7 flood; it interstitials every visitor. Turn off after.
Managed Challenge vs hard Block
Managed Challenge lets humans through (Turnstile-style) while stopping bots — fewer false positives than outright Block on shared IPs.
L3/L4 volumetric DDoS
Handled automatically by CF's anycast network (and AWS Shield on AWS). Your job is L7: rate limits, WAF rules, CAPTCHA, caching static responses.

Quick tips & gotchas

  • Rate limit by identity (email/user/API key), not IP alone — botnets rotate thousands of IPs and NAT makes many users share one IP.
  • Set app.set('trust proxy', ...) behind any proxy/LB, or req.ip is the proxy and everyone shares a single bucket. Don't set it to true blindly — spoofable X-Forwarded-For.
  • In-memory rate limiters reset on restart and are per-process; anything running >1 instance needs a Redis/shared store.
  • 6-digit OTPs with no attempt cap fall in minutes. Prefer 256-bit crypto.randomBytes tokens; if you must use OTPs, cap attempts, expire fast, and single-use them.
  • Store a HASH of reset tokens/OTPs, never the raw value — a DB leak otherwise = account takeover.
  • Always verify CAPTCHA tokens server-side; the client widget passing means nothing. Turnstile/reCAPTCHA tokens are single-use and expire (~5 min).
  • Keep /forgot responses generic and constant-time ('if an account exists...') to block user enumeration.
  • CAPTCHA + edge WAF are defense-in-depth on top of app rate limiting, not replacements. And lock your origin to Cloudflare IPs or the whole edge is bypassable.
