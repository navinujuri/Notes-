Intro Class - DSA — Cheat Sheet

Cohort 2.0 · 7 lessons · quick reference

DSA vs CP + How to Approach

What is DSA
Data Structures (how you store data) + Algorithms (how you process it). Success = correct output at the best time/space Big-O, not just working code.
DSA vs CP
DSA = interview-focused: standard problems, correctness + optimal complexity. CP = timed contests: math tricks, observations, ranking/speed. For jobs start with DSA; CP later sharpens speed.
Big-O ladder (fastest to slowest)
O(1) < O(log n) < O(n) < O(n log n) < O(n^2) < O(2^n) < O(n!)
~10^8 simple ops/sec. Use constraints to pick target: n<=10^4 -> O(n^2) fine; n<=10^6 -> need O(n) or O(n log n).
Universal problem-solving flow
1) Brute force first (get it correct). 2) Find the bottleneck. 3) Optimize with hashing / sorting / two-pointers / prefix sums. Always eyeball edge cases before submitting.

Choice of Language & Fast I/O

Which language?
C++ is the DSA/CP default (STL + raw speed). Java = rich libs but verbose. Python = concise but slow (TLE risk). JS is fine for web devs. Logic is language-agnostic — pick one, master its std lib.
C++ competitive boilerplate
#include <bits/stdc++.h>
using namespace std;
int main() {
    ios_base::sync_with_stdio(false);
    cin.tie(NULL);
    int t; cin >> t;
    while (t--) {
        // solve()
    }
}
sync_with_stdio(false) + cin.tie(NULL) are mandatory for large inputs or you TLE on I/O alone.
Compile & run C++
g++ -std=c++17 -O2 sol.cpp -o sol && ./sol < in.txt
-O2 enables optimization; redirect stdin from a file for quick local testing.
JavaScript (Node) full-stdin read
const lines = require('fs').readFileSync(0, 'utf8').split('\n');
let i = 0;
const t = +lines[i++];
while (t--) {
  const [n, k] = lines[i++].split(' ').map(Number);
}
readFileSync(0) reads all of stdin at once. Batch output into one console.log — many small logs are slow.
Python fast input
import sys
input = sys.stdin.readline
t = int(input())
Default input() is slow for large T; rebinding to sys.stdin.readline avoids TLE.

Arrays / Lists

Declare & init
vector<int> a(n, 0);      // n zeros
vector<int> b = {1, 2, 3};
vector<vector<int>> g(n, vector<int>(m, 0)); // 2D
JS: const a = new Array(n).fill(0); let b = [1,2,3];
Add / remove
a.push_back(x); a.pop_back();   // back: O(1)
JS: a.push()/a.pop() O(1); a.shift()/a.unshift() are O(n) — avoid in loops.
Iterate
for (int x : a) cout << x << ' ';
for (int i = 0; i < a.size(); i++) { }
C++ operator[] has NO bounds check — out-of-range is undefined behavior, not an error.
Sort (asc / desc)
sort(a.begin(), a.end());                  // asc
sort(a.begin(), a.end(), greater<int>());  // desc
JS: a.sort((x,y)=>x-y). NEVER call bare a.sort() on numbers — it sorts lexicographically (10 < 2).
Max / min / sum / reverse
*max_element(a.begin(), a.end());
accumulate(a.begin(), a.end(), 0LL);  // 0LL -> long long
reverse(a.begin(), a.end());
JS: Math.max(...a); a.reduce((s,x)=>s+x,0); a.reverse(). Use 0LL seed to avoid int overflow on big sums.

Strings

Read input
string s; cin >> s;   // one token (stops at whitespace)
getline(cin, s);      // full line incl. spaces
After cin >> x, call cin.ignore() before getline() to drop the leftover newline.
Char <-> int
int d   = s[i] - '0';   // digit char -> value
int idx = c   - 'a';    // 'a'..'z' -> 0..25
char c  = 'a' + idx;
JS: +ch for digit, ch.charCodeAt(0)-97 for index, String.fromCharCode(97+idx).
Frequency count (fixed alphabet)
int cnt[26] = {0};
for (char c : s) cnt[c - 'a']++;
A size-26 array beats unordered_map when the alphabet is fixed — faster, no hashing.
Substring & reverse
s.substr(pos, len);
reverse(s.begin(), s.end());
JS: s.slice(start,end). JS strings are immutable — build with an array + arr.join('') for heavy concatenation.
Compare
if (s1 == s2) { }   // C++: value comparison
Java uses s1.equals(s2) (== compares references!). JS === is value comparison — fine.

Problem 1: Good Pairs (LeetCode 1512)

Problem
Count pairs (i, j) with i < j and nums[i] == nums[j].
Brute force O(n^2)
int cnt = 0;
for (int i = 0; i < n; i++)
  for (int j = i + 1; j < n; j++)
    if (a[i] == a[j]) cnt++;
Fine for n<=10^4; TLEs beyond. This is your baseline before optimizing.
Optimal O(n) — count-so-far
unordered_map<int,int> freq;
long long good = 0;
for (int x : a) { good += freq[x]; freq[x]++; }
Each new x forms a pair with every earlier equal value -> add freq[x] BEFORE incrementing. This trick kills most O(n^2) pair problems.
Formula variant
for (auto& [v, f] : freq) good += 1LL * f * (f - 1) / 2;
A value seen f times yields C(f,2)=f*(f-1)/2 pairs. Use 1LL* to avoid int overflow.
JS one-pass
const m = new Map(); let good = 0;
for (const x of a) { good += m.get(x)||0; m.set(x,(m.get(x)||0)+1); }

Problem 2: Chef and Happy Strings

Problem
A string is 'happy' if no two adjacent characters are equal. Count how many substrings are happy (i.e. alternating).
Optimal O(n) — running-run count
long long ans = 0, run = 0;
for (int i = 0; i < n; i++) {
    if (i > 0 && s[i] != s[i-1]) run++;
    else run = 1;
    ans += run;   // # happy substrings ending at i
}
run = length of the alternating streak ending at i = number of happy substrings ending there. Resets to 1 on a repeat.
Closed form per segment
// maximal alternating segment of length L -> L*(L+1)/2 substrings
Equivalent to the running-run sum; handy for a math-only derivation.
Variant: min deletions to make happy
int del = 0;
for (int i = 1; i < n; i++) if (s[i] == s[i-1]) del++;
Every adjacent equal pair needs one deletion. Watch for which variant the statement actually asks.

Problem 3 & Core Intro Patterns

Note on Problem 3
dsa7 is unlabeled in the course — a follow-up problem drilling the same toolkit below. Reach for these four patterns first on any intro array/string problem.
Hashing / frequency
Detect duplicates, anagrams, counts, first-unique. int cnt[26] for fixed alphabet, unordered_map<int,int> otherwise. Turns 'search for a match' into O(1) lookup.
Prefix sum (range queries O(1))
vector<long long> pre(n + 1, 0);
for (int i = 0; i < n; i++) pre[i+1] = pre[i] + a[i];
// sum of a[l..r] = pre[r+1] - pre[l]
O(n) build, O(1) per query. Core of the 'Intro Class 2' track (Highest Altitude, subarray sums).
Two pointers (sorted array)
int l = 0, r = n - 1;
while (l < r) {
    int s = a[l] + a[r];
    if (s == tgt) break;
    s < tgt ? l++ : r--;
}
Two-Sum on a sorted array in O(n) / O(1) space — no hashmap needed.
Two Sum via hashmap (unsorted)
unordered_map<int,int> seen;
for (int i = 0; i < n; i++) {
    if (seen.count(tgt - a[i])) { /* i, seen[tgt-a[i]] */ }
    seen[a[i]] = i;
}
O(n) when input isn't sorted; check for complement before inserting current.

Quick tips & gotchas

  • Read the constraints FIRST — n's magnitude picks your complexity, and value ranges pick your data type (int vs long long).
  • Counting pairs/substrings overflows 32-bit int fast: n*(n-1)/2 exceeds int around n~65536. Use long long in C++ (Number is safe to 2^53 in JS).
  • The 'add count-so-far, then increment' hashmap pattern collapses many O(n^2) pair problems to O(n) (see Good Pairs).
  • JS gotcha: [10,2,1].sort() gives [1,10,2]. Always pass a comparator: .sort((a,b)=>a-b).
  • C++ operator[] on vector/string is unchecked — out-of-range reads are UB, not exceptions. Guard your indices.
  • For a fixed alphabet use int cnt[26] instead of a hashmap — no hashing overhead, cache-friendly, and auto-sorted by letter.
  • Always compile C++ with -O2 and enable fast I/O (sync_with_stdio(false)); many 'wrong' TLEs are just slow I/O.
  • Test the four killer edge cases every time: empty, single element, all-same, all-distinct.
